# 42. Auditing Row Access Using QUERY_HISTORY in Snowflake

This concept explores how to leverage Snowflake's `QUERY_HISTORY` view to audit access to data, specifically focusing on how row access policies influence query results and what information can be gleaned about policy application. It's crucial for understanding data governance, security compliance, and debugging policy enforcement.

**Short Explanation**

Auditing row access policies using `QUERY_HISTORY` means examining past queries to see if and how row access policies affected the data returned. This helps database administrators verify that sensitive data is being protected as intended. It's important for ensuring compliance with data privacy regulations and maintaining data integrity. It's commonly used in security audits, debugging policy issues, and monitoring data access patterns.

-   **What problem does this SQL feature solve?** It solves the problem of verifying and monitoring the correct application of row access policies, ensuring that users only see the data they are authorized to access.
-   **Why is it important in databases or analytics?** It's vital for data security, compliance (e.g., GDPR, HIPAA), and understanding data usage patterns, especially in multi-tenant environments or when dealing with sensitive information.
-   **Where is it commonly used in real workflows?** In security compliance checks, post-incident analysis, policy debugging, and regular security reviews to ensure data access controls are functioning effectively.

## Implementation Example

```sql
SELECT
    query_id,
    query_text,
    user_name,
    start_time,
    end_time,
    execution_status,
    rows_produced,
    query_tag
FROM
    snowflake.account_usage.query_history
WHERE
    start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP()) -- Look at queries from the last 7 days
    AND query_text ILIKE '%SELECT%FROM%MY_SECURE_TABLE%' -- Filter for queries accessing a specific table
    AND query_tag LIKE '%ROW_ACCESS_AUDIT%' -- Optional: If you tag queries affected by policies
    AND query_id IN (
        SELECT
            query_id
        FROM
            snowflake.account_usage.access_history
        WHERE
            object_modified_by_row_access_policy = TRUE
            AND query_start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    )
ORDER BY
    start_time DESC;
```

## Explanation of the Code

This query analyzes the `QUERY_HISTORY` view, optionally joined with `ACCESS_HISTORY`, to find queries that interacted with a specific table and were potentially affected by row access policies.

-   **`SELECT query_id, query_text, user_name, start_time, end_time, execution_status, rows_produced, query_tag`**:
    -   **What it does**: Specifies the columns to retrieve from the `QUERY_HISTORY` view.
    -   **How it changes the result set**: These columns provide essential metadata about each query, including its unique ID, the SQL text executed, the user who ran it, start and end times, its success status, the number of rows returned, and any custom `QUERY_TAG` associated.

-   **`FROM snowflake.account_usage.query_history`**:
    -   **What it does**: Indicates that the data is being retrieved from Snowflake's `QUERY_HISTORY` view, which contains metadata for all queries executed in the account.
    -   **How it changes the result set**: This is the primary data source for the audit, providing raw query information before any filtering.

-   **`WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())`**:
    -   **What it does**: Filters queries to include only those executed within the last 7 days from the current timestamp.
    -   **How it changes the result set**: Narrows down the results to a recent time window, making the audit more focused and efficient.

-   **`AND query_text ILIKE '%SELECT%FROM%MY_SECURE_TABLE%'`**:
    -   **What it does**: Filters queries whose text contains "SELECT" and "FROM MY\_SECURE\_TABLE" (case-insensitive).
    -   **How it changes the result set**: Targets queries that specifically accessed the table you are interested in auditing for row access policy effects.

-   **`AND query_tag LIKE '%ROW_ACCESS_AUDIT%'`**:
    -   **What it does**: (Optional) Filters queries that have a `QUERY_TAG` containing "ROW\_ACCESS\_AUDIT". This assumes you've instrumented your policies or applications to set this tag when row access policies are active.
    -   **How it changes the result set**: Further refines the selection to potentially only include queries known to be subject to row access policies, if tags are used.

-   **`AND query_id IN (SELECT query_id FROM snowflake.account_usage.access_history WHERE object_modified_by_row_access_policy = TRUE AND query_start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP()))`**:
    -   **What it does**: This subquery identifies `query_id`s from `ACCESS_HISTORY` where a row access policy modified the accessed object (table/view) within the last 7 days. The outer query then uses these `query_id`s to filter `QUERY_HISTORY`.
    -   **How it changes the result set**: This is the most crucial part for auditing row access. It ensures that only queries where a row access policy was actively applied and potentially altered the result are included in the final output. `object_modified_by_row_access_policy` is a direct indicator of policy impact.

-   **`ORDER BY start_time DESC`**:
    -   **What it does**: Sorts the results in descending order based on the query start time.
    -   **How it changes the result set**: Presents the most recent relevant queries first, making it easier to review current activity.

## When to Use

1.  **Compliance Audits**: When auditors need to verify that data access controls (specifically row access policies) are effectively protecting sensitive data as per regulatory requirements (e.g., GDPR, HIPAA, SOX).
    *   *Scenario*: A company needs to prove that employees in department A cannot view salary information for employees in department B, even if they query the same `EMPLOYEES` table.
2.  **Debugging Row Access Policies**: When a row access policy isn't behaving as expected (e.g., users are seeing too much or too little data), `QUERY_HISTORY` combined with `ACCESS_HISTORY` can help diagnose why and how the policy was applied.
    *   *Scenario*: A user reports they can't see any data in a table, but they should see their own records. An audit helps determine if the policy was applied, what user ran the query, and if the policy logic needs adjustment.
3.  **Monitoring Data Access**: To proactively monitor who is accessing sensitive tables and how row access policies are influencing the data they retrieve, helping identify potential misuse or unexpected access patterns.
    *   *Scenario*: A security team wants to know if any non-administrative users have attempted to query a table containing highly confidential customer financial data, and confirm that the row access policy successfully filtered their results.

## When Not to Use

1.  **Real-time Enforcement**: `QUERY_HISTORY` and `ACCESS_HISTORY` are for *auditing* past events, not for *enforcing* policies in real-time. Policies themselves handle real-time filtering.
    *   *Situation*: You cannot use `QUERY_HISTORY` to dynamically prevent a user from executing a query or seeing specific rows *as they happen*. This is the job of the row access policy itself.
2.  **Fine-grained Performance Metrics for Individual Queries**: While `QUERY_HISTORY` provides `execution_status` and `rows_produced`, it's not the primary tool for detailed, low-level performance analysis or query plan investigation for a single problematic query.
    *   *Situation*: If you need to optimize a slow query, `QUERY_HISTORY` gives you the basic timing, but you'd typically use `QUERY_PROFILE` or `EXPLAIN` for deep dive performance analysis, not just `QUERY_HISTORY`'s output.
3.  **General User Activity Tracking Without Policy Impact**: If the goal is simply to track all user queries, regardless of whether a row access policy was involved, filtering by `object_modified_by_row_access_policy = TRUE` is an unnecessary and restrictive filter.
    *   *Situation*: A business team wants a report of all queries run by their department last week. Using the row access policy specific filters would exclude many relevant queries.

## Common Mistakes

1.  **Ignoring `ACCESS_HISTORY`**: Relying solely on `QUERY_HISTORY` and `query_text` patterns to infer row access policy application. `QUERY_HISTORY` alone won't tell you if a policy *actually* affected the result; `ACCESS_HISTORY`'s `object_modified_by_row_access_policy` column is key.
2.  **Not Understanding Data Latency**: `ACCOUNT_USAGE` views (like `QUERY_HISTORY` and `ACCESS_HISTORY`) have a latency of up to 120 minutes. Auditing results may not reflect actions taken just moments ago.
3.  **Overly Broad or Narrow Time Windows**: Searching too far back can lead to massive result sets and slow queries. Searching too narrowly might miss relevant events. Balancing the `start_time` filter is crucial.
4.  **Misinterpreting `rows_produced`**: A low `rows_produced` count for a sensitive table might indicate a successful policy application, but it could also mean the user simply queried for data that doesn't exist or is genuinely sparse. Context is vital.

## Expected Output

The resulting dataset will be a list of queries that accessed `MY_SECURE_TABLE` (or any specified table) within the defined time frame, and for which a row access policy was applied, potentially modifying the result set. Each row will represent a single query with its key metadata.

| QUERY_ID                             | QUERY_TEXT                                                      | USER_NAME  | START_TIME                | END_TIME                  | EXECUTION_STATUS | ROWS_PRODUCED | QUERY_TAG           |
| :----------------------------------- | :-------------------------------------------------------------- | :--------- | :------------------------ | :------------------------ | :--------------- | :------------ | :------------------ |
| `01b2c3d4-e5f6-7890-1234-567890abcdef` | `SELECT * FROM MY_SECURE_TABLE WHERE REGION = 'US';`           | `analyst1` | `2026-03-13 10:30:00.000` | `2026-03-13 10:30:01.200` | `SUCCESS`        | `150`         | `ROW_ACCESS_AUDIT`  |
| `01c2d3e4-f5a6-7890-1234-567890abcdef` | `SELECT COUNT(*) FROM MY_SECURE_TABLE;`                         | `manager2` | `2026-03-12 14:15:30.000` | `2026-03-12 14:15:30.800` | `SUCCESS`        | `1`           | `ROW_ACCESS_AUDIT`  |
| `01d2e3f4-a5b6-7890-1234-567890abcdef` | `SELECT CUSTOMER_ID, BALANCE FROM MY_SECURE_TABLE LIMIT 10;` | `admin_user` | `2026-03-11 09:00:05.000` | `2026-03-11 09:00:06.100` | `SUCCESS`        | `10`          | `FULL_ACCESS_QUERY` |

-   **QUERY\_ID**: Unique identifier for the executed query.
-   **QUERY\_TEXT**: The full SQL statement executed by the user.
-   **USER\_NAME**: The Snowflake user who ran the query.
-   **START\_TIME**: When the query began execution.
-   **END\_TIME**: When the query finished execution.
-   **EXECUTION\_STATUS**: Indicates if the query completed successfully or failed.
-   **ROWS\_PRODUCED**: The number of rows returned by the query *after* any row access policy filtering.
-   **QUERY\_TAG**: An optional custom tag that can be set for queries, potentially indicating if a policy was expected to be involved.
