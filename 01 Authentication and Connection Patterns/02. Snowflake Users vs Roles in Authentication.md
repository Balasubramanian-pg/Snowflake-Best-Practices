[02. Snowflake Users vs Roles in Authentication](01 Authentication and Connection Patterns/02. Snowflake Users vs Roles in Authentication.md)

EXPLANATION
A User object represents a unique identity (human or service) responsible for establishing a connection via authentication credentials such as passwords or key pairs. A Role object is a named collection of privileges granted to users, defining the authorization scope and specific access rights available during an active session.

KEY POINTS

*   **Identity vs. Authorization**: The User object answers the question "Who is connecting?" while the Role object answers "What can they do?". Authentication validates the User; Authorization validates the Role.
*   **Credential Ownership**: Authentication mechanisms (passwords, RSA keys for key-pair authentication, or SSO configurations) are properties of the User object, never the Role.
*   **Role-Based Access Control (RBAC)**: Snowflake enforces a strict RBAC model where privileges (e.g., `SELECT` on a table) are granted to Roles, and Roles are granted to Users.
*   **Many-to-Many Relationship**: A single User can be assigned multiple Roles, and a single Role can be assigned to multiple Users, allowing for flexible organizational modeling.
*   **Session Context**: When a User authenticates, they must assume a specific Role (either explicitly or via a configured `DEFAULT_ROLE`) to perform any actions, as Users themselves hold no direct privileges on data objects.

> [!IMPORTANT]
> While Snowflake technically allows granting privileges directly to a User, it is strongly discouraged. Best practice dictates that all privileges be assigned to Roles to maintain a clean, auditable security hierarchy.

> [!NOTE]
> A User can switch between granted Roles within a single session using the `USE ROLE` command without re-authenticating, provided they have been granted those roles.

SQL EXAMPLE
The following SQL demonstrates the separation of concerns: creating an authorization container (Role), creating an identity (User), and linking them.

SQL

```sql
-- 1. Switch to a role capable of managing security objects
USE ROLE SECURITYADMIN;

-- 2. Create the Role (The Authorization Container)
CREATE OR REPLACE ROLE data_analyst_role
    COMMENT = 'Role for standard data analysis tasks';

-- 3. Grant Privileges to the Role (Not the User)
GRANT USAGE ON WAREHOUSE compute_wh TO ROLE data_analyst_role;
GRANT USAGE ON DATABASE sales_db TO ROLE data_analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE data_analyst_role;

-- 4. Create the User (The Identity)
CREATE OR REPLACE USER jane_doe
    PASSWORD = 'ComplexPassword123!'
    LOGIN_NAME = 'jdoe'
    FIRST_NAME = 'Jane'
    LAST_NAME = 'Doe'
    EMAIL = 'jane.doe@example.com'
    DEFAULT_ROLE = data_analyst_role
    MUST_CHANGE_PASSWORD = TRUE;

-- 5. Grant the Role to the User (Linking Identity to Authorization)
GRANT ROLE data_analyst_role TO USER jane_doe;
```

CODE EXPLANATION

*   **Step 1**: The session context is set to `SECURITYADMIN`, which is the system-defined role responsible for managing users and roles.
*   **Step 2**: A new role named `data_analyst_role` is created. At this stage, it has no permissions and is assigned to no one.
*   **Step 3**: Specific access rights (Usage on warehouse/database and Select on tables) are granted to the *Role*. This defines what the role is allowed to do.
*   **Step 4**: A new user `jane_doe` is created with authentication details (Password). The `DEFAULT_ROLE` property ensures that when Jane logs in, she automatically assumes the `data_analyst_role` context.
*   **Step 5**: The final step explicitly grants the `data_analyst_role` to `jane_doe`. Without this step, Jane could authenticate (log in) but would not be able to assume the role or access the data defined in Step 3.

WHEN TO USE

*   **Employee Onboarding**: When provisioning access for a new team member, you create a unique User for auditability but assign them an existing Role (e.g., `marketing_analyst`) shared by their peers.
*   **Service Account Creation**: When configuring ETL tools or BI dashboards, you create a specific User for the application (e.g., `tableau_service_user`) and assign it a Role with the minimum privileges required to extract data.
