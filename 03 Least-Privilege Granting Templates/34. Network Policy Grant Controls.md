# Network Policy Grant Controls in Snowflake

This concept addresses how Snowflake administrators manage network access to Snowflake accounts, ensuring that only approved IP addresses can connect. It exists to enhance security by restricting incoming traffic to a defined set of trusted networks.

**Short Explanation**

Network Policy Grant Controls in Snowflake allow you to define rules that specify which IP addresses are permitted or blocked from accessing your Snowflake account. It solves the problem of unauthorized network access, acting as a virtual firewall for your data warehouse. This is important for data security and compliance, ensuring that sensitive data is only accessible from secure environments. It's commonly used in organizations to enforce strict access policies, especially for production environments or highly regulated data.

- What problem does this SQL feature solve? It solves the problem of unauthorized network access to a Snowflake account.
- Why is it important in databases or analytics? It's crucial for security and compliance, preventing data breaches by restricting access to known, secure networks.
- Where is it commonly used in real workflows? In enterprise environments to secure production Snowflake accounts, often integrated with corporate VPNs or specific office IP ranges.

## Implementation Example

```sql
-- 1. Create a Network Policy
CREATE NETWORK POLICY my_network_policy
    ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.1')
    BLOCKED_IP_LIST = ('192.168.1.100');

-- 2. Grant the Network Policy to a user
ALTER USER jsmith SET NETWORK_POLICY = my_network_policy;

-- 3. Grant the Network Policy to an account (as default for all users)
ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy;
```

## Explanation of the Code

- **`CREATE NETWORK POLICY my_network_policy`**:
    - **What it does**: This statement defines a new network policy named `my_network_policy`.
    - **How it changes the result set**: It creates a named security object in Snowflake that contains network access rules.

- **`ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.1')`**:
    - **What it does**: Specifies a list of IP addresses or CIDR ranges that are permitted to connect.
    - **How it changes the result set**: Any connection attempts from IPs not in this list (unless explicitly blocked) will be denied if this policy is active.

- **`BLOCKED_IP_LIST = ('192.168.1.100')`**:
    - **What it does**: Specifies a list of IP addresses or CIDR ranges that are explicitly denied connection, even if they are also included in `ALLOWED_IP_LIST`. This acts as an override.
    - **How it changes the result set**: Connections from these IPs will always be denied if this policy is active.

- **`ALTER USER jsmith SET NETWORK_POLICY = my_network_policy;`**:
    - **What it does**: Applies the `my_network_policy` to a specific user named `jsmith`.
    - **How it changes the result set**: User `jsmith` will now be subject to the network access rules defined in `my_network_policy`.

- **`ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy;`**:
    - **What it does**: Applies the `my_network_policy` as the default for the entire Snowflake account.
    - **How it changes the result set**: All users in the account, unless they have a specific user-level network policy, will be subject to `my_network_policy`. User-level policies take precedence over account-level policies.

## When to Use

1.  **Securing Production Environments**: When you want to ensure that your production Snowflake account is only accessible from your corporate network, VPN, or specific cloud environments.
    *   *Scenario*: Your data analysts access Snowflake from the office (IP range `203.0.113.0/24`) or via a corporate VPN (IP `198.51.100.5`). You apply a network policy with these IPs to the account.
2.  **Compliance Requirements**: When regulatory frameworks (like HIPAA, GDPR, PCI DSS) mandate strict control over data access points.
    *   *Scenario*: Your company handles sensitive customer data, and auditors require proof that access is restricted to a limited, secure set of IP addresses. You implement and document network policies.
3.  **Restricting Third-Party Tool Access**: To limit access for integration tools or third-party applications to their specific, known outbound IP addresses.
    *   *Scenario*: A data integration tool connects to Snowflake from a fixed IP address (`203.0.113.10`). You create a network policy allowing only this IP for the dedicated service account user.

## When Not to Use

1.  **Overly Restrictive for Development/Test Environments**: Applying very strict network policies to non-production environments can hinder development and testing.
    *   *Scenario*: Developers frequently work from various locations or need to use different tools with dynamic IPs. A strict policy would block their legitimate access, causing frustration and inefficiency.
2.  **Dynamic IP Addresses for Users**: When users legitimately access Snowflake from various locations with frequently changing public IP addresses (e.g., traveling employees without VPN access).
    *   *Scenario*: A sales team accesses reports via Snowflake from client sites or coffee shops, where their public IP changes constantly. Applying a network policy would prevent them from connecting.
3.  **Single Sign-On (SSO) with IdP-initiated Flow**: While network policies secure direct connections, for browser-based SSO, the initial connection might come from the IdP. However, the policy still applies to the user's subsequent access attempts. Overly broad blocking could inadvertently affect legitimate SSO.
    *   *Scenario*: Your organization uses Okta for SSO. If you block common proxy IPs without considering how Okta proxies connections, users might be unable to log in, even though the issue isn't direct login but the network path taken. (This is more nuanced, but generally, network policies secure the *Snowflake connection*.)

## Common Mistakes

1.  **Forgetting to Apply the Policy**: Creating a network policy but not assigning it to any user or the account, rendering it ineffective.
2.  **Incorrect CIDR Notation**: Misunderstanding CIDR blocks, leading to either overly permissive or overly restrictive policies (e.g., using `/32` for a range, or `/24` when only a single IP is intended).
3.  **Confusing `ALLOWED_IP_LIST` and `BLOCKED_IP_LIST`**: Not understanding that `BLOCKED_IP_LIST` takes precedence, or listing IPs in `BLOCKED_IP_LIST` that are not expected to be in `ALLOWED_IP_LIST`.
4.  **Not Testing Thoroughly**: Deploying a network policy without testing access from both allowed and blocked IPs, leading to accidental lockouts.
5.  **Lack of Documentation/Change Management**: Not documenting which IPs are included and why, or failing to update policies when network configurations change, causing unexpected access issues.

## Expected Output

The operations (`CREATE NETWORK POLICY`, `ALTER USER SET NETWORK_POLICY`, `ALTER ACCOUNT SET NETWORK_POLICY`) do not return a dataset in the traditional sense, but rather a confirmation message indicating success.

For example, after `CREATE NETWORK POLICY my_network_policy ...`, the output would typically be:

```
Statement executed successfully.
```

If you query `SHOW NETWORK POLICIES;`, you would see a row for `my_network_policy`:

| created_on | name | comment | entries | owner | owner_role_type |
| :--------- | :--- | :------ | :------ | :---- | :-------------- |
| YYYY-MM-DD HH:MM:SS.SSS -0700 | MY_NETWORK_POLICY | | ALLOWED: 192.168.1.0/24, 10.0.0.1 BLOCKED: 192.168.1.100 | ACCOUNTADMIN | ROLE |

- **created_on**: Timestamp of when the policy was created.
- **name**: The name of the network policy.
- **comment**: Any optional comment added to the policy.
- **entries**: A summary of the allowed and blocked IP lists.
- **owner**: The role that owns the network policy.
- **owner_role_type**: The type of role.

This table shows the existence and configuration of the network policy itself. The "output" in terms of user experience is the enforcement of access rules: permitted connections succeed, and non-permitted connections fail with an access error.
