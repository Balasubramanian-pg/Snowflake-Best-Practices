# BI Tool Access Control in Snowflake

This concept addresses how to manage and secure data access for Business Intelligence (BI) tools and their users within a Snowflake data warehouse. It ensures that users connecting via BI tools only see and interact with the data they are authorized to access, aligning with an organization's security policies and compliance requirements.

**Short Explanation**

BI Tool Access Control in Snowflake means setting up permissions so that when people use tools like Tableau or Power BI, they only see data they're allowed to see. This prevents unauthorized data exposure, keeps sensitive information safe, and ensures compliance with regulations. It's crucial for maintaining data governance across all reporting and analytical activities.

- **What problem does this SQL feature solve?** It solves the problem of data security and governance when integrating third-party BI tools, preventing unauthorized data access or accidental exposure of sensitive information.
- **Why is it important in databases or analytics?** It's vital for maintaining data integrity, confidentiality, and compliance (e.g., GDPR, HIPAA) across all data consumption layers, especially when data is accessed by diverse user roles.
- **Where is it commonly used in real workflows?** It's used in virtually every organization that uses Snowflake as a data warehouse and connects BI tools for reporting, dashboards, and ad-hoc analysis.

## Implementation Example

Let's imagine we have a sales team and a marketing team, and each should only see their respective sales or marketing data. We'll use roles and secure views to enforce this.

```sql
-- 1. Create base tables
CREATE TABLE SALES.PUBLIC.ORDERS (
    ORDER_ID INTEGER,
    CUSTOMER_ID INTEGER,
    PRODUCT VARCHAR,
    AMOUNT DECIMAL(10, 2),
    ORDER_DATE DATE
);

CREATE TABLE MARKETING.PUBLIC.CAMPAIGNS (
    CAMPAIGN_ID INTEGER,
    CAMPAIGN_NAME VARCHAR,
    BUDGET DECIMAL(10, 2),
    START_DATE DATE,
    END_DATE DATE
);

-- Insert some dummy data
INSERT INTO SALES.PUBLIC.ORDERS VALUES
(1, 101, 'Laptop', 1200.00, '2026-01-15'),
(2, 102, 'Monitor', 300.00, '2026-01-16');

INSERT INTO MARKETING.PUBLIC.CAMPAIGNS VALUES
(10, 'Winter Sale', 5000.00, '2026-02-01', '2026-02-28'),
(11, 'Spring Promo', 7500.00, '2026-03-01', '2026-03-31');

-- 2. Create roles for BI users
CREATE ROLE BI_SALES_ANALYST;
CREATE ROLE BI_MARKETING_ANALYST;

-- 3. Grant access to BI roles on relevant databases/schemas
GRANT USAGE ON DATABASE SALES TO ROLE BI_SALES_ANALYST;
GRANT USAGE ON SCHEMA SALES.PUBLIC TO ROLE BI_SALES_ANALYST;
GRANT SELECT ON ALL TABLES IN SCHEMA SALES.PUBLIC TO ROLE BI_SALES_ANALYST;

GRANT USAGE ON DATABASE MARKETING TO ROLE BI_MARKETING_ANALYST;
GRANT USAGE ON SCHEMA MARKETING.PUBLIC TO ROLE BI_MARKETING_ANALYST;
GRANT SELECT ON ALL TABLES IN SCHEMA MARKETING.PUBLIC TO ROLE BI_MARKETING_ANALYST;

-- Optional: Create secure views for more granular control (e.g., row-level security)
CREATE SECURE VIEW SALES.PUBLIC.VW_SALES_DATA AS
SELECT ORDER_ID, CUSTOMER_ID, PRODUCT, AMOUNT, ORDER_DATE
FROM SALES.PUBLIC.ORDERS
WHERE AMOUNT > 200; -- Example: Only show orders above a certain amount for BI sales

CREATE SECURE VIEW MARKETING.PUBLIC.VW_MARKETING_DATA AS
SELECT CAMPAIGN_ID, CAMPAIGN_NAME, BUDGET, START_DATE, END_DATE
FROM MARKETING.PUBLIC.CAMPAIGNS
WHERE BUDGET > 1000; -- Example: Only show campaigns above a certain budget for BI marketing

-- Revoke select from base tables if using secure views, and grant on views instead
REVOKE SELECT ON ALL TABLES IN SCHEMA SALES.PUBLIC FROM ROLE BI_SALES_ANALYST;
GRANT SELECT ON VIEW SALES.PUBLIC.VW_SALES_DATA TO ROLE BI_SALES_ANALYST;

REVOKE SELECT ON ALL TABLES IN SCHEMA MARKETING.PUBLIC FROM ROLE BI_MARKETING_ANALYST;
GRANT SELECT ON VIEW MARKETING.PUBLIC.VW_MARKETING_DATA TO ROLE BI_MARKETING_ANALYST;

-- 4. Assign users to roles (example user 'BI_USER_SALES' and 'BI_USER_MARKETING')
CREATE USER BI_USER_SALES PASSWORD='password123' DEFAULT_ROLE=BI_SALES_ANALYST;
CREATE USER BI_USER_MARKETING PASSWORD='password456' DEFAULT_ROLE=BI_MARKETING_ANALYST;

GRANT ROLE BI_SALES_ANALYST TO USER BI_USER_SALES;
GRANT ROLE BI_MARKETING_ANALYST TO USER BI_USER_MARKETING;

-- When BI_USER_SALES connects via a BI tool, they will only see data from VW_SALES_DATA
-- When BI_USER_MARKETING connects, they will only see data from VW_MARKETING_DATA
```

## Explanation of the Code

This code sets up a fundamental access control model in Snowflake for BI tools, leveraging roles and secure views.

- **`CREATE TABLE`**: Defines two base tables, `ORDERS` and `CAMPAIGNS`, simulating raw sales and marketing data.
- **`INSERT INTO`**: Populates the base tables with sample data.
- **`CREATE ROLE BI_SALES_ANALYST; CREATE ROLE BI_MARKETING_ANALYST;`**: These statements define distinct roles for different groups of BI users.
    - **What it does**: Creates logical groups to which privileges can be assigned.
    - **How it changes the result set**: It doesn't directly change data but establishes identities for privilege assignment.
- **`GRANT USAGE ON DATABASE ... TO ROLE ...;`**: Grants the ability to access specific databases.
    - **What it does**: Allows a role to "see" and interact with a database.
    - **How it changes the result set**: No direct change to data; it's a foundational permission.
- **`GRANT USAGE ON SCHEMA ... TO ROLE ...;`**: Grants the ability to access specific schemas within a database.
    - **What it does**: Allows a role to "see" and interact with a schema.
    - **How it changes the result set**: Similar to database usage, it's a foundational permission.
- **`GRANT SELECT ON ALL TABLES IN SCHEMA ... TO ROLE ...;`**: Grants read access to all existing tables within a specified schema to a role.
    - **What it does**: Allows users with this role to query data from these tables.
    - **How it changes the result set**: Users can now retrieve rows from these tables.
- **`CREATE SECURE VIEW SALES.PUBLIC.VW_SALES_DATA AS ...;`**: Creates a secure view over the `ORDERS` table, adding a filtering condition (e.g., `AMOUNT > 200`).
    - **What it does**: A view acts as a virtual table. `SECURE` means its definition (the underlying query) is hidden from users who can query the view. It provides an abstraction layer and can enforce row-level security.
    - **How it changes the result set**: Filters the rows returned based on the `WHERE` clause defined in the view, and potentially hides underlying table structure.
- **`REVOKE SELECT ON ALL TABLES ... FROM ROLE ...;`**: Removes previous `SELECT` privileges.
    - **What it does**: Removes the ability for a role to query specific objects.
    - **How it changes the result set**: Users can no longer retrieve data from the revoked objects.
- **`GRANT SELECT ON VIEW ... TO ROLE ...;`**: Grants read access to the newly created secure views.
    - **What it does**: Allows roles to query the secure views.
    - **How it changes the result set**: Users can now retrieve rows *filtered by the view's definition* from these views.
- **`CREATE USER BI_USER_SALES ... DEFAULT_ROLE=BI_SALES_ANALYST;`**: Creates new users and assigns them a default role.
    - **What it does**: Establishes user accounts that BI tools will use for connecting. The `DEFAULT_ROLE` ensures they automatically assume the correct permissions upon login.
    - **How it changes the result set**: No direct change to data; it's about user authentication and initial privilege context.
- **`GRANT ROLE BI_SALES_ANALYST TO USER BI_USER_SALES;`**: Assigns roles to users.
    - **What it does**: Makes the defined roles available for users to activate (or automatically use as default).
    - **How it changes the result set**: No direct change; it links users to their potential access privileges.

## When to Use

1.  **Enforcing Data Security and Compliance**: When sensitive data (e.g., PII, financial records) is present, and you need to ensure only authorized users or teams can access it, especially through third-party tools.
    *   **Scenario**: A healthcare organization uses Snowflake for patient data and needs to ensure that only doctors can see patient health records, while administrative staff can only see appointment schedules. BI tool access control, possibly with row-level security views based on user roles, is critical.
2.  **Streamlining Access for Different Business Units**: When various departments (sales, marketing, finance) need access to different subsets of data for their BI dashboards and reports, without seeing each other's specific metrics.
    *   **Scenario**: A global retail company has sales data segmented by region. Regional sales managers should only see data for their specific region in their Tableau dashboards. Roles and secure views can filter data based on the connecting user's assigned region.
3.  **Simplifying Data Consumption for BI Developers**: Providing curated, pre-filtered, or joined views to BI developers or data analysts so they don't have to write complex queries against raw tables, and are implicitly constrained by access policies.
    *   **Scenario**: Data analysts need to build reports on customer lifetime value, which requires joining several tables and aggregating data. A secure view provides the pre-joined, pre-calculated data, ensuring they only get relevant, authorized segments, making their BI development faster and safer.

## When Not to Use

1.  **Overly Granular Security for Small Datasets or Non-Sensitive Data**: Applying overly complex row-level or column-level security to small, non-sensitive datasets where the overhead of managing these rules outweighs the security benefit.
    *   **Reason**: If data is public or non-confidential, and the dataset is small, the complexity of maintaining intricate access controls for BI tools can introduce unnecessary overhead and performance impacts without significant gain.
2.  **When Performance is Critically Impacted by Complex Views**: If secure views involve very complex joins or filters that significantly degrade query performance for BI tools, impacting user experience.
    *   **Reason**: Secure views add a layer of processing. While generally optimized, extremely complex views, especially those with UDFs or numerous joins for every query, can slow down BI dashboards. In such cases, alternative strategies like materializing data marts with pre-aggregated, pre-filtered data might be better.
3.  **For Simple Data Sharing with Known and Trusted Partners**: If sharing entire datasets with a single, trusted partner or internal team who requires full access to a specific schema and has their own robust internal security.
    *   **Reason**: In situations where a full schema or database is intended to be shared without any internal differentiation in access, granting direct permissions to a dedicated role (or even the `PUBLIC` role for truly public data) might be simpler and more efficient than building multiple layers of views and roles that aren't strictly necessary.

## Common Mistakes

1.  **Not Using Dedicated BI Roles**: Granting `ACCOUNTADMIN` or `SYSADMIN` to BI service accounts or users, bypassing all security mechanisms.
    *   **Issue**: This provides unrestricted access to all data, undermining the entire security model and posing a massive data breach risk.
2.  **Forgetting `USAGE` Grants**: Creating roles and granting `SELECT` on tables/views but forgetting to grant `USAGE` on the parent database and schema.
    *   **Issue**: Users will get "Object does not exist" errors because they can't navigate to the database or schema where the objects reside, even if they have `SELECT` on the object itself.
3.  **Over-reliance on `PUBLIC` Role**: Granting broad permissions (like `SELECT` on schemas or tables) to the `PUBLIC` role.
    *   **Issue**: Any user created in Snowflake automatically inherits the `PUBLIC` role. This means if you grant `SELECT` on sensitive data to `PUBLIC`, every user can see it, irrespective of other roles assigned to them.
4.  **Inadequate Testing with BI Tools**: Not thoroughly testing the access control policies using the actual BI tool and various user accounts.
    *   **Issue**: What looks correct in Snowflake's UI or command line might behave differently or expose unintended data when accessed through a BI tool due to different connection settings, default roles, or context.
5.  **Not Documenting Role Hierarchies and Grants**: Failing to clearly document who has access to what, and why.
    *   **Issue**: Leads to confusion, difficulty auditing, and "role creep" where users accumulate too many permissions over time, making it hard to manage security effectively.

## Expected Output

When a BI tool connects with a user assigned to `BI_SALES_ANALYST` role and queries `SALES.PUBLIC.VW_SALES_DATA`, it will retrieve a dataset containing only sales orders where the `AMOUNT` is greater than 200.

| ORDER\_ID | CUSTOMER\_ID | PRODUCT  | AMOUNT | ORDER\_DATE |
| :-------- | :----------- | :------- | :----- | :---------- |
| 1         | 101          | Laptop   | 1200.00| 2026-01-15  |

-   **ORDER\_ID**: A unique identifier for each sales order.
-   **CUSTOMER\_ID**: A unique identifier for the customer who placed the order.
-   **PRODUCT**: The name of the product sold.
-   **AMOUNT**: The monetary value of the order, filtered to be above 200.
-   **ORDER\_DATE**: The date the order was placed.

Conversely, if a BI tool connects with a user assigned to `BI_MARKETING_ANALYST` role and queries `MARKETING.PUBLIC.VW_MARKETING_DATA`, it will retrieve a dataset containing only marketing campaigns where the `BUDGET` is greater than 1000.

| CAMPAIGN\_ID | CAMPAIGN\_NAME | BUDGET  | START\_DATE | END\_DATE |
| :---------- | :------------- | :------ | :---------- | :-------- |
| 10          | Winter Sale    | 5000.00 | 2026-02-01  | 2026-02-28|
| 11          | Spring Promo   | 7500.00 | 2026-03-01  | 2026-03-31|

-   **CAMPAIGN\_ID**: A unique identifier for the marketing campaign.
-   **CAMPAIGN\_NAME**: The name of the marketing campaign.
-   **BUDGET**: The allocated budget for the campaign, filtered to be above 1000.
-   **START\_DATE**: The start date of the campaign.
-   **END\_DATE**: The end date of the campaign.
