# Conditional Masking Based on User Role in Snowflake

Conditional masking based on user role in Snowflake is a powerful data governance feature that allows you to dynamically redact or obfuscate sensitive data in a column based on the role of the user querying the data. This ensures that only authorized users see the full, unmasked data, while others see a masked version, enhancing data security and compliance without creating multiple versions of the data.

**Short Explanation**

This SQL feature solves the problem of fine-grained access control to sensitive data at the column level. It's important in databases and analytics for maintaining data privacy, adhering to regulatory requirements like GDPR or HIPAA, and preventing unauthorized data exposure. It's commonly used in real workflows to protect personally identifiable information (PII), financial data, or confidential business metrics when accessed by different user groups (e.g., analysts vs. data stewards).

- What problem does this SQL feature solve? It solves the problem of controlling access to specific data within a column, showing different data to different users based on their security roles.
- Why is it important in databases or analytics? It's crucial for data security, privacy compliance, and enabling data sharing while minimizing risk.
- Where is it commonly used in real workflows? It's used to mask sensitive columns like email addresses, social security numbers, or salary information for users who don't have the explicit role to view them.

## Implementation Example

```sql
-- 1. Create a sample table with sensitive data
CREATE TABLE employees (
    employee_id INT,
    employee_name VARCHAR,
    email_address VARCHAR,
    salary DECIMAL(10, 2)
);

-- 2. Insert sample data
INSERT INTO employees (employee_id, employee_name, email_address, salary) VALUES
(1, 'Alice Smith', 'alice.s@example.com', 75000.00),
(2, 'Bob Johnson', 'bob.j@example.com', 82000.00),
(3, 'Charlie Brown', 'charlie.b@example.com', 68000.00);

-- 3. Create a masking policy
CREATE MASKING POLICY email_masking_policy AS (val VARCHAR) RETURNS VARCHAR ->
    CASE
        WHEN CURRENT_ROLE() IN ('HR_ANALYST', 'SECURITY_ADMIN') THEN val
        ELSE '**********' || SUBSTR(val, INSTR(val, '@')) -- Masks everything before '@'
    END;

CREATE MASKING POLICY salary_masking_policy AS (val DECIMAL(10, 2)) RETURNS DECIMAL(10, 2) ->
    CASE
        WHEN CURRENT_ROLE() = 'HR_ANALYST' THEN val
        ELSE 0.00
    END;

-- 4. Apply the masking policies to the columns
ALTER TABLE employees MODIFY COLUMN email_address SET MASKING POLICY email_masking_policy;
ALTER TABLE employees MODIFY COLUMN salary SET MASKING POLICY salary_masking_policy;

-- 5. Test with different roles (assuming roles exist and user is granted these)

-- As a HR_ANALYST (full access)
-- USE ROLE HR_ANALYST;
-- SELECT * FROM employees;

-- As a FINANCE_ANALYST (email masked, salary masked)
-- USE ROLE FINANCE_ANALYST;
-- SELECT * FROM employees;

-- As a PUBLIC role (email masked, salary masked)
-- USE ROLE PUBLIC;
-- SELECT * FROM employees;
```

## Explanation of the Code

This example demonstrates how to set up and apply conditional masking policies in Snowflake:

- **`CREATE TABLE employees`**: This statement defines a sample table named `employees` with columns like `employee_id`, `employee_name`, `email_address`, and `salary`. This is the base data we want to protect.
- **`INSERT INTO employees`**: Populates the `employees` table with some sample records.
- **`CREATE MASKING POLICY email_masking_policy AS (val VARCHAR) RETURNS VARCHAR -> ...`**:
    - **What it does**: This creates a masking policy named `email_masking_policy`. It takes a `VARCHAR` value (`val`) as input and returns a `VARCHAR`.
    - **How it changes the result set**: The `CASE` statement inside the policy checks the `CURRENT_ROLE()` of the user. If the role is `HR_ANALYST` or `SECURITY_ADMIN`, the original `val` (the email address) is returned unmasked. For any other role, a masked version is returned, showing asterisks before the `@` symbol and the domain name. This effectively alters the data seen by unauthorized roles in the `email_address` column.
- **`CREATE MASKING POLICY salary_masking_policy AS (val DECIMAL(10, 2)) RETURNS DECIMAL(10, 2) -> ...`**:
    - **What it does**: Similar to the email policy, this creates a masking policy for the `salary` column, which is of `DECIMAL` type.
    - **How it changes the result set**: It returns the original `val` (salary) only if the `CURRENT_ROLE()` is `HR_ANALYST`. For all other roles, it returns `0.00`, completely obfuscating the actual salary value.
- **`ALTER TABLE employees MODIFY COLUMN email_address SET MASKING POLICY email_masking_policy;`**:
    - **What it does**: This statement applies the `email_masking_policy` to the `email_address` column of the `employees` table.
    - **How it changes the result set**: From this point on, any query on `employees.email_address` will automatically invoke `email_masking_policy`, and the returned values for that column will be conditionally masked based on the user's role.
- **`ALTER TABLE employees MODIFY COLUMN salary SET MASKING POLICY salary_masking_policy;`**:
    - **What it does**: This statement applies the `salary_masking_policy` to the `salary` column.
    - **How it changes the result set**: Similarly, any query on `employees.salary` will now be subject to the `salary_masking_policy`, returning the original or masked salary based on the user's role.
- **`SELECT * FROM employees;` (under different roles)**: This shows how different roles would perceive the data. The `SELECT` clause retrieves data, and the applied masking policies automatically modify the output for `email_address` and `salary` before it's returned to the user, effectively changing the result set based on the `CURRENT_ROLE()`.

## When to Use

1.  **Protecting PII (Personally Identifiable Information)**: When you have columns containing sensitive personal data (e.g., email addresses, phone numbers, partial credit card numbers) that only specific teams (like HR or customer support) should see in full, while others (like general analysts) should see a masked version.
    *   **Example Scenario**: An `orders` table contains a `customer_email` column. You want marketing analysts to see only the domain (e.g., `**********@example.com`) but customer service agents to see the full email.
2.  **Compliance with Regulations**: When your data needs to comply with privacy regulations such as GDPR, HIPAA, or CCPA, which mandate strict controls over sensitive data access.
    *   **Example Scenario**: A `patient_records` table holds `diagnosis` and `medication` information. Only medical staff roles (`DOCTOR`, `NURSE`) should view this data unmasked, while administrative roles see a generic "Masked" value.
3.  **Tiered Data Access for Business Metrics**: When different levels of detail for business metrics (e.g., exact revenue figures, profit margins) are accessible based on seniority or department.
    *   **Example Scenario**: A `financial_transactions` table has a `transaction_amount` column. Senior management roles (`EXECUTIVE`) can see the exact amounts, but junior analysts (`JUNIOR_ANALYST`) only see rounded values or NULLs.

## When Not to Use

1.  **Simple Column-Level Access Control**: If you only need to restrict access to entire columns (e.g., a role can see `employee_name` but not `salary` at all), traditional `GRANT` and `REVOKE` permissions on columns are simpler and more efficient than masking policies that return NULL or a fixed value.
2.  **Performance-Critical Workloads with Complex Policies**: While Snowflake's masking policies are optimized, excessively complex policies with many `CASE` statements or calls to expensive UDFs (User-Defined Functions) could introduce minor overhead, especially on very large tables or high-frequency queries. If performance is paramount and simpler access control methods suffice, avoid overly intricate masking.
3.  **Data Deletion or Permanent Redaction**: Masking policies only obfuscate data at query time; they do not alter the underlying data. If the requirement is to permanently delete or redact data from storage, masking policies are not the right tool. Data retention policies or `DELETE` statements are needed for such cases.

## Common Mistakes

1.  **Over-complicating Masking Logic**: Developers often create overly complex `CASE` statements or nested conditions within masking policies when simpler logic would suffice. This can make policies harder to understand, maintain, and debug.
2.  **Not Testing with All Relevant Roles**: Failing to test the masking policy thoroughly with all intended user roles (including `PUBLIC`) can lead to unintended data exposure or, conversely, accidental over-masking, preventing legitimate access.
3.  **Ignoring Performance Implications for Complex UDFs**: While `CURRENT_ROLE()` is lightweight, if a masking policy calls a user-defined function (UDF) that performs complex computations or external lookups, it can impact query performance, especially on large datasets.
4.  **Assuming Masking Policies Alter Underlying Data**: A common misconception is that masking policies change the actual data stored in the table. They only transform the data returned to the user at query time. The raw, sensitive data remains in storage.

## Expected Output

The resulting dataset will vary depending on the `CURRENT_ROLE()` of the user executing the `SELECT` statement. The `employee_id` and `employee_name` columns will always show the original values. The `email_address` and `salary` columns will show either the original unmasked values or masked values as defined by their respective policies.

**Example Table (as viewed by `HR_ANALYST` role):**

| EMPLOYEE_ID | EMPLOYEE_NAME | EMAIL_ADDRESS         | SALARY   |
|-------------|---------------|-----------------------|----------|
| 1           | Alice Smith   | alice.s@example.com   | 75000.00 |
| 2           | Bob Johnson   | bob.j@example.com     | 82000.00 |
| 3           | Charlie Brown | charlie.b@example.com | 68000.00 |

*Explanation:* For an `HR_ANALYST`, both `email_address` and `salary` are returned in their original, unmasked form because the policies grant full access to this role.

**Example Table (as viewed by `FINANCE_ANALYST` or `PUBLIC` role):**

| EMPLOYEE_ID | EMPLOYEE_NAME | EMAIL_ADDRESS         | SALARY |
|-------------|---------------|-----------------------|--------|
| 1           | Alice Smith   | **********@example.com | 0.00   |
| 2           | Bob Johnson   | **********@example.com | 0.00   |
| 3           | Charlie Brown | **********@example.com | 0.00   |

*Explanation:* For a `FINANCE_ANALYST` or `PUBLIC` role, the `email_address` column shows a masked version (asterisks before `@`), and the `salary` column shows `0.00`, as per the defined masking policies for roles other than `HR_ANALYST`.
