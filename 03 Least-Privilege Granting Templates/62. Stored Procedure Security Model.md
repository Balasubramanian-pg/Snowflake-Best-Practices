# Stored Procedure Security Model in Snowflake

Stored procedures in Snowflake allow you to encapsulate SQL logic and procedural code (JavaScript, Python, Java, Scala, etc.) to perform complex operations. The security model defines how access to these procedures and the data they interact with is managed, ensuring data integrity and controlling what users can execute and what resources the procedures can access. This model exists to provide granular control over the execution environment and privileges associated with stored procedures, essential for enterprise-level data management and security.

**Short Explanation**

The security model for Snowflake stored procedures dictates who can define, execute, and inspect procedures, and what permissions the procedure itself has when running. It primarily uses two execution models: `OWNER` and `CALLER`. This allows administrators to choose whether a procedure runs with the permissions of its creator or the user executing it, solving critical security and access control problems. It's important in databases for maintaining strict separation of duties and preventing unauthorized data access or modification, commonly used in ETL processes, data governance, and application integrations.

- What problem does this SQL feature solve? It solves the problem of controlling privileges for code execution, ensuring that sensitive operations can be performed without granting extensive permissions directly to end-users.
- Why is it important in databases or analytics? It's crucial for security, compliance, and simplifying permission management, allowing complex operations to be executed safely by users with limited direct access.
- Where is it commonly used in real workflows? In data warehousing for automated data transformations, data loading (ETL/ELT), administrative tasks, and exposing controlled data access to business users via applications.

## Implementation Example

```sql
-- Create a table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    amount DECIMAL(10, 2),
    commission DECIMAL(10, 2)
);

-- Populate with some sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 1200.00, 120.00),
(2, 'Mouse', 25.00, 2.50),
(3, 'Monitor', 300.00, 30.00);

-- Create a role for analysts
CREATE ROLE analyst_role;
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE analyst_role;
GRANT USAGE ON DATABASE my_db TO ROLE analyst_role;
GRANT USAGE ON SCHEMA my_db.public TO ROLE analyst_role;

-- Grant SELECT to analyst_role on sales_data, but not on commission
GRANT SELECT ON sales_data TO ROLE analyst_role;
REVOKE SELECT ON sales_data(commission) FROM ROLE analyst_role; -- Ensure no direct access to commission

-- Create a stored procedure that calculates total sales, including commission, but is executed by the owner
CREATE OR REPLACE PROCEDURE calculate_total_sales_owner_rights()
RETURNS TABLE (total_sales DECIMAL(10,2), total_commission DECIMAL(10,2))
LANGUAGE SQL
EXECUTE AS OWNER
AS
$$
BEGIN
    RETURN TABLE(
        SELECT SUM(amount) AS total_sales, SUM(commission) AS total_commission
        FROM sales_data
    );
END;
$$;

-- Grant EXECUTE privilege on the procedure to analyst_role
GRANT EXECUTE ON PROCEDURE calculate_total_sales_owner_rights() TO ROLE analyst_role;

-- Assume 'analyst_user' has 'analyst_role' granted to them.
-- Now, 'analyst_user' can call the procedure.
-- CALL calculate_total_sales_owner_rights();

-- Example of a procedure with CALLER rights (for comparison, not part of the main problem solved here)
CREATE OR REPLACE PROCEDURE get_product_names_caller_rights()
RETURNS TABLE (product_name VARCHAR)
LANGUAGE SQL
EXECUTE AS CALLER
AS
$$
BEGIN
    RETURN TABLE(
        SELECT product_name FROM sales_data
    );
END;
$$;

-- Grant EXECUTE to analyst_role on the CALLER procedure.
GRANT EXECUTE ON PROCEDURE get_product_names_caller_rights() TO ROLE analyst_role;
-- If analyst_user tries to call this, it will only succeed if they have SELECT on sales_data.
-- CALL get_product_names_caller_rights();
```

## Explanation of the Code

- `CREATE OR REPLACE TABLE sales_data (...)`: This statement creates a new table named `sales_data` to hold our sample data, including sensitive `commission` information.
- `CREATE ROLE analyst_role; GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE analyst_role; ...`: These commands create a new role, `analyst_role`, and grant it necessary privileges (warehouse usage, database usage, schema usage) to operate within the environment.
- `GRANT SELECT ON sales_data TO ROLE analyst_role; REVOKE SELECT ON sales_data(commission) FROM ROLE analyst_role;`: The `analyst_role` is granted `SELECT` access to the `sales_data` table, but explicitly denied `SELECT` access to the `commission` column. This is a key part of the security setup, demonstrating that the user role *cannot* directly see commission data.
- `CREATE OR REPLACE PROCEDURE calculate_total_sales_owner_rights() ... EXECUTE AS OWNER ...`: This defines a stored procedure named `calculate_total_sales_owner_rights`.
    - `EXECUTE AS OWNER`: This is the critical part of the security model. It specifies that when this procedure runs, it will execute with the privileges of the user who *created* (owns) the procedure, not the user who *calls* it. Since the owner likely has `SELECT` access to all columns, including `commission`, the procedure can access it.
    - `SELECT SUM(amount) AS total_sales, SUM(commission) AS total_commission FROM sales_data;`: Inside the procedure, it calculates the sum of `amount` and `commission`. Because it runs `EXECUTE AS OWNER`, it successfully accesses the `commission` column, even though the `analyst_role` calling it does not have direct access.
- `GRANT EXECUTE ON PROCEDURE calculate_total_sales_owner_rights() TO ROLE analyst_role;`: This command grants the `analyst_role` the permission to execute the stored procedure. This means users with `analyst_role` can call the procedure and get aggregated commission data without ever having direct access to the raw `commission` column.
- `CREATE OR REPLACE PROCEDURE get_product_names_caller_rights() ... EXECUTE AS CALLER ...`: This defines another stored procedure, this time with `EXECUTE AS CALLER`.
    - `EXECUTE AS CALLER`: This means the procedure runs with the privileges of the user *calling* it. If `analyst_role` calls this, it will only succeed if `analyst_role` has `SELECT` on `sales_data` (which it does for product_name). If it tried to select `commission`, it would fail for `analyst_role`.

## When to Use

1.  **Data Masking/Controlled Access**: When you need to expose aggregated or partial sensitive data to users who should not have direct access to the raw underlying data. For example, an analyst can see total commissions via a procedure but not individual commission values.
2.  **Encapsulating Complex ETL/ELT Logic**: For complex data transformation pipelines where the procedure needs elevated privileges (e.g., creating tables, loading data into different schemas) that end-users should not possess.
3.  **Administrative Tasks**: Automating routine maintenance or administrative operations (like cloning databases, managing access, data retention policies) where the procedure needs higher permissions than the typical user triggering it.

## When Not to Use

1.  **Simple Queries with Direct Access**: If users already have all the necessary permissions to access the underlying tables and perform the query themselves, creating a stored procedure just for `EXECUTE AS OWNER` adds unnecessary overhead.
2.  **Caller-Specific Logic**: When the procedure's behavior or outcome *must* respect the calling user's individual row-level or column-level security policies. In such cases, `EXECUTE AS CALLER` is appropriate, not `EXECUTE AS OWNER`.
3.  **Over-Privileging**: Using `EXECUTE AS OWNER` unnecessarily can create security vulnerabilities if the procedure's owner has too many privileges and the procedure is not thoroughly vetted. A malicious or buggy procedure could exploit these elevated rights.

## Common Mistakes

1.  **Forgetting `GRANT EXECUTE`**: Developers often define the procedure but forget to grant the `EXECUTE` privilege to the roles or users who need to run it, leading to "permission denied" errors.
2.  **Misunderstanding `EXECUTE AS OWNER` vs. `EXECUTE AS CALLER`**: Confusing the two execution contexts can lead to either security gaps (procedure has too many rights) or permission errors (procedure fails because the caller lacks rights).
3.  **Not Revoking Direct Access**: Relying solely on `EXECUTE AS OWNER` without also revoking direct access to sensitive data for the calling roles/users. If users can bypass the procedure and query the raw data directly, the security model is compromised.
4.  **Inadequate Owner Permissions**: If the owner of an `EXECUTE AS OWNER` procedure doesn't have all the necessary permissions for the procedure's operations, the procedure will fail, even though the intent was to use the owner's privileges.

## Implementation Example

```sql
-- First, let's create a table that holds some sensitive employee salary data.
CREATE OR REPLACE TABLE employee_salaries (
    employee_id INT,
    employee_name VARCHAR,
    department VARCHAR,
    salary DECIMAL(10, 2)
);

-- And put some data in it.
INSERT INTO employee_salaries VALUES
(101, 'Alice Smith', 'Sales', 75000.00),
(102, 'Bob Johnson', 'HR', 60000.00),
(103, 'Charlie Brown', 'Sales', 80000.00);

-- Now, create a role for "analysts" who should not see individual salaries directly.
CREATE ROLE analytics_team_role;

-- Grant minimal privileges to the analytics team role.
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE analytics_team_role;
GRANT USAGE ON DATABASE my_db TO ROLE analytics_team_role;
GRANT USAGE ON SCHEMA my_db.public TO ROLE analytics_team_role;

-- Important: Grant SELECT on the table but explicitly REVOKE SELECT on the 'salary' column.
GRANT SELECT ON employee_salaries TO ROLE analytics_team_role;
REVOKE SELECT ON employee_salaries(salary) FROM ROLE analytics_team_role;

-- Create a stored procedure that calculates the average salary for a department.
-- This procedure will execute with the privileges of its OWNER.
CREATE OR REPLACE PROCEDURE get_avg_department_salary(dept_name VARCHAR)
RETURNS TABLE (department VARCHAR, average_salary DECIMAL(10,2))
LANGUAGE SQL
EXECUTE AS OWNER -- This is the key: runs with the creator's permissions
AS
$$
BEGIN
    RETURN TABLE(
        SELECT
            department,
            AVG(salary) AS average_salary
        FROM
            employee_salaries
        WHERE
            department = :dept_name
        GROUP BY
            department
    );
END;
$$;

-- Grant the analytics_team_role permission to execute this stored procedure.
GRANT EXECUTE ON PROCEDURE get_avg_department_salary(VARCHAR) TO ROLE analytics_team_role;

-- Now, a user with analytics_team_role can call this procedure
-- even though they cannot directly select the 'salary' column.
-- CALL get_avg_department_salary('Sales');
```

## Explanation of the Code

- `CREATE OR REPLACE TABLE employee_salaries (...)`: This sets up a table to store employee data, including their `salary`, which we consider sensitive.
- `INSERT INTO employee_salaries VALUES (...)`: Populates the table with some example employee records.
- `CREATE ROLE analytics_team_role;`: Defines a new role for analysts.
- `GRANT USAGE ON WAREHOUSE ...; GRANT USAGE ON DATABASE ...; GRANT USAGE ON SCHEMA ...;`: These grants provide the `analytics_team_role` the basic permissions to connect and work within the database environment.
- `GRANT SELECT ON employee_salaries TO ROLE analytics_team_role; REVOKE SELECT ON employee_salaries(salary) FROM ROLE analytics_team_role;`: This is critical! The `analytics_team_role` can see most of the `employee_salaries` table, but it is explicitly prevented from seeing the individual `salary` column directly. This establishes the security boundary.
- `CREATE OR REPLACE PROCEDURE get_avg_department_salary(dept_name VARCHAR) ... LANGUAGE SQL EXECUTE AS OWNER ...`: This defines our stored procedure.
    - `EXECUTE AS OWNER`: This clause is the heart of the security model here. It instructs Snowflake that when this procedure is executed, it should use the privileges of the user who *created* (owns) the procedure. If the owner has `SELECT` access to the `salary` column (which is assumed for the procedure creator), the procedure will successfully access it.
    - `SELECT department, AVG(salary) AS average_salary FROM employee_salaries WHERE department = :dept_name GROUP BY department;`: The logic inside the procedure calculates the average salary for a given department. Because it runs with `OWNER` rights, it can access the `salary` column, even if the calling role (`analytics_team_role`) cannot.
- `GRANT EXECUTE ON PROCEDURE get_avg_department_salary(VARCHAR) TO ROLE analytics_team_role;`: This command allows any user assigned the `analytics_team_role` to execute the `get_avg_department_salary` procedure. They can now get the average salary without direct access to individual salaries.

## When to Use

1.  **Delegated Access to Sensitive Data**: When you want to provide aggregated or transformed views of sensitive data (like salaries, PII, financial details) to users who should not have direct row-level or column-level access to the raw information.
    *   **Scenario**: Your "HR Business Partners" need to see average salaries by department but should never view individual employee salaries. An `EXECUTE AS OWNER` procedure can calculate and expose the average without granting direct `SELECT` on the `salary` column to the HR role.
2.  **Encapsulating Administrative or High-Privilege Tasks**: For complex operations that require elevated permissions (e.g., creating/dropping tables, managing schemas, loading data into restricted areas) which ordinary users or applications should trigger but not directly perform.
    *   **Scenario**: A "Data Loader" role needs to run a nightly process that truncates a staging table and inserts new data. The `TRUNCATE TABLE` and `INSERT` commands might be too powerful for the role to have directly. An `EXECUTE AS OWNER` procedure can wrap these operations, and the `Data Loader` role is only granted `EXECUTE` on the procedure.
3.  **Building Secure Data APIs/Microservices**: When creating interfaces for applications or external systems where the application users don't need direct database credentials but rely on specific, pre-defined operations that adhere to security policies.
    *   **Scenario**: A web application displays a "company sales performance dashboard." The application connects to Snowflake via a generic user, but calls various `EXECUTE AS OWNER` stored procedures to fetch specific, pre-computed metrics without ever needing `SELECT` on all underlying sales tables.

## When Not to Use

1.  **When Caller's Context is Paramount**: If the procedure's logic needs to strictly adhere to the calling user's current session parameters, user-specific row-level security, or dynamically changing privileges.
    *   **Reason**: `EXECUTE AS OWNER` procedures execute within the owner's context for most session parameters and object resolution, overriding the caller's session settings in many cases. Use `EXECUTE AS CALLER` instead if the caller's context is essential.
2.  **For Simple Queries Where Direct Access is Already Granted**: If the users already have all the necessary permissions to directly query the underlying data, wrapping it in an `EXECUTE AS OWNER` procedure adds unnecessary complexity and overhead without security benefits.
    *   **Reason**: It complicates development, debugging, and maintenance without adding value to access control.
3.  **Over-Privileging without Due Diligence**: If the owner of the `EXECUTE AS OWNER` procedure has very broad permissions and the procedure itself is not thoroughly reviewed and secured, it can become a potential security vulnerability.
    *   **Reason**: A flaw in the procedure could be exploited, allowing unauthorized access or manipulation of data under the owner's elevated privileges.

## Common Mistakes

1.  **Forgetting to Grant `EXECUTE`**: Developers often define the `EXECUTE AS OWNER` procedure but forget to grant the `EXECUTE` privilege on that procedure to the roles or users who need to run it. This results in "permission denied" errors when users try to call it.
2.  **Not Understanding the `EXECUTE AS` Clause**: Confusing `EXECUTE AS OWNER` with `EXECUTE AS CALLER` can lead to either procedures failing due to insufficient permissions (when `OWNER` was intended) or inadvertently exposing more privileges than desired (when `CALLER` was intended). `EXECUTE AS OWNER` is the default if `EXECUTE AS` is omitted.
3.  **Relying Solely on Procedures without Revoking Direct Access**: The security benefit of `EXECUTE AS OWNER` procedures is maximized when combined with carefully restricted direct access to underlying objects for calling roles. If users can bypass the procedure and query the raw sensitive data directly, the security model is compromised.
4.  **Implicit Assumptions about Session Context**: Expecting an `EXECUTE AS OWNER` procedure to fully respect the caller's session variables or parameters. While some parameters are inherited, many are not, leading to unexpected behavior or failures if the procedure relies on specific session settings from the caller.
