# Attaching Row Access Policies to Views in Snowflake

Attaching Row Access Policies to Views in Snowflake is a powerful security feature that allows for granular control over data access. It essentially filters rows returned by a view based on the user's role or other context, ensuring that users only see the data they are authorized to access, even when querying a view.

**Short Explanation**

This feature enables you to apply row-level security directly to views, which act as virtual tables. It solves the problem of needing to create multiple views for different access levels or complex `WHERE` clauses in every query. This is important for maintaining data privacy, compliance with regulations like GDPR or HIPAA, and simplifying data governance. It's commonly used in scenarios where a single underlying table or view needs to be shared among various departments or external users, each with different data visibility requirements.

- **What problem does this SQL feature solve?** It solves the challenge of enforcing row-level security dynamically and consistently across a shared dataset, preventing unauthorized users from seeing sensitive rows without modifying the base tables or creating numerous specialized views.
- **Why is it important in databases or analytics?** It's crucial for data governance, regulatory compliance (e.g., ensuring only authorized personnel see specific customer records), and preventing data leakage, especially in multi-tenant environments or when sharing data with diverse user groups.
- **Where is it commonly used in real workflows?** It's often used in analytics platforms, data warehouses, and applications where different user roles (e.g., sales, marketing, finance) need access to the same dataset but with varying levels of row visibility, such as only seeing data relevant to their region or customers.

## Implementation Example

```sql
-- 1. Create a base table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2),
    customer_id INT
);

INSERT INTO sales_data VALUES
(1, 'Laptop', 'East', 1200.00, 101),
(2, 'Monitor', 'West', 300.00, 102),
(3, 'Keyboard', 'East', 75.00, 101),
(4, 'Mouse', 'South', 50.00, 103),
(5, 'Tablet', 'West', 600.00, 102),
(6, 'Webcam', 'East', 100.00, 104);

-- 2. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_column VARCHAR) RETURNS BOOLEAN ->
    CURRENT_ROLE() = 'ADMIN_ROLE' OR region_column = 'East';

-- 3. Create a View over the base table
CREATE OR REPLACE VIEW regional_sales_view AS
SELECT
    sale_id,
    product_name,
    region,
    sales_amount
FROM
    sales_data;

-- 4. Apply the Row Access Policy to the View
ALTER VIEW regional_sales_view ADD ROW ACCESS POLICY sales_policy ON (region);

-- 5. Test access (assuming roles exist or simulate with different CURRENT_ROLE() calls)
-- Connect as ADMIN_ROLE:
-- SELECT * FROM regional_sales_view; -- Should see all rows

-- Connect as a user with NO specific role or a different role than ADMIN_ROLE:
-- SELECT * FROM regional_sales_view; -- Should only see 'East' region data
```

## Explanation of the Code

This example first sets up a `sales_data` table, then defines a `ROW ACCESS POLICY` called `sales_policy`. A `VIEW`, `regional_sales_view`, is created over the `sales_data` table. Finally, the `sales_policy` is attached to this view.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to simulate a dataset that needs row-level security. It populates it with some example sales records, including a `region` column that will be used for policy enforcement.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_column VARCHAR) RETURNS BOOLEAN -> ...`**: This defines the actual row access policy.
    - `sales_policy AS (region_column VARCHAR)`: Declares a policy named `sales_policy` that expects one argument, `region_column` of type `VARCHAR`.
    - `RETURNS BOOLEAN ->`: Indicates that the policy function will return a boolean value (`TRUE` to show the row, `FALSE` to hide it).
    - `CURRENT_ROLE() = 'ADMIN_ROLE' OR region_column = 'East'`: This is the core logic. Rows are visible if the `CURRENT_ROLE()` is `ADMIN_ROLE` OR if the `region_column` for that row is 'East'. All other rows will be filtered out for users who are not `ADMIN_ROLE`.
- **`CREATE OR REPLACE VIEW regional_sales_view AS SELECT ... FROM sales_data;`**: This creates a view named `regional_sales_view` which selects all relevant columns from the `sales_data` table. At this stage, the view itself doesn't have any security applied.
- **`ALTER VIEW regional_sales_view ADD ROW ACCESS POLICY sales_policy ON (region);`**: This is the crucial step where the `sales_policy` is applied to the `regional_sales_view`. The `ON (region)` clause specifies that the `region` column from the view should be passed as the `region_column` argument to the `sales_policy`.
- **`SELECT * FROM regional_sales_view;`**: This is a query used to demonstrate the effect of the policy.
    - **What it does**: When a user queries `regional_sales_view`, Snowflake automatically evaluates the attached `sales_policy` for each row using the `region` column's value from that row and the `CURRENT_ROLE()` of the user.
    - **How it changes the result set**: If the policy evaluates to `TRUE` for a row, that row is included in the result set. If it evaluates to `FALSE`, the row is excluded, making it invisible to the user. This happens transparently to the querying user.

## When to Use

1.  **Compliance and Data Privacy:** When you need to enforce strict data access rules to comply with regulations (e.g., GDPR, HIPAA) or internal privacy policies, ensuring that sensitive data (like patient records, financial details, or PII) is only visible to authorized personnel.
    *   **Example Scenario:** A healthcare database where doctors can only see their own patients' records, while hospital administrators can see all records. A view over the patient table with a policy tied to the doctor's ID would achieve this.
2.  **Multi-tenant Applications:** In a multi-tenant environment where different clients or organizations share the same database tables but must never see each other's data.
    *   **Example Scenario:** A SaaS platform where each company stores its data in the same `customer_transactions` table. A view with a row access policy can ensure that users from Company A only see transactions belonging to Company A, based on a `tenant_id` column.
3.  **Simplified Data Sharing:** To provide different views of the same underlying data to various user groups or external partners without creating multiple physical tables or complex application-level filtering.
    *   **Example Scenario:** A sales team needs to see sales data only for their assigned region, while a global manager needs to see all sales data. A single view with a row access policy checking the user's role or assigned region can serve both needs efficiently.

## When Not to Use

1.  **Simple Table-Level Permissions:** If all users need to see all rows in a table, and permission is purely at the table level (either they can query the table or they can't). In this case, standard `GRANT/REVOKE` permissions on the table or view are sufficient and simpler.
    *   **Reasoning:** Introducing a row access policy adds overhead and complexity for no additional benefit if row-level security isn't required.
2.  **Performance Critical Workloads with Complex Policies:** For extremely high-throughput query environments where the row access policy involves very complex or expensive computations (e.g., subqueries, UDFs, or lookups against large auxiliary tables for every row).
    *   **Reasoning:** While Snowflake optimizes policies, a highly complex policy evaluated for every row on a massive table could introduce noticeable latency. Simpler filters or pre-filtered materialized views might be more efficient.
3.  **Small Tables with Infrequent Access:** For small, non-sensitive lookup tables or tables that are rarely queried.
    *   **Reasoning:** The overhead of defining, managing, and evaluating a policy might outweigh the security benefits if the data volume is low and sensitivity is minimal. Standard view definitions or application-level filtering might be simpler to maintain.

## Common Mistakes

1.  **Forgetting to Grant `APPLY ROW ACCESS POLICY`:** Users (or roles) need the `APPLY ROW ACCESS POLICY` privilege on the policy to attach it to a view. Forgetting this leads to permission errors.
2.  **Incorrectly Mapping Columns:** Specifying the wrong column from the view in the `ON (...)` clause when attaching the policy. The column(s) passed to the policy must correspond correctly to the policy's arguments.
3.  **Policy Logic Bugs:** Writing a policy logic that inadvertently hides too much data, too little data, or creates an infinite loop if the policy itself queries data it's meant to protect. Thorough testing with various roles is essential.
4.  **Performance Impact Neglect:** Not considering the potential performance implications of a complex policy, especially when applied to very large views that are heavily queried. Always test performance after applying policies.
5.  **Not Understanding Policy Precedence:** If a table also has a row access policy, and a view built on that table also has one, understanding how these policies interact (they are ANDed together) is crucial to avoid unexpected filtering.

## Expected Output

When `ADMIN_ROLE` queries `regional_sales_view`:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT |
| :------ | :----------- | :----- | :----------- |
| 1       | Laptop       | East   | 1200.00      |
| 2       | Monitor      | West   | 300.00       |
| 3       | Keyboard     | East   | 75.00        |
| 4       | Mouse        | South  | 50.00        |
| 5       | Tablet       | West   | 600.00       |
| 6       | Webcam       | East   | 100.00       |

When a non-`ADMIN_ROLE` user (or a user whose role doesn't match `ADMIN_ROLE` and whose region isn't 'East') queries `regional_sales_view`:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT |
| :------ | :----------- | :----- | :----------- |
| 1       | Laptop       | East   | 1200.00      |
| 3       | Keyboard     | East   | 75.00        |
| 6       | Webcam       | East   | 100.00       |

-   **SALE_ID**: A unique identifier for each sale.
-   **PRODUCT_NAME**: The name of the product sold.
-   **REGION**: The geographical region where the sale occurred. This column is used by the row access policy to determine visibility.
-   **SALES_AMOUNT**: The monetary value of the sale.

The output will contain only the rows for which the attached `sales_policy` evaluates to `TRUE` for the querying user, based on their current role and the `REGION` column's value.
