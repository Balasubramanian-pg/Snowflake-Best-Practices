# Combining Row Access Policies with Column Masking in Snowflake

Combining Row Access Policies with Column Masking in Snowflake is a powerful data governance strategy that allows for fine-grained control over data access. It ensures that users only see the rows of data they are authorized to view and, for those rows, only the columns they are permitted to access, with sensitive data often obfuscated or masked. This dual approach provides comprehensive security and compliance capabilities, particularly for sensitive data in regulated industries.

**Short Explanation**

This concept involves applying both Row Access Policies and Column Masking Policies to tables in Snowflake. Row Access Policies filter the rows returned based on a user's role or other attributes, while Column Masking Policies dynamically obfuscate or redact sensitive column data before it's displayed to unauthorized users.

*   **What problem does this SQL feature solve?** It solves the problem of granular data access control, allowing different users or roles to see different subsets of data (rows) and different levels of detail within those subsets (columns).
*   **Why is it important in databases or analytics?** It's crucial for meeting compliance requirements (e.g., GDPR, HIPAA, CCPA), preventing data breaches, and ensuring data privacy by restricting sensitive information only to authorized individuals.
*   **Where is it commonly used in real workflows?** It's commonly used in healthcare for patient data, finance for customer financial records, human resources for employee information, and any scenario where data segregation and protection are paramount.

## Implementation Example

```sql
-- Create a table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    customer_email VARCHAR,
    product_name VARCHAR,
    sale_amount DECIMAL(10, 2),
    region VARCHAR,
    employee_id INT
);

-- Insert some sample data
INSERT INTO sales_data VALUES
(1, 'Alice Smith', 'alice@example.com', 'Laptop', 1200.00, 'East', 101),
(2, 'Bob Johnson', 'bob@example.com', 'Mouse', 25.00, 'West', 102),
(3, 'Charlie Brown', 'charlie@example.com', 'Keyboard', 75.00, 'East', 101),
(4, 'Diana Prince', 'diana@example.com', 'Monitor', 300.00, 'South', 103),
(5, 'Eve Adams', 'eve@example.com', 'Webcam', 50.00, 'East', 101);

-- Create roles for demonstration
CREATE ROLE sales_east;
CREATE ROLE sales_west;
CREATE ROLE hr_analyst;

-- Grant select on the table to roles
GRANT SELECT ON TABLE sales_data TO ROLE sales_east;
GRANT SELECT ON TABLE sales_data TO ROLE sales_west;
GRANT SELECT ON TABLE sales_data TO ROLE hr_analyst;

-- Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy
AS (region VARCHAR, employee_id INT) RETURNS BOOLEAN ->
  CURRENT_ROLE() IN ('ACCOUNTADMIN', 'HR_ANALYST') OR -- HR and Admin can see all rows
  (CURRENT_ROLE() = 'SALES_EAST' AND region = 'East') OR
  (CURRENT_ROLE() = 'SALES_WEST' AND region = 'West');

-- Create Masking Policies
CREATE OR REPLACE MASKING POLICY mask_email_policy
AS (val VARCHAR) RETURNS VARCHAR ->
  CASE
    WHEN CURRENT_ROLE() IN ('ACCOUNTADMIN', 'HR_ANALYST') THEN val
    ELSE '***MASKED***'
  END;

CREATE OR REPLACE MASKING POLICY mask_customer_name_policy
AS (val VARCHAR) RETURNS VARCHAR ->
  CASE
    WHEN CURRENT_ROLE() IN ('ACCOUNTADMIN', 'HR_ANALYST') THEN val
    WHEN CURRENT_ROLE() IN ('SALES_EAST', 'SALES_WEST') THEN LEFT(val, 1) || '*****'
    ELSE '***MASKED***'
  END;

-- Apply policies to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region, employee_id);
ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY mask_email_policy;
ALTER TABLE sales_data MODIFY COLUMN customer_name SET MASKING POLICY mask_customer_name_policy;

-- Test as different roles
-- USE ROLE SALES_EAST;
-- SELECT * FROM sales_data;

-- USE ROLE HR_ANALYST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example first sets up a `sales_data` table with various columns, including sensitive ones like `customer_name` and `customer_email`. It then creates different roles to simulate distinct user groups.

*   `CREATE OR REPLACE TABLE sales_data`: Defines a table to store sales information, including columns that will be subject to policies.
*   `INSERT INTO sales_data VALUES(...)`: Populates the table with sample data for demonstration.
*   `CREATE ROLE sales_east`, `sales_west`, `hr_analyst`: Establishes distinct roles with varying data access needs.
*   `GRANT SELECT ON TABLE sales_data TO ROLE ...`: Gives the created roles basic `SELECT` privileges on the table.
*   `CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy`:
    *   **What it does:** This clause defines a function that takes `region` and `employee_id` as input and returns a `BOOLEAN`.
    *   **How it changes the result set:** When applied, only rows where the policy condition evaluates to `TRUE` for the `CURRENT_ROLE()` are returned. For `SALES_EAST`, only rows where `region = 'East'` are visible. For `HR_ANALYST` and `ACCOUNTADMIN`, all rows are visible.
*   `CREATE OR REPLACE MASKING POLICY mask_email_policy` and `mask_customer_name_policy`:
    *   **What it does:** These clauses define functions that take a column value (`val`) and return a modified `VARCHAR` based on the `CURRENT_ROLE()`.
    *   **How it changes the result set:** When applied to a column, the actual data in that column is transformed. For `mask_email_policy`, if the role is not `ACCOUNTADMIN` or `HR_ANALYST`, the email is replaced with `'***MASKED***'`. For `mask_customer_name_policy`, sales roles see a partially masked name, while other unauthorized roles see a fully masked name.
*   `ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region, employee_id)`: Applies the `sales_region_policy` to the entire `sales_data` table, linking the policy to the `region` and `employee_id` columns for evaluation.
*   `ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY mask_email_policy`: Attaches `mask_email_policy` to the `customer_email` column, ensuring its contents are masked according to the policy.
*   `ALTER TABLE sales_data MODIFY COLUMN customer_name SET MASKING POLICY mask_customer_name_policy`: Attaches `mask_customer_name_policy` to the `customer_name` column, masking its contents dynamically.
*   `USE ROLE ...; SELECT * FROM sales_data;`: These commented lines demonstrate how different roles would perceive the data, showing the combined effect of row filtering and column masking.

## When to Use

1.  **Regulatory Compliance (e.g., HIPAA, GDPR):** When dealing with sensitive personal identifiable information (PII) or protected health information (PHI) where access must be strictly controlled and audited.
    *   *Example Scenario:* A hospital system needs to ensure that only doctors and nurses can view full patient records for their assigned patients, while billing staff can only see billing-related rows and masked patient identifiers.
2.  **Multi-Tenant Architectures:** In SaaS applications or data platforms where multiple customers share the same underlying database, but each customer should only see their own data.
    *   *Example Scenario:* A CRM platform stores data for multiple companies in one large table. Each company's users should only see their company's customer data, and certain sensitive contact details might be masked for lower-level employees.
3.  **Internal Data Segregation:** When different departments or teams within an organization require access to different subsets of data, often with varying levels of detail or sensitivity.
    *   *Example Scenario:* An HR department has access to all employee records, but the marketing department only sees anonymized employee demographics, and line managers only see performance data for their direct reports, with salary details masked.

## When Not to Use

1.  **Overkill for Simple Access Control:** If data access needs are very basic (e.g., full table access for one role, no access for others), policies can add unnecessary complexity.
    *   *Situation:* A small internal tool with only two user roles: "Admin" (sees everything) and "Viewer" (sees everything read-only). Standard `GRANT/REVOKE` permissions are sufficient.
2.  **Performance Critical Workloads on Very Large Tables:** While Snowflake policies are optimized, complex policies with many conditions or deeply nested functions can introduce slight overhead. If queries on extremely large tables are already pushing performance limits and data sensitivity is low, policies might be avoided.
    *   *Situation:* A data warehouse used for high-volume, low-latency analytics where slight query performance degradation due to policy evaluation is unacceptable, and the data is not considered highly sensitive.
3.  **When Data Needs to Be Permanently Redacted:** If sensitive data needs to be irreversibly removed or transformed at ingestion or permanently stored in a masked state, policies are not the right tool. Policies apply dynamic masking at query time.
    *   *Situation:* A system that needs to permanently anonymize customer names or addresses for a public dataset, where the original sensitive data should not be retrievable by any means, even by administrators. This requires data transformation during ETL, not policy-based dynamic masking.

## Common Mistakes

1.  **Forgetting to Grant Policy Privileges:** Not granting the `APPLY MASKING POLICY` or `APPLY ROW ACCESS POLICY` privilege to the role that creates or alters the table, leading to permission errors when applying policies.
2.  **Incorrect Policy Logic:** Errors in the `CASE` statements or `BOOLEAN` conditions within the policies, causing data to be incorrectly masked or filtered, or even making data inaccessible to authorized users.
3.  **Mixing Policies Inefficiently:** Applying too many masking policies to a single column or having overly complex row access policies, which can make debugging difficult and potentially impact query performance if not designed well.
4.  **Not Testing Policies with Different Roles:** Assuming policies work as intended without thoroughly testing them by switching to various roles (`USE ROLE <role_name>;`) to verify the expected data visibility and masking.
5.  **Policy Dependency Issues:** Modifying or dropping columns that are referenced by an active row access policy without first detaching or updating the policy, leading to errors.

The resulting dataset will vary based on the `CURRENT_ROLE()` executing the `SELECT` statement.

**Example for `SALES_EAST` role:**

Only rows where `region = 'East'` are visible, and `customer_name` is partially masked, while `customer_email` is fully masked.
