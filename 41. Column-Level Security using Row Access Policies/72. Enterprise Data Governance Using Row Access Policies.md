# Enterprise Data Governance Using Row Access Policies in Snowflake

Row Access Policies in Snowflake provide a powerful, centralized, and granular way to control access to individual rows within a table based on a user's role or other attributes. This mechanism is crucial for implementing robust enterprise data governance strategies, ensuring that sensitive data is only visible to authorized personnel, while maintaining a single, consistent data set.

**Short Explanation**

Row Access Policies act like dynamic filters on a table. When a user queries a table with an active policy, the policy's conditions are automatically applied to restrict which rows they can see, without them even knowing it. This solves the problem of needing to create multiple views or tables for different user groups, simplifying data management and enhancing security by preventing unauthorized data exposure. It's important in databases and analytics for ensuring compliance with regulations like GDPR or HIPAA, and is commonly used in scenarios where data access needs to be restricted based on department, region, or user roles.

- **What problem does this SQL feature solve?** It solves the problem of granular data access control, allowing different users to see different subsets of data within the same table, based on predefined rules, without needing to create separate tables or views for each access level.
- **Why is it important in databases or analytics?** It's vital for data security, privacy, and regulatory compliance (e.g., GDPR, HIPAA, CCPA), ensuring that sensitive information is protected at the row level and only accessible to authorized individuals.
- **Where is it commonly used in real workflows?** It's used in enterprises for multi-tenant applications, financial data where access depends on roles (e.g., analyst vs. manager), healthcare data where patient records are restricted, and any scenario requiring data segregation based on user attributes or classifications.

## Implementation Example

```sql
-- 1. Create a sample table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2),
    customer_name VARCHAR,
    sensitive_info VARCHAR
);

-- 2. Insert some sample data
INSERT INTO sales_data VALUES
    (1, 'Laptop', 'North', 1200.00, 'Alice', 'Confidential A'),
    (2, 'Mouse', 'South', 25.00, 'Bob', 'Public'),
    (3, 'Keyboard', 'North', 75.00, 'Charlie', 'Confidential B'),
    (4, 'Monitor', 'East', 300.00, 'David', 'Confidential A'),
    (5, 'Webcam', 'West', 50.00, 'Eve', 'Public');

-- 3. Create a mapping table for user roles and their allowed regions/access levels
CREATE OR REPLACE TABLE user_access_control (
    user_role VARCHAR,
    allowed_region VARCHAR,
    access_level VARCHAR
);

INSERT INTO user_access_control VALUES
    ('ANALYST_NORTH', 'North', 'Confidential A'),
    ('ANALYST_SOUTH', 'South', 'Public'),
    ('ADMIN', NULL, NULL); -- Admin can see everything

-- 4. Create the Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_data_policy
ON sales_data
FOR ROW USING (region, sensitive_info)
RETURNS BOOLEAN ->
    -- Check if the current role is 'ADMIN' (sees all rows)
    -- OR if there's a match in user_access_control for the current role and region/access level
    (SELECT CURRENT_ROLE() = 'ADMIN') OR
    EXISTS (
        SELECT 1 FROM user_access_control uac
        WHERE uac.user_role = CURRENT_ROLE()
          AND (uac.allowed_region IS NULL OR uac.allowed_region = sales_data.region)
          AND (uac.access_level IS NULL OR uac.access_level = sales_data.sensitive_info)
    );

-- 5. Add the Row Access Policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_data_policy ON (region, sensitive_info);

-- 6. Grant necessary privileges for policy to evaluate (important for non-ADMIN roles)
GRANT SELECT ON TABLE user_access_control TO ROLE ANALYST_NORTH;
GRANT SELECT ON TABLE user_access_control TO ROLE ANALYST_SOUTH;

-- Now test with different roles (assuming roles 'ANALYST_NORTH', 'ANALYST_SOUTH', 'ADMIN' exist and are granted appropriately)
-- Use a role to test, e.g.:
-- USE ROLE ANALYST_NORTH;
-- SELECT * FROM sales_data;

-- USE ROLE ANALYST_SOUTH;
-- SELECT * FROM sales_data;

-- USE ROLE ADMIN;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This code sets up a demonstration of a Snowflake Row Access Policy to control data visibility based on user roles and data sensitivity.

-   **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` that will hold our business information, including `sensitive_info` which will be used in our policy.
-   **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with example records, including different regions and sensitive information flags.
-   **`CREATE OR REPLACE TABLE user_access_control (...)`**: This is a lookup table that defines which roles have access to which regions or sensitivity levels. This provides flexibility for managing permissions outside the policy definition itself.
-   **`INSERT INTO user_access_control VALUES (...)`**: Adds sample user role mappings to the `user_access_control` table. Notice 'ADMIN' has `NULL` for region and access level, implying full access.
-   **`CREATE OR REPLACE ROW ACCESS POLICY sales_data_policy ON sales_data FOR ROW USING (region, sensitive_info) RETURNS BOOLEAN -> ...`**:
    -   `CREATE ROW ACCESS POLICY sales_data_policy`: Defines a new row access policy named `sales_data_policy`.
    -   `ON sales_data`: Specifies that this policy is intended to be applied to the `sales_data` table.
    -   `FOR ROW USING (region, sensitive_info)`: Indicates that the policy will evaluate the `region` and `sensitive_info` columns of each row in `sales_data` to determine access. These columns are passed as arguments to the policy's SQL expression.
    -   `RETURNS BOOLEAN -> ...`: The core logic of the policy. For each row, this expression must return `TRUE` for the row to be visible, or `FALSE` for it to be filtered out.
        -   `(SELECT CURRENT_ROLE() = 'ADMIN')`: This part checks if the user's current role is 'ADMIN'. If it is, the policy returns `TRUE`, allowing the admin to see all rows.
        -   `EXISTS (SELECT 1 FROM user_access_control uac WHERE uac.user_role = CURRENT_ROLE() AND (...) )`: If not 'ADMIN', this checks if there's an entry in the `user_access_control` table that matches the `CURRENT_ROLE()` and the `region` and `sensitive_info` of the current row being evaluated.
            -   `(uac.allowed_region IS NULL OR uac.allowed_region = sales_data.region)`: Allows access if the `allowed_region` in the control table is `NULL` (meaning all regions are allowed for that role) OR if the `allowed_region` matches the row's `region`.
            -   `(uac.access_level IS NULL OR uac.access_level = sales_data.sensitive_info)`: Similar logic for the `sensitive_info` column.
        -   `How it changes the result set`: If the policy returns `FALSE` for a row, that row is effectively removed from the user's query result. If it returns `TRUE`, the row is included.
-   **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_data_policy ON (region, sensitive_info);`**: This statement applies the `sales_data_policy` to the `sales_data` table. From this point forward, any query on `sales_data` will automatically invoke the policy.
-   **`GRANT SELECT ON TABLE user_access_control TO ROLE ANALYST_NORTH;`**: This is crucial. Roles that need to evaluate the policy must have `SELECT` privileges on any lookup tables (like `user_access_control`) used within the policy, otherwise the policy evaluation will fail.

## When to Use

1.  **Regulatory Compliance (GDPR, HIPAA, CCPA):** When you need to restrict access to sensitive customer data (e.g., personal identifiable information, health records) based on user authorization, location, or department.
    *   **Scenario:** A healthcare provider needs to ensure that only medical personnel directly involved in a patient's care can view their full medical history, while administrative staff can only see basic demographic information.
2.  **Multi-tenant Applications:** In a software-as-a-service (SaaS) application where multiple customers share the same underlying database, but each customer should only see their own data.
    *   **Scenario:** A project management tool stores data for various companies in a single table. A row access policy ensures that users from "Company A" can only see projects and tasks belonging to "Company A".
3.  **Role-Based Data Segmentation:** When different internal departments or roles within an organization require access to different subsets of data within a large operational table.
    *   **Scenario:** A global sales report table contains sales figures for all regions. A "North America Sales Manager" role should only see data for the North America region, while a "Global Sales Director" can see all regions.

## When Not to Use

1.  **Simple Table-Level Access Control:** If the entire table should be accessible or restricted for a role, traditional `GRANT SELECT ON TABLE` or `REVOKE SELECT ON TABLE` is simpler and more performant.
    *   **Situation:** You have a `lookup_table` that is either entirely public or entirely private. Applying a row access policy adds unnecessary overhead.
2.  **Performance-Critical Workloads with Complex Policies:** Policies with complex subqueries or joins within the `RETURNS BOOLEAN` expression can introduce significant overhead, especially on very large tables or high-concurrency environments.
    *   **Situation:** A policy involves joining multiple large tables to determine row access for every query, leading to slow query execution times for frequently accessed data.
3.  **When User Roles and Data Access Rules Change Frequently and Rapidly:** While policies offer flexibility, very frequent changes to the underlying lookup tables or the policy logic itself can be harder to manage and audit compared to simpler view-based approaches in some cases.
    *   **Situation:** A dynamic reporting system where user-defined access rules are created and destroyed constantly, leading to potential complexity in maintaining and debugging many intertwined policies.

## Common Mistakes

1.  **Forgetting to Grant Permissions to Policy Owners/Roles:** The owner of the policy (the role that created it) or roles that execute queries against the protected table must have `SELECT` privileges on any tables referenced within the policy's body (e.g., the `user_access_control` table in the example). Failing to do so results in policy evaluation errors.
2.  **Overly Complex Policy Logic:** Writing highly intricate SQL within the policy function can lead to performance bottlenecks. Remember that the policy's logic is executed for *every row* for *every query*.
3.  **Not Testing with Different Roles:** Developers often test policies with an `ADMIN` role, which typically bypasses the policy. It's crucial to test with the specific roles (`ANALYST_NORTH`, `ANALYST_SOUTH` in our example) that the policy is designed to restrict to ensure it's functioning as expected.
4.  **Implicitly Relying on `CURRENT_ROLE()`:** While powerful, `CURRENT_ROLE()` can be misleading if users frequently switch roles. Consider using `CURRENT_AVAILABLE_ROLES()` or a custom user attribute table for more robust identity-based access control.
