# Read Only Consumer Roles in Snowflake

Read-only consumer roles in Snowflake are a fundamental security concept designed to grant users or other roles access to data solely for querying, without the ability to modify, insert, or delete any information. This concept exists to enforce the principle of least privilege, ensuring data integrity and security by limiting permissions to only what is necessary for a user's function.

**Short Explanation**

These roles provide secure access to data for reporting, analytics, and data consumption without risking accidental or malicious data alteration. They solve the problem of granting access while maintaining strict control over data modifications. This is crucial for environments where many users need to view sensitive data but only a few should be able to change it, making them essential in data warehousing and business intelligence workflows.

- **What problem does this SQL feature solve?** It solves the problem of securely sharing data for consumption (reading) while preventing any unauthorized modifications to the underlying data.
- **Why is it important in databases or analytics?** It's critical for data governance, compliance, and maintaining data integrity in complex data ecosystems.
- **Where is it commonly used in real workflows?** Widely used for business intelligence dashboards, reporting tools, data analysts, and external partners who need to query data but not alter it.

## Implementation Example

```sql
-- Step 1: Create a read-only role
CREATE ROLE REPORTING_ANALYST_RO;

-- Step 2: Grant USAGE on the database to the read-only role
GRANT USAGE ON DATABASE ANALYTICS_DB TO ROLE REPORTING_ANALYST_RO;

-- Step 3: Grant USAGE on the schema to the read-only role
GRANT USAGE ON SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;

-- Step 4: Grant SELECT on all tables in the schema to the read-only role
GRANT SELECT ON ALL TABLES IN SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;

-- Step 5: Grant SELECT on all future tables in the schema (for ongoing read access)
GRANT SELECT ON FUTURE TABLES IN SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;

-- Step 6: Grant the read-only role to a user or another role
GRANT ROLE REPORTING_ANALYST_RO TO USER JOHN_DOE;
```

## Explanation of the Code

This code demonstrates how to establish a read-only role and assign it permissions.

- `CREATE ROLE REPORTING_ANALYST_RO;`: This statement defines a new role named `REPORTING_ANALYST_RO`.
  - **What it does**: Creates a new security identity within Snowflake.
  - **How it changes the result set**: This is a DDL (Data Definition Language) statement; it doesn't return a result set but creates a new role in the system.
- `GRANT USAGE ON DATABASE ANALYTICS_DB TO ROLE REPORTING_ANALYST_RO;`: This grants the `REPORTING_ANALYST_RO` role permission to access the `ANALYTICS_DB` database. `USAGE` is necessary to access objects within the database.
  - **What it does**: Allows the role to see and navigate into the specified database.
  - **How it changes the result set**: No result set. It modifies access control.
- `GRANT USAGE ON SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;`: This grants the `REPORTING_ANALYST_RO` role permission to access the `SALES_SCHEMA` within `ANALYTICS_DB`. Similar to database `USAGE`, schema `USAGE` is required to access objects inside.
  - **What it does**: Allows the role to see and navigate into the specified schema.
  - **How it changes the result set**: No result set. It modifies access control.
- `GRANT SELECT ON ALL TABLES IN SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;`: This is the core read-only permission. It grants the `REPORTING_ANALYST_RO` role the ability to `SELECT` (query) data from all *existing* tables within the `ANALYTICS_DB.SALES_SCHEMA`.
  - **What it does**: Enables the role to retrieve data from tables.
  - **How it changes the result set**: No result set. It modifies access control.
- `GRANT SELECT ON FUTURE TABLES IN SCHEMA ANALYTICS_DB.SALES_SCHEMA TO ROLE REPORTING_ANALYST_RO;`: This statement is crucial for maintaining read-only access for newly created tables. It ensures that any table created *after* this grant in the `ANALYTICS_DB.SALES_SCHEMA` will automatically have `SELECT` permission for `REPORTING_ANALYST_RO`.
  - **What it does**: Sets up future grants, automatically applying `SELECT` to new tables.
  - **How it changes the result set**: No result set. It modifies access control.
- `GRANT ROLE REPORTING_ANALYST_RO TO USER JOHN_DOE;`: This assigns the `REPORTING_ANALYST_RO` to the user `JOHN_DOE`, allowing `JOHN_DOE` to assume this role and its associated permissions.
  - **What it does**: Makes the `REPORTING_ANALYST_RO` available to `JOHN_DOE`.
  - **How it changes the result set**: No result set. It modifies user-role assignments.

## When to Use

1.  **Business Intelligence and Reporting**: When BI tools, dashboards, or reporting applications need to fetch data for visualization and analysis without any modification capabilities.
    *   *Scenario*: A team of marketing analysts needs to run reports on sales data but should not be able to accidentally delete or update customer records.
2.  **Data Sharing with External Partners**: When you need to provide data access to vendors, clients, or auditors who only require viewing privileges.
    *   *Scenario*: An external auditing firm needs to inspect financial transaction data for compliance purposes, but you want to ensure they cannot alter any records.
3.  **Ad-hoc Analysis for Data Consumers**: For data scientists or data analysts performing exploratory data analysis where data integrity is paramount.
    *   *Scenario*: A junior data analyst is exploring a new dataset and needs to query tables, but the organization wants to prevent any accidental `UPDATE` or `DELETE` statements from impacting production data.

## When Not to Use

1.  **For Data Loading or ETL Processes**: When a process needs to insert, update, or delete data (e.g., an ETL pipeline), a read-only role will lack the necessary permissions.
    *   *Inefficiency*: The ETL process would fail with permission errors, requiring a different role with write access.
2.  **When Data Governance Requires Row/Column-Level Security**: While read-only roles control write access, they don't inherently control *which* rows or columns can be seen. If you need fine-grained access control on subsets of data, additional mechanisms (like secure views or row access policies) are required in conjunction with, or instead of, a simple read-only role.
    *   *Inefficiency*: Granting a read-only role might expose sensitive data columns or rows to users who should not see them, necessitating a more complex security setup.
3.  **For Administrative Tasks**: Roles requiring database, schema, or warehouse management (e.g., creating tables, managing users, resizing warehouses) cannot use a read-only role.
    *   *Inefficiency*: An administrator using a read-only role would be unable to perform their duties, leading to constant permission failures.

## Common Mistakes

1.  **Forgetting `USAGE` Grants**: Users often grant `SELECT` on tables but forget to grant `USAGE` on the parent database and schema, leading to "object does not exist" errors, even though `SELECT` was technically granted.
2.  **Not Granting on `FUTURE TABLES`**: Only granting `SELECT ON ALL TABLES` means that any tables created after the grant will not be accessible to the read-only role, requiring manual grants for each new table.
3.  **Granting Too Broadly**: Creating a single read-only role and granting it `SELECT` on every schema/database, which can lead to over-permissioning if different consumers need access to different datasets.
4.  **Mixing Read-Only with Write Privileges**: Accidentally granting `INSERT`, `UPDATE`, or `DELETE` to a role intended to be read-only, compromising data integrity.

## Expected Output

When a user assigned the `REPORTING_ANALYST_RO` role executes a `SELECT` query, they will receive the data from the specified tables as if they were any other user with `SELECT` permissions. However, any attempt to `INSERT`, `UPDATE`, `DELETE`, or `TRUNCATE` will result in a permission error.
