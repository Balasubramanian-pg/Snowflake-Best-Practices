# 08. The SYSADMIN Role for Object Creation

The SYSADMIN role is a system-defined role designed specifically for creating and managing all custom database objects and virtual warehouses within an account.
Utilizing this role ensures that all user-created objects roll up to a centralized administrative hierarchy, preventing orphaned objects and simplifying access management.

# Key Points

* The SYSADMIN role possesses the inherent privileges required to create databases, schemas, tables, and virtual warehouses, making it the primary role for infrastructure provisioning.
* It strictly isolates object management from user and security management (handled by USERADMIN/SECURITYADMIN) and account-level configuration (handled by ACCOUNTADMIN).
* > [!IMPORTANT]
  > Always grant custom roles to the SYSADMIN role; this ensures that the system administrator retains the ability to manage objects created by those custom roles.
* Using SYSADMIN prevents the ACCOUNTADMIN role from becoming cluttered with object ownership, strictly adhering to the principle of least privilege.
* Objects created by SYSADMIN can be easily managed, modified, or dropped by authorized administrators without requiring top-level account intervention.

# Implementation

The following Snowflake SQL script demonstrates how to assume the SYSADMIN role to provision foundational data architecture, including compute resources and storage containers.

```sql
USE ROLE SYSADMIN;

CREATE OR REPLACE WAREHOUSE dev_wh 
  WITH WAREHOUSE_SIZE = 'XSMALL' 
  AUTO_SUSPEND = 60 
  AUTO_RESUME = TRUE;

CREATE OR REPLACE DATABASE analytics_db;

CREATE OR REPLACE SCHEMA analytics_db.staging;

CREATE OR REPLACE TABLE analytics_db.staging.raw_events (
  event_id STRING, 
  event_timestamp TIMESTAMP
);
```

# What does the Query do?

The SQL script executes the following operations step by step:

Step 1 switches the active session context to the `SYSADMIN` role, ensuring that all subsequently created objects are owned by this centralized administrative role rather than a default or highly privileged account role.

Step 2 provisions a new virtual warehouse (`dev_wh`) with specific compute sizing and auto-suspend policies to efficiently manage query execution resources and control costs.

Step 3 creates a new logical database (`analytics_db`) to serve as the top-level container for data storage and organization within the account.

Step 4 generates a schema (`staging`) within the newly created database and deploys a foundational table (`raw_events`) to store incoming data, completing the standard object hierarchy.

# When Should We Use This? 

* When initially setting up the data architecture, including databases, schemas, and compute warehouses, to ensure all objects are centrally managed and owned by the correct system role.
* When establishing a role-based access control (RBAC) hierarchy where custom roles need to roll up to a single administrative role for ongoing maintenance, oversight, and object lifecycle management.
