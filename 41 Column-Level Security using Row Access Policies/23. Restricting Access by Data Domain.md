# Restricting Access by Data Domain in Snowflake

Restricting access by data domain in Snowflake involves implementing granular control over who can see what data based on specific business classifications or categories. This is a crucial security measure to ensure data privacy, compliance, and proper data governance within an organization. It prevents unauthorized users from accessing sensitive information even if they have general access to a table.

**Short Explanation**

This concept allows you to filter the rows a user can see in a table based on their assigned data domains or roles. It solves the problem of broad table access, where a user might need to query a table but should only see data relevant to their specific domain, like a particular region or department. It's important for maintaining data security and adhering to regulatory requirements, especially in large enterprises with diverse data and user bases. This is commonly used in multi-tenant environments, or when different teams are responsible for different subsets of data within the same analytical platform.

- **What problem does this SQL feature solve?** It solves the problem of providing row-level security, ensuring users only see data they are authorized to access, preventing data leakage and unauthorized exposure of sensitive information within a shared dataset.
- **Why is it important in databases or analytics?** It's vital for data governance, regulatory compliance (like GDPR, HIPAA), and maintaining trust by enforcing strict data access policies, allowing for secure data sharing and analysis without duplicating data.
- **Where is it commonly used in real workflows?** It's frequently used in financial services (customer data by region), healthcare (patient data by department), multi-national companies (sales data by country), and SaaS providers (tenant data isolation).

## Implementation Example

```sql
-- Create a sample table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    amount DECIMAL(10, 2),
    region VARCHAR,
    sale_date DATE
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 1200.00, 'EMEA', '2025-01-15'),
(2, 'Mouse', 25.00, 'AMER', '2025-01-16'),
(3, 'Keyboard', 75.00, 'EMEA', '2025-01-17'),
(4, 'Monitor', 300.00, 'APAC', '2025-01-18'),
(5, 'Webcam', 50.00, 'AMER', '2025-01-19');

-- Create a security policy (Row Access Policy)
CREATE OR REPLACE ROW ACCESS POLICY sales_policy
AS (region_col VARCHAR) RETURNS BOOLEAN ->
    CURRENT_ROLE() IN ('ACCOUNTADMIN', 'ANALYST_GLOBAL') OR
    (CURRENT_ROLE() = 'ANALYST_EMEA' AND region_col = 'EMEA') OR
    (CURRENT_ROLE() = 'ANALYST_AMER' AND region_col = 'AMER') OR
    (CURRENT_ROLE() = 'ANALYST_APAC' AND region_col = 'APAC');

-- Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);

-- Grant roles to a user (example)
-- GRANT ROLE ANALYST_EMEA TO USER some_user;

-- Test access for a user with ANALYST_EMEA role
-- USE ROLE ANALYST_EMEA;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example demonstrates how to implement row-level security using a `ROW ACCESS POLICY` in Snowflake.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a new table named `sales_data` with columns for sale ID, product, amount, region, and sale date. This is the table on which access will be restricted.
- **`INSERT INTO sales_data VALUES (...)`**: This populates the `sales_data` table with example records, including different `region` values that will be used for access control.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_col VARCHAR) RETURNS BOOLEAN -> ...`**:
    - **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy`**: This defines a new row access policy named `sales_policy`.
    - **`AS (region_col VARCHAR) RETURNS BOOLEAN ->`**: This specifies that the policy expects one argument, `region_col` (which will correspond to the `region` column in our table), and it returns a boolean value.
    - **`CURRENT_ROLE() IN ('ACCOUNTADMIN', 'ANALYST_GLOBAL') OR ...`**: This is the core logic. It returns `TRUE` (allowing the row to be seen) if:
        - The current user's role is `ACCOUNTADMIN` or `ANALYST_GLOBAL` (these roles see all data).
        - OR if the `CURRENT_ROLE()` matches a specific analyst role (e.g., `ANALYST_EMEA`) AND the `region_col` for that row matches the corresponding region (e.g., `'EMEA'`). This limits analysts to their assigned regions.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);`**:
    - **`ALTER TABLE sales_data`**: This command modifies the `sales_data` table.
    - **`ADD ROW ACCESS POLICY sales_policy ON (region)`**: This applies the previously defined `sales_policy` to the `sales_data` table. The `region` column of the `sales_data` table is passed as the `region_col` argument to the policy function. Any queries on `sales_data` will now automatically invoke this policy to filter rows based on the user's role.
- **`USE ROLE ANALYST_EMEA; SELECT * FROM sales_data;`**: (Commented out in the example but essential for testing). This would simulate a user with the `ANALYST_EMEA` role querying the `sales_data` table. Due to the policy, they would only see rows where the `region` is 'EMEA'.

## When to Use

1.  **Multi-tenant applications**: If you're building an application where multiple customers share the same underlying data tables, but each customer should only see their own data.
    *   *Scenario*: A SaaS analytics platform stores data for various companies in one large table. A row access policy ensures Company A's users only see Company A's data, and Company B's users only see Company B's data.
2.  **Regulatory compliance**: When specific data privacy regulations (e.g., GDPR, HIPAA) mandate that certain individuals or departments can only access specific subsets of sensitive data.
    *   *Scenario*: In a healthcare database, doctors should only see patient records from their department or for patients they are treating, not all patient records across the entire hospital.
3.  **Internal departmental data segregation**: To ensure that different departments within a large organization can only view and analyze data relevant to their operational domain.
    *   *Scenario*: A global sales report table where the EMEA sales team should only see sales figures for the EMEA region, and the AMER sales team only for the AMER region.

## When Not to Use

1.  **Simple table-level access control is sufficient**: If you only need to grant or revoke access to entire tables for specific roles, row-level security adds unnecessary overhead.
    *   *Situation*: A small team works on a dataset where everyone needs to see all rows. `GRANT SELECT ON TABLE` is simpler and more efficient.
2.  **Performance-critical queries on very large tables without proper indexing**: Row access policies add a filter condition to every query, which can impact performance, especially if the filtering column is not optimized or if the policy logic is complex.
    *   *Situation*: A table with billions of rows and a frequently queried policy column might suffer significant performance degradation if not carefully implemented and indexed.
3.  **Data segregation is handled at the application layer**: If your application already has robust mechanisms to filter data presented to users based on their login and permissions, implementing a duplicate layer of row-level security in the database might be redundant and harder to maintain.
    *   *Situation*: A front-end application already queries data and applies business logic to filter results for specific users, making database-level row access policies unnecessary overhead.

## Common Mistakes

1.  **Not testing with different roles**: Developers often test with `ACCOUNTADMIN` or their own elevated roles, which might bypass policies, leading to a false sense of security. Always test policies with the actual roles that will be subject to them.
2.  **Overly complex policy logic**: Creating policies with very intricate `WHERE` clause conditions or multiple subqueries can severely impact query performance and become difficult to debug and maintain.
3.  **Forgetting to apply the policy**: Defining a `ROW ACCESS POLICY` is only half the battle; it must be explicitly applied to the target table(s) using `ALTER TABLE ... ADD ROW ACCESS POLICY`. Forgetting this step means the policy isn't active.
4.  **Implicitly granting access**: If the policy allows access for `ACCOUNTADMIN` or other highly privileged roles without proper safeguards, these roles could accidentally or intentionally view data they shouldn't.

## Expected Output

The `SELECT * FROM sales_data;` query, when executed by a user with the `ANALYST_EMEA` role, would only return rows where the `REGION` column is 'EMEA'.

**Example Table (for `ANALYST_EMEA` role):**

| SALE_ID | PRODUCT_NAME | AMOUNT | REGION | SALE_DATE |
| :------ | :----------- | :----- | :----- | :-------- |
| 1       | Laptop       | 1200.00 | EMEA   | 2025-01-15 |
| 3       | Keyboard     | 75.00  | EMEA   | 2025-01-17 |

- **SALE_ID**: A unique identifier for each sale.
- **PRODUCT_NAME**: The name of the product sold.
- **AMOUNT**: The monetary value of the sale.
- **REGION**: The geographical region where the sale occurred. This is the column used by the row access policy to filter data.
- **SALE_DATE**: The date the sale was made.
