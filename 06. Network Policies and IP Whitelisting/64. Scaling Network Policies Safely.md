# Scaling Network Policies Safely in Snowflake

Snowflake Network Policies are a critical security feature that allows organizations to control inbound network access to their Snowflake account, ensuring that only authorized users and systems can connect. Historically, network policies directly contained lists of allowed or blocked IP addresses. However, Snowflake has introduced Network Rules as a more flexible and scalable approach to managing these policies, especially when dealing with complex or evolving network landscapes. This shift enables safer and more manageable scaling of network access controls.

**Short Explanation**

Network Policies in Snowflake act as a virtual firewall, defining which IP addresses are permitted or denied access. With the introduction of Network Rules, these policies now reference reusable sets of IP addresses or other network identifiers, rather than embedding them directly.

- **What problem does this SQL feature solve?** It addresses the challenge of securely managing and scaling network access to Snowflake, especially for large organizations with numerous IP ranges, remote users, or integrations. It prevents unauthorized network access and helps meet compliance requirements.
- **Why is it important in databases or analytics?** In data platforms like Snowflake, sensitive data is often stored and processed. Network policies are crucial for establishing a strong perimeter defense, preventing data breaches from external networks, and ensuring that only trusted sources can interact with the data.
- **Where is it commonly used in real workflows?** It's used across all Snowflake environments, from development to production. Common scenarios include restricting access to corporate networks (offices, VPNs), specific cloud provider endpoints (AWS PrivateLink, Azure Private Link), or particular users and security integrations.

## Implementation Example

This example demonstrates how to create a network rule with a list of allowed IP addresses and then apply it to a network policy, which is subsequently activated at the account level.

```sql
-- Step 1: Create a Network Rule to define allowed IP addresses
CREATE OR REPLACE NETWORK RULE my_corp_network_rule
  TYPE = IPV4
  VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')
  MODE = INGRESS
  COMMENT = 'Allowed IP ranges for corporate network and a specific user';

-- Step 2: Create a Network Policy that references the Network Rule
CREATE OR REPLACE NETWORK POLICY my_security_policy
  ALLOWED_NETWORK_RULE_LIST = ('my_corp_network_rule')
  BLOCKED_NETWORK_RULE_LIST = ()
  COMMENT = 'Policy restricting access to corporate IP ranges via network rule';

-- Step 3: Activate the Network Policy at the account level
ALTER ACCOUNT SET NETWORK_POLICY = my_security_policy;

-- Optional: Activate the Network Policy for a specific user
-- ALTER USER 'JOHN_DOE' SET NETWORK_POLICY = my_security_policy;

-- Optional: View the currently active account network policy
SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;

-- Optional: View all network rules
SHOW NETWORK RULES;

-- Optional: View all network policies
SHOW NETWORK POLICIES;
```

## Explanation of the Code

- `CREATE OR REPLACE NETWORK RULE my_corp_network_rule ...`: This statement defines a reusable object called a "network rule."
    - `TYPE = IPV4`: Specifies that the rule will operate on IPv4 addresses. Other types like VPCE IDs (for AWS PrivateLink) or LinkIDs (for Azure Private Link) are also supported.
    - `VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')`: This is the core of the rule, listing the actual IP addresses or CIDR blocks that are allowed or blocked. The CIDR notation (`/24`) indicates a range of IP addresses.
    - `MODE = INGRESS`: This signifies that the rule applies to incoming traffic to Snowflake.
    - `COMMENT = '...'`: Provides a descriptive text for the rule, which is helpful for documentation and auditing.
    - **How it changes the result set:** It doesn't directly change a dataset but creates a security object that will be referenced by network policies to filter connection attempts.

- `CREATE OR REPLACE NETWORK POLICY my_security_policy ...`: This statement creates a network policy that utilizes the previously defined network rule.
    - `ALLOWED_NETWORK_RULE_LIST = ('my_corp_network_rule')`: Instead of listing IPs directly, this policy refers to the `my_corp_network_rule`. This makes policies much more manageable, especially if IP ranges change or are shared across multiple policies.
    - `BLOCKED_NETWORK_RULE_LIST = ()`: An optional list for network rules defining explicitly blocked network identifiers. If an IP is in both `ALLOWED` and `BLOCKED` lists, `BLOCKED` takes precedence.
    - **How it changes the result set:** Similar to network rules, this creates a security object. It acts as the container for network rules and is the object that can be applied to accounts, users, or security integrations.

- `ALTER ACCOUNT SET NETWORK_POLICY = my_security_policy;`: This command activates the `my_security_policy` for the entire Snowflake account.
    - **How it changes the result set:** Once applied, all subsequent connection attempts to the Snowflake account will be evaluated against the rules defined in `my_security_policy`. Connections from IPs not listed in the `ALLOWED_NETWORK_RULE_LIST` (or explicitly in an `ALLOWED_IP_LIST` if using the legacy method) will be denied.

- `ALTER USER 'JOHN_DOE' SET NETWORK_POLICY = my_security_policy;` (Optional): This applies the network policy to a specific user, overriding any account-level policy for that user.
    - **How it changes the result set:** Only connections from the allowed IP addresses will be successful for `JOHN_DOE`. This provides granular control for individual users or groups.

## When to Use

1.  **Restricting access to corporate networks:** When you want to ensure that access to Snowflake is only possible from your company's offices, VPNs, or specific cloud environments.
    *   *Scenario:* A financial institution needs to prevent access to its Snowflake data warehouse from outside its secure corporate network to comply with strict regulatory requirements.
2.  **Granular user-specific access control:** For specific users or groups who require different network access rules than the general account-level policy.
    *   *Scenario:* A team of external consultants needs temporary access from their office IP, while internal users have a broader corporate network policy. You can apply a specific policy to the consultant's user account.
3.  **Managing large and dynamic IP lists:** When your allowed IP ranges are numerous or frequently change, using Network Rules makes management much easier. Instead of modifying multiple policies, you only update the underlying rule.
    *   *Scenario:* A global company has many regional offices, each with its own set of public IPs. Instead of defining these IPs in each policy, they create a single network rule `GLOBAL_OFFICE_IPS` and reference it across all relevant network policies.

## When Not to Use

1.  **If a single IP list is sufficient and static:** For very small organizations with a simple, unchanging set of allowed IPs, directly using `ALLOWED_IP_LIST` within a Network Policy (the legacy method) might seem simpler, though Network Rules are now the recommended best practice for reusability.
2.  **For outbound traffic control:** Network Policies (and Network Rules with `MODE = INGRESS`) are designed to control *inbound* access to Snowflake. They are not used to restrict outbound connections from Snowflake to external destinations (e.g., external stages, external functions). For outbound control, Snowflake's External Network Access features are used.
3.  **When physical network segmentation is already in place and sufficient:** In extremely isolated environments where direct network segmentation (e.g., private cloud, dedicated connections like PrivateLink with strict routing) already guarantees zero public access, the additional layer of IP-based network policies might be redundant for basic access, but still useful for fine-grained control within the private network.

## Common Mistakes

1.  **Locking yourself out:** Forgetting to include your current IP address (or the IP of your administrative access point) in the allowed list when creating or activating a new network policy. This can lead to being unable to access your Snowflake account. Snowflake typically prevents activation if it detects this.
2.  **Not using Network Rules for scalability:** Sticking to the older method of embedding long lists of IP addresses directly in Network Policies. This makes policies harder to manage, update, and audit, especially as the number of allowed IPs grows or changes.
3.  **Misunderstanding precedence:** Assuming that user-level policies always override account-level policies in all scenarios. While user-level policies are more granular, understanding the full hierarchy (user > integration > account) is crucial. Also, confusing `ALLOWED_IP_LIST` and `BLOCKED_IP_LIST` behavior (blocked takes precedence).
4.  **Mixing `IPV4` with PrivateLink identifiers:** Attempting to define network rules with `TYPE = IPV4` for private connectivity (AWS PrivateLink, Azure Private Link) where VPCE IDs or LinkIDs should be used instead.
5.  **Forgetting to activate the policy:** Creating a network policy but not activating it at the account, user, or integration level means it will not be enforced.

## Expected Output

When a network policy is successfully applied, the direct output is typically a confirmation message from Snowflake, not a data set. However, the *expected outcome* in terms of system behavior is that only connections originating from the specified allowed IP addresses (defined in the referenced network rules) will be able to successfully connect to the Snowflake account. Attempts from any other IP address will be rejected.

If you query for the activated policy, you'd see something like this:

| KEY            | VALUE             | DEFAULT   | LEVEL   |
| :------------- | :---------------- | :-------- | :------ |
| NETWORK_POLICY | MY_SECURITY_POLICY | NULL      | ACCOUNT |

-   **KEY**: The parameter name being displayed.
-   **VALUE**: The name of the network policy currently active for the account. In this case, `MY_SECURITY_POLICY`.
-   **DEFAULT**: Indicates if this is the default value (NULL means it's explicitly set).
-   **LEVEL**: The scope at which the parameter is set (e.g., `ACCOUNT`, `USER`).

This table shows that `MY_SECURITY_POLICY` is now the active network policy for the entire Snowflake account, meaning its rules are now enforced for all incoming connections.
