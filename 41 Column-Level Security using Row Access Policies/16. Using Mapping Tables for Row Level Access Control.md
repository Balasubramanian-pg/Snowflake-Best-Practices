# Using Mapping Tables for Row Level Access Control in Snowflake

Mapping tables for row-level access control in Snowflake provide a dynamic and flexible way to restrict which rows a user or role can see in a given table. This mechanism ensures data security and compliance by allowing different users to view only the data relevant and authorized for them, without duplicating data or creating multiple views.

**Short Explanation**

This approach involves creating a separate table (the "mapping table") that links users or roles to specific data dimensions (e.g., regions, departments, customer IDs). A secure view is then created over the sensitive data table, incorporating a `JOIN` with the mapping table and a `WHERE` clause to filter rows based on the current user's or role's permissions defined in the mapping table.

- **What problem does this SQL feature solve?** It solves the problem of enforcing fine-grained data access policies, ensuring users only see data they are authorized for, which is crucial for compliance and data privacy.
- **Why is it important in databases or analytics?** It's vital for maintaining data governance, preventing unauthorized data exposure, and simplifying data management by avoiding hardcoded access rules within applications.
- **Where is it commonly used in real workflows?** It's commonly used in multi-tenant applications, financial services, healthcare, and any scenario where different departments, regions, or client groups need restricted access to a shared dataset.

## Implementation Example

```sql
-- 1. Create a sample sensitive data table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sale_amount DECIMAL(10, 2),
    sale_date DATE
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 'North', 1200.00, '2023-01-05'),
(2, 'Mouse', 'South', 25.00, '2023-01-06'),
(3, 'Keyboard', 'North', 75.00, '2023-01-07'),
(4, 'Monitor', 'East', 300.00, '2023-01-08'),
(5, 'Webcam', 'South', 50.00, '2023-01-09'),
(6, 'Printer', 'West', 150.00, '2023-01-10');

-- 2. Create the mapping table
CREATE OR REPLACE TABLE user_region_access (
    username VARCHAR,
    allowed_region VARCHAR
);

-- Insert access rules: user JOHN can see North and South, user JANE can see East and West
INSERT INTO user_region_access VALUES
('JOHN', 'North'),
('JOHN', 'South'),
('JANE', 'East'),
('JANE', 'West');

-- 3. Create a secure view that enforces row-level security
CREATE OR REPLACE SECURE VIEW sales_data_secure AS
SELECT
    sd.sale_id,
    sd.product_name,
    sd.region,
    sd.sale_amount,
    sd.sale_date
FROM
    sales_data sd
JOIN
    user_region_access ura ON sd.region = ura.allowed_region
WHERE
    ura.username = CURRENT_USER(); -- CURRENT_USER() is a Snowflake context function
```

## Explanation of the Code

This example sets up a secure view using a mapping table to control row-level access.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This defines our main fact table containing sensitive sales information, including `region`.
- **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with example records across different regions.
- **`CREATE OR REPLACE TABLE user_region_access (...)`**: This is the mapping table. It stores associations between `username` and `allowed_region`. This table dictates which regions each user is permitted to see.
- **`INSERT INTO user_region_access VALUES (...)`**: Populates the mapping table, defining that user 'JOHN' can access 'North' and 'South' data, while user 'JANE' can access 'East' and 'West'.
- **`CREATE OR REPLACE SECURE VIEW sales_data_secure AS ...`**: This is the core of the solution.
    - **`SELECT ... FROM sales_data sd`**: Selects all relevant columns from the original `sales_data` table, aliased as `sd`.
    - **`JOIN user_region_access ura ON sd.region = ura.allowed_region`**: This `JOIN` clause connects the `sales_data` table with the `user_region_access` mapping table. It links sales records to access rules based on matching regions.
        - **What it does**: It combines rows from `sales_data` and `user_region_access` where the `region` in `sales_data` matches an `allowed_region` in `user_region_access`.
        - **How it changes the result set**: For each row in `sales_data`, it finds corresponding access rules. Rows without a match in the mapping table for the current user's permitted regions will be implicitly excluded by the `WHERE` clause.
    - **`WHERE ura.username = CURRENT_USER()`**: This `WHERE` clause is critical. It filters the joined results to only show rows where the `username` in the mapping table matches the `CURRENT_USER()` Snowflake context function.
        - **What it does**: It applies a filter based on the currently logged-in user. `CURRENT_USER()` dynamically returns the username of the session.
        - **How it changes the result set**: It restricts the output to only those rows from `sales_data` that are associated with the regions permitted for the `CURRENT_USER()` in the `user_region_access` table. If 'JOHN' queries the view, only 'North' and 'South' region data will be visible. If 'JANE' queries, only 'East' and 'West' region data will be visible.

## When to Use

1.  **Multi-tenant data architectures**: When different customers or internal departments share the same underlying data table but should only see their specific records.
    *   **Example Scenario**: A SaaS application where `CUSTOMER_A` should only see data related to `CUSTOMER_A_ID`, and `CUSTOMER_B` only sees `CUSTOMER_B_ID`.
2.  **Compliance and regulatory requirements**: When strict data access policies must be enforced (e.g., GDPR, HIPAA) to prevent unauthorized disclosure of sensitive information.
    *   **Example Scenario**: In healthcare, doctors or staff should only see patient records from their assigned clinic or department.
3.  **Dynamic access control based on roles or attributes**: When access rules are complex and can change frequently, or depend on user roles or attributes rather than static permissions.
    *   **Example Scenario**: A manager role can see data for all employees in their department, while an individual employee can only see their own records. The mapping table can link roles to departments.

## When Not to Use

1.  **Very high-performance, simple direct table access**: If the underlying table is extremely large, and queries need to be as fast as possible without any filtering, and there are no security concerns for all users to see all data.
    *   **Reason**: The `JOIN` operation, even with well-indexed tables, adds a slight overhead compared to direct table access.
2.  **Static, unchangeable permissions for a small number of users**: If access rules are fixed, simple, and rarely change for a very small set of users, hardcoded views or grants might be simpler to manage initially.
    *   **Reason**: The overhead of maintaining the mapping table and view might be excessive for trivial cases.
3.  **When column-level security is the primary requirement**: If the need is to hide certain columns for some users, rather than specific rows.
    *   **Reason**: Mapping tables for RLS are designed for row filtering. Column-level access control requires different mechanisms (e.g., Snowflake's column-level security features or projecting only allowed columns in a view).

## Common Mistakes

1.  **Forgetting to grant access to the secure view**: Users will get "object does not exist" errors if they don't have `SELECT` privilege on the `sales_data_secure` view.
2.  **Not securing the mapping table**: If users can directly query or modify the `user_region_access` mapping table, they could potentially bypass the security controls. The mapping table should only be accessible to administrators.
3.  **Using `CURRENT_ROLE()` instead of `CURRENT_USER()` incorrectly**: Depending on the granularity needed, developers might confuse when to use `CURRENT_USER()` (specific user) versus `CURRENT_ROLE()` (current active role) in the `WHERE` clause of the secure view. It's crucial to align the mapping table with the chosen context function.
4.  **Inefficient `JOIN` conditions or missing indexes (in general SQL systems)**: While Snowflake handles many performance aspects, complex `JOIN` predicates or very large mapping tables without proper clustering keys could potentially impact query performance if not designed carefully.

## Expected Output

When a user (`JOHN` or `JANE`) queries the `sales_data_secure` view, they will only see the sales records from the regions they are allowed to access based on the `user_region_access` table. The output will be a subset of the `sales_data` table's columns and rows.

**Example Table for User JOHN (who can access 'North' and 'South'):**

| SALE_ID | PRODUCT_NAME | REGION | SALE_AMOUNT | SALE_DATE  |
| :------ | :----------- | :----- | :---------- | :--------- |
| 1       | Laptop       | North  | 1200.00     | 2023-01-05 |
| 2       | Mouse        | South  | 25.00       | 2023-01-06 |
| 3       | Keyboard     | North  | 75.00       | 2023-01-07 |
| 5       | Webcam       | South  | 50.00       | 2023-01-09 |

-   **SALE_ID**: Unique identifier for each sale.
-   **PRODUCT_NAME**: The name of the product sold.
-   **REGION**: The geographical region where the sale occurred. This column is used for filtering access.
-   **SALE_AMOUNT**: The monetary value of the sale.
-   **SALE_DATE**: The date the sale took place.

The table for user JOHN shows only rows where the `REGION` is 'North' or 'South', effectively hiding data from 'East' and 'West' regions. The structure remains identical to the original `sales_data` table, but the visible rows are filtered according to the user's permissions defined in the mapping table.
