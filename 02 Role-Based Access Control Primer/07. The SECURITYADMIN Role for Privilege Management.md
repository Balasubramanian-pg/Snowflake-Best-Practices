# 07. The SECURITYADMIN Role for Privilege Management

The `SECURITYADMIN` role is a highly privileged system role dedicated to managing access control, user provisioning, and role hierarchies within the data platform. It operates globally to grant, revoke, and audit privileges across the entire account without requiring access to the underlying data itself.

# Key Points

* It possesses the `MANAGE GRANTS` global privilege, allowing it to modify any grant in the system, including those on objects owned by other roles.
* It inherently inherits the privileges of the `USERADMIN` role, meaning it can create, alter, and drop users and roles across the account.
* It is the canonical role for implementing and maintaining the Principle of Least Privilege (PoLP) and centralized Role-Based Access Control (RBAC) architectures.
* Unlike the `SYSADMIN` role, it does not inherently have access to create or manage databases, virtual warehouses, or schema objects unless explicitly granted.
* > [!IMPORTANT]
  > Because `SECURITYADMIN` can modify access to any object in the system, its usage should be strictly limited to dedicated security administrators to prevent unauthorized privilege escalation.

# Implementation

```sql
USE ROLE SECURITYADMIN;

CREATE ROLE data_analyst_role;

GRANT USAGE ON DATABASE analytics_db TO ROLE data_analyst_role;
GRANT USAGE ON SCHEMA analytics_db.public TO ROLE data_analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics_db.public TO ROLE data_analyst_role;

GRANT ROLE data_analyst_role TO USER jsmith;
```

# What does the Query do?

Step 1 switches the active session context to the `SECURITYADMIN` role to ensure the subsequent access control commands are executed with the correct administrative authority.

Step 2 creates a new custom role named `data_analyst_role` (leveraging the inherited `USERADMIN` privileges) to group specific access rights for analytical users.

Step 3 grants the necessary `USAGE` and `SELECT` privileges on the `analytics_db` database, schema, and underlying tables to the newly created role, establishing the access boundary.

Step 4 assigns the `data_analyst_role` to the user `jsmith`, effectively provisioning the user with the read-only access required to query the analytics tables.

# When Should We Use This? 

* Centralizing the management of Role-Based Access Control (RBAC) hierarchies, ensuring that object owners (like `SYSADMIN`) do not have to manually manage individual user access.
* Conducting security audits, provisioning new departments, or bulk-revoking privileges during offboarding processes without exposing sensitive table data to the security team.
