# Restricting Snowflake Public Endpoints

This concept refers to the security practice of limiting network access to your Snowflake account via its public endpoints. By default, Snowflake accounts are accessible from any public IP address, which might not align with an organization's security policies. Restricting these endpoints enhances security by ensuring only authorized networks can connect.

**Short Explanation**

Restricting Snowflake public endpoints involves configuring network policies to define which IP addresses or ranges are allowed to connect to your Snowflake account. This acts as a firewall at the Snowflake account level, preventing unauthorized access attempts from unknown locations. It's crucial for compliance and protecting sensitive data, especially in environments with strict data governance requirements. This is commonly used in enterprise settings where data access needs to be tightly controlled and limited to corporate networks, VPNs, or specific cloud environments.

- **What problem does this SQL feature solve?** It solves the problem of unauthorized network access to Snowflake accounts, reducing the attack surface and enhancing data security.
- **Why is it important in databases or analytics?** It's critical for protecting sensitive data, meeting compliance regulations (like HIPAA, GDPR, PCI DSS), and ensuring only trusted environments can interact with your data warehouse.
- **Where is it commonly used in real workflows?** It's used in any organization that processes sensitive data and requires strict network perimeter security, often as part of their overall cloud security posture.

## Implementation Example

While endpoint restriction isn't done with SQL *queries* in the traditional sense, it's configured using SQL `ALTER ACCOUNT` statements for network policies within Snowflake.

```sql
-- 1. Create a Network Policy
CREATE NETWORK POLICY my_network_policy
    ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.100')
    COMMENT = 'Allow access only from corporate VPN and specific office IP';

-- 2. Activate the Network Policy for the account
ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy;

-- 3. (Optional) Deactivate the Network Policy
-- ALTER ACCOUNT UNSET NETWORK_POLICY;
```

## Explanation of the Code

This code block demonstrates how to create and apply a network policy in Snowflake to restrict access.

- **`CREATE NETWORK POLICY my_network_policy`**: This SQL command initiates the creation of a new network policy named `my_network_policy`.
- **`ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.100')`**: This clause defines the core of the network policy, specifying a list of IP addresses or CIDR blocks that are permitted to connect to the Snowflake account. Any connection attempt from an IP address not on this list will be denied.
    - `192.168.1.0/24`: This is a CIDR block representing all IP addresses from 192.168.1.0 to 192.168.1.255.
    - `10.0.0.100`: This is a single, specific IP address.
- **`COMMENT = 'Allow access only from corporate VPN and specific office IP'`**: This adds a descriptive comment to the network policy, which is useful for documentation and understanding its purpose.
- **`ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy;`**: This command applies the previously created `my_network_policy` to the entire Snowflake account. Once executed, only IPs in the `ALLOWED_IP_LIST` will be able to connect.
    - **What it does**: This statement activates the specified network policy at the account level.
    - **How it changes the result set**: This doesn't change a query result set directly but controls which users (from which IPs) can even execute queries against the account.
- **`ALTER ACCOUNT UNSET NETWORK_POLICY;`**: This (optional) command removes the network policy from the account, reverting to the default behavior of allowing connections from any public IP.

## When to Use

1.  **Enhanced Security for Sensitive Data**: When your Snowflake account stores highly sensitive data (e.g., PII, financial records, health data) that requires restricted access.
    *   *Example Scenario*: A financial institution storing customer transaction data wants to ensure that analysts can only access Snowflake from their secure corporate network or VPN, not from personal devices or public Wi-Fi.
2.  **Meeting Compliance Requirements**: For organizations needing to comply with industry regulations (e.g., HIPAA, GDPR, PCI DSS) that mandate strict network access controls.
    *   *Example Scenario*: A healthcare provider must demonstrate that access to patient data in Snowflake is restricted to approved internal networks, aligning with HIPAA's security rules.
3.  **Preventing Unauthorized Access**: To reduce the overall attack surface and protect against brute-force attacks or unauthorized connection attempts from unknown IP addresses.
    *   *Example Scenario*: An organization experiences repeated login attempts from suspicious IP ranges and decides to implement a network policy to whitelist only known, legitimate access points.

## When Not to Use

1.  **Highly Dynamic or Remote Workforces Without VPN**: If users frequently access Snowflake from various, untrusted locations without a consistent VPN, a restrictive network policy will block their legitimate access.
    *   *Example Scenario*: A startup with a fully remote team scattered globally, where not all team members use a corporate VPN, would find a global network policy too restrictive and disruptive.
2.  **Publicly Accessible Applications or Integrations**: When Snowflake needs to be accessed by external, public-facing applications or services that do not have static IP addresses or are distributed globally.
    *   *Example Scenario*: A web application that directly queries Snowflake for real-time data and is hosted on a dynamic cloud platform (e.g., serverless functions with changing IPs) would be blocked by a static IP whitelist.
3.  **During Initial Setup or Troubleshooting**: Applying a strict network policy prematurely during initial setup or when debugging connectivity issues can lead to lockout and significant delays.
    *   *Example Scenario*: A new team is onboarding to Snowflake, and a network policy is immediately enforced, preventing them from connecting and configuring their environments because their IPs haven't been added yet.

## Common Mistakes

1.  **Locking Yourself Out**: Forgetting to include your own current IP address or the IP range of your administrative access point in the `ALLOWED_IP_LIST`.
2.  **Using Overly Broad IP Ranges**: Specifying a `0.0.0.0/0` (allow all) or a very wide CIDR block, which defeats the purpose of restricting access.
3.  **Not Updating Policies**: Failing to update the `ALLOWED_IP_LIST` when corporate network configurations change, or new legitimate access points are introduced.
4.  **Confusing Account vs. User-Level Policies**: Not understanding that network policies can be set at the account level (global) or specific user levels, and misapplying the scope.
5.  **Forgetting to Activate**: Creating a network policy but forgetting to use `ALTER ACCOUNT SET NETWORK_POLICY = ...` to activate it for the account.

## Expected Output

The "output" of these commands is not a dataset in the traditional sense, but rather a change in the security configuration of your Snowflake account. When you query network policies or your account settings, you would see the policy's details.

**Querying Network Policy Status**

To check if a network policy is set for your account, you can use:

```sql
SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;
```

**Example Output (Illustrative Table)**

| key            | value             | default | description                                           | level   |
| :------------- | :---------------- | :------ | :---------------------------------------------------- | :------ |
| NETWORK_POLICY | MY_NETWORK_POLICY | NULL    | Specifies the network policy for the account.         | ACCOUNT |

This output indicates that a network policy named `MY_NETWORK_POLICY` is currently active for the entire Snowflake account. The `value` column shows the name of the active policy. If no policy were active, `value` would be `NULL` or empty, depending on the client.
