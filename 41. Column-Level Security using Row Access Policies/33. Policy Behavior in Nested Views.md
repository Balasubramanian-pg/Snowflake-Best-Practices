# Policy Behavior in Nested Views in Snowflake

In Snowflake, **Policy Behavior in Nested Views** refers to how data governance policies, specifically Row Access Policies (RAPs) and Masking Policies, are applied and evaluated when multiple views are stacked on top of each other, eventually querying a base table. This is crucial for maintaining fine-grained access control and data security in complex data architectures. It ensures that regardless of how many layers of views are created, the defined security policies are consistently enforced.

**Short Explanation**

When you have a series of views (e.g., `View1` built on `BaseTable`, and `View2` built on `View1`), and policies are applied at different layers, Snowflake has a specific order of evaluation. This ensures that data access is always restricted according to the most stringent policies in place, providing robust security.

- **What problem does this SQL feature solve?** It solves the problem of consistent and inherited data security. Without this, a user might bypass security policies applied to a base table by querying a view built on top of it that doesn't explicitly have the policy applied. It prevents policy circumvention through view stacking.
- **Why is it important in databases or analytics?** It's critical for data governance, compliance (like GDPR, HIPAA), and multi-tenant architectures where different users or groups need varying levels of data access or obfuscation, even when using shared data sources.
- **Where is it commonly used in real workflows?** It's used when data is exposed through multiple analytical layers, such as reporting dashboards, data marts, or specialized user interfaces, where the underlying tables are protected by RAPs or Masking Policies.

## Implementation Example

Let's illustrate with a Row Access Policy applied to a base table, and then a view built on that table.

```sql
-- 1. Create a sample base table
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2)
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 'North', 1200.00),
(2, 'Mouse', 'South', 50.00),
(3, 'Keyboard', 'North', 150.00),
(4, 'Monitor', 'West', 300.00);

-- 2. Create a Row Access Policy (RAP)
CREATE OR REPLACE ROW ACCESS POLICY region_policy AS (region_col VARCHAR) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'ANALYST_NORTH' AND region_col = 'North' THEN TRUE
        WHEN CURRENT_ROLE() = 'ANALYST_SOUTH' AND region_col = 'South' THEN TRUE
        WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE
        ELSE FALSE
    END;

-- 3. Apply the RAP to the base table
ALTER TABLE sales_data ADD ROW ACCESS POLICY region_policy ON (region);

-- 4. Create a view on the base table
CREATE OR REPLACE VIEW sales_summary_view AS
SELECT
    order_id,
    product_name,
    region,
    sales_amount
FROM
    sales_data
WHERE sales_amount > 100;

-- 5. (Optional) Apply another RAP to the view itself (nested policy)
-- CREATE OR REPLACE ROW ACCESS POLICY product_type_policy AS (product VARCHAR) RETURNS BOOLEAN ->
-- CASE
-- WHEN CURRENT_ROLE() = 'PRODUCT_VIEWER' AND product IN ('Laptop', 'Monitor') THEN TRUE
-- WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE
-- ELSE FALSE
-- END;
-- ALTER VIEW sales_summary_view ADD ROW ACCESS POLICY product_type_policy ON (product_name);
```

## Explanation of the Code

This example focuses on a Row Access Policy, but the principle extends to Masking Policies.

- `CREATE OR REPLACE TABLE sales_data (...)`: This creates a basic table holding sales information, including a `region` column which will be used for row-level security.
- `CREATE OR REPLACE ROW ACCESS POLICY region_policy AS (region_col VARCHAR) RETURNS BOOLEAN ->...`: This defines a Row Access Policy named `region_policy`.
    - **What it does**: It takes a `region_col` (which will be mapped to the `region` column of the `sales_data` table) and returns `TRUE` or `FALSE` for each row.
    - **How it changes the result set**: Only rows for which the policy expression evaluates to `TRUE` for the `CURRENT_ROLE()` will be returned. For example, `ANALYST_NORTH` can only see rows where `region` is 'North'. `ADMIN_ROLE` can see all rows.
- `ALTER TABLE sales_data ADD ROW ACCESS POLICY region_policy ON (region);`: This command binds the `region_policy` to the `sales_data` table, specifically using the `region` column as the input to the policy.
    - **What it does**: Activates the row-level security on the `sales_data` table.
    - **How it changes the result set**: Every query against `sales_data` will now implicitly filter rows based on `region_policy`.
- `CREATE OR REPLACE VIEW sales_summary_view AS SELECT... FROM sales_data WHERE sales_amount > 100;`: This creates a view named `sales_summary_view` that selects data from the `sales_data` table and applies a filter on `sales_amount`.
    - **What it does**: Defines a logical representation of data from `sales_data`.
    - **How it changes the result set**: Filters `sales_data` to only include orders with `sales_amount` greater than 100 before any further policy evaluation.
- `ALTER VIEW sales_summary_view ADD ROW ACCESS POLICY product_type_policy ON (product_name);` (commented out in example): If this were uncommented, it would apply a *second* Row Access Policy directly to the `sales_summary_view`.
    - **What it does**: Adds an additional layer of row-level security specific to the view.
    - **How it changes the result set**: When a query accesses `sales_summary_view`, Snowflake evaluates the policy on the *base table first*, then the policy on the *view*. Both must evaluate to `TRUE` for a row to be returned.

**Policy Evaluation Order in Nested Views:**
Snowflake evaluates policies in a specific order:
1. **Row Access Policy on the base table** is always evaluated first.
2. Then, **Row Access Policy on the view** (if any) is evaluated.
3. **Masking Policies** are evaluated after Row Access Policies.

This means if `View2` is built on `View1` which is built on `BaseTable`, and all have RAPs, the policies will be applied from `BaseTable` -> `View1` -> `View2` sequentially. For a row to be visible, all applicable policies in the nested hierarchy must evaluate to `TRUE`.

## When to Use

1. **Multi-Tenant Data Architectures**: When different user groups (tenants) should only see their specific data, even if they're querying the same underlying tables or views.
    * *Example Scenario*: A sales organization has regional sales managers. Each manager should only see sales data from their assigned region. RAPs on base tables or views ensure this.
2. **Layered Data Access for Analytics**: When you have multiple layers of views for different analytical purposes (e.g., raw data view, transformed view, aggregated view) and need to ensure security is maintained across all layers.
    * *Example Scenario*: A `transactions` base table has PII masked by a policy. A `daily_sales_report` view is built on `transactions`, and an `executive_dashboard` view is built on `daily_sales_report`. Masking policies should seamlessly apply through all these views.
3. **Regulatory Compliance**: To enforce strict data visibility rules to meet compliance requirements (e.g., only authorized personnel can see specific types of data).
    * *Example Scenario*: Patient medical records in a `patients_data` table must be restricted based on the querying doctor's department. This policy must apply even if an `anonymized_patient_view` is created for research purposes.

## When Not to Use

1. **Performance Critical Workloads on Very Large Tables (without proper optimization)**: Policies add overhead because Snowflake must evaluate the policy expression for every row. If not properly optimized (e.g., policy columns aligned with clustering keys), this can impact performance significantly on massive tables.
    * *Situation*: Querying a base table with billions of rows and a complex RAP that involves subqueries or many joins in its definition, where the policy column is not clustered.
2. **Simple, Non-Sensitive Data**: Applying policies to data that doesn't require any form of row-level security or data masking adds unnecessary complexity and overhead.
    * *Situation*: A lookup table for country codes that is publicly accessible and contains no sensitive information.
3. **When Access Control is Best Handled by Standard Privileges**: For broad access control (e.g., a user can or cannot see an entire table), standard `GRANT SELECT` privileges are more efficient and simpler to manage than policies.
    * *Situation*: A user role should either have full access to a specific `product_catalog` table or no access at all. Using a policy to hide all rows for certain roles would be over-engineering.

## Common Mistakes

1. **Ignoring Policy Evaluation Order**: Assuming a policy on a view overrides a policy on a base table, leading to security gaps or unexpected results. Remember, base table policy *first*, then view policy.
2. **Performance Impact Neglect**: Not testing the performance implications of policies, especially on large datasets or with complex policy logic. This can lead to slow queries and poor user experience.
3. **Over-complicating Policy Logic**: Writing overly complex policy expressions with many `CASE` statements, subqueries, or joins within the policy definition. This makes them hard to understand, debug, and can hurt performance.
4. **Inconsistent Policy Application**: Applying policies inconsistently across different layers or related tables, leading to fragmented security or difficulty in auditing.
5. **Lack of Documentation**: Not documenting which policies are applied to which tables/views, and why, makes maintenance and troubleshooting extremely difficult.

## Expected Output

When a user queries a view that has policies applied either directly to it or to its underlying objects, the resulting dataset will only contain rows that satisfy *all* applicable policies, as determined by the `TRUE` return value of the policy expressions. Masked columns will appear obfuscated if a masking policy is also in effect.

**Example Table (Query by `ANALYST_NORTH` role on `sales_summary_view`)**

| ORDER_ID | PRODUCT_NAME | REGION | SALES_AMOUNT |
|----------|--------------|--------|--------------|
| 1 | Laptop | North | 1200.00 |
| 3 | Keyboard | North | 150.00 |

**Explanation of Columns:**
- `ORDER_ID`: The unique identifier for the sales order.
- `PRODUCT_NAME`: The name of the product sold.
- `REGION`: The geographic region where the sale occurred.
- `SALES_AMOUNT`: The total amount of the sale.

In this example, an `ANALYST_NORTH` user querying `sales_summary_view` would only see orders from the 'North' region where `sales_amount` is greater than `100`, because:
1. The `sales_amount > 100` filter from the view definition is applied.
2. The `region_policy` on the `sales_data` table restricts rows to `REGION = 'North'` for `ANALYST_NORTH`.
All rows not meeting these criteria would be filtered out and not appear in the result set.
