# Segregating Tenant Data by Schema vs Database in Snowflake

**Description:**
This topic discusses the best practices for segregating tenant data in Snowflake using schema and database structures. It highlights the pros and cons of using schema versus database for data segregation. By segregating tenant data, organizations can improve data security, reduce data complexity, and enhance data governance.

## Short Explanation
**Overview:** Segregating tenant data is crucial for multi-tenant environments to ensure data isolation and security.

**Problem Solved:** It solves the problem of data leakage and unauthorized access in multi-tenant environments.

**Importance:** It matters in analytics and warehousing as it enables secure data sharing and collaboration.

**Use Cases:**
- Multi-tenancy in SaaS applications
- Data partitioning in data warehouses

### Typical Scenario
A Software as a Service (SaaS) provider wants to host multiple tenants' data in a single Snowflake account. The provider needs to segregate each tenant's data to ensure data security and isolation.

### Core Snowflake SQL Objects Used
`CREATE SCHEMA`, `CREATE DATABASE`

### Code Source Execution
```sql
CREATE DATABASE tenant_data;
CREATE SCHEMA tenant_data.tenant1;
CREATE TABLE tenant_data.tenant1.customer (id INT, name STRING);
CREATE SCHEMA tenant_data.tenant2;
CREATE TABLE tenant_data.tenant2.customer (id INT, name STRING);
```

### Example Execution Logic Step-by-Step
In this example, we create a database named 'tenant_data' and two schemas 'tenant1' and 'tenant2' under it. Each schema contains a 'customer' table. This structure allows for data segregation at the schema level.

### Implementation Example
To implement segregation at the database level, you can create separate databases for each tenant:
CREATE DATABASE tenant1;
CREATE DATABASE tenant2;

### When to Use
- Use separate databases for large-scale tenants with complex data structures
- Use separate schemas for smaller tenants with simpler data structures

### When Not to Use
- Avoid using a single schema or database for all tenants in a large-scale multi-tenant environment

### Common Mistakes
- Not segregating data properly, leading to data leakage
- Overly complex schema or database structures that are hard to manage
