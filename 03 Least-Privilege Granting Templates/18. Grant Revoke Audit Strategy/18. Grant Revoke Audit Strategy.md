# Grant Revoke Audit Strategy in Snowflake

The Grant Revoke Audit Strategy in Snowflake refers to the systematic approach of managing, tracking, and reviewing access permissions (grants and revokes) within a Snowflake environment. This strategy is crucial for maintaining data security, ensuring compliance with regulatory requirements, and achieving granular control over who can access and manipulate data. It leverages Snowflake's robust Role-Based Access Control (RBAC) model and auditing capabilities to provide visibility and control.

**Short Explanation**

This strategy helps control data access by defining roles with specific permissions (grants) and removing unnecessary access (revokes). It solves the problem of unauthorized data access and ensures compliance by providing clear audit trails of all permission changes. It's vital for maintaining a secure data environment and is commonly used in regulated industries and any organization handling sensitive data.

- What problem does this SQL feature solve? It solves the problem of uncontrolled data access, reducing the risk of data breaches and ensuring that users only have the necessary permissions to perform their jobs.
- Why is it important in databases or analytics? It's important for security, compliance, and maintaining data integrity in complex database and analytics environments.
- Where is it commonly used in real workflows? It's used in security governance, compliance audits (like GDPR, HIPAA, SOX), and routine access management for data engineers, analysts, and administrators.

## Implementation Example

```sql
-- Step 1: Create custom roles following the principle of least privilege
CREATE ROLE DATA_ANALYST;
CREATE ROLE DATA_ENGINEER;
CREATE ROLE FINANCE_VIEWER;

-- Step 2: Grant specific privileges to roles
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE DATA_ANALYST;
GRANT USAGE ON DATABASE PROD_DB TO ROLE DATA_ANALYST;
GRANT USAGE ON SCHEMA PROD_DB.SALES TO ROLE DATA_ANALYST;
GRANT SELECT ON ALL TABLES IN SCHEMA PROD_DB.SALES TO ROLE DATA_ANALYST;
GRANT SELECT ON FUTURE TABLES IN SCHEMA PROD_DB.SALES TO ROLE DATA_ANALYST;

GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE DATA_ENGINEER;
GRANT USAGE ON DATABASE PROD_DB TO ROLE DATA_ENGINEER;
GRANT USAGE ON SCHEMA PROD_DB.RAW_DATA TO ROLE DATA_ENGINEER;
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA PROD_DB.RAW_DATA TO ROLE DATA_ENGINEER;
GRANT ALL PRIVILEGES ON FUTURE TABLES IN SCHEMA PROD_DB.RAW_DATA TO ROLE DATA_ENGINEER;

GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE FINANCE_VIEWER;
GRANT USAGE ON DATABASE PROD_DB TO ROLE FINANCE_VIEWER;
GRANT USAGE ON SCHEMA PROD_DB.FINANCE TO ROLE FINANCE_VIEWER;
GRANT SELECT ON TABLE PROD_DB.FINANCE.BUDGET_REPORTS TO ROLE FINANCE_VIEWER;

-- Step 3: Grant roles to users
GRANT ROLE DATA_ANALYST TO USER ANNA_SMITH;
GRANT ROLE DATA_ENGINEER TO USER JOHN_DOE;
GRANT ROLE FINANCE_VIEWER TO USER JANE_DOE;

-- Step 4: Revoke privileges as needed (e.g., if a user's role changes or data access is no longer required)
REVOKE SELECT ON TABLE PROD_DB.FINANCE.BUDGET_REPORTS FROM ROLE DATA_ANALYST;
REVOKE ROLE DATA_ENGINEER FROM USER JOHN_DOE;

-- Step 5: Audit grants and revokes using Account Usage views
SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_ROLES WHERE GRANTED_TO = 'ROLE' AND GRANTEE_NAME = 'DATA_ANALYST';
SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_USERS WHERE GRANTED_TO = 'USER' AND GRANTEE_NAME = 'ANNA_SMITH';
SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY WHERE USER_NAME = 'JOHN_DOE';
```

## Explanation of the Code

This example demonstrates a basic grant/revoke strategy and how to audit it in Snowflake.

- **`CREATE ROLE <role_name>;`**: This statement creates new custom roles. In Snowflake, permissions are granted to roles, and roles are then granted to users, promoting the RBAC model.
- **`GRANT USAGE ON WAREHOUSE <warehouse_name> TO ROLE <role_name>;`**: Grants the ability for the specified role to use a virtual warehouse for query execution.
- **`GRANT USAGE ON DATABASE <database_name> TO ROLE <role_name>;`**: Grants the ability for the specified role to access and use a database. This is a prerequisite for accessing objects within the database.
- **`GRANT USAGE ON SCHEMA <schema_name> TO ROLE <role_name>;`**: Grants the ability for the specified role to access and use a schema within a database. This is a prerequisite for accessing objects within the schema.
- **`GRANT SELECT ON ALL TABLES IN SCHEMA <schema_name> TO ROLE <role_name>;`**: Grants the `SELECT` privilege on all *existing* tables within a specified schema to a role. This allows the role to read data from these tables.
- **`GRANT SELECT ON FUTURE TABLES IN SCHEMA <schema_name> TO ROLE <role_name>;`**: Grants the `SELECT` privilege on any tables that will be created *in the future* within a specified schema to a role. This automates permission assignment for new objects.
- **`GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA <schema_name> TO ROLE <role_name>;`**: Grants all available privileges (e.g., SELECT, INSERT, UPDATE, DELETE) on all *existing* tables within a schema to a role.
- **`GRANT ALL PRIVILEGES ON FUTURE TABLES IN SCHEMA <schema_name> TO ROLE <role_name>;`**: Grants all available privileges on any tables that will be created *in the future* within a specified schema to a role.
- **`GRANT ROLE <role_name> TO USER <user_name>;`**: Assigns a previously created role to a specific user, allowing that user to inherit all privileges granted to that role.
- **`REVOKE SELECT ON TABLE <table_name> FROM ROLE <role_name>;`**: Removes the `SELECT` privilege on a specific table from a role. This is used to restrict access that is no longer needed.
- **`REVOKE ROLE <role_name> FROM USER <user_name>;`**: Removes a role from a user, revoking all privileges associated with that role for that user.
- **`SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_ROLES;`**: This query accesses a view in Snowflake's `ACCOUNT_USAGE` schema, which provides historical data about which privileges have been granted to roles.
- **`SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_USERS;`**: This query accesses a view in Snowflake's `ACCOUNT_USAGE` schema, showing which roles have been granted to users.
- **`SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY;`**: This view tracks user login attempts, including success and failure, which can be useful for auditing suspicious activity.

## When to Use

1. **Establishing Initial Access Controls**: When setting up a new Snowflake environment or onboarding new teams, this strategy is essential for defining clear, role-based access to data and resources from the outset.
    * *Scenario*: A new data science team needs access to specific analytical datasets. You create a `DATA_SCIENTIST` role, grant `SELECT` privileges on the necessary tables, and then grant this role to each data scientist.
2. **Compliance and Regulatory Requirements**: Organizations subject to regulations like HIPAA, GDPR, SOX, or PCI DSS need robust mechanisms to prove who accessed what data and when. Regular auditing of grants and revokes provides the necessary trails.
    * *Scenario*: An external auditor requests a report on all users who have had access to PII (Personally Identifiable Information) data in the past year. You can query `ACCOUNT_USAGE.GRANTS_TO_ROLES` and `ACCOUNT_USAGE.GRANTS_TO_USERS` to compile this report.
3. **Managing Privilege Lifecycle and "Least Privilege"**: As users' roles change or projects conclude, their access needs to be adjusted. The strategy ensures that users only have the minimum necessary permissions at any given time, preventing privilege escalation or data exposure.
    * *Scenario*: An intern finishes a project that required `INSERT` permissions on a staging table. You `REVOKE INSERT` on that table from the intern's temporary role, ensuring they no longer have write access.

## When Not to Use

1. **Granting `ACCOUNTADMIN` for routine tasks**: The `ACCOUNTADMIN` role has the highest privileges and should be used sparingly for account-level configurations only. Granting it for day-to-day operations bypasses the RBAC strategy and creates significant security risks.
    * *Situation*: A developer wants full control over everything for a quick test. Instead of using `ACCOUNTADMIN`, they should be granted a role with specific, limited privileges relevant to their testing.
2. **Granting privileges directly to users (User-Based Access Control - UBAC) for long-term access**: While Snowflake supports granting privileges directly to users, relying on UBAC for persistent access breaks the RBAC model, making access management complex and auditing difficult.
    * *Situation*: Instead of creating a `REPORT_USER` role and granting it to new report viewers, an administrator individually grants `SELECT` on various tables directly to each new user. This leads to privilege sprawl and inconsistency.
3. **Ignoring "future grants" for dynamically growing schemas**: If you have schemas where new tables are frequently created (e.g., data ingestion pipelines), failing to use `GRANT... ON FUTURE TABLES` will result in new objects being inaccessible until manual grants are applied, causing operational overhead and potential data access gaps.
    * *Situation*: A data pipeline continuously lands new daily tables into a `RAW_DATA` schema, but the `DATA_ANALYST` role only has `SELECT ON ALL TABLES`. The analysts can't query new tables until an administrator manually grants `SELECT` on each new table.

## Common Mistakes

1. **Over-privileging Roles**: Granting more permissions than necessary to roles (e.g., `ALL PRIVILEGES` when `SELECT` suffices). This violates the principle of least privilege and increases the attack surface.
2. **Not Using a Role Hierarchy**: Failing to structure roles hierarchically can lead to redundant grants and make managing complex permissions cumbersome. Effective hierarchies simplify auditing and management.
3. **Lack of Regular Auditing**: Not routinely reviewing who has what access, which can lead to "privilege creep" where users retain access they no longer need, creating security vulnerabilities.
4. **Inconsistent Application of Future Grants**: Neglecting to use `ON FUTURE` grants for schemas where objects are frequently added. This results in manual grant management for new objects and potential access gaps.
5. **Forgetting to Revoke Access**: When an employee leaves or changes roles, forgetting to revoke their previous access can leave sensitive data vulnerable.

## Expected Output

The resulting dataset from auditing queries in `SNOWFLAKE.ACCOUNT_USAGE` provides a comprehensive record of all grants and revokes. Each row typically represents a specific privilege grant or role assignment.

| GRANTED_ON | GRANTED_TO | GRANTEE_NAME | PRIVILEGE | NAME | GRANT_OPTION | CREATED_ON | GRANTED_BY |
| :--------- | :--------- | :----------- | :-------- | :--- | :----------- | :--------- | :--------- |
| WAREHOUSE | ROLE | DATA_ANALYST | USAGE | COMPUTE_WH | FALSE | 2026-03-14 10:00:00.000 | SECURITYADMIN |
| DATABASE | ROLE | DATA_ANALYST | USAGE | PROD_DB | FALSE | 2026-03-14 10:01:00.000 | SECURITYADMIN |

| TABLE | ROLE | DATA_ANALYST | SELECT | TABLE2 | FALSE | 2026-03-14 10:04:00.000 | SECURITYADMIN |

**Explanation of Columns:**

- **`GRANTED_ON`**: The type of object the privilege was granted on (e.g., WAREHOUSE, DATABASE, SCHEMA, TABLE).
- **`GRANTED_TO`**: Indicates whether the privilege was granted to a `ROLE` or `USER`.
- **`GRANTEE_NAME`**: The name of the role or user that received the privilege.
- **`PRIVILEGE`**: The specific permission granted (e.g., `USAGE`, `SELECT`, `ALL PRIVILEGES`).
- **`NAME`**: The identifier of the object on which the privilege was granted (e.g., `COMPUTE_WH`, `PROD_DB`, `SALES`, `TABLE1`).
- **`GRANT_OPTION`**: A boolean indicating if the grantee can further grant this privilege to others.
- **`CREATED_ON`**: The timestamp when the grant was created.
- **`GRANTED_BY`**: The role that performed the grant operation.

This output provides a clear, auditable record of access permissions within Snowflake, crucial for security and compliance.
