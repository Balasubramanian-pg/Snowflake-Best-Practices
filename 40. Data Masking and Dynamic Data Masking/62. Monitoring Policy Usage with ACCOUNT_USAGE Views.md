# Monitoring Policy Usage with ACCOUNT_USAGE Views in Snowflake

This concept focuses on utilizing Snowflake's `ACCOUNT_USAGE` schema to track and understand how data governance policies (like masking policies, row access policies, and tag-based masking) are being applied and used across your Snowflake account. It exists to provide administrators with visibility into policy effectiveness, compliance auditing, and resource consumption related to policy enforcement.

**Short Explanation**

Monitoring policy usage with `ACCOUNT_USAGE` views involves querying specific views that record metadata and activity related to data governance policies. These views help you see which policies are active, on which tables/columns they are applied, and how often they are being evaluated during query execution. This is important for ensuring data security compliance, optimizing policy performance, and understanding the impact of policies on your data ecosystem. It's commonly used in security audits, cost management for policy evaluation, and policy lifecycle management workflows.

- What problem does this SQL feature solve? It solves the problem of lacking visibility into the application and usage patterns of data governance policies across a Snowflake account.
- Why is it important in databases or analytics? It's crucial for compliance, security auditing, performance tuning of policies, and understanding the operational impact of data governance.
- Where is it commonly used in real workflows? It's used by security teams for auditing, data governance teams for policy management, and operations teams for performance monitoring and cost analysis.

## Implementation Example

```sql
SELECT
    policy_name,
    policy_kind,
    reference_database_name,
    reference_schema_name,
    reference_entity_name,
    reference_entity_domain,
    created,
    deleted
FROM
    snowflake.account_usage.policy_references
WHERE
    deleted IS NULL
ORDER BY
    created DESC;
```

## Explanation of the Code

This query retrieves information about all active policy references in the Snowflake account.

- `SELECT policy_name, policy_kind, reference_database_name, reference_schema_name, reference_entity_name, reference_entity_domain, created, deleted`: This clause specifies the columns to be retrieved.
    - `policy_name`: The name of the data governance policy (e.g., masking policy).
    - `policy_kind`: The type of policy (e.g., 'MASKING_POLICY', 'ROW_ACCESS_POLICY').
    - `reference_database_name`: The database where the policy is applied.
    - `reference_schema_name`: The schema where the policy is applied.
    - `reference_entity_name`: The table or column name to which the policy is attached.
    - `reference_entity_domain`: Specifies if the policy is applied to a table or a column.
    - `created`: Timestamp when the policy reference was created.
    - `deleted`: Timestamp when the policy reference was deleted (NULL if still active).
    - What it does: It selects specific metadata attributes about the policies and their application.
    - How it changes the result set: It filters down the available columns from the `POLICY_REFERENCES` view to only those explicitly listed, shaping the output data.

- `FROM snowflake.account_usage.policy_references`: This clause specifies the source of the data.
    - `snowflake.account_usage.policy_references`: This is a view within the `ACCOUNT_USAGE` schema that contains records of all policy attachments to tables or columns.
    - What it does: It identifies the `POLICY_REFERENCES` view as the data source.
    - How it changes the result set: It provides the raw dataset from which all subsequent filtering and ordering operations will be performed.

- `WHERE deleted IS NULL`: This clause filters the rows.
    - `deleted IS NULL`: This condition ensures that only currently active policy references are returned, excluding those that have been deleted.
    - What it does: It applies a filter to include only records where the `deleted` column is `NULL`.
    - How it changes the result set: It removes all rows corresponding to policies that are no longer active, narrowing down the results to only current policy usage.

- `ORDER BY created DESC`: This clause sorts the result set.
    - `created DESC`: This orders the results by the `created` timestamp in descending order, showing the most recently created policy references first.
    - What it does: It arranges the returned rows based on the creation date of the policy reference.
    - How it changes the result set: It sorts the final filtered rows, making it easier to review the latest policy applications.

## When to Use

1.  **Auditing and Compliance:** When you need to demonstrate to auditors which data governance policies are active, where they are applied, and who created them, ensuring regulatory compliance.
    *   *Example Scenario:* A financial institution needs to prove that sensitive customer data in specific columns is consistently masked across all production tables.
2.  **Policy Lifecycle Management:** To get an overview of all currently active policies and their target entities before making changes, retiring old policies, or implementing new ones.
    *   *Example Scenario:* A data steward wants to identify all tables affected by a particular masking policy before modifying its definition to avoid unintended consequences.
3.  **Performance Monitoring and Troubleshooting:** To identify if a large number of policies are being applied to a single table or column, which could impact query performance, or to verify if a policy was correctly applied to a new table.
    *   *Example Scenario:* A user reports slow query performance on a table. An administrator can check `POLICY_REFERENCES` to see if an excessive number of policies are active on that table, potentially contributing to the slowdown.

## When Not to Use

1.  **Real-time Policy Evaluation Details:** `ACCOUNT_USAGE` views are not for real-time monitoring of policy evaluation during active query execution. There's a latency (typically a few hours) before data appears in `ACCOUNT_USAGE`.
    *   *Inefficient because:* It won't give immediate feedback on the impact of a query or the number of rows processed by a policy *right now*.
2.  **Detailed Query Performance Metrics per Policy:** While you can infer potential performance impacts, `ACCOUNT_USAGE` doesn't provide granular metrics like CPU time or I/O spent *specifically* on each policy evaluation within a single query.
    *   *Inefficient because:* For deep performance analysis of individual queries and their policy overhead, you'd need to look at query profile details or potentially other monitoring tools.
3.  **Modifying Policy Definitions:** `ACCOUNT_USAGE` views are read-only. You cannot use them to alter, create, or drop policies; they are purely for informational and monitoring purposes.
    *   *Inefficient because:* Attempting to use these views for DDL operations is impossible and will result in errors.

## Common Mistakes

1.  **Expecting Real-time Data:** A common mistake is assuming that changes made to policies or their application will immediately reflect in `ACCOUNT_USAGE` views. There's usually a delay of a few hours.
2.  **Not Joining with Other ACCOUNT_USAGE Views:** Relying solely on `POLICY_REFERENCES` might not give a complete picture. For example, to see policy usage over time (evaluations per query), you might need to join with `QUERY_HISTORY` and parse query text.
3.  **Ignoring the `deleted` Column:** Forgetting to filter out deleted policy references when looking for currently active policies can lead to reviewing outdated or irrelevant information. Always filter `WHERE deleted IS NULL` for active policies.

## Expected Output

The resulting dataset will show a list of all currently active data governance policy applications within your Snowflake account. Each row represents a single instance where a policy is attached to a database, schema, table, or column.

| POLICY_NAME        | POLICY_KIND        | REFERENCE_DATABASE_NAME | REFERENCE_SCHEMA_NAME | REFERENCE_ENTITY_NAME | REFERENCE_ENTITY_DOMAIN | CREATED                     | DELETED |
| :----------------- | :----------------- | :---------------------- | :-------------------- | :-------------------- | :---------------------- | :-------------------------- | :------ |
| email_mask_policy  | MASKING_POLICY     | PROD_DB                 | CUSTOMER_DATA         | EMAIL                 | COLUMN                  | 2024-01-15 10:30:00.000 +0000 | NULL    |
| hr_access_policy   | ROW_ACCESS_POLICY  | HR_DB                   | EMPLOYEE_INFO         | EMPLOYEES             | TABLE                   | 2024-02-01 09:00:00.000 +0000 | NULL    |
| ssn_redact_policy  | MASKING_POLICY     | PROD_DB                 | CUSTOMER_DATA         | SSN                   | COLUMN                  | 2024-03-10 14:15:00.000 +0000 | NULL    |

- `POLICY_NAME`: The name of the policy, helping you identify which specific rule is in effect.
- `POLICY_KIND`: Tells you if it's a masking policy, row access policy, etc.
- `REFERENCE_DATABASE_NAME`, `REFERENCE_SCHEMA_NAME`, `REFERENCE_ENTITY_NAME`, `REFERENCE_ENTITY_DOMAIN`: These columns pinpoint exactly where the policy is applied (e.g., the `email_mask_policy` is on the `EMAIL` column in the `CUSTOMER_DATA` schema of `PROD_DB`).
- `CREATED`: Shows when the policy was applied, useful for auditing timeline.
- `DELETED`: This will be `NULL` for all rows in this specific query's output, indicating the policy reference is still active.
