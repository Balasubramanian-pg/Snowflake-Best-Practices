# User Account Expiry Enforcement in Snowflake

User account expiry enforcement in Snowflake refers to the mechanism by which user accounts are automatically disabled or locked after a specified period of inactivity or a set expiration date. This feature exists to enhance security by preventing the prolonged use of dormant accounts, which could pose a risk if compromised.

**Short Explanation**

This SQL feature addresses the security problem of stale user accounts accumulating in a system. It's important in databases and analytics to maintain a secure environment by automatically revoking access for users who no longer need it or haven't logged in for a long time. It's commonly used in compliance-driven environments, like financial or healthcare industries, and for managing temporary access for contractors or seasonal employees.

- What problem does this SQL feature solve? It solves the problem of security vulnerabilities arising from dormant or abandoned user accounts that could be exploited.
- Why is it important in databases or analytics? It's crucial for maintaining data security, adhering to compliance regulations (like GDPR, HIPAA), and ensuring good security hygiene by reducing the attack surface.
- Where is it commonly used in real workflows? In organizations with strict security policies, managing large user bases, or when provisioning temporary access for external partners or specific projects.

## Implementation Example

```sql
-- Create a new user with a password expiry set to 90 days from creation
CREATE USER temporary_analyst
PASSWORD = 'StrongPassword123!'
LOGIN_NAME = 'temp_analyst'
COMMENT = 'Temporary account for a contractor'
PASSWORD_EXPIRE_AFTER_DAYS = 90;

-- Alter an existing user to enforce password expiry after 60 days
ALTER USER existing_user
SET PASSWORD_EXPIRE_AFTER_DAYS = 60;

-- View user information to check password expiry settings
DESCRIBE USER temporary_analyst;

-- Example of setting a global parameter for password expiry for all new users (applies to new users only, existing users need explicit alteration)
ALTER ACCOUNT SET PASSWORD_EXPIRE_AFTER_DAYS = 30;
```

## Explanation of the Code

- `CREATE USER temporary_analyst ... PASSWORD_EXPIRE_AFTER_DAYS = 90;`:
  - What it does: This statement creates a new Snowflake user named `temporary_analyst`.
  - How it changes the result set: It adds a new row to Snowflake's internal user metadata. The `PASSWORD_EXPIRE_AFTER_DAYS = 90` clause sets a policy for this specific user, meaning their password will expire 90 days after its creation (or last reset), requiring them to set a new one. This ensures access isn't indefinite.

- `ALTER USER existing_user SET PASSWORD_EXPIRE_AFTER_DAYS = 60;`:
  - What it does: This statement modifies an existing user named `existing_user`.
  - How it changes the result set: It updates the metadata for `existing_user`, setting their password expiry policy to 60 days. This means their current password will expire 60 days from the last time it was set, forcing a password change.

- `DESCRIBE USER temporary_analyst;`:
  - What it does: This command retrieves and displays detailed information about the specified user, `temporary_analyst`.
  - How it changes the result set: It returns a temporary dataset showing various properties of the user, including the `PASSWORD_EXPIRE_AFTER_DAYS` setting, allowing an administrator to verify the policy.

- `ALTER ACCOUNT SET PASSWORD_EXPIRE_AFTER_DAYS = 30;`:
  - What it does: This statement sets a default password expiry policy at the account level.
  - How it changes the result set: It modifies an account-level parameter. From this point forward, any *new* users created in this Snowflake account will inherit a `PASSWORD_EXPIRE_AFTER_DAYS` setting of 30, unless explicitly overridden during user creation. It does not affect existing users.

## When to Use

1.  **Compliance Requirements:** When regulations (e.g., PCI DSS, HIPAA) mandate periodic password changes or user access reviews.
    *   *Scenario:* A financial institution needs to ensure all user passwords are reset every 90 days to comply with industry standards.
2.  **Temporary Access Management:** For contractors, consultants, or temporary project members whose access should automatically cease or be reviewed after a specific project duration.
    *   *Scenario:* A contractor is granted access for a 6-month project, and their account is set to expire after 180 days, ensuring their access is not left open indefinitely.
3.  **Enhanced Security Posture:** To reduce the risk associated with dormant accounts that might be forgotten but still hold privileges, preventing potential compromise.
    *   *Scenario:* An organization wants to implement a general policy that if an employee leaves or changes roles, their old account access becomes invalid if not actively managed, or if a password hasn't been changed for a year.

## When Not to Use

1.  **Service Accounts/Automated Processes:** For system-to-system integrations or automated jobs where passwords are not interactively managed by a human and expiring them would break critical processes.
    *   *Inefficiency:* An ETL process running daily using a service account would repeatedly fail if its password expires, requiring manual intervention to reset and update connection strings.
2.  **Federated Identity or SSO:** When Snowflake users authenticate exclusively through an external Identity Provider (IdP) using Federated Identity Management (e.g., Okta, Azure AD), as password management is handled by the IdP, not Snowflake.
    *   *Inefficiency:* Enforcing password expiry within Snowflake when SSO is in use is redundant and can cause confusion or conflicts, as the actual authentication happens outside Snowflake.
3.  **Accounts with Very Infrequent but Necessary Access:** For administrators or auditors who rarely log in but need access only in emergencies or specific annual reviews, frequent password expiry could be an unnecessary burden.
    *   *Inefficiency:* An auditor who logs in once a year for compliance checks would find their password expired every few months, leading to constant password resets for very low-frequency usage.

## Common Mistakes

1.  **Applying to Service Accounts:** Forgetting to exempt service accounts from expiry policies, leading to broken automated workflows.
2.  **Not Communicating Expiry:** Failing to inform users about the password expiry policy, causing frustration and helpdesk calls when they suddenly can't log in.
3.  **Confusing Account Expiry with Password Expiry:** While related, `PASSWORD_EXPIRE_AFTER_DAYS` specifically refers to password age. There isn't a direct "account expiry" feature in Snowflake that disables an account after a set number of days *from creation* without regard to password age; inactive accounts are managed differently (e.g., through network policies or roles being revoked).
4.  **Setting Global Policy without Review:** Applying a `PASSWORD_EXPIRE_AFTER_DAYS` at the account level without considering its impact on all user types, including exceptions.

## Expected Output

When you run `DESCRIBE USER temporary_analyst;`, the resulting dataset will show various properties of the user. The relevant columns for this concept would be something like this:

| property               | value               |
| :--------------------- | :------------------ |
| NAME                   | TEMPORARY_ANALYST   |
| LOGIN_NAME             | temp_analyst        |
| DEFAULT_WAREHOUSE      | (null)              |
| DEFAULT_NAMESPACE      | (null)              |
| DEFAULT_ROLE           | (null)              |
| HAS_PASSWORD           | true                |
| DISABLED               | false               |
| LAST_SUCCESS_LOGIN     | (null)              |
| LAST_FAIL_LOGIN        | (null)              |
| EXPIRES_AT             | (null)              |
| PASSWORD_EXPIRE_AFTER_DAYS | 90                  |
| PASSWORD_LAST_SET_TIME | 2026-03-14 10:00:00.000 +0000 |
| MINS_TO_UNLOCK         | (null)              |

**Explanation of Columns:**

-   **PASSWORD_EXPIRE_AFTER_DAYS**: This column will show `90`, indicating that this user's password must be changed every 90 days.
-   **PASSWORD_LAST_SET_TIME**: This column would display the timestamp when the password was last set. The expiry date is calculated based on this timestamp plus the `PASSWORD_EXPIRE_AFTER_DAYS` value.
-   **EXPIRES_AT**: This column shows when the *user's access token* might expire, not the password itself, and is often null for standard users. The `PASSWORD_EXPIRE_AFTER_DAYS` effectively acts as an expiry for the password's validity.
