# Compliance Requirements For User Management in Snowflake

Managing users in Snowflake with compliance in mind means enforcing controls like least-privilege access, auditing, and data protection. It exists because regulated industries and enterprise teams need to prove who can see or change data, when, and under what policy.

**Short Explanation**

Write 2-3 sentences explaining the idea in simple terms.

- What problem does this SQL feature solve? It solves uncontrolled access and missing audit trails by requiring roles, privileges, and user policies that align with compliance standards (e.g., GDPR, HIPAA, SOX).  
- Why is it important in databases or analytics? Because analytics platforms hold sensitive business and customer data, and regulators demand accountability for access and changes.  
- Where is it commonly used in real workflows? In onboarding/offboarding employees, periodic access reviews, and implementing separation of duties for finance or healthcare data.

## Implementation Example

```sql
-- 1. Create a compliance-oriented role with least privileges
CREATE ROLE IF NOT EXISTS analyst_eu_readonly;

-- 2. Grant read-only privilege on a specific database/schema
GRANT USAGE ON DATABASE sales_db TO ROLE analyst_eu_readonly;
GRANT USAGE ON SCHEMA sales_db.eu_data TO ROLE analyst_eu_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.eu_data TO ROLE analyst_eu_readonly;

-- 3. Create a user and assign the role
CREATE USER IF NOT EXISTS alice_smith
  PASSWORD = 'TempPass!2026'
  DEFAULT_ROLE = analyst_eu_readonly
  MUST_CHANGE_PASSWORD = TRUE
  COMMENT = 'EU analyst - GDPR review required';

GRANT ROLE analyst_eu_readonly TO USER alice_smith;

-- 4. Enable auditing: show current grants for review
SHOW GRANTS TO ROLE analyst_eu_readonly;
```

## Explanation of the Code

- CREATE ROLE: defines a role that encapsulates compliance policy (least-privilege). It creates a container for grants without giving direct user privileges.  
- GRANT USAGE: allows the role to access the database and schema; restricts scope so the result set of accessible objects is limited.  
- GRANT SELECT: gives read-only access to tables; changes the result set of queries the user can run to include only those columns/rows returned by SELECT.  
- CREATE USER: creates a login with MUST_CHANGE_PASSWORD to enforce credential hygiene; DEFAULT_ROLE sets the starting privileges for the session, reducing accidental access.  
- GRANT ROLE TO USER: assigns the policy-bearing role to the user, so the user inherits only the privileges defined.  
- SHOW GRANTS: outputs grants metadata; used for access reviews and does not affect query results.

## When to Use

- Onboarding contractors who need read-only access to a regulated schema for a short project.  
- Enforcing separation of duties: finance analysts get SELECT, finance managers get INSERT/UPDATE via separate roles.  
- Periodic compliance audits where you extract SHOW GRANTS output to prove least-privilege design.

## When Not to Use

- Giving direct privileges to users instead of through roles (breaks auditing and complicates reviews).  
- Using ACCOUNTADMIN or SECURITYADMIN for everyday queries (violates least-privilege).  
- Assigning overly broad warehouse or database privileges when schema-level grants suffice (increases breach impact).

## Common Mistakes

- Creating users without MUST_CHANGE_PASSWORD or MFA policy.  
- Forgetting to GRANT SELECT on FUTURE TABLES IN SCHEMA, so new tables are invisible or access drifts.  
- Assigning multiple powerful roles to one user, making it impossible to show a clean audit trail.

## Expected Output

The resulting dataset from the final SHOW GRANTS lists role grants and privileges. Example:

| created_on | privilege | granted_on | name | granted_to | grantee_name |
|------------|-----------|------------|------|------------|--------------|
| 2026-03-14 | USAGE | DATABASE | SALES_DB | ROLE | ANALYST_EU_READONLY |
| 2026-03-14 | SELECT | TABLE | ORDERS | ROLE | ANALYST_EU_READONLY |
| 2026-03-14 | SELECT | TABLE | CUSTOMERS | ROLE | ANALYST_EU_READONLY |

Columns show when the privilege was created, which privilege it is, the object type/name, and the role/user it’s granted to. This output is used for evidence in compliance reviews.
