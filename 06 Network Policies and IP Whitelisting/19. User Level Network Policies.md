# User Level Network Policies in Snowflake

User Level Network Policies in Snowflake allow administrators to control network access for individual users, restricting which IP addresses or ranges can connect to Snowflake. This feature enhances security by ensuring that users can only access Snowflake from approved locations, preventing unauthorized access from untrusted networks.

**Short Explanation**

User Level Network Policies let you specify allowed and blocked IP addresses for a single Snowflake user. This helps secure your data by enforcing access restrictions on a per-user basis, ensuring that even if credentials are compromised, access is limited to authorized network origins. It's important for organizations with strict security requirements or users who access Snowflake from varying, but controlled, environments.

- **What problem does this SQL feature solve?** It solves the problem of unauthorized network access to Snowflake for specific users, adding an extra layer of security beyond just authentication.
- **Why is it important in databases or analytics?** It's crucial for compliance, data governance, and protecting sensitive data by limiting the "attack surface" for individual user accounts.
- **Where is it commonly used in real workflows?** It's often used for privileged users (like database administrators), external contractors, or users accessing Snowflake from specific corporate VPNs, ensuring they only connect from secure, pre-approved locations.

## Implementation Example

```sql
-- Create a Network Policy named 'analyst_restricted_policy'
CREATE NETWORK POLICY analyst_restricted_policy
  ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.5')
  BLOCKED_IP_LIST = ('172.16.0.0/16');

-- Apply the Network Policy to a specific user
ALTER USER data_analyst_user
  SET NETWORK_POLICY = analyst_restricted_policy;

-- Unset the Network Policy from the user (if needed)
ALTER USER data_analyst_user
  UNSET NETWORK_POLICY;
```

## Explanation of the Code

This code snippet demonstrates how to create a network policy and then apply it to a specific user in Snowflake.

- **`CREATE NETWORK POLICY analyst_restricted_policy ...`**: This statement creates a new network policy named `analyst_restricted_policy`.
  - **What it does**: Defines a set of rules for network access.
  - **How it changes the result set**: This is a DDL (Data Definition Language) statement; it does not return a result set but creates or modifies a database object.
- **`ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.5')`**: This clause within the `CREATE NETWORK POLICY` statement specifies the IP addresses or CIDR ranges from which connections are permitted.
  - **What it does**: Allows connections only from the listed IP addresses or ranges.
  - **How it changes the result set**: Not applicable as it's part of a DDL statement.
- **`BLOCKED_IP_LIST = ('172.16.0.0/16')`**: This clause within the `CREATE NETWORK POLICY` statement specifies the IP addresses or CIDR ranges from which connections are explicitly denied.
  - **What it does**: Denies connections from the listed IP addresses or ranges, even if they might otherwise be covered by an allowed list (blocked list takes precedence).
  - **How it changes the result set**: Not applicable.
- **`ALTER USER data_analyst_user SET NETWORK_POLICY = analyst_restricted_policy;`**: This statement applies the previously defined `analyst_restricted_policy` to the `data_analyst_user`.
  - **What it does**: Links the network policy to a specific user, enforcing its rules for that user.
  - **How it changes the result set**: Not applicable; it modifies user properties.
- **`ALTER USER data_analyst_user UNSET NETWORK_POLICY;`**: This statement removes the network policy from the `data_analyst_user`.
  - **What it does**: Disassociates the network policy from the user, reverting to a higher-level network policy (account-level) or unrestricted access if no other policies apply.
  - **How it changes the result set**: Not applicable.

## When to Use

1.  **Restricting privileged users:** For roles like database administrators or security officers who handle sensitive data, ensure they can only connect from highly secure, monitored corporate networks or VPNs.
    *   *Example:* An admin user `db_admin` is set to a policy allowing access only from the company's datacenter IP range.
2.  **External consultants or contractors:** When granting temporary access to external parties, restrict their connectivity to specific, known IP addresses they will use, enhancing control over their access points.
    *   *Example:* A contractor `external_dev` has a policy limiting their access to the IP address of their client site or approved VPN.
3.  **Specific application service accounts:** For service accounts that should only connect from particular application servers, assign a user-level policy to lock down their access to those server IPs.
    *   *Example:* A data ingestion service user `etl_svc` is assigned a policy that only allows connections from the IP addresses of the ETL servers.

## When Not to Use

1.  **Widespread access for many users:** For a large number of users who require flexible access from various locations (e.g., remote workers with dynamic IPs, public Wi-Fi), setting individual user-level policies can become an unmanageable administrative burden. Account-level policies are generally more appropriate here.
2.  **Highly dynamic IP environments:** If users frequently change networks and have non-static public IP addresses, maintaining and updating their individual network policies would be impractical and lead to constant access issues.
3.  **As the sole security measure:** Network policies are a layer of defense, not a replacement for strong authentication (MFA), role-based access control (RBAC), and regular security audits. Relying solely on network policies without these other measures is insufficient.

## Common Mistakes

1.  **Confusing user-level with account-level policies:** Forgetting that user-level policies override account-level policies for that specific user, potentially leading to unintended access restrictions or allowances.
2.  **Incorrect IP address/CIDR notation:** Typos or misunderstandings in IP address ranges (e.g., using `/24` instead of `/32` for a single IP) can either block legitimate users or inadvertently open access.
3.  **Over-blocking legitimate access:** Being too restrictive without fully understanding the user's connection patterns, leading to users being unable to connect and requiring frequent policy adjustments.
4.  **Not testing policies after implementation:** Failing to verify that the policy works as intended by attempting connections from both allowed and blocked IP addresses.

## Expected Output

Applying a user-level network policy does not produce a direct SQL output or dataset. Its effect is on the network connectivity to Snowflake. When a user attempts to connect to Snowflake, the system checks if their connecting IP address is permitted by their assigned network policy.

If the `data_analyst_user` attempts to connect from an IP address within `192.168.1.0/24` or exactly `10.0.0.5`, the connection will succeed. If they try to connect from an IP address within `172.16.0.0/16`, the connection will be explicitly denied. For any other IP address not specified in either list, the connection would also be denied because `ALLOWED_IP_LIST` implicitly denies anything not listed.

This is a conceptual output, representing the outcome of a connection attempt:

| Connection Attempt Source IP | Network Policy Status | Result          |
| :-------------------------- | :-------------------- | :-------------- |
| `192.168.1.10`              | `analyst_restricted_policy` | Connection Success |
| `10.0.0.5`                  | `analyst_restricted_policy` | Connection Success |
| `172.16.0.20`               | `analyst_restricted_policy` | Connection Denied  |
| `198.51.100.1`              | `analyst_restricted_policy` | Connection Denied  |

**Explanation of Columns:**

-   **Connection Attempt Source IP:** The IP address from which the user is trying to connect to Snowflake.
-   **Network Policy Status:** Indicates which network policy is active for the user at the time of connection.
-   **Result:** Whether the connection attempt was successful or denied based on the rules defined in the network policy.
