# Row Access Policies

Row Access Policies in Snowflake provide a robust, attribute-based access control mechanism to dynamically filter rows returned by a query. They ensure that users only see the data they are authorized to access, based on specific criteria defined in the policy. This is crucial for data privacy, compliance, and multi-tenant data segregation.

**Short Explanation**

Row Access Policies allow you to define rules that restrict which rows a user can see in a table or view. When a query is executed against a protected table, the policy evaluates the user's role or attributes and filters the data in real-time, showing only the permitted rows.

- **What problem does this SQL feature solve?** It solves the problem of granular row-level security, ensuring that sensitive data is only accessible to authorized individuals, thereby helping with compliance (like GDPR, HIPAA) and data segregation in shared environments.
- **Why is it important in databases or analytics?** It's important for maintaining data privacy, preventing unauthorized data exposure, simplifying access control management for large datasets, and enabling secure multi-tenant applications.
- **Where is it commonly used in real workflows?** Commonly used in financial institutions for customer data, healthcare for patient records, and multi-tenant SaaS applications where each tenant should only see their own data.

## Implementation Example

```sql
-- 1. Create a table with sensitive data
CREATE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    sale_amount DECIMAL(10, 2),
    region VARCHAR,
    sales_rep_id INT
);

-- Insert some sample data
INSERT INTO sales_data VALUES
    (1, 'Laptop', 1200.00, 'EAST', 101),
    (2, 'Mouse', 25.00, 'WEST', 102),
    (3, 'Keyboard', 75.00, 'EAST', 101),
    (4, 'Monitor', 300.00, 'SOUTH', 103),
    (5, 'Webcam', 50.00, 'WEST', 102),
    (6, 'Headphones', 150.00, 'EAST', 104);

-- 2. Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_column VARCHAR) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE
        WHEN CURRENT_ROLE() = 'EAST_SALES' AND region_column = 'EAST' THEN TRUE
        WHEN CURRENT_ROLE() = 'WEST_SALES' AND region_column = 'WEST' THEN TRUE
        ELSE FALSE
    END;

-- 3. Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);

-- 4. Create roles and grant privileges (for demonstration)
CREATE ROLE ADMIN_ROLE;
CREATE ROLE EAST_SALES;
CREATE ROLE WEST_SALES;

GRANT SELECT ON TABLE sales_data TO ROLE ADMIN_ROLE;
GRANT SELECT ON TABLE sales_data TO ROLE EAST_SALES;
GRANT SELECT ON TABLE sales_data TO ROLE WEST_SALES;

-- Grant necessary privileges to the policy owner (if different)
GRANT USAGE ON WAREHOUSE your_warehouse TO ADMIN_ROLE;
GRANT USAGE ON DATABASE your_database TO ADMIN_ROLE;
GRANT USAGE ON SCHEMA your_schema TO ADMIN_ROLE;

-- Assign users to roles (example: user 'JOHN' gets 'EAST_SALES' role)
-- GRANT ROLE EAST_SALES TO USER JOHN;

-- 5. Test the policy (run as different roles)
-- USE ROLE ADMIN_ROLE;
-- SELECT * FROM sales_data;

-- USE ROLE EAST_SALES;
-- SELECT * FROM sales_data;

-- USE ROLE WEST_SALES;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

The example demonstrates the creation and application of a Row Access Policy in Snowflake.

- **`CREATE TABLE sales_data (...)`**: This statement defines a sample table named `sales_data` to store sales records, including a `region` column that will be used for access control.
- **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with sample data, representing sales from different regions.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_column VARCHAR) RETURNS BOOLEAN -> ...`**:
    - This is the core of the policy definition.
    - `CREATE OR REPLACE ROW ACCESS POLICY sales_policy`: Creates a new row access policy named `sales_policy`.
    - `AS (region_column VARCHAR)`: Defines the input parameter for the policy, which is expected to be the `region` column from the table it will be applied to.
    - `RETURNS BOOLEAN -> ...`: Indicates that the policy will return a boolean value (`TRUE` to include the row, `FALSE` to exclude it).
    - `CASE WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE ... ELSE FALSE END;`: This is the policy logic. It checks the `CURRENT_ROLE()` of the user executing the query.
        - If the user has `ADMIN_ROLE`, they see all rows (`TRUE`).
        - If the user has `EAST_SALES` role and the `region_column` for that row is `'EAST'`, they see the row (`TRUE`).
        - Similarly for `WEST_SALES` and `'WEST'`.
        - All other combinations result in `FALSE`, meaning the row is filtered out.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);`**:
    - This statement applies the `sales_policy` to the `sales_data` table.
    - `ON (region)` specifies that the `region` column of `sales_data` should be passed as the `region_column` argument to the `sales_policy`.
- **`CREATE ROLE ...`, `GRANT SELECT ...`, `GRANT USAGE ...`**: These commands set up roles and grant the necessary `SELECT` privileges on the `sales_data` table to these roles, allowing different users to test the policy.
- **`USE ROLE ...; SELECT * FROM sales_data;`**: These commented lines illustrate how different roles would query the `sales_data` table and observe dynamic filtering based on the applied policy.

## When to Use

1.  **Multi-tenant applications**: When a single table stores data for multiple customers or tenants, and each tenant should only see their own data.
    *   *Example:* A SaaS platform storing customer order data in a `CUSTOMER_ORDERS` table. A row access policy can ensure that `CUSTOMER_A` only sees orders belonging to `CUSTOMER_A`.
2.  **Compliance and data privacy**: To meet regulatory requirements (e.g., HIPAA, GDPR) by restricting access to sensitive information based on user attributes or roles.
    *   *Example:* In a `PATIENT_RECORDS` table, doctors might only see records for their assigned patients, while administrative staff might see only non-medical demographic data for all patients.
3.  **Role-based data segregation**: When different departments or roles within an organization need access to specific subsets of data in a larger dataset.
    *   *Example:* A `COMPANY_FINANCE` table where users with the `ACCOUNTING` role can see all transactions, but users with the `SALES_REPORTING` role can only see sales-related transactions from their specific region.

## When Not to Use

1.  **Simple column-level security**: If the requirement is only to restrict access to specific columns, not rows, column-level security (e.g., using `MASKING POLICY` or simple `VIEW`s) might be more appropriate and less complex.
    *   *Inefficiency:* Applying a row access policy for column-level restrictions adds unnecessary overhead.
2.  **Performance-critical queries on very large tables with complex policies**: While Snowflake optimizes policy evaluation, overly complex policies on extremely large tables might introduce some query latency. Consider materialized views or pre-filtered datasets if performance is paramount and access patterns are static.
    *   *Inefficiency:* The policy is evaluated for *every* query, potentially impacting scan and join performance if the policy logic is resource-intensive.
3.  **When data access is static and can be handled by simple views**: If the access control logic is very straightforward (e.g., `WHERE department = 'IT'`) and doesn't depend on the `CURRENT_ROLE()` or other dynamic session context, a simple view might be easier to manage.
    *   *Complexity:* Introducing a policy for static filtering adds an extra layer of abstraction that might not be necessary.

## Common Mistakes

1.  **Not granting `SELECT` on the policy to the roles**: Users will receive an error indicating insufficient privileges if the roles trying to query the protected table do not have `SELECT` privilege granted on the policy itself (if it's not owned by `ACCOUNTADMIN`).
2.  **Forgetting to apply the policy to the table**: After creating the policy, it must be explicitly linked to a table or view using `ALTER TABLE ... ADD ROW ACCESS POLICY ...`.
3.  **Incorrectly referencing columns in the policy or `ON` clause**: The column passed to the policy in the `ON` clause must match the parameter name and data type in the `CREATE ROW ACCESS POLICY` statement.
4.  **Over-complicating policy logic**: Keeping the policy logic as simple as possible is crucial for readability, maintainability, and performance. Complex nested `CASE` statements or subqueries within the policy can be hard to debug and may affect query performance.

## Expected Output

The resulting dataset from querying a table protected by a row access policy will be a filtered version of the original table. Only rows that satisfy the policy's conditions for the current user's context (e.g., `CURRENT_ROLE()`) will be returned.

**Example Table (if queried by a user with `EAST_SALES` role):**

| sale_id | product_name | sale_amount | region | sales_rep_id |
| :------ | :----------- | :---------- | :----- | :----------- |
| 1       | Laptop       | 1200.00     | EAST   | 101          |
| 3       | Keyboard     | 75.00       | EAST   | 101          |
| 6       | Headphones   | 150.00      | EAST   | 104          |

-   **sale_id**: Unique identifier for the sale.
-   **product_name**: Name of the product sold.
-   **sale_amount**: The monetary value of the sale.
-   **region**: The geographical region where the sale occurred.
-   **sales_rep_id**: The ID of the sales representative.

The example above shows only rows where the `region` is 'EAST', as expected for a user assigned to the `EAST_SALES` role. Rows from 'WEST' or 'SOUTH' regions are automatically filtered out and not visible to this user.
