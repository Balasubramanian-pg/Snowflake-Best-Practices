# Network Policy Architecture Design in Snowflake

Network Policies in Snowflake define allowed and blocked network locations that can connect to Snowflake. They enhance security by restricting access to your Snowflake account based on IP addresses, ensuring that only authorized networks can initiate connections. This mechanism is crucial for maintaining data governance and compliance, especially in regulated environments.

**Short Explanation**

Network Policies act like a firewall for your Snowflake account, allowing you to specify which IP addresses or ranges are permitted to connect. This feature solves the problem of unauthorized network access, making your data more secure by limiting connection points. It's important in databases for preventing breaches and is commonly used in enterprise security architectures to enforce strict access controls.

- What problem does this SQL feature solve? It solves the problem of unauthorized network access to a Snowflake account.
- Why is it important in databases or analytics? It's important for enhancing security, meeting compliance requirements, and preventing data exfiltration by restricting access to known, trusted networks.
- Where is it commonly used in real workflows? It's used in organizations with strict security policies, cloud environments requiring VNet/VPC integration, and when connecting BI tools or ETL processes from specific, secure locations.

## Implementation Example

```sql
-- Create a network policy allowing specific IP addresses
CREATE NETWORK POLICY my_corporate_network_policy
    ALLOWED_IP_LIST = ('203.0.113.1', '203.0.113.2', '192.0.2.0/24')
    COMMENT = 'Policy to allow connections only from corporate VPN and office IPs';

-- Apply the network policy to an account
ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;

-- (Optional) Disable the policy for a specific user (overrides account policy)
ALTER USER JOHN_DOE SET NETWORK_POLICY = NULL;

-- (Optional) Apply a different policy to a specific user
CREATE NETWORK POLICY analyst_network_policy
    ALLOWED_IP_LIST = ('198.51.100.10')
    COMMENT = 'Policy for specific analyst VPN';
ALTER USER JANE_SMITH SET NETWORK_POLICY = analyst_network_policy;
```

## Explanation of the Code

This example demonstrates how to create, apply, and manage Network Policies in Snowflake.

- `CREATE NETWORK POLICY my_corporate_network_policy`: This statement initializes a new network policy named `my_corporate_network_policy`.
    - What it does: Creates a new security object that defines network access rules.
    - How it changes the result set: It doesn't directly change a result set but creates a new object in the Snowflake environment.

- `ALLOWED_IP_LIST = ('203.0.113.1', '203.0.113.2', '192.0.2.0/24')`: This clause specifies the list of IP addresses or CIDR blocks that are permitted to connect.
    - What it does: Defines the allowed inbound IP addresses.
    - How it changes the result set: N/A, part of policy definition.

- `COMMENT = 'Policy to allow connections only from corporate VPN and office IPs'`: Adds a descriptive comment to the policy.
    - What it does: Provides metadata for the policy.
    - How it changes the result set: N/A, part of policy definition.

- `ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;`: This statement applies the newly created policy at the account level.
    - What it does: Enforces the network policy for all users and connections to the Snowflake account, unless explicitly overridden.
    - How it changes the result set: N/A, changes account configuration.

- `ALTER USER JOHN_DOE SET NETWORK_POLICY = NULL;`: This statement removes any user-specific network policy, causing the user to inherit the account-level policy (if one exists). If no account policy, no network restrictions apply to the user.
    - What it does: Resets a user's network policy setting.
    - How it changes the result set: N/A, changes user configuration.

- `ALTER USER JANE_SMITH SET NETWORK_POLICY = analyst_network_policy;`: This statement applies a specific network policy to a user, overriding any account-level policy for that user.
    - What it does: Sets a user-specific network access rule.
    - How it changes the result set: N/A, changes user configuration.

## When to Use

1.  **Enhancing Security for Corporate Networks**: When you want to ensure that only connections originating from your corporate offices or VPNs can access your Snowflake account.
    *   *Example Scenario*: A company requires all database access to come through its secure internal network or approved VPN, blocking direct internet access.
2.  **Compliance Requirements**: When regulatory frameworks (like HIPAA, PCI DSS, GDPR) demand strict control over network access to sensitive data.
    *   *Example Scenario*: A healthcare provider stores patient data in Snowflake and must prove that only authorized internal systems can connect to the database.
3.  **Restricting Access for Specific Users or Roles**: To provide granular network access control for users who might need different access patterns (e.g., a contractor accessing from a specific, temporary IP).
    *   *Example Scenario*: An external consultant needs temporary access, but only from their specific client-provided VPN IP, while internal employees follow the main corporate policy.

## When Not to Use

1.  **Highly Dynamic IP Environments**: If users frequently connect from locations with constantly changing public IP addresses (e.g., public Wi-Fi, various mobile hotspots).
    *   *Reason*: Maintaining an up-to-date `ALLOWED_IP_LIST` would be cumbersome and lead to frequent lockout issues.
2.  **Public-Facing Applications without Fixed IPs**: For applications that need to connect to Snowflake but run on infrastructure with dynamic or unpredictable outbound IP addresses (e.g., certain serverless functions or containerized services without fixed egress IPs).
    *   *Reason*: It becomes impractical to whitelist all possible IP ranges, and such configurations often benefit more from private connectivity solutions like Private Link.
3.  **Sole Security Measure**: Relying solely on Network Policies for security without implementing other layers like strong authentication (MFA), role-based access control (RBAC), and data encryption.
    *   *Reason*: Network Policies are a perimeter defense; they don't protect against threats originating from within the allowed network or compromised credentials.

## Common Mistakes

1.  **Forgetting to Apply the Policy**: Creating a network policy but failing to activate it at the account or user level, leaving the account unsecured.
2.  **Incorrect IP/CIDR Notation**: Using incorrect IP addresses or CIDR block notations, leading to legitimate users being blocked or unintended IPs gaining access.
3.  **Overlapping or Conflicting Policies**: Applying policies at both the account and user level without understanding the precedence, leading to unexpected access behaviors (user-level policies override account-level ones).
4.  **Not Whitelisting Snowflake's Own IP Ranges**: In very restrictive environments, sometimes users forget that Snowflake's internal services (like Snowpipe, external stages, or specific cloud provider features) might also need to access the account, requiring their IP ranges to be whitelisted.
5.  **Not Testing Thoroughly**: Implementing a policy without thoroughly testing access from all expected locations, potentially locking out critical users or applications.

## Expected Output

Applying a Network Policy does not produce a direct SQL result set. Instead, it alters the network accessibility of your Snowflake account or specific users. If you query `SHOW NETWORK POLICIES;` or `SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;`, you would see information about the defined policies and their assignments.

Example Output (from `SHOW NETWORK POLICIES;`):

| created_on                     | name                           | owner | comment                                                  | entries                                                     |
| :----------------------------- | :----------------------------- | :---- | :------------------------------------------------------- | :---------------------------------------------------------- |
| 2026-03-14 10:00:00.000 -0700 | MY_CORPORATE_NETWORK_POLICY    | SYSADMIN | Policy to allow connections only from corporate VPN and office IPs | `ALLOWED_IP_LIST = ('203.0.113.1', '203.0.113.2', '192.0.2.0/24')` |
| 2026-03-14 10:05:00.000 -0700 | ANALYST_NETWORK_POLICY         | SYSADMIN | Policy for specific analyst VPN                          | `ALLOWED_IP_LIST = ('198.51.100.10')`                       |

- `created_on`: Timestamp indicating when the policy was created.
- `name`: The name given to the network policy.
- `owner`: The role that owns the network policy.
- `comment`: The description provided for the policy.
- `entries`: The actual IP list defined for the policy.
