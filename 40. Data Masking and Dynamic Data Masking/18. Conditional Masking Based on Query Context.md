# Conditional Masking Based on Query Context in Snowflake

Conditional Masking Based on Query Context in Snowflake refers to the ability to dynamically mask sensitive data based on the characteristics of the query being executed, such as the role of the user, the IP address from which the query originated, or other session-level parameters. This advanced security feature ensures that data exposure is minimized and controlled precisely according to defined policies, enhancing data privacy and compliance.

**Short Explanation**

This SQL feature solves the problem of granular data security, allowing different users or applications to see different versions of the same data within the same table, based on their context. It's important in databases and analytics because it provides robust, row-level and column-level security without needing to create multiple copies of data or complex views for each access level. It is commonly used in real workflows where compliance regulations (like GDPR, HIPAA) demand strict control over sensitive data access, such as financial institutions, healthcare providers, and any organization handling personal identifiable information (PII).

## Implementation Example

```sql
-- Create a table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    customer_name VARCHAR,
    customer_email VARCHAR,
    sale_amount DECIMAL(10, 2),
    sale_date DATE
);

-- Insert some sample data
INSERT INTO sales_data (sale_id, customer_name, customer_email, sale_amount, sale_date) VALUES
    (1, 'Alice Wonderland', 'alice.w@example.com', 150.75, '2025-01-10'),
    (2, 'Bob Thebuilder', 'bob.t@example.com', 200.00, '2025-01-11'),
    (3, 'Charlie Chaplin', 'charlie.c@example.com', 50.25, '2025-01-12');

-- Create a masking policy
CREATE OR REPLACE MASKING POLICY email_masking_policy AS (val VARCHAR) RETURNS VARCHAR ->
    CASE
        WHEN CURRENT_ROLE() IN ('ANALYST_ROLE', 'DATA_STEWARD_ROLE') THEN val
        WHEN CURRENT_ROLE() = 'SALES_REP_ROLE' THEN '******' || SUBSTR(val, INSTR(val, '@'))
        ELSE '***MASKED***'
    END;

-- Apply the masking policy to the customer_email column
ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_masking_policy;

-- Test with different roles (assuming roles are created and granted)
-- USE ROLE ANALYST_ROLE;
-- SELECT customer_name, customer_email FROM sales_data;

-- USE ROLE SALES_REP_ROLE;
-- SELECT customer_name, customer_email FROM sales_data;

-- USE ROLE PUBLIC; -- Or any other role without specific access
-- SELECT customer_name, customer_email FROM sales_data;
```

## Explanation of the Code

This example demonstrates how to implement conditional masking on the `customer_email` column of the `sales_data` table based on the `CURRENT_ROLE()` of the user executing the query.

-   **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to store sales information, including sensitive customer email addresses.
-   **`INSERT INTO sales_data (...)`**: Populates the `sales_data` table with some example records.
-   **`CREATE OR REPLACE MASKING POLICY email_masking_policy AS (val VARCHAR) RETURNS VARCHAR -> ...`**: This is the core of the conditional masking.
    -   It defines a new masking policy named `email_masking_policy`.
    -   `AS (val VARCHAR) RETURNS VARCHAR` specifies that this policy takes a `VARCHAR` input (the original column value) and returns a `VARCHAR` (the masked value).
    -   The `CASE` statement inside the policy logic is where the "query context" comes into play:
        -   `WHEN CURRENT_ROLE() IN ('ANALYST_ROLE', 'DATA_STEWARD_ROLE') THEN val`: If the user's current role is `ANALYST_ROLE` or `DATA_STEWARD_ROLE`, they see the original, unmasked `val` (email address).
        -   `WHEN CURRENT_ROLE() = 'SALES_REP_ROLE' THEN '******' || SUBSTR(val, INSTR(val, '@'))`: If the user's current role is `SALES_REP_ROLE`, they see a partially masked email, revealing only the domain (e.g., `******@example.com`).
        -   `ELSE '***MASKED***'`: For any other role, the email address is completely replaced with a generic `***MASKED***` string.
-   **`ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_masking_policy;`**: This statement applies the `email_masking_policy` to the `customer_email` column in the `sales_data` table. Once applied, any query on this column will trigger the masking policy.

**How it changes the result set:**
The masking policy acts as a filter on the column's *display value*. Depending on the `CURRENT_ROLE()` of the user querying the `customer_email` column, the actual data returned will be different (original, partially masked, or fully masked) without altering the underlying stored data.

## When to Use

1.  **Data Governance and Compliance (GDPR, HIPAA):** When sensitive personal data (PII, PHI) needs to be protected, and access must be restricted based on user roles or departmental responsibilities.
    *   *Example Scenario:* A healthcare database stores patient health records (PHI). Doctors need to see full patient details, while billing staff only need to see anonymized patient IDs and financial information, and external auditors need highly masked data.
2.  **Multi-tenancy and SaaS Applications:** In environments where multiple customers share the same database infrastructure, but each customer's data must remain isolated or masked from others.
    *   *Example Scenario:* A SaaS CRM platform stores customer contact information. Support agents should only see masked customer email addresses or phone numbers for privacy, unless explicitly granted full access for a specific support ticket.
3.  **Analytics and Reporting:** To allow analysts to perform aggregate analysis on sensitive data without exposing individual sensitive records directly.
    *   *Example Scenario:* Business intelligence dashboards display sales performance, including customer demographics. Analysts might need to see aggregated demographic data but should only see masked individual customer identifiers like names or exact addresses.

## When Not to Use

1.  **Simple Access Control Needs:** If data access control can be handled simply by granting or revoking `SELECT` privileges on entire tables or columns, dynamic masking might be an over-engineering.
    *   *Situation:* A table contains no sensitive data, or all users either need full access or no access to a specific column.
2.  **Performance Critical Workloads:** While Snowflake's masking policies are optimized, applying complex `CASE` statements based on many context variables for every query on a very large, frequently queried column can introduce a slight performance overhead.
    *   *Situation:* A high-throughput OLTP system where every millisecond counts, and the masking logic is excessively complex or applied to columns used in frequent `WHERE` clauses.
3.  **When Data Needs to Be Transformed, Not Just Masked:** If the requirement is to permanently alter, encrypt, or pseudonymize data before it's even stored, or for data migration purposes, masking policies are not the right tool.
    *   *Situation:* Data needs to be irreversibly anonymized for public release or long-term archival, where the original data must no longer exist even to privileged users.

## Common Mistakes

1.  **Forgetting to Apply the Policy:** Creating a masking policy but neglecting to apply it to the target column using `ALTER TABLE ... MODIFY COLUMN ... SET MASKING POLICY ...`. The policy won't take effect until it's explicitly applied.
2.  **Incorrect Role Hierarchies or Grants:** Assuming a role has access because it's a higher-level role, without properly configuring role hierarchies or granting the necessary privileges for the roles to access the masked data (if unmasked access is intended).
3.  **Overly Complex Policy Logic:** Writing extremely intricate `CASE` statements with many conditions or expensive functions. This can make the policy hard to maintain, debug, and potentially impact query performance.
4.  **Not Testing All Contexts:** Only testing the masking policy with one or two user roles, overlooking edge cases or unintended data exposure for roles not explicitly handled in the `CASE` statement.

## Expected Output

The resulting dataset will show the `customer_name` as it is, but the `customer_email` column will display different values based on the `CURRENT_ROLE()` of the user executing the `SELECT` statement.

**Example Table (when queried by `ANALYST_ROLE`):**

| customer_name | customer_email        |
| :------------ | :-------------------- |
| Alice Wonderland | alice.w@example.com   |
| Bob Thebuilder | bob.t@example.com     |
| Charlie Chaplin | charlie.c@example.com |

**Example Table (when queried by `SALES_REP_ROLE`):**

| customer_name | customer_email        |
| :------------ | :-------------------- |
| Alice Wonderland | ******@example.com   |
| Bob Thebuilder | ******@example.com     |
| Charlie Chaplin | ******@example.com |

**Example Table (when queried by `PUBLIC` role or any other unspecified role):**

| customer_name | customer_email |
| :------------ | :------------- |
| Alice Wonderland | ***MASKED***   |
| Bob Thebuilder | ***MASKED***   |
| Charlie Chaplin | ***MASKED***   |

**Explanation of the columns:**

-   `customer_name`: This column will always show the original customer's full name, as no masking policy was applied to it.
-   `customer_email`: This column's content will vary. It will show the original email, a partially masked email (revealing only the domain), or a fully masked string (`***MASKED***`), depending on the role of the user running the query, as defined in the `email_masking_policy`.
