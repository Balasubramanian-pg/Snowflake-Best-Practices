# Managing Department Based Row Access in Snowflake

Snowflake Row Access Policies (RAPs) are a powerful security feature that allows you to implement fine-grained access control, restricting which rows of data a user can see based on their role, attributes, or other dynamic criteria. This concept is crucial for multi-tenant architectures, regulatory compliance, and protecting sensitive information, enabling different departments to view only their relevant data within the same table.

**Short Explanation**

Row Access Policies in Snowflake dynamically filter rows in a table at query time, ensuring users only see data they are authorized to access. They solve the problem of needing to create separate, duplicate tables or views for different user groups, simplifying data management and improving security. This is particularly important in scenarios where sensitive data needs to be segmented by organizational units like departments, allowing for a single source of truth while maintaining strict access controls.

- **What problem does this SQL feature solve?** It solves the problem of restricting data visibility at the row level without creating multiple copies of data or complex view hierarchies.
- **Why is it important in databases or analytics?** It's vital for data governance, compliance (e.g., GDPR, HIPAA), and multi-tenant environments where different user groups (like departments) should only access their own data.
- **Where is it commonly used in real workflows?** Commonly used in finance for restricting access to specific transactions, in HR for managing employee data, and in sales for department-specific customer or sales records.

## Implementation Example

This example demonstrates how to create a Row Access Policy that filters data based on a user's current role matching a department ID in a `users_department_mapping` table.

```sql
-- 1. Create a table with sensitive department data
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    department_id VARCHAR,
    sale_amount DECIMAL(10, 2)
);

-- 2. Insert some sample data
INSERT INTO sales_data (order_id, customer_name, department_id, sale_amount) VALUES
(101, 'Alice', 'SALES_EAST', 1500.00),
(102, 'Bob', 'SALES_WEST', 2200.50),
(103, 'Charlie', 'MARKETING', 800.00),
(104, 'David', 'SALES_EAST', 300.75),
(105, 'Eve', 'SALES_WEST', 5000.00);

-- 3. Create a mapping table for roles to department_ids
CREATE OR REPLACE TABLE users_department_mapping (
    role_name VARCHAR,
    department_id VARCHAR
);

-- 4. Populate the mapping table
INSERT INTO users_department_mapping (role_name, department_id) VALUES
('SALES_EAST_ANALYST', 'SALES_EAST'),
('SALES_WEST_ANALYST', 'SALES_WEST'),
('MARKETING_ANALYST', 'MARKETING'),
('DATA_ADMIN', NULL); -- DATA_ADMIN can see all departments

-- 5. Create a custom role (if not already existing)
CREATE ROLE IF NOT EXISTS SALES_EAST_ANALYST;
CREATE ROLE IF NOT EXISTS SALES_WEST_ANALYST;
CREATE ROLE IF NOT EXISTS MARKETING_ANALYST;
CREATE ROLE IF NOT EXISTS DATA_ADMIN;

-- Grant necessary privileges to the custom roles
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE SALES_EAST_ANALYST, SALES_WEST_ANALYST, MARKETING_ANALYST, DATA_ADMIN;
GRANT USAGE ON DATABASE <your_database_name> TO ROLE SALES_EAST_ANALYST, SALES_WEST_ANALYST, MARKETING_ANALYST, DATA_ADMIN;
GRANT USAGE ON SCHEMA <your_schema_name> TO ROLE SALES_EAST_ANALYST, SALES_WEST_ANALYST, MARKETING_ANALYST, DATA_ADMIN;

-- Grant SELECT on the mapping table to all roles that need to use the policy
GRANT SELECT ON TABLE users_department_mapping TO ROLE SALES_EAST_ANALYST, SALES_WEST_ANALYST, MARKETING_ANALYST, DATA_ADMIN;

-- 6. Create the Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY department_row_access_policy
AS (department_id VARCHAR) RETURNS BOOLEAN ->
    -- Allow DATA_ADMIN to see all rows
    CURRENT_ROLE() = 'DATA_ADMIN'
    OR
    -- Allow other roles to see rows matching their assigned department_id
    EXISTS (
        SELECT 1
        FROM users_department_mapping udm
        WHERE udm.role_name = CURRENT_ROLE()
          AND udm.department_id = department_id
    );

-- 7. Apply the Row Access Policy to the sales_data table
ALTER TABLE sales_data ADD ROW ACCESS POLICY department_row_access_policy ON (department_id);

-- 8. Test the policy as different roles (e.g., using USE ROLE)
-- USE ROLE SALES_EAST_ANALYST;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_WEST_ANALYST;
-- SELECT * FROM sales_data;

-- USE ROLE DATA_ADMIN;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

- **`CREATE OR REPLACE TABLE sales_data (...)`**: Defines a table `sales_data` to hold transactional information, including a `department_id` column which will be used for filtering.
- **`INSERT INTO sales_data (...)`**: Populates the `sales_data` table with sample records across different departments.
- **`CREATE OR REPLACE TABLE users_department_mapping (...)`**: Creates a lookup table that maps custom roles to specific `department_id` values. This table is crucial for dynamic, role-based filtering.
- **`INSERT INTO users_department_mapping (...)`**: Populates the mapping table, associating roles like `SALES_EAST_ANALYST` with `SALES_EAST` department data. A `DATA_ADMIN` role is included with a `NULL` department to signify unrestricted access.
- **`CREATE ROLE IF NOT EXISTS...`**: Creates custom roles that will be used to simulate different department users.
- **`GRANT USAGE ON...`**: Grants basic permissions (warehouse, database, schema usage) to the new roles.
- **`GRANT SELECT ON TABLE users_department_mapping...`**: Grants `SELECT` permission on the `users_department_mapping` table to the roles. This is necessary because the Row Access Policy will query this table, and the policy is executed under the context of the querying role.
- **`CREATE OR REPLACE ROW ACCESS POLICY department_row_access_policy AS (department_id VARCHAR) RETURNS BOOLEAN ->...`**: This is the core of the row-level security.
    - `AS (department_id VARCHAR) RETURNS BOOLEAN`: Defines the policy, specifying that it takes a `department_id` as an input and returns a boolean (`TRUE` to show the row, `FALSE` to hide it).
    - `CURRENT_ROLE() = 'DATA_ADMIN'`: This condition allows the `DATA_ADMIN` role to bypass the filtering and see all rows.
    - `OR EXISTS (SELECT 1 FROM users_department_mapping udm WHERE udm.role_name = CURRENT_ROLE() AND udm.department_id = department_id)`: This is the dynamic filtering logic. For any other role, it checks if there is an entry in the `users_department_mapping` table where the `role_name` matches the `CURRENT_ROLE()` and the `department_id` in the `sales_data` table matches the `department_id` assigned to that role in the mapping table. If a match is found, the `EXISTS` clause returns `TRUE`, and the row is visible.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY department_row_access_policy ON (department_id)`**: This command applies the created `department_row_access_policy` to the `sales_data` table, specifying that the `department_id` column from `sales_data` should be passed as the argument to the policy.

## When to Use

1. **Multi-Tenant Data Architectures:** When a single database table stores data for multiple departments, clients, or business units, and each unit should only see its own data.
    * *Scenario:* A `CUSTOMER_ORDERS` table contains orders from various regions. A `SALES_NORTH` role should only see orders from the 'North' region.
2. **Regulatory Compliance:** To meet data privacy regulations that mandate strict data segregation, such as preventing users from seeing Personally Identifiable Information (PII) belonging to individuals outside their authorized scope.
    * *Scenario:* An `EMPLOYEE_HR_DATA` table has sensitive employee records. HR managers for specific departments should only view data for employees within their department.
3. **Simplified Data Management:** When you want to maintain a single, consolidated dataset for analytical purposes but need to present filtered views to different stakeholders without creating numerous separate views or data extracts.
    * *Scenario:* A `GLOBAL_SALES_REPORT` table is used by analysts worldwide, but each regional analyst should only see sales figures for their assigned territory.

## When Not to Use

1. **Performance Critical Queries on Very Large Tables with Complex Policies:** If the policy involves complex lookups (like joining mapping tables) on extremely large tables, it can introduce significant query overhead, especially if the policy column is not aligned with clustering keys.
    * *Situation:* A policy with multiple `EXISTS` subqueries on a fact table with billions of rows, where the policy's filter column is not part of the table's clustering key.
2. **When Column-Level Security is Sufficient:** If the requirement is to hide or mask specific columns of data rather than entire rows, Dynamic Data Masking policies might be a more appropriate and sometimes more performant solution.
    * *Situation:* Users need to see all customer records, but only certain roles should see the `credit_card_number` column.
3. **Simple Table-Level Access Control:** For basic access control where entire tables or views should be visible or hidden based on roles, standard `GRANT` and `REVOKE` privileges on tables/views are simpler and more efficient.
    * *Situation:* Only the `FINANCE` role should have access to the entire `GENERAL_LEDGER` table, and other roles should not see it at all.

## Common Mistakes

1. **Forgetting to Grant `SELECT` on Mapping Tables:** The policy itself (which runs under the context of the user's role) needs `SELECT` privileges on any lookup tables it references. If not granted, the policy will fail silently or return no rows.
2. **Not Testing with Different Roles:** Developers often test policies with high-privileged roles (like `ACCOUNTADMIN` or `SYSADMIN`) which might bypass the policy, leading to a false sense of security. Always test with the specific limited roles the policy is intended for.
3. **Overly Complex Policy Logic:** Policies that involve many joins, subqueries, or complex conditional logic can become difficult to manage, debug, and can significantly degrade query performance. Keep policies as simple and efficient as possible.
4. **Not Understanding Policy Application Order with Masking Policies:** If both a Row Access Policy and a Masking Policy are applied to the same table, the Row Access Policy is evaluated *first*. Misunderstanding this order can lead to unexpected results.
5. **Hardcoding Values in Policy:** Instead of using a mapping table, sometimes developers hardcode department IDs directly into the policy. This makes the policy rigid and requires modification and re-application whenever department structures or role assignments change.

## Expected Output

When a `SELECT * FROM sales_data;` query is run, the resulting dataset will only contain rows where the `department_id` matches the `department_id` associated with the `CURRENT_ROLE()` in the `users_department_mapping` table, or all rows if the `CURRENT_ROLE()` is `DATA_ADMIN`.

*Explanation:* This table shows only the sales data belonging to the 'SALES\_EAST' department. The `ORDER_ID`, `CUSTOMER_NAME`, `DEPARTMENT_ID`, and `SALE_AMOUNT` columns are included for each visible row.

*Explanation:* This table shows only the sales data belonging to the 'SALES\_WEST' department.
