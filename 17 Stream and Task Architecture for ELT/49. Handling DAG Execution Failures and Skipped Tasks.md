# Handling DAG Execution Failures and Skipped Tasks
Handling DAG execution failures and skipped tasks is crucial in maintaining the reliability and efficiency of data pipelines. A Directed Acyclic Graph (DAG) is a collection of tasks that are executed in a specific order, and failures or skipped tasks can have significant impacts on the overall pipeline. By implementing proper handling mechanisms, developers can ensure that their pipelines are robust and can recover from failures. This can be achieved through the use of retries, notifications, and task skipping. Effective handling of DAG execution failures and skipped tasks is essential for data pipeline management.

## Implementation Example
```sql
FROM datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.bash_operator import BashOperator
from airflow.operators.python_operator import PythonOperator

default_args = {
    'owner': 'airflow',
    'depends_on_past': False,
    'start_date': datetime(2023, 3, 20),
    'retries': 1,
    'retry_delay': timedelta(minutes=5),
}

dag = DAG(
    'example_dag',
    default_args=default_args,
    schedule_interval=timedelta(days=1),
)

def example_task():
    # task code here
    pass

task = BashOperator(
    task_id='example_task',
    bash_command='echo "Hello World!"',
    dag=dag,
)

task2 = PythonOperator(
    task_id='example_task2',
    python_callable=example_task,
    dag=dag,
)
```
Few examples and alternatives of the code include using `SimpleHttpOperator` for HTTP requests, `MySqlOperator` for MySQL queries, and `PythonOperator` for custom Python tasks.

## Explanation of the Code
* The `default_args` dictionary is used to define default arguments for the DAG, including the owner, dependency on past runs, start date, number of retries, and retry delay.
* The `DAG` object is created with the specified default arguments, schedule interval, and DAG ID.
* The `BashOperator` is used to execute a bash command, while the `PythonOperator` is used to execute a custom Python function.
* The `task_id` parameter is used to specify a unique ID for each task.
* The `dag` parameter is used to assign the task to the specified DAG.
* The `retries` parameter is used to specify the number of times a task should be retried in case of failure.
* The `retry_delay` parameter is used to specify the delay between retries.

## When to use
This approach can be used in the following scenarios:
* When a task is prone to failures due to external factors, such as network connectivity issues or third-party API downtime. In this case, implementing retries can help ensure that the task is executed successfully.
* When a task has a significant impact on the overall pipeline, such as a critical data processing task. In this case, implementing proper handling mechanisms can help prevent pipeline failures.

## When not to use
This approach may not be suitable in the following scenarios:
* When a task has a low impact on the overall pipeline, such as a non-critical data processing task. In this case, implementing retries may not be necessary.
* When a task has a high risk of causing data inconsistencies or corruption, such as a task that writes to a database. In this case, implementing retries may exacerbate the issue.

## Common Mistakes
* Not specifying a sufficient number of retries, leading to premature task failure.
* Not specifying a sufficient retry delay, leading to excessive resource usage.
* Not handling task failures properly, leading to pipeline failures.
* Not monitoring task execution and retry attempts, leading to undetected issues.

The expected output of the DAG execution will depend on the specific tasks and their outcomes. However, in general, the output will include:
