# Deny All By Default Strategy in Snowflake

This strategy is a fundamental security principle where access to resources is denied to all users and roles by default, unless explicitly granted. It's a cornerstone of robust security models, ensuring that only authorized entities can perform specific actions on data and objects. The purpose is to minimize the attack surface and prevent unauthorized data access or manipulation.

**Short Explanation**

The "deny all by default" strategy means that if you don't specifically give someone permission to do something, they can't do it. It addresses the problem of accidental over-privileging, reducing security risks by limiting access to only what's absolutely necessary. This is important in databases and analytics to safeguard sensitive information and maintain data integrity, and it's commonly used in setting up initial security policies and managing user permissions in any regulated environment.

- What problem does this SQL feature solve? It solves the problem of accidental data exposure or unauthorized access by ensuring that access is always restricted unless explicitly allowed.
- Why is it important in databases or analytics? It's crucial for maintaining data security, compliance with regulations, and preventing data breaches, especially with sensitive analytical datasets.
- Where is it commonly used in real workflows? It's used when defining security policies, creating new users and roles, managing object privileges, and adhering to principle of least privilege in data warehousing and data lake environments.

## Implementation Example

```sql
-- Step 1: Create a custom role
CREATE ROLE ANALYST_ROLE;

-- Step 2: Grant the custom role to a user
GRANT ROLE ANALYST_ROLE TO USER JANE_DOE;

-- Step 3: Deny all access to a specific schema for public (implicitly, this is often the default)
-- No explicit DENY statement is usually needed here because it's deny-by-default.
-- Instead, we focus on granting specific access to the ANALYST_ROLE.

-- Step 4: Grant specific usage on a database and schema to the custom role
GRANT USAGE ON DATABASE PRODUCTION_DB TO ROLE ANALYST_ROLE;
GRANT USAGE ON SCHEMA PRODUCTION_DB.SALES_DATA TO ROLE ANALYST_ROLE;

-- Step 5: Grant specific select privilege on a table to the custom role
GRANT SELECT ON TABLE PRODUCTION_DB.SALES_DATA.CUSTOMER_SALES TO ROLE ANALYST_ROLE;

-- Step 6: Verify Jane_Doe (with ANALYST_ROLE activated) can query the table
-- As JANE_DOE:
USE ROLE ANALYST_ROLE;
SELECT * FROM PRODUCTION_DB.SALES_DATA.CUSTOMER_SALES LIMIT 10;
-- This will succeed.

-- Attempt to access an unauthorized table (assuming no other grants)
-- As JANE_DOE:
SELECT * FROM PRODUCTION_DB.FINANCE_DATA.EXPENSE_REPORTS LIMIT 10;
-- This will fail because no explicit grant was given for FINANCE_DATA.EXPENSE_REPORTS.
```

## Explanation of the Code

This example demonstrates the "Deny All By Default" principle by showing how access is implicitly denied unless explicitly granted.

- `CREATE ROLE ANALYST_ROLE;`: This creates a new security role named `ANALYST_ROLE`. This is the first step in defining a new set of permissions that will be applied.
- `GRANT ROLE ANALYST_ROLE TO USER JANE_DOE;`: This assigns the newly created `ANALYST_ROLE` to a specific user, `JANE_DOE`. This user will now inherit any privileges granted to `ANALYST_ROLE`.
- `GRANT USAGE ON DATABASE PRODUCTION_DB TO ROLE ANALYST_ROLE;`: This grants the `USAGE` privilege on the `PRODUCTION_DB` database to `ANALYST_ROLE`.
    - **What it does**: Allows the role to access objects within the database.
    - **How it changes the result set**: It doesn't directly change a query's result set, but it's a prerequisite for any queries to succeed on objects within that database. Without this, even `SELECT` statements would fail.
- `GRANT USAGE ON SCHEMA PRODUCTION_DB.SALES_DATA TO ROLE ANALYST_ROLE;`: This grants the `USAGE` privilege on the `SALES_DATA` schema within `PRODUCTION_DB` to `ANALYST_ROLE`.
    - **What it does**: Allows the role to access objects within that specific schema.
    - **How it changes the result set**: Similar to database `USAGE`, it enables access to schema objects, allowing queries to eventually return results.
- `GRANT SELECT ON TABLE PRODUCTION_DB.SALES_DATA.CUSTOMER_SALES TO ROLE ANALYST_ROLE;`: This is the most granular grant, allowing `ANALYST_ROLE` to perform `SELECT` queries on the `CUSTOMER_SALES` table.
    - **What it does**: Permits retrieving data from the specified table.
    - **How it changes the result set**: Enables the role to execute `SELECT` statements against `CUSTOMER_SALES`, thereby generating a result set from that table.
- `SELECT * FROM PRODUCTION_DB.SALES_DATA.CUSTOMER_SALES LIMIT 10;` (as JANE_DOE): This query will succeed because all necessary `USAGE` and `SELECT` privileges were explicitly granted.
- `SELECT * FROM PRODUCTION_DB.FINANCE_DATA.EXPENSE_REPORTS LIMIT 10;` (as JANE_DOE): This query will fail because no `USAGE` or `SELECT` privileges were granted for the `FINANCE_DATA` schema or the `EXPENSE_REPORTS` table to `ANALYST_ROLE` (and thus to `JANE_DOE`). This failure demonstrates the "Deny All By Default" principle in action.

## When to Use

1.  **Establishing Initial Security Policies**: When setting up a new Snowflake account or a new project within an existing account, it's best practice to start with a "deny all by default" approach for all custom roles and users. This ensures a secure baseline from the beginning.
    *   *Example scenario*: A new data warehouse is being deployed. Instead of opening access broadly, the security team defines specific roles (e.g., `DATA_ENGINEER`, `BI_ANALYST`, `DATA_SCIENTIST`) and grants them only the minimum necessary permissions on databases, schemas, and tables.
2.  **Managing Access to Sensitive Data**: For highly sensitive data (e.g., PII, financial records, health data), the "deny all by default" strategy is critical. Even within a permitted schema, specific tables or columns might require explicit grants.
    *   *Example scenario*: A table `PATIENT_RECORDS` contains confidential health information. A `RESEARCH_ANALYST` role is created and only granted `SELECT` access to specific non-identifiable columns, while `FULL_ACCESS_ADMINS` are granted access to all columns. Any new user without these explicit grants cannot see the data.
3.  **Adhering to Least Privilege Principle**: This strategy directly supports the security principle of "least privilege," where users and roles are given only the permissions absolutely required to perform their job functions, and nothing more.
    *   *Example scenario*: A contractor needs to load data into a staging table. They are granted `INSERT` and `SELECT` on that specific staging table only. They cannot `DELETE`, `UPDATE`, or `TRUNCATE` the table, nor can they access any other tables in the database.

## When Not to Use

1.  **Small, Non-Sensitive Development Environments**: In very small, isolated development environments with no sensitive data and a limited number of trusted users, the overhead of strictly enforcing "deny all by default" for every single object might be more trouble than it's worth. Broader, more permissive grants might be acceptable for agility, *provided there's no path to production or sensitive data*.
    *   *Inefficiency*: Constantly granting granular permissions for every new temporary object can slow down rapid prototyping.
2.  **When Granting Too Broadly with Future Intent**: If you grant `OWNERSHIP` or `ALL PRIVILEGES` on an entire schema or database, you're essentially bypassing the "deny all by default" principle for new objects created within that scope. This can lead to over-privileging down the line.
    *   *Wrong approach*: Granting `ALL PRIVILEGES ON SCHEMA DEV_SCHEMA TO DEVELOPER_ROLE` is too broad if developers only need to create tables and select from others. This negates the security benefit for objects they don't own.
3.  **When Inherited Roles Provide Sufficient Granularity**: If a well-designed role hierarchy already exists and higher-level roles implicitly provide the necessary access without exposing sensitive data, over-engineering granular grants at lower levels might be redundant.
    *   *Inefficiency*: If `SALES_ADMIN` role already has `SELECT` on all sales tables, creating another `SALES_READ_ONLY` role and explicitly granting `SELECT` on every single sales table to it, when `SALES_ADMIN` could just be a parent, might be overly complex.

## Common Mistakes

1.  **Forgetting `USAGE` Grants**: Developers often focus on `SELECT`, `INSERT`, `UPDATE` grants but forget to grant `USAGE` on the parent database and schema. Without `USAGE` on the containing database and schema, no operations on objects within them are possible, leading to "object does not exist or not authorized" errors.
2.  **Over-Granting at the Database/Schema Level**: While trying to adhere to "deny all by default," some administrators might grant `ALL PRIVILEGES` on a database or schema thinking it's simpler. This negates the principle for any objects within that scope, including future ones, creating a potential security hole.
3.  **Not Understanding Role Hierarchy and Inheritance**: Snowflake's role hierarchy means privileges are inherited. A common mistake is granting a specific privilege to a role, then granting that role to another role, leading to unintended expanded access if not carefully planned. Forgetting to `USE ROLE` the correct context when testing permissions is also common.
4.  **Implicit vs. Explicit Grants**: Assuming a user *doesn't* have access because a `DENY` statement wasn't issued. In Snowflake (and SQL generally), it's about what you *grant*. If you don't grant it, it's denied by default. Explicit `DENY` is rarely used and often makes security harder to manage than explicit `GRANT` statements.

## Expected Output

When adhering to the "Deny All By Default" strategy, the expected output for a user attempting an unauthorized action is an error indicating insufficient privileges. For authorized actions, the output is the requested data or confirmation of the action.

**Example Table (for an authorized `SELECT` operation):**

| CUSTOMER_ID | ORDER_DATE | TOTAL_AMOUNT |
|-------------|------------|--------------|
| 1001        | 2026-01-15 | 125.50       |
| 1002        | 2026-01-16 | 300.75       |
| 1003        | 2026-01-16 | 50.00        |

- **CUSTOMER_ID**: An integer representing a unique customer identifier.
- **ORDER_DATE**: The date the customer placed the order.
- **TOTAL_AMOUNT**: The total monetary value of the order.

This table represents the data retrieved from `PRODUCTION_DB.SALES_DATA.CUSTOMER_SALES` when `JANE_DOE` (with `ANALYST_ROLE`) successfully executes a `SELECT` query, as she was explicitly granted `SELECT` access. If she tried to select from a table without explicit grants, the output would be an authorization error message instead of data.
