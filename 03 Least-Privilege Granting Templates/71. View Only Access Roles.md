# View Only Access Roles in Snowflake

View Only Access Roles in Snowflake are security constructs designed to grant users the ability to read data from tables, views, and other database objects without permitting any modifications (insert, update, delete). They exist to enforce the principle of least privilege, ensuring users can access necessary information for reporting or analysis without posing a risk to data integrity.

**Short Explanation**

These roles provide read-only permissions, which means users can query data but cannot change it. This solves the problem of unauthorized data modification, making it crucial for maintaining data security and compliance. They are commonly used in scenarios where business intelligence tools, analysts, or auditors need to access data for reporting or insights without the ability to alter the underlying data.

- What problem does this SQL feature solve? It solves the problem of granting data access for analysis and reporting while preventing unintended or unauthorized data modifications.
- Why is it important in databases or analytics? It's important for robust security, compliance, and data governance, ensuring data integrity and auditability.
- Where is it commonly used in real workflows? It's widely used for BI dashboards, data analysis tasks, regulatory compliance audits, and granting access to third-party tools that only need to read data.

## Implementation Example

```sql
-- Create a new custom role for view-only access
CREATE ROLE ANALYST_VIEWER;

-- Grant usage on the database to the new role
GRANT USAGE ON DATABASE my_production_db TO ROLE ANALYST_VIEWER;

-- Grant usage on a specific schema within the database
GRANT USAGE ON SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;

-- Grant SELECT privilege on all existing tables in the schema
GRANT SELECT ON ALL TABLES IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;

-- Grant SELECT privilege on all future tables in the schema (important for new tables)
GRANT SELECT ON FUTURE TABLES IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;

-- Grant SELECT privilege on all existing views in the schema
GRANT SELECT ON ALL VIEWS IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;

-- Grant SELECT privilege on all future views in the schema
GRANT SELECT ON FUTURE VIEWS IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;

-- Grant the ANALYST_VIEWER role to a user
GRANT ROLE ANALYST_VIEWER TO USER jdoe;

-- Grant the ANALYST_VIEWER role to another role (e.g., a functional role)
GRANT ROLE ANALYST_VIEWER TO ROLE BI_TEAM_ROLE;
```

## Explanation of the Code

This SQL example outlines the steps to create and configure a view-only access role in Snowflake.

- `CREATE ROLE ANALYST_VIEWER;`: This command initializes a new role named `ANALYST_VIEWER`. This is the first step to creating a custom set of permissions.
- `GRANT USAGE ON DATABASE my_production_db TO ROLE ANALYST_VIEWER;`: This grants the `ANALYST_VIEWER` role the ability to "use" or navigate into the specified database. It doesn't grant access to data directly, but allows the role to access objects within it.
- `GRANT USAGE ON SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;`: Similar to database usage, this grants the role permission to "use" or navigate within the `public` schema of `my_production_db`.
- `GRANT SELECT ON ALL TABLES IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;`: This is the core command that provides read access. It grants the `SELECT` privilege on all *existing* tables within the `public` schema to the `ANALYST_VIEWER` role. `SELECT` allows querying data.
- `GRANT SELECT ON FUTURE TABLES IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;`: This is a crucial step for maintainability. It ensures that any *new* tables created in the `public` schema will automatically inherit the `SELECT` privilege for the `ANALYST_VIEWER` role, avoiding manual grants for each new table.
- `GRANT SELECT ON ALL VIEWS IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;`: Similar to tables, this grants `SELECT` privilege on all *existing* views in the schema.
- `GRANT SELECT ON FUTURE VIEWS IN SCHEMA my_production_db.public TO ROLE ANALYST_VIEWER;`: This ensures any *new* views created in the schema will also automatically inherit the `SELECT` privilege.
- `GRANT ROLE ANALYST_VIEWER TO USER jdoe;`: This assigns the newly configured `ANALYST_VIEWER` role to a specific user, `jdoe`, allowing `jdoe` to inherit all permissions granted to `ANALYST_VIEWER`.
- `GRANT ROLE ANALYST_VIEWER TO ROLE BI_TEAM_ROLE;`: This demonstrates granting the `ANALYST_VIEWER` role to another role (`BI_TEAM_ROLE`). This creates a role hierarchy, where `BI_TEAM_ROLE` inherits all privileges of `ANALYST_VIEWER`, which is a common practice for managing permissions for teams.

## When to Use

1.  **Business Intelligence and Reporting:** When analysts or BI tools need to query data for dashboards and reports, but should not have the ability to alter the source data.
    *   *Scenario:* A marketing team needs to analyze campaign performance data in the `marketing_campaigns` table. A `MARKETING_ANALYST_VIEWER` role is created and granted `SELECT` access to this table.
2.  **Compliance and Auditing:** For auditors or regulatory bodies who require access to historical or current data for review, ensuring they can verify information without accidental or intentional data changes.
    *   *Scenario:* An external auditor needs to review financial transaction data. A `FINANCIAL_AUDITOR_VIEWER` role is created and granted `SELECT` access to relevant financial tables.
3.  **Third-Party Integrations:** When integrating external applications or services that only need to consume data from your Snowflake instance, such as data visualization platforms or data science notebooks.
    *   *Scenario:* A data science platform needs to ingest customer data for model training. A `DATA_SCIENCE_FEED_VIEWER` role is created and granted `SELECT` access to the `customer_profiles` table.

## When Not to Use

1.  **Data Ingestion and ETL Processes:** For processes that specifically involve loading, transforming, or modifying data.
    *   *Situation:* An ETL pipeline is responsible for inserting new records into a `daily_sales` table. A view-only role would prevent these `INSERT` operations.
2.  **Application User Accounts Requiring Write Access:** When user accounts (e.g., from a web application) are designed to submit or update data.
    *   *Situation:* A CRM application user needs to update customer contact details. A view-only role would block the `UPDATE` statements.
3.  **Database Administration Tasks:** For administrators who need full control over schema, tables, roles, and other database objects.
    *   *Situation:* A DBA needs to create new tables, manage users, or modify role grants. A view-only role would severely restrict their administrative capabilities.

## Common Mistakes

1.  **Forgetting `FUTURE` Grants:** Only granting `SELECT ON ALL TABLES` but forgetting `SELECT ON FUTURE TABLES`. This means new tables created after the grant won't be accessible to the view-only role, leading to broken reports or queries.
2.  **Over-granting USAGE:** Granting `USAGE ON WAREHOUSE` (which is necessary) but then also granting `OPERATE ON WAREHOUSE` by mistake, which gives more privileges than intended for a view-only role.
3.  **Granting SELECT on `INFORMATION_SCHEMA` or `SNOWFLAKE` databases:** While sometimes necessary for introspection, granting broad `SELECT` access on these can expose metadata or account details that might not be intended for general view-only users.
4.  **Not testing the role:** Creating a role and assigning privileges without logging in as a user with that role to verify that only the intended read access (and no write access) is available.

## Expected Output

The expected output when a user assigned the `ANALYST_VIEWER` role executes a `SELECT` statement on an authorized table is the dataset containing the requested columns and rows from that table. If they attempt an `INSERT`, `UPDATE`, or `DELETE`, they will receive a "SQL compilation error: Insufficient privileges" message.

Example of a successful `SELECT` query:

```
SELECT order_id, customer_id, order_date, total_amount FROM my_production_db.public.orders LIMIT 2;
```

Resulting Dataset:

| ORDER_ID | CUSTOMER_ID | ORDER_DATE | TOTAL_AMOUNT |
| :------- | :---------- | :--------- | :----------- |
| 1001     | CUST-001    | 2026-03-01 | 150.75       |
| 1002     | CUST-002    | 2026-03-02 | 225.00       |

Explanation of columns:
- `ORDER_ID`: A unique identifier for each order.
- `CUSTOMER_ID`: An identifier for the customer who placed the order.
- `ORDER_DATE`: The date when the order was placed.
- `TOTAL_AMOUNT`: The total monetary value of the order.

If the user tries to run `INSERT INTO my_production_db.public.orders VALUES (1003, 'CUST-003', '2026-03-14', 50.00);`, the expected output would be an error message indicating insufficient privileges, as the role only has `SELECT` access.
