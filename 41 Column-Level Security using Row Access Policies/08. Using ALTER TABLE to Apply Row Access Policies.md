# Using ALTER TABLE to Apply Row Access Policies in Snowflake

This concept introduces how to enforce data security at the row level in Snowflake by linking a row access policy to a table using the `ALTER TABLE` statement. This mechanism ensures that only authorized users can view specific rows of data based on defined conditions, which is crucial for compliance and data governance.

**Short Explanation**

`ALTER TABLE ... ADD ROW ACCESS POLICY` attaches a pre-defined policy to a table, filtering rows based on access rules. This solves the problem of granular data access control, preventing unauthorized users from seeing sensitive information. It's important for maintaining data privacy and compliance (e.g., GDPR, HIPAA) and is commonly used in data warehousing and analytics to segment data views for different user roles.

- What problem does this SQL feature solve? It solves the problem of restricting access to specific rows within a table based on user roles or other contextual information, ensuring data privacy and regulatory compliance.
- Why is it important in databases or analytics? It's vital for implementing fine-grained access control, safeguarding sensitive data, and creating a single source of truth while still serving diverse user needs with tailored data views.
- Where is it commonly used in real workflows? It's used in scenarios like multi-tenant applications where different customers should only see their own data, or in financial and healthcare industries to comply with strict data access regulations.

## Implementation Example

```sql
-- 1. Create a sample table
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    customer_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sale_amount DECIMAL(10, 2)
);

-- 2. Insert some sample data
INSERT INTO sales_data (order_id, customer_id, product_name, region, sale_amount) VALUES
(1, 101, 'Laptop', 'EAST', 1200.00),
(2, 102, 'Mouse', 'WEST', 25.00),
(3, 101, 'Keyboard', 'EAST', 75.00),
(4, 103, 'Monitor', 'NORTH', 300.00),
(5, 102, 'Webcam', 'WEST', 50.00),
(6, 104, 'Printer', 'SOUTH', 150.00);

-- 3. Create a row access policy
-- This policy allows users to see all data if they are 'ADMIN'
-- or only data from their assigned region if they have a 'SALES_ANALYST' role
-- and their current role matches the region.
CREATE OR REPLACE ROW ACCESS POLICY region_access_policy
AS (region VARCHAR) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'ADMIN' THEN TRUE
        WHEN CURRENT_ROLE() = 'SALES_ANALYST_EAST' AND region = 'EAST' THEN TRUE
        WHEN CURRENT_ROLE() = 'SALES_ANALYST_WEST' AND region = 'WEST' THEN TRUE
        WHEN CURRENT_ROLE() = 'SALES_ANALYST_NORTH' AND region = 'NORTH' THEN TRUE
        WHEN CURRENT_ROLE() = 'SALES_ANALYST_SOUTH' AND region = 'SOUTH' THEN TRUE
        ELSE FALSE
    END;

-- 4. Apply the row access policy to the table
ALTER TABLE sales_data
ADD ROW ACCESS POLICY region_access_policy ON (region);

-- 5. Grant roles for testing (assuming these roles exist)
-- GRANT ROLE ADMIN TO USER <your_user_name>;
-- GRANT ROLE SALES_ANALYST_EAST TO USER <your_user_name>;
-- GRANT ROLE SALES_ANALYST_WEST TO USER <your_user_name>;

-- 6. Test with different roles (e.g., using USE ROLE command)
-- USE ROLE ADMIN;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_ANALYST_EAST;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_ANALYST_WEST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

- `CREATE OR REPLACE TABLE sales_data (...)`: This creates a standard table named `sales_data` to hold our sample sales records. It's the base table to which the policy will be applied.
- `INSERT INTO sales_data (...) VALUES (...)`: This populates the `sales_data` table with example data, including an `order_id`, `customer_id`, `product_name`, `region`, and `sale_amount`.
- `CREATE OR REPLACE ROW ACCESS POLICY region_access_policy AS (region VARCHAR) RETURNS BOOLEAN -> ...`: This defines the row access policy named `region_access_policy`.
    - **What it does:** It's a schema-level object that evaluates to `TRUE` or `FALSE` for each row, determining if the row is visible. It takes a column (or columns) from the target table as input.
    - **How it changes the result set:** This policy uses `CURRENT_ROLE()` to check the active role of the user. If the user is `ADMIN`, they see all rows (`TRUE`). If they are a `SALES_ANALYST` for a specific region (e.g., `SALES_ANALYST_EAST`) and the row's `region` matches, they see that row (`TRUE`). Otherwise, the row is filtered out (`FALSE`).
- `ALTER TABLE sales_data ADD ROW ACCESS POLICY region_access_policy ON (region);`: This is the core statement.
    - **What it does:** It links the previously defined `region_access_policy` to the `sales_data` table, specifying that the `region` column of the table should be passed as an argument to the policy.
    - **How it changes the result set:** Once applied, any `SELECT` query on `sales_data` will implicitly invoke `region_access_policy` for each row, using the `region` column's value from that row as input to the policy. Only rows for which the policy returns `TRUE` will be included in the query's result set.

## When to Use

1.  **Multi-tenant Data Sharing:** When different customers or departments share a single table but should only see their own specific data.
    *   *Example Scenario:* A SaaS application stores data for multiple companies in one large `transactions` table. A row access policy ensures each company's users only see their own transactions, based on a `company_id` column.
2.  **Regulatory Compliance:** To meet strict data privacy regulations like GDPR, HIPAA, or CCPA, where specific data must be restricted to authorized personnel.
    *   *Example Scenario:* A `patient_records` table needs to hide sensitive medical notes from all users except those with a `MEDICAL_STAFF` role and who are assigned to that specific patient.
3.  **Role-Based Data Segmentation:** When different internal teams or roles require access to only a subset of data relevant to their responsibilities.
    *   *Example Scenario:* In a global sales dataset, regional sales managers (e.g., `APAC_SALES_MANAGER`) should only see sales figures for their respective regions.

## When Not to Use

1.  **Simple Column-Level Security:** If you only need to restrict access to entire columns, not rows, `GRANT/REVOKE` permissions on columns or using secure views are more straightforward and potentially more performant.
    *   *Inefficient Scenario:* Trying to hide a `social_security_number` column from most users using a row access policy that always returns `FALSE` if the user is not an `HR_MANAGER`. A secure view that excludes the column is better.
2.  **Very Frequent Policy Changes or Extremely Complex Logic:** While flexible, highly dynamic policies that change frequently or involve extremely complex logic within the policy UDF can introduce maintenance overhead and potentially impact query performance.
    *   *Inefficient Scenario:* A policy that performs multiple complex joins or subqueries for every row evaluation, especially on very large tables, could become a performance bottleneck.
3.  **Low-Security Data:** For tables that contain only publicly available or non-sensitive data, applying a row access policy adds unnecessary overhead without security benefits.
    *   *Inefficient Scenario:* Applying a row access policy to a `product_catalog` table that only contains public product names and descriptions.

## Common Mistakes

1.  **Forgetting to Grant Privileges on the Policy:** Users must have `SELECT` privileges on the table and `APPLY ROW ACCESS POLICY` privilege on the policy itself (or `OWNERSHIP` on the table if they are applying it). If not, the policy won't be applied, or users won't be able to query the table.
2.  **Incorrect Policy Logic:** Errors in the `CASE` statement or `AND`/`OR` conditions within the policy can lead to incorrect data visibility (either showing too much or too little data). Thorough testing with various roles is essential.
3.  **Performance Impact:** Not optimizing the policy UDF. If the policy logic involves expensive operations (e.g., joins, subqueries, or calls to external functions) and is applied to very large tables, it can significantly degrade query performance.
4.  **Not Handling `NULL` Values in Policy Columns:** If the column passed to the policy can contain `NULL`s, the policy logic must explicitly handle these cases, otherwise, rows with `NULL`s might be unexpectedly filtered or included.

## Expected Output

The resulting dataset from a `SELECT * FROM sales_data` query will vary depending on the `CURRENT_ROLE()` of the user executing the query.

**Example 1: User with `ADMIN` role**
When the user has the `ADMIN` role, the policy returns `TRUE` for all rows.

| ORDER_ID | CUSTOMER_ID | PRODUCT_NAME | REGION | SALE_AMOUNT |
| :------- | :---------- | :----------- | :----- | :---------- |
| 1 | 101 | Laptop | EAST | 1200.00 |
| 2 | 102 | Mouse | WEST | 25.00 |
| 3 | 101 | Keyboard | EAST | 75.00 |
| 4 | 103 | Monitor | NORTH | 300.00 |
| 5 | 102 | Webcam | WEST | 50.00 |
| 6 | 104 | Printer | SOUTH | 150.00 |

*Explanation:* All columns are visible, and all rows are returned as the `ADMIN` role bypasses regional restrictions.

**Example 2: User with `SALES_ANALYST_EAST` role**
When the user has the `SALES_ANALYST_EAST` role, the policy returns `TRUE` only for rows where `region = 'EAST'`.

| ORDER_ID | CUSTOMER_ID | PRODUCT_NAME | REGION | SALE_AMOUNT |
| :------- | :---------- | :----------- | :----- | :---------- |
| 1 | 101 | Laptop | EAST | 1200.00 |

*Explanation:* Only rows pertaining to the 'EAST' region are displayed. Other rows are filtered out by the row access policy.
