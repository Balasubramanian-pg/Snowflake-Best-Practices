# Handling Policy Conflicts in Snowflake

In Snowflake, data governance features like Row Access Policies (RAPs) and Masking Policies (MPs) are crucial for enforcing data security and compliance. Policy conflicts arise when multiple policies apply to the same data, and understanding their precedence is vital to ensure data is protected as intended. Snowflake has a defined order of evaluation to resolve these conflicts.

**Short Explanation**

Policy conflicts in Snowflake occur when a piece of data is subject to more than one data governance policy, such as a row access policy and a masking policy, or even multiple masking policies. Snowflake resolves these by applying a specific order of precedence, ensuring that the most restrictive or appropriate policy is enforced.

- **What problem does this SQL feature solve?** It provides a clear, predictable mechanism for data protection when various security policies overlap, preventing ambiguous data access or masking scenarios.
- **Why is it important in databases or analytics?** It ensures consistent and reliable data governance, critical for compliance (e.g., GDPR, HIPAA) and preventing unauthorized data exposure, even in complex data environments.
- **Where is it commonly used in real workflows?** It's used in scenarios requiring fine-grained access control and dynamic data masking, such as multi-tenant applications, protecting PII (Personally Identifiable Information) in analytics datasets, or implementing role-based data visibility.

## Implementation Example

Let's imagine a scenario where we have a `CUSTOMER_DATA` table. We want to:
1. Limit rows visible based on the user's region (Row Access Policy).
2. Mask sensitive `EMAIL` and `PHONE_NUMBER` columns (Masking Policies).
3. Have a tag-based masking policy for `PII_DATA` which includes `EMAIL` but a direct column masking policy for `EMAIL` is also present.

```sql
-- 1. Create a Masking Policy for email
CREATE OR REPLACE MASKING POLICY email_mask_policy AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('ANALYST_ROLE') THEN val
    ELSE '***MASKED_EMAIL***'
  END;

-- 2. Create a Masking Policy for phone numbers
CREATE OR REPLACE MASKING POLICY phone_mask_policy AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('SUPPORT_ROLE') THEN val
    ELSE '***MASKED_PHONE***'
  END;

-- 3. Create a Row Access Policy to filter by region
CREATE OR REPLACE ROW ACCESS POLICY region_rap AS (region STRING) RETURNS BOOLEAN ->
  CASE
    WHEN CURRENT_ROLE() = 'US_ANALYST_ROLE' THEN region = 'US'
    WHEN CURRENT_ROLE() = 'EU_ANALYST_ROLE' THEN region = 'EU'
    ELSE FALSE
  END;

-- 4. Create the customer data table
CREATE OR REPLACE TABLE CUSTOMER_DATA (
    CUSTOMER_ID INT,
    CUSTOMER_NAME STRING,
    EMAIL STRING,
    PHONE_NUMBER STRING,
    REGION STRING,
    ORDER_AMOUNT DECIMAL(10, 2)
);

-- Insert sample data
INSERT INTO CUSTOMER_DATA VALUES
(1, 'Alice', 'alice@example.com', '111-222-3333', 'US', 150.75),
(2, 'Bob', 'bob@example.com', '444-555-6666', 'EU', 200.00),
(3, 'Charlie', 'charlie@example.com', '777-888-9999', 'US', 50.25),
(4, 'David', 'david@example.com', '000-111-2222', 'EU', 300.50);

-- 5. Apply the policies to the table and columns
ALTER TABLE CUSTOMER_DATA ADD ROW ACCESS POLICY region_rap ON (REGION);
ALTER TABLE CUSTOMER_DATA MODIFY COLUMN EMAIL SET MASKING POLICY email_mask_policy;
ALTER TABLE CUSTOMER_DATA MODIFY COLUMN PHONE_NUMBER SET MASKING POLICY phone_mask_policy;

-- 6. Example of Tag-based Masking Policy (higher precedence than direct column policy)
-- First, create a tag
CREATE TAG PII_DATA COMMENT = 'Tag for Personally Identifiable Information';
-- Assign the tag to the EMAIL column
ALTER TABLE CUSTOMER_DATA MODIFY COLUMN EMAIL SET TAG PII_DATA = 'email_pii';

-- Create a different masking policy for the PII_DATA tag
CREATE OR REPLACE MASKING POLICY tag_email_mask_policy AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('ADMIN_ROLE') THEN val
    ELSE '###TAG_MASKED_EMAIL###'
  END;

-- Apply the tag-based masking policy to the tag
ALTER TAG PII_DATA SET MASKING POLICY tag_email_mask_policy;

-- Querying as different roles to observe conflict resolution
-- Assume roles 'US_ANALYST_ROLE', 'EU_ANALYST_ROLE', 'ANALYST_ROLE', 'SUPPORT_ROLE', 'ADMIN_ROLE' exist.

-- As US_ANALYST_ROLE (sees US data, email masked by tag policy, phone by general MP)
-- SELECT * FROM CUSTOMER_DATA;

-- As EU_ANALYST_ROLE (sees EU data, email masked by tag policy, phone by general MP)
-- SELECT * FROM CUSTOMER_DATA;

-- As ANALYST_ROLE (no row access, email masked by tag policy, phone masked by general MP)
-- SELECT * FROM CUSTOMER_DATA;

-- As ADMIN_ROLE (no row access, email unmasked by tag policy, phone masked by general MP)
-- SELECT * FROM CUSTOMER_DATA;
```

## Explanation of the Code

This example sets up a table with a row access policy and multiple masking policies, demonstrating policy conflict resolution.

- `CREATE OR REPLACE MASKING POLICY email_mask_policy AS ...`: Defines a masking policy that hides email addresses unless the `CURRENT_ROLE()` is `ANALYST_ROLE`.
- `CREATE OR REPLACE MASKING POLICY phone_mask_policy AS ...`: Defines a masking policy that hides phone numbers unless the `CURRENT_ROLE()` is `SUPPORT_ROLE`.
- `CREATE OR REPLACE ROW ACCESS POLICY region_rap AS ...`: Defines a row access policy that filters rows based on the `REGION` column, allowing `US_ANALYST_ROLE` to see 'US' data and `EU_ANALYST_ROLE` to see 'EU' data.
- `CREATE OR REPLACE TABLE CUSTOMER_DATA ...`: Creates a sample table to apply the policies to.
- `INSERT INTO CUSTOMER_DATA ...`: Populates the table with sample data.
- `ALTER TABLE CUSTOMER_DATA ADD ROW ACCESS POLICY region_rap ON (REGION);`: Attaches the `region_rap` to the `CUSTOMER_DATA` table. This policy is evaluated first.
- `ALTER TABLE CUSTOMER_DATA MODIFY COLUMN EMAIL SET MASKING POLICY email_mask_policy;`: Directly assigns `email_mask_policy` to the `EMAIL` column.
- `ALTER TABLE CUSTOMER_DATA MODIFY COLUMN PHONE_NUMBER SET MASKING POLICY phone_mask_policy;`: Directly assigns `phone_mask_policy` to the `PHONE_NUMBER` column.
- `CREATE TAG PII_DATA ...`: Creates a tag named `PII_DATA`.
- `ALTER TABLE CUSTOMER_DATA MODIFY COLUMN EMAIL SET TAG PII_DATA = 'email_pii';`: Assigns the `PII_DATA` tag to the `EMAIL` column.
- `CREATE OR REPLACE MASKING POLICY tag_email_mask_policy AS ...`: Creates another masking policy specifically to be applied via the `PII_DATA` tag. This one unmasks for `ADMIN_ROLE`.
- `ALTER TAG PII_DATA SET MASKING POLICY tag_email_mask_policy;`: Attaches `tag_email_mask_policy` to the `PII_DATA` tag.

**How it changes the result set:**
- **Row Access Policy:** The `region_rap` always executes first. It filters the rows that a user can see based on their role and the `REGION` column. If a role does not match any condition, no rows are returned.
- **Masking Policy (Precedence):** When both a direct column masking policy (`email_mask_policy`) and a tag-based masking policy (`tag_email_mask_policy` via `PII_DATA` tag) are applied to the same column (`EMAIL`), the **direct column masking policy takes precedence**. So, `email_mask_policy` will be active for `EMAIL` despite the `PII_DATA` tag having its own masking policy. If `email_mask_policy` was not present, the `tag_email_mask_policy` would apply.
- The `phone_mask_policy` applies to `PHONE_NUMBER` independently.

**Snowflake's Policy Precedence:**
1. **Row Access Policy:** Always evaluated first to determine which rows are returned.
2. **Masking Policies:** If a column has both a direct masking policy and a tag-based masking policy, the **direct column masking policy takes precedence**. If only one is present, that one applies.

## When to Use

1.  **Multi-tenant Data Environments:** When different tenants (or departments within an organization) store their data in the same table, and each should only see their own rows.
    *   *Scenario:* A `SALES_DATA` table containing sales records from various regions, where analysts from 'NORTH' can only see 'NORTH' region sales, and 'SOUTH' analysts see 'SOUTH'. A Row Access Policy would be ideal here.
2.  **Protecting Sensitive Data based on Roles:** Dynamically mask sensitive information (e.g., PII, financial data) based on the user's role or other context.
    *   *Scenario:* An `EMPLOYEE_HR` table where HR managers can see unmasked salaries, but regular employees or basic reporting roles see salaries masked (e.g., `******`). Masking Policies are used for this.
3.  **Compliance and Regulatory Requirements:** Adhering to data privacy regulations (e.g., GDPR, HIPAA, CCPA) by ensuring specific data types are never exposed to unauthorized users.
    *   *Scenario:* Medical records where patient names or addresses are masked for researchers, but visible to doctors or authorized personnel. A combination of Row Access and Masking Policies ensures stringent control.

## When Not to Use

1.  **Performance Critical Workloads with Complex RAPs:** While powerful, highly complex Row Access Policies on very large tables (billions of rows) without proper clustering can introduce performance overhead as Snowflake evaluates the policy for each row.
    *   *Situation:* Applying a RAP that joins to multiple large mapping tables or uses non-performant functions on a non-clustered, massive fact table frequently accessed by high-volume queries.
2.  **Simple Table-Level Access Control:** If access is simply "all or nothing" for an entire table, standard `GRANT`/`REVOKE` permissions on the table are more straightforward and efficient.
    *   *Situation:* Granting `SELECT` on `PUBLIC.MY_TABLE` to `FINANCE_ROLE`. Using a RAP here would be overkill and add unnecessary complexity.
3.  **Overlapping and Conflicting Business Logic in Policies:** If policies are designed poorly with conflicting business rules that lead to unpredictable outcomes or unexpected data visibility/masking.
    *   *Situation:* A masking policy for emails allows `ROLE_A` to see unmasked data, but another masking policy for the same email column (applied via a tag) masks it for `ROLE_A`. While Snowflake has precedence rules, such ambiguity in design can indicate a flawed security architecture.

## Common Mistakes

1.  **Ignoring Policy Precedence:** Developers might assume policies apply in a certain order or combine intuitively, leading to unintended data exposure or over-masking if Snowflake's defined precedence (RAP first, then MP, with direct MP over tag-based MP) is not understood.
2.  **Creating Overly Complex Policies:** Policies that involve many conditions, subqueries, or joins can negatively impact query performance, especially RAPs on large datasets.
3.  **Lack of Testing with Different Roles:** Not thoroughly testing data visibility and masking for every relevant role can lead to security gaps or users being unable to access necessary data.
4.  **Forgetting to Apply Policies:** Creating policies is only half the battle; they must be actively applied to tables/columns/tags using `ALTER TABLE` or `ALTER TAG` commands to take effect.
5.  **Hardcoding Values in Policies:** Using fixed values instead of context functions (`CURRENT_ROLE()`, `CURRENT_USER()`, `SESSION_CONTEXT()`) makes policies inflexible and hard to manage across different environments or for different users.

## Expected Output

When a user queries `SELECT * FROM CUSTOMER_DATA;`, the resulting dataset will first be filtered by the Row Access Policy, and then the columns `EMAIL` and `PHONE_NUMBER` will be masked based on the applicable Masking Policies and the user's `CURRENT_ROLE()`.

**Example Output (as 'US_ANALYST_ROLE'):**

| CUSTOMER_ID | CUSTOMER_NAME | EMAIL                 | PHONE_NUMBER      | REGION | ORDER_AMOUNT |
| :---------- | :------------ | :-------------------- | :---------------- | :----- | :----------- |
| 1           | Alice         | ***MASKED_EMAIL***    | ***MASKED_PHONE*** | US     | 150.75       |
| 3           | Charlie       | ***MASKED_EMAIL***    | ***MASKED_PHONE*** | US     | 50.25        |

**Explanation of Columns:**
-   **CUSTOMER_ID, CUSTOMER_NAME, REGION, ORDER_AMOUNT:** These columns are not subject to masking policies in this example, so their original values are shown.
-   **EMAIL:** Masked by `email_mask_policy` because `US_ANALYST_ROLE` is not `ANALYST_ROLE`. (The direct column masking policy `email_mask_policy` takes precedence over `tag_email_mask_policy`).
-   **PHONE_NUMBER:** Masked by `phone_mask_policy` because `US_ANALYST_ROLE` is not `SUPPORT_ROLE`.

Only rows where `REGION = 'US'` are returned due to `region_rap` evaluating first.
