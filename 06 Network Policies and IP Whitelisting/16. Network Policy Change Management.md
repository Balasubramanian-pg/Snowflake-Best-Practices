# Network Policy Change Management in Snowflake

Network Policy Change Management in Snowflake refers to the systematic process of creating, modifying, and applying network policies to control inbound network access to your Snowflake account. These policies act like a firewall, specifying which IP addresses or ranges are allowed or blocked from connecting to Snowflake, thus enhancing security by limiting potential attack vectors. The introduction of Network Rules has significantly streamlined this process, allowing for more modular and reusable definitions of IP lists.

**Short Explanation**

Network policies restrict access to your Snowflake account based on IP addresses, ensuring only authorized networks can connect. This feature solves the problem of unauthorized network access, making your data more secure by limiting connection points. It's crucial in databases for preventing breaches and is commonly used in enterprise security architectures to enforce strict access controls.

- **What problem does this SQL feature solve?** It solves the problem of unauthorized access to Snowflake accounts by restricting connections to a defined set of IP addresses.
- **Why is it important in databases or analytics?** It's vital for maintaining data security, compliance with regulations, and preventing data breaches by controlling the network perimeter.
- **Where is it commonly used in real workflows?** Enterprises use it to restrict access to their corporate networks, VPNs, or specific cloud environments, ensuring that only trusted sources can connect to Snowflake.

## Implementation Example

This example demonstrates how to create a Network Rule, create a Network Policy that uses it, and then apply that policy at the account level.

```sql
-- Step 1: Create a Network Rule to define allowed IPs
CREATE OR REPLACE NETWORK RULE corporate_office_ips_rule
    TYPE = IPV4
    MODE = INGRESS
    VALUE_LIST = ('203.0.113.0/24', '198.51.100.1/32')
    COMMENT = 'Allowed IP ranges for corporate offices';

-- Step 2: Create a Network Policy referencing the Network Rule
CREATE OR REPLACE NETWORK POLICY corporate_access_policy
    ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips_rule')
    COMMENT = 'Network policy to allow access from corporate office IPs';

-- Step 3: Apply the Network Policy to the account
ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;

-- Step 4: Verify the active network policy (optional)
SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;

-- Step 5: Modify the Network Rule (example of change management)
ALTER NETWORK RULE corporate_office_ips_rule
    SET VALUE_LIST = ('203.0.113.0/24', '198.51.100.1/32', '203.0.113.128/25')
    COMMENT = 'Updated allowed IP ranges including a new subnet';
```

## Explanation of the Code

- `CREATE OR REPLACE NETWORK RULE corporate_office_ips_rule...`:
    - **What it does:** This statement defines a reusable network rule named `corporate_office_ips_rule`. Network rules are schema-level objects that group network identifiers (like IP addresses).
    - **How it changes the result set:** It creates a new database object (a network rule) that can be referenced by network policies. It does not directly affect user access until applied through a policy.
    - `TYPE = IPV4`: Specifies that the identifiers are IPv4 addresses.
    - `MODE = INGRESS`: Indicates that this rule is for incoming traffic to Snowflake.
    - `VALUE_LIST = (...)`: Defines the specific IP addresses or CIDR blocks that this rule encompasses.
    - `COMMENT =...`: Adds a descriptive comment to the rule.

- `CREATE OR REPLACE NETWORK POLICY corporate_access_policy...`:
    - **What it does:** This statement creates a network policy named `corporate_access_policy`. Network policies enforce access restrictions based on the rules they contain.
    - **How it changes the result set:** It creates a new security object. It does not enforce access until associated with an account, user, or security integration.
    - `ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips_rule')`: This is the crucial part for change management. Instead of listing IPs directly, it references the previously defined network rule. This makes it easier to update IP lists without altering the policy itself.

- `ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;`:
    - **What it does:** This statement activates the `corporate_access_policy` at the account level. Once active, all connections to this Snowflake account must originate from the IP addresses specified in the `corporate_office_ips_rule` referenced by `corporate_access_policy`.
    - **How it changes the result set:** It modifies the account parameter `NETWORK_POLICY`, thereby enforcing the access restrictions for all users and integrations connecting to the account.

- `SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;`:
    - **What it does:** This command is used to verify which network policy is currently active for the Snowflake account.
    - **How it changes the result set:** It returns metadata about the `NETWORK_POLICY` parameter, including its current value (the name of the active policy).

- `ALTER NETWORK RULE corporate_office_ips_rule SET VALUE_LIST = (...);`:
    - **What it does:** This is a key aspect of change management. It modifies the `VALUE_LIST` of an existing network rule.
    - **How it changes the result set:** Crucially, any network policies that reference `corporate_office_ips_rule` automatically inherit this change. There's no need to modify the network policy itself. This simplifies updates and reduces the risk of errors.

## When to Use

1. **Restricting Access to Corporate Networks:** To ensure only employees connected to the company's internal network or VPN can access Snowflake.
    * **Scenario:** A company wants to prevent external access to its sensitive data in Snowflake. They create a network rule for their corporate IP ranges and apply a policy referencing it to the entire Snowflake account.
2. **Granular User-Specific Access:** When certain users require access from different locations than the general account policy.
    * **Scenario:** A data analyst frequently works remotely. An account-level policy restricts access to the corporate VPN. A separate, more permissive network policy (or rule) is applied specifically to the analyst's user account, allowing access from their home office IP. User-level policies override account-level policies.[AELq]
3. **Managing IP Changes with Ease:** When IP addresses for allowed access points (e.g., VPN gateways) change frequently.
    * **Scenario:** A company's VPN provider updates its IP address ranges. Instead of modifying multiple network policies, the administrator only needs to update the single network rule that defines the VPN IPs, and all associated policies will reflect the change automatically.

## When Not to Use

1. **To control outbound traffic from Snowflake:** Network policies and rules control *inbound* access to Snowflake. For outbound traffic (e.g., from UDFs to external APIs), you need to use external network access features with `EGRESS` mode network rules.
2. **As the sole security measure:** Relying only on network policies is insufficient. They are perimeter defenses. Strong authentication (MFA), role-based access control (RBAC), and data encryption are also critical.
3. **For extremely dynamic IP environments without automation:** While network rules improve change management, manually updating them for very frequently changing IPs (e.g., mobile devices with dynamic public IPs) can still be cumbersome and prone to errors without an automated process.

## Common Mistakes

1. **Forgetting to apply the policy:** Creating network rules and policies without actually activating them at the account, user, or security integration level means they will have no effect.
2. **Locking yourself out:** Implementing a restrictive network policy without including your current IP address (or a safe bypass mechanism) can prevent you from accessing your own Snowflake account. Snowflake provides a `MINS_TO_BYPASS_NETWORK_POLICY` setting, but it requires Snowflake support to activate for an emergency.
3. **Not testing thoroughly:** Deploying a network policy without comprehensive testing from all expected access points can lead to unexpected access denials for critical users or applications.
4. **Mixing old and new methods:** Using `ALLOWED_IP_LIST`/`BLOCKED_IP_LIST` directly in a policy alongside `ALLOWED_NETWORK_RULE_LIST` can lead to confusion and is generally not recommended, especially for new policies. Always prefer network rules.

## Expected Output

After successfully creating the network rule and policy and applying it to the account, `SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;` would return a result set indicating the active policy.

| key | value | default | level | description |
| :------------- | :-------------------- | :------------------ | :------ | :-------------------------------------------- |
| NETWORK_POLICY | `corporate_access_policy` | NULL | ACCOUNT | Network policy controlling client IP addresses |

**Explanation of Columns:**

- **key**: The name of the parameter being displayed, in this case, `NETWORK_POLICY`.
- **value**: The current value of the parameter, which is the name of the network policy applied to the account (`corporate_access_policy`).
- **default**: The default value for the parameter, which is `NULL` for `NETWORK_POLICY`, meaning no policy is applied by default.
- **level**: Indicates at what level the parameter is set (e.g., `ACCOUNT`, `USER`, `SESSION`).
- **description**: A brief explanation of what the parameter controls.

This output confirms that the `corporate_access_policy` is active at the account level, enforcing the network restrictions defined within it.
