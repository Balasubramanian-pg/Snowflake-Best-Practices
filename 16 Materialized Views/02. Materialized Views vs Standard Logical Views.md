# Materialized Views vs Standard Logical Views

> [!NOTE]
> Materialized views and standard logical views are two types of database objects used to simplify data retrieval and improve query performance. They exist to provide a layer of abstraction between the user and the underlying data, making it easier to manage complex queries and data transformations.

**Short Explanation**

Materialized views are pre-computed views that store the result of a query in a physical table, while standard logical views are virtual tables that do not store data and instead rely on the underlying query to retrieve the data. Materialized views are useful for improving query performance by reducing the computational overhead of complex queries, while standard logical views are useful for simplifying complex queries and making them easier to maintain.

### Example SQL Code

```sql
-- Create a materialized view
CREATE MATERIALIZED VIEW sales_data AS
SELECT order_date, customer_name, order_total
FROM orders
WHERE order_total > 1000;

-- Create a standard logical view
CREATE VIEW sales_data AS
SELECT order_date, customer_name, order_total
FROM orders
WHERE order_total > 1000;
```

### Explanation of the Code

#### Materialized View

* The `CREATE MATERIALIZED VIEW` statement creates a new materialized view called `sales_data`.
* The `SELECT` statement defines the columns to be included in the view.
* The `FROM` statement specifies the underlying table (`orders`) that contains the data.
* The `WHERE` clause filters the data to include only orders with a total value greater than 1000.

#### Standard Logical View

* The `CREATE VIEW` statement creates a new standard logical view called `sales_data`.
* The `SELECT` statement defines the columns to be included in the view.
* The `FROM` statement specifies the underlying table (`orders`) that contains the data.
* The `WHERE` clause filters the data to include only orders with a total value greater than 1000.

### When to Use

> [!TIP]
> Use materialized views when you need to improve query performance by reducing the computational overhead of complex queries. Use standard logical views when you need to simplify complex queries and make them easier to maintain.

Typical use cases:

* Creating aggregated metrics for dashboards
* Filtering datasets for analysis
* Joining datasets to enrich information

Example scenarios:

* Creating a materialized view to store the result of a complex query that is frequently executed
* Creating a standard logical view to simplify a complex query and make it easier to maintain

### When Not to Use

> [!WARNING]
> Avoid using materialized views when the underlying data is frequently updated or changed, as this can lead to stale data in the materialized view. Avoid using standard logical views when the query performance is critical, as they can introduce additional computational overhead.

Examples:

* When row level detail must remain unchanged
* When a different SQL feature performs the task more efficiently
* When the transformation should occur earlier in the data pipeline

### Common Mistakes

> [!IMPORTANT]
> These errors frequently appear when using materialized views and standard logical views.

Examples:

* Forgetting to update the materialized view when the underlying data changes
* Referencing columns that do not exist in the view
* Mixing aggregate and non-aggregate columns incorrectly
* Confusing similar SQL constructs

### Expected Output

The resulting dataset from a materialized view should be a physical table that stores the pre-computed result of the query. The resulting dataset from a standard logical view should be a virtual table that relies on the underlying query to retrieve the data.
