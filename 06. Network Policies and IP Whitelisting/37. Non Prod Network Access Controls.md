# Non-Prod Network Access Controls in Snowflake

Network Access Controls in Snowflake, specifically for non-production environments, refer to the practice of restricting network connectivity to your Snowflake instance. This ensures that only authorized IP addresses can access your development, testing, or staging environments, adding a crucial layer of security. It's like having a bouncer at the door, but for your data warehouse.

**Short Explanation**

This SQL feature lets you define specific IP addresses or IP ranges that are allowed to connect to your Snowflake account or specific users. It solves the problem of unauthorized network access by preventing connection attempts from untrusted locations, which is vital for protecting sensitive data, even in non-production settings. It's important for maintaining data security and compliance, ensuring that only known and trusted sources, like your corporate VPN or specific client IPs, can initiate connections. You'll often see this used by security administrators to lock down access to different environments within an organization.

## Implementation Example

```sql
-- 1. Create a Network Rule for non-production IP addresses
CREATE NETWORK RULE non_prod_allowed_ips
    TYPE = IPV4
    VALUE_LIST = ('192.168.10.0/24', '172.16.5.10')
    MODE = INGRESS
    COMMENT = 'Allowed IP addresses for non-production environments';

-- 2. Create a Network Policy referencing the Network Rule
CREATE NETWORK POLICY non_prod_network_policy
    ALLOWED_NETWORK_RULE_LIST = ('non_prod_allowed_ips')
    COMMENT = 'Network policy for non-production environments';

-- 3. Apply the Network Policy to a specific non-production user
ALTER USER non_prod_analyst SET NETWORK_POLICY = non_prod_network_policy;

-- (Optional) Apply to the account for all non-prod environments if all should be restricted
-- ALTER ACCOUNT SET NETWORK_POLICY = non_prod_network_policy;
```

## Explanation of the Code

This example demonstrates how to set up network access controls in Snowflake using network rules and policies.

* **`CREATE NETWORK RULE non_prod_allowed_ips...`**:
    * **What it does**: This statement creates a new schema-level object called a "network rule". Network rules group network identifiers (like IP addresses) into logical units.
    * **How it changes the result set**: This is a DDL (Data Definition Language) statement that creates a metadata object in your Snowflake account. It does not produce a query result set.
    * **`TYPE = IPV4`**: Specifies that the rule will contain IPv4 addresses.
    * **`VALUE_LIST = ('192.168.10.0/24', '172.16.5.10')`**: Defines the specific IP addresses or CIDR blocks that are part of this rule. Only connections originating from these IPs will be considered "allowed" by policies referencing this rule.
    * **`MODE = INGRESS`**: Indicates that this rule is for incoming requests to Snowflake.

* **`CREATE NETWORK POLICY non_prod_network_policy...`**:
    * **What it does**: This statement creates a network policy that references the previously defined network rule. A network policy uses network rules to control inbound network traffic to the Snowflake service.
    * **How it changes the result set**: This is a DDL statement that creates a metadata object. It does not produce a query result set.
    * **`ALLOWED_NETWORK_RULE_LIST = ('non_prod_allowed_ips')`**: This is the core of the policy, specifying that only connections matching the `non_prod_allowed_ips` network rule are permitted.

* **`ALTER USER non_prod_analyst SET NETWORK_POLICY = non_prod_network_policy;`**:
    * **What it does**: This statement applies the `non_prod_network_policy` to a specific user named `non_prod_analyst`. User-level network policies take precedence over account-level policies for that specific user.
    * **How it changes the result set**: This is a DDL statement that modifies user-level settings; it does not operate on a result set.

## When to Use

1. **Restricting Access to Development/Test Environments**: When you want to ensure that only developers or testers from specific corporate networks or VPNs can access non-production Snowflake instances.
    * **Example Scenario**: A company has a dedicated team working on a new feature in a `DEV` environment. They want to ensure only their office IPs and secure VPN access points can connect to this environment.
2. **Securing Data Pipelines and ETL Tools**: When data ingestion or transformation tools running on specific servers need access to Snowflake, and you want to restrict that access to only their known egress IP addresses.
    * **Example Scenario**: An ETL tool hosted in a cloud environment has a static set of egress IP addresses. These IPs are whitelisted to push data into a `STAGING` Snowflake database, blocking any other external connections.
3. **Ensuring Compliance for Sensitive Non-Prod Data**: Even if it's not production data, sometimes non-production environments contain sanitized but still sensitive data. Network policies help meet compliance requirements by limiting potential access points.
    * **Example Scenario**: A `UAT` (User Acceptance Testing) environment contains anonymized customer data for testing. Regulatory compliance dictates strict access controls, so network policies are used to limit access to only the UAT team's VPN.

## When Not to Use

1. **Highly Dynamic IP Environments**: When users frequently connect from locations with constantly changing public IP addresses (e.g., mobile devices, home networks without VPN).
    * **Reason**: Whitelisting individual dynamic IPs is impractical and can lead to frequent access issues, frustrating users.
2. **Public-Facing Applications**: If you are building a public application that needs to connect to Snowflake (e.g., a customer-facing dashboard directly querying Snowflake).
    * **Reason**: Network policies are designed for controlled, known access points, not for open internet access. In such cases, other authentication/authorization mechanisms (like secure APIs or intermediate services) are more appropriate.
3. **When Granular Object-Level Security is Sufficient**: If your primary concern is controlling *what* data users can see within Snowflake (e.g., row-level or column-level security) rather than *where* they connect from.
    * **Reason**: While network policies are about perimeter defense, they don't replace in-database access controls. If your users are already inside your network, network policies won't prevent them from accessing unauthorized data if other Snowflake roles/privileges are not correctly set up.

## Common Mistakes

1. **Forgetting to Activate Policy**: Creating a network policy but forgetting to apply it to the account or specific users, leaving the environment vulnerable.
2. **Incorrect IP/CIDR Syntax**: Mismatched or incorrectly formatted IP addresses or CIDR blocks, leading to legitimate users being blocked or unauthorized access being granted.
3. **No Emergency Break-Glass Access**: Not having a plan or method to access the account if all allowed IPs become unavailable (e.g., VPN issues or misconfigurations). It's crucial to have at least one highly secured, restricted IP that can always access it for administrative purposes.
4. **Overly Broad IP Ranges**: Using very broad CIDR blocks (e.g., `0.0.0.0/0` in an allowed list) defeats the purpose of granular network security by effectively allowing connections from anywhere.
5. **Conflicting Policies**: Applying a network policy at the account level and then applying a different, potentially conflicting, policy at the user level without understanding that user-level policies override account-level policies.

## Expected Output

The concept of "Non-Prod Network Access Controls" does not directly produce a data set from a SQL query in the traditional sense. Instead, the `CREATE NETWORK RULE`, `CREATE NETWORK POLICY`, and `ALTER USER`/`ALTER ACCOUNT` commands are DDL operations that modify the security configuration of your Snowflake environment.

If you were to query metadata views about the applied network policies (e.g., `SHOW NETWORK POLICIES` or `SHOW NETWORK RULES`), you might see a record reflecting the configuration.

Here's an example of what metadata you might retrieve if you queried for network policies after applying the example:

| NAME | TYPE | VALUE | COMMENT |
| :-------------------------- | :------ | :------------------------------------------------------------------ | :------------------------------------------ |
| NON_PROD_NETWORK_POLICY | NETWORK | {'ALLOWED_NETWORK_RULE_LIST': ['NON_PROD_ALLOWED_IPS']} | Network policy for non-production environments |
| NON_PROD_ALLOWED_IPS | NETWORK\_RULE | {'TYPE': 'IPV4', 'VALUE_LIST': ['192.168.10.0/24', '172.16.5.10'], 'MODE': 'INGRESS'} | Allowed IP addresses for non-production environments |

* **NAME**: The unique identifier of the network policy or network rule.
* **TYPE**: Indicates whether it's a `NETWORK` policy or a `NETWORK_RULE`.
* **VALUE**: Contains the detailed configuration, such as the list of allowed network rules for a policy, or the IP addresses/type/mode for a rule.
* **COMMENT**: A descriptive note about the policy or rule.

When the network policy is active, any attempt to connect to Snowflake from an IP address *not* in the defined `VALUE_LIST` (or referenced by the `ALLOWED_NETWORK_RULE_LIST`) would result in a connection error, effectively blocking access.
