# Restricting Access by Business Unit in Snowflake

Restricting access by business unit in Snowflake involves implementing security measures to ensure that users can only view or modify data relevant to their specific business unit. This is crucial for data governance, compliance, and preventing unauthorized data exposure across different departments within an organization. It leverages Snowflake's robust security features like roles, access control, and secure views or row-access policies to achieve fine-grained control over data visibility.

**Short Explanation**

This concept focuses on segmenting data access based on a user's organizational unit, ensuring data privacy and adherence to internal policies. It solves the problem of broad data access by limiting users to only the information directly pertinent to their department or business function. This is important for maintaining data security, complying with regulations like GDPR or HIPAA, and ensuring data integrity by preventing accidental or malicious cross-departmental data exposure. It's commonly used in large organizations with multiple business units, such as financial institutions, healthcare providers, or global corporations, where data segregation is paramount.

## Implementation Example

```sql
-- 1. Create roles for different business units
CREATE ROLE FINANCE_ANALYST;
CREATE ROLE HR_MANAGER;
CREATE ROLE SALES_REP;

-- 2. Grant necessary privileges to base roles (e.g., read access to a base table)
GRANT USAGE ON DATABASE MY_DB TO ROLE FINANCE_ANALYST, HR_MANAGER, SALES_REP;
GRANT USAGE ON SCHEMA MY_DB.APP_DATA TO ROLE FINANCE_ANALYST, HR_MANAGER, SALES_REP;
GRANT SELECT ON TABLE MY_DB.APP_DATA.EMPLOYEE_DATA TO ROLE FINANCE_ANALYST, HR_MANAGER, SALES_REP;

-- 3. Create a mapping table for user-to-business-unit and a business unit-to-data mapping
CREATE TABLE MY_DB.SECURITY.USER_BUSINESS_UNIT_MAPPING (
    USER_NAME VARCHAR,
    BUSINESS_UNIT VARCHAR
);

CREATE TABLE MY_DB.SECURITY.BUSINESS_UNIT_DATA_MAPPING (
    BUSINESS_UNIT VARCHAR,
    DATA_SCOPE VARCHAR -- e.g., 'Finance', 'HR', 'Sales'
);

INSERT INTO MY_DB.SECURITY.USER_BUSINESS_UNIT_MAPPING VALUES
('JOHN_DOE', 'Finance'),
('JANE_SMITH', 'HR'),
('BOB_JOHNSON', 'Sales');

INSERT INTO MY_DB.SECURITY.BUSINESS_UNIT_DATA_MAPPING VALUES
('Finance', 'Finance'),
('HR', 'HR'),
('Sales', 'Sales');

-- 4. Implement a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY MY_DB.SECURITY.EMPLOYEE_DATA_ROW_ACCESS_POLICY
AS
(EMPLOYEE_BUSINESS_UNIT VARCHAR) RETURNS BOOLEAN ->
  CURRENT_ROLE() IN ('ACCOUNTADMIN') OR
  EXISTS (
    SELECT 1 FROM MY_DB.SECURITY.USER_BUSINESS_UNIT_MAPPING ubum
    JOIN MY_DB.SECURITY.BUSINESS_UNIT_DATA_MAPPING bdwm ON ubum.BUSINESS_UNIT = bdwm.BUSINESS_UNIT
    WHERE ubum.USER_NAME = CURRENT_USER()
      AND bdwm.DATA_SCOPE = EMPLOYEE_BUSINESS_UNIT
  );

-- 5. Apply the Row Access Policy to the table
ALTER TABLE MY_DB.APP_DATA.EMPLOYEE_DATA ADD ROW ACCESS POLICY MY_DB.SECURITY.EMPLOYEE_DATA_ROW_ACCESS_POLICY ON (BUSINESS_UNIT);

-- 6. Grant roles to users (assuming users are already created)
GRANT ROLE FINANCE_ANALYST TO USER JOHN_DOE;
GRANT ROLE HR_MANAGER TO USER JANE_SMITH;
GRANT ROLE SALES_REP TO USER BOB_JOHNSON;

-- 7. Test (as JOHN_DOE, assuming their current role is FINANCE_ANALYST)
-- SELECT * FROM MY_DB.APP_DATA.EMPLOYEE_DATA;
```

## Explanation of the Code

This implementation uses Snowflake's Row Access Policies to dynamically filter rows based on the current user's business unit.

*   **`CREATE ROLE`**: This statement defines distinct roles (`FINANCE_ANALYST`, `HR_MANAGER`, `SALES_REP`) for each business unit.
    *   **What it does:** Creates named entities that can be granted privileges and assigned to users.
    *   **How it changes the result set:** It doesn't directly change a result set but establishes the security context for users.
*   **`GRANT USAGE ON DATABASE/SCHEMA`, `GRANT SELECT ON TABLE`**: These statements provide fundamental access to the database, schema, and the base `EMPLOYEE_DATA` table for all business unit roles.
    *   **What it does:** Assigns permissions to roles, allowing them to interact with specified database objects.
    *   **How it changes the result set:** This gives broad `SELECT` access, which will then be refined by the row access policy.
*   **`CREATE TABLE MY_DB.SECURITY.USER_BUSINESS_UNIT_MAPPING`, `CREATE TABLE MY_DB.SECURITY.BUSINESS_UNIT_DATA_MAPPING`**: These create auxiliary tables. `USER_BUSINESS_UNIT_MAPPING` links Snowflake users to their business units, and `BUSINESS_UNIT_DATA_MAPPING` defines what data scopes (e.g., 'Finance', 'HR') each business unit role is allowed to see.
    *   **What it does:** Stores metadata essential for the row access policy to function.
    *   **How it changes the result set:** These tables do not directly alter data accessible to users, but their contents are critical for the policy's logic.
*   **`INSERT INTO ... VALUES`**: Populates the mapping tables with example data.
    *   **What it does:** Adds records to the mapping tables.
    *   **How it changes the result set:** Provides the lookup data for the row access policy.
*   **`CREATE OR REPLACE ROW ACCESS POLICY MY_DB.SECURITY.EMPLOYEE_DATA_ROW_ACCESS_POLICY ...`**: This is the core of the dynamic access control. The policy defines a boolean function (`RETURNS BOOLEAN`) that, for each row in the `EMPLOYEE_DATA` table, determines if the current user can see that row. It checks if the current role is `ACCOUNTADMIN` (allowing full access) or if there's a match between the `CURRENT_USER()`'s mapped business unit and the `EMPLOYEE_BUSINESS_UNIT` of the row being evaluated.
    *   **What it does:** Filters rows from a table based on a dynamic condition evaluated at query runtime, considering the `CURRENT_USER()` and their associated roles/business units.
    *   **How it changes the result set:** This significantly restricts the number of rows returned to a user, showing only those rows where the `EMPLOYEE_BUSINESS_UNIT` matches the user's authorized business units.
*   **`ALTER TABLE MY_DB.APP_DATA.EMPLOYEE_DATA ADD ROW ACCESS POLICY ...`**: Attaches the defined row access policy to the `EMPLOYEE_DATA` table, specifying which column (`BUSINESS_UNIT`) in the main table maps to the policy's input parameter.
    *   **What it does:** Activates the row access policy on the specified table.
    *   **How it changes the result set:** Ensures that all queries on `MY_DB.APP_DATA.EMPLOYEE_DATA` automatically apply the row-level filtering defined by the policy.
*   **`GRANT ROLE ... TO USER ...`**: Assigns the previously created roles to individual users.
    *   **What it does:** Links users to the permissions defined by the roles.
    *   **How it changes the result set:** It enables users to assume specific business unit roles, which in turn influences what data they can see through the row access policy.
*   **`SELECT * FROM MY_DB.APP_DATA.EMPLOYEE_DATA` (as a specific user)**: This simulates a user querying the data.
    *   **What it does:** Retrieves data from the `EMPLOYEE_DATA` table.
    *   **How it changes the result set:** The row access policy will implicitly add a `WHERE` clause based on the user's business unit, returning only relevant rows.

## When to Use

1.  **Compliance with Regulations:** When dealing with sensitive data (e.g., HR records, financial transactions, patient health information) that must be strictly segregated by department or legal entity to comply with regulations like GDPR, HIPAA, or SOX.
    *   **Scenario:** A healthcare organization needs to ensure that only HR personnel can see employee salary details, while medical staff can only access patient treatment records within their specific clinic.
2.  **Multi-Tenant Architectures:** In a data platform serving multiple internal business units or external clients, where each unit/client should only see their own data, even if it resides in the same physical table.
    *   **Scenario:** A SaaS analytics platform hosts data for multiple client companies. Each client's users should only be able to query their own company's data within shared tables.
3.  **Internal Data Segmentation:** For large enterprises with distinct functional departments (Finance, HR, Sales, Marketing) that require data separation to prevent accidental data leaks or unauthorized access between teams.
    *   **Scenario:** A global retail company wants to ensure that sales managers in Europe can only view sales data for European regions, and finance analysts can only access financial records relevant to their budget.

## When Not to Use

1.  **Small, Homogeneous Datasets:** For small organizations or datasets where all users legitimately need access to all data, implementing row access policies adds unnecessary complexity without significant security benefits.
    *   **Situation:** A startup with a single team where everyone shares responsibilities and needs a holistic view of all operational data.
2.  **Performance Critical Workloads with Very Complex Policies:** While Snowflake's row access policies are optimized, an extremely complex policy with multiple joins to large mapping tables might introduce minor performance overhead for very high-volume, performance-sensitive queries.
    *   **Situation:** A real-time dashboard requiring sub-second response times on billions of rows, and the policy involves multiple subqueries or complex logic that could be simplified by data-level separation if possible.
3.  **Static, Coarse-Grained Access Control is Sufficient:** If access control can be adequately managed at a table or schema level (e.g., granting `SELECT` on an entire table to specific roles) and no row-level filtering is required.
    *   **Situation:** One table contains only "public" product information accessible to everyone, and another table contains "confidential" customer data accessible only to a specific customer service role.

## Common Mistakes

1.  **Not Testing with Different Users/Roles:** Developers might test policies only with `ACCOUNTADMIN` or their own elevated role, leading to a false sense of security where the policy isn't actually restricting access for regular users.
2.  **Incorrectly Mapping Users to Business Units:** Errors in the `USER_BUSINESS_UNIT_MAPPING` or `BUSINESS_UNIT_DATA_MAPPING` tables can result in users seeing too much data (security risk) or too little data (hindering productivity).
3.  **Overlooking `ACCOUNTADMIN` Bypass:** Forgetting that `ACCOUNTADMIN` (and roles with `MANAGE GRANTS`) bypasses all row access policies, which is a feature but also a potential oversight if not understood.
4.  **Performance Impact with Suboptimal Policy Logic:** Writing inefficient policy functions (e.g., using `JOIN` conditions that don't leverage proper indexing or involve full table scans within the policy) can slow down queries significantly.
5.  **Forgetting to Grant Privileges on Mapping Tables:** The role executing the query needs `SELECT` privileges on the mapping tables used within the row access policy, otherwise the policy evaluation will fail.

## Expected Output

The expected output when a user (e.g., `JOHN_DOE` who is a `FINANCE_ANALYST`) queries the `EMPLOYEE_DATA` table will be a subset of the original table, containing only those rows where the `BUSINESS_UNIT` column matches 'Finance'. Other rows, even if present in the base table, will be hidden from this user.

**Example Table (what JOHN_DOE would see):**

| EMPLOYEE_ID | EMPLOYEE_NAME | SALARY | BUSINESS_UNIT | HIRE_DATE |
| :---------- | :------------ | :----- | :------------ | :-------- |
| 101         | Alice         | 75000  | Finance       | 2022-01-15 |
| 105         | Charlie       | 90000  | Finance       | 2021-07-20 |

*   **EMPLOYEE\_ID**: Unique identifier for the employee.
*   **EMPLOYEE\_NAME**: Name of the employee.
*   **SALARY**: Employee's salary (visible to Finance).
*   **BUSINESS\_UNIT**: The business unit the employee belongs to. For `JOHN_DOE`, this would only show 'Finance' related rows.
*   **HIRE\_DATE**: The date the employee was hired.
