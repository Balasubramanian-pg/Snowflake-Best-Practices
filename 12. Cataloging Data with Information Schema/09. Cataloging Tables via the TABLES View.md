# Cataloging Tables via the TABLES View in Snowflake

**Description:**
The TABLES view in Snowflake's Information Schema provides a comprehensive catalog of tables in a database, allowing users to retrieve metadata about tables, such as table names, schema names, and table types. This enables efficient management and discovery of tables in Snowflake. By querying the TABLES view, users can filter and retrieve specific table information.

## Short Explanation
**Overview:** The TABLES view provides a centralized catalog for tables in Snowflake.

**Problem Solved:** Manually tracking and managing table metadata across multiple databases and schemas.

**Importance:** Essential for data governance, data discovery, and efficient data management in Snowflake.

**Use Cases:**
- Data cataloging
- Table discovery
- Metadata management

### Typical Scenario
A data engineer wants to retrieve a list of all tables in a specific schema, along with their table types and creation dates. They can query the TABLES view in Snowflake to achieve this.

### Core Snowflake SQL Objects Used
`INFORMATION_SCHEMA.TABLES`

### Code Source Execution
```sql
SELECT * FROM INFORMATION_SCHEMA.TABLES WHERE TABLE_SCHEMA = 'MY_SCHEMA';
```

### Example Execution Logic Step-by-Step
The query retrieves all columns (*) from the TABLES view where the table schema matches 'MY_SCHEMA'.

### Implementation Example
To catalog tables in a specific database, use the following query:
```
SELECT TABLE_NAME, TABLE_SCHEMA, TABLE_TYPE, CREATED
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'MY_SCHEMA';
```

### Explanation of the Code
- The query selects specific columns (TABLE_NAME, TABLE_SCHEMA, TABLE_TYPE, CREATED) from the TABLES view.
- The WHERE clause filters the results to only include tables in the specified schema ('MY_SCHEMA').

### When to Use
- When you need to retrieve metadata about tables in Snowflake
- When you want to filter tables by schema or table type

### When Not to Use
- When you need to retrieve data from the tables themselves (use SELECT statements instead)
- When you need to perform DDL operations on tables (use CREATE, ALTER, DROP statements instead)

### Common Mistakes
- Forgetting to specify the schema or database when querying the TABLES view
- Assuming the TABLES view contains data from the tables themselves (it only contains metadata)
