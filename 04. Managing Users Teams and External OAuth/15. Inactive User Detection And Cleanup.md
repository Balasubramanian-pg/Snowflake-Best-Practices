# Inactive User Detection And Cleanup in Snowflake

Inactive user detection and cleanup in Snowflake refers to the process of identifying user accounts that have not accessed the system for a specified period and subsequently taking action, such as disabling or removing them. This practice is crucial for maintaining security, reducing licensing costs, and ensuring compliance with organizational policies. It exists because dormant accounts can pose a security risk (e.g., if compromised, they might go unnoticed) and unnecessarily consume resources.

**Short Explanation**

This SQL concept involves querying Snowflake's metadata to find users who haven't logged in recently. The problem it solves is mitigating security vulnerabilities from stale accounts and optimizing resource allocation by removing unneeded access. It's important in databases for robust security postures and efficient cost management, especially in large organizations. It's commonly used in IT administration and security auditing workflows.

## Implementation Example

```sql
-- Step 1: Identify inactive users (e.g., no login in the last 90 days)
SELECT
    u.name AS username,
    u.login_name,
    u.last_success_login AS last_login_time,
    u.created_on AS user_created_on,
    DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) AS days_since_last_login
FROM
    SNOWFLAKE.ACCOUNT_USAGE.USERS u
WHERE
    u.last_success_login IS NULL OR DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) > 90
    AND u.disabled = FALSE
    AND u.name NOT IN ('SNOWFLAKE_ADMIN', 'ACCOUNTADMIN'); -- Exclude critical system users

-- Step 2: (Optional) Generate commands to disable identified inactive users
SELECT
    'ALTER USER ' || u.name || ' SET DISABLED = TRUE;' AS disable_command
FROM
    SNOWFLAKE.ACCOUNT_USAGE.USERS u
WHERE
    (u.last_success_login IS NULL OR DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) > 90)
    AND u.disabled = FALSE
    AND u.name NOT IN ('SNOWFLAKE_ADMIN', 'ACCOUNTADMIN');

-- Step 3: (Optional) Generate commands to drop identified inactive users (use with extreme caution)
SELECT
    'DROP USER ' || u.name || ';' AS drop_command
FROM
    SNOWFLAKE.ACCOUNT_USAGE.USERS u
WHERE
    (u.last_success_login IS NULL OR DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) > 180) -- Longer period for dropping
    AND u.disabled = TRUE -- Only drop if already disabled
    AND u.name NOT IN ('SNOWFLAKE_ADMIN', 'ACCOUNTADMIN');
```

## Explanation of the Code

-   **`SELECT u.name AS username, ...`**: This clause specifies the columns we want to retrieve, renaming `name` to `username` for clarity. It dictates what information will be in the final result set.
-   **`FROM SNOWFLAKE.ACCOUNT_USAGE.USERS u`**: This clause indicates that we are querying the `USERS` view within the `ACCOUNT_USAGE` schema of the `SNOWFLAKE` database. This view contains metadata about all users in the account. It defines the source data for the query.
-   **`WHERE u.last_success_login IS NULL OR DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) > 90 AND u.disabled = FALSE AND u.name NOT IN (...)`**: This clause filters the rows from the `USERS` view.
    -   `u.last_success_login IS NULL`: Includes users who have never successfully logged in.
    -   `DATEDIFF('day', u.last_success_login, CURRENT_TIMESTAMP()) > 90`: Includes users whose last successful login was more than 90 days ago.
    -   `u.disabled = FALSE`: Excludes users who are already disabled.
    -   `u.name NOT IN (...)`: Excludes specific critical system users from the results.
    -   This clause restricts the rows that meet the specified conditions, thus changing the result set to include only inactive, enabled users (excluding critical ones).
-   **`CURRENT_TIMESTAMP()`**: A Snowflake function that returns the current UTC timestamp, used here to calculate the duration since the last login.
-   **`DATEDIFF('day', date1, date2)`**: A Snowflake function that calculates the difference between two date/timestamp expressions based on the specified date part (e.g., 'day').
-   **`'ALTER USER ' || u.name || ' SET DISABLED = TRUE;' AS disable_command`**: This constructs a string literal that represents a SQL command to disable a user. The `||` operator concatenates strings. This transforms the data into executable commands.

## When to Use

1.  **Security Audits**: Regularly (e.g., quarterly) to identify and disable dormant accounts that could be exploited if compromised, strengthening overall security posture.
2.  **Cost Optimization**: In environments with per-user licensing or where unused accounts contribute to administrative overhead, to reduce unnecessary costs by deactivating or removing them.
3.  **Compliance Requirements**: To meet regulatory requirements (e.g., GDPR, SOX) that mandate periodic review and removal of access for inactive users, ensuring data access controls are up-to-date.

## When Not to Use

1.  **Without Proper Review**: Never automate the dropping or disabling of users without human review and confirmation, as critical service accounts or temporarily inactive users might be inadvertently affected.
2.  **For Service Accounts**: Do not apply this logic directly to service accounts or accounts used by applications, as these typically do not have interactive logins and would appear "inactive" but are essential for system operations.
3.  **Short Inactivity Periods**: Avoid using very short inactivity periods (e.g., less than 30 days) for automatic cleanup, as users on vacation, leave, or infrequent project involvement could be prematurely affected.

## Common Mistakes

1.  **Forgetting to Exclude System/Service Accounts**: Accidentally disabling or dropping critical Snowflake system accounts (like ACCOUNTADMIN) or service accounts that don't have interactive logins, leading to system outages.
2.  **Insufficient Inactivity Period**: Setting too short a dormancy period, causing legitimate users who are temporarily away or have infrequent access needs to be flagged as inactive.
3.  **Lack of Notification/Warning**: Disabling or dropping users without prior notification, leading to frustration, lost work, and support tickets when users suddenly lose access.
4.  **No Archiving or Audit Trail**: Failing to maintain a record of disabled/dropped users, including the reason and date, which can hinder auditing or recovery efforts if an account needs to be reinstated.

## Expected Output

The resulting dataset from the first query (identifying inactive users) should look like a list of users, their login details, and the duration of their inactivity. Each row represents an inactive user identified by the query conditions.

| USERNAME     | LOGIN_NAME     | LAST_LOGIN_TIME       | USER_CREATED_ON       | DAYS_SINCE_LAST_LOGIN |
| :----------- | :------------- | :-------------------- | :-------------------- | :-------------------- |
| JDOE_ANALYST | john.doe@comp  | 2025-10-20 10:00:00.0 | 2024-05-15 08:30:00.0 | 145                   |
| MSMITH_DEV   | mary.smith@comp | NULL                  | 2025-01-01 09:00:00.0 | NULL                  |
| PWILL_TEMP   | peter.will@comp | 2025-11-01 14:00:00.0 | 2025-08-01 11:00:00.0 | 134                   |

-   **USERNAME**: The display name of the user.
-   **LOGIN_NAME**: The login identifier for the user.
-   **LAST_LOGIN_TIME**: The timestamp of the user's last successful login. `NULL` indicates they have never logged in.
-   **USER_CREATED_ON**: The timestamp when the user account was created.
-   **DAYS_SINCE_LAST_LOGIN**: The number of days passed since the `LAST_LOGIN_TIME` relative to the current timestamp. `NULL` for users who never logged in.
