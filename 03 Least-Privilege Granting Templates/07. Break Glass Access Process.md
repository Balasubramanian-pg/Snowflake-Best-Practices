# Break Glass Access Process in Snowflake

The "Break Glass Access Process" in Snowflake is a critical security procedure designed to grant emergency, elevated access to the Snowflake environment when normal authentication and authorization mechanisms are unavailable or compromised. It exists to prevent total lockout from the system during severe outages (e.g., identity provider failure) or critical incidents, ensuring that authorized personnel can still access and remediate urgent issues.

**Short Explanation**

This SQL concept outlines a pre-defined, secure process to gain temporary, highly privileged access to Snowflake in emergencies. It solves the problem of operational paralysis when standard access methods fail, ensuring business continuity. It's crucial in databases for incident response and disaster recovery, commonly used in scenarios like identity provider outages or security breaches to regain control.

- **What problem does this SQL feature solve?** It solves the problem of complete system inaccessibility during critical incidents, such as identity provider outages, where normal login procedures are non-functional, or during security breaches requiring immediate, deep access for remediation.
- **Why is it important in databases or analytics?** It's vital for maintaining operational resilience and security. Without it, an organization could be locked out of its data warehouse, leading to prolonged downtime, data loss, or inability to respond to a security incident.
- **Where is it commonly used in real workflows?** It's part of an organization's overall incident response and disaster recovery plan, specifically for scenarios affecting authentication systems or requiring root-level access that bypasses typical controls.

## Implementation Example

While the "Break Glass Access Process" itself isn't a single SQL query, it involves the existence of pre-configured Snowflake users and roles. The "implementation" in Snowflake largely focuses on user and role setup, and then the actual *process* for using these in an emergency.

Here's an example of setting up a dedicated "break glass" user and role:

```sql
-- 1. Create a dedicated "break glass" role with ACCOUNTADMIN privileges
CREATE ROLE BREAK_GLASS_ADMIN;
GRANT ROLE ACCOUNTADMIN TO ROLE BREAK_GLASS_ADMIN;

-- 2. Create a dedicated "break glass" user
-- This user should NOT be associated with an SSO provider or have MFA directly tied to a single device.
-- The password (and OTP secret if used) should be securely stored in an offline, physical vault.
CREATE USER EMERGENCY_USER
    PASSWORD = 'VeryStrongAndComplexPassword123!' -- To be changed immediately upon first use
    DEFAULT_ROLE = BREAK_GLASS_ADMIN
    MUST_CHANGE_PASSWORD = TRUE -- Enforce password change on first login
    COMMENT = 'Dedicated user for emergency break-glass access. Credentials stored securely offline.';

-- 3. Grant the break-glass role to the emergency user
GRANT ROLE BREAK_GLASS_ADMIN TO USER EMERGENCY_USER;

-- 4. (Optional but recommended) Configure MFA for this user with an OTP secret
-- The OTP secret should also be stored securely offline with the password.
ALTER USER EMERGENCY_USER SET CLIENT_MFA_EXT_AUTHN_REQUIRED = TRUE;
-- The OTP secret is generated during the user's first MFA setup, which should be done
-- by a security team member and then immediately stored offline.
```

## Explanation of the Code

This isn't a query that *does* the break-glass, but sets up the necessary components.

- **`CREATE ROLE BREAK_GLASS_ADMIN;`**: This statement creates a new custom role specifically for emergency access. It is named clearly to indicate its purpose.
- **`GRANT ROLE ACCOUNTADMIN TO ROLE BREAK_GLASS_ADMIN;`**: This grants the highest possible privilege (`ACCOUNTADMIN`) to the `BREAK_GLASS_ADMIN` role. This is crucial because emergency access might require broad permissions to diagnose and fix systemic issues.
- **`CREATE USER EMERGENCY_USER ...;`**: This creates a dedicated Snowflake user account.
    - `PASSWORD = '...'`: A strong, temporary password is set. This password (and any MFA details) should be stored in a highly secure, offline location (e.g., a physical safe, an encrypted USB drive in a secure location), separate from normal access controls.
    - `DEFAULT_ROLE = BREAK_GLASS_ADMIN`: Ensures this user automatically assumes the emergency administrative role upon login.
    - `MUST_CHANGE_PASSWORD = TRUE`: Forces the user to change their password immediately after the first login, enhancing security after its initial use.
    - `COMMENT = '...'`: Provides metadata about the user's purpose for auditing and clarity.
- **`GRANT ROLE BREAK_GLASS_ADMIN TO USER EMERGENCY_USER;`**: Explicitly assigns the highly privileged `BREAK_GLASS_ADMIN` role to the `EMERGENCY_USER`.
- **`ALTER USER EMERGENCY_USER SET CLIENT_MFA_EXT_AUTHN_REQUIRED = TRUE;`**: This line enforces Multi-Factor Authentication (MFA) for the `EMERGENCY_USER`. For break-glass, this typically means a Time-based One-Time Password (TOTP) secret that is also stored securely offline, not tied to an individual's personal device.

## When to Use

1.  **Identity Provider (IdP) Outage:** If your organization uses an external IdP (like Okta, Azure AD, PingFederate) for Snowflake authentication and that IdP experiences an outage, preventing all users from logging in.
    *   **Scenario:** Your company's Okta instance is down globally, and no one can log into Snowflake. The break-glass user, with credentials stored independently, allows a designated security or operations team member to access Snowflake and investigate/remediate.
2.  **Accidental Lockout:** When configuration errors (e.g., incorrect network policies, overly restrictive roles, or misconfigured SSO) accidentally lock all or critical administrative users out of the Snowflake account.
    *   **Scenario:** A new network policy is deployed that inadvertently blocks all corporate IP ranges, preventing all users, including `ACCOUNTADMIN`s, from connecting. The break-glass user, potentially exempt from network policies or using a pre-approved external IP, can be used to revert the problematic policy.
3.  **Critical Security Incident:** During a severe security breach where normal administrative accounts may be compromised, or the incident response requires immediate, broad access to audit, isolate, or respond within Snowflake.
    *   **Scenario:** Evidence suggests an attacker has compromised several administrative accounts. The break-glass user is used by the incident response team to bypass potentially compromised accounts, suspend suspicious users, terminate active sessions, and implement new security controls.

## When Not to Use

1.  **Routine Administrative Tasks:** It should never be used for daily operations, regular configuration changes, or minor troubleshooting.
    *   **Reason:** The purpose of break-glass is for *emergencies only*. Using it for routine tasks bypasses normal auditing and access controls, increasing the risk of misuse or undetected compromise. Regular administrative accounts with least privilege should always be preferred.
2.  **Non-Critical Outages:** For issues that do not prevent all administrative access or do not pose an immediate, severe business impact.
    *   **Reason:** The process should have strict activation protocols due to its highly privileged nature. Overusing it desensitizes personnel to its importance and can lead to lax security practices. For example, if only one specific user's login is failing, the standard password reset process should be followed.
3.  **As a Substitute for Proper Access Management:** If an organization consistently finds itself needing to use break-glass because their regular access controls are poorly designed or too restrictive for legitimate tasks.
    *   **Reason:** This indicates a fundamental flaw in access management strategy. Instead of relying on break-glass, the organization should re-evaluate and improve its role-based access control (RBAC), SSO integration, and other security policies to support efficient daily operations without needing emergency overrides.

## Common Mistakes

1.  **Insecure Storage of Credentials:** Storing the break-glass password (and MFA secret) in easily accessible digital files, shared drives, or relying on individual memory.
2.  **Lack of Independent MFA:** Not configuring MFA for the break-glass user, or tying it to the same IdP or device that might be affected by the emergency. The MFA method should be entirely separate and offline (e.g., a physical TOTP token).
3.  **Infrequent Testing:** Not periodically testing the break-glass procedure (including credential retrieval and login) to ensure it works as expected. This can lead to discovery of expired credentials or broken processes during an actual emergency.
4.  **No Post-Mortem/Audit:** Failing to conduct a thorough review and audit every time break-glass access is used. This includes logging all actions, rotating credentials, and analyzing why it was needed.
5.  **Over-Privileging Regular Users:** Relying on break-glass because regular `ACCOUNTADMIN` users are deemed too risky, which is a symptom of not having strong enough regular administrative controls.

## Expected Output

There is no direct "output" in the sense of a SQL query result for the "Break Glass Access Process" itself. The "output" is a *successful and auditable restoration of access and control* to the Snowflake environment during a critical incident.

If we were to look at the effects of the `CREATE USER` and `CREATE ROLE` commands used in the setup example, you would see:

**Table: SNOWFLAKE.ACCOUNT_USAGE.ROLES**

| ROLE_NAME         | OWNER         | COMMENT                                         |
| :---------------- | :------------ | :---------------------------------------------- |
| BREAK_GLASS_ADMIN | SYSADMIN      | Role for emergency break-glass access           |

**Explanation:** This row indicates that a new role named `BREAK_GLASS_ADMIN` has been successfully created and is owned by `SYSADMIN`.

**Table: SNOWFLAKE.ACCOUNT_USAGE.USERS**

| USER_NAME       | DEFAULT_ROLE      | MUST_CHANGE_PASSWORD | COMMENT                                                               |
| :-------------- | :---------------- | :------------------- | :-------------------------------------------------------------------- |
| EMERGENCY_USER | BREAK_GLASS_ADMIN | TRUE                 | Dedicated user for emergency break-glass access. Credentials stored... |

**Explanation:** This row confirms the `EMERGENCY_USER` account has been created, is set to use `BREAK_GLASS_ADMIN` as its default role, requires a password change on first login, and has the descriptive comment.

Additionally, after the `ALTER USER ... SET CLIENT_MFA_EXT_AUTHN_REQUIRED = TRUE;` command, the user's configuration for MFA would be updated internally by Snowflake.
