# Network Policy Rollback Procedures in Snowflake

Network policies in Snowflake control access to your Snowflake account based on IP addresses, ensuring that only authorized networks can connect. While Snowflake doesn't have a direct "rollback" command for network policies, the rollback process involves reapplying a previous, known-good configuration. This is crucial for maintaining security and preventing accidental lockouts due to misconfigured policies. The ability to revert to a working state quickly is a fundamental aspect of managing security configurations in any critical system.

**Short Explanation**

A network policy rollback in Snowflake means restoring a previous, functional network policy configuration after a problematic change. Since there isn't an explicit "undo" feature, it typically involves reactivating a previously defined policy or re-applying a known-good set of IP addresses and rules. This ensures continuous, secure access while correcting any errors introduced by recent modifications.

- **What problem does this SQL feature solve?** It addresses the problem of accidental lockouts or unintended access restrictions caused by errors in network policy updates. It allows administrators to quickly restore a secure and functional state.
- **Why is it important in databases or analytics?** It's vital for maintaining business continuity and security. Incorrect network policies can block legitimate users, ETL processes, or BI tools, halting operations. A quick rollback minimizes downtime and potential data exposure.
- **Where is it commonly used in real workflows?** It's used as part of incident response procedures after a network policy change causes connectivity issues, or during planned maintenance where a change needs to be reverted if not successful. It's also part of change management processes to ensure a fallback plan exists.

## Implementation Example

While there isn't a single "rollback" command, here's how you might revert to a previous account-level network policy named `GOOD_NETWORK_POLICY` after an issue with `BAD_NETWORK_POLICY`.

```sql
-- Assume BAD_NETWORK_POLICY was applied and caused issues
ALTER ACCOUNT SET NETWORK_POLICY = BAD_NETWORK_POLICY;

-- To "rollback" or revert to a known good state:
-- First, identify the previously good network policy (e.g., GOOD_NETWORK_POLICY)
-- Or, if the intention is to temporarily remove any account-level policy:
-- ALTER ACCOUNT SET NETWORK_POLICY = NULL;

-- In this example, we re-apply a known good policy
USE ROLE SECURITYADMIN;
ALTER ACCOUNT SET NETWORK_POLICY = GOOD_NETWORK_POLICY;

-- To verify the change
SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;
```

## Explanation of the Code

This example demonstrates how to switch between different network policies or remove an active one.

- `ALTER ACCOUNT SET NETWORK_POLICY = BAD_NETWORK_POLICY;`: This line, representing the problematic change, sets the account's network policy to `BAD_NETWORK_POLICY`.
    - **What it does**: Modifies the `NETWORK_POLICY` parameter at the account level.
    - **How it changes the result set**: This is a DDL command; it doesn't return a result set. It changes the account's security configuration, affecting which IP addresses can connect to Snowflake.
- `USE ROLE SECURITYADMIN;`: Before modifying account-level parameters like `NETWORK_POLICY`, you need a role with sufficient privileges, typically `SECURITYADMIN` or higher.
    - **What it does**: Changes the current active role for the session.
    - **How it changes the result set**: N/A. It alters the context of subsequent commands.
- `ALTER ACCOUNT SET NETWORK_POLICY = GOOD_NETWORK_POLICY;`: This is the actual "rollback" action. It reassigns the account's network policy to a previously defined and verified `GOOD_NETWORK_POLICY`.
    - **What it does**: Reconfigures the account to use `GOOD_NETWORK_POLICY`, effectively replacing `BAD_NETWORK_POLICY`.
    - **How it changes the result set**: N/A. It changes the account's network access rules.
- `ALTER ACCOUNT SET NETWORK_POLICY = NULL;`: (Alternative rollback) This command would remove any active account-level network policy, allowing connections from any IP address (unless user-level policies are in place). This might be used as an emergency measure to restore access.
    - **What it does**: Unsets the `NETWORK_POLICY` parameter for the account.
    - **How it changes the result set**: N/A. It removes the account's network access restrictions.
- `SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;`: This command is used for verification.
    - **What it does**: Displays the current value of the `NETWORK_POLICY` parameter for the account.
    - **How it changes the result set**: Returns a table showing parameter name, key, value, default, and level (account, user, etc.), confirming which network policy is active.

## When to Use

1. **Post-Deployment Verification Failure**: Immediately after deploying a new network policy or an update, if connectivity tests fail for expected users or systems, revert to the previous working policy.
    * *Example Scenario*: A new policy `NEW_CORPORATE_NP` is applied. Suddenly, the BI tool connecting from a whitelisted IP address can no longer access Snowflake. The DBA would quickly `ALTER ACCOUNT SET NETWORK_POLICY = OLD_CORPORATE_NP;`.
2. **Accidental Lockout**: If an administrator inadvertently blocks their own access or critical system access, a rollback to a less restrictive or previously functional policy is necessary to regain control.
    * *Example Scenario*: An admin updates an `ALLOWED_IP_LIST` and accidentally omits their own IP or the IP of a critical data loading service. They would need to use a pre-existing "break-glass" policy or a previous version to restore access.
3. **Security Incident Response**: If a network policy is suspected to be compromised or misconfigured, leading to unauthorized access attempts, reverting to a known secure configuration can be a quick containment measure.
    * *Example Scenario*: Unusual login attempts are detected from an IP range that *should* be blocked, indicating an issue with the current policy. Revert to a validated secure policy while investigating.

## When Not to Use

1. **Without a Valid Backup/Known Good State**: Do not attempt a "rollback" if you don't have a reliable previous policy or configuration to revert to. Blindly setting `NETWORK_POLICY = NULL` could open up access to unintended IPs.
2. **When the Issue is Not Network Policy Related**: If connectivity problems persist after a rollback, the issue might lie elsewhere (e.g., firewall changes outside Snowflake, VPN issues, incorrect user credentials). Continuously rolling back policies won't solve the underlying problem.
3. **For Minor Adjustments**: For small, incremental changes to an existing network policy (e.g., adding a single IP), directly modifying the policy (using `ALTER NETWORK POLICY... ADD/REMOVE...` or `CREATE OR REPLACE NETWORK POLICY`) is generally more efficient than a full rollback/reapplication cycle, especially with the newer network rules approach.

## Common Mistakes

1. **Forgetting to Apply the Policy**: Creating a `NETWORK POLICY` object is only the first step; it must be explicitly applied to the account, user, or security integration using `ALTER ACCOUNT SET NETWORK_POLICY = <policy_name>;` or `ALTER USER SET NETWORK_POLICY = <policy_name>;`. Developers might create a new policy but forget to activate it, leading to a false sense of security.
2. **Not Including Your Own IP**: When defining a new `ALLOWED_IP_LIST` or network rule, administrators often accidentally omit their own current IP address, leading to self-lockout.
3. **Lack of Testing**: Implementing a policy without thoroughly testing access from all expected locations can lead to critical systems being locked out. Always test changes in a controlled environment or with a user-specific policy before applying account-wide.
4. **Misunderstanding Precedence**: User-level network policies override account-level policies, and security integration-level policies override both. Applying policies at multiple levels without understanding this hierarchy can lead to unexpected access behaviors.
5. **Not Using Network Rules (New Approach)**: Relying on the old `ALLOWED_IP_LIST` and `BLOCKED_IP_LIST` directly within network policies rather than using the newer, more modular `NETWORK RULES` can make management and updates cumbersome, especially when dealing with many IPs or complex scenarios.

## Expected Output

Applying a `NETWORK_POLICY` doesn't produce a direct SQL result set that shows data. Instead, it alters the network accessibility. To verify, you would query the system parameters or views.

If you run `SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;` after a rollback, the expected output would be a single row showing the successfully applied network policy.

| key | value | default | level | description |
| :------------- | :-------------------- | :------ | :------ | :-------------------------------------------------- |
| NETWORK_POLICY | `GOOD_NETWORK_POLICY` | NULL | ACCOUNT | Current network policy for the Snowflake account. |

**Explanation of Columns:**

- **`key`**: The name of the parameter being displayed (`NETWORK_POLICY`).
- **`value`**: The name of the network policy currently active at the account level (e.g., `GOOD_NETWORK_POLICY`). This confirms the rollback was successful.
- **`default`**: The default value for the parameter, which is `NULL` for `NETWORK_POLICY` (meaning no policy is active by default).
- **`level`**: Indicates at which level the parameter is set (e.g., `ACCOUNT`).
-   **`description`**: A brief explanation of the parameter.
