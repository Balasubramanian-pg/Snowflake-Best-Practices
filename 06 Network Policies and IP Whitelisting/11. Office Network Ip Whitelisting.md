# Office Network IP Whitelisting in Snowflake

Office Network IP Whitelisting in Snowflake is a security feature that allows you to restrict access to your Snowflake account based on the IP addresses from which connection attempts originate. This ensures that only authorized users connecting from approved network locations (like your corporate office network or VPN) can access your data, significantly reducing the risk of unauthorized access from external or untrusted sources. It acts as a virtual firewall at the account, user, or integration level within Snowflake.

**Short Explanation**

This SQL feature solves the problem of unauthorized access by ensuring connections to Snowflake only come from predefined, trusted IP addresses or ranges. It's crucial in databases and analytics to protect sensitive data from external threats and comply with security regulations. Commonly, it's used to enforce corporate security policies, allowing access only from secure office environments or approved VPNs.

- **What problem does this SQL feature solve?** It prevents unauthorized access to your Snowflake account by limiting connection origins to known and trusted IP addresses, thereby mitigating external attack vectors.
- **Why is it important in databases or analytics?** It's a fundamental layer of security for protecting sensitive data, ensuring data integrity and confidentiality, and helping organizations meet compliance requirements like GDPR or HIPAA.
- **Where is it commonly used in real workflows?** It's typically implemented to secure access for employees working from a corporate office, remote employees using a corporate VPN, or automated tools and services that connect to Snowflake from a fixed set of IP addresses.

## Implementation Example

Snowflake recommends using Network Rules, which are schema-level objects that group network identifiers, in conjunction with Network Policies for managing IP whitelisting. This provides greater flexibility and reusability.

First, create a Network Rule defining the allowed IP addresses:

```sql
-- Create a Network Rule for office IP addresses
CREATE NETWORK RULE office_network_ips
    TYPE = IPV4
    VALUE_LIST = ('203.0.113.0/24', '198.51.100.10', '192.0.2.0/28')
    MODE = INGRESS
    COMMENT = 'Allowed IP addresses for the corporate office network';

-- Then, create a Network Policy that references this Network Rule
CREATE NETWORK POLICY corporate_access_policy
    ALLOWED_NETWORK_RULE_LIST = ('office_network_ips')
    COMMENT = 'Network policy allowing access only from specified office IPs';

-- Finally, activate the Network Policy for the entire account
ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;

-- Alternatively, activate for a specific user
-- ALTER USER <username> SET NETWORK_POLICY = corporate_access_policy;
```

## Explanation of the Code

This example demonstrates the modern approach to IP whitelisting in Snowflake using Network Rules and Network Policies.

- `CREATE NETWORK RULE office_network_ips...`: This statement defines a new `NETWORK RULE` named `office_network_ips`.
    - `TYPE = IPV4`: Specifies that the rule contains IPv4 addresses.
    - `VALUE_LIST = ('203.0.113.0/24', '198.51.100.10', '192.0.2.0/28')`: This clause lists the specific IP addresses and CIDR blocks (ranges of IP addresses) that will be permitted. These would be your office's public IP addresses.
    - `MODE = INGRESS`: Indicates that this rule applies to incoming traffic to Snowflake.
    - `COMMENT = 'Allowed IP addresses for the corporate office network'`: Provides a descriptive comment for the rule.
- `CREATE NETWORK POLICY corporate_access_policy...`: This statement creates a `NETWORK POLICY` named `corporate_access_policy`.
    - `ALLOWED_NETWORK_RULE_LIST = ('office_network_ips')`: This is the crucial part that links the network policy to the previously defined `office_network_ips` network rule. It specifies that only connections originating from the IP addresses listed in `office_network_ips` are allowed.
    - `COMMENT = 'Network policy allowing access only from specified office IPs'`: A descriptive comment for the policy.
- `ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;`: This command activates the `corporate_access_policy` at the account level. Once activated, this policy applies to all users and all connections attempting to access this Snowflake account. Any connection attempt from an IP address not listed in the `office_network_ips` rule will be denied.
- `ALTER USER <username> SET NETWORK_POLICY = corporate_access_policy;`: (Commented out) This alternative command shows how to apply the network policy to a specific user instead of the entire account. This allows for more granular control, where different users might have different allowed IP ranges.

## When to Use

1. **Enhancing Overall Account Security:** When you want to ensure that all access to your Snowflake account (for all users and integrations) is restricted to a set of known and trusted IP addresses, such as your corporate network IPs or VPN endpoints.
    * *Example Scenario:* Your company policy mandates that no one can access cloud data warehousing services from personal home networks or public Wi-Fi. By whitelisting your office and corporate VPN IP ranges, you enforce this policy across the board.
2. **Securing Service Accounts and Automated Processes:** When you have service accounts or automated ETL/ELT tools that connect to Snowflake, it's critical to restrict their access to only the specific IP addresses of the servers or services running these processes.
    * *Example Scenario:* A data pipeline tool (e.g., dbt Cloud, Fivetran, Airflow) needs to connect to Snowflake. You whitelist the egress IP addresses provided by these services to ensure only they can connect using the service account credentials.
3. **Meeting Compliance Requirements:** Many industry regulations (e.g., PCI DSS, HIPAA, GDPR) require strict access controls to sensitive data. IP whitelisting helps demonstrate compliance by limiting the network perimeter from which data can be accessed.
    * *Example Scenario:* An organization handling patient health information (PHI) must adhere to HIPAA. Implementing IP whitelisting ensures that PHI in Snowflake is only accessible from approved, secure internal networks, reducing the risk of data breaches.

## When Not to Use

1. **Highly Distributed or Transient User Bases Without Centralized Network Control:** If your users are globally distributed and frequently connect from varying, unpredictable IP addresses (e.g., consultants, field sales without a mandatory VPN), and you don't have a corporate VPN for all of them.
    * *Situation:* Forcing IP whitelisting in such a scenario would lead to constant access issues and management overhead, potentially locking out legitimate users.
2. **When IP Addresses are Dynamic and Cannot be Controlled:** For users or services whose external IP addresses change frequently and are outside your control (e.g., some mobile network providers, certain cloud-based developer workstations that assign dynamic public IPs).
    * *Situation:* Constantly updating the whitelist for dynamic IPs is impractical and can lead to frequent service interruptions. Other security measures like strong MFA and client certificates might be more appropriate.
3. **As the Sole Security Measure:** Relying solely on IP whitelisting is not sufficient for comprehensive security. It should always be combined with other layers like strong authentication (MFA), Role-Based Access Control (RBAC), and data encryption.
    * *Situation:* If an attacker gains access to an allowed IP address (e.g., through a compromised office workstation), whitelisting alone won't prevent unauthorized access to Snowflake.

## Common Mistakes

1. **Forgetting to include your current IP address:** When activating a network policy, if your current IP address is not included in the allowed list, you can immediately lock yourself out of the account.
2. **Not using CIDR notation correctly or missing IP ranges:** Incorrectly specifying IP addresses or CIDR blocks can lead to legitimate users being blocked or, conversely, leaving unintended gaps in your security. Forgetting to include all necessary office or VPN IP ranges is a common oversight.
3. **Applying a policy too broadly without testing:** Activating an account-level network policy without thoroughly testing it with a smaller group of users or in a non-production environment can cause widespread access issues for your entire organization. Snowflake recommends testing user-level policies first.
4. **Neglecting to update policies when network infrastructure changes:** If your office moves, your VPN provider changes, or your network architecture is reconfigured, the whitelisted IP addresses must be updated accordingly. Forgetting to do so will result in users being locked out.

## Expected Output

After successfully applying the network policy, any attempt to connect to the Snowflake account from an IP address *not* present in the `office_network_ips` rule will result in a connection error or denied access. Connections from the whitelisted IP addresses will proceed as normal.

**Example of connection attempt outcomes:**

| IP Address Attempting Connection | Connection Status | Explanation |
| :------------------------------- | :---------------- | :---------------------------------------------------------------------------------------------------------- |
| 203.0.113.15 | Success | This IP is within the `203.0.113.0/24` CIDR block in the `office_network_ips` rule. |
| 198.51.100.10 | Success | This specific IP is listed in the `office_network_ips` rule. |
| 1.2.3.4 | Denied | This IP address is not included in the `office_network_ips` rule, so access is blocked by `corporate_access_policy`. |

The "Expected Output" isn't a dataset returned by a query, but rather the behavior of the Snowflake system regarding connection attempts based on their originating IP address. The "columns" in this context would be aspects of a connection attempt and the system's response.
