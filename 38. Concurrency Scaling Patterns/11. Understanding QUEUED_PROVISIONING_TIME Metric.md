# QUEUED_PROVISIONING_TIME Metric in Snowflake

The `QUEUED_PROVISIONING_TIME` metric in Snowflake measures the amount of time a warehouse spends waiting for resources to become available before it can start provisioning. This metric helps understand delays in query execution caused by insufficient compute resources. It exists to provide visibility into potential bottlenecks related to warehouse scaling and resource contention.

**Short Explanation**

`QUEUED_PROVISIONING_TIME` indicates how long a query (or a warehouse acting on behalf of a query) was stuck in a queue because the necessary compute resources were not immediately available to provision a new or scale-up an existing warehouse. It directly addresses the problem of understanding why queries might be slow to start, even when they appear to be "running" from a user perspective. This is crucial for optimizing Snowflake costs and performance, as high provisioning times can point to undersized warehouses or inefficient scaling policies. It's commonly used in performance tuning, capacity planning, and cost optimization workflows.

- **What problem does this SQL feature solve?** It helps identify resource contention issues by showing how much time is lost due to warehouses waiting to start or scale up.
- **Why is it important in databases or analytics?** High values can indicate performance bottlenecks, sub-optimal warehouse configurations, or a need to adjust auto-scaling policies, impacting user experience and query latency.
- **Where is it commonly used in real workflows?** Performance monitoring, cost management, and resource allocation discussions, especially in data warehousing and analytics environments with fluctuating workloads.

## Implementation Example

```sql
SELECT
    QUERY_ID,
    QUERY_TEXT,
    EXECUTION_STATUS,
    QUEUED_PROVISIONING_TIME,
    START_TIME,
    END_TIME
FROM
    SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    QUEUED_PROVISIONING_TIME > 0
    AND START_TIME >= DATEADD(hour, -24, CURRENT_TIMESTAMP())
ORDER BY
    QUEUED_PROVISIONING_TIME DESC;
```

## Explanation of the Code

This SQL query retrieves information about queries that experienced provisioning queue time within the last 24 hours.

- **`SELECT QUERY_ID, QUERY_TEXT, EXECUTION_STATUS, QUEUED_PROVISIONING_TIME, START_TIME, END_TIME`**: This clause specifies the columns to be retrieved from the `QUERY_HISTORY` view.
    - **What it does**: It selects specific attributes of each query.
    - **How it changes the result set**: It determines the schema and content of the output, showing the query identifier, the actual SQL text, its final status, the time spent in provisioning queue, and the start and end timestamps.
- **`FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY`**: This clause indicates the table or view from which to retrieve data.
    - **What it does**: It specifies that we are querying the `QUERY_HISTORY` view, which contains metadata about all queries executed in the Snowflake account.
    - **How it changes the result set**: It provides the source dataset for the query.
- **`WHERE QUEUED_PROVISIONING_TIME > 0 AND START_TIME >= DATEADD(hour, -24, CURRENT_TIMESTAMP())`**: This clause filters the rows based on specified conditions.
    - **What it does**: It includes only queries that had a non-zero `QUEUED_PROVISIONING_TIME` (meaning they actually waited for provisioning) and started within the last 24 hours.
    - **How it changes the result set**: It reduces the number of rows to only those relevant to recent provisioning delays, making the analysis more focused.
- **`ORDER BY QUEUED_PROVISIONING_TIME DESC`**: This clause sorts the result set.
    - **What it does**: It orders the filtered queries in descending order based on their `QUEUED_PROVISIONING_TIME`.
    - **How it changes the result set**: It arranges the output so that queries with the longest provisioning delays appear first, highlighting the most significant issues.

## When to Use

1.  **Performance Troubleshooting**: When users report slow query starts or overall slow performance, checking `QUEUED_PROVISIONING_TIME` can help determine if the delay is due to warehouse resource availability.
    *   *Example Scenario*: A daily reporting dashboard starts running late. Investigating `QUEUED_PROVISIONING_TIME` for the dashboard's queries might reveal consistent delays due to waiting for the associated virtual warehouse to scale up, indicating the need for a larger warehouse or adjusted scaling policy.
2.  **Capacity Planning**: To understand if existing virtual warehouse sizes and auto-scaling configurations are adequate for current and projected workloads.
    *   *Example Scenario*: Regularly monitoring `QUEUED_PROVISIONING_TIME` across all warehouses can show trends. If it consistently increases during peak hours, it suggests that the current scaling limits or warehouse sizes are insufficient, informing decisions to increase max clusters or warehouse size.
3.  **Cost Optimization**: To ensure that the Snowflake account is not over-provisioned or under-provisioned, impacting both performance and credit consumption.
    *   *Example Scenario*: High `QUEUED_PROVISIONING_TIME` might lead to queries running longer overall, tying up resources. Conversely, if no `QUEUED_PROVISIONING_TIME` is observed, but queries are still slow, the issue might be query optimization rather than warehouse scaling, preventing unnecessary cost increases by just upsizing warehouses.

## When Not to Use

1.  **For Analyzing Query Execution Time After Provisioning**: `QUEUED_PROVISIONING_TIME` only measures the time *before* the warehouse is ready. It doesn't tell you anything about how long the query actually ran once resources were allocated.
    *   *Situation*: A query has a high `EXECUTION_TIME` but zero `QUEUED_PROVISIONING_TIME`. This metric is irrelevant here; you should look at `COMPILATION_TIME`, `LOCAL_DISK_IO_TIME`, `REMOTE_DISK_IO_TIME`, etc.
2.  **When Queries Are Constantly Failing Due to Syntax Errors**: If queries are failing early due to incorrect SQL, `QUEUED_PROVISIONING_TIME` will likely be negligible or zero, as the issue is not resource contention.
    *   *Situation*: A new SQL script is failing with "SQL compilation error". `QUEUED_PROVISIONING_TIME` is not the metric to check; the error message itself is the primary diagnostic.
3.  **To Diagnose Network Latency Issues**: If users are experiencing delays fetching results, but queries show low `QUEUED_PROVISIONING_TIME` and reasonable `EXECUTION_TIME`, the problem might be network-related between the client and Snowflake, which this metric does not cover.
    *   *Situation*: Reports are slow to load in an external BI tool. Snowflake metrics show good performance. `QUEUED_PROVISIONING_TIME` won't help; network diagnostics from the client side are needed.

## Common Mistakes

1.  **Confusing it with `QUEUED_OVERLOAD_TIME`**: These are distinct. `QUEUED_OVERLOAD_TIME` occurs when a warehouse is fully scaled out and cannot handle more queries, so queries wait for *existing* resources to free up. `QUEUED_PROVISIONING_TIME` is about waiting for *new* resources to become available (i.e., a warehouse to start or scale up).
2.  **Ignoring Auto-Suspend and Auto-Resume Settings**: High `QUEUED_PROVISIONING_TIME` can be frequent if warehouses are set to auto-suspend very quickly during periods of intermittent usage, leading to repeated provisioning delays.
3.  **Not Correlating with Warehouse Scaling Policies**: Failing to check the `MIN_CLUSTER_COUNT` and `MAX_CLUSTER_COUNT` settings for the affected warehouses. A low `MAX_CLUSTER_COUNT` might be preventing the warehouse from scaling sufficiently to meet demand, leading to high provisioning times.
