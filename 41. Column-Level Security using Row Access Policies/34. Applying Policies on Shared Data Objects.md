# Applying Policies on Shared Data Objects in Snowflake

This concept explores how data governance policies, such as row access policies and masking policies, are applied to data objects that are shared with other Snowflake accounts. It's crucial for maintaining data security and compliance even when data moves beyond the original account.

**Short Explanation**

Applying policies on shared data objects in Snowflake means that data providers can define and enforce access controls and data masking rules on their shared data, and these policies remain active and effective when consumers access that data. This feature solves the problem of maintaining granular control over sensitive data even after it's shared, ensuring compliance and preventing unauthorized access or exposure of information. It's important for creating secure data ecosystems and is commonly used in scenarios where data is shared with partners, customers, or different departments within an organization while adhering to strict governance requirements.

- **What problem does this SQL feature solve?** It solves the challenge of maintaining data security, compliance, and granular access control over sensitive data when it is shared with external (or internal) Snowflake accounts.
- **Why is it important in databases or analytics?** It ensures that data governance policies are consistently applied, preventing data leakage, unauthorized access, and non-compliance, which is vital for trust and regulatory adherence in data sharing.
- **Where is it commonly used in real workflows?** It's used when sharing sensitive customer data with analytics partners, financial data with auditors, or medical records with research institutions, where specific columns need to be masked or rows filtered based on the consumer's role or access rights.

## Implementation Example

```sql
-- In the Provider Account
-- 1. Create a database for sharing
CREATE DATABASE sales_data_share;
USE DATABASE sales_data_share;
CREATE SCHEMA public;

-- 2. Create a table with sensitive data
CREATE TABLE customer_info (
    customer_id INT,
    customer_name VARCHAR,
    email VARCHAR,
    phone_number VARCHAR,
    region VARCHAR,
    sensitive_notes VARCHAR
);

INSERT INTO customer_info VALUES
(1, 'Alice Smith', 'alice@example.com', '111-222-3333', 'East', 'High value customer'),
(2, 'Bob Johnson', 'bob@example.com', '444-555-6666', 'West', 'Prefers email communication'),
(3, 'Charlie Brown', 'charlie@example.com', '777-888-9999', 'East', 'Has ongoing support ticket');

-- 3. Create a masking policy for the email column
CREATE MASKING POLICY email_mask_policy AS (val string) RETURNS string ->
  CASE
    WHEN CURRENT_ROLE() = 'ANALYST_ROLE' THEN val
    ELSE '***MASKED***'
  END;

-- 4. Create a row access policy for the region
CREATE ROW ACCESS POLICY region_access_policy AS (region_col string) RETURNS BOOLEAN ->
  CASE
    WHEN CURRENT_ROLE() = 'EAST_ANALYST_ROLE' AND region_col = 'East' THEN TRUE
    WHEN CURRENT_ROLE() = 'WEST_ANALYST_ROLE' AND region_col = 'West' THEN TRUE
    WHEN CURRENT_ROLE() = 'FULL_ACCESS_ROLE' THEN TRUE
    ELSE FALSE
  END;

-- 5. Apply the policies to the table columns
ALTER TABLE customer_info MODIFY COLUMN email SET MASKING POLICY email_mask_policy;
ALTER TABLE customer_info ADD ROW ACCESS POLICY region_access_policy ON (region);

-- 6. Grant usage on policies to the share (this is crucial for policies to work in consumer accounts)
GRANT USAGE ON MASKING POLICY email_mask_policy TO SHARE my_data_share;
GRANT USAGE ON ROW ACCESS POLICY region_access_policy TO SHARE my_data_share;

-- 7. Create a share and add the database to it
CREATE SHARE my_data_share;
GRANT USAGE ON DATABASE sales_data_share TO SHARE my_data_share;
GRANT USAGE ON SCHEMA sales_data_share.public TO SHARE my_data_share;
GRANT SELECT ON TABLE sales_data_share.public.customer_info TO SHARE my_data_share;

-- 8. Add consumer accounts to the share (e.g., 'ABCDEFG123')
ALTER SHARE my_data_share ADD ACCOUNTS = ABCDEFG123;

-- In the Consumer Account (ABCDEFG123)
-- 9. Create database from share
CREATE DATABASE shared_sales_data FROM SHARE provider_account.my_data_share;

-- 10. Grant roles to users (e.g., assuming user 'JOHN' needs East region access)
USE ROLE SECURITYADMIN;
CREATE ROLE EAST_ANALYST_ROLE; -- Must match role name in provider account policies
GRANT ROLE EAST_ANALYST_ROLE TO USER JOHN;

-- 11. Query the shared data as user JOHN (with EAST_ANALYST_ROLE)
USE ROLE EAST_ANALYST_ROLE;
SELECT customer_id, customer_name, email, phone_number, region FROM shared_sales_data.public.customer_info;
```

## Explanation of the Code

This example demonstrates how policies defined in a provider account are enforced in a consumer account through Snowflake's Secure Data Sharing.

- **`CREATE DATABASE sales_data_share;`**: Sets up a new database in the provider account to house the data intended for sharing.
- **`CREATE TABLE customer_info (...)`**: Creates a sample table containing customer information, including sensitive fields like `email` and `sensitive_notes`.
- **`CREATE MASKING POLICY email_mask_policy AS (...)`**: Defines a masking policy named `email_mask_policy`.
    - **What it does**: This policy will mask the `email` column's value with `***MASKED***` for any role except `ANALYST_ROLE`. If the user querying has `ANALYST_ROLE`, they see the original email.
    - **How it changes the result set**: It modifies the data directly returned from the `email` column, replacing actual values with a masked string based on the current role.
- **`CREATE ROW ACCESS POLICY region_access_policy AS (...)`**: Defines a row access policy named `region_access_policy`.
    - **What it does**: This policy filters rows based on the `region_col` (which maps to the `region` column in the table). Users with `EAST_ANALYST_ROLE` can only see rows where `region` is 'East', `WEST_ANALYST_ROLE` for 'West', and `FULL_ACCESS_ROLE` sees all rows. Other roles see no rows.
    - **How it changes the result set**: It removes entire rows from the result set if the `region` value does not match the allowed regions for the current role.
- **`ALTER TABLE customer_info MODIFY COLUMN email SET MASKING POLICY email_mask_policy;`**: Applies the `email_mask_policy` to the `email` column of `customer_info`.
- **`ALTER TABLE customer_info ADD ROW ACCESS POLICY region_access_policy ON (region);`**: Applies the `region_access_policy` to the `customer_info` table, using the `region` column for evaluation.
- **`GRANT USAGE ON MASKING POLICY... TO SHARE...;`**: This is a critical step. It explicitly grants usage on the masking policy to the share. Without this, the policy would not be evaluated in the consumer account.
- **`GRANT USAGE ON ROW ACCESS POLICY... TO SHARE...;`**: Similarly, this grants usage on the row access policy to the share, enabling its enforcement in consumer accounts.
- **`CREATE SHARE my_data_share;`**: Initializes a secure share object.
- **`GRANT USAGE ON DATABASE... TO SHARE...;`**, **`GRANT USAGE ON SCHEMA... TO SHARE...;`**, **`GRANT SELECT ON TABLE... TO SHARE...;`**: These commands grant the necessary permissions for the share to include the database, schema, and the table within the shared object.
- **`ALTER SHARE my_data_share ADD ACCOUNTS = ABCDEFG123;`**: Adds a specific consumer Snowflake account (identified by its account locator, e.g., `ABCDEFG123`) to the share, allowing them to access the shared data.
- **`CREATE DATABASE shared_sales_data FROM SHARE provider_account.my_data_share;`**: In the consumer account, this command creates a read-only database linked to the shared data from the provider.
- **`CREATE ROLE EAST_ANALYST_ROLE;`** and **`GRANT ROLE EAST_ANALYST_ROLE TO USER JOHN;`**: In the consumer account, this creates a role with the same name as used in the provider's row access policy and assigns it to a user. The *name* of the role is crucial for policy evaluation.
- **`SELECT... FROM shared_sales_data.public.customer_info;`**: When the user `JOHN` (with `EAST_ANALYST_ROLE`) queries the shared data, the policies from the provider account are evaluated. The masking policy applies because `JOHN` does not have `ANALYST_ROLE`, and the row access policy filters to only 'East' region data.

## When to Use

1. **Compliance with Data Privacy Regulations:** When you need to share data that contains personally identifiable information (PII) or other sensitive data, and regulations like GDPR or HIPAA require strict access controls and data anonymization.
    * **Example Scenario:** A healthcare provider shares patient treatment data with a research institution. A masking policy is applied to patient names and addresses, and a row access policy ensures researchers only see data for patients who have given consent.
2. **Multi-Tenant Data Sharing:** In a SaaS model where you share a common dataset with multiple customers, but each customer should only see their own data, or a masked version of other customers' data.
    * **Example Scenario:** A marketing analytics platform shares campaign performance data with various clients. A row access policy ensures each client only sees their own campaign data, while a masking policy might anonymize specific competitive metrics if a client accidentally gains access to another's aggregated view.
3. **Internal Departmental Data Governance:** When different departments within a large organization need access to a central dataset, but their access privileges vary based on their role and data sensitivity.
    * **Example Scenario:** A sales department shares its customer database with the finance department. Finance needs to see full customer details for billing, but a support team viewing the same data might only need to see customer names and masked contact information, filtered to only their assigned region.

## When Not to Use

1. **Over-complicating Simple Data Sharing:** For data that is non-sensitive, already anonymized, or where full access is acceptable for all consumers, applying policies can introduce unnecessary complexity and overhead.
    * **Reason:** Policies add a layer of processing to every query, which can slightly impact performance. If there's no security or compliance reason, it's simpler to share directly.
2. **Performance-Critical Analytical Workloads:** If the shared data is part of a highly performance-sensitive analytical pipeline where every millisecond counts, and the policies involve complex logic or join external tables for evaluation.
    * **Reason:** While Snowflake's policy engine is optimized, complex policies, especially row access policies that might involve subqueries or UDFs, can add latency. For extremely high-throughput systems, pre-filtering or pre-masking data before sharing might be more efficient.
3. **When Policies Are Best Managed at the Consumer Level:** In scenarios where the consumer account has specific and dynamic internal governance requirements that the provider cannot foresee or manage universally.
    * **Reason:** While provider-managed policies offer strong centralized control, if consumers need highly custom or frequently changing access rules based on their internal user roles, it might be more agile for the provider to share raw (non-sensitive) data and let the consumer apply their own policies.

## Common Mistakes

1. **Forgetting to Grant Policy Usage to the Share:** The most common mistake. Policies applied to tables in the provider account will not be enforced in the consumer account unless `GRANT USAGE ON MASKING POLICY <policy_name> TO SHARE <share_name>;` and `GRANT USAGE ON ROW ACCESS POLICY <policy_name> TO SHARE <share_name>;` commands are executed.
2. **Mismatched Role Names in Consumer Account:** The roles referenced in the provider's policies (e.g., `ANALYST_ROLE`, `EAST_ANALYST_ROLE`) must exist with the exact same names in the consumer account for the policies to evaluate correctly. If the consumer account uses different role names, the policies might default to the "ELSE" condition, leading to unintended masking or filtering.
3. **Overly Complex Policy Logic:** Creating masking or row access policies with very complex SQL logic, numerous `CASE` statements, or subqueries can impact query performance, especially on large datasets.
4. **Not Testing Policies from Consumer Perspective:** Providers sometimes test policies only within their own account. It's crucial to thoroughly test the shared data from a consumer account perspective with different roles to ensure policies are applied as intended.
5. **Sharing Policies Directly Instead of Table with Policy:** You share the database object (table, view) that *has* the policies applied to it, not the policies themselves as standalone objects. The `GRANT USAGE` on the policies to the share is what links them.

## Expected Output

When a user in the consumer account queries the `customer_info` table with the `EAST_ANALYST_ROLE`, they would see only the customer(s) from the 'East' region. Additionally, the `email` column for these customers would be masked because the `EAST_ANALYST_ROLE` is not the `ANALYST_ROLE` that was granted full access in the masking policy.
