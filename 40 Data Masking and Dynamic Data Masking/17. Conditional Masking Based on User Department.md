# Conditional Masking Based on User Department in Snowflake

This concept refers to dynamically obscuring sensitive data in Snowflake based on the department or role of the user querying the data. It leverages Snowflake's security features, primarily masking policies, to ensure that users only see the data they are authorized to view, protecting sensitive information from unauthorized access while still allowing legitimate access for those who need it.

**Short Explanation**

Conditional masking based on user department in Snowflake automatically hides or partially hides data (like redacting or nulling out values) for users who don't belong to specific departments. This solves the problem of data exposure by enforcing fine-grained access control directly at the column level. It's crucial in databases for maintaining data privacy and regulatory compliance (e.g., GDPR, HIPAA), and is commonly used in analytics, reporting, and data sharing scenarios where different teams have varying data access privileges.

- **What problem does this SQL feature solve?** It solves the problem of unauthorized data exposure and ensures that sensitive information is only revealed to authorized users, preventing data breaches and aiding compliance.
- **Why is it important in databases or analytics?** It's vital for implementing robust data governance, protecting personal identifiable information (PII), and ensuring that data consumers only see relevant and permitted data, thus improving data security and integrity.
- **Where is it commonly used in real workflows?** It's used in organizations with diverse departments (e.g., HR, Finance, Marketing, Sales) that share a common data warehouse but require strict separation of duties regarding sensitive data like salaries, customer contact details, or proprietary financial figures.

## Implementation Example

```sql
-- 1. Create a sample table with sensitive data
CREATE TABLE employee_data (
    employee_id INT,
    employee_name VARCHAR(100),
    department VARCHAR(50),
    salary DECIMAL(10, 2),
    ssn VARCHAR(20) -- Sensitive data
);

-- 2. Insert sample data
INSERT INTO employee_data VALUES
(1, 'Alice Johnson', 'HR', 75000.00, 'XXX-XX-1234'),
(2, 'Bob Smith', 'Sales', 60000.00, 'XXX-XX-5678'),
(3, 'Charlie Brown', 'Finance', 90000.00, 'XXX-XX-9012'),
(4, 'Diana Prince', 'HR', 80000.00, 'XXX-XX-3456');

-- 3. Create custom roles for departments (if not already existing)
CREATE ROLE IF NOT EXISTS HR_DEPARTMENT_ROLE;
CREATE ROLE IF NOT EXISTS FINANCE_DEPARTMENT_ROLE;
CREATE ROLE IF NOT EXISTS SALES_DEPARTMENT_ROLE;

-- Grant roles to users (example for a user 'analyst_user')
-- GRANT ROLE HR_DEPARTMENT_ROLE TO USER analyst_user;
-- GRANT ROLE FINANCE_DEPARTMENT_ROLE TO USER analyst_user;
-- GRANT ROLE SALES_DEPARTMENT_ROLE TO USER analyst_user;

-- 4. Create a masking policy
CREATE OR REPLACE MASKING POLICY ssn_mask_policy AS (val VARCHAR) RETURNS VARCHAR ->
    CASE
        WHEN CURRENT_ROLE() IN ('HR_DEPARTMENT_ROLE', 'FINANCE_DEPARTMENT_ROLE') THEN val -- HR and Finance can see SSN
        ELSE '***-**-****' -- Other roles see masked SSN
    END;

-- 5. Apply the masking policy to the SSN column
ALTER TABLE employee_data MODIFY COLUMN ssn SET MASKING POLICY ssn_mask_policy;

-- 6. Test with a user having a specific role (e.g., 'analyst_user' with SALES_DEPARTMENT_ROLE)
-- If 'analyst_user' has SALES_DEPARTMENT_ROLE granted:
-- USE ROLE SALES_DEPARTMENT_ROLE;
-- SELECT employee_id, employee_name, department, salary, ssn FROM employee_data;

-- Test with a user having HR_DEPARTMENT_ROLE
-- USE ROLE HR_DEPARTMENT_ROLE;
-- SELECT employee_id, employee_name, department, salary, ssn FROM employee_data;

-- Test with a user having no specific department role (or a different role not listed in policy)
-- USE ROLE PUBLIC; -- Or any other non-privileged role
-- SELECT employee_id, employee_name, department, salary, ssn FROM employee_data;
```

## Explanation of the Code

- **`CREATE TABLE employee_data (...)`**: This statement creates a table named `employee_data` to store employee information, including a `ssn` column which is considered sensitive.
- **`INSERT INTO employee_data VALUES (...)`**: This populates the `employee_data` table with sample records for different departments.
- **`CREATE ROLE IF NOT EXISTS ...`**: These commands create custom roles representing different departments. Snowflake roles are used here to manage department-specific permissions.
- **`CREATE OR REPLACE MASKING POLICY ssn_mask_policy AS (val VARCHAR) RETURNS VARCHAR -> ...`**:
    - This is the core of the conditional masking. It defines a masking policy named `ssn_mask_policy`.
    - `AS (val VARCHAR) RETURNS VARCHAR`: Specifies that the policy takes a `VARCHAR` input (the column value) and returns a `VARCHAR` (the masked or unmasked value).
    - `CASE WHEN CURRENT_ROLE() IN ('HR_DEPARTMENT_ROLE', 'FINANCE_DEPARTMENT_ROLE') THEN val ELSE '***-**-****' END;`: This `CASE` expression implements the conditional logic.
        - **`CURRENT_ROLE()`**: A Snowflake function that returns the primary role currently active for the session.
        - **`IN ('HR_DEPARTMENT_ROLE', 'FINANCE_DEPARTMENT_ROLE')`**: Checks if the user's active role is one of the specified department roles.
        - If the user's role is `HR_DEPARTMENT_ROLE` or `FINANCE_DEPARTMENT_ROLE`, the original `val` (SSN) is returned, meaning the data is unmasked.
        - **`ELSE '***-**-****'`**: For any other role, a masked string `***-**-****` is returned, effectively hiding the actual SSN.
- **`ALTER TABLE employee_data MODIFY COLUMN ssn SET MASKING POLICY ssn_mask_policy;`**: This command applies the `ssn_mask_policy` to the `ssn` column of the `employee_data` table. Once applied, all queries on this column will automatically invoke the masking policy.
- **`SELECT employee_id, employee_name, department, salary, ssn FROM employee_data;`**: This is a sample query to demonstrate how the masking policy works when executed by users with different roles.
    - **`SELECT`**: Specifies the columns to retrieve.
    - **`FROM`**: Indicates the table to query.
    - **`How it changes the result set`**: The `ssn` column's value in the result set will change dynamically based on the `CURRENT_ROLE()` of the user executing the `SELECT` statement. For authorized roles, the full SSN is shown; for unauthorized roles, it's masked.

## When to Use

1.  **Compliance and Regulatory Requirements:** When sensitive data (like PII, PCI, HIPAA-protected health information) must be restricted to specific departments or roles to meet regulatory standards.
    *   **Scenario:** A healthcare provider needs to store patient medical records. Only doctors and nurses (via their assigned roles) should see full diagnoses, while administrative staff might only see masked or partial information for scheduling purposes.
2.  **Internal Data Governance:** To enforce internal data security policies where different business units have varying levels of access to shared datasets, even within the same organization.
    *   **Scenario:** A company stores all employee compensation data in a single table. The HR department role can see full salaries and bonuses, while team managers (different role) can only see aggregated or averaged salary data for their direct reports, not individual figures.
3.  **Third-Party Data Sharing with Anonymization:** When sharing a dataset with external vendors or partners who require some data for analytics but should not have access to sensitive identifiers.
    *   **Scenario:** A marketing analytics firm needs access to customer demographic data, but actual customer names and contact information must be masked, while geographical and age data remain unmasked.

## When Not to Use

1.  **Overly Complex Logic:** If the masking logic becomes extremely convoluted with many nested `CASE` statements based on numerous conditions and roles, it can be difficult to manage and debug.
    *   **Situation:** Masking requirements depend on a combination of user department, time of day, specific data values, and an external system's authorization, leading to an unwieldy masking policy.
2.  **Performance Critical Workloads on Very Large Tables:** While Snowflake's masking policies are optimized, complex policy evaluations on columns that are frequently queried by many different roles on extremely large tables might introduce a slight overhead.
    *   **Situation:** A high-frequency transactional system relies on microsecond response times for queries on a table with billions of rows, and the masking policy involves several complex subqueries or external function calls.
3.  **Simple Row-Level Security Needs:** If the requirement is to restrict access to entire rows based on user department (e.g., an HR user only sees HR employees, a Sales user only sees Sales leads), Row Access Policies are generally a more direct and efficient solution than trying to achieve this with column masking.
    *   **Situation:** Finance users should only see financial transactions, and HR users should only see HR records, with no overlap. Trying to mask entire rows of non-departmental data using column masking policies would be inefficient and incorrect; Row Access Policies are designed for this.

## Common Mistakes

1.  **Forgetting to Grant Roles:** Developers often define roles and policies but forget to grant the necessary department roles to the actual users who need to see the unmasked data. This results in all users seeing masked data.
2.  **Incorrect `CURRENT_ROLE()` Usage:** Assuming `CURRENT_ROLE()` will always return the highest-privileged role or a specific role when a user might have multiple roles activated or switch roles. The policy evaluates based on the *active* role in the session, which might not be the intended department role. `IS_ROLE_IN_SESSION()` can sometimes be a more robust alternative for checking if a role is *granted* to the current session, regardless of whether it's the *active* primary role.
3.  **Overlapping or Conflicting Policies:** Applying multiple masking policies to the same column, or having a complex interaction between masking policies and other security features (like row access policies), leading to unexpected masking behavior or access issues. Snowflake only allows one masking policy per column.
4.  **Inadequate Testing with Different Roles:** Not thoroughly testing the masking policy with various user roles (including those with no specific department role, and those with the privileged department roles) can lead to data exposure or unintended data blocking.

## Expected Output

The resulting dataset will show all columns of the `employee_data` table. The `ssn` column will display its original, unmasked value only if the querying user's active role is `HR_DEPARTMENT_ROLE` or `FINANCE_DEPARTMENT_ROLE`. For all other roles, the `ssn` column will display the masked value `***-**-****`.
