# Cross Account Data Sharing Controls in Snowflake

Snowflake's Cross Account Data Sharing allows organizations to securely share live, read-only data with other Snowflake accounts without physically moving or copying the data. This revolutionary approach eliminates the need for traditional ETL processes, data replication, and managing multiple copies of datasets, fostering real-time collaboration and data consistency across different teams, departments, or even external partners.

**Short Explanation**

This SQL concept enables a data provider to make specific database objects accessible to one or more consumer Snowflake accounts. The provider retains full control over the data and its security, while consumers can query the shared data directly from their own accounts as if it were local. It solves the problem of inefficient data exchange, ensures data is always current, and simplifies compliance by centralizing data governance in the provider's account. It's commonly used for collaborative analytics, inter-company data exchange, and creating data marketplaces.

- **What problem does this SQL feature solve?** It eliminates the overhead, latency, and costs associated with traditional data sharing methods like ETL pipelines, data replication, or file transfers.
- **Why is it important in databases or analytics?** It enables real-time insights, ensures data consistency, and facilitates secure collaboration without compromising data governance or security.
- **Where is it commonly used in real workflows?** Collaborative analytics, financial reporting, supply chain tracking, marketing analysis, and data monetization through data marketplaces.

## Implementation Example

Here's a simplified example of a data provider creating a share and granting access to a consumer:

```sql
-- As Data Provider (ACCOUNTADMIN role recommended for setup)

-- 1. Create a database to hold the data to be shared
CREATE DATABASE sales_db;
USE DATABASE sales_db;

-- 2. Create a schema and a table with some data
CREATE SCHEMA public;
CREATE TABLE public.daily_sales (
    sale_date DATE,
    product_id VARCHAR,
    quantity INTEGER,
    revenue DECIMAL(10, 2)
);

INSERT INTO public.daily_sales VALUES
('2026-03-13', 'PROD_A', 10, 150.00),
('2026-03-13', 'PROD_B', 5, 75.50);

-- 3. Create a Share object
CREATE SHARE sales_data_share;

-- 4. Grant USAGE privilege on the database and schema to the share
GRANT USAGE ON DATABASE sales_db TO SHARE sales_data_share;
GRANT USAGE ON SCHEMA sales_db.public TO SHARE sales_data_share;

-- 5. Grant SELECT privilege on the table to the share
GRANT SELECT ON TABLE sales_db.public.daily_sales TO SHARE sales_data_share;

-- 6. Add the consumer account to the share (replace 'XYZ12345' with actual consumer account locator)
ALTER SHARE sales_data_share ADD ACCOUNTS = XYZ12345;

-- As Data Consumer (e.g., SYSADMIN role)

-- 1. Create a database from the shared data
CREATE DATABASE shared_sales_db FROM SHARE provider_account.sales_data_share;
-- Note: 'provider_account' is the name or locator of the provider's account.

-- 2. Query the shared data
SELECT * FROM shared_sales_db.public.daily_sales WHERE sale_date = CURRENT_DATE();
```

## Explanation of the Code

This example demonstrates the typical workflow for setting up cross-account data sharing.

**Provider Side:**
- `CREATE DATABASE sales_db;`: Creates a new database where the data to be shared will reside.
  - **What it does:** Defines a logical container for schemas and tables.
  - **How it changes the result set:** This is a DDL command; it doesn't directly return a result set but creates the database object.
- `USE DATABASE sales_db;`: Sets the current database context.
  - **What it does:** Specifies which database subsequent commands will operate on.
  - **How it changes the result set:** No direct result set; changes the session context.
- `CREATE SCHEMA public;`: Creates a schema within the database.
  - **What it does:** Organizes database objects logically.
  - **How it changes the result set:** DDL command; creates the schema object.
- `CREATE TABLE public.daily_sales (...);`: Defines a table structure.
  - **What it does:** Creates a table to store transactional sales data.
  - **How it changes the result set:** DDL command; creates the table object.
- `INSERT INTO public.daily_sales VALUES (...);`: Populates the table with data.
  - **What it does:** Adds records to the `daily_sales` table.
  - **How it changes the result set:** DML command; modifies the table's contents.
- `CREATE SHARE sales_data_share;`: Creates a named share object.
  - **What it does:** Initializes a share entity that will encapsulate the data to be shared.
  - **How it changes the result set:** DDL command; creates the share object.
- `GRANT USAGE ON DATABASE sales_db TO SHARE sales_data_share;`: Grants permission to use the database.
  - **What it does:** Allows the share to reference objects within `sales_db`. Without this, objects in the database cannot be granted.
  - **How it changes the result set:** DDL command; establishes a privilege.
- `GRANT USAGE ON SCHEMA sales_db.public TO SHARE sales_data_share;`: Grants permission to use the schema.
  - **What it does:** Allows the share to reference objects within the `public` schema.
  - **How it changes the result set:** DDL command; establishes a privilege.
- `GRANT SELECT ON TABLE sales_db.public.daily_sales TO SHARE sales_data_share;`: Grants read access to the table.
  - **What it does:** Specifies that the `daily_sales` table can be queried by consumers of this share. Crucially, shared data is always read-only.[ajyM]
  - **How it changes the result set:** DDL command; establishes a privilege.
- `ALTER SHARE sales_data_share ADD ACCOUNTS = XYZ12345;`: Adds a consumer account to the share.
  - **What it does:** Designates a specific Snowflake account (identified by its locator) as a recipient of this share.
  - **How it changes the result set:** DDL command; modifies the share configuration.

**Consumer Side:**
- `CREATE DATABASE shared_sales_db FROM SHARE provider_account.sales_data_share;`: Creates a local database from the share.
  - **What it does:** The consumer creates a read-only database in their account, which points to the shared data in the provider's account.
  - **How it changes the result set:** DDL command; creates a new database object (a "shared database").
- `SELECT * FROM shared_sales_db.public.daily_sales WHERE sale_date = CURRENT_DATE();`: Queries the shared table.
  - **What it does:** Retrieves data from the `daily_sales` table as provided by the sharing account.
  - **How it changes the result set:** Returns a dataset containing rows from `daily_sales` where `sale_date` matches the current date.

## When to Use

1. **Cross-departmental or cross-organizational collaboration:** When different business units or external partners need access to common datasets without data duplication.
    * *Example Scenario:* A retail company sharing sales data with its regional marketing teams or external market research agencies for real-time analysis.
2. **Data monetization or data marketplaces:** Providers can offer datasets to multiple consumers, sometimes via the Snowflake Marketplace, as a data product.
    * *Example Scenario:* A weather data provider offering historical and real-time weather information to various industries (e.g., agriculture, logistics) as a paid service.
3. **Real-time data integration:** When consumers need immediate access to the most up-to-date information from a provider without building complex ETL processes.
    * *Example Scenario:* A manufacturing company sharing inventory levels or production line data with a logistics partner to optimize shipping schedules.

## When Not to Use

1. **When write access is required by the consumer:** Shared data in Snowflake is always read-only. If the consumer needs to modify, insert, or delete data, data sharing is not the appropriate solution.[ajyM]
    * *Reason:* Snowflake Secure Data Sharing fundamentally operates on a read-only principle for consumers.
2. **When data needs to be stored locally by the consumer for performance-critical, highly isolated workloads:** While shared data can be queried efficiently, extreme performance requirements for highly specific, localized operations might benefit from having a copy of the data (though this negates many sharing benefits).
    * *Reason:* Although shared data performs well, in rare, highly specialized cases, a physically localized copy might offer marginal performance gains for certain extreme workloads.
3. **When sharing non-shareable objects:** Not all Snowflake objects can be shared. For example, non-secure views, stored procedures, file stages, tasks, and streams cannot be directly shared via this mechanism.
    * *Reason:* The sharing mechanism is designed for specific data objects (tables, secure views, UDFs, etc.), and trying to share unsupported types will fail.

## Common Mistakes

1. **Forgetting to grant `USAGE` on the database and schema:** Providers often grant `SELECT` on tables but forget that `USAGE` privileges on the containing database and schema are also required for the share to be functional.[ajyM]
2. **Assuming shared data is writable:** New users might mistakenly believe consumers can modify shared data, leading to incorrect architectural decisions or failed operations on the consumer side.[ajyM]
3. **Over-sharing data:** Granting access to more data than necessary can create security and compliance risks. Best practice is to limit exposure using secure views or sharing only specific tables/schemas.[LGpX]
4. **Not monitoring shared data usage:** Providers should track which consumers access the data, how often queries are run, and which objects are used for governance, cost optimization, and compliance.[LGpX]

## Expected Output

The resulting dataset from the consumer's query `SELECT * FROM shared_sales_db.public.daily_sales WHERE sale_date = CURRENT_DATE();` should contain the sales records for the current date from the provider's `daily_sales` table.

Here's an example table structure and data that the consumer would see if `CURRENT_DATE()` were '2026-03-13':

| SALE_DATE | PRODUCT_ID | QUANTITY | REVENUE |
| :--------- | :--------- | :------- | :------ |
| 2026-03-13 | PROD_A | 10 | 150.00 |
| 2026-03-13 | PROD_B | 5 | 75.50 |

- **SALE_DATE (DATE):** The date on which the sale occurred.
- **PRODUCT_ID (VARCHAR):** A unique identifier for the product sold.
- **QUANTITY (INTEGER):** The number of units of the product sold.
- **REVENUE (DECIMAL(10, 2)):** The total revenue generated from that sale item.
