# IP Whitelisting Fundamentals in Snowflake

IP Whitelisting in Snowflake, primarily managed through **Network Policies** and more recently **Network Rules**, is a crucial security feature that controls which network IP addresses are permitted to access your Snowflake account. It acts like a digital bouncer, only allowing connections from pre-approved locations, thus significantly reducing the risk of unauthorized access to your data.

**Short Explanation**

This SQL feature solves the problem of unauthorized network access by creating a list of trusted IP addresses or ranges. It's important for database and analytics security as it ensures only legitimate sources, like your corporate network or specific applications, can connect to your Snowflake environment. You'll commonly use it in workflows requiring strict data governance and compliance, such as restricting access for specific user groups or integrating with other secure services like Azure Data Factory.

- **What problem does this SQL feature solve?** It prevents unauthorized access to your Snowflake account by restricting connections to a predefined set of IP addresses.
- **Why is it important in databases or analytics?** It's a fundamental security layer, protecting sensitive data from external threats and ensuring compliance with data protection regulations.
- **Where is it commonly used in real workflows?** It's used to secure connections from corporate networks, VPNs, specific applications (like ETL tools), and even for individual users requiring restricted access.

## Implementation Example

```sql
-- Step 1: Create a Network Rule (modern approach)
CREATE NETWORK RULE my_office_ips_rule
  TYPE = IPV4
  VALUE_LIST = ('203.0.113.10', '198.51.100.0/24')
  MODE = INGRESS
  COMMENT = 'Allowed IP addresses for My Office and VPN subnet';

-- Step 2: Create a Network Policy referencing the Network Rule
CREATE NETWORK POLICY my_corporate_network_policy
  ALLOWED_NETWORK_RULE_LIST = ('my_office_ips_rule')
  COMMENT = 'Network policy for corporate access';

-- Step 3: Activate the Network Policy for the entire account
ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;

-- OR, activate the Network Policy for a specific user
-- ALTER USER my_analytics_user SET NETWORK_POLICY = my_corporate_network_policy;

-- Step 4: (Optional) To view active network policy on the account
SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;

-- Step 5: (Optional) To view active network policy for a user
-- SELECT "POLICY_NAME" FROM TABLE("SNOWFLAKE"."INFORMATION_SCHEMA"."POLICY_REFERENCES"(REF_ENTITY_DOMAIN => 'USER', REF_ENTITY_NAME => 'MY_ANALYTICS_USER')) WHERE "POLICY_KIND" = 'NETWORK_POLICY';
```

## Explanation of the Code

This example demonstrates the modern approach using `NETWORK RULES` and `NETWORK POLICIES`.

- **`CREATE NETWORK RULE my_office_ips_rule...`**:
    - **What it does**: This statement defines a reusable network rule named `my_office_ips_rule`.
    - **How it changes the result set**: It creates a new object in Snowflake's metadata, storing a list of specific IP addresses (`203.0.113.10`) and CIDR blocks (`198.51.100.0/24`) that are to be considered for allowing access. `TYPE = IPV4` specifies it's for IPv4 addresses, and `MODE = INGRESS` indicates it applies to incoming connections. This rule itself doesn't block or allow anything; it's a building block.[AELq]

- **`CREATE NETWORK POLICY my_corporate_network_policy...`**:
    - **What it does**: This statement creates a network policy that references one or more network rules.
    - **How it changes the result set**: It creates a policy object (`my_corporate_network_policy`) that now specifies `my_office_ips_rule` as an `ALLOWED_NETWORK_RULE_LIST`. This policy defines the "allow" list of network origins.

- **`ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;`**:
    - **What it does**: This command activates the `my_corporate_network_policy` for the entire Snowflake account.
    - **How it changes the result set**: Once executed, any attempt to connect to this Snowflake account from an IP address *not* present in `my_office_ips_rule` will be denied. It immediately restricts access to the account.

- **`ALTER USER my_analytics_user SET NETWORK_POLICY = my_corporate_network_policy;`**:
    - **What it does**: (Commented out in the example) This command would activate the policy for a specific user, overriding any account-level policy for that user.[AELq]
    - **How it changes the result set**: If activated, `my_analytics_user` would only be able to connect from the specified IPs, regardless of the account-level policy.

- **`SHOW PARAMETERS LIKE 'NETWORK_POLICY' IN ACCOUNT;`**:
    - **What it does**: Retrieves the currently active network policy at the account level.
    - **How it changes the result set**: Returns a row indicating the `NETWORK_POLICY` parameter and its current value for the account.

- **`SELECT "POLICY_NAME" FROM TABLE("SNOWFLAKE"."INFORMATION_SCHEMA"."POLICY_REFERENCES"(REF_ENTITY_DOMAIN => 'USER', REF_ENTITY_NAME => 'MY_ANALYTICS_USER')) WHERE "POLICY_KIND" = 'NETWORK_POLICY';`**:
    - **What it does**: (Commented out) This query checks if a network policy is applied to a specific user.
    - **How it changes the result set**: Returns the name of the network policy if one is active for the specified user.

## When to Use

1. **Enhancing Security for Sensitive Data**: When you have highly sensitive data in Snowflake and need to ensure that only connections originating from your secure corporate network or specific data centers can access it.
    * **Example Scenario**: A financial institution storing customer transaction data restricts access to its Snowflake account to only IP addresses within its approved corporate offices and VPN ranges.
2. **Compliance Requirements**: When regulatory compliance (e.g., HIPAA, GDPR, PCI DSS) mandates strict control over data access points.
    * **Example Scenario**: A healthcare provider uses IP whitelisting to meet HIPAA requirements by ensuring that patient health information (PHI) in Snowflake is only accessible from authorized clinical systems within their network.
3. **Integrating with Specific Applications/Services**: When a particular application, ETL tool, or cloud service needs to connect to Snowflake, and its IP addresses are known and stable.
    * **Example Scenario**: An Azure Data Factory pipeline needs to load data into Snowflake. The specific IP ranges used by the Azure region are whitelisted in Snowflake to allow the ADF service to connect.

## When Not to Use

1. **Highly Dynamic or Unknown IP Addresses**: When users or applications connect from constantly changing or unknown public IP addresses, making it impractical to maintain a whitelist.
    * **Inefficiency**: Constantly updating the whitelist becomes an operational burden and can lead to legitimate users being locked out.
2. **Overly Restrictive for Global/Remote Access**: If your users are globally distributed and frequently access Snowflake from various public Wi-Fi networks, home networks, or traveling.
    * **Inefficiency**: Enforcing a strict IP whitelist would severely hinder user productivity and access, requiring VPNs or other complex solutions for every connection.
3. **As the Only Security Measure**: Relying solely on IP whitelisting without other security controls like strong authentication (MFA), role-based access control, and data encryption.
    * **Wrong Approach**: IP whitelisting is a perimeter defense; it doesn't protect against threats originating *within* the allowed network or against compromised credentials.

## Common Mistakes

1. **Forgetting Your Own IP Address**: Activating an account-level network policy without including your current IP address in the allowed list, which can lock you out of the account.
2. **Not Including All Necessary IPs**: Failing to whitelist all IP addresses that legitimate users or services might use, leading to connection failures and operational disruptions (e.g., forgetting a specific VPN range or a secondary office location).
3. **Ignoring Precedence**: Misunderstanding that user-level network policies override account-level policies, and integration-level policies override both. Applying a broad account-level policy might not affect a user with a more restrictive user-level policy.[AELq]
4. **Using Legacy `ALLOWED_IP_LIST`/`BLOCKED_IP_LIST` Instead of Network Rules**: While still functional, using direct IP lists within `CREATE NETWORK POLICY` is the older approach. Snowflake recommends using `NETWORK RULES` for better organization, reusability, and management, especially with more complex scenarios.

## Expected Output

After successfully implementing an IP whitelisting policy, the expected outcome is that users and applications can only connect to your Snowflake account from the explicitly defined IP addresses or ranges. Connections originating from any other IP address will be rejected at the network level before they even reach the authentication stage.

Here's an example of what a successful connection would look like for a whitelisted IP and a failed connection for a non-whitelisted IP:

| Connection Attempt Source IP | Connection Status | Details |
| :-------------------------- | :---------------- | :------------------------------------------- |
| `203.0.113.10` | Successful | User `johndoe` connects from the office VPN. |
| `198.51.100.55` | Successful | ETL tool connects from an authorized subnet. |
| `192.168.1.10` | Failed | User `janedoe` attempts connection from home (not whitelisted). |

**Explanation of the columns:**

- **Connection Attempt Source IP**: The IP address from which the connection attempt originates.
- **Connection Status**: Indicates whether the connection was successfully established or rejected by the network policy.
-   **Details**: Provides context for the connection, such as the user or application making the attempt, and why it succeeded or failed based on the whitelisting rules.
