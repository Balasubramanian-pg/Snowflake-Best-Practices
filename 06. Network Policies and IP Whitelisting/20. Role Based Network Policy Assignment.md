# Role Based Network Policy Assignment in Snowflake

This concept allows you to control network access to Snowflake for users based on the roles they have. Instead of assigning network policies directly to individual users, which can be cumbersome and error-prone in large organizations, you assign them to roles. When a user assumes a particular role, they inherit the network policy associated with that role. This simplifies administration, ensures consistent security, and makes it easier to manage access as user roles change.

**Short Explanation**

Role-based network policy assignment in Snowflake addresses the challenge of managing network access for many users by linking network policies to roles rather than individual users. This is important because it centralizes security management, allowing administrators to define allowed IP ranges for specific job functions or teams, thereby enhancing security and compliance. It's commonly used in enterprises to enforce strict access controls, especially for sensitive data access or specific operational environments.

- **What problem does this SQL feature solve?** It solves the problem of granular and scalable network access control, avoiding the need to manually update network policies for each user.
- **Why is it important in databases or analytics?** It ensures that only authorized network locations can connect to the Snowflake environment, crucial for protecting sensitive data and meeting regulatory requirements.
- **Where is it commonly used in real workflows?** In large organizations, it's used to segregate network access for different departments (e.g., finance, development, operations) or for specific types of user accounts (e.g., service accounts, standard users).

## Implementation Example

```sql
-- 1. Create a Network Policy
CREATE NETWORK POLICY my_finance_np
    ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.10');

-- 2. Create a Role
CREATE ROLE finance_analyst;

-- 3. Assign the Network Policy to the Role
ALTER ROLE finance_analyst SET NETWORK_POLICY = my_finance_np;

-- 4. Grant the Role to a User
GRANT ROLE finance_analyst TO USER jane_doe;

-- (Optional) 5. Unset the network policy from the role
-- ALTER ROLE finance_analyst UNSET NETWORK_POLICY;

-- (Optional) 6. Drop the network policy
-- DROP NETWORK POLICY my_finance_np;
```

## Explanation of the Code

This example demonstrates the full lifecycle of creating and assigning a role-based network policy.

- `CREATE NETWORK POLICY my_finance_np ...`:
    - **What it does:** This statement defines a new network policy named `my_finance_np`.
    - **How it changes the result set:** It doesn't directly change a result set, but it creates a new security object in Snowflake that can be referenced.
    - `ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.10')`: Specifies the IP addresses or CIDR ranges that are permitted to connect when this policy is active.

- `CREATE ROLE finance_analyst;`:
    - **What it does:** This statement creates a new custom role named `finance_analyst`.
    - **How it changes the result set:** Creates a new role object in Snowflake.

- `ALTER ROLE finance_analyst SET NETWORK_POLICY = my_finance_np;`:
    - **What it does:** This is the core command that assigns the previously created network policy (`my_finance_np`) to the `finance_analyst` role.
    - **How it changes the result set:** Modifies the `finance_analyst` role object to associate it with the specified network policy. When a user with this role attempts to connect, this policy will be evaluated.

- `GRANT ROLE finance_analyst TO USER jane_doe;`:
    - **What it does:** Grants the `finance_analyst` role to a specific user, `jane_doe`.
    - **How it changes the result set:** Links the user `jane_doe` to the `finance_analyst` role, making them subject to the role's network policy when that role is active.

## When to Use

1.  **Enforcing Location-Based Access:** When specific departments or teams should only access Snowflake from their office network or designated VPNs.
    *   *Example Scenario:* The finance team can only access Snowflake from the corporate headquarters' IP range, ensuring financial data security.
2.  **Segregating Access for Service Accounts:** To restrict automated processes or applications to connect only from specific server IPs.
    *   *Example Scenario:* An ETL tool running on a particular server needs to connect to Snowflake; a network policy ensures it only connects from that server's IP address.
3.  **Simplifying Compliance and Auditing:** To meet regulatory requirements that mandate strict network access controls for certain data types.
    *   *Example Scenario:* For PCI DSS compliance, roles accessing credit card data are assigned network policies that limit access to certified secure environments.

## When Not to Use

1.  **When a User Needs Global Access:** If a user (e.g., a highly mobile executive or an external consultant) legitimately needs to connect from many different, changing IP addresses.
    *   *Inefficiency:* Constantly updating network policies for individual dynamic IPs would be unmanageable.
2.  **For Small, Static User Bases:** In very small organizations with only a few users where direct user-level policy assignment is simpler to manage and less overhead.
    *   *Overhead:* Creating roles and policies might add unnecessary complexity for a handful of users.
3.  **When Network Policies Conflict or Overlap Excessively:** If a user has multiple roles with conflicting network policies, the most permissive (union of allowed IPs) might apply, or the most restrictive (intersection), leading to unpredictable behavior or accidental blocking.
    *   *Confusion:* Without clear understanding of policy precedence, it can lead to confusion and unintended access behavior.

## Common Mistakes

1.  **Forgetting to Activate the Role:** A user needs to explicitly use the `USE ROLE` command or have the role set as their default for the network policy associated with that role to take effect.
2.  **Not Understanding Policy Precedence:** If a network policy is set at the account level AND assigned to a user AND assigned to a role the user is using, understanding which policy takes precedence (account > user > role if the user has no user-level policy set, otherwise user-level overrides role-level) can be tricky and lead to unexpected access denials or allowances.
3.  **Using Overly Broad IP Ranges:** Defining policies with large CIDR blocks (e.g., `/16` or `/8`) defeats the purpose of granular control and can expose your Snowflake account to wider network threats.

## Expected Output

When a user `jane_doe` logs in and activates the `finance_analyst` role, Snowflake will evaluate the `my_finance_np` network policy. If `jane_doe` attempts to connect from an IP address *not* in `192.168.1.0/24` or `10.0.0.10`, the connection will be blocked. If they connect from an allowed IP, access will be granted.

No direct table output is generated by the policy assignment itself, but its effect is on connection attempts. If we were to query metadata, we might see the policy linked to the role.

| ROLE_NAME        | NETWORK_POLICY_NAME | ALLOWED_IP_LIST           |
|------------------|---------------------|---------------------------|
| FINANCE_ANALYST  | MY_FINANCE_NP       | ['192.168.1.0/24', '10.0.0.10'] |
| (other roles)    | (other policies)    | (other IPs)               |
