# Policy Evaluation Order with Masking Policies in Snowflake

This concept addresses how Snowflake determines which masking policy to apply when multiple policies could potentially affect the same column. It's crucial for data governance and security, ensuring that sensitive data is consistently protected according to defined rules, especially in complex scenarios with overlapping policies.

**Short Explanation**

Policy evaluation order defines the hierarchy and precedence of masking policies when a column is subject to multiple policies. Snowflake applies a set of rules to decide which policy "wins" and is enforced. This ensures predictable data access control.

*   **What problem does this SQL feature solve?** It solves the ambiguity and potential security gaps that could arise if multiple masking policies are applicable to a single column, by providing a clear, deterministic method for policy selection.
*   **Why is it important in databases or analytics?** It's vital for maintaining data security, regulatory compliance (like GDPR or HIPAA), and ensuring that users only see data they are authorized to view, even in complex data environments.
*   **Where is it commonly used in real workflows?** It's used in environments with granular data access requirements, where different roles or groups need varying levels of access to the same sensitive data (e.g., PII in a customer table, financial data).

## Implementation Example

```sql
-- Assume these policies and roles already exist:
-- policy_strict_mask: Masks PII completely for non-authorized roles.
-- policy_partial_mask: Partially masks PII (e.g., last 4 digits) for analysts.
-- role_admin: Has full access.
-- role_analyst: Has partial access.
-- role_public: Has no access to sensitive data.

-- Scenario: A column 'email' has policy_strict_mask applied directly.
-- Then, policy_partial_mask is applied to the database, which also covers 'email'.

-- Direct policy on the column
ALTER TABLE employees MODIFY COLUMN email SET MASKING POLICY policy_strict_mask;

-- Policy applied to the database (or schema)
ALTER DATABASE my_company_db SET MASKING POLICY policy_partial_mask ON (email);

-- Querying the email column as different roles
-- As role_public
SELECT email FROM employees;

-- As role_analyst (assuming policy_partial_mask allows partial view for this role)
SELECT email FROM employees;

-- As role_admin (assuming policy_strict_mask allows full view for this role)
SELECT email FROM employees;
```

## Explanation of the Code

This example demonstrates how different masking policies can be applied at different levels (column vs. database/schema). The key is understanding Snowflake's precedence rules, which dictate which policy will be active.

*   `ALTER TABLE employees MODIFY COLUMN email SET MASKING POLICY policy_strict_mask;`: This statement applies `policy_strict_mask` directly to the `email` column of the `employees` table. This is the most specific level of policy application.
*   `ALTER DATABASE my_company_db SET MASKING POLICY policy_partial_mask ON (email);`: This statement applies `policy_partial_mask` at the database level, specifically targeting the `email` column within that database. This is a less specific level of application than a direct column policy.
*   `SELECT email FROM employees;` (as `role_public`, `role_analyst`, `role_admin`): These queries simulate accessing the `email` column from different user roles. The output for `email` will vary based on the active policy for each role, considering Snowflake's policy evaluation order.

**How it changes the result set:**
Snowflake applies masking policies in a specific order of precedence, from most specific to least specific:
1.  **Column-level policy:** A policy explicitly set on a column (e.g., `ALTER TABLE ... MODIFY COLUMN ... SET MASKING POLICY`). This is the highest precedence.
2.  **Table-level policy (if applicable):** A policy set on a column as part of a table-level policy (though Snowflake's primary mechanism is column-specific or object-specific).
3.  **Schema-level policy:** A policy set on a schema that applies to columns within that schema (e.g., `ALTER SCHEMA ... ADD MASKING POLICY ... ON (column_name)`).
4.  **Database-level policy:** A policy set on a database that applies to columns within that database (e.g., `ALTER DATABASE ... ADD MASKING POLICY ... ON (column_name)`).

In the example, the `policy_strict_mask` applied directly to the `email` column will take precedence over `policy_partial_mask` applied at the database level for the `email` column. Therefore, for `role_analyst` to see a partial mask, `policy_strict_mask` would need to have logic that allows partial masking for that role, or the `policy_strict_mask` wouldn't be present at all. If `policy_strict_mask` always strictly masks for `role_analyst`, then `role_analyst` would see the strict mask, overriding the database-level policy.

## When to Use

1.  **Layered Security Requirements:** When different levels of data access are needed for different departments or roles within the same organization. For example, a global `email` column might have a strict mask for most users, but a specific analytics team needs to see the last four characters for segmentation.
2.  **Data Compliance:** To meet regulatory requirements (e.g., PII masking). You might have a default policy to mask all PII for everyone, but then grant specific roles access to unmasked data or a specific format of masked data based on their duties and authorization.
3.  **Third-Party Data Sharing:** When sharing data with external partners, you might apply a highly restrictive policy at the database or schema level for the shared data, ensuring no sensitive information leaves your control, while internal users still access less restricted views.

## When Not to Use

1.  **Over-Complication:** If a single, straightforward masking policy is sufficient for a column across all use cases and roles, avoid creating multiple policies and relying on evaluation order, as it can unnecessarily complicate management.
2.  **Performance-Critical Workloads:** While Snowflake's masking policies are optimized, an extremely large number of overlapping policies on many columns across various levels could theoretically add minimal overhead. For extremely high-throughput, low-latency scenarios where every millisecond counts, simplicity is often best.
3.  **When Data Redaction is Required:** Masking policies alter the data viewed, but the underlying data remains unchanged. If you need to permanently redact or delete sensitive data from the source, masking policies are not the right tool; you'd need data manipulation (DELETE/UPDATE) or specific data redaction tools.

## Common Mistakes

1.  **Misunderstanding Precedence:** Developers often assume a database-level policy will override a column-level policy, or vice-versa, without fully grasping the specific-to-general rule (column > table > schema > database). This can lead to data being over-masked or, worse, under-masked.
2.  **Conflicting Policy Logic:** Creating policies that have contradictory logic for the same role, leading to unexpected masking behavior. For example, one policy might allow a role to see full data, while another at a lower precedence might try to mask it – the higher precedence policy will always win.
3.  **Lack of Testing:** Failing to thoroughly test policy application from the perspective of various roles. It's crucial to `GRANT ROLE <role_name> TO USER <user_name>;` and then `USE ROLE <role_name>;` to verify that the expected masking is applied for each role.

The expected output is the masked value of the `email` column, which will vary based on the active role executing the query and the evaluation order of the applied masking policies.

| ***********              | bo**@example.com          | bob.johnson@example.com |

**Explanation of Columns:**

*   **EMAIL (as `role_public`)**: This column shows the `email` value when queried by a user with `role_public`. Since this role likely has no special privileges defined in either policy, the `policy_strict_mask` (due to its higher precedence) would apply a full mask (e.g., replacing with asterisks).
*   **EMAIL (as `role_analyst`)**: This column shows the `email` value when queried by a user with `role_analyst`. If `policy_strict_mask` was designed to allow partial viewing for `role_analyst` (e.g., revealing the domain), or if `policy_strict_mask` was not present and `policy_partial_mask` applied, then `role_analyst` would see a partially masked email (e.g., `an**@example.com`).
*   **EMAIL (as `role_admin`)**: This column shows the `email` value when queried by a user with `role_admin`. Assuming `role_admin` is explicitly allowed to see unmasked data within `policy_strict_mask` (the highest precedence policy), they would see the original, unmasked email address.
