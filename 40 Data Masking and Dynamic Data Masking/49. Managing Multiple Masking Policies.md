# Managing Multiple Masking Policies in Snowflake

This concept addresses the scenario where a single column in a Snowflake table needs to be subject to more than one data masking policy, based on different conditions or roles. It's a powerful feature for granular data security and compliance, ensuring that sensitive data is protected differently for various users or use cases.

**Short Explanation**

Managing multiple masking policies means applying more than one masking rule to a single data column. Snowflake uses a concept called "policy precedence" to determine which masking policy takes effect when multiple policies are assigned. This allows for complex, context-aware data protection.

- **What problem does this SQL feature solve?** It solves the problem of needing dynamic, condition-based data masking where different users or situations require different levels of data visibility on the same column.
- **Why is it important in databases or analytics?** It's crucial for meeting strict data governance, privacy regulations (like GDPR, HIPAA), and internal security policies, especially in environments with diverse user roles and data access needs.
- **Where is it commonly used in real workflows?** It's used in scenarios like financial reporting (masking account numbers differently for auditors vs. customer service), healthcare analytics (masking patient IDs differently for researchers vs. billing staff), or development environments (masking production data for non-production users).

## Implementation Example

```sql
-- Create a table with sensitive data
CREATE TABLE customer_data (
    customer_id INT,
    full_name VARCHAR,
    email VARCHAR,
    credit_card_number VARCHAR
);

-- Insert some sample data
INSERT INTO customer_data VALUES
(1, 'Alice Wonderland', 'alice@example.com', '1234-5678-9012-3456'),
(2, 'Bob The Builder', 'bob@example.com', '9876-5432-1098-7654');

-- Create a masking policy for full masking
CREATE MASKING POLICY full_mask_credit_card AS (val string) RETURNS string ->
    CASE
        WHEN CURRENT_ROLE() IN ('SECURITY_ADMIN', 'ANALYST') THEN val
        ELSE '************'
    END;

-- Create a masking policy for partial masking (last 4 digits visible)
CREATE MASKING POLICY partial_mask_credit_card AS (val string) RETURNS string ->
    CASE
        WHEN CURRENT_ROLE() IN ('FINANCE_ADMIN') THEN val
        WHEN CURRENT_ROLE() IN ('CUSTOMER_SUPPORT') THEN '************' || SUBSTR(val, -4)
        ELSE '************'
    END;

-- Apply the full masking policy to the column
ALTER TABLE customer_data MODIFY COLUMN credit_card_number SET MASKING POLICY full_mask_credit_card;

-- Apply the partial masking policy to the same column (this creates a conflict for resolution)
ALTER TABLE customer_data MODIFY COLUMN credit_card_number SET MASKING POLICY partial_mask_credit_card;

-- Now, observe policy precedence. Snowflake uses the policy with higher precedence (last one applied if no hierarchy specified).
-- For example, if FINANCE_ADMIN views the data:
-- SELECT credit_card_number FROM customer_data;
```

*(Note: Snowflake's policy precedence for multiple policies on the same column without a specific policy hierarchy or conditional application within a single policy can be complex. The example above demonstrates applying multiple policies, where the last applied policy often takes precedence or a combined policy is needed for true "multiple" dynamic masking. A single, comprehensive policy with nested `CASE` statements is often preferred for managing different masking levels on one column.)*

## Explanation of the Code

- `CREATE TABLE customer_data (...)`: This statement creates a sample table `customer_data` to hold information, including a sensitive `credit_card_number`.
- `INSERT INTO customer_data VALUES (...)`: Populates the `customer_data` table with example records.
- `CREATE MASKING POLICY full_mask_credit_card AS (val string) RETURNS string -> ...`: This defines a masking policy named `full_mask_credit_card`.
    - **What it does**: If the `CURRENT_ROLE()` is `SECURITY_ADMIN` or `ANALYST`, the original value (`val`) is returned. Otherwise, it returns `************`, completely masking the credit card number.
    - **How it changes the result set**: For roles other than `SECURITY_ADMIN` or `ANALYST`, the `credit_card_number` column will display `************`.
- `CREATE MASKING POLICY partial_mask_credit_card AS (val string) RETURNS string -> ...`: This defines another masking policy named `partial_mask_credit_card`.
    - **What it does**: If the `CURRENT_ROLE()` is `FINANCE_ADMIN`, the original value is returned. If the role is `CUSTOMER_SUPPORT`, it shows `************` followed by the last four digits. For any other role, it fully masks.
    - **How it changes the result set**: For `CUSTOMER_SUPPORT`, `credit_card_number` will show partial data; for `FINANCE_ADMIN`, full data; for others, full mask.
- `ALTER TABLE customer_data MODIFY COLUMN credit_card_number SET MASKING POLICY ...`: These statements apply the created masking policies to the `credit_card_number` column. When multiple policies are applied to the same column, Snowflake determines which policy (or combination of policies) is effective based on internal precedence rules. Often, it's about combining logic into one policy or the last policy applied taking precedence unless a more complex policy is designed to handle multiple conditions.

## When to Use

1.  **Regulatory Compliance:** When different regulatory bodies or internal policies dictate varied access levels to the same sensitive data.
    *   *Scenario:* A `patient_ssn` column in a healthcare database needs to be fully masked for general users, partially visible (e.g., last 4 digits) for billing specialists, and fully visible for specific compliance officers.
2.  **Role-Based Data Access:** When different user roles within an organization require distinct views of sensitive information.
    *   *Scenario:* A `salary` column in an HR database is fully visible to HR administrators, but only visible as a salary band (e.g., "$50k-$70k") to line managers, and completely masked for general employees.
3.  **Multi-Tenant Environments:** In applications serving multiple clients, where each client's data must be protected according to their specific agreements.
    *   *Scenario:* A `customer_bank_account` column might be masked with different patterns or entirely hidden based on the tenant ID associated with the user accessing the data.

## When Not to Use

1.  **Simple Masking Requirements:** If a column only ever needs a single, static masking rule regardless of user or context. Applying multiple policies unnecessarily complicates management.
    *   *Inefficiency:* Over-engineering for a simple scenario leads to more complex policy definitions and potential confusion without added benefit.
2.  **Performance-Critical Read-Heavy Workloads:** While masking policies are generally optimized, overly complex nested `CASE` statements within a single policy or evaluating multiple policies on every read can introduce minor overhead.
    *   *Inefficiency:* If response time is absolutely paramount and the masking logic is extremely intricate, evaluating complex policies on every `SELECT` operation could marginally impact performance, though usually negligible in Snowflake.
3.  **Unclear Access Requirements:** If the data access rules for different roles are not clearly defined or frequently change.
    *   *Wrong approach:* Without clear requirements, creating multiple, possibly conflicting policies can lead to incorrect data exposure or unnecessary data hiding, causing operational issues.

## Common Mistakes

1.  **Conflicting Policies without Resolution:** Applying multiple policies to a single column without understanding Snowflake's policy precedence or how to consolidate logic into a single effective policy.
    *   *Result:* Unexpected masking behavior, where either the wrong policy applies or data is not masked as intended.
2.  **Over-Complicating Single Policy Logic:** Instead of managing truly separate policies, developers sometimes cram all conditions into one massive `CASE` statement, making it hard to read, maintain, and test.
    *   *Result:* Unmaintainable code; changes for one role might inadvertently affect another, and debugging becomes a nightmare.
3.  **Not Testing All Role Scenarios:** Failing to test the masking behavior from the perspective of every relevant role (`SECURITY_ADMIN`, `ANALYST`, `CUSTOMER_SUPPORT`, etc.) after applying policies.
    *   *Result:* Security vulnerabilities where sensitive data is exposed to unauthorized users, or legitimate users are blocked from necessary data.

## Expected Output

When managing multiple masking policies effectively (typically by consolidating conditional logic into a single, comprehensive policy per column), the `credit_card_number` column would show different values based on the role of the user querying the `customer_data` table.

For a user with the `SECURITY_ADMIN` or `FINANCE_ADMIN` role:

| CUSTOMER_ID | FULL_NAME         | EMAIL             | CREDIT_CARD_NUMBER     |
|-------------|-------------------|-------------------|------------------------|
| 1           | Alice Wonderland  | alice@example.com | 1234-5678-9012-3456    |
| 2           | Bob The Builder   | bob@example.com   | 9876-5432-1098-7654    |

For a user with the `CUSTOMER_SUPPORT` role:

| CUSTOMER_ID | FULL_NAME         | EMAIL             | CREDIT_CARD_NUMBER     |
|-------------|-------------------|-------------------|------------------------|
| 1           | Alice Wonderland  | alice@example.com | ************3456       |
| 2           | Bob The Builder   | bob@example.com   | ************7654       |

For a user with a role not explicitly allowed full or partial view (e.g., a `GENERAL_USER` role):

| CUSTOMER_ID | FULL_NAME         | EMAIL             | CREDIT_CARD_NUMBER     |
|-------------|-------------------|-------------------|------------------------|
| 1           | Alice Wonderland  | alice@example.com | ************           |
| 2           | Bob The Builder   | bob@example.com   | ************           |

**Explanation of Columns:**
- `CUSTOMER_ID`: The unique identifier for the customer, unaffected by masking.
- `FULL_NAME`: The customer's full name, unaffected by masking.
- `EMAIL`: The customer's email address, unaffected by masking.
- `CREDIT_CARD_NUMBER`: This column shows the masked (or unmasked) credit card number based on the current user's role and the logic defined in the applied masking policies.
