# Compliance Use Cases for Row Level Security in Snowflake

Row Level Security (RLS) in Snowflake is a powerful feature that allows organizations to control access to individual rows in a table based on the characteristics of the user querying the data. It ensures that users only see the data they are authorized to view, which is crucial for maintaining data privacy, adhering to regulatory requirements, and implementing data governance strategies. By dynamically filtering data at the row level, RLS helps prevent unauthorized data exposure without needing to create multiple copies of data or complex view structures.

**Short Explanation**

Row Level Security dynamically filters data based on user roles or attributes, ensuring that each user only sees the relevant rows. It solves the problem of granular data access control, which is vital for protecting sensitive information and complying with regulations like GDPR or CCPA. RLS is commonly used in multi-tenant applications or for segregating data by department or region in analytical workflows.

- **What problem does this SQL feature solve?** RLS solves the problem of ensuring granular data access control, where different users or roles need to see different subsets of data within the same table, without exposing sensitive information.
- **Why is it important in databases or analytics?** It's critical for protecting sensitive data, maintaining privacy, and meeting regulatory compliance requirements (e.g., GDPR, HIPAA, CCPA) by enforcing data minimization and robust audit trails.
- **Where is it commonly used in real workflows?** It's widely used in multi-tenant environments, for departmental data segregation (e.g., sales data by region), and in scenarios where access to customer-specific or patient-specific information must be restricted to authorized personnel only.

## Implementation Example

Here's an example of implementing Row Level Security in Snowflake using a Row Access Policy to restrict sales data visibility based on a user's assigned region.

```sql
-- 1. Create a table for sales data
CREATE OR REPLACE TABLE sales_data (
    sale_id INTEGER,
    region STRING,
    amount FLOAT,
    customer_email STRING
);

-- Insert sample data
INSERT INTO sales_data (sale_id, region, amount, customer_email) VALUES
(1, 'EAST', 100.00, 'alice@example.com'),
(2, 'WEST', 150.00, 'bob@example.com'),
(3, 'EAST', 200.00, 'charlie@example.com'),
(4, 'NORTH', 50.00, 'diana@example.com'),
(5, 'WEST', 120.00, 'eve@example.com');

-- 2. Create an entitlements table to map roles to regions
CREATE OR REPLACE TABLE role_region_mapping (
    role_name STRING,
    region STRING
);

-- Insert mappings
INSERT INTO role_region_mapping (role_name, region) VALUES
('SALES_EAST_ANALYST', 'EAST'),
('SALES_WEST_ANALYST', 'WEST'),
('SALES_GLOBAL_ANALYST', 'EAST'),
('SALES_GLOBAL_ANALYST', 'WEST'),
('SALES_GLOBAL_ANALYST', 'NORTH');

-- 3. Create the Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy
AS (region STRING)
RETURNS BOOLEAN ->
    EXISTS (
        SELECT 1 FROM role_region_mapping
        WHERE role_name = CURRENT_ROLE()
        AND region = sales_region_policy.region
    )
    OR
    EXISTS (
        SELECT 1 FROM role_region_mapping
        WHERE role_name = CURRENT_ROLE()
        AND region = 'ALL' -- Allows a global role to see all regions if mapped
    );

-- 4. Apply the Row Access Policy to the sales_data table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region);

-- 5. Create roles and grant them to a user (example)
-- Assuming user 'TEST_USER' exists
-- GRANT ROLE SALES_EAST_ANALYST TO USER TEST_USER;
-- GRANT ROLE SALES_WEST_ANALYST TO USER TEST_USER;
-- GRANT ROLE SALES_GLOBAL_ANALYST TO USER TEST_USER;

-- 6. Test as a specific role
-- USE ROLE SALES_EAST_ANALYST;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_WEST_ANALYST;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_GLOBAL_ANALYST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This implementation uses a Snowflake Row Access Policy to enforce RLS.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table `sales_data` to hold transactional information, including a `region` column that will be used for filtering.
- **`INSERT INTO sales_data (...)`**: Populates the `sales_data` table with example records across different regions.
- **`CREATE OR REPLACE TABLE role_region_mapping (...)`**: This creates a lookup table `role_region_mapping` which stores the association between Snowflake roles and the regions they are authorized to access. This decouples access logic from the policy itself, making it more scalable and maintainable.
- **`INSERT INTO role_region_mapping (...)`**: Populates the mapping table with example role-to-region assignments. For instance, `SALES_EAST_ANALYST` can see `EAST` region data.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy AS (region STRING) RETURNS BOOLEAN ->...`**: This defines the actual Row Access Policy.
    - `AS (region STRING)`: Specifies that the policy expects a column named `region` of type `STRING` from the table it's applied to.
    - `RETURNS BOOLEAN`: Indicates that the policy will return `TRUE` if a row should be visible, and `FALSE` if it should be hidden.
    - `EXISTS (SELECT 1 FROM role_region_mapping WHERE role_name = CURRENT_ROLE() AND region = sales_region_policy.region)`: This is the core logic. For each row in `sales_data`, it checks if the `CURRENT_ROLE()` (the role of the user executing the query) is present in the `role_region_mapping` table and if that role is authorized for the `region` of the current row being evaluated.
    - `OR EXISTS (SELECT 1 FROM role_region_mapping WHERE role_name = CURRENT_ROLE() AND region = 'ALL')`: This additional condition allows for "global" roles. If a role is mapped to `'ALL'` in the `role_region_mapping` table, that role can see data from any region.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region)`**: This command applies the `sales_region_policy` to the `sales_data` table. From this point forward, any query on `sales_data` will automatically have this policy applied, filtering rows based on the querying user's active role and the `region` column.
- **`USE ROLE <ROLE_NAME>; SELECT * FROM sales_data;`**: These are example commands to demonstrate how different roles would perceive the `sales_data` table after the policy is applied.

## When to Use

1. **Regulatory Compliance (GDPR, HIPAA, CCPA):** When regulations require strict data access based on user authorization, such as limiting access to Personally Identifiable Information (PII) or Protected Health Information (PHI) to specific teams or individuals. For example, a customer service agent should only see data for customers they are actively serving, or a medical researcher only anonymized patient data.
2. **Multi-Tenant Architectures:** In SaaS applications where different customers share the same database infrastructure but must only access their own data. RLS ensures that each tenant's data is isolated from others, preventing cross-tenant data leakage.
3. **Departmental/Regional Data Segregation:** For large organizations where different departments or regional teams should only see data relevant to their operational scope. For instance, a sales manager in the "EAST" region only sees sales figures for the "EAST" region.

## When Not to Use

1. **Overhead on Small, Non-Sensitive Tables:** Applying RLS to small tables with non-sensitive data can introduce unnecessary query overhead without providing significant security benefits. The performance impact of policies should be considered.
2. **Complex Business Logic Requiring Aggregations First:** If the access control logic is extremely complex and requires aggregations or complex transformations *before* row filtering (which RLS applies at query time), secure views might be a better approach, though Row Access Policies are generally preferred for maintainability.
3. **Column-Level Security Needs:** RLS only restricts entire rows. If the requirement is to mask or redact specific columns (e.g., credit card numbers, email addresses) while still showing the row, Dynamic Data Masking is the appropriate solution, often used in conjunction with RLS for comprehensive security.

## Common Mistakes

1. **Embedding Logic Directly in Policy:** Hardcoding role names or entitlements directly within the Row Access Policy definition instead of using a separate entitlements table. This makes policies hard to maintain, scale, and audit.
2. **Lack of Testing Across Roles:** Not thoroughly testing the RLS policy with all relevant roles (and non-relevant roles) to ensure that access is granted correctly and denied appropriately. This can lead to either over-exposure or under-exposure of data.
3. **Ignoring Performance Impact:** Not considering the potential performance overhead of complex RLS policies, especially on very large tables or with frequently accessed data. Sub-optimal policy logic or unindexed entitlement tables can slow down queries.

## Expected Output

When a user queries the `sales_data` table with the `sales_region_policy` enabled, the result will be a subset of the original table, containing only the rows that the user's active role is authorized to see based on the `region` column.

**Example Table (if `SALES_EAST_ANALYST` queries `sales_data`):**

| SALE_ID | REGION | AMOUNT | CUSTOMER_EMAIL |
|---------|--------|--------|-------------------|
| 1 | EAST | 100.00 | alice@example.com |
| 3 | EAST | 200.00 | charlie@example.com |

- **SALE_ID**: The unique identifier for each sale.
- **REGION**: The geographical region where the sale occurred. This column is used by the RLS policy to filter data.
- **AMOUNT**: The monetary value of the sale.
- **CUSTOMER_EMAIL**: The email address of the customer.

In this example, the `SALES_EAST_ANALYST` role is configured to only see data from the 'EAST' region. Therefore, any query executed under this role on the `sales_data` table would only return the two rows where the `REGION` is 'EAST'. Rows from 'WEST' and 'NORTH' would be automatically filtered out by the Row Access Policy before being presented to the user.
