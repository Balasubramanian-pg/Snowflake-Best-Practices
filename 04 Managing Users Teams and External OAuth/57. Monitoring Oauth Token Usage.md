# Monitoring OAuth Token Usage in Snowflake

Monitoring OAuth token usage in Snowflake is a crucial aspect of maintaining a robust security posture. OAuth (Open Authorization) is an open standard for access delegation, commonly used in Snowflake to allow third-party applications to access Snowflake data on behalf of a user, without sharing the user's actual credentials. It exists to provide secure, delegated access, ensuring that applications only have the necessary permissions for a limited time.

**Short Explanation**

Monitoring OAuth token usage helps you track which applications and users are leveraging OAuth for access to your Snowflake environment. This is essential for auditing security, identifying unusual access patterns, and ensuring compliance with data governance policies. By observing token generation, refresh, and revocation, you can detect potential misuse or unauthorized access.

- **What problem does this SQL feature solve?** It helps prevent unauthorized access and data breaches by providing visibility into how and when delegated access is being used.
- **Why is it important in databases or analytics?** In data platforms like Snowflake, where sensitive data often resides, tracking all forms of access, especially delegated access via OAuth, is paramount for security and compliance.
- **Where is it commonly used in real workflows?** Security audits, compliance reporting (e.g., GDPR, HIPAA), incident response investigations, and general security monitoring dashboards.

## Implementation Example

While Snowflake doesn't have a direct view specifically named `OAUTH_TOKEN_USAGE`, you can monitor OAuth-related activities through the `LOGIN_HISTORY` and `QUERY_HISTORY` views within the `SNOWFLAKE.ACCOUNT_USAGE` schema. These views capture details about successful and failed logins, including the authentication method used, and all executed queries.

To monitor OAuth logins:

```sql
SELECT
    EVENT_TIMESTAMP,
    USER_NAME,
    CLIENT_IP,
    AUTHENTICATION_METHOD,
    IS_SUCCESS,
    ERROR_MESSAGE
FROM
    SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY
WHERE
    AUTHENTICATION_METHOD = 'OAUTH'
    AND EVENT_TIMESTAMP >= DATEADD(day, -30, CURRENT_TIMESTAMP())
ORDER BY
    EVENT_TIMESTAMP DESC;
```

To monitor queries executed via OAuth authenticated sessions:

```sql
SELECT
    QUERY_ID,
    QUERY_TEXT,
    USER_NAME,
    ROLE_NAME,
    WAREHOUSE_NAME,
    START_TIME,
    END_TIME,
    EXECUTION_STATUS
FROM
    SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    -- Assuming your OAuth users or roles have a specific naming convention,
    -- or you can join with LOGIN_HISTORY to filter by sessions authenticated with OAUTH.
    -- For simplicity, let's assume we filter by a specific user or a general pattern
    USER_NAME IN (SELECT DISTINCT USER_NAME FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY WHERE AUTHENTICATION_METHOD = 'OAUTH')
    AND START_TIME >= DATEADD(day, -30, CURRENT_TIMESTAMP())
ORDER BY
    START_TIME DESC;
```

## Explanation of the Code

**First Query (LOGIN_HISTORY):**

- `SELECT EVENT_TIMESTAMP, USER_NAME, CLIENT_IP, AUTHENTICATION_METHOD, IS_SUCCESS, ERROR_MESSAGE`: This clause specifies the columns we want to retrieve.
    - `EVENT_TIMESTAMP`: When the login attempt occurred.
    - `USER_NAME`: The user involved in the login.
    - `CLIENT_IP`: The IP address from which the login attempt originated.
    - `AUTHENTICATION_METHOD`: This is key, as it tells us how the user authenticated.
    - `IS_SUCCESS`: Whether the login attempt was successful.
    - `ERROR_MESSAGE`: Any error message if the login failed.
- `FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY`: Specifies the view containing login activity data.
- `WHERE AUTHENTICATION_METHOD = 'OAUTH' AND EVENT_TIMESTAMP >= DATEADD(day, -30, CURRENT_TIMESTAMP())`: This filters the results to only include login attempts that used OAuth as the authentication method within the last 30 days.
- `ORDER BY EVENT_TIMESTAMP DESC`: Sorts the results by the most recent login attempts first.

**Second Query (QUERY_HISTORY):**

- `SELECT QUERY_ID, QUERY_TEXT, USER_NAME, ROLE_NAME, WAREHOUSE_NAME, START_TIME, END_TIME, EXECUTION_STATUS`: This clause selects details about executed queries.
    - `QUERY_ID`: Unique identifier for the query.
    - `QUERY_TEXT`: The actual SQL query executed.
    - `USER_NAME`: The user who ran the query.
    - `ROLE_NAME`: The role used to execute the query.
    - `WAREHOUSE_NAME`: The virtual warehouse used.
    - `START_TIME`, `END_TIME`: When the query started and ended.
    - `EXECUTION_STATUS`: The status of the query execution.
- `FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY`: Specifies the view containing query execution data.
- `WHERE USER_NAME IN (SELECT DISTINCT USER_NAME FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY WHERE AUTHENTICATION_METHOD = 'OAUTH') AND START_TIME >= DATEADD(day, -30, CURRENT_TIMESTAMP())`: This filters queries to those executed by users who have authenticated via OAuth, specifically within the last 30 days. This subquery effectively identifies users who logged in using OAuth.
- `ORDER BY START_TIME DESC`: Sorts the results by the most recent query executions first.

## When to Use

1.  **Security Audits:** Regularly review OAuth login and activity to ensure that delegated access is being used as expected and to detect any unauthorized access attempts.
    *   *Scenario:* A monthly audit requires a report of all OAuth-authenticated sessions and the data they accessed.
2.  **Compliance Reporting:** For regulatory compliance (e.g., SOC 2, PCI DSS), you need to demonstrate control over who accesses sensitive data, and how. OAuth monitoring provides evidence for delegated access.
    *   *Scenario:* An auditor requests proof that only authorized third-party applications integrated via OAuth accessed PII data.
3.  **Troubleshooting Application Access:** If an application integrated via OAuth is experiencing connectivity issues or unexpected permissions, monitoring token usage can help diagnose the problem.
    *   *Scenario:* A data visualization tool connected via OAuth suddenly can't access a specific table; monitoring `QUERY_HISTORY` for that OAuth user reveals permission errors.

## When Not to Use

1.  **Direct User Authentication Issues:** If a user is having trouble logging in with their username and password, focusing solely on OAuth token usage might be irrelevant.
    *   *Situation:* A user reports they cannot log in to the Snowflake UI. Checking `LOGIN_HISTORY` filtered by `OAUTH` won't directly help if they are using standard authentication.
2.  **Performance Tuning of Queries Not Related to OAuth Apps:** While `QUERY_HISTORY` is excellent for performance, filtering it strictly by OAuth users when the performance issue is broader will limit diagnostic scope.
    *   *Situation:* Overall warehouse performance is slow. Filtering `QUERY_HISTORY` only for OAuth-related queries might miss the real culprits that are using traditional authentication.
3.  **Monitoring Non-Access Specific Security Configurations:** OAuth monitoring focuses on access patterns. It won't directly tell you about other security aspects like network policies, data masking configurations, or encryption settings.
    *   *Situation:* You need to verify if a new network policy is correctly blocking certain IP ranges. OAuth usage logs won't provide this information directly.

## Common Mistakes

1.  **Not correlating `LOGIN_HISTORY` with `QUERY_HISTORY`:** Only looking at logins or only looking at queries in isolation misses the full picture of an OAuth session's activity.
2.  **Ignoring failed login attempts:** Focusing only on successful OAuth logins can hide attempts at unauthorized access or misconfigurations.
3.  **Lack of alerting on anomalous patterns:** Not setting up alerts for unusual OAuth activity (e.g., logins from new IP addresses, too many failed attempts, or access to sensitive data by unexpected OAuth clients) can delay incident detection.
4.  **Assuming all external access is OAuth:** Other methods like service accounts with key-pair authentication also exist, and OAuth monitoring won't cover them.

## Expected Output

The `LOGIN_HISTORY` query provides a list of login events, indicating which were successful and which failed, along with the authentication method.

**Example Table: `LOGIN_HISTORY` (filtered for OAuth)**

| EVENT_TIMESTAMP         | USER_NAME       | CLIENT_IP      | AUTHENTICATION_METHOD | IS_SUCCESS | ERROR_MESSAGE |
| :---------------------- | :-------------- | :------------- | :-------------------- | :--------- | :------------ |
| 2026-02-28 10:30:05.123 | APP_ANALYTICS_U | 192.168.1.100  | OAUTH                 | TRUE       | NULL          |
| 2026-02-28 10:28:15.456 | APP_ANALYTICS_U | 192.168.1.100  | OAUTH                 | FALSE      | Invalid token |

- **EVENT_TIMESTAMP:** Shows the exact time the login attempt occurred.
- **USER_NAME:** Identifies the user (or service user) attempting to log in.
- **CLIENT_IP:** The IP address from which the connection originated.
- **AUTHENTICATION_METHOD:** Clearly states 'OAUTH' for these specific records.
- **IS_SUCCESS:** Indicates if the login was successful (TRUE) or not (FALSE).
- **ERROR_MESSAGE:** Provides details if the login failed.

The `QUERY_HISTORY` query provides a detailed log of all SQL statements executed by users who authenticated via OAuth.

**Example Table: `QUERY_HISTORY` (filtered for OAuth users)**

| QUERY_ID                             | QUERY_TEXT                                | USER_NAME       | ROLE_NAME        | WAREHOUSE_NAME | START_TIME              | END_TIME                | EXECUTION_STATUS |
| :----------------------------------- | :---------------------------------------- | :-------------- | :--------------- | :------------- | :---------------------- | :---------------------- | :--------------- |
| 123abc45-de6f-7890-1234-567890abcdef | `SELECT COUNT(*) FROM SALES_DATA;`        | APP_ANALYTICS_U | ANALYST_ROLE     | REPORT_WH      | 2026-02-28 10:30:10.000 | 2026-02-28 10:30:12.000 | SUCCESS          |
| fedcba98-7654-3210-fedc-ba9876543210 | `SELECT * FROM CUSTOMER_PII LIMIT 10;`    | APP_ANALYTICS_U | ANALYST_ROLE     | REPORT_WH      | 2026-02-28 10:31:00.000 | 2026-02-28 10:31:01.000 | SUCCESS          |

- **QUERY_ID:** A unique identifier for each executed query.
- **QUERY_TEXT:** The actual SQL statement that was run.
- **USER_NAME:** The user account that executed the query.
- **ROLE_NAME:** The role under which the query was executed.
- **WAREHOUSE_NAME:** The compute resource used for the query.
- **START_TIME, END_TIME:** The start and end times of the query execution.
- **EXECUTION_STATUS:** Indicates if the query ran successfully.
