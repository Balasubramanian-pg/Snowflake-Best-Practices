# Building Concurrency Observability Dashboards in Snowflake

Concurrency observability dashboards in Snowflake are tools that allow database administrators and developers to monitor and analyze the simultaneous execution of queries and other operations within their Snowflake environment. They exist to provide insights into how effectively the data warehouse is handling multiple workloads, identify potential bottlenecks, and ensure optimal performance and resource utilization.

**Short Explanation**

This concept focuses on visualizing how many queries and tasks are running at the same time in Snowflake. It helps identify if the system is overloaded or underutilized, ensuring smooth operations.
- **What problem does this SQL feature solve?** It solves the problem of understanding database performance under varying loads, helping pinpoint concurrency issues like excessive queueing or resource contention.
- **Why is it important in databases or analytics?** It's crucial for maintaining performance, optimizing costs, and ensuring that users get timely results, especially in high-demand analytical environments.
- **Where is it commonly used in real workflows?** Data platform teams and DevOps engineers use these dashboards to continuously monitor Snowflake, troubleshoot performance problems, and plan for capacity.

## Implementation Example

```sql
SELECT
    DATE_TRUNC('hour', start_time) AS hour_of_day,
    warehouse_name,
    COUNT(query_id) AS total_queries,
    AVG(execution_time) AS avg_execution_time_ms,
    AVG(queued_provisioning_time + queued_repair_time + queued_overload_time) AS avg_queue_time_ms,
    COUNT(DISTINCT session_id) AS unique_sessions,
    SUM(CASE WHEN query_type IN ('SELECT', 'CREATE TABLE AS SELECT', 'INSERT') THEN 1 ELSE 0 END) AS dml_queries,
    SUM(CASE WHEN query_type NOT IN ('SELECT', 'CREATE TABLE AS SELECT', 'INSERT') THEN 1 ELSE 0 END) AS ddl_queries
FROM
    snowflake.account_usage.query_history
WHERE
    start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    AND warehouse_name IS NOT NULL
GROUP BY
    1, 2
ORDER BY
    hour_of_day DESC, warehouse_name;
```

## Explanation of the Code

This SQL query analyzes the `query_history` view in Snowflake's `account_usage` schema to gather metrics relevant to concurrency over the last 7 days.

- **SELECT** specifies the columns to be retrieved:
    - `DATE_TRUNC('hour', start_time) AS hour_of_day`: This truncates the `start_time` of each query to the nearest hour, allowing aggregation of data on an hourly basis. It helps group query activity by time.
    - `warehouse_name`: The name of the virtual warehouse where the query ran. This helps analyze concurrency per warehouse.
    - `COUNT(query_id) AS total_queries`: Counts the total number of queries executed within each hour for each warehouse. This indicates the volume of work.
    - `AVG(execution_time) AS avg_execution_time_ms`: Calculates the average time queries spent executing, in milliseconds. This provides insight into query efficiency.
    - `AVG(queued_provisioning_time + queued_repair_time + queued_overload_time) AS avg_queue_time_ms`: Computes the average time queries spent waiting in queues due to various reasons. High queue times indicate concurrency bottlenecks.
    - `COUNT(DISTINCT session_id) AS unique_sessions`: Counts the number of distinct user sessions active within the hour for each warehouse. This reflects the number of concurrent users.
    - `SUM(CASE WHEN query_type IN ('SELECT', 'CREATE TABLE AS SELECT', 'INSERT') THEN 1 ELSE 0 END) AS dml_queries`: Counts Data Manipulation Language (DML) queries.
    - `SUM(CASE WHEN query_type NOT IN ('SELECT', 'CREATE TABLE AS SELECT', 'INSERT') THEN 1 ELSE 0 END) AS ddl_queries`: Counts Data Definition Language (DDL) and other query types. These help categorize workloads.

- **FROM** `snowflake.account_usage.query_history`: Specifies the source table, which is a view provided by Snowflake containing historical query metadata for the account. It provides the raw data for analysis.

- **WHERE** `start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP()) AND warehouse_name IS NOT NULL`: Filters the data to only include queries that started within the last 7 days and were executed on a named warehouse (excluding internal Snowflake processes). This limits the analysis to a recent and relevant timeframe.

- **GROUP BY** `1, 2`: Groups the result set by the `hour_of_day` (first selected column) and `warehouse_name` (second selected column). This aggregates the metrics hourly for each warehouse.

- **ORDER BY** `hour_of_day DESC, warehouse_name`: Sorts the final result set, showing the most recent hours first, and then by warehouse name for consistency. It organizes the output for easier readability.

## When to Use

1.  **Performance Troubleshooting**: When users report slow query performance or increased wait times, these dashboards can quickly identify periods of high concurrency or excessive queueing on specific warehouses.
    *   *Example Scenario*: A daily reporting job is running much slower than usual. The dashboard might show a sudden spike in `total_queries` and `avg_queue_time_ms` around the job's start time, indicating contention.
2.  **Capacity Planning**: To make informed decisions about scaling warehouses up or down, or creating new warehouses. Observing peak concurrency allows for proactive resource allocation.
    *   *Example Scenario*: The dashboard consistently shows high `total_queries` and low `avg_execution_time_ms` during business hours, but spikes in `avg_queue_time_ms` during nightly ETL loads, suggesting the need for a larger warehouse for ETL.
3.  **Cost Optimization**: Identifying underutilized warehouses or times when concurrency is low can lead to opportunities to suspend or resize warehouses, reducing Snowflake credits consumption.
    *   *Example Scenario*: A particular warehouse shows very few `total_queries` and `unique_sessions` during off-peak hours. The dashboard helps justify setting up auto-suspend for this warehouse or resizing it downwards.

## When Not to Use

1.  **Debugging Specific Query Logic**: While concurrency dashboards show *when* performance issues occur, they don't explain *why* a particular query is slow due to inefficient SQL or data modeling.
    *   *Situation*: A single complex query is taking hours to run. The dashboard might show it as one of many running queries, but won't provide details on its execution plan or data scan inefficiencies.
2.  **Real-time Micro-optimizations**: For extremely low-latency, real-time monitoring of individual query execution steps, more specialized tools like Snowflake's Query Profile are more appropriate than a dashboard summarizing historical concurrency.
    *   *Situation*: A developer needs to understand the exact stages and operators a query is using to identify bottlenecks within that specific query.
3.  **Analyzing Data Freshness**: These dashboards focus on resource utilization and query execution, not the timeliness or freshness of the underlying data being queried.
    *   *Situation*: A user complains that a report shows outdated data. The dashboard would tell you if the query ran successfully, but not if the data source was refreshed properly.

## Common Mistakes

1.  **Ignoring Queue Time**: Focusing solely on `execution_time` without considering `queued_time` can mask concurrency issues. A query might have a fast execution time but spend a long time waiting in a queue.
2.  **One-Size-Fits-All Warehouse**: Using a single warehouse for all workloads (ETL, reporting, ad-hoc analysis) without monitoring concurrency often leads to resource contention and poor performance for some or all users.
3.  **Not Segmenting by Warehouse**: Aggregating all query history without breaking it down by `warehouse_name` makes it impossible to identify which specific resources are bottlenecked or underutilized.

## Expected Output

The resulting dataset should provide an hourly summary of query activity and performance metrics for each virtual warehouse over the specified period. It offers a structured view of concurrency patterns.

| HOUR_OF_DAY         | WAREHOUSE_NAME | TOTAL_QUERIES | AVG_EXECUTION_TIME_MS | AVG_QUEUE_TIME_MS | UNIQUE_SESSIONS | DML_QUERIES | DDL_QUERIES |
| :------------------ | :------------- | :------------ | :-------------------- | :---------------- | :-------------- | :---------- | :---------- |
| 2026-03-14 10:00:00 | REPORTING_WH   | 150           | 550                   | 10                | 12              | 140         | 10          |
| 2026-03-14 09:00:00 | ETL_WH         | 80            | 1200                  | 500               | 3               | 75          | 5           |
| 2026-03-14 09:00:00 | REPORTING_WH   | 120           | 600                   | 5                 | 10              | 110         | 10          |

-   **HOUR_OF_DAY**: The hour timestamp, showing when the query activity occurred.
-   **WAREHOUSE_NAME**: The specific Snowflake virtual warehouse where the queries ran.
-   **TOTAL_QUERIES**: The total number of queries executed within that hour on that warehouse.
-   **AVG_EXECUTION_TIME_MS**: The average time, in milliseconds, that queries spent actively running.
-   **AVG_QUEUE_TIME_MS**: The average time, in milliseconds, queries spent waiting to be executed.
-   **UNIQUE_SESSIONS**: The count of distinct user sessions that submitted queries.
-   **DML_QUERIES**: The count of data manipulation language queries (e.g., SELECT, INSERT).
-   **DDL_QUERIES**: The count of data definition language queries (e.g., CREATE, ALTER).
