# Using Clustering Keys to Improve Query Efficiency in Snowflake

Clustering keys are a Snowflake performance feature that tells the engine how to physically group rows inside a table’s micro-partitions. By ordering data on columns you filter or join by most often, Snowflake can skip whole chunks of data (partition pruning) and run queries faster while scanning less. They exist to give Snowflake a nudge when natural loading order isn’t enough for big, frequently queried tables.

**Short Explanation**

- What problem does this SQL feature solve?  
  Large tables accumulate data in scattered micro-partitions; queries that filter on date, region, or IDs end up scanning everything. Clustering keys co-locate similar values so those scans can be pruned.  
- Why is it important in databases or analytics?  
  Less data scanned means faster response times and lower compute costs, which matters for multi-terabyte fact tables and daily SLAs.  
- Where is it commonly used in real workflows?  
  In analytics pipelines for time-series sales logs, event logs, or IoT tables where queries repeatedly filter by date, customer_id, or status.  

## Implementation Example

```sql
CREATE OR REPLACE TABLE sales_events (
    event_time TIMESTAMP,
    customer_id NUMBER,
    region STRING,
    amount NUMBER
)
CLUSTER BY (event_time, region);

SELECT
    region,
    SUM(amount) AS daily_rev
FROM sales_events
WHERE event_time BETWEEN '2025-12-01' AND '2025-12-07'
GROUP BY region
ORDER BY daily_rev DESC;
```

## Explanation of the Code

- **CREATE TABLE … CLUSTER BY (event_time, region)**  
  Defines the clustering key. Snowflake will try to group rows with nearby event_time values and the same region in the same micro-partitions.  
- **SELECT**  
  Returns the columns we want: region and the aggregated sum.  
- **FROM sales_events**  
  Source table that is clustered; pruning kicks in here.  
- **WHERE event_time BETWEEN …**  
  Filter on the leading clustering column; Snowflake uses micro-partition metadata to skip partitions outside the date range, shrinking the result set early.  
- **GROUP BY region**  
  Aggregates the already pruned rows; because region is the second key, rows are already partly co-located, helping the aggregation.  
- **ORDER BY daily_rev DESC**  
  Sorts the final aggregated output; it doesn’t affect pruning—it only orders the result set returned to the client.  

## When to Use

- Large fact tables (multi-TB) with recurring filters on a date or timestamp column.  
- Workloads that join or filter on a low-to-moderate cardinality column like region or status.  
- Queries that scan ranges (BETWEEN, >, <) rather than point lookups, where partition pruning gives measurable benefit.  

## When Not to Use

- Tiny tables that are fully scanned anyway; clustering adds maintenance cost for no gain.  
- Tables with heavy, continuous DML on the clustering columns, causing reclustering overhead to outweigh query savings.  
- High-cardinality keys like UUIDs or nanosecond timestamps used directly—pruning is minimal and reclustering is expensive.  

## Common Mistakes

- Choosing 5+ columns in the clustering key, which raises cost without improving pruning.  
- Clustering on columns not actually used in WHERE/JOIN predicates, so queries never benefit.  
- Expecting instant reorganization; clustering is background and existing rows aren’t reordered until reclustering runs.  

## Expected Output

The result set is a small aggregated table listing each region with its total revenue for the week, sorted highest to lowest.

| REGION | DAILY_REV |
|--------|-----------|
| West   | 145320.50 |
| East   | 98210.00  |
| North  | 76045.25  |

- **region**: the grouping key from the table.  
- **daily_rev**: sum of amount for matching rows after pruning; only micro-partitions covering the specified dates are scanned. 【nTPO】
