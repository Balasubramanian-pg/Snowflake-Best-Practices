# Row Access Policies in Snowflake

Row Access Policies in Snowflake provide a robust, centralized, and granular security model that filters the individual rows returned by a query at runtime. They exist to enforce data security and privacy requirements by ensuring that users only see the data they are authorized to access, even when querying the same table. This mechanism allows organizations to meet compliance regulations and maintain data confidentiality without modifying the underlying tables or views.

**Short Explanation**

Row Access Policies allow you to define conditions that determine which rows a user can see from a table based on their role, user, or other attributes. This feature solves the problem of granular data access control, preventing unauthorized users from viewing sensitive information within a dataset. It's incredibly important for maintaining data privacy and regulatory compliance in modern data warehousing and analytics, commonly used in scenarios where different departments or roles need varied access to the same data, such as sales regions, HR data, or customer information.

- **What problem does this SQL feature solve?** It solves the problem of granular data access control, allowing different users to see different subsets of data from the same table without creating multiple views or tables.
- **Why is it important in databases or analytics?** It's crucial for data privacy, security, and regulatory compliance (e.g., GDPR, HIPAA) by ensuring data confidentiality at the row level.
- **Where is it commonly used in real workflows?** It's used in environments where data needs to be segmented by user roles, departments, geographical regions, or specific user attributes, such as multi-tenant applications or sensitive corporate data.

## Implementation Example

```sql
-- 1. Create a table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2),
    customer_email VARCHAR
);

-- 2. Insert some sample data
INSERT INTO sales_data (sale_id, product_name, region, sales_amount, customer_email) VALUES
(1, 'Laptop', 'East', 1200.00, 'alice@example.com'),
(2, 'Mouse', 'West', 25.00, 'bob@example.com'),
(3, 'Keyboard', 'East', 75.00, 'charlie@example.com'),
(4, 'Monitor', 'Central', 300.00, 'diana@example.com'),
(5, 'Webcam', 'West', 50.00, 'eve@example.com');

-- 3. Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy
AS (region VARCHAR, customer_email VARCHAR) RETURNS BOOLEAN ->
    -- Allow users with 'SALES_EAST' role to see 'East' region sales
    (CURRENT_ROLE() IN ('SALES_EAST') AND region = 'East') OR
    -- Allow users with 'SALES_WEST' role to see 'West' region sales
    (CURRENT_ROLE() IN ('SALES_WEST') AND region = 'West') OR
    -- Allow users with 'DATA_ANALYST' role to see all sales but mask customer_email
    (CURRENT_ROLE() IN ('DATA_ANALYST')) OR
    -- Allow the data owner/admin to see everything
    (IS_ROLE_IN_SESSION('ACCOUNTADMIN'));

-- 4. Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region, customer_email);

-- 5. Create roles and grant privileges
CREATE OR REPLACE ROLE SALES_EAST;
CREATE OR REPLACE ROLE SALES_WEST;
CREATE OR REPLACE ROLE DATA_ANALYST;

GRANT SELECT ON TABLE sales_data TO ROLE SALES_EAST;
GRANT SELECT ON TABLE sales_data TO ROLE SALES_WEST;
GRANT SELECT ON TABLE sales_data TO ROLE DATA_ANALYST;

-- Example queries (to be run by users with different roles):
-- USE ROLE SALES_EAST;
-- SELECT * FROM sales_data;

-- USE ROLE SALES_WEST;
-- SELECT * FROM sales_data;

-- USE ROLE DATA_ANALYST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example demonstrates how to create a `ROW ACCESS POLICY` and apply it to a table in Snowflake.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to simulate a dataset that needs row-level security. It defines columns like `sale_id`, `product_name`, `region`, `sales_amount`, and `customer_email`.
- **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with example records, including sales in different regions and associated customer emails.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region VARCHAR, customer_email VARCHAR) RETURNS BOOLEAN -> ...`**: This is the core of the row access policy definition.
    - `sales_policy`: The name of the policy.
    - `AS (region VARCHAR, customer_email VARCHAR)`: Specifies the column(s) from the target table that will be passed into the policy's SQL expression. These columns are used to evaluate the access conditions for each row.
    - `RETURNS BOOLEAN ->`: Indicates that the policy's SQL expression must return a boolean value (TRUE to allow access to the row, FALSE to deny).
    - `(CURRENT_ROLE() IN ('SALES_EAST') AND region = 'East') OR ...`: This is the policy body, which contains the logic for filtering rows. It uses `CURRENT_ROLE()` to check the active role of the user querying the table and `IS_ROLE_IN_SESSION()` for broader checks. It defines that:
        - Users with the `SALES_EAST` role can only see rows where the `region` is 'East'.
        - Users with the `SALES_WEST` role can only see rows where the `region` is 'West'.
        - Users with the `DATA_ANALYST` role can see all rows (though masking could be added separately with masking policies).
        - Users with the `ACCOUNTADMIN` role can see all rows.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region, customer_email);`**: This statement applies the `sales_policy` to the `sales_data` table. When this policy is active, any query on `sales_data` will first pass the `region` and `customer_email` values of each row to the `sales_policy` to determine visibility.
- **`CREATE OR REPLACE ROLE ...` and `GRANT SELECT ON TABLE ...`**: These commands set up specific roles (`SALES_EAST`, `SALES_WEST`, `DATA_ANALYST`) and grant them `SELECT` privileges on the `sales_data` table. This is essential because users need privileges to query the table, and the row access policy then dictates *which* rows they can see based on their active role.

## When to Use

1.  **Compliance and Regulatory Requirements:** When dealing with sensitive data that falls under regulations like GDPR, HIPAA, or CCPA, and you need to ensure that only authorized individuals can view specific rows of data.
    *   *Example Scenario:* A healthcare database contains patient records. A row access policy can ensure that doctors only see records for their assigned patients, and administrators only see metadata without patient details, while ensuring data scientists can access anonymized data.
2.  **Multi-Tenant Applications:** In a SaaS application where multiple organizations (tenants) share the same underlying database tables. Each tenant should only see their own data.
    *   *Example Scenario:* A CRM application stores customer data for multiple companies in a single `customers` table. A row access policy could filter rows based on a `tenant_id` column, ensuring each company's users only access their own customer information.
3.  **Departmental or Regional Data Segregation:** When different departments or geographical regions within an organization need access to the same logical table, but only to data relevant to their specific area.
    *   *Example Scenario:* A global sales database where sales managers for the 'Europe' region should only see sales records from Europe, and 'Asia' managers only from Asia, even though all data resides in one `sales` table.

## When Not to Use

1.  **Simple Table-Level Access Control:** If the requirement is simply to grant or deny access to an entire table or view for a specific role. Standard `GRANT` and `REVOKE` DCL statements are more straightforward and efficient.
    *   *Inefficient Scenario:* Using a row access policy to always filter out all rows if a user doesn't have a specific role, instead of just revoking `SELECT` privileges on the table.
2.  **Performance-Critical OLTP Workloads:** While Snowflake's policies are optimized, applying complex row access policies to very high-throughput OLTP (Online Transaction Processing) tables with extreme latency requirements might introduce overhead, especially if the policy logic is intricate or involves many joins.
    *   *Inefficient Scenario:* A policy that performs subqueries or complex calculations for every row access in a table that handles millions of writes and reads per second.
3.  **Masking Specific Columns:** If the goal is to redact, tokenize, or mask sensitive data within a *column* (e.g., hiding credit card numbers or replacing names with 'XXXX'), Masking Policies are the appropriate solution, not Row Access Policies. Row Access Policies hide entire rows.
    *   *Wrong Approach Scenario:* Trying to use a row access policy to display '*******' instead of the actual `customer_email` for certain roles; this would require a masking policy.

## Common Mistakes

1.  **Overlapping or Conflicting Policies:** Applying multiple row access policies to the same table or defining policies with conflicting logic can lead to unexpected filtering results or even errors, making debugging difficult.
2.  **Not Testing with Different Roles:** Developers often test policies with high-privilege roles (like `ACCOUNTADMIN`) which might bypass the policy, leading to a false sense of security. Thorough testing with all relevant roles is crucial.
3.  **Performance Impact Neglect:** Designing overly complex policy logic or policies that join back to large tables can significantly impact query performance, especially on large datasets. Always consider the performance implications and optimize policy logic.
4.  **Forgetting to Apply the Policy:** Creating a row access policy but forgetting to use `ALTER TABLE ... ADD ROW ACCESS POLICY ...` means the policy will not be enforced, leaving data exposed.
5.  **Using `IS_ROLE_IN_SESSION` vs. `CURRENT_ROLE` Incorrectly:** `CURRENT_ROLE()` returns the active role, while `IS_ROLE_IN_SESSION()` checks if a role is in the current session's hierarchy. Misunderstanding their behavior can lead to incorrect access grants.

## Expected Output

When a user queries the `sales_data` table, the expected output will be a subset of the original table's rows, determined by the row access policy based on the user's active role.

For example, if a user with the `SALES_EAST` role queries `SELECT * FROM sales_data;`, the resulting dataset will only contain rows where the `region` is 'East'.

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT | CUSTOMER_EMAIL |
| :------ | :----------- | :----- | :----------- | :------------- |
| 1       | Laptop       | East   | 1200.00      | alice@example.com |
| 3       | Keyboard     | East   | 75.00        | charlie@example.com |

**Explanation of Columns:**

-   **SALE_ID:** The unique identifier for each sale.
-   **PRODUCT_NAME:** The name of the product sold.
-   **REGION:** The geographical region where the sale occurred. This column is critical for the policy's filtering logic.
-   **SALES_AMOUNT:** The amount of money generated by the sale.
-   **CUSTOMER_EMAIL:** The email address of the customer who made the purchase. This column could also be used for filtering or combined with masking policies.
