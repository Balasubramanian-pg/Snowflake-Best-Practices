# Network Policy Governance Model in Snowflake

Network Policies in Snowflake are a crucial security feature that allow administrators to control inbound network traffic to their Snowflake account. They define rules that permit or deny access based on the source IP address of the connection attempt, acting as a virtual firewall for your data warehouse. This feature exists to bolster security by ensuring that only trusted networks and devices can connect to your Snowflake environment, thereby protecting sensitive data from unauthorized access.

**Short Explanation**

This SQL feature restricts access to your Snowflake account by specifying allowed or blocked IP addresses. It solves the problem of unauthorized network access, even if credentials are valid, by acting as a perimeter defense. It's important in databases for preventing breaches and is commonly used in enterprise security architectures to enforce strict access controls and meet compliance requirements.

- **What problem does this SQL feature solve?** It prevents unauthorized users or systems from connecting to your Snowflake account from untrusted network locations, even if they possess valid login credentials.
- **Why is it important in databases or analytics?** It's a fundamental layer of security, crucial for protecting sensitive data, ensuring data governance, and complying with various industry regulations (like HIPAA, PCI DSS, GDPR) that mandate strict control over network access.
- **Where is it commonly used in real workflows?** Enterprises use it to restrict access to their corporate networks, VPNs, or specific cloud environments, ensuring that only internal systems or approved external tools can interact with Snowflake.

## Implementation Example

Snowflake recently introduced Network Rules as building blocks for Network Policies, offering greater flexibility. Here's an example demonstrating both the creation of a network rule and its application via a network policy.

```sql
-- Step 1: Create a Network Rule to define allowed IP addresses
CREATE NETWORK RULE corporate_office_ips_rule
  TYPE = IPV4
  VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')
  MODE = INGRESS
  COMMENT = 'Allowed IP ranges for corporate offices and specific workstations.';

-- Step 2: Create a Network Policy that references the Network Rule
CREATE OR REPLACE NETWORK POLICY my_corporate_network_policy
  ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips_rule')
  COMMENT = 'Policy to allow connections only from corporate VPN and office IPs via network rule.';

-- Step 3: Apply the Network Policy at the account level
ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;

-- Optional: Apply a different policy to a specific user (overrides account policy for that user)
-- CREATE NETWORK RULE analyst_vpn_rule
```

## Explanation of the Code

This example demonstrates how to create, apply, and manage Network Policies using Network Rules in Snowflake.

- **`CREATE NETWORK RULE corporate_office_ips_rule ...`**:
    - **What it does**: This statement initializes a new network rule named `corporate_office_ips_rule`. Network Rules are reusable objects that define a set of network identifiers (like IP addresses).
    - **How it changes the result set**: This creates a new security object within Snowflake's metadata. It does not directly affect query result sets but defines a set of allowed ingress network locations.
    - `TYPE = IPV4`: Specifies that the rule defines IPv4 addresses.
    - `VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')`: Lists the specific IP addresses and CIDR ranges that this rule will allow.
    - `MODE = INGRESS`: Indicates that this rule applies to incoming connections to Snowflake.
    - `COMMENT = 'Allowed IP ranges for corporate offices and specific workstations.'`: Provides a descriptive comment for the rule.

- **`CREATE OR REPLACE NETWORK POLICY my_corporate_network_policy ...`**:
    - **What it does**: This statement creates a new network policy named `my_corporate_network_policy`. Network policies leverage network rules to define the actual access control. `OR REPLACE` ensures that if a policy with this name already exists, it will be replaced.
    - **How it changes the result set**: This creates or updates a security object. It doesn't affect query result sets directly.
    - `ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips_rule')`: This is the core of the policy, specifying which previously defined network rules contain the allowed IP addresses. Snowflake will only permit connections from IP addresses specified in these rules.
    - `COMMENT = 'Policy to allow connections only from corporate VPN and office IPs via network rule.'`: Adds a descriptive comment to the policy.

- **`ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;`**:
    - **What it does**: This statement activates the `my_corporate_network_policy` at the account level. Once set, this policy applies to all users and connections to the Snowflake account, unless overridden by a user-specific policy.
    - **How it changes the result set**: N/A. This is an account-level configuration change that modifies how connections are authenticated before any queries can be executed.

- **`ALTER USER JANE_SMITH SET NETWORK_POLICY = analyst_network_policy;` (Commented out)**:
    - **What it does**: This would apply a specific network policy to a single user (`JANE_SMITH`), overriding any account-level policy for that user. This allows for granular control.
    - **How it changes the result set**: N/A. This is a user-level configuration change.

## When to Use

1.  **Enhancing Overall Account Security:** To create a baseline security posture for your entire Snowflake account, ensuring that only connections from known and trusted networks (e.g., corporate offices, specific cloud VPCs) are allowed.
    *   **Example Scenario:** A company wants to ensure that all access to its Snowflake data warehouse can only originate from its headquarters' IP range or its cloud-based application servers.
2.  **Compliance with Regulatory Requirements:** When industry regulations (e.g., HIPAA, PCI DSS, GDPR) demand strict control over network access to sensitive data.
    *   **Example Scenario:** A healthcare provider stores patient data in Snowflake and must prove that only authorized internal systems and secured VPN connections can connect to the database.
3.  **Restricting Access for Specific Users or Roles:** To provide more granular control for certain user groups who might need different access policies (e.g., remote analysts using a specific VPN endpoint).
    *   **Example Scenario:** A team of external consultants needs temporary access to Snowflake, but only from their designated secure VPN IP address, which is different from the company's main network policy.

## When Not to Use

1.  **Sole Security Measure:** Relying solely on Network Policies for security without implementing other layers like strong authentication (MFA), role-based access control (RBAC), and data encryption.
    *   **Reason:** Network Policies are a perimeter defense; they don't protect against threats originating from within the allowed network or compromised credentials.
2.  **Dynamic IP Environments Without Automation:** In scenarios where user IP addresses frequently change (e.g., mobile users without a fixed VPN or cloud services with dynamic IPs) and there isn't an automated mechanism to update the allowed list.
    *   **Reason:** Manually updating policies for constantly changing IPs becomes an operational burden and can lead to legitimate users being locked out.
3.  **Overly Permissive "Allowed" Lists:** Creating a policy with an `ALLOWED_IP_LIST` that includes broad, unsecured public IP ranges without careful consideration.
    *   **Reason:** This defeats the purpose of the network policy, providing a false sense of security and leaving the account vulnerable.

## Common Mistakes

1.  **Forgetting to Apply the Policy:** Creating network policies but failing to activate them at the account or user level.
    *   **Impact:** The policy does not take effect, leaving the account unsecured.
2.  **Locking Yourself Out:** Applying a network policy that does not include your current IP address in the `ALLOWED_IP_LIST` (or if your IP is in `BLOCKED_IP_LIST`).
    *   **Impact:** You will be unable to connect to your Snowflake account. Snowflake provides a `MINS_TO_BYPASS_NETWORK_POLICY` feature to mitigate this but it's meant for emergencies and not routine operations.
3.  **Not Testing Thoroughly:** Implementing a policy without thoroughly testing access from all expected locations and users, including automated tools and services.
    *   **Impact:** Critical users or applications might be unintentionally locked out, causing operational disruptions.
4.  **Mixing Old and New Approaches Carelessly:** In the context of Snowflake's newer Network Rules, some developers might still rely on directly embedding `ALLOWED_IP_LIST` within the policy, missing the benefits of reusability and modularity offered by Network Rules.
    *   **Impact:** Less organized, harder to manage policies, especially for complex environments.

## Expected Output

The Network Policy Governance Model itself does not produce a "dataset" in the traditional sense, as it's a security configuration rather than a data operation. However, the *effect* of its implementation is that connection attempts to Snowflake will either succeed or fail based on the defined rules.

When a Network Policy is active, you would expect:

1.  **Successful Connection:** If a user attempts to connect from an IP address included in the policy's `ALLOWED_IP_LIST` (or any `ALLOWED_NETWORK_RULE_LIST` entries).
2.  **Blocked Connection (Login Error):** If a user attempts to connect from an IP address *not* in the allowed list, or specifically listed in a `BLOCKED_IP_LIST` (or `BLOCKED_NETWORK_RULE_LIST`).

Here's what an example "output" (an event in login history) might look like, along with an explanation of columns:

| EVENT_TIMESTAMP             | EVENT_TYPE   | USER_NAME   | CLIENT_IP       | IS_SUCCESS  | ERROR_MESSAGE                                                 |
| :-------------------------- | :----------- | :---------- | :-------------- | :---------- | :------------------------------------------------------------ |
| 2026-03-14 09:30:00.123 -0700 | LOGIN        | JANE_SMITH  | 198.51.100.10   | TRUE        | NULL                                                        |
| 2026-03-14 09:31:30.456 -0700 | LOGIN        | JOHN_DOE    | 203.0.113.5     | TRUE        | NULL                                                        |
| 2026-03-14 09:32:15.789 -0700 | LOGIN        | ALICE_W     | 10.0.0.100      | FALSE       | IP address 10.0.0.100 is not allowed to access Snowflake.     |

**Explanation of the columns in the example login history (from `LOGIN_HISTORY` view):**

-   **`EVENT_TIMESTAMP`**: The timestamp when the login attempt occurred.
-   **`EVENT_TYPE`**: The type of event; in this case, `LOGIN`.
-   **`USER_NAME`**: The name of the user attempting to log in.
-   **`CLIENT_IP`**: The IP address from which the login attempt originated.
-   **`IS_SUCCESS`**: A boolean indicating whether the login attempt was successful (`TRUE`) or failed (`FALSE`).
-   **`ERROR_MESSAGE`**: If the login failed, this column contains the reason for the failure, which for network policies would explicitly mention the IP not being allowed.

This table shows two successful logins from allowed IPs and one failed login from an IP blocked by the network policy.
