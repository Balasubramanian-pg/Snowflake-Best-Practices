# Using Secure Views with Row Access Policies in Snowflake

This concept combines two powerful Snowflake security features: Secure Views and Row Access Policies, to enforce fine-grained access control on data. It allows you to restrict which rows a user can see in a table, even when accessing the data through a view, ensuring data privacy and compliance.

**Short Explanation**

Secure Views prevent users from seeing the underlying view definition, protecting sensitive business logic. Row Access Policies then act as a dynamic filter, allowing or denying access to specific rows in a table based on user roles or other conditions. This combination ensures that users only ever see the data they are authorized for, regardless of how they query it.

-   **What problem does this SQL feature solve?** It solves the problem of fine-grained access control, where different users or roles need to see different subsets of the same data within a table, without duplicating data or complex application-level filtering.
-   **Why is it important in databases or analytics?** It's crucial for data governance, compliance (e.g., GDPR, HIPAA), and multi-tenant architectures, ensuring that sensitive information is protected and users only access data relevant to their permissions.
-   **Where is it commonly used in real workflows?** It's used in scenarios like multi-tenant dashboards, protecting employee payroll data, customer data isolation for sales teams, or ensuring compliance in financial reporting.

## Implementation Example

```sql
-- 1. Create a base table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    customer_region VARCHAR,
    sale_amount DECIMAL(10, 2),
    sales_rep_id INT
);

-- Insert some sample data
INSERT INTO sales_data (order_id, customer_name, customer_region, sale_amount, sales_rep_id) VALUES
(101, 'Alice Johnson', 'East', 150.00, 1001),
(102, 'Bob Williams', 'West', 230.50, 1002),
(103, 'Charlie Brown', 'East', 75.25, 1001),
(104, 'Diana Prince', 'North', 500.00, 1003),
(105, 'Eve Adams', 'West', 120.00, 1002);

-- 2. Create a Row Access Policy
-- This policy allows users to see rows if their current role is 'ADMIN'
-- OR if their sales_rep_id matches the sales_rep_id in the row,
-- OR if their region matches the customer_region in the row (for 'REGIONAL_ANALYST' role).
CREATE OR REPLACE ROW ACCESS POLICY sales_policy
AS (sales_rep_id_col INT, customer_region_col VARCHAR) RETURNS BOOLEAN ->
    (CURRENT_ROLE() = 'ADMIN') OR
    (CURRENT_ROLE() = 'SALES_REP' AND sales_rep_id_col = CURRENT_USER_ID()) OR
    (CURRENT_ROLE() = 'REGIONAL_ANALYST' AND customer_region_col = (SELECT GET_CUSTOM_USER_PROPERTY('REGION')));

-- Note: CURRENT_USER_ID() is a placeholder for a function that returns the sales_rep_id of the current user.
-- GET_CUSTOM_USER_PROPERTY('REGION') is a placeholder for a function that returns the region of the current user.
-- In a real scenario, you would have a user mapping table or custom functions to derive these values.

-- 3. Apply the Row Access Policy to the base table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (sales_rep_id, customer_region);

-- 4. Create a Secure View on top of the base table
CREATE OR REPLACE SECURE VIEW sales_summary_secure_v AS
SELECT
    order_id,
    customer_name,
    customer_region,
    sale_amount
FROM
    sales_data;

-- 5. Grant access to the view (not the base table)
GRANT SELECT ON VIEW sales_summary_secure_v TO ROLE SALES_REP;
GRANT SELECT ON VIEW sales_summary_secure_v TO ROLE REGIONAL_ANALYST;
GRANT SELECT ON VIEW sales_summary_secure_v TO ROLE ADMIN;

-- Simulate user roles and custom properties for testing
-- These would typically be handled by Snowflake user/role management and custom functions/tables.
-- For a 'SALES_REP' with ID 1001
-- For a 'REGIONAL_ANALYST' for 'East' region
-- For 'ADMIN' role
```

## Explanation of the Code

-   **`CREATE OR REPLACE TABLE sales_data (...)`**: This creates our base table containing sales information, including sensitive details like `sale_amount` and `sales_rep_id`.
-   **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (...) RETURNS BOOLEAN -> (...)`**: This defines the row access policy.
    -   `AS (sales_rep_id_col INT, customer_region_col VARCHAR)`: Specifies the columns from the `sales_data` table that the policy will evaluate for each row.
    -   `RETURNS BOOLEAN`: The policy must return `TRUE` for a row to be visible, `FALSE` otherwise.
    -   `CURRENT_ROLE() = 'ADMIN'`: Allows users with the 'ADMIN' role to see all rows.
    -   `CURRENT_ROLE() = 'SALES_REP' AND sales_rep_id_col = CURRENT_USER_ID()`: This is a conditional predicate. If the current user's role is 'SALES_REP', they can only see rows where the `sales_rep_id_col` matches their `CURRENT_USER_ID()` (a placeholder for a function that would fetch the user's ID).
    -   `CURRENT_ROLE() = 'REGIONAL_ANALYST' AND customer_region_col = (SELECT GET_CUSTOM_USER_PROPERTY('REGION'))`: If the role is 'REGIONAL_ANALYST', they can only see rows from their assigned `REGION` (another placeholder).
-   **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (sales_rep_id, customer_region)`**: This statement applies the created `sales_policy` to the `sales_data` table, linking specific columns from the table to the policy parameters.
-   **`CREATE OR REPLACE SECURE VIEW sales_summary_secure_v AS SELECT ... FROM sales_data;`**: This creates a secure view on top of the `sales_data` table.
    -   **What it does:** It defines a virtual table `sales_summary_secure_v`.
    -   **How it changes the result set:** It selects specific columns, effectively acting as a projection. The `SECURE` keyword is critical here: it hides the view's definition from non-privileged users. This means users cannot see that the view is built directly on `sales_data`, which might contain columns they shouldn't know about, or logic they shouldn't bypass. The Row Access Policy applied to `sales_data` will automatically filter the rows returned by this view based on the user's permissions.
-   **`GRANT SELECT ON VIEW sales_summary_secure_v TO ROLE ...`**: These statements grant `SELECT` privileges on the *view* to different roles, not directly on the base table. This is a best practice, as users interact only with the view, which transparently enforces the row access policy.

## When to Use

1.  **Multi-tenant Data Architectures:** When multiple clients or departments share the same underlying data table, but each should only see their own specific rows. For example, a `transactions` table shared by several retail brands, where each brand's user should only see their brand's transactions.
2.  **Regulatory Compliance (GDPR, HIPAA):** To ensure that personal identifiable information (PII) or protected health information (PHI) is only accessible to authorized personnel. For instance, doctors only see their patients' records, or HR staff only see records for their department.
3.  **Data Monetization/Productization:** When building data products where different subscribers pay for different tiers of data access (e.g., premium users get more detailed or broader datasets). The secure view with RAP can dynamically adjust the data visibility.

## When Not to Use

1.  **Simple Column-Level Security:** If you only need to restrict access to specific *columns* for all users, column-level security (using masking policies or simply selecting fewer columns in a non-secure view) is simpler and more efficient than using a row access policy.
2.  **Performance Critical Workloads with Very Complex Policies:** While Snowflake optimizes RAPs, extremely complex policies involving many joins or expensive subqueries within the policy function itself can sometimes impact query performance, especially on very large tables. Simple filtering in a `WHERE` clause on a standard view might be preferred if performance is paramount and data sensitivity allows.
3.  **Static Data Subsets:** If access to rows is completely static and doesn't depend on user roles or attributes (e.g., "users in Group A always see rows 1-100"), then creating separate tables or materialized views for each subset might offer better performance and simpler management, though this duplicates data.

## Common Mistakes

1.  **Forgetting the `SECURE` Keyword:** Creating a regular view instead of a `SECURE VIEW`. A non-secure view's definition is visible to users with `SELECT` privileges, potentially exposing the underlying table structure, column names, or even sensitive logic in the view definition, even if a RAP is on the base table.
2.  **Granting Access to the Base Table:** Granting `SELECT` privileges directly on the base table (`sales_data` in the example) to end-user roles. This bypasses the secure view entirely and allows users to query the base table directly, where only the Row Access Policy protects the data. The secure view should be the *only* access point.
3.  **Overly Complex Policy Logic:** Writing Row Access Policies with overly complex `CASE` statements, many `OR` conditions, or subqueries that don't perform well. This can lead to performance degradation, as the policy is evaluated for every row accessed.
4.  **Not Handling Null Values in Policy Columns:** If the columns used in the `ROW ACCESS POLICY` (e.g., `sales_rep_id`, `customer_region`) can be `NULL` in the base table, the policy logic must explicitly account for how `NULL` values should be handled to avoid unintended data visibility or hiding.

## Expected Output

The resulting dataset from querying `sales_summary_secure_v` should contain rows filtered according to the `sales_policy` based on the `CURRENT_ROLE()` and any custom user properties. For an 'ADMIN' user, all rows would be visible. For a 'SALES\_REP' with `CURRENT_USER_ID()` = 1001, only their assigned sales data would appear.

Here's an example of what a `SALES_REP` with ID `1001` (who is also associated with the 'East' region) would see when querying `sales_summary_secure_v`:

| ORDER\_ID | CUSTOMER\_NAME | CUSTOMER\_REGION | SALE\_AMOUNT |
| :-------- | :------------- | :--------------- | :---------- |
| 101       | Alice Johnson  | East             | 150.00      |
| 103       | Charlie Brown  | East             | 75.25       |

**Explanation of columns:**

-   `ORDER_ID`: The unique identifier for a sales order.
-   `CUSTOMER_NAME`: The name of the customer.
-   `CUSTOMER_REGION`: The geographic region of the customer.
-   `SALE_AMOUNT`: The monetary value of the sale.

Rows `102`, `104`, and `105` are filtered out because they do not match the `sales_rep_id` of `1001` (or the 'East' region if the policy allowed region-based access for this role).
