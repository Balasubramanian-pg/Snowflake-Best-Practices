# 14. Designing a Clean Role Hierarchy

Designing a clean role hierarchy involves structuring access control by separating object privileges from user assignments to ensure scalable and secure permissions management.
This approach minimizes administrative overhead by allowing roles to inherit permissions logically based on organizational functions and data access requirements.

# Key Points

* Separate access roles (which grant privileges on database objects) from functional roles (which map to business functions and are assigned to users).
* Utilize role inheritance to build a hierarchy where higher-level functional roles automatically inherit the permissions of lower-level roles.
* Adhere to the principle of least privilege by ensuring each access role only contains the minimum permissions necessary for a specific task.
* Avoid granting privileges directly to users; always grant privileges to access roles, access roles to functional roles, and functional roles to users.
* > [!IMPORTANT] Always attach the highest-level custom roles to a centralized administrative role (like `SYSADMIN`) to prevent orphaned roles and maintain governance.

# Implementation

Provide a realistic Snowflake SQL example demonstrating the concept.

```sql
USE ROLE USERADMIN;

-- Step 1: Create Access Roles and Functional Roles
CREATE ROLE ar_sales_read;
CREATE ROLE ar_sales_write;
CREATE ROLE fr_sales_analyst;
CREATE ROLE fr_sales_manager;

-- Step 2: Grant object privileges to Access Roles
USE ROLE SECURITYADMIN;
GRANT USAGE ON DATABASE sales_db TO ROLE ar_sales_read;
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE ar_sales_read;
GRANT INSERT, UPDATE ON ALL TABLES IN SCHEMA sales_db.public TO ROLE ar_sales_write;

-- Step 3: Build the hierarchy by granting Access Roles to Functional Roles
GRANT ROLE ar_sales_read TO ROLE fr_sales_analyst;
GRANT ROLE ar_sales_read TO ROLE fr_sales_manager;
GRANT ROLE ar_sales_write TO ROLE fr_sales_manager;
GRANT ROLE fr_sales_analyst TO ROLE fr_sales_manager;

-- Step 4: Assign Functional Roles to users and attach the hierarchy to SYSADMIN
GRANT ROLE fr_sales_analyst TO USER jdoe;
GRANT ROLE fr_sales_manager TO ROLE SYSADMIN;
```

> [!TIP]
> Use a consistent naming convention (e.g., `ar_` for access roles and `fr_` for functional roles) to make the hierarchy instantly understandable to other administrators.

# What does the Query do?

Explain the SQL step by step:

Step 1 creates dedicated access roles designed exclusively to hold database object privileges, alongside functional roles that map directly to real-world job titles.

Step 2 applies the principle of least privilege by granting specific database, schema, and table permissions only to the designated access roles, keeping object-level grants isolated.

Step 3 constructs the hierarchy by granting access roles to functional roles, and establishes inheritance so the `fr_sales_manager` automatically inherits all capabilities of the `fr_sales_analyst`.

Step 4 assigns the functional role to a specific user for daily operations and attaches the top of the custom hierarchy to the `SYSADMIN` role to ensure centralized management and prevent orphaned objects.

# When Should We Use This? 

* When scaling a data platform across multiple departments where users require varying levels of access to shared datasets without creating a tangled, unmanageable web of direct user grants.
* During security audits or compliance reviews (e.g., SOC2, GDPR) where administrators must easily demonstrate who has access to what data and prove that the principle of least privilege is strictly enforced.
