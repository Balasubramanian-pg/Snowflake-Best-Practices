# Point-in-Time Stream Offsets using Time Travel
Point-in-Time Stream Offsets using Time Travel allows you to access and manipulate data at a specific point in time. This feature is useful for auditing, debugging, and recovering data. It provides a way to query data as it existed at a previous point in time, enabling you to track changes and identify issues. By using Time Travel, you can also restore data to a previous state, ensuring data consistency and integrity.

## Implementation Example
```sql
-- Create a table
CREATE TABLE example_table (
    id INT,
    name VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Insert data
INSERT INTO example_table (id, name) VALUES (1, 'John');
INSERT INTO example_table (id, name) VALUES (2, 'Jane');

-- Use Time Travel to query data at a specific point in time
SELECT * FROM example_table AT TIMESTAMP '2022-01-01 12:00:00';
```
Few examples and alternatives of SQL Code:
- Use `AT` clause to specify the point in time.
- Use `BEFORE` clause to specify the point in time before a specific event.
- Use `AFTER` clause to specify the point in time after a specific event.

## Explanation of the Code
* The `CREATE TABLE` statement creates a new table with a timestamp column that defaults to the current timestamp.
* The `INSERT INTO` statements insert data into the table with the current timestamp.
* The `SELECT` statement uses the `AT` clause to query data at a specific point in time.
* The `AT` clause specifies the point in time as a timestamp value.
* The `TIMESTAMP` function is used to specify the point in time as a timestamp value.
* The `SELECT` statement returns all columns (`*`) from the table at the specified point in time.
* The `example_table` is the table being queried, and the `AT` clause is used to specify the point in time.

## When to use
1. **Auditing and Logging**: Use Point-in-Time Stream Offsets to track changes to data over time, enabling auditing and logging capabilities.
2. **Data Recovery**: Use Point-in-Time Stream Offsets to recover data to a previous state, ensuring data consistency and integrity.
3. **Debugging**: Use Point-in-Time Stream Offsets to identify issues and debug data-related problems.

## When not to use
1. **Real-time Data**: Avoid using Point-in-Time Stream Offsets for real-time data, as it may introduce latency and affect performance.
2. **High-Volume Data**: Avoid using Point-in-Time Stream Offsets for high-volume data, as it may impact performance and increase storage requirements.
3. **Simple Queries**: Avoid using Point-in-Time Stream Offsets for simple queries, as it may add unnecessary complexity.

## Common Mistakes
* Using the wrong timestamp value, resulting in incorrect data being returned.
* Forgetting to specify the point in time, resulting in the current data being returned.
* Using the `AT` clause with an invalid timestamp value, resulting in an error.
* Not considering the impact of time zones on timestamp values.

## Expected Output
The expected output of the `SELECT` statement will be the data as it existed at the specified point in time. For example:
| id | name | created_at          |
|----|------|----------------------|
| 1  | John | 2022-01-01 12:00:00 |
| 2  | Jane | 2022-01-01 12:00:00 |
Note that the actual output will depend on the data in the table and the point in time specified.
