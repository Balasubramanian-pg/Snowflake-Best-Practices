# Dormant Account Locking Policy in Snowflake

The Dormant Account Locking Policy in Snowflake is a security feature designed to automatically lock user accounts that have been inactive for a specified period. This policy helps enhance data security by preventing unauthorized access to dormant accounts, reducing the risk of compromise through forgotten or untended credentials. It exists to enforce stricter access controls and minimize potential attack vectors within the Snowflake environment.

**Short Explanation**

This policy tackles the security problem of inactive user accounts that could be exploited by malicious actors. It's important in databases for maintaining a strong security posture and ensuring compliance with various regulatory standards by automatically managing access. It's commonly used in enterprise environments to manage large numbers of user accounts efficiently and securely.

- What problem does this SQL feature solve? It solves the security risk associated with dormant user accounts, which can be vulnerable to unauthorized access if credentials are compromised or accounts are left unattended.
- Why is it important in databases or analytics? It's crucial for maintaining a robust security posture, protecting sensitive data, and meeting compliance requirements (like SOC 2, HIPAA, GDPR) by enforcing automated access control.
- Where is it commonly used in real workflows? In organizations with many users, particularly those handling sensitive data, to automatically de-provision or restrict access for users who are no longer actively using the system.

## Implementation Example

While not a SQL query in the traditional sense of data manipulation, the policy is configured using SQL DDL (Data Definition Language) commands.

```sql
ALTER ACCOUNT SET ACCOUNT_LOCK_AFTER_DAYS = 90;
ALTER ACCOUNT SET ACCOUNT_LOCK_VIEW_ONLY_AFTER_DAYS = 30;
ALTER ACCOUNT SET MIN_PASSWORD_LENGTH = 12; -- Example of other related security policies
```

## Explanation of the Code

This example demonstrates how to configure account locking policies at the account level in Snowflake.

- **`ALTER ACCOUNT`**: This clause is used to modify account-level parameters.
- **`SET ACCOUNT_LOCK_AFTER_DAYS = 90`**: This parameter specifies that a user account will be completely locked (preventing any login) if it remains inactive for 90 consecutive days. It changes the account's security behavior by automatically disabling access for long-inactive users.
- **`SET ACCOUNT_LOCK_VIEW_ONLY_AFTER_DAYS = 30`**: This parameter indicates that after 30 days of inactivity, a user account will be restricted to a "view-only" state. This means the user can log in and view data but cannot perform any write operations. It changes the account's permissions, restricting functionality for moderately inactive users.
- **`SET MIN_PASSWORD_LENGTH = 12`**: (Optional, but related) This parameter sets a minimum length requirement for user passwords, further enhancing security. While not directly a dormant account policy, it's often configured as part of a comprehensive security strategy.

## When to Use

1.  **Ensuring Compliance**: When an organization needs to meet regulatory compliance standards (e.g., GDPR, HIPAA, SOC 2) that mandate strict access control and account lifecycle management.
    *   *Scenario:* A healthcare company must ensure that former employees or long-inactive contractors cannot access patient data, even if their credentials were not explicitly revoked.
2.  **Managing Large User Bases**: In environments with a high turnover rate or a large number of users, making manual account management cumbersome and prone to errors.
    *   *Scenario:* A large enterprise with thousands of Snowflake users needs an automated way to secure accounts of employees who have left the company or changed roles, without manual intervention for every single case.
3.  **Reducing Attack Surface**: To proactively reduce the risk of compromised dormant accounts being used to access sensitive data.
    *   *Scenario:* An attacker gains access to credentials for a user account that hasn't been used in months. If the dormant account locking policy is in place, the account would already be locked, preventing the attacker from logging in.

## When Not to Use

1.  **Highly Critical Service Accounts**: For service accounts or integration users that are designed to run automated tasks and may not have interactive logins for long periods.
    *   *Scenario:* An ETL process runs daily but the service account used for it doesn't log in interactively. Locking this account based on inactivity would break critical data pipelines.
2.  **Seldom-Used Administrative Accounts**: For accounts designated for emergency access or disaster recovery that are rarely, if ever, used for normal operations.
    *   *Scenario:* A "break glass" administrative account is kept for extreme emergencies. Locking it due to dormancy would prevent its use when it's most needed.
3.  **Accounts for Infrequent but Important Access**: Users who only need to access Snowflake a few times a year for specific reporting or auditing purposes.
    *   *Scenario:* An external auditor only logs in once a quarter to pull specific compliance reports. A strict dormant account policy might lock their account every time, requiring frequent unlocking requests and hindering their work.

## Common Mistakes

1.  **Setting Policies Too Aggressively**: Configuring very short lock periods (e.g., 7 days) without considering legitimate infrequent users or service accounts, leading to frequent account lockouts and administrative overhead.
2.  **Not Communicating Policies**: Failing to inform users about the dormant account locking policy, resulting in confusion and frustration when their accounts are unexpectedly locked.
3.  **Ignoring Service Accounts**: Applying the policy universally without exempting or specially handling non-human service accounts, which often do not have interactive logins and can be critical for automated processes.
4.  **Lack of Review and Adjustment**: Not regularly reviewing the effectiveness and impact of the policy, leading to outdated lock periods or issues that go unaddressed.

## Expected Output

The dormant account locking policy doesn't produce a direct dataset in the same way a `SELECT` statement does. Instead, its "output" is a change in the security behavior and state of user accounts in Snowflake. When the policy is active, the expected outcome is that user accounts meeting the dormancy criteria will have their login capabilities restricted or revoked.

An example conceptual table illustrating the *state* of user accounts under such a policy would look like this:

| USERNAME     | LAST_LOGIN_DATE | DAYS_INACTIVE | ACCOUNT_STATUS       |
| :----------- | :-------------- | :------------ | :------------------- |
| ANALYST_JANE | 2026-03-01      | 13            | ACTIVE               |
| REPORT_BOT   | 2026-01-05      | 68            | ACTIVE (View-Only)   |
| OLD_DEV_MARK | 2025-10-15      | 150           | LOCKED               |

**Explanation of Columns:**

-   **USERNAME**: The identifier for the user account.
-   **LAST\_LOGIN\_DATE**: The date of the user's last successful login. This is the key metric used by the dormant account policy.
-   **DAYS\_INACTIVE**: The calculated number of days since the `LAST_LOGIN_DATE`. This helps to show how the policy rules are being applied.
-   **ACCOUNT\_STATUS**: The current state of the user's account as enforced by the policy.
    *   `ACTIVE`: The user can log in and perform all assigned actions.
    *   `ACTIVE (View-Only)`: The user can log in but is restricted to view-only operations (if `ACCOUNT_LOCK_VIEW_ONLY_AFTER_DAYS` is set and the inactivity period is met).
    *   `LOCKED`: The user cannot log in at all (if `ACCOUNT_LOCK_AFTER_DAYS` is set and the inactivity period is met).
