# 04. The Principle of Least Privilege in Snowflake

The Principle of Least Privilege (PoLP) ensures that users, roles, and applications are granted only the minimum level of access necessary to perform their required tasks.
In Snowflake, this is achieved through a robust Role-Based Access Control (RBAC) model that separates system administration from data access and enforces granular privileges on specific database objects.

# Key Points

* **Role-Based Access Control (RBAC):** Privileges are never assigned directly to users; instead, they are granted to roles, which are then assigned to users or other roles to create a secure, manageable hierarchy.
* **Separation of Duties:** Snowflake provides distinct system-defined roles (e.g., `SYSADMIN` for object creation, `SECURITYADMIN` for user/role management) to prevent any single role from having unchecked power.
* **Granular Object Privileges:** Access can be restricted down to specific actions (e.g., `SELECT`, `INSERT`, `USAGE`) on specific objects (e.g., databases, schemas, tables, warehouses).
* **Future Grants:** Snowflake allows administrators to define privileges for objects that do not yet exist using `ON FUTURE` grants, ensuring new objects automatically adhere to least privilege policies.
* **Managed Access Schemas:** These schemas centralize privilege management by allowing only the schema owner (rather than individual object owners) to grant access to objects within the schema.

# Implementation

The following script demonstrates how to create a custom role and grant it the exact minimum permissions required to read a single table using a specific compute warehouse.

```sql
-- Step 1: Create a dedicated, restricted role
USE ROLE USERADMIN;
CREATE ROLE data_analyst_ro;

-- Step 2: Grant minimum required compute access
USE ROLE SYSADMIN;
GRANT USAGE ON WAREHOUSE reporting_wh TO ROLE data_analyst_ro;

-- Step 3: Grant minimum required database and schema access
GRANT USAGE ON DATABASE sales_db TO ROLE data_analyst_ro;
GRANT USAGE ON SCHEMA sales_db.public TO ROLE data_analyst_ro;

-- Step 4: Grant specific table-level access (Read-only)
GRANT SELECT ON TABLE sales_db.public.q3_revenue TO ROLE data_analyst_ro;

-- Step 5: Assign the restricted role to a specific user
USE ROLE SECURITYADMIN;
GRANT ROLE data_analyst_ro TO USER alice_smith;
```

# What does the Query do?

> [!IMPORTANT]  
> Notice the switching of roles (`USERADMIN`, `SYSADMIN`, `SECURITYADMIN`) throughout the script. This enforces the separation of duties, which is a core component of implementing the Principle of Least Privilege effectively.

* **Step 1:** Switches to the `USERADMIN` role to create a new, purpose-built role (`data_analyst_ro`) designed specifically for read-only analytical tasks.
* **Step 2:** Switches to the `SYSADMIN` role and grants `USAGE` on a specific virtual warehouse, allowing the role to execute queries without granting permission to modify, resize, or suspend the compute resource.
* **Step 3:** Grants `USAGE` privileges on the target database and schema. This is the strict prerequisite permission required to navigate the logical hierarchy; without it, the user cannot access the schema's contents, even if table-level permissions are granted.
* **Step 4:** Grants the `SELECT` privilege exclusively on the `q3_revenue` table, ensuring the role can read data from this specific table but cannot modify it or access any other tables residing in the schema.
* **Step 5:** Switches to the `SECURITYADMIN` role to assign the newly configured least-privilege role to the user `alice_smith`, granting her exactly the access she needs to do her job and nothing more.

# When Should We Use This? 

* When onboarding new employees, contractors, or third-party applications that require access to specific datasets without risking unauthorized data modification or exposure of sensitive PII.
* When designing a production environment where automated ETL/ELT service accounts need precise permissions to write to staging tables but must be strictly restricted from dropping databases or altering warehouse configurations.
