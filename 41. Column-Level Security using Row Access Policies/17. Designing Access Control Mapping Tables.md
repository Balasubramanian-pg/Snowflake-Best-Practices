# Designing Access Control Mapping Tables in Snowflake

Access control mapping tables in Snowflake are specialized tables used in conjunction with Row Access Policies (RAPs) or Dynamic Data Masking (DDM) to define granular data access based on user roles, attributes, or other criteria. They externalize the access logic from the policy definition, making security management more flexible and scalable. Essentially, these tables act as a lookup mechanism for policies to determine *who* can see *what* data.

**Short Explanation**

These tables store rules that map users or roles to specific data entitlements, such as regions, departments, or access levels. When a user queries data, Snowflake's Row Access Policies or Dynamic Data Masking policies consult these mapping tables in real-time to decide which rows the user can view or how specific column data should be masked.

* **What problem does this SQL feature solve?** This feature solves the problem of managing complex and dynamic access rules without hardcoding them directly into security policies. It centralizes access logic, allowing for easier updates and auditing.
* **Why is it important in databases or analytics?** It's crucial for implementing Row-Level Security (RLS) and Column-Level Security (CLS) efficiently, ensuring that users only see data relevant to their role or permissions, which is vital for compliance and data privacy.
* **Where is it commonly used in real workflows?** Commonly used in scenarios requiring multi-tenant data isolation, regional data restrictions, PII protection, or segregation of duties, especially in large enterprises with diverse data access requirements.

## Implementation Example

Let's imagine a scenario where we have sales data, and sales managers should only see data for their assigned regions.

```sql
-- 1. Create a mapping table for sales regions to roles
CREATE TABLE governance.security.sales_region_roles_map (
    region VARCHAR,
    role_name VARCHAR
);

-- Insert data into the mapping table
INSERT INTO governance.security.sales_region_roles_map (region, role_name) VALUES
('North America', 'SALES_MANAGER_NA'),
('Europe', 'SALES_MANAGER_EU'),
('Asia', 'SALES_MANAGER_ASIA');

-- 2. Create the main sales data table
CREATE TABLE sales.public.transactions (
    transaction_id INT,
    product_name VARCHAR,
    amount DECIMAL(10, 2),
    region VARCHAR,
    sale_date DATE
);

-- Insert sample data
INSERT INTO sales.public.transactions (transaction_id, product_name, amount, region, sale_date) VALUES
(1, 'Laptop', 1200.00, 'North America', '2026-03-01'),
(2, 'Mouse', 25.00, 'Europe', '2026-03-02'),
(3, 'Keyboard', 75.00, 'Asia', '2026-03-03'),
(4, 'Monitor', 300.00, 'North America', '2026-03-04'),
(5, 'Webcam', 50.00, 'Europe', '2026-03-05');

-- 3. Create a Row Access Policy that uses the mapping table
CREATE OR REPLACE ROW ACCESS POLICY governance.security.sales_region_policy
AS (region_col VARCHAR) RETURNS BOOLEAN ->
    -- Allow ACCOUNTADMIN to see all data
    (CURRENT_ROLE() = 'ACCOUNTADMIN') OR
    -- Otherwise, check if the current role is in the mapping table for the region
    EXISTS (
        SELECT 1
        FROM governance.security.sales_region_roles_map m
        WHERE m.region = region_col
          AND m.role_name = CURRENT_ROLE()
    );

-- 4. Apply the Row Access Policy to the sales transactions table
ALTER TABLE sales.public.transactions
ADD ROW ACCESS POLICY governance.security.sales_region_policy ON (region);

-- Grant privileges for testing
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE SALES_MANAGER_NA;
GRANT USAGE ON DATABASE SALES TO ROLE SALES_MANAGER_NA;
GRANT USAGE ON SCHEMA SALES.PUBLIC TO ROLE SALES_MANAGER_NA;
GRANT SELECT ON TABLE sales.public.transactions TO ROLE SALES_MANAGER_NA;
```

## Explanation of the Code

This example demonstrates how to set up Row-Level Security using a mapping table.

* **`CREATE TABLE governance.security.sales_region_roles_map (...)`**: This statement creates the mapping table.
    * **What it does**: Defines a table `sales_region_roles_map` in the `governance.security` schema to store which `region` is accessible by which `role_name`.
    * **How it changes the result set**: This table itself doesn't directly change a query's result set, but it provides the lookup data that the Row Access Policy will use to filter rows.
* **`INSERT INTO governance.security.sales_region_roles_map (...)`**: Populates the mapping table with example data.
    * **What it does**: Associates specific regions (e.g., 'North America') with corresponding roles (e.g., 'SALES_MANAGER_NA').
    * **How it changes the result set**: Provides the active mappings for the policy to use.
* **`CREATE TABLE sales.public.transactions (...)`**: Creates the main data table that needs protection.
    * **What it does**: Sets up a table `transactions` containing sales data, including a `region` column.
    * **How it changes the result set**: This is the base table whose rows will be filtered.
* **`CREATE OR REPLACE ROW ACCESS POLICY governance.security.sales_region_policy AS (region_col VARCHAR) RETURNS BOOLEAN ->...`**: This defines the Row Access Policy.
    * **What it does**: Creates a policy named `sales_region_policy`. This policy will be evaluated for each row in a table it's applied to. It takes a column value (`region_col`) as input and returns `TRUE` if the row should be visible, `FALSE` otherwise.
    * **How it changes the result set**:
        * `CURRENT_ROLE() = 'ACCOUNTADMIN'`: Allows the `ACCOUNTADMIN` role to bypass the policy and see all rows. This is a common practice for superusers.
        * `EXISTS (SELECT 1 FROM governance.security.sales_region_roles_map m WHERE m.region = region_col AND m.role_name = CURRENT_ROLE())`: This is the core of the mapping table usage. For every row, it checks if the `region_col` of that row, combined with the `CURRENT_ROLE()` of the user, has an entry in the `sales_region_roles_map`. If a match is found, `EXISTS` returns `TRUE`, and the row is returned. If no match, it returns `FALSE`, and the row is filtered out.
* **`ALTER TABLE sales.public.transactions ADD ROW ACCESS POLICY governance.security.sales_region_policy ON (region);`**: Applies the policy to the `transactions` table.
    * **What it does**: Binds the `sales_region_policy` to the `transactions` table, specifying that the `region` column from the `transactions` table should be passed as `region_col` to the policy's evaluation logic.
    * **How it changes the result set**: From this point on, any `SELECT` query on `sales.public.transactions` will have its rows filtered by the `sales_region_policy` based on the user's current role.

## When to Use

1. **Complex Row-Level Security (RLS)**: When access to rows depends on multiple factors (e.g., region AND department, or project AND user group). Mapping tables centralize these complex rules.
    * *Example Scenario*: A healthcare database where doctors can only see patient records from their assigned hospital, and nurses only records from their specific ward within that hospital.
2. **Dynamic Data Masking (DDM) with Variable Conditions**: To mask sensitive column data differently based on the user's role or other attributes, and these rules need to be easily configurable.
    * *Example Scenario*: In a financial application, a general user might see `****-****-****-1234` for a credit card number, while a fraud investigator role sees the full number. The mapping table would define which roles get which masking level.
3. **Centralized Access Management**: When you need a single source of truth for access entitlements that can be easily updated without modifying SQL code in policies.
    * *Example Scenario*: An HR system where new employees or changes in departmental access can be managed by simply adding or updating records in a mapping table, rather than rewriting security policies.

## When Not to Use

1. **Simple, Static Access Rules**: For very basic RLS or DDM where a policy can easily check a single `CURRENT_ROLE()` or `CURRENT_USER()` condition without needing an external lookup.
    * *Situation*: If all members of a `HR_ANALYST` role should always see all rows, a simple `CURRENT_ROLE() = 'HR_ANALYST'` in the policy is sufficient without a mapping table.
2. **Performance-Critical Operations with Very Large Mapping Tables**: While Snowflake optimizes policy evaluation, extremely large and frequently changing mapping tables that are not properly indexed or utilized with memoizable functions can introduce overhead.
    * *Situation*: A real-time analytics dashboard querying billions of rows where every millisecond counts, and the mapping table has millions of entries leading to potential performance degradation if not carefully managed.
3. **Column-Level Access without Masking/Filtering**: If the requirement is simply to prevent users from seeing certain columns *at all*, rather than masking their content or filtering rows. This is typically handled by granting `SELECT` privileges only on specific columns or creating views that exclude sensitive columns.
    * *Situation*: A user should never see the `SSN` column. Instead of a DDM mapping table, simply don't grant `SELECT` on `SSN` or provide a view without it.

## Common Mistakes

1. **Forgetting to grant `SELECT` on the mapping table to the relevant roles**: The roles that need to be evaluated by the policy (e.g., `SALES_MANAGER_NA`) must have `SELECT` privilege on the mapping table, otherwise the policy evaluation will fail, and users might get "access denied" errors or no rows at all.
2. **Not placing the mapping table in the same database (or schema) as the protected data**: For optimal performance and ease of management, Snowflake recommends placing mapping tables in the same database as the protected table, especially if using `IS_DATABASE_ROLE_IN_SESSION`.
3. **Over-complicating mapping table logic**: Designing overly complex mapping tables or policy subqueries can lead to performance issues and make debugging difficult. Keep the logic as straightforward as possible.
4. **Not protecting the mapping table itself**: If sensitive information is stored in the mapping table (e.g., specific user-to-data mappings), it's crucial to apply appropriate access controls (like another Row Access Policy or granular grants) to the mapping table itself to prevent unauthorized access to the security logic.
5. **Not using memoizable functions for subqueries (for performance)**: For mapping table lookups within Row Access Policies, Snowflake suggests replacing subqueries with memoizable functions to improve query performance on policy-protected tables.

## Expected Output

When a user with a specific role (e.g., `SALES_MANAGER_NA`) queries the `sales.public.transactions` table, they should only see rows where the `region` matches their assigned region(s) in the `governance.security.sales_region_roles_map`.

**Example Table `sales.public.transactions` (as seen by `SALES_MANAGER_NA` role):**

| TRANSACTION_ID | PRODUCT_NAME | AMOUNT | REGION | SALE_DATE |
| :------------- | :----------- | :----- | :------------ | :--------- |
| 1 | Laptop | 1200.00 | North America | 2026-03-01 |
| 4 | Monitor | 300.00 | North America | 2026-03-04 |

**Explanation of columns:**

* **TRANSACTION_ID**: Unique identifier for the sale.
* **PRODUCT_NAME**: Name of the product sold.
* **AMOUNT**: The monetary value of the transaction.
* **REGION**: The geographic region where the sale occurred. This column is used by the Row Access Policy to filter results.
* **SALE_DATE**: The date of the transaction.

The output shows only transactions from 'North America', as the `SALES_MANAGER_NA` role is mapped to this region in the `sales_region_roles_map` table, and the `sales_region_policy` filters out all other regions.
