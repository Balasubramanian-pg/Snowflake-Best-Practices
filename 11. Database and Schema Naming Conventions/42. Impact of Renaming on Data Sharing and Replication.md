# Impact of Renaming on Data Sharing and Replication in Snowflake

**Description:**
Renaming database and schema objects in Snowflake can have significant impacts on data sharing and replication. When objects are renamed, references to them in shared databases or replicated data may break, leading to errors and data inconsistencies. Understanding these implications is crucial for maintaining data integrity and availability.

## Short Explanation
**Overview:** Renaming objects in Snowflake can affect data sharing and replication by breaking object references.

**Problem Solved:** This topic addresses the challenges of maintaining data consistency and availability when renaming database and schema objects in Snowflake.

**Importance:** It matters in analytics and warehousing because data sharing and replication are critical for collaboration and high availability.

**Use Cases:**
- Renaming a database object used in a shared database
- Renaming a schema object replicated across multiple databases

### Typical Scenario
A company has a shared database across teams, and one team renames a schema object used by another team, causing errors in the dependent team's queries.

### Core Snowflake SQL Objects Used
`ALTER DATABASE`, `ALTER SCHEMA`

### Code Source Execution
```sql
ALTER DATABASE my_database RENAME TO my_new_database;
```

### Example Execution Logic Step-by-Step
When you rename a database or schema object, Snowflake does not automatically update references to that object in shared databases or replicated data. For example, if you have a view in a shared database that references a table in the renamed database, the view will fail to resolve the table name.

### Implementation Example
To mitigate these issues, use fully qualified object names and consider using synonyms or views to abstract object references. For instance, create a synonym for the renamed object: CREATE SYNONYM my_synonym FOR my_new_database.my_schema.my_table;

### When Not to Use
