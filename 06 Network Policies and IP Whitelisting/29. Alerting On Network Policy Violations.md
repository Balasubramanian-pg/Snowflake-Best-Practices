# Alerting On Network Policy Violations in Snowflake

This concept addresses the critical need to proactively monitor and be notified about security breaches related to network access in your Snowflake environment. Network policies in Snowflake restrict which IP addresses can connect to your account. Alerting on violations ensures that any unauthorized access attempts are immediately flagged, allowing for a swift response to potential security threats.

**Short Explanation**

This SQL concept involves setting up monitoring within Snowflake to detect and report when connections originate from IP addresses not permitted by your defined network policies. It solves the problem of not knowing when unauthorized connection attempts occur, which is vital for maintaining data security and compliance. It's important in databases to ensure only approved sources can access sensitive data, and it's commonly used in security operations centers (SOCs) or by database administrators to enforce and verify access controls.

## Implementation Example

```sql
SELECT
    event_timestamp,
    event_type,
    user_name,
    client_ip_address,
    network_policy_name
FROM
    snowflake.account_usage.login_history
WHERE
    event_type = 'FAILED_LOGIN'
    AND client_ip_address NOT IN (SELECT ip_address FROM your_allowed_ips_table) -- Assuming a table of known allowed IPs
    AND DATEDIFF(hour, event_timestamp, CURRENT_TIMESTAMP()) <= 24 -- Check for events in the last 24 hours
ORDER BY
    event_timestamp DESC;
```

## Explanation of the Code

This query analyzes login attempts to identify potential network policy violations.

*   `SELECT event_timestamp, event_type, user_name, client_ip_address, network_policy_name`:
    *   **What it does:** Specifies the columns to retrieve from the `login_history` table.
    *   **How it changes the result set:** It projects only these five specific columns into the final output.
*   `FROM snowflake.account_usage.login_history`:
    *   **What it does:** Indicates the source table for the data, which is Snowflake's built-in view for login activity.
    *   **How it changes the result set:** Defines the initial dataset from which rows will be selected and filtered.
*   `WHERE event_type = 'FAILED_LOGIN'`:
    *   **What it does:** Filters the results to include only failed login attempts.
    *   **How it changes the result set:** Removes all successful login records, focusing on attempts that didn't go through.
*   `AND client_ip_address NOT IN (SELECT ip_address FROM your_allowed_ips_table)`:
    *   **What it does:** Further filters failed logins to those originating from IP addresses *not* present in a predefined table of allowed IP addresses. This is the core of detecting policy violations.
    *   **How it changes the result set:** Excludes failed logins from legitimately allowed IPs (e.g., mistyped password from a corporate office).
*   `AND DATEDIFF(hour, event_timestamp, CURRENT_TIMESTAMP()) <= 24`:
    *   **What it does:** Filters records to only include events that occurred within the last 24 hours.
    *   **How it changes the result set:** Limits the output to recent activity, making the alert timely.
*   `ORDER BY event_timestamp DESC`:
    *   **What it does:** Sorts the results by the `event_timestamp` in descending order.
    *   **How it changes the result set:** Presents the most recent violations at the top of the output.

## When to Use

1.  **Continuous Security Monitoring:** Integrate this query into a scheduled task or an external SIEM (Security Information and Event Management) system to run regularly (e.g., every hour).
    *   *Example Scenario:* A scheduled task runs hourly, detects a login attempt from an unrecognized IP in China, and triggers an email alert to the security team.
2.  **Post-Incident Analysis:** Use it to investigate suspicious activity after a security alert or an observed anomaly.
    *   *Example Scenario:* A user reports unusual account activity; you run this query over a longer period to see if there were any network policy violations preceding the reported incident.
3.  **Compliance Auditing:** Provide evidence of network access control enforcement for regulatory compliance (e.g., GDPR, HIPAA).
    *   *Example Scenario:* During an annual audit, you can demonstrate that mechanisms are in place to detect and report unauthorized network access attempts to your Snowflake environment.

## When Not to Use

1.  **Without Properly Defined Allowed IPs:** If `your_allowed_ips_table` is empty, incomplete, or not maintained, this query will generate many false positives or miss actual violations.
    *   *Inefficiency:* Constant false alerts can lead to alert fatigue, making real threats harder to spot.
2.  **As a Sole Security Measure:** This is a detection mechanism, not a preventative one. It won't stop the connection attempt itself, only alert you after it occurs.
    *   *Inefficiency:* Relying solely on this means you're always reacting, not preventing. Robust network policies should be in place first.
3.  **On Very High-Volume, Legitimate Failed Logins:** If your legitimate users frequently mistype passwords from allowed IPs, this query (without the `NOT IN your_allowed_ips_table` clause) could return a lot of noise.
    *   *Inefficiency:* While the example includes the `NOT IN` clause, if this clause is omitted or `your_allowed_ips_table` is not accurate, the query might return too much irrelevant data, obscuring actual threats.

## Common Mistakes

1.  **Not Maintaining the `your_allowed_ips_table`:** If the table of allowed IPs isn't kept up-to-date with legitimate new IP ranges or changes, the alerts will either miss valid violations or generate excessive false positives.
2.  **Ignoring `event_type`:** Just looking for any login history entry without filtering for `FAILED_LOGIN` might include successful logins from allowed IPs, creating noise.
3.  **Lack of Alerting Integration:** Running the query manually without integrating it into an automated alerting system (like Snowflake Tasks + external notification services or a SIEM) means human intervention is always needed to check for violations.
4.  **Not considering `network_policy_name`:** While not directly used in the `WHERE` clause of the example, ignoring the `network_policy_name` in the output can make it harder to understand which specific policy was theoretically violated if multiple policies are in place.

## Expected Output

The resulting dataset will show records of failed login attempts that originated from IP addresses *not* explicitly listed as allowed in `your_allowed_ips_table`, ordered by the most recent events. Each row represents a suspicious connection attempt that should be investigated.

| EVENT\_TIMESTAMP           | EVENT\_TYPE   | USER\_NAME | CLIENT\_IP\_ADDRESS | NETWORK\_POLICY\_NAME |
| :------------------------- | :------------ | :--------- | :------------------ | :-------------------- |
| 2026-03-14 09:30:15.000 +0000 | FAILED\_LOGIN | jane\_doe  | 192.168.1.100       | external\_access\_np  |
| 2026-03-14 09:25:40.000 +0000 | FAILED\_LOGIN | system     | 203.0.113.45        | security\_policy\_01 |
| 2026-03-14 09:05:01.000 +0000 | FAILED\_LOGIN | admin\_user| 172.16.0.200        | NULL                  |

*   **EVENT\_TIMESTAMP:** The exact time the failed login attempt occurred.
*   **EVENT\_TYPE:** Will always be 'FAILED\_LOGIN' in this filtered output.
*   **USER\_NAME:** The username that was attempted to be used.
*   **CLIENT\_IP\_ADDRESS:** The IP address from which the failed login attempt originated, which is not in the allowed list.
*   **NETWORK\_POLICY\_NAME:** The network policy associated with the connection, if applicable. (Can be NULL if no specific policy was directly triggered, but the IP is still external).
