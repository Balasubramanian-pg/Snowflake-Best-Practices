# Restricting Account Usage Views in Snowflake

This concept addresses how to control access to Snowflake's Account Usage views, which provide metadata and historical data about your Snowflake account's operations. These views are crucial for monitoring, auditing, and cost analysis, but their sensitive nature means access often needs to be restricted to specific roles or users.

**Short Explanation**

Restricting Account Usage views means setting permissions so only authorized personnel can query these system-provided views. This prevents unauthorized access to sensitive operational data, ensuring data governance and security within your Snowflake environment. It's important for maintaining a secure and auditable data platform, commonly implemented by granting `IMPORT SHARE` privilege to custom roles.

- This SQL feature solves the problem of uncontrolled access to sensitive operational metadata, protecting against information leakage and misuse.
- It's important in databases and analytics for security, compliance, and proper separation of duties, ensuring that only those who need to see operational data can.
- It is commonly used in real workflows by administrators, security teams, and finance departments for auditing, monitoring resource consumption, and managing costs.

## Implementation Example

```sql
-- Create a custom role for viewing account usage
CREATE ROLE ACCOUNT_USAGE_VIEWER;

-- Grant the necessary privilege to the custom role
-- The IMPORT SHARE privilege is required to access Account Usage views
GRANT IMPORT SHARE ON ACCOUNT TO ROLE ACCOUNT_USAGE_VIEWER;

-- Grant the custom role to a user
GRANT ROLE ACCOUNT_USAGE_VIEWER TO USER ANALYST_USER;

-- Grant the custom role to another role (e.g., SYSADMIN) if needed
GRANT ROLE ACCOUNT_USAGE_VIEWER TO ROLE SYSADMIN;

-- Now, as ANALYST_USER or a user with SYSADMIN role,
-- you can query Account Usage views, e.g.:
USE ROLE ACCOUNT_USAGE_VIEWER;
SELECT
    query_id,
    query_text,
    user_name,
    start_time,
    end_time,
    total_elapsed_time,
    warehouse_name,
    warehouse_size
FROM
    SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(hour, -24, CURRENT_TIMESTAMP())
ORDER BY
    start_time DESC
LIMIT 10;
```

## Explanation of the Code

- **`CREATE ROLE ACCOUNT_USAGE_VIEWER;`**: This statement creates a new custom role specifically designed to hold permissions for viewing account usage data. This adheres to the principle of least privilege, allowing fine-grained control over access.
- **`GRANT IMPORT SHARE ON ACCOUNT TO ROLE ACCOUNT_USAGE_VIEWER;`**: This is the critical step. The `IMPORT SHARE` privilege on the `ACCOUNT` object is what grants access to the `SNOWFLAKE.ACCOUNT_USAGE` schema and its views.
    - **What it does**: It allows the `ACCOUNT_USAGE_VIEWER` role to read data from shared schemas, specifically the `ACCOUNT_USAGE` schema provided by Snowflake.
    - **How it changes the result set**: Without this grant, any queries attempted against `SNOWFLAKE.ACCOUNT_USAGE` views by a user with `ACCOUNT_USAGE_VIEWER` role (or any role inheriting it) would result in a "no privileges" error. With it, the queries will execute successfully, returning the requested operational data.
- **`GRANT ROLE ACCOUNT_USAGE_VIEWER TO USER ANALYST_USER;`**: This assigns the newly created custom role to a specific user named `ANALYST_USER`.
    - **What it does**: It makes the `ACCOUNT_USAGE_VIEWER` role available to `ANALYST_USER`, allowing them to activate and use its permissions.
    - **How it changes the result set**: The user `ANALYST_USER` will now be able to switch to `ACCOUNT_USAGE_VIEWER` role and query the `ACCOUNT_USAGE` views.
- **`GRANT ROLE ACCOUNT_USAGE_VIEWER TO ROLE SYSADMIN;`**: This statement makes `ACCOUNT_USAGE_VIEWER` a sub-role of `SYSADMIN`.
    - **What it does**: It allows users who have the `SYSADMIN` role to inherit the permissions of `ACCOUNT_USAGE_VIEWER` without explicitly switching roles.
    - **How it changes the result set**: Any user with `SYSADMIN` activated will now implicitly have access to `ACCOUNT_USAGE` views.
- **`USE ROLE ACCOUNT_USAGE_VIEWER;`**: This command activates the `ACCOUNT_USAGE_VIEWER` role for the current session.
    - **What it does**: It switches the current context to use the permissions associated with `ACCOUNT_USAGE_VIEWER`.
    - **How it changes the result set**: Queries run after this command will be evaluated based on the privileges granted to `ACCOUNT_USAGE_VIEWER`.
- **`SELECT ... FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY WHERE ... LIMIT ...;`**: This is a sample query accessing one of the `ACCOUNT_USAGE` views.
    - **`SELECT`**: Specifies the columns to retrieve (query_id, query_text, user_name, etc.).
    - **`FROM`**: Indicates the source of the data, in this case, the `QUERY_HISTORY` view within the `SNOWFLAKE.ACCOUNT_USAGE` schema.
    - **`WHERE`**: Filters the results to show only queries from the last 24 hours.
    - **`ORDER BY`**: Sorts the results by start time in descending order.
    - **`LIMIT`**: Restricts the number of rows returned to 10.
    - **How it changes the result set**: This clause defines which specific operational data is retrieved from the `QUERY_HISTORY` view, how it's filtered, sorted, and how many rows are returned.

## When to Use

1.  **Security and Compliance**: When sensitive operational data (like user activity, query text, or resource consumption) needs to be viewed only by authorized personnel to meet compliance regulations (e.g., GDPR, HIPAA) or internal security policies.
    *   *Example Scenario*: A financial institution needs to ensure that only its security and audit teams can access logs of all SQL queries executed, to detect potential insider threats or data breaches.
2.  **Cost Management and Resource Monitoring**: When you want to grant specific finance or operations teams the ability to monitor Snowflake credit consumption, warehouse usage, or storage costs without giving them broader administrative privileges.
    *   *Example Scenario*: A FinOps team needs to analyze daily warehouse usage and query costs to optimize spending, but they don't need permissions to create or manage actual warehouses or databases.
3.  **Separation of Duties**: To implement a strong separation of duties, ensuring that roles responsible for data management are distinct from roles responsible for monitoring the platform's operational aspects.
    *   *Example Scenario*: Data engineers manage data pipelines and tables, while a separate "Platform Operations" team uses Account Usage views to monitor system health and performance, preventing one team from having too much control.

## When Not to Use

1.  **Granting Broad `ACCOUNTADMIN` Privileges**: If you already grant `ACCOUNTADMIN` to a user, explicitly granting `IMPORT SHARE` to a sub-role for them is redundant, as `ACCOUNTADMIN` already has all privileges.
    *   *Inefficiency*: It adds unnecessary complexity to the role hierarchy without providing additional security or functionality.
2.  **When Access is Not Required**: If a user or role genuinely does not need to see any operational or metadata from the `ACCOUNT_USAGE` schema, do not grant them the privilege.
    *   *Security Risk*: Over-granting access, even to seemingly innocuous views, can inadvertently expose sensitive information or make auditing harder.
3.  **For Performance Monitoring of Specific Queries (Developer Level)**: Developers often need to see the execution plan and statistics for their own queries. While `ACCOUNT_USAGE.QUERY_HISTORY` contains this, `QUERY_HISTORY` in the `INFORMATION_SCHEMA` (which requires less privilege and is often database/schema scoped) might be more appropriate for individual query debugging.
    *   *Overkill/Wrong Scope*: Granting `IMPORT SHARE` for `ACCOUNT_USAGE` views might be too broad for a developer just needing to debug their own queries in a specific database.

## Common Mistakes

1.  **Forgetting `IMPORT SHARE`**: Developers often forget that simply `GRANT`ing `SELECT` on individual `ACCOUNT_USAGE` views (e.g., `QUERY_HISTORY`) isn't enough; the `IMPORT SHARE` privilege on the `ACCOUNT` level is fundamentally required to access the shared `SNOWFLAKE.ACCOUNT_USAGE` schema itself.
2.  **Granting `ACCOUNTADMIN` for Monitoring**: Instead of creating a specific role, organizations sometimes default to granting `ACCOUNTADMIN` to individuals who only need to monitor account usage, which provides far too many privileges.
3.  **Assuming `SNOWFLAKE.ACCOUNT_USAGE` is a regular database**: It's a special shared database, and its access mechanism is different from user-defined databases. Attempting to `CREATE SCHEMA` or `CREATE TABLE` within `SNOWFLAKE` will fail.
4.  **Not using custom roles**: Directly granting `IMPORT SHARE` to built-in roles like `PUBLIC` or `SYSADMIN` can make it harder to audit who has access and violates the principle of least privilege.
5.  **Confusing `ACCOUNT_USAGE` with `INFORMATION_SCHEMA`**: While both provide metadata, `ACCOUNT_USAGE` is account-wide and historical (data can be delayed), whereas `INFORMATION_SCHEMA` is more real-time and scoped to the current database. Using the wrong one for the task can lead to incomplete data or performance issues.

## Expected Output

The resulting dataset will contain rows representing past queries executed in the Snowflake account, with details about each query. Each row corresponds to a single query execution.

| QUERY_ID                             | QUERY_TEXT                                  | USER_NAME   | START_TIME                  | END_TIME                    | TOTAL_ELAPSED_TIME | WAREHOUSE_NAME | WAREHOUSE_SIZE |
| :----------------------------------- | :------------------------------------------ | :---------- | :-------------------------- | :-------------------------- | :----------------- | :------------- | :------------- |
| 1a2b3c4d-5e6f-7890-abcd-ef1234567890 | `SELECT * FROM MY_TABLE WHERE ID = 1;`      | ANALYST_USER | 2026-03-13 14:30:00.000 +0000 | 2026-03-13 14:30:01.200 +0000 | 1200               | COMPUTE_WH     | XSMALL         |
| 0f1e2d3c-4b5a-6789-0fed-cb9876543210 | `INSERT INTO LOGS (MSG) VALUES ('New entry');` | ETL_USER    | 2026-03-13 15:05:10.500 +0000 | 2026-03-13 15:05:11.000 +0000 | 500                | SMALL_WH       | SMALL          |
| 98765432-10fe-dcba-9876-543210fedcba | `SHOW DATABASES;`                           | ADMIN_USER  | 2026-03-13 16:10:20.100 +0000 | 2026-03-13 16:10:20.300 +0000 | 200                | COMPUTE_WH     | XSMALL         |

- **QUERY_ID**: A unique identifier for the executed query.
- **QUERY_TEXT**: The actual SQL text of the query that was run.
- **USER_NAME**: The name of the user who executed the query.
- **START_TIME**: The timestamp when the query execution started.
- **END_TIME**: The timestamp when the query execution finished.
- **TOTAL_ELAPSED_TIME**: The total time, in milliseconds, the query took to execute.
- **WAREHOUSE_NAME**: The name of the virtual warehouse used to execute the query.
- **WAREHOUSE_SIZE**: The size of the virtual warehouse (e.g., XSMALL, SMALL, MEDIUM).
