# Managing Country Based Row Access in Snowflake

Country-based row access in Snowflake is a data governance mechanism that restricts which rows a user can see in a table based on their assigned country or geographical region. This is crucial for complying with data residency regulations, privacy laws (like GDPR or CCPA), and internal policies that dictate where data can be accessed and by whom. It ensures that users only view data relevant and permissible to their jurisdiction.

**Short Explanation**

This SQL feature solves the problem of data access control based on geographical location, ensuring that users only see data that aligns with their country's regulations or internal business rules. It's important in databases and analytics for maintaining data privacy, adhering to compliance standards, and preventing unauthorized access to sensitive regional data. It's commonly used in global enterprises, financial institutions, and healthcare organizations to manage data access across different countries.

- What problem does this SQL feature solve? It solves the problem of enforcing data access policies based on a user's country or region, ensuring compliance and privacy.
- Why is it important in databases or analytics? It's vital for data governance, regulatory compliance, and protecting sensitive information across global operations.
- Where is it commonly used in real workflows? Global businesses, multinational corporations, and any organization dealing with country-specific data regulations.

## Implementation Example

```sql
-- 1. Create a sample table
CREATE TABLE customer_data (
    customer_id INT,
    customer_name VARCHAR,
    country VARCHAR,
    sensitive_info VARCHAR
);

-- 2. Insert sample data
INSERT INTO customer_data VALUES
(1, 'Alice', 'USA', 'SSN_123'),
(2, 'Bob', 'Canada', 'Passport_456'),
(3, 'Charlie', 'UK', 'NHS_789'),
(4, 'David', 'USA', 'SSN_012'),
(5, 'Eve', 'Germany', 'TaxID_345');

-- 3. Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY country_access_policy
AS (country_col VARCHAR) RETURNS BOOLEAN ->
    -- User's current role is allowed to see all countries for example (ADMIN role)
    -- OR the user's mapped country matches the row's country
    -- OR the user's secondary roles contain a role for that country
    (CURRENT_ROLE() = 'ACCOUNTADMIN') OR
    (EXISTS (SELECT 1 FROM user_country_mapping WHERE user_name = CURRENT_USER() AND country = country_col)) OR
    (ARRAY_CONTAINS(country_col::VARIANT, CURRENT_SECONDARY_ROLES()));

-- 4. Create a table for user-country mapping (or use roles directly)
CREATE TABLE user_country_mapping (
    user_name VARCHAR,
    country VARCHAR
);

INSERT INTO user_country_mapping VALUES
('USER_USA', 'USA'),
('USER_UK', 'UK'),
('USER_GERMANY', 'Germany');

-- 5. Apply the policy to the table
ALTER TABLE customer_data ADD ROW ACCESS POLICY country_access_policy ON (country);

-- 6. Test with a user (assuming USER_USA is logged in)
-- As USER_USA (role could be assigned 'USA' secondary role or mapped in user_country_mapping table)
SELECT * FROM customer_data;
```

## Explanation of the Code

This example demonstrates how to set up country-based row access using Snowflake's Row Access Policies.

- `CREATE TABLE customer_data (...)`: This statement creates a sample table named `customer_data` which will hold customer information, including their `country`. This is the table whose rows we want to protect.
- `INSERT INTO customer_data VALUES (...)`: This populates the `customer_data` table with some example records, each associated with a specific country.
- `CREATE OR REPLACE ROW ACCESS POLICY country_access_policy AS (country_col VARCHAR) RETURNS BOOLEAN -> (...)`: This defines the core logic for row access.
    - `country_col VARCHAR`: This specifies that the policy will take a column of type `VARCHAR` as input, which will typically be the `country` column from the `customer_data` table.
    - `RETURNS BOOLEAN`: The policy must return `TRUE` for a row to be visible, and `FALSE` for it to be hidden.
    - `(CURRENT_ROLE() = 'ACCOUNTADMIN') OR ...`: This is the policy's condition. It allows `ACCOUNTADMIN` to see all rows.
    - `(EXISTS (SELECT 1 FROM user_country_mapping WHERE user_name = CURRENT_USER() AND country = country_col))`: This part checks if the currently logged-in user (`CURRENT_USER()`) has an entry in the `user_country_mapping` table that grants them access to the `country_col` of the current row. This is one way to manage user-to-country entitlements.
    - `(ARRAY_CONTAINS(country_col::VARIANT, CURRENT_SECONDARY_ROLES()))`: This alternative condition checks if the row's `country_col` value is present in the `CURRENT_SECONDARY_ROLES()` of the user. This is a common and flexible way to assign country-specific roles (e.g., a role named 'USA' could grant access to 'USA' data).
- `CREATE TABLE user_country_mapping (...)`: This creates a lookup table to explicitly map users to countries they are allowed to access. This offers granular control.
- `INSERT INTO user_country_mapping VALUES (...)`: Populates the mapping table.
- `ALTER TABLE customer_data ADD ROW ACCESS POLICY country_access_policy ON (country);`: This is the crucial step that attaches the created row access policy to the `customer_data` table, specifically linking it to the `country` column. From this point forward, every query on `customer_data` will implicitly filter rows based on `country_access_policy`.
- `SELECT * FROM customer_data;`: This is a test query. When executed by `USER_USA`, only rows where `country` is 'USA' would be returned (assuming `USER_USA` is mapped to 'USA' or has a 'USA' secondary role).

## When to Use

1.  **Regulatory Compliance:** When your data processing must adhere to specific regional data privacy laws like GDPR (Europe), CCPA (California), or similar regulations that restrict data access based on the user's location or the data's origin.
    *   *Scenario:* A healthcare company needs to ensure that patient records from the EU are only accessible by authorized personnel within the EU, even if the database is globally distributed.
2.  **Internal Business Policy:** To enforce internal data governance policies where different regions or business units should only access data relevant to their operations.
    *   *Scenario:* A multinational sales organization wants sales managers in the APAC region to only see sales data from APAC countries, and not from EMEA or Americas.
3.  **Data Residency Requirements:** When there are strict data residency requirements that dictate certain data must physically reside and be accessed only within a specific country's borders.
    *   *Scenario:* A financial institution handles sensitive client investment data that, by law, cannot be viewed by anyone outside the country where the client resides.

## When Not to Use

1.  **Performance Critical Workloads on Small Tables:** For very small tables or high-performance, low-latency queries where the overhead of policy evaluation on every query might introduce unacceptable latency, especially if the filtering is simple and could be handled at the application layer.
    *   *Situation:* A lookup table with only 10 rows that needs to be accessed thousands of times per second.
2.  **Overly Complex Policy Logic:** If the row access logic becomes extremely intricate, involving multiple nested conditions, joins, or external lookups within the policy itself, it can become difficult to maintain and debug, and may negatively impact query performance.
    *   *Situation:* A policy that needs to check 10 different user attributes against 5 different data columns across 3 lookup tables.
3.  **Static Data Access:** When data access patterns are static and don't change frequently, and security requirements are not country-specific. Simple role-based access control (RBAC) at the table or schema level might be more straightforward and efficient.
    *   *Situation:* A table containing public product descriptions that all users should see, regardless of their location.

## Common Mistakes

1.  **Forgetting to Grant Access to the Policy:** The user who owns the policy (or a role with ownership) must grant privileges on the policy to the roles that will be querying the protected table. Without this, users won't be able to query the table at all, not just be filtered.
2.  **Inefficient Policy Logic:** The logic within the row access policy is executed for every row, for every query. If the policy includes expensive subqueries, functions, or joins, it can significantly degrade query performance. Optimizing the policy's SQL is crucial.
3.  **Not Handling Nulls or Missing Data:** If the column the policy relies on (`country` in our example) can contain `NULL` values or values not covered by the policy logic, unexpected access or denial of access can occur. The policy must explicitly handle these edge cases.
4.  **Overlooking `ACCOUNTADMIN` Access:** While `ACCOUNTADMIN` bypasses policies by default, testing should always be done with non-admin roles to ensure the policy functions as intended. Relying on `ACCOUNTADMIN` for testing can mask policy errors.

## Expected Output

The expected output is a filtered dataset where only rows that satisfy the row access policy for the current user are returned. Rows where the `country` column matches the countries the user is authorized to view will be included, and all other rows will be silently excluded.

Example Table (if `USER_USA` is querying):

| customer_id | customer_name | country | sensitive_info |
| :---------- | :------------ | :------ | :------------- |
| 1           | Alice         | USA     | SSN_123        |
| 4           | David         | USA     | SSN_012        |

-   **customer_id**: The unique identifier for the customer.
-   **customer_name**: The name of the customer.
-   **country**: The country associated with the customer's data. This column is used by the row access policy to determine visibility.
-   **sensitive_info**: Other sensitive data about the customer.

This table shows only the rows where the `country` is 'USA', as `USER_USA` is configured to only access data from the USA. All other country-specific rows (`Canada`, `UK`, `Germany`) are hidden from this user.
