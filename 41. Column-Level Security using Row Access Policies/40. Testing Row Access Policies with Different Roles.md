# Testing Row Access Policies with Different Roles in Snowflake

Testing Row Access Policies with Different Roles in Snowflake involves verifying that data filtering based on policies works as expected for users assigned various roles. This ensures data security and compliance by confirming that users only see the data they are authorized to access, mimicking real-world access scenarios.

**Short Explanation**

This concept focuses on validating that your defined Row Access Policies correctly restrict data visibility based on a user's role. It solves the problem of ensuring granular data security in a multi-user environment by confirming that different roles indeed get different data views. It's critical in databases to prevent unauthorized data exposure and is commonly used in audit processes, security testing, and UAT (User Acceptance Testing) for data-sensitive applications.

- **What problem does this SQL feature solve?** It solves the problem of ensuring that Row Access Policies are correctly applied and enforce the intended data visibility restrictions for different user roles, thereby maintaining data security and compliance.
- **Why is it important in databases or analytics?** It's vital for maintaining data governance, preventing data breaches, and complying with regulations (like GDPR, HIPAA) by guaranteeing that sensitive information is only accessible to authorized personnel.
- **Where is it commonly used in real workflows?** It's frequently used during the development, testing, and deployment phases of Snowflake solutions that involve sensitive data, particularly in industries like finance, healthcare, and government.

## Implementation Example

```sql
-- Assume a table 'SALES_DATA' with columns: PRODUCT_ID, REGION, SALES_AMOUNT, EMPLOYEE_ROLE
-- And a Row Access Policy 'sales_policy' defined as:
--   CREATE ROW ACCESS POLICY sales_policy AS (region VARCHAR) RETURNS BOOLEAN ->
--     CASE
--       WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE
--       WHEN CURRENT_ROLE() = 'EAST_REGION_ANALYST' THEN region = 'EAST'
--       WHEN CURRENT_ROLE() = 'WEST_REGION_ANALYST' THEN region = 'WEST'
--       ELSE FALSE
--     END;
-- And the policy is attached to SALES_DATA:
--   ALTER TABLE SALES_DATA ADD ROW ACCESS POLICY sales_policy ON (REGION);

-- 1. Test as ADMIN_ROLE (should see all data)
USE ROLE ADMIN_ROLE;
SELECT PRODUCT_ID, REGION, SALES_AMOUNT FROM SALES_DATA ORDER BY REGION;

-- 2. Test as EAST_REGION_ANALYST (should only see 'EAST' region data)
USE ROLE EAST_REGION_ANALYST;
SELECT PRODUCT_ID, REGION, SALES_AMOUNT FROM SALES_DATA ORDER BY REGION;

-- 3. Test as WEST_REGION_ANALYST (should only see 'WEST' region data)
USE ROLE WEST_REGION_ANALYST;
SELECT PRODUCT_ID, REGION, SALES_AMOUNT FROM SALES_DATA ORDER BY REGION;

-- 4. Test as a role with no specific access (should see no data)
USE ROLE PUBLIC; -- Or any other role not explicitly handled by the policy
SELECT PRODUCT_ID, REGION, SALES_AMOUNT FROM SALES_DATA ORDER BY REGION;
```

## Explanation of the Code

This example demonstrates how to switch between different Snowflake roles and query a table protected by a Row Access Policy to observe the varying results.

- **`USE ROLE <role_name>;`**:
    - **What it does**: This command changes the current active role for the session to the specified `<role_name>`.
    - **How it changes the result set**: It doesn't directly change the result set but rather changes the context under which subsequent queries are executed. The `CURRENT_ROLE()` function within the Row Access Policy will then return the newly activated role, dictating which rows are visible to the user.

- **`SELECT PRODUCT_ID, REGION, SALES_AMOUNT FROM SALES_DATA ORDER BY REGION;`**:
    - **What it does**: This is a standard `SELECT` statement that retrieves product ID, region, and sales amount from the `SALES_DATA` table.
    - **How it changes the result set**: When a Row Access Policy is active on `SALES_DATA`, this `SELECT` query will only return rows for which the policy's condition evaluates to `TRUE` for the `CURRENT_ROLE()`. The `ORDER BY REGION` clause simply sorts the visible rows by region.

The core of the testing lies in observing how the results of the `SELECT` statement change as the `USE ROLE` command is executed with different roles, confirming that the `sales_policy` filters data correctly based on the active role.

## When to Use

1.  **Policy Development & Debugging**: When initially creating or modifying a Row Access Policy, you need to verify that it functions as intended for all relevant roles before deployment.
    *   *Scenario*: You've just written a policy to restrict customer data by sales territory. You'd switch between roles like 'EAST_SALES_MANAGER' and 'WEST_SALES_MANAGER' to confirm they only see their respective territories.
2.  **User Acceptance Testing (UAT)**: During the UAT phase, business users or data owners should confirm that the security measures align with their expectations and compliance requirements.
    *   *Scenario*: A compliance officer, playing the role of a 'FINANCE_ANALYST', queries a financial transactions table to ensure they can only see data for their allowed departments.
3.  **Regular Security Audits**: Periodically, security teams need to audit data access to ensure policies are still effective and no unauthorized data exposure has occurred due to changes in roles, users, or policies.
    *   *Scenario*: An auditor connects to Snowflake, assumes a 'DATA_STEWARD' role, and queries sensitive employee data to verify that the Row Access Policy correctly hides salaries from this role, despite the role having broader data access in other areas.

## When Not to Use

1.  **For General Data Exploration (as an end-user)**: An end-user who simply wants to query data shouldn't be involved in `USE ROLE` commands for testing policies. Their role context should already be set correctly.
    *   *Situation*: A business analyst is querying sales figures for their daily report. They should simply `SELECT` from the table, not switch roles to test policy behavior.
2.  **When the policy is not applied**: If a Row Access Policy hasn't been attached to the target table, using `USE ROLE` to test its effects on that table will yield no filtering, as no policy is active.
    *   *Situation*: You've defined a `hr_policy` but forgot to `ALTER TABLE employees ADD ROW ACCESS POLICY hr_policy ON (department);`. Testing with `USE ROLE` will show all rows regardless of the role.
3.  **As a Substitute for Proper Role Hierarchy Design**: While testing with roles is crucial, it shouldn't replace a well-designed role hierarchy where privileges and inherited roles are structured logically.
    *   *Situation*: Instead of creating a granular role for every possible data slice, use a more generic role and rely heavily on complex policies to derive access. This can become unmanageable and hard to test.

## Common Mistakes

1.  **Forgetting to `USE ROLE`**: Developers often forget to explicitly switch roles before executing `SELECT` statements, leading them to see results based on their default or previous role, not the one they intend to test.
2.  **Incorrect Policy Logic**: The policy itself might contain errors (e.g., typos in role names, incorrect `CASE` conditions, or faulty column references), causing unexpected data visibility.
3.  **Missing Policy Application**: The Row Access Policy might be defined but not actually attached to the table (`ALTER TABLE ... ADD ROW ACCESS POLICY ...`), leading to no filtering whatsoever.
4.  **Testing with an `ACCOUNTADMIN` role**: The `ACCOUNTADMIN` role bypasses Row Access Policies by default. Testing policies with this role will always show all data, which is misleading for policy validation.
5.  **Not verifying negative cases**: Only testing that a role *can* see its allowed data is insufficient. It's crucial to also test that it *cannot* see unauthorized data.

## Expected Output

The expected output will vary based on the role currently active.

**1. When `ADMIN_ROLE` is active:**
The output should include all rows from the `SALES_DATA` table, as the `ADMIN_ROLE` is designed to bypass regional restrictions.

| PRODUCT_ID | REGION | SALES_AMOUNT |
| :--------- | :----- | :----------- |
| P101       | EAST   | 1500.00      |
| P102       | WEST   | 2000.00      |
| P103       | NORTH  | 1200.00      |
| P104       | SOUTH  | 800.00       |

**2. When `EAST_REGION_ANALYST` is active:**
The output should only include rows where `REGION` is 'EAST'.

| PRODUCT_ID | REGION | SALES_AMOUNT |
| :--------- | :----- | :----------- |
| P101       | EAST   | 1500.00      |

**3. When `WEST_REGION_ANALYST` is active:**
The output should only include rows where `REGION` is 'WEST'.

| PRODUCT_ID | REGION | SALES_AMOUNT |
| :--------- | :----- | :----------- |
| P102       | WEST   | 2000.00      |

**4. When `PUBLIC` (or an unauthorized role) is active:**
The output should be an empty result set, as the policy dictates `FALSE` for roles not explicitly granted access.

| PRODUCT_ID | REGION | SALES_AMOUNT |
| :--------- | :----- | :----------- |
|            |        |              |

**Explanation of Columns:**
- `PRODUCT_ID`: The unique identifier for a product.
- `REGION`: The geographical region where the sale occurred. This column is central to the Row Access Policy's filtering logic.
- `SALES_AMOUNT`: The monetary value of the sale.
