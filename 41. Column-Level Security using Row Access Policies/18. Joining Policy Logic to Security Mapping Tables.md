# Joining Policy Logic to Security Mapping Tables in Snowflake

This concept addresses how to dynamically apply security policies, like row-access policies or masking policies, based on user roles or attributes stored in separate mapping tables within Snowflake. It allows for flexible and scalable data access control by separating the policy definition from the user-specific permissions. This is crucial for implementing fine-grained access control (FGAC) effectively.

**Short Explanation**

This idea involves creating a security policy that doesn't hardcode specific user roles or values. Instead, it references a separate lookup table (the "security mapping table") that contains the current assignments of users or roles to data access levels. The policy logic then joins with this mapping table to determine whether a user can see a particular row or an unmasked column value.

- **What problem does this SQL feature solve?** It solves the problem of rigid access control that requires frequent policy updates when user roles or permissions change, and the challenge of managing complex access rules for large datasets and many users.
- **Why is it important in databases or analytics?** It's vital for implementing scalable, maintainable, and dynamic data governance, ensuring that sensitive data is only accessible by authorized individuals, which is critical for compliance and security.
- **Where is it commonly used in real workflows?** It's commonly used in enterprises that need fine-grained access control for sensitive data (e.g., customer PII, financial records, healthcare data) where access depends on the user's role, department, or other attributes that might change over time.

## Implementation Example

```sql
-- Create sample tables for demonstration
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_id INT,
    region VARCHAR,
    amount DECIMAL(10, 2),
    customer_name VARCHAR,
    customer_email VARCHAR
);

CREATE OR REPLACE TABLE user_security_map (
    user_name VARCHAR,
    allowed_region VARCHAR
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 101, 'EAST', 150.00, 'Alice', 'alice@example.com'),
(2, 102, 'WEST', 200.00, 'Bob', 'bob@example.com'),
(3, 103, 'EAST', 120.00, 'Charlie', 'charlie@example.com'),
(4, 104, 'SOUTH', 300.00, 'David', 'david@example.com');

INSERT INTO user_security_map VALUES
('ANALYST_EAST', 'EAST'),
('ANALYST_WEST', 'WEST');

-- Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY sales_row_access_policy
AS (region VARCHAR) RETURNS BOOLEAN ->
    EXISTS (
        SELECT 1
        FROM user_security_map usm
        WHERE usm.user_name = CURRENT_ROLE()
          AND usm.allowed_region = region
    ) OR
    CURRENT_ROLE() = 'ADMIN';

-- Apply the policy to the sales_data table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_row_access_policy ON (region);

-- Create a masking policy for customer_email
CREATE OR REPLACE MASKING POLICY email_masking_policy AS (val STRING) RETURNS STRING ->
    CASE
        WHEN CURRENT_ROLE() = 'ADMIN' THEN val
        WHEN EXISTS (SELECT 1 FROM user_security_map WHERE user_name = CURRENT_ROLE()) THEN '***MASKED***'
        ELSE val
    END;

-- Apply the masking policy to the customer_email column
ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_masking_policy;

-- Grant select on tables to a test role
GRANT SELECT ON TABLE sales_data TO ROLE ANALYST_EAST;
GRANT SELECT ON TABLE sales_data TO ROLE ANALYST_WEST;
GRANT SELECT ON TABLE user_security_map TO ROLE ANALYST_EAST;
GRANT SELECT ON TABLE user_security_map TO ROLE ANALYST_WEST;

-- Test as ANALYST_EAST
-- USE ROLE ANALYST_EAST;
-- SELECT * FROM sales_data;

-- Test as ANALYST_WEST
-- USE ROLE ANALYST_WEST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example demonstrates how a `ROW ACCESS POLICY` and a `MASKING POLICY` are defined to query a `user_security_map` table, enabling dynamic access control.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement defines a sample table containing sales transaction information.
- **`CREATE OR REPLACE TABLE user_security_map (...)`**: This defines a mapping table that links user roles (represented by `user_name`) to specific `allowed_region`s. This is the "security mapping table."
- **`INSERT INTO ... VALUES (...)`**: These statements populate the sample tables with data.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_row_access_policy AS (region VARCHAR) RETURNS BOOLEAN -> ...`**:
    - **What it does**: This creates a row-access policy named `sales_row_access_policy`. When applied, this policy will filter rows from a table based on its `region` column.
    - **How it changes the result set**: For every row, the policy evaluates if the `region` of that row is present in the `user_security_map` for the `CURRENT_ROLE()`, or if the `CURRENT_ROLE()` is 'ADMIN'. If the condition is `TRUE`, the row is included; otherwise, it's excluded. This effectively limits users to seeing only data for regions they are authorized for.
    - **`EXISTS (SELECT 1 FROM user_security_map usm WHERE usm.user_name = CURRENT_ROLE() AND usm.allowed_region = region)`**: This is the core logic. It performs a subquery (an implicit join) against the `user_security_map` table to check if the user's `CURRENT_ROLE()` is explicitly allowed access to the `region` of the current row being evaluated.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_row_access_policy ON (region)`**:
    - **What it does**: This applies the `sales_row_access_policy` to the `sales_data` table, using the `region` column as the policy argument.
    - **How it changes the result set**: From this point on, any query on `sales_data` will be implicitly filtered by the policy based on the `region` column and the user's current role.
- **`CREATE OR REPLACE MASKING POLICY email_masking_policy AS (val STRING) RETURNS STRING -> ...`**:
    - **What it does**: This creates a masking policy that will transform the value of a string column.
    - **How it changes the result set**: It checks if the `CURRENT_ROLE()` is 'ADMIN'. If so, it returns the original `val`. If the `CURRENT_ROLE()` exists in the `user_security_map` (indicating it's an analyst role), it masks the email with '***MASKED***'. Otherwise, it returns the original value, potentially for roles not managed by this specific policy. This ensures sensitive email addresses are hidden from analysts.
- **`ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_masking_policy`**:
    - **What it does**: Applies the `email_masking_policy` to the `customer_email` column in the `sales_data` table.
    - **How it changes the result set**: Any query selecting `customer_email` will now have the column's values masked according to the policy's logic.
- **`GRANT SELECT ON TABLE sales_data TO ROLE ANALYST_EAST;`**: Grants permission for specific roles to query the `sales_data` table.

## When to Use

1.  **Dynamic Role-Based Access Control (RBAC)**: When different user roles need access to different subsets of data (e.g., sales teams seeing only their region's data, HR staff seeing only their department's employee records).
    *   **Scenario**: A global sales organization wants to ensure that sales representatives can only view sales data for their assigned geographical region, and this assignment might change frequently without requiring policy modifications.
2.  **Sensitive Data Masking based on User Privileges**: When certain columns contain sensitive information (like PII, financial details) that should be masked or tokenized for most users but fully visible to a few authorized individuals.
    *   **Scenario**: A healthcare provider needs to mask patient names or diagnoses for most administrative staff but allow doctors and specific researchers to view the unmasked data.
3.  **Centralized Management of Access Rules**: When an organization has many data sets and users, and wants to manage access rules centrally in a single or a few security mapping tables rather than embedding them directly in numerous policies.
    *   **Scenario**: A large financial institution has hundreds of tables with sensitive data. Instead of creating and maintaining a distinct policy for every table and every role combination, they use mapping tables to control which roles have access to which data classifications.

## When Not to Use

1.  **Small, Static Datasets with Simple Access Needs**: For very small datasets where access rules are few, stable, and unlikely to change, the overhead of managing mapping tables and policy joins might be unnecessary.
    *   **Inefficiency**: Adds complexity and potentially minor performance overhead without significant benefit.
2.  **Performance-Critical Workloads with Extremely Large Mapping Tables**: While Snowflake's query optimizer is efficient, if the mapping tables become exceptionally large (billions of rows) and the policy is called for every row in a very large fact table during high-volume transactional queries, it could introduce latency.
    *   **Inefficiency**: The join operation within the policy evaluation could become a bottleneck for real-time applications if not designed carefully.
3.  **Access Control Based on External Identity Providers Only**: If access control is entirely managed and enforced by an external identity provider (IdP) and Snowflake only needs to honor those coarse-grained permissions without further internal logic.
    *   **Wrong Approach**: In such cases, policies within Snowflake might duplicate or conflict with the IdP's role mapping.

## Common Mistakes

1.  **Granting SELECT on Mapping Table to EVERYONE**: While the policies need to read the mapping table, granting `SELECT` on `user_security_map` to `PUBLIC` or `ALL ROLES` would expose the entire mapping table, potentially revealing security configurations or user assignments. Instead, grant `SELECT` only to the specific roles or `SECURITYADMIN` role, or ensure the policy is created by a role that has the necessary permissions. The policy's execution context usually handles this implicitly.
2.  **Performance Overheads from Inefficient Joins**: Using complex subqueries or joining large mapping tables without proper indexing (though Snowflake handles this well automatically in many cases) can lead to performance degradation, especially if the policy is applied to very large tables.
3.  **Lack of Auditability and Governance for Mapping Tables**: If the `user_security_map` table itself is not properly managed, version-controlled, and audited, changes to access control can be made without proper oversight, leading to potential security vulnerabilities or compliance issues.

## Expected Output

When a user with the `ANALYST_EAST` role queries `SELECT * FROM sales_data;`, the resulting dataset will only show rows where the `region` is 'EAST'. Additionally, the `customer_email` column will be masked.

| SALE_ID | PRODUCT_ID | REGION | AMOUNT | CUSTOMER_NAME | CUSTOMER_EMAIL |
| :------ | :--------- | :----- | :----- | :------------ | :------------- |
| 1       | 101        | EAST   | 150.00 | Alice         | ***MASKED***   |
| 3       | 103        | EAST   | 120.00 | Charlie       | ***MASKED***   |

-   **SALE_ID**: The unique identifier for the sale. (Unfiltered, Unmasked)
-   **PRODUCT_ID**: The identifier for the product sold. (Unfiltered, Unmasked)
-   **REGION**: The geographical region of the sale. (Filtered by row-access policy)
-   **AMOUNT**: The monetary amount of the sale. (Unfiltered, Unmasked)
-   **CUSTOMER_NAME**: The name of the customer. (Unfiltered, Unmasked)
-   **CUSTOMER_EMAIL**: The email address of the customer. (Masked by masking policy)
