# 07. Break Glass Access Process in Snowflake

The "Break Glass Access Process" in Snowflake refers to a pre-defined, highly restricted, and auditable procedure to gain emergency access to data or systems when standard access methods are unavailable or compromised. It exists to provide a last resort mechanism for critical situations, such as a security incident, an outage impacting normal operations, or a key administrator being unavailable.

**Short Explanation**

This process is a security and operational protocol designed for emergency scenarios where immediate access to sensitive data or systems is crucial to mitigate severe business impact. It addresses situations where regular access procedures might fail or be insufficient, ensuring business continuity while maintaining strict control and auditability. It's important in databases and analytics to prevent data loss, ensure system recovery, and respond to security threats, and is commonly used in incident response plans and disaster recovery strategies.

- What problem does this SQL feature solve? This concept solves the problem of needing immediate, emergency access to data or Snowflake accounts when standard access methods (e.g., VPNs, SSO, regular credentials) are inoperable or insufficient, ensuring that critical operations can continue or be restored.
- Why is it important in databases or analytics? It's crucial for maintaining business continuity, enabling rapid incident response (like containing a data breach), and facilitating disaster recovery by providing a failsafe for administrative access to the data warehouse.
- Where is it commonly used in real workflows? It is a standard component of enterprise security policies, incident response plans, and regulatory compliance frameworks (like SOX, HIPAA) where strict control over emergency access is mandated.

## Implementation Example

While "Break Glass Access Process" isn't a single SQL command, it often involves a pre-provisioned Snowflake user role and user, designed *only* for these emergencies. Here's how you might set up the user and role, and a hypothetical "break glass" action (though the actual process is manual and external to SQL):

```sql
-- Step 1: Create a dedicated 'break glass' role
CREATE ROLE BREAK_GLASS_ADMIN;

-- Step 2: Grant minimal but sufficient privileges for emergencies
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE BREAK_GLASS_ADMIN;
GRANT USAGE ON DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;
GRANT USAGE ON ALL SCHEMAS IN DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;
GRANT SELECT ON ALL TABLES IN DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;
GRANT OPERATE ON WAREHOUSE COMPUTE_WH TO ROLE BREAK_GLASS_ADMIN; -- For emergency warehouse control

-- Step 3: Create a dedicated 'break glass' user
CREATE USER BREAK_GLASS_USER
  PASSWORD = '<strong_complex_password>' -- Password stored securely OFFLINE, NOT in plain text here
  DEFAULT_ROLE = BREAK_GLASS_ADMIN
  DEFAULT_WAREHOUSE = COMPUTE_WH
  MUST_CHANGE_PASSWORD = TRUE
  COMMENT = 'Emergency access user - password stored securely offline and only used in critical situations.';

-- Step 4: Grant the user the break glass role
GRANT ROLE BREAK_GLASS_ADMIN TO USER BREAK_GLASS_USER;

-- Step 5: Simulate an emergency action (this part is conceptual for demonstration)
-- The actual "break glass" involves physical retrieval of credentials, manual login, and execution of necessary commands.
USE ROLE BREAK_GLASS_ADMIN;
SELECT CURRENT_USER(), CURRENT_ROLE(), CURRENT_WAREHOUSE();
SELECT COUNT(*) FROM PROD_DB.PUBLIC.SENSITIVE_DATA; -- Example emergency data access
ALTER WAREHOUSE COMPUTE_WH RESUME; -- Example emergency operation
```

## Explanation of the Code

This code sets up the necessary Snowflake objects for a "Break Glass" scenario.

- `CREATE ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Creates a new security role named `BREAK_GLASS_ADMIN`.
    - How it changes the result set: It defines a new access identity within Snowflake that can be granted specific privileges.
- `GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Grants the ability to use the `COMPUTE_WH` warehouse to the `BREAK_GLASS_ADMIN` role.
    - How it changes the result set: The `BREAK_GLASS_ADMIN` role now has permission to run queries using this warehouse.
- `GRANT USAGE ON DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Grants the ability to use the `PROD_DB` database to the `BREAK_GLASS_ADMIN` role.
    - How it changes the result set: The `BREAK_GLASS_ADMIN` role can now access objects within this database.
- `GRANT USAGE ON ALL SCHEMAS IN DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Grants the ability to use all existing schemas within `PROD_DB` to the `BREAK_GLASS_ADMIN` role.
    - How it changes the result set: The `BREAK_GLASS_ADMIN` role can now access objects within these schemas.
- `GRANT SELECT ON ALL TABLES IN DATABASE PROD_DB TO ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Grants read-only (SELECT) permission on all existing tables within `PROD_DB` to the `BREAK_GLASS_ADMIN` role.
    - How it changes the result set: The `BREAK_GLASS_ADMIN` role can now query data from all tables in `PROD_DB`.
- `GRANT OPERATE ON WAREHOUSE COMPUTE_WH TO ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Grants the ability to start, stop, and resize the `COMPUTE_WH` warehouse.
    - How it changes the result set: The `BREAK_GLASS_ADMIN` role can now manage the operational state of the warehouse.
- `CREATE USER BREAK_GLASS_USER ...;`:
    - What it does: Creates a new user named `BREAK_GLASS_USER` with a specific password (to be securely stored offline), default role, default warehouse, and forces a password change on first login.
    - How it changes the result set: A new login identity is established in Snowflake.
- `GRANT ROLE BREAK_GLASS_ADMIN TO USER BREAK_GLASS_USER;`:
    - What it does: Assigns the `BREAK_GLASS_ADMIN` role to the `BREAK_GLASS_USER`.
    - How it changes the result set: The `BREAK_GLASS_USER` now inherits all privileges granted to the `BREAK_GLASS_ADMIN` role.
- `USE ROLE BREAK_GLASS_ADMIN;`:
    - What it does: Switches the current session's active role to `BREAK_GLASS_ADMIN`.
    - How it changes the result set: Subsequent operations in the session will be executed with the privileges of `BREAK_GLASS_ADMIN`.
- `SELECT CURRENT_USER(), CURRENT_ROLE(), CURRENT_WAREHOUSE();`:
    - What it does: Retrieves the name of the current user, active role, and active warehouse for the session.
    - How it changes the result set: Returns a single row showing the current context of the Snowflake session.
- `SELECT COUNT(*) FROM PROD_DB.PUBLIC.SENSITIVE_DATA;`:
    - What it does: Counts all rows in the `SENSITIVE_DATA` table within the `PROD_DB.PUBLIC` schema.
    - How it changes the result set: Returns a single row with one column, representing the total number of records.
- `ALTER WAREHOUSE COMPUTE_WH RESUME;`:
    - What it does: Resumes a suspended virtual warehouse.
    - How it changes the result set: Changes the state of the `COMPUTE_WH` warehouse from suspended to running.

## When to Use

1.  **System-wide Outage/Loss of Admin Access:** When standard administrative accounts are inaccessible due to an identity provider (IdP) outage, compromised credentials, or configuration errors.
    *   *Scenario:* Your SSO provider is down, preventing all regular administrators from logging into Snowflake. The "break glass" user, with its direct Snowflake credentials, can be used to gain access and diagnose/resolve the issue.
2.  **Critical Data Incident Response:** In the event of a suspected data breach or unauthorized activity where immediate action is required to isolate data or analyze logs.
    *   *Scenario:* Security alerts indicate anomalous activity on a sensitive table, but the regular security team's access is somehow restricted. The "break glass" account can be used to immediately revoke permissions from suspicious accounts or query audit logs.
3.  **Disaster Recovery or Business Continuity:** During a major disaster where primary IT infrastructure is offline, necessitating a complete rebuild or recovery process within Snowflake.
    *   *Scenario:* A regional data center failure affects your regular management tools. The "break glass" account could be used from an unaffected location to verify the status of Snowflake objects or initiate data restoration processes.

## When Not to Use

1.  **Routine Administrative Tasks:** It should never be used for daily operations, regular user management, or standard data loading/querying.
    *   *Inefficiency/Wrong:* Using it for everyday tasks bypasses audit controls and increases the risk of accidental misuse or compromise of the emergency credentials.
2.  **Testing or Development:** The "break glass" account should not be used in non-production environments or for testing new features.
    *   *Inefficiency/Wrong:* This can blur the lines between emergency and regular access, making it harder to distinguish legitimate emergency use from unauthorized access attempts in real crises.
3.  **When Standard Access is Available:** If normal authentication and authorization mechanisms are functioning, the "break glass" process should not be initiated.
    *   *Inefficiency/Wrong:* Activating the "break glass" process has significant operational overhead (retrieving credentials, detailed logging, post-incident review) that is unnecessary when standard processes work.

## Common Mistakes

1.  **Weak or Compromised Credentials:** Using easy-to-guess passwords or storing the "break glass" credentials insecurely (e.g., in a shared, unencrypted document).
2.  **Lack of Auditability/Monitoring:** Not implementing robust logging and alerting for when the "break glass" account is accessed or used. This defeats its primary purpose of being a strictly auditable last resort.
3.  **Over-Privileging the Account:** Granting the "break glass" user excessive permissions beyond what is absolutely necessary for emergency situations. This increases the blast radius if the account is compromised.
4.  **Failure to Test:** Not periodically testing the "break glass" process to ensure the credentials are valid and the procedure works as expected.
5.  **Not Rotating Credentials:** Failing to rotate the "break glass" password regularly or immediately after every use.

## Expected Output

When the `SELECT CURRENT_USER(), CURRENT_ROLE(), CURRENT_WAREHOUSE();` command is executed by the `BREAK_GLASS_USER` using the `BREAK_GLASS_ADMIN` role, the resulting dataset will confirm the identity and context of the emergency session.

| CURRENT_USER()   | CURRENT_ROLE()    | CURRENT_WAREHOUSE() |
| :--------------- | :---------------- | :------------------ |
| BREAK_GLASS_USER | BREAK_GLASS_ADMIN | COMPUTE_WH          |

This table shows that the user logged in as `BREAK_GLASS_USER`, is operating with the `BREAK_GLASS_ADMIN` role, and is using the `COMPUTE_WH` virtual warehouse. This output is crucial for auditing and verifying that the correct emergency access account was used during an incident.
