# 10. Understanding the PUBLIC Role

The PUBLIC role is a default, automatically granted pseudo-role assigned to every user and role within the database environment.
It serves as the baseline level of access, meaning any privileges granted to PUBLIC are universally inherited by all users in the system.

# Key Points

* Every user and role in the system is automatically granted the PUBLIC role upon creation, and this implicit grant cannot be dropped or revoked.
* > [!WARNING]
  > Any object privileges granted to the PUBLIC role are immediately accessible to every user, making it highly risky to grant it access to sensitive data or powerful compute resources.
* By default, the PUBLIC role typically owns no objects, but users can create objects under it if the role is explicitly granted the necessary schema privileges.
* It is a strict security best practice to strip the PUBLIC role of all default privileges and only grant it access to universally safe, non-sensitive resources.
* When a user logs in without specifying a default role, the session often defaults to the PUBLIC role, providing only the most basic baseline permissions required to connect.

# Implementation

```sql
USE ROLE ACCOUNTADMIN;

REVOKE ALL PRIVILEGES ON WAREHOUSE compute_wh FROM ROLE PUBLIC;

CREATE DATABASE shared_company_data;
CREATE SCHEMA shared_company_data.public_reference;

GRANT USAGE ON DATABASE shared_company_data TO ROLE PUBLIC;
GRANT USAGE ON SCHEMA shared_company_data.public_reference TO ROLE PUBLIC;
GRANT SELECT ON ALL TABLES IN SCHEMA shared_company_data.public_reference TO ROLE PUBLIC;
```

# What does the Query do?

Step 1 assumes the highly privileged `ACCOUNTADMIN` role to ensure sufficient permissions to modify global role grants and create new top-level objects.

Step 2 revokes all default privileges on a specific compute warehouse from the `PUBLIC` role to prevent unauthorized compute consumption and unexpected costs from default users.

Step 3 creates a new database and schema specifically designed to house non-sensitive, universally accessible reference data.

Step 4 grants usage and read-only (`SELECT`) privileges on the new database, schema, and its tables to the `PUBLIC` role, ensuring all users can query this reference data without needing additional role grants.

# When Should We Use This? 

* When establishing baseline security postures by actively removing default compute or data access from the PUBLIC role to enforce the principle of least privilege.
* When providing organization-wide access to non-sensitive, universally required datasets (like date dimensions, zip code mappings, or company holiday calendars) without the administrative overhead of managing individual role grants.
