# Row Access Behavior in Tableau Queries in Snowflake

This concept describes how row access policies, defined in Snowflake, interact with queries generated by Tableau. It's crucial for ensuring data security and compliance by filtering data at the row level before it even reaches the Tableau dashboard.

**Short Explanation**

Row access policies in Snowflake allow you to define rules that restrict which rows a user can see in a table, based on their roles or other attributes. When Tableau queries Snowflake, these policies are automatically applied, meaning Tableau only receives the data rows that the querying user is authorized to see. This solves the problem of granular data security, is important for protecting sensitive data, and is commonly used in compliance-driven environments where different users require different views of the same dataset.

- **What problem does this SQL feature solve?** It solves the problem of implementing fine-grained, row-level security on data, ensuring users only see data they are authorized for, regardless of the application querying it.
- **Why is it important in databases or analytics?** It's critical for data governance, regulatory compliance (like GDPR, HIPAA), and internal security policies, preventing unauthorized access to sensitive information.
- **Where is it commonly used in real workflows?** It's often used in analytical platforms like Tableau where multiple users with varying access levels need to interact with the same underlying data source, but only see their permitted subset.

## Implementation Example

```sql
-- 1. Create a sample table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2),
    employee_id INT
);

-- 2. Insert some sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 'East', 1200.00, 101),
(2, 'Mouse', 'West', 25.00, 102),
(3, 'Keyboard', 'East', 75.00, 101),
(4, 'Monitor', 'North', 300.00, 103),
(5, 'Webcam', 'West', 50.00, 102),
(6, 'Headphones', 'East', 150.00, 101);

-- 3. Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy
AS (employee_id_col INT) RETURNS BOOLEAN ->
    CURRENT_ROLE() IN ('SALES_MANAGER') OR
    employee_id_col = CURRENT_USER();

-- 4. Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (employee_id);

-- 5. (Simulated) A Tableau query for sales data
-- If CURRENT_USER() is 'USER_101' and CURRENT_ROLE() is 'ANALYST'
SELECT
    region,
    SUM(sales_amount) AS total_sales
FROM
    sales_data
GROUP BY
    region;
```

## Explanation of the Code

This example demonstrates how to set up and apply a row access policy, and how it implicitly affects a query, such as one Tableau might generate.

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to hold sales transaction information, including a `region` and `employee_id`.
- **`INSERT INTO sales_data VALUES (...)`**: This populates the `sales_data` table with example rows, associating sales with different regions and employee IDs.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (employee_id_col INT) RETURNS BOOLEAN -> ...`**: This defines the `sales_policy`.
    - `AS (employee_id_col INT)`: Declares that the policy takes a column named `employee_id_col` (which will be mapped to `sales_data.employee_id`) as an input.
    - `RETURNS BOOLEAN ->`: Specifies that the policy returns `TRUE` if the row should be visible, `FALSE` otherwise.
    - `CURRENT_ROLE() IN ('SALES_MANAGER') OR employee_id_col = CURRENT_USER()`: This is the core logic. It allows a row to be visible if:
        - The current user has the `SALES_MANAGER` role (allowing them to see all rows), OR
        - The `employee_id_col` of the row matches the `CURRENT_USER()` (meaning individual employees only see their own sales).
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (employee_id);`**: This statement applies the `sales_policy` to the `sales_data` table, linking the policy's `employee_id_col` parameter to the table's `employee_id` column.
- **`SELECT region, SUM(sales_amount) AS total_sales FROM sales_data GROUP BY region;`**: This is a simplified example of a query Tableau might generate.
    - **`SELECT region, SUM(sales_amount) AS total_sales`**: Selects the `region` and calculates the sum of `sales_amount` for each region.
    - **`FROM sales_data`**: Specifies the table to retrieve data from.
    - **`GROUP BY region`**: Aggregates the results by `region`.
    - **How it changes the result set**: When this query is executed by a user, Snowflake's security framework implicitly adds a `WHERE` clause derived from the `sales_policy` to the query. For instance, if `CURRENT_USER()` is 'USER_101', the effective query would become something like:
        ```sql
        SELECT region, SUM(sales_amount) AS total_sales
        FROM sales_data
        WHERE (CURRENT_ROLE() IN ('SALES_MANAGER') OR employee_id = 'USER_101') -- This is implicitly added
        GROUP BY region;
        ```
        This ensures that only rows passing the policy's criteria are included in the aggregation.

## When to Use

1.  **Granular Data Security**: When different users or groups need to see only specific subsets of rows within the same table.
    *   **Example Scenario**: A sales team where each salesperson should only see their own sales records, but sales managers can see all sales records for their region.
2.  **Regulatory Compliance**: To enforce strict data access regulations (e.g., HIPAA for patient data, GDPR for personal data) that require masking or filtering sensitive information based on user roles or attributes.
    *   **Example Scenario**: In a healthcare database, doctors can see full patient records, but billing personnel can only see anonymized billing information, and researchers can only access aggregated, de-identified data.
3.  **Multi-Tenancy Architectures**: In applications where data from multiple tenants (customers) resides in a single table, but each tenant should only access their own data.
    *   **Example Scenario**: A SaaS application storing customer data from various companies in a single `customer_transactions` table. A row access policy ensures each company's users only see their company's transactions.

## When Not to Use

1.  **Performance Critical Queries on Extremely Large Tables**: While optimized, applying row access policies adds an overhead. For tables with billions of rows where every millisecond counts for broad analytical queries, this overhead might be noticeable.
    *   **Reason**: Each query needs to evaluate the policy logic for every row, which can increase processing time.
2.  **When Column-Level Security is Sufficient**: If the requirement is only to restrict access to certain columns, not rows, then column-level masking policies are a more direct and often more performant solution.
    *   **Reason**: Applying a row access policy adds complexity when a simpler masking policy would achieve the desired security for columns.
3.  **For Simple Static Filters**: If the filtering logic is always the same and doesn't depend on the user's role or attributes, a simple `WHERE` clause in the view or query itself is more straightforward and potentially better optimized.
    *   **Reason**: Row access policies are dynamic. If the filter is static, the dynamic nature is unnecessary overhead.

## Common Mistakes

1.  **Incorrect Policy Logic**: Defining a policy that either unintentionally grants too much access or restricts too much, leading to data exposure or data unavailability.
2.  **Not Testing All User Roles**: Policies must be thoroughly tested with all expected user roles and attributes to ensure they behave as intended for every scenario.
3.  **Performance Impact Overlooked**: Applying complex policies to very large tables without considering the potential performance degradation, especially with non-optimized policy conditions.
4.  **Forgetting to Apply the Policy**: Creating a policy but forgetting to link it to the target table using `ALTER TABLE ... ADD ROW ACCESS POLICY`.
5.  **Policy Dependent on External Functions**: While powerful, if a policy calls an external function, performance can suffer, and troubleshooting becomes more complex.

## Expected Output

The resulting dataset from a Tableau query (or any query) on `sales_data` will be a subset of the original `sales_data` table, filtered according to the active row access policy and the `CURRENT_USER()` and `CURRENT_ROLE()` at the time of query execution. The data returned to Tableau will only contain rows that the user is authorized to see.

**Example Table (if `CURRENT_USER()` is 'USER_101', role is 'ANALYST'):**

| REGION | TOTAL_SALES |
|:-------|:------------|
| East   | 1425.00     |

**Explanation of Columns:**
- **REGION**: The geographical region associated with the sales.
- **TOTAL_SALES**: The aggregated sum of `sales_amount` for the visible sales records in that region.

In this example, 'USER_101' can only see their own `employee_id = 101` sales. From the sample data, employee 101 has sales of 1200.00 (Laptop) and 75.00 (Keyboard), and 150.00 (Headphones) all in the 'East' region. The total for 'USER_101' in 'East' would be 1200 + 75 + 150 = 1425.00. Rows for `employee_id` 102 and 103 would be implicitly filtered out before aggregation.
