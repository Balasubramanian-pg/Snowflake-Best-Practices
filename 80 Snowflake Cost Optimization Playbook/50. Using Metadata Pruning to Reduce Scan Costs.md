# Using Metadata Pruning to Reduce Scan Costs in Snowflake

Metadata pruning is Snowflake’s automatic optimization that skips scanning micro-partitions that don’t contain relevant data. It exists to cut compute costs and speed up queries by using partition-level metadata (min/max values, null counts, etc.) instead of reading full tables.

**Short Explanation**

- Solves the problem of scanning massive tables when only a small subset of rows match filters.  
- Important because it reduces credit consumption and query latency, especially in analytics workloads with time-series or segmented data.  
- Commonly used in real workflows like dashboards, incremental loads, and selective filters on clustering columns.

## Implementation Example

```sql
SELECT order_id, order_date, customer_id, total_amount
FROM sales.orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-01-31'
  AND region = 'EMEA';
```

## Explanation of the Code

- **SELECT**: Chooses the columns to return (`order_id`, `order_date`, `customer_id`, `total_amount`). It defines the output schema but does not affect pruning.  
- **FROM**: Specifies the source table (`sales.orders`). Snowflake reads micro-partition metadata from this table’s catalog.  
- **WHERE**: Applies filters (`order_date` range and `region`). Snowflake uses min/max metadata for these columns to discard partitions that cannot contribute rows, shrinking the result set before scanning.

No GROUP BY, JOIN, HAVING, or ORDER BY are present; if added, they would operate *after* pruning has already reduced the scan.

## When to Use

- Filtering fact tables by date ranges when the table is clustered or naturally ordered by date.  
- Running selective lookups on a partition-friendly column like region, status, or customer segment.  
- Dashboard queries that repeatedly slice a large table by a known filter, where metadata is sufficient to prune aggressively.

## When Not to Use

- Queries with no filter predicates; pruning cannot help if all partitions must be read.  
- Filters on columns with poor clustering or no correlation to partition pruning metadata (random IDs).  
- Extremely selective point lookups better served by a key lookup or index-like access pattern instead of full scans.

## Common Mistakes

- Assuming pruning works on any column—if the table isn’t clustered by the filtered column, benefits diminish.  
- Using non-deterministic expressions in WHERE (e.g., wrapped in functions) that prevent metadata use.  
- Expecting pruning to reduce network output size; it only reduces scan work, not result cardinality after filtering.

## Expected Output

The resulting dataset includes only orders in January 2024 for the EMEA region. Columns: `order_id` (unique identifier), `order_date` (date of order), `customer_id` (buyer key), `total_amount` (numeric value). Example:

| order_id | order_date | customer_id | total_amount |
|----------|------------|-------------|--------------|
| 10234    | 2024-01-05 | C123        | 1250.00      |
| 10257    | 2024-01-19 | C789        | 980.50       |

The table shows filtered rows; unneeded partitions were never scanned thanks to metadata pruning.
