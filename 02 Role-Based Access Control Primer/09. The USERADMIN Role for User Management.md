# 09. The USERADMIN Role for User Management

The `USERADMIN` role is a built-in system role dedicated to the creation, modification, and management of user accounts and custom roles within the platform.
By isolating user and role management from broader security or system administration, it enforces the principle of least privilege and ensures clear separation of duties.

# Key Points

* The `USERADMIN` role holds the specific privileges required to execute `CREATE USER` and `CREATE ROLE` statements across the account.

* It automatically inherits the `OWNERSHIP` privilege on any user or role it creates, allowing it to alter, drop, or manage those specific objects.

* Unlike the `SECURITYADMIN` role, `USERADMIN` cannot manage global object grants or modify access control policies, limiting its scope strictly to identity provisioning.

* It operates within the standard role hierarchy, typically sitting below `SECURITYADMIN` (which inherits `USERADMIN` privileges) and `ACCOUNTADMIN`.

* Best practices dictate granting `USERADMIN` only to dedicated identity management personnel or automated provisioning systems to prevent unauthorized account creation.

# Implementation

> [!TIP]
> Always execute user and role creation scripts using the `USERADMIN` role rather than `ACCOUNTADMIN` to maintain a secure, least-privilege environment.

```sql
USE ROLE USERADMIN;

CREATE ROLE data_analyst_role
  COMMENT = 'Role for data analysis team members';

CREATE USER jsmith
  PASSWORD = 'StrongPassword123!'
  LOGIN_NAME = 'jsmith@example.com'
  DISPLAY_NAME = 'John Smith'
  DEFAULT_ROLE = data_analyst_role
  MUST_CHANGE_PASSWORD = TRUE;

GRANT ROLE data_analyst_role TO USER jsmith;
```

# What does the Query do?

Step 1: The `USE ROLE USERADMIN;` command switches the current session context to the `USERADMIN` role, ensuring all subsequent actions are executed with user and role management privileges.

Step 2: The `CREATE ROLE` statement provisions a new custom role named `data_analyst_role` and assigns a descriptive comment. 
> [!IMPORTANT]
> Because `USERADMIN` created the role, it inherently possesses the `OWNERSHIP` privilege over `data_analyst_role`, meaning only `USERADMIN` (or higher roles) can drop or alter it.

Step 3: The `CREATE USER` statement creates a new user account for John Smith, configuring essential authentication credentials, setting a default role, and forcing a password reset upon first login.

Step 4: The `GRANT ROLE` statement attaches the newly created `data_analyst_role` to the `jsmith` user, establishing the foundational access identity for that individual.

# When Should We Use This? 

* When onboarding new employees or integrating automated identity providers (like Okta or Azure AD) that require permissions to provision users and roles without needing full security administration rights.
* When implementing a strict separation of duties where the team responsible for creating user accounts is different from the team responsible for granting database access privileges.
