# Using CURRENT_ROLE in Row Access Policies in Snowflake

This concept leverages the `CURRENT_ROLE()` function within Snowflake's Row Access Policies to implement dynamic, role-based access control at the row level. It allows organizations to restrict data visibility based on the specific role a user has assumed, ensuring that different users see different subsets of the same table data, thereby enhancing data security and compliance.

**Short Explanation**

`CURRENT_ROLE()` in Row Access Policies dynamically filters rows a user can see based on their active Snowflake role. This feature solves the problem of fine-grained access control, ensuring users only view data relevant and permissible to their specific duties. It's crucial for security and compliance, especially in multi-tenant environments or when sensitive data needs strict segregation. It's commonly used in analytics and data warehousing to enforce least privilege access.

- **What problem does this SQL feature solve?** It solves the problem of granular, dynamic row-level security, allowing different users (based on their active roles) to see different data from the same table without creating multiple views or tables.
- **Why is it important in databases or analytics?** It's vital for data governance, compliance (e.g., GDPR, HIPAA), and ensuring that sensitive data is only exposed to authorized personnel. It simplifies security management and reduces the risk of data breaches.
- **Where is it commonly used in real workflows?** It's frequently used in financial institutions, healthcare organizations, and any enterprise dealing with confidential or regulated data where access must be strictly controlled based on user roles and responsibilities.

## Implementation Example

```sql
-- Create a sample table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_category VARCHAR,
    region VARCHAR,
    amount DECIMAL(10, 2),
    sensitive_flag BOOLEAN
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 'Electronics', 'North', 1500.00, FALSE),
(2, 'Clothing', 'South', 300.00, FALSE),
(3, 'Electronics', 'East', 2200.00, TRUE),
(4, 'Home Goods', 'West', 750.00, FALSE),
(5, 'Clothing', 'North', 500.00, TRUE),
(6, 'Electronics', 'South', 1800.00, FALSE);

-- Create roles
CREATE OR REPLACE ROLE analyst_north;
CREATE OR REPLACE ROLE analyst_south;
CREATE OR REPLACE ROLE compliance_officer;

-- Grant usage on database and schema, and select on the table to roles
GRANT USAGE ON DATABASE my_database TO ROLE analyst_north, analyst_south, compliance_officer;
GRANT USAGE ON SCHEMA my_database.my_schema TO ROLE analyst_north, analyst_south, compliance_officer;
GRANT SELECT ON TABLE sales_data TO ROLE analyst_north, analyst_south, compliance_officer;

-- Create the row access policy
CREATE OR REPLACE ROW ACCESS POLICY sales_rap AS (region VARCHAR, sensitive_flag BOOLEAN) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'ANALYST_NORTH' THEN region = 'North'
        WHEN CURRENT_ROLE() = 'ANALYST_SOUTH' THEN region = 'South'
        WHEN CURRENT_ROLE() = 'COMPLIANCE_OFFICER' THEN sensitive_flag = TRUE OR sensitive_flag = FALSE -- See all
        ELSE FALSE -- Default to no access
    END;

-- Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_rap ON (region, sensitive_flag);

-- Test as ANALYST_NORTH
USE ROLE analyst_north;
SELECT * FROM sales_data;

-- Test as ANALYST_SOUTH
USE ROLE analyst_south;
SELECT * FROM sales_data;

-- Test as COMPLIANCE_OFFICER
USE ROLE compliance_officer;
SELECT * FROM sales_data;
```

## Explanation of the Code

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` with columns for `sale_id`, `product_category`, `region`, `amount`, and a `sensitive_flag`.
- **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with sample records that include different regions and `sensitive_flag` values to demonstrate policy application.
- **`CREATE OR REPLACE ROLE ...`**: Defines three distinct roles: `analyst_north`, `analyst_south`, and `compliance_officer`. These roles will be used to test the row access policy.
- **`GRANT USAGE ON DATABASE ...`, `GRANT USAGE ON SCHEMA ...`, `GRANT SELECT ON TABLE ...`**: These `GRANT` statements provide necessary permissions for the created roles to access the database, schema, and `SELECT` from the `sales_data` table. This is important because the policy only filters what's visible; basic access must still be granted.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_rap AS (...) RETURNS BOOLEAN -> ...`**: This is the core of the example.
    - `CREATE ROW ACCESS POLICY sales_rap AS (...)`: Defines a row access policy named `sales_rap`.
    - `(region VARCHAR, sensitive_flag BOOLEAN)`: Specifies the columns from the target table that the policy will evaluate. These columns are passed as arguments to the policy.
    - `RETURNS BOOLEAN ->`: Indicates that the policy function will return a boolean value, where `TRUE` means the row is accessible, and `FALSE` means it's filtered out.
    - `CASE WHEN CURRENT_ROLE() = 'ANALYST_NORTH' THEN region = 'North' ... END`: This `CASE` statement uses `CURRENT_ROLE()` to determine the active role of the user executing the query.
        - If the `CURRENT_ROLE()` is `ANALYST_NORTH`, the policy returns `TRUE` only for rows where `region` is 'North'. This changes the result set by filtering out all rows not in the 'North' region for this role.
        - If the `CURRENT_ROLE()` is `ANALYST_SOUTH`, it returns `TRUE` only for rows where `region` is 'South'. This changes the result set by filtering out all rows not in the 'South' region for this role.
        - If the `CURRENT_ROLE()` is `COMPLIANCE_OFFICER`, it returns `TRUE` for all rows (`sensitive_flag = TRUE OR sensitive_flag = FALSE` effectively means all rows). This allows the compliance officer to see the full dataset.
        - `ELSE FALSE`: For any other role not explicitly handled, the policy defaults to `FALSE`, filtering out all rows.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_rap ON (region, sensitive_flag)`**: This statement applies the `sales_rap` policy to the `sales_data` table, specifying which columns from the table should be passed to the policy for evaluation. This statement fundamentally changes how `SELECT` queries on `sales_data` will behave, as the policy will now be enforced.
- **`USE ROLE ... SELECT * FROM sales_data;`**: These lines demonstrate how different roles would perceive the `sales_data` table. Each `SELECT` statement, when executed under a specific role, will return a different subset of data based on the policy.

## When to Use

1.  **Multi-Tenant Architectures:** When a single database serves multiple clients or departments, and each client/department should only see their own data.
    *   *Example:* An analytics platform where each client's data is tagged with a `client_id`, and `CURRENT_ROLE()` is used to ensure analysts for `Client A` only see data where `client_id = 'A'`.
2.  **Regulatory Compliance (GDPR, HIPAA):** To enforce strict data privacy rules where certain types of data (e.g., PII, health records) are restricted to specific, authorized roles.
    *   *Example:* A healthcare database where doctors can see patient medical records based on their department or assigned patients, while billing staff only see financial details. `CURRENT_ROLE()` would dynamically filter records based on the user's role and associated patient access.
3.  **Tiered Data Access:** When different levels of users (e.g., junior analyst, senior analyst, manager) require varying degrees of access to the same dataset, with managers typically having broader views.
    *   *Example:* A sales dashboard where junior analysts can only see sales data for their assigned region, while sales managers can view data for all regions under their purview.

## When Not to Use

1.  **Overly Complex Logic:** If the row access logic becomes extremely intricate with many `CASE` statements and `CURRENT_ROLE()` comparisons, it can impact performance and maintainability. Consider simplifying roles or data segregation.
    *   *Scenario:* A policy with dozens of `WHEN` clauses based on complex role hierarchies and combinations of attributes, making it difficult to debug and slow to execute.
2.  **Static Data Segregation:** If data access never changes or only needs to be filtered once for an entire group, traditional views might be simpler and more performant.
    *   *Scenario:* A dataset where all users of a specific team should always see the same subset of data. A simple `VIEW` with a `WHERE` clause could be defined and granted to that team's role, avoiding the overhead of a dynamic policy.
3.  **High-Performance Transactional Workloads:** While Snowflake policies are optimized, very high-volume, low-latency transactional systems might experience performance degradation due to the overhead of policy evaluation on every query.
    *   *Scenario:* An operational database for a trading system where sub-millisecond query responses are critical, and row-level security is enforced through application logic rather than database policies.

## Common Mistakes

1.  **Forgetting to grant `SELECT` permissions:** Users might think the policy itself grants access. The policy only *filters*; underlying `SELECT` permission on the table is still required.
2.  **Incorrectly assuming `CURRENT_ROLE()` behavior:** `CURRENT_ROLE()` returns the *active* role, not necessarily the *primary* role or all roles a user has. Users might switch roles, altering their perceived access.
3.  **Overlooking performance impacts:** While powerful, complex row access policies, especially those involving joins or subqueries within the policy logic, can add overhead. Testing performance under load is crucial.
4.  **Not testing with different roles:** Developers often test with their administrative role, which might have privileges that bypass or simplify the policy, leading to a false sense of security. Always test policies by assuming the specific roles they are intended for.
5.  **Policy argument type mismatch:** The column types specified in the `ROW ACCESS POLICY sales_rap AS (region VARCHAR, sensitive_flag BOOLEAN)` must exactly match the data types of the columns in the table (`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_rap ON (region, sensitive_flag)`). Mismatches will lead to errors.
