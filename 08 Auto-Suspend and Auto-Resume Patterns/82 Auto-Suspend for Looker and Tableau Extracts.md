# 82 Auto-Suspend for Looker and Tableau Extracts in Snowflake

Snowflake's `AUTO_SUSPEND` feature automatically pauses a virtual warehouse after a specified period of inactivity, stopping credit consumption. This is crucial for cost management, especially with BI tools like Looker and Tableau that often generate short, bursty queries or extract data periodically, leading to idle warehouse time.

**Short Explanation**

Auto-suspend in Snowflake is a setting for virtual warehouses that automatically shuts them down after a period of no activity. This prevents unnecessary credit usage when warehouses are idle, which is particularly useful for analytical workloads from tools like Looker and Tableau.

* **What problem does this SQL feature solve?** It solves the problem of incurring costs for idle compute resources. Snowflake bills per second of warehouse runtime, so an active but unused warehouse still consumes credits.
* **Why is it important in databases or analytics?** It's critical for cost optimization in cloud data warehousing. For BI tools, queries are often intermittent, and without auto-suspend, warehouses would remain active and costly between user interactions or scheduled extract refreshes.
* **Where is it commonly used in real workflows?** It's widely used across all types of Snowflake workloads, but particularly emphasized for development/QA environments, ad-hoc query warehouses, and dedicated warehouses serving BI tools like Looker and Tableau, where query patterns are often sporadic.

## Implementation Example

```sql
-- Create a new warehouse specifically for BI tools with auto-suspend set to 60 seconds
CREATE WAREHOUSE bi_queries_wh
  WAREHOUSE_SIZE = SMALL
  AUTO_SUSPEND = 60
  AUTO_RESUME = TRUE
  COMMENT = 'Warehouse for Looker and Tableau queries/extracts with aggressive auto-suspend';

-- Alternatively, alter an existing warehouse to enable auto-suspend
ALTER WAREHOUSE existing_analytics_wh
  SET AUTO_SUSPEND = 120,
      AUTO_RESUME = TRUE;
```

## Explanation of the Code

* **`CREATE WAREHOUSE bi_queries_wh...`**: This statement creates a new virtual warehouse named `bi_queries_wh`.
    * **`WAREHOUSE_SIZE = SMALL`**: Specifies the initial size of the compute cluster. Smaller warehouses are generally good for concurrent, short queries often generated by BI tools.
    * **`AUTO_SUSPEND = 60`**: This is the core setting. It tells Snowflake to suspend this warehouse after 60 seconds (1 minute) of continuous inactivity. This is often considered a sweet spot for short, bursty BI workloads.
    * **`AUTO_RESUME = TRUE`**: This ensures that the warehouse automatically starts up again when a new query is submitted to it, eliminating manual intervention.
    * **`COMMENT = '...'`**: A descriptive comment for the warehouse.
* **`ALTER WAREHOUSE existing_analytics_wh SET AUTO_SUSPEND = 120, AUTO_RESUME = TRUE;`**: This statement modifies an existing warehouse.
    * **`SET AUTO_SUSPEND = 120`**: Changes the auto-suspend time to 120 seconds (2 minutes).
    * **`AUTO_RESUME = TRUE`**: Ensures auto-resume is enabled.

In both cases, these clauses directly control the lifecycle of the warehouse, optimizing credit consumption by powering it down when not in use and automatically bringing it back online when needed.

## When to Use

1. **BI Tool Warehouses (Looker, Tableau):** When dedicated warehouses serve interactive dashboards or data extract jobs that run intermittently. Setting `AUTO_SUSPEND` to a short duration (e.g., 60-120 seconds) prevents costs from accumulating between user interactions or scheduled refresh cycles.
2. **Development/QA Environments:** For warehouses used by developers for ad-hoc querying and testing. These environments often have unpredictable usage patterns, making aggressive auto-suspend essential to prevent idle spend.
3. **ETL/Batch Workloads:** While some ETL jobs are long-running, if you have frequent, short batch processes, an auto-suspend of 1 minute or less, paired with `AUTO_RESUME`, can be highly effective. The warehouse suspends quickly after a job finishes and resumes for the next.

## When Not to Use

1. **Workloads Heavily Reliant on Warehouse Cache:** If queries frequently re-use cached data for performance (e.g., highly repetitive interactive dashboards where the cache provides significant speedup), an aggressive auto-suspend will clear the cache, potentially increasing query latency. In such cases, a slightly longer auto-suspend (e.g., 5-10 minutes) might be considered to keep the cache "warm".
2. **Very Frequent, Continuous Workloads:** For warehouses processing a constant stream of queries with almost no idle time (e.g., high-throughput APIs), suspending the warehouse would lead to frequent restarts, adding latency and potentially increasing credit consumption due to cold starts. In these rare cases, `AUTO_SUSPEND = 0` (effectively disabling it) might be considered, but with careful cost monitoring.
3. **When Auto-Resume is Disabled:** If `AUTO_RESUME` is `FALSE`, the warehouse will suspend and require manual intervention to restart, which is highly inefficient and disruptive for BI users or automated processes. `AUTO_SUSPEND` should almost always be paired with `AUTO_RESUME = TRUE`.

## Common Mistakes

1. **Setting `AUTO_SUSPEND` Too Long:** The most common mistake is leaving the default `AUTO_SUSPEND` of 5 minutes or setting it to a value that's too high for the workload. This directly leads to paying for significant idle time, especially for bursty BI queries.
2. **Disabling `AUTO_SUSPEND` Entirely:** While tempting for "convenience," disabling auto-suspend (`AUTO_SUSPEND = 0`) is a major cost driver, as the warehouse will run continuously even with no activity. This can lead to thousands of dollars in wasted spend annually.
3. **Not Understanding Cache Impact:** Developers might set `AUTO_SUSPEND` too aggressively (e.g., 30 seconds) for cache-sensitive workloads, leading to degraded performance due to frequent cache clearing and cold starts.

## Expected Output

The `AUTO_SUSPEND` setting itself doesn't produce a direct dataset output. Its effect is on the operational behavior and cost of the Snowflake warehouse. When correctly configured, the expected "output" is a more cost-efficient warehouse operation.

However, if we were to monitor the warehouse state and credit usage, we would see:

| TIMESTAMP | WAREHOUSE_NAME | STATE | CREDITS_USED_PER_MINUTE | DURATION_SECONDS |
| :-------------------------- | :------------- | :-------- | :---------------------- | :--------------- |
| 2026-03-14 10:00:00.000 +0000 | BI_QUERIES_WH | RESUMING | 0.00 | 5 |
| 2026-03-14 10:00:05.000 +0000 | BI_QUERIES_WH | ACTIVE | 0.03 | 10 |
| 2026-03-14 10:00:15.000 +0000 | BI_QUERIES_WH | ACTIVE | 0.03 | 45 |
| 2026-03-14 10:01:00.000 +0000 | BI_QUERIES_WH | SUSPENDING | 0.00 | 5 |
| 2026-03-14 10:01:05.000 +0000 | BI_QUERIES_WH | SUSPENDED | 0.00 | 0 |

* **TIMESTAMP**: The time of the state change or observation.
* **WAREHOUSE_NAME**: The name of the virtual warehouse.
* **STATE**: The current state of the warehouse (e.g., `RESUMING`, `ACTIVE`, `SUSPENDING`, `SUSPENDED`).
* **CREDITS_USED_PER_MINUTE**: The approximate credit consumption during that state. Note that credits are only consumed when the warehouse is `ACTIVE`.
* **DURATION_SECONDS**: How long the warehouse remained in that state.

This example shows the `bi_queries_wh` resuming, becoming active to process queries for about a minute (10 + 45 seconds), then suspending after 60 seconds of inactivity, thus preventing further credit consumption until the next query arrives.
