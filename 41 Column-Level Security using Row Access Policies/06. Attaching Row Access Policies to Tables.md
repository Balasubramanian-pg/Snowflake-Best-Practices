# Attaching Row Access Policies to Tables in Snowflake

Row Access Policies in Snowflake provide a robust, centralized, and fine-grained security model, allowing administrators to filter rows returned by a query based on the user's role or other attributes. This feature ensures data privacy and compliance by controlling which users can see which data rows in a table or view.

**Short Explanation**

Row Access Policies dynamically filter data rows returned from a table or view based on conditions defined in the policy. This means different users can query the same table but see different subsets of data based on their permissions or other context.

- What problem does this SQL feature solve? It solves the problem of fine-grained access control to data at the row level, preventing unauthorized users from viewing sensitive information.
- Why is it important in databases or analytics? It's crucial for compliance with data privacy regulations (like GDPR, HIPAA), internal security policies, and for multi-tenant environments where different customers or departments share a single database but should only see their own data.
- Where is it commonly used in real workflows? It's used in financial services to restrict access to customer accounts, in healthcare to limit access to patient records, and in multi-tenant SaaS applications to ensure data isolation.

## Implementation Example

```sql
-- Create a sample table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sale_amount NUMBER(10, 2),
    employee_id INT
);

-- Insert sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 'North', 1500.00, 101),
(2, 'Mouse', 'South', 25.00, 102),
(3, 'Keyboard', 'North', 75.00, 101),
(4, 'Monitor', 'East', 300.00, 103),
(5, 'Webcam', 'South', 50.00, 102);

-- Create a row access policy
CREATE OR REPLACE ROW ACCESS POLICY employee_sales_policy
AS (employee_id_col INT) RETURNS BOOLEAN ->
  CURRENT_ROLE() IN ('ADMIN_ROLE')
      OR CURRENT_ROLE() = 'SALES_MANAGER_NORTH' AND employee_id_col IN (101)
      OR CURRENT_ROLE() = 'SALES_MANAGER_SOUTH' AND employee_id_col IN (102);

-- Apply the row access policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY employee_sales_policy ON (employee_id);

-- Create roles for testing
CREATE OR REPLACE ROLE ADMIN_ROLE;
CREATE OR REPLACE ROLE SALES_MANAGER_NORTH;
CREATE OR REPLACE ROLE SALES_MANAGER_SOUTH;
CREATE OR REPLACE ROLE ANALYST_ROLE; -- A role with no access defined in policy

-- Grant USAGE on warehouse and database/schema
GRANT USAGE ON WAREHOUSE compute_wh TO ROLE ADMIN_ROLE, SALES_MANAGER_NORTH, SALES_MANAGER_SOUTH, ANALYST_ROLE;
GRANT USAGE ON DATABASE my_db TO ROLE ADMIN_ROLE, SALES_MANAGER_NORTH, SALES_MANAGER_SOUTH, ANALYST_ROLE;
GRANT USAGE ON SCHEMA my_db.public TO ROLE ADMIN_ROLE, SALES_MANAGER_NORTH, SALES_MANAGER_SOUTH, ANALYST_ROLE;

-- Grant SELECT on the table
GRANT SELECT ON TABLE sales_data TO ROLE ADMIN_ROLE, SALES_MANAGER_NORTH, SALES_MANAGER_SOUTH, ANALYST_ROLE;

-- Test as different roles (imagine switching roles and running SELECT * FROM sales_data;)
-- USE ROLE ADMIN_ROLE; -- Would see all rows
-- USE ROLE SALES_MANAGER_NORTH; -- Would see rows where employee_id = 101
-- USE ROLE SALES_MANAGER_SOUTH; -- Would see rows where employee_id = 102
-- USE ROLE ANALYST_ROLE; -- Would see no rows if no condition matches
```

## Explanation of the Code

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to store sales records, including an `employee_id` column which will be used by our policy.
- **`INSERT INTO sales_data VALUES (...)`**: This populates the `sales_data` table with some dummy data for demonstration.
- **`CREATE OR REPLACE ROW ACCESS POLICY employee_sales_policy AS (employee_id_col INT) RETURNS BOOLEAN -> ...`**: This defines the actual row access policy.
    - `AS (employee_id_col INT)`: Declares the input parameter for the policy, which is the column from the table that the policy will evaluate against.
    - `RETURNS BOOLEAN -> ...`: Specifies that the policy function returns a boolean value (true to show the row, false to hide it).
    - `CURRENT_ROLE() IN ('ADMIN_ROLE') OR ...`: This is the core logic. It checks the `CURRENT_ROLE()` of the user. If the user's role is `ADMIN_ROLE`, they see all rows. If it's `SALES_MANAGER_NORTH`, they see rows where `employee_id_col` is `101`. If `SALES_MANAGER_SOUTH`, they see rows where `employee_id_col` is `102`.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY employee_sales_policy ON (employee_id);`**: This is the crucial step where the defined policy (`employee_sales_policy`) is attached to the `sales_data` table, specifically associating the policy's input parameter (`employee_id_col`) with the `employee_id` column of the table. From this point on, any `SELECT` query on `sales_data` will implicitly apply this policy.
- **`CREATE OR REPLACE ROLE ...`**: These statements create various roles to demonstrate the different access levels.
- **`GRANT USAGE ON WAREHOUSE ...` and `GRANT USAGE ON DATABASE/SCHEMA ...`**: These are standard grants required for any role to interact with a warehouse, database, and schema in Snowflake.
- **`GRANT SELECT ON TABLE sales_data TO ROLE ...`**: This grants the `SELECT` privilege on the `sales_data` table to all the created roles. Even with this grant, the attached Row Access Policy will still filter the rows.
- **When a `SELECT` query is executed (e.g., `SELECT * FROM sales_data;`)**:
    - **What it does**: The Row Access Policy is automatically evaluated for each row in the `sales_data` table.
    - **How it changes the result set**: Only rows for which the policy's condition evaluates to `TRUE` (based on the `CURRENT_ROLE()` and the `employee_id` of that row) will be included in the final result set returned to the user. Rows where the condition is `FALSE` are effectively filtered out and not shown.

## When to Use

1.  **Regulatory Compliance**: When strict data privacy regulations (e.g., GDPR, HIPAA, CCPA) require that only authorized individuals can view certain sensitive rows of data.
    *   *Example Scenario*: A healthcare database needs to ensure that patient records are only accessible by medical staff assigned to that specific patient, or administrators for auditing purposes.
2.  **Multi-Tenant Architectures**: In applications where multiple customers or departments share the same underlying database tables but each entity should only see its own data.
    *   *Example Scenario*: A SaaS application stores data for multiple companies in a single `transactions` table. A Row Access Policy ensures that users from Company A can only see transactions belonging to Company A.
3.  **Role-Based Data Segmentation**: To implement complex business rules for data visibility based on user roles, regions, or hierarchical structures within an organization.
    *   *Example Scenario*: Sales managers should only see sales data for their specific region, while regional directors see data for all regions under them, and executives see all sales data.

## When Not to Use

1.  **Simple Column-Level Security**: If the requirement is only to restrict access to entire columns (e.g., hiding a `salary` column from non-HR users), Column-Level Security (Masking Policies) or simply not granting `SELECT` on specific columns is more appropriate and efficient.
    *   *Inefficient Scenario*: Applying a Row Access Policy that always returns `TRUE` if a user is in a certain role, just to simulate column visibility.
2.  **High-Performance OLTP Workloads with Frequent Small Queries**: The overhead of evaluating the policy for every row on every query can introduce latency, especially for frequently accessed tables in transactional systems where performance is critical.
    *   *Inefficient Scenario*: A highly concurrent system processing thousands of tiny customer lookup queries per second against a table with a complex Row Access Policy.
3.  **Static Data Filtering that Rarely Changes**: If the filtering logic is simple and constant (e.g., always hide deleted records, `WHERE is_active = TRUE`), it's often more straightforward and sometimes more performant to include the `WHERE` clause directly in views or application queries, rather than relying on a policy.
    *   *Inefficient Scenario*: Using a Row Access Policy that always filters `WHERE status = 'ACTIVE'` instead of creating a view `active_users AS SELECT * FROM users WHERE status = 'ACTIVE'`.

## Common Mistakes

1.  **Forgetting to Grant Policy Ownership/Usage**: Users or roles attempting to query a table protected by a policy must have `SELECT` privilege on the table AND `APPLY` privilege on the schema containing the policy, and `USAGE` on the policy itself. Forgetting these grants can lead to "Insufficient privileges" errors.
2.  **Overly Complex Policy Logic**: Designing policies with too many `OR` conditions, subqueries, or expensive functions can severely degrade query performance. Keep policy logic as simple and efficient as possible.
3.  **Not Testing with Different Roles**: Developers often test with an `ADMIN` role, which bypasses many policy restrictions. It's crucial to test queries with all relevant user roles to ensure the policy works as intended and doesn't inadvertently expose or hide data.
4.  **Mixing Policies Inadvertently**: Applying a Row Access Policy to a view that itself has a Row Access Policy on its underlying tables can lead to unexpected filtering or performance issues. Understand the order of operations and how policies cascade.
5.  **Lack of Visibility for Database Administrators**: Without proper logging or monitoring, it can be challenging for administrators to understand why certain users are seeing specific data subsets, especially when policies are complex and involve multiple roles and conditions.

## Expected Output

When a user queries `SELECT * FROM sales_data;`, the resulting dataset will only contain rows that satisfy the conditions defined in the `employee_sales_policy` based on the user's `CURRENT_ROLE()`.

**Example Table for `SALES_MANAGER_NORTH` (assuming `employee_id = 101`):**

| SALE_ID | PRODUCT_NAME | REGION | SALE_AMOUNT | EMPLOYEE_ID |
| :------ | :----------- | :----- | :---------- | :---------- |
| 1       | Laptop       | North  | 1500.00     | 101         |
| 3       | Keyboard     | North  | 75.00       | 101         |

**Explanation of columns:**

-   `SALE_ID`: Unique identifier for the sale.
-   `PRODUCT_NAME`: Name of the product sold.
-   `REGION`: Geographical region where the sale occurred.
-   `SALE_AMOUNT`: Monetary value of the sale.
-   `EMPLOYEE_ID`: Identifier of the employee who made the sale.

For `SALES_MANAGER_NORTH`, only sales made by `employee_id = 101` are returned because the Row Access Policy filters out all other rows based on the `CURRENT_ROLE()` check. If an `ADMIN_ROLE` executed the same query, they would see all five rows from the original `sales_data` table. If `ANALYST_ROLE` executed it, they would see an empty result set, as their role is not explicitly granted access to any `employee_id` in the policy.
