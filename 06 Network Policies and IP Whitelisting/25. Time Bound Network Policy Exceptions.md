# Time Bound Network Policy Exceptions in Snowflake

This concept refers to the ability to define exceptions to a Snowflake network policy that are only active for a specific duration. It allows administrators to temporarily grant access from IP addresses or ranges that would otherwise be blocked, without permanently altering the network policy. This feature exists to provide flexibility and temporary access control in secure Snowflake environments, crucial for scenarios like troubleshooting, external vendor access, or scheduled maintenance.

**Short Explanation**

Time-bound network policy exceptions allow temporary deviations from strict network access rules in Snowflake. They solve the problem of needing to grant short-term access without compromising overall security or requiring manual changes that might be forgotten. This is important for maintaining a secure data warehouse while enabling necessary operational flexibility. It's commonly used in situations requiring limited-time access for specific tasks or users, like an auditor needing to connect from a non-standard IP for a few days.

- **What problem does this SQL feature solve?** It solves the challenge of granting temporary, controlled access to Snowflake when the standard network policy would deny it, without having to permanently modify the policy.
- **Why is it important in databases or analytics?** It enhances security by allowing administrators to tightly control network access while still providing operational agility for specific, temporary needs, reducing the risk of leaving broad access rules in place indefinitely.
- **Where is it commonly used in real workflows?** It's used for scenarios such as granting a third-party vendor temporary access for data integration, allowing a remote developer to troubleshoot an issue from a new location for a limited time, or enabling scheduled system-to-system connections for a maintenance window.

## Implementation Example

```sql
ALTER ACCOUNT SET NETWORK_POLICY = 'my_strict_network_policy';

CREATE OR REPLACE NETWORK POLICY my_temporary_exception_policy
ALLOWED_IP_LIST = ('203.0.113.10/32')
COMMENT = 'Temporary access for external audit team';

ALTER ACCOUNT ADD NETWORK_POLICY my_temporary_exception_policy FOR 24 HOURS;

-- After 24 hours, the exception will automatically expire.
-- You can check current active network policies with:
SHOW NETWORK POLICIES;
DESCRIBE NETWORK POLICY my_temporary_exception_policy;
```

## Explanation of the Code

This SQL example demonstrates how to set up a time-bound network policy exception.

- `ALTER ACCOUNT SET NETWORK_POLICY = 'my_strict_network_policy';`
    - **What it does:** This statement applies an existing network policy named `my_strict_network_policy` to the entire Snowflake account. This policy is assumed to be restrictive, allowing access only from approved IP ranges.
    - **How it changes the result set:** It doesn't directly change a dataset but rather enforces the network access rules for all connections to the Snowflake account.

- `CREATE OR REPLACE NETWORK POLICY my_temporary_exception_policy ALLOWED_IP_LIST = ('203.0.113.10/32') COMMENT = 'Temporary access for external audit team';`
    - **What it does:** This creates or replaces a new, separate network policy named `my_temporary_exception_policy`. This policy is specifically designed to allow access from a single IP address (`203.0.113.10/32`). It includes a comment for clarity.
    - **How it changes the result set:** This command defines a new security object (network policy) within Snowflake, but does not yet activate it for the account.

- `ALTER ACCOUNT ADD NETWORK_POLICY my_temporary_exception_policy FOR 24 HOURS;`
    - **What it does:** This is the core command for creating the time-bound exception. It adds `my_temporary_exception_policy` to the account's active network policies, but critically, it specifies that this policy will only be active for 24 hours. During this period, connections from `203.0.113.10` will be allowed even if `my_strict_network_policy` would normally block them.
    - **How it changes the result set:** It modifies the account's effective network access rules, temporarily allowing connections from the specified IP while the time-bound policy is active.

- `SHOW NETWORK POLICIES;`
    - **What it does:** This command lists all network policies defined in the current account or session.
    - **How it changes the result set:** It returns metadata about existing network policies, including their names, descriptions, and whether they are currently applied.

- `DESCRIBE NETWORK POLICY my_temporary_exception_policy;`
    - **What it does:** This command provides detailed information about a specific network policy, such as its allowed and blocked IP lists.
    - **How it changes the result set:** It displays the configuration details of `my_temporary_exception_policy`.

## When to Use

1.  **Granting temporary vendor access:** An external consultant needs access to perform a data migration or integration for a limited time (e.g., 48 hours). You can set up an exception for their specific IP range that automatically expires.
2.  **Scheduled maintenance windows:** When a specific tool or service needs to connect from a new or non-standard IP address during a planned maintenance activity that will last a few hours.
3.  **Troubleshooting from an unusual location:** A developer or support team member needs to connect from a home network or a temporary remote location to diagnose an urgent issue, and their IP is not part of the standard allowed list. A time-bound exception provides quick, secure, temporary access.

## When Not to Use

1.  **Permanent IP whitelist additions:** If an IP address or range needs to be allowed indefinitely, it should be added directly to the main, permanent network policy instead of using a temporary exception.
2.  **Broad, unrestricted access:** This feature is for targeted, temporary exceptions. It should not be used to bypass strong network policies by allowing wide, non-specific IP ranges for long durations, as this undermines security.
3.  **Frequent, repetitive temporary access:** If the same "temporary" exception is constantly being re-applied, it indicates a flaw in the primary network policy design. The regularly needed IP should be integrated into the main policy.

## Common Mistakes

1.  **Forgetting to specify the duration:** Creating a new network policy but applying it to the account without the `FOR <duration>` clause makes it a permanent addition, not a temporary exception, potentially leaving security gaps.
2.  **Using overly broad IP ranges for exceptions:** Even for temporary access, specifying `0.0.0.0/0` or large CIDR blocks defeats the purpose of granular security. Always use the smallest necessary IP range.
3.  **Not verifying expiration:** Assuming the exception will expire without regularly reviewing `SHOW NETWORK POLICIES` or `DESCRIBE NETWORK POLICY` to confirm the expiration time can lead to unintended prolonged access.
4.  **Misunderstanding priority:** When multiple network policies are applied, it's important to understand how Snowflake resolves overlapping IP rules, though for `ADD NETWORK_POLICY FOR <duration>`, the temporary policy effectively acts as an override.

## Expected Output

The expected output after successfully applying a time-bound network policy exception is that connections originating from the specified temporary IP address(es) will be allowed to access Snowflake, whereas they would have been blocked by the standing network policy. After the specified duration, these connections will automatically revert to being blocked if they are not covered by other active policies.

When you run `SHOW NETWORK POLICIES;`, you would see something like this:

| name                          | owner     | comment                                | entries_in_allowed_ip_list | entries_in_blocked_ip_list | applied_on_account | applied_on_user | effective_on_account | effective_on_user | created_on             | expires_on             |
|:------------------------------|:----------|:---------------------------------------|:---------------------------|:---------------------------|:-------------------|:----------------|:---------------------|:------------------|:-----------------------|:-----------------------|
| MY_STRICT_NETWORK_POLICY      | ACCOUNTADMIN | Main policy for secure access          | 5                          | 0                          | TRUE               | FALSE           | TRUE                 | FALSE             | 2026-03-01 10:00:00.000 | NULL                   |
| MY_TEMPORARY_EXCEPTION_POLICY | ACCOUNTADMIN | Temporary access for external audit team | 1                          | 0                          | FALSE              | FALSE           | TRUE                 | FALSE             | 2026-03-14 10:00:00.000 | 2026-03-15 10:00:00.000 |

**Explanation of Columns:**

-   **`name`**: The name of the network policy.
-   **`comment`**: A descriptive comment for the policy.
-   **`effective_on_account`**: Indicates that `MY_TEMPORARY_EXCEPTION_POLICY` is currently active at the account level (due to the `ALTER ACCOUNT ADD` command).
-   **`expires_on`**: This crucial column shows the timestamp when the `MY_TEMPORARY_EXCEPTION_POLICY` will automatically cease to be effective, indicating its time-bound nature.
