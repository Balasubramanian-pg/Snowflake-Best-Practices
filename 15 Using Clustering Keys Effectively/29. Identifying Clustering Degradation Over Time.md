# Identifying Clustering Degradation Over Time

**Description:**
This topic explains how to monitor and detect clustering degradation over time in Snowflake, ensuring optimal performance and data organization. Clustering is crucial for efficient data retrieval, and degradation can impact query performance. By identifying degradation, users can take corrective actions to maintain optimal clustering. This is achieved through a combination of system functions and SQL queries.

## Short Explanation
**Overview:** Clustering is a crucial aspect of Snowflake's data organization, allowing for efficient data retrieval. Over time, clustering can degrade due to data changes, leading to suboptimal query performance.

**Problem Solved:** Identifying clustering degradation allows users to detect and address performance issues before they impact business operations.

**Importance:** Maintaining optimal clustering ensures efficient data retrieval, reducing query execution times and improving overall system performance.

**Use Cases:**
- Monitoring clustering degradation in a data warehouse
- Optimizing query performance in a Snowflake database

### Typical Scenario
A data warehouse administrator notices a gradual increase in query execution times over time. They suspect clustering degradation and want to identify the affected tables and clusters.

### Example Execution Logic Step-by-Step
The query retrieves information about clustered tables and their corresponding clustering columns. By analyzing this information, users can identify potential clustering degradation.

### Implementation Example
To monitor clustering degradation, create a view that retrieves clustering information:

CREATE VIEW clustering_info AS
SELECT
  TABLE_NAME,
  CLUSTERING_KEY,
  SYSTEM$CLUSTERING_DEPTH(OBJECT_CONSTRUCT('table_name', TABLE_NAME)) AS clustering_depth
FROM
  INFORMATION_SCHEMA.TABLES
WHERE
  TABLE_TYPE = 'BASE TABLE';

### Explanation of the Code
- The view, 'clustering_info', retrieves the table name, clustering key, and clustering depth for each table.
- The clustering depth is obtained using the SYSTEM$CLUSTERING_DEPTH function, which returns the average number of partitions that need to be scanned for a given table.

### When to Use
- When monitoring query performance and suspecting clustering degradation
- When optimizing data organization and clustering for better performance

### When Not to Use
- When clustering is not used or is disabled for a table
- When data is not frequently updated or changed

- Not updating clustering keys after significant data changes
