# Policy Inheritance Across Views in Snowflake

Policy inheritance across views in Snowflake refers to how security policies, specifically Row Access Policies and Masking Policies, are applied and propagated when data is accessed through views. When a policy is applied to a base table, that policy can implicitly affect views created on top of that table. Similarly, policies can be directly applied to views themselves.

**Short Explanation**

This concept ensures that data governance and security rules, such as who can see certain rows or what data is masked, remain enforced even when users interact with data through views rather than directly querying base tables. It solves the problem of maintaining consistent data security by preventing views from inadvertently bypassing established policies. This is important for regulatory compliance, data privacy, and maintaining control over sensitive information, commonly used in scenarios involving secure data sharing, role-based access control, and dynamic data masking in analytical and operational databases.

- **What problem does this SQL feature solve?** It ensures that data access policies (row-level security and data masking) are consistently applied, regardless of whether users query the base table or a view built on top of it, preventing security bypasses.
- **Why is it important in databases or analytics?** It's crucial for maintaining data privacy, meeting compliance requirements (like GDPR, HIPAA), and enforcing fine-grained access control, especially in environments where data is accessed by various user groups through different abstraction layers.
- **Where is it commonly used in real workflows?** In secure data sharing, multi-tenant environments, and any system where different roles need varying levels of access to the same underlying data, often involving sensitive personal identifiable information (PII) or financial data.

## Implementation Example

Here's an example demonstrating a Row Access Policy applied to a base table and how it impacts a view built on that table.

```sql
-- 1. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY employee_policy AS
(employee_region VARCHAR) RETURNS BOOLEAN ->
  CASE
    WHEN CURRENT_ROLE() IN ('ADMIN_ROLE', 'HR_ROLE') THEN TRUE
    WHEN CURRENT_ROLE() = 'WEST_SALES' AND employee_region = 'West' THEN TRUE
    WHEN CURRENT_ROLE() = 'EAST_SALES' AND employee_region = 'East' THEN TRUE
    ELSE FALSE
  END;

-- 2. Create a base table
CREATE OR REPLACE TABLE employees (
    employee_id INT,
    employee_name VARCHAR,
    salary DECIMAL(10, 2),
    employee_region VARCHAR
);

-- 3. Insert some data
INSERT INTO employees VALUES
(1, 'Alice', 60000, 'West'),
(2, 'Bob', 75000, 'East'),
(3, 'Charlie', 50000, 'North'),
(4, 'Diana', 80000, 'West');

-- 4. Apply the Row Access Policy to the base table
ALTER TABLE employees ADD ROW ACCESS POLICY employee_policy ON (employee_region);

-- 5. Create a view on top of the base table
CREATE OR REPLACE VIEW regional_employees_view AS
SELECT
    employee_id,
    employee_name,
    salary,
    employee_region
FROM employees;

-- 6. Test with different roles (simulate by using specific roles in your session)
-- User with role 'WEST_SALES' querying the view:
-- USE ROLE WEST_SALES;
-- SELECT * FROM regional_employees_view;

-- User with role 'ADMIN_ROLE' querying the view:
-- USE ROLE ADMIN_ROLE;
-- SELECT * FROM regional_employees_view;
```

## Explanation of the Code

- **`CREATE OR REPLACE ROW ACCESS POLICY employee_policy...`**:
  - **What it does**: Defines a Row Access Policy named `employee_policy`. This policy dictates which rows a user can see based on their current role and the `employee_region` column.
  - **How it changes the result set**: It creates the policy definition, but does not yet apply it to any data.

- **`CREATE OR REPLACE TABLE employees (...)`**:
  - **What it does**: Creates a base table `employees` with columns for employee information, including their region.
  - **How it changes the result set**: It defines the schema for the `employees` table.

- **`INSERT INTO employees VALUES (...)`**:
  - **What it does**: Populates the `employees` table with sample data.
  - **How it changes the result set**: It adds rows to the `employees` table.

- **`ALTER TABLE employees ADD ROW ACCESS POLICY employee_policy ON (employee_region)`**:
  - **What it does**: Attaches the `employee_policy` to the `employees` table, specifying that the `employee_region` column is used for policy evaluation.
  - **How it changes the result set**: At query runtime, this policy will filter rows from the `employees` table based on the `CURRENT_ROLE()` and the `employee_region` column.

- **`CREATE OR REPLACE VIEW regional_employees_view AS SELECT... FROM employees`**:
  - **What it does**: Creates a view named `regional_employees_view` that selects all columns from the `employees` base table.
  - **How it changes the result set**: This view itself does not define new data but provides an alternative access path to the `employees` table. Critically, the row access policy applied to the underlying `employees` table *is automatically inherited* by this view. This means when a user queries `regional_employees_view`, the `employee_policy` will still be evaluated against the `employees` table before the view's results are returned.

## When to Use

1. **Enforcing Row-Level Security on Shared Data**: When sharing data with external partners or different internal departments, you can create views that expose specific subsets of data (e.g., only data relevant to their region or project). By applying policies to the underlying tables, you ensure these views always respect access restrictions, even if the view itself doesn't explicitly define them.
    * **Scenario**: A company shares sales data with regional sales managers. A row access policy on the `SALES` table restricts each manager to only see sales from their assigned region. Views built on `SALES` (e.g., `REGIONAL_SALES_PERFORMANCE`) will automatically inherit this policy.

2. **Dynamic Data Masking for Sensitive Columns**: To protect PII or other sensitive information, masking policies can be applied to columns in base tables. Views selecting these columns will automatically inherit the masking behavior, ensuring that sensitive data is masked for unauthorized users, even through the view.
    * **Scenario**: An `EMPLOYEES` table contains sensitive `SOCIAL_SECURITY_NUMBER` and `SALARY` columns. A masking policy might show `*****` for SSN and a rounded value for `SALARY` for most users, but full values for HR. A view `EMPLOYEE_DETAILS` will present masked data as per the policy.

3. **Simplifying Compliance Audits**: By having policies defined at the table level and automatically inherited by views, auditing data access becomes simpler. You can verify the policies on base tables, knowing that views will correctly enforce them, reducing the surface area for compliance checks.
    * **Scenario**: A financial institution needs to prove that customer financial data is only accessible by authorized personnel. By applying policies to core `CUSTOMER_ACCOUNT` tables, any views created for reporting or analysis purposes will automatically comply, simplifying audit trails.

## When Not to Use

1. **When Performance is Critical for Unfiltered Queries**: If a view is primarily used by a small number of highly privileged users who need to perform complex analytics on the *entire* dataset without any filtering, applying policies that involve complex logic or mapping tables to the underlying table can introduce unnecessary overhead.
    * **Reason**: Policy evaluation happens at query runtime, and for very large tables, this can add latency if the filtering logic is computationally intensive.

2. **When Policies Need to be Explicitly Different for a View**: If a view needs a *different* or *more permissive* policy than the base table, relying solely on inheritance might not be suitable. While policies can be applied directly to views, this can lead to complex policy interactions if not managed carefully.
    * **Reason**: Snowflake evaluates the row access policy on the table first, then the policy on the view. This means the view's policy is applied *after* the table's policy has already filtered rows, and it cannot "undo" the table's filtering.

3. **For Simple Data Subsetting Without Security Concerns**: If the view's purpose is merely to simplify complex joins or provide a predefined subset of columns/rows without any security implications, adding policies to the underlying tables might be an over-engineering if not truly necessary for data governance.
    * **Reason**: Over-applying policies can complicate debugging and maintenance, adding unnecessary layers where simple `WHERE` clauses in view definitions might suffice for non-security related filtering.

## Common Mistakes

1. **Assuming Views Bypass Policies**: A common misconception is that creating a view on a table protected by a policy will somehow bypass that policy. Snowflake explicitly ensures that policies on base tables are applied *before* the view's results are returned.
2. **Not Understanding Policy Evaluation Order**: When both a base table and a view built on it have row access policies, the table's policy is *always* evaluated first, then the view's policy. This means the view's policy can further restrict but cannot expand the rows returned by the table's policy.
3. **Performance Impact of Complex Policies**: Overly complex policy logic, especially those involving joins to large mapping tables within the policy function, can significantly impact query performance, particularly on large datasets, without proper optimization.
4. **Inconsistent Column Names**: The columns specified in the policy signature must match in data type with the columns it is applied to, though the names can differ. A common mistake is to mismatch data types.

## Expected Output

When a user queries the `regional_employees_view`, the output will depend on the `CURRENT_ROLE()` of the user executing the query, due to the inherited `employee_policy`.

**Example Table (if `WEST_SALES` role queries `regional_employees_view`):**

| EMPLOYEE_ID | EMPLOYEE_NAME | SALARY | EMPLOYEE_REGION |
| :---------- | :------------ | :----- | :-------------- |
| 1 | Alice | 60000 | West |
| 4 | Diana | 80000 | West |

**Explanation of Columns:**

- **EMPLOYEE_ID**: Unique identifier for the employee.
- **EMPLOYEE_NAME**: The name of the employee.
- **SALARY**: The employee's salary.
- **EMPLOYEE_REGION**: The region where the employee works.

In this example, if the `WEST_SALES` role queries the view, only employees from the 'West' region are visible because the inherited `employee_policy` filters out rows from other regions. If an `ADMIN_ROLE` queries the view, all four rows would be visible.
