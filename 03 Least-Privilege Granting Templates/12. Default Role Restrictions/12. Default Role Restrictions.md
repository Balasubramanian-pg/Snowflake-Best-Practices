# Default Role Restrictions in Snowflake

Default role restrictions in Snowflake define which roles are activated by default when a user logs in. This feature exists to enhance security and streamline user experience by ensuring users start with a predefined set of privileges, preventing accidental over-privileging and making it easier to manage access.

**Short Explanation**

This Snowflake feature allows administrators to specify a limited set of roles that are automatically enabled for a user upon login, even if the user has many more roles granted. It solves the problem of users inadvertently operating with more permissions than necessary, which can lead to security vulnerabilities. It's important in databases for maintaining a strong security posture and simplifying access control, commonly used in environments with strict "least privilege" principles.

- What problem does this SQL feature solve? It prevents users from automatically activating all granted roles, enforcing a "least privilege" principle from the start.
- Why is it important in databases or analytics? It's crucial for security, ensuring users only have the necessary permissions active at any given time, reducing the risk of accidental data exposure or modification.
- Where is it commonly used in real workflows? In organizations with strict security policies, large user bases, or complex role hierarchies where fine-grained access control is paramount.

## Implementation Example

```sql
-- Granting roles to a user
GRANT ROLE ANALYST_ROLE TO USER JANE_DOE;
GRANT ROLE DEVELOPER_ROLE TO USER JANE_DOE;
GRANT ROLE DATA_ADMIN_ROLE TO USER JANE_DOE;

-- Setting default roles for the user, restricting them to specific roles
ALTER USER JANE_DOE SET DEFAULT_ROLES = (ANALYST_ROLE, DEVELOPER_ROLE);

-- Verify the default roles set for the user (optional)
SHOW USERS LIKE 'JANE_DOE';
```

## Explanation of the Code

- `GRANT ROLE ANALYST_ROLE TO USER JANE_DOE;`: This SQL statement grants the `ANALYST_ROLE` to the user `JANE_DOE`.
  - What it does: Assigns a specific role (and its associated privileges) to a user.
  - How it changes the result set: It doesn't directly change a dataset but updates the system's metadata regarding user-role assignments.

- `ALTER USER JANE_DOE SET DEFAULT_ROLES = (ANALYST_ROLE, DEVELOPER_ROLE);`: This statement is the core of setting default role restrictions.
  - What it does: Specifies a subset of roles that will be activated by default for `JANE_DOE` when they log in. Even if `DATA_ADMIN_ROLE` is granted, it won't be active initially.
  - How it changes the result set: Modifies the user's login behavior by controlling which roles are active by default.

- `SHOW USERS LIKE 'JANE_DOE';`: This statement is used for verification.
  - What it does: Retrieves information about the specified user, including their default roles.
  - How it changes the result set: Returns a system-generated result set displaying user properties.

## When to Use

1.  **Enforcing Least Privilege**: When you want to ensure users start their session with only the minimum necessary permissions, requiring them to explicitly activate higher-privileged roles.
    *   *Example Scenario*: A data analyst has roles for querying specific datasets (`ANALYST_ROLE`) and a higher-privileged role for managing data definitions (`DDL_MANAGER_ROLE`). You want them to always start with `ANALYST_ROLE` active.
2.  **Streamlining User Experience**: For users who have many roles but primarily work within a few specific ones. Setting defaults reduces clutter and confusion.
    *   *Example Scenario*: A user is part of 10 different project teams, each with its own role. However, their day-to-day work only involves 2-3 of these roles. Setting those 2-3 as default makes login simpler.
3.  **Security Audits and Compliance**: To meet regulatory requirements that demand strict control over access and active permissions.
    *   *Example Scenario*: An organization needs to demonstrate that users do not have elevated privileges active by default, complying with industry-specific security standards.

## When Not to Use

1.  **When a User Needs All Granted Privileges Instantly**: If a user genuinely needs all their granted roles active immediately upon login for their primary tasks.
    *   *Example Scenario*: A database administrator who consistently works across all aspects of the database and needs all their administrative roles active from the start of every session.
2.  **Overly Restrictive for Common Tasks**: If setting default roles makes everyday tasks cumbersome by forcing users to manually switch roles frequently for routine operations.
    *   *Example Scenario*: A user's workflow involves constantly switching between two roles that are not both set as default, leading to frustration and inefficiency.
3.  **When Roles are Granted Dynamically or Temporarily**: If roles are frequently granted and revoked for specific, short-term tasks, managing `DEFAULT_ROLES` constantly could become an administrative burden.
    *   *Example Scenario*: A project where users are temporarily granted a new role for a day, and setting it as a default role for that short period is unnecessary overhead.

## Common Mistakes

1.  **Not including essential roles in `DEFAULT_ROLES`**: Users might log in and find they can't perform even basic tasks because their primary working role wasn't set as default.
2.  **Assuming `DEFAULT_ROLES` also restricts *grantable* roles**: `DEFAULT_ROLES` only controls which roles are active on login, not which roles a user can eventually `USE` or `SET ROLE` to.
3.  **Forgetting to update `DEFAULT_ROLES` when user responsibilities change**: If a user's primary role changes, their `DEFAULT_ROLES` must be updated, otherwise they'll log in with outdated permissions.
4.  **Setting a role as default that the user is not actually granted**: This will result in an error or the role simply not being activated, leading to confusion.

## Expected Output

The `SHOW USERS LIKE 'JANE_DOE';` command will return a table with information about the user `JANE_DOE`. The `DEFAULT_ROLES` column in this table would show the roles that are automatically activated upon login for `JANE_DOE`.

| name | default_warehouse | default_namespace | default_role | default_secondary_roles | default_roles |
| :--- | :---------------- | :---------------- | :----------- | :---------------------- | :------------ |
| JANE\_DOE | MY\_WH | MY\_DB.MY\_SCHEMA | ANALYST\_ROLE | NONE | ANALYST\_ROLE, DEVELOPER\_ROLE |

- `default_roles`: This column will list the specific roles (`ANALYST_ROLE`, `DEVELOPER_ROLE` in this example) that are set to be active for `JANE_DOE` when they log in to Snowflake. Other granted roles will not be active unless explicitly switched by the user.
