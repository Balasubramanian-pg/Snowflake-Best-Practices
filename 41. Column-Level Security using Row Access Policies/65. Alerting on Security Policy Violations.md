# Alerting on Security Policy Violations in Snowflake

Alerting on Security Policy Violations in Snowflake refers to the process of setting up automated mechanisms to detect and notify administrators or security teams when activities within Snowflake breach predefined security rules, configurations, or best practices. This proactive approach helps in identifying and responding to potential threats, unauthorized access, or misconfigurations that could expose sensitive data or compromise the environment. It exists to ensure data governance, maintain compliance, and enhance the overall security posture of a Snowflake data platform.

**Short Explanation**

This SQL feature enables automated detection of security issues or policy breaches within your Snowflake environment. It helps you quickly spot things like unauthorized data access, unusual login patterns, or changes to critical security settings. It's important for maintaining a secure database, complying with regulations, and protecting sensitive information. Commonly, it's used in real-time monitoring dashboards, incident response workflows, and compliance auditing.

- **What problem does this SQL feature solve?** It solves the problem of manual, reactive security monitoring by automating the detection of policy violations, reducing the time to identify and respond to threats.
- **Why is it important in databases or analytics?** It's crucial for protecting sensitive data, ensuring regulatory compliance (like GDPR, HIPAA), and maintaining the integrity and availability of data.
- **Where is it commonly used in real workflows?** It's used by security operations teams for continuous threat detection, by data governance teams for compliance auditing, and by platform administrators for monitoring system health and configuration drift.

## Implementation Example

While Snowflake offers native `ALERT` objects for general data conditions, monitoring security policy violations often involves combining Snowflake's built-in auditing capabilities (like `ACCOUNT_USAGE` schema) with custom alerts or external security analytics tools like SnowAlert.[RgpB]

Here's an example using a native Snowflake `ALERT` to detect unusual login failures, which could indicate a brute-force attack or unauthorized access attempt. This alert leverages the `LOGIN_HISTORY` view in the `ACCOUNT_USAGE` schema.

```sql
-- Create an email notification integration (replace with your actual email details)
CREATE NOTIFICATION INTEGRATION my_email_integration
    TYPE = EMAIL
    ENABLED = TRUE
    ALLOWED_RECIPIENTS = ('security@example.com', 'admin@example.com');

-- Create a custom role with necessary privileges for the alert (best practice)
CREATE ROLE ALERT_MONITOR_ROLE;
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE ALERT_MONITOR_ROLE;
GRANT USAGE ON DATABASE SNOWFLAKE TO ROLE ALERT_MONITOR_ROLE;
GRANT USAGE ON SCHEMA SNOWFLAKE.ACCOUNT_USAGE TO ROLE ALERT_MONITOR_ROLE;
GRANT SELECT ON TABLE SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY TO ROLE ALERT_MONITOR_ROLE;
GRANT USAGE ON INTEGRATION MY_EMAIL_INTEGRATION TO ROLE ALERT_MONITOR_ROLE;
GRANT EXECUTE ALERT ON ACCOUNT TO ROLE ALERT_MONITOR_ROLE; -- For serverless alerts
-- If using a warehouse for alert execution, ensure the role has USAGE on it.

-- Create the alert
CREATE OR REPLACE ALERT FAILED_LOGIN_ATTEMPTS_ALERT
    WAREHOUSE = COMPUTE_WH -- Specify a warehouse for alert execution
    SCHEDULE = '15 MINUTE' -- Check every 15 minutes
    IF (EXISTS (
        SELECT
            USER_NAME,
            CLIENT_IP,
            COUNT(*) AS FAILED_ATTEMPTS
        FROM
            SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY
        WHERE
            IS_SUCCESS = FALSE
            AND EVENT_TIMESTAMP >= DATEADD(minute, -15, CURRENT_TIMESTAMP()) -- Look at the last 15 minutes
            AND REPORTED_CLIENT_TYPE = 'SNOWSQL' -- Filter for a specific client type, adjust as needed
        GROUP BY
            USER_NAME,
            CLIENT_IP
        HAVING
            COUNT(*) >= 5 -- Trigger if 5 or more failed attempts from the same user/IP
    ))
    THEN CALL SYSTEM$SEND_EMAIL(
        'MY_EMAIL_INTEGRATION',
        'security@example.com',
        'Snowflake Security Alert: Multiple Failed Login Attempts',
        'Multiple failed login attempts detected. Please investigate the user(s) and IP address(es) involved.'
    );

-- Resume the alert after creation
ALTER ALERT FAILED_LOGIN_ATTEMPTS_ALERT RESUME;
```

## Explanation of the Code

This example focuses on creating a native Snowflake `ALERT` object to monitor failed login attempts.

- **`CREATE NOTIFICATION INTEGRATION my_email_integration...`**:
    - **What it does**: Defines an integration object that allows Snowflake to send emails to specified recipients.
    - **How it changes the result set**: This statement creates a configuration for sending external notifications; it doesn't directly affect any dataset but enables the alert action.

- **`CREATE ROLE ALERT_MONITOR_ROLE; GRANT... TO ROLE ALERT_MONITOR_ROLE;`**:
    - **What it does**: Establishes a dedicated role (`ALERT_MONITOR_ROLE`) with the minimum necessary privileges to execute the alert and access the required `ACCOUNT_USAGE` views. This follows the principle of least privilege.
    - **How it changes the result set**: This creates a role and grants permissions. It doesn't modify data, but it defines who can execute the alert.

- **`CREATE OR REPLACE ALERT FAILED_LOGIN_ATTEMPTS_ALERT...`**:
    - **What it does**: This is the core statement that defines the alert.
    - **`WAREHOUSE = COMPUTE_WH`**: Specifies the virtual warehouse that will execute the SQL query defined in the alert.
    - **`SCHEDULE = '15 MINUTE'`**: Sets how often the alert condition (the `IF (EXISTS (...))` query) will be evaluated. In this case, every 15 minutes.
    - **`IF (EXISTS (...))`**:
        - **What it does**: This clause contains the condition that, if met, will trigger the alert's action. It checks if any rows are returned by the subquery.
        - **`SELECT USER_NAME, CLIENT_IP, COUNT(*) AS FAILED_ATTEMPTS FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY`**: Selects user name, client IP, and counts login attempts from the `LOGIN_HISTORY` view within Snowflake's `ACCOUNT_USAGE` schema. This view provides auditing information on login activities.
        - **`WHERE IS_SUCCESS = FALSE AND EVENT_TIMESTAMP >= DATEADD(minute, -15, CURRENT_TIMESTAMP())`**: Filters the login history to only include *unsuccessful* login attempts within the *last 15 minutes*.
        - **`GROUP BY USER_NAME, CLIENT_IP`**: Groups the failed attempts by the user and their originating IP address.
        - **`HAVING COUNT(*) >= 5`**: This is the critical condition. If any user from a specific IP address has 5 or more failed login attempts within the last 15 minutes, the `EXISTS` condition becomes true, and the alert fires.
        - **How it changes the result set**: The subquery inside `EXISTS` produces a temporary result set. The `EXISTS` clause then checks if this result set is non-empty.

- **`THEN CALL SYSTEM$SEND_EMAIL(...)`**:
    - **What it does**: Specifies the action to take when the `IF` condition is met. In this case, it calls the `SYSTEM$SEND_EMAIL` stored procedure to send an email notification using the previously defined `my_email_integration`.
    - **How it changes the result set**: This action sends an email; it does not produce a SQL result set.

- **`ALTER ALERT FAILED_LOGIN_ATTEMPTS_ALERT RESUME;`**:
    - **What it does**: Alerts are created in a suspended state and must be explicitly resumed to start evaluating their conditions on schedule.
    - **How it changes the result set**: This command activates the alert; it does not affect data.

## When to Use

1. **Detecting Brute-Force Attacks**:
    * **Scenario**: You want to know immediately if there are repeated failed login attempts from a single user or IP address, which could indicate a brute-force attack or credential stuffing.
    * **Example**: As implemented above, an alert triggers if 5 or more failed logins occur for the same user/IP within 15 minutes.

2. **Monitoring Privileged Access Changes**:
    * **Scenario**: You need to track and be notified of any grants or revocations of highly privileged roles (e.g., `ACCOUNTADMIN`, `SECURITYADMIN`) to users or other roles.
    * **Example**: An alert could query `SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_ROLES` or `GRANTS_TO_USERS` for recent changes involving these specific roles.

3. **Identifying Unauthorized Data Access Patterns**:
    * **Scenario**: A user or role might suddenly start querying sensitive tables they rarely accessed before, or querying an unusually high volume of data from sensitive tables.
    * **Example**: An alert could monitor `SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY` for queries against tables tagged as 'Sensitive_PII' by users outside a specific department or those with standard access privileges.

## When Not to Use

1. **For Simple Data Quality Checks without Security Implications**:
    * **Situation**: If you're merely checking for null values in a non-critical column or row counts in a table that isn't security-sensitive.
    * **Why**: While Snowflake alerts can be used for general data monitoring, using them for non-security-critical data quality issues might clutter security alert channels and dilute the focus on actual security violations. Separate data quality monitoring tools or less disruptive notifications might be more appropriate.

2. **When an External SIEM or SOAR Platform is Already Integrated**:
    * **Situation**: Your organization already has a robust Security Information and Event Management (SIEM) or Security Orchestration, Automation, and Response (SOAR) platform that ingests Snowflake audit logs (e.g., `LOGIN_HISTORY`, `QUERY_HISTORY`) and has its own rule engine for security incident detection.
    * **Why**: Duplicating alerting logic can lead to alert fatigue, inconsistent incident handling, and increased management overhead. It's often better to centralize security monitoring and alerting within the dedicated security platform, letting Snowflake primarily feed its audit logs.

3. **For Real-Time Blockage or Remediation**:
    * **Situation**: You need to instantly block a user or roll back a malicious action as soon as it's detected.
    * **Why**: Snowflake `ALERT` objects are primarily for *detection and notification*. While they can call stored procedures, there's an inherent latency in their scheduled execution and the time it takes for an action to complete. For immediate, preventative measures, other Snowflake features like network policies, session policies, or external security tools with real-time enforcement capabilities might be more suitable.

## Common Mistakes

1. **Over-alerting or Under-alerting**:
    * **Mistake**: Setting alert thresholds too low, leading to excessive notifications ("alert fatigue"), or too high, missing actual violations.
    * **Remedy**: Start with reasonable thresholds, monitor the alert frequency, and fine-tune them based on actual operational patterns and observed false positives/negatives.

2. **Lack of Role-Based Access Control (RBAC) for Alerts**:
    * **Mistake**: Creating alerts using the `ACCOUNTADMIN` role or other highly privileged roles, violating the principle of least privilege.
    * **Remedy**: Always create a dedicated, custom role with only the necessary privileges (e.g., `USAGE` on warehouse, `SELECT` on `ACCOUNT_USAGE` views, `EXECUTE ALERT` on account) for the alert to run under.

3. **Not Testing Alert Conditions Thoroughly**:
    * **Mistake**: Deploying alerts without validating if their `IF (EXISTS (...))` conditions correctly identify the desired security events and don't produce unexpected results.
    * **Remedy**: Test the SQL query within the `IF` condition against historical data in `ACCOUNT_USAGE` or by simulating conditions to ensure it behaves as expected before scheduling the alert.

4. **Ignoring Alert Management and Lifecycle**:
    * **Mistake**: Creating alerts and then forgetting to monitor their execution, update their logic, or disable them when no longer needed.
    * **Remedy**: Regularly review alert history (`INFORMATION_SCHEMA.ALERT_HISTORY`), update alert definitions as security requirements evolve, and establish a process for suspending or dropping alerts that are obsolete or misconfigured.

## Expected Output

When the `FAILED_LOGIN_ATTEMPTS_ALERT` condition is met (e.g., 5 or more failed logins from the same user/IP within 15 minutes), the `SYSTEM$SEND_EMAIL` procedure is called. The "output" is an email sent to the specified recipients, rather than a SQL result set returned to the user.

An example of the email content that would be sent:

**Subject**: Snowflake Security Alert: Multiple Failed Login Attempts

**Body**: Multiple failed login attempts detected. Please investigate the user(s) and IP address(es) involved.

If you were to query the `INFORMATION_SCHEMA.ALERT_HISTORY` view to see a record of the alert firing, the resulting dataset would look something like this (actual columns may vary but these are key):

| ALERT_NAME | STATE | SCHEDULED_TIME | COMPLETED_TIME | RETURN_VALUE | ERROR_MESSAGE |
| :--------------------------- | :-------- | :-------------------------- | :-------------------------- | :--------------------------------------------- | :------------ |
| FAILED_LOGIN_ATTEMPTS_ALERT | SUCCEEDED | 2026-03-14 14:30:00.000 +0000 | 2026-03-14 14:30:05.123 +0000 | Email sent to security@example.com, admin@example.com | NULL |
| FAILED_LOGIN_ATTEMPTS_ALERT | FAILED | 2026-03-14 14:15:00.000 +0000 | 2026-03-14 14:15:02.456 +0000 | NULL | Email integration failed |

- **`ALERT_NAME`**: The name of the alert that was executed.
- **`STATE`**: Indicates if the alert execution `SUCCEEDED` or `FAILED`.
- **`SCHEDULED_TIME`**: The timestamp when the alert was scheduled to run.
- **`COMPLETED_TIME`**: The timestamp when the alert execution finished.
- **`RETURN_VALUE`**: The result of the action taken (e.g., confirmation of email sent).
-   **`ERROR_MESSAGE`**: If the alert execution failed, this column would contain details about the error.
