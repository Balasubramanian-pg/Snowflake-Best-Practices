# Network Policies Overview in Snowflake

Network policies in Snowflake are security features that restrict network access to your Snowflake account based on IP addresses. They act as firewalls, ensuring that only approved network locations can connect to your Snowflake instance, enhancing data security by limiting potential attack vectors.

**Short Explanation**

Network policies define allowed (and optionally blocked) IP addresses for connecting to Snowflake, acting like a security gate. They solve the problem of unauthorized network access, making your data more secure by ensuring only trusted sources can connect. This is crucial for compliance and protecting sensitive data in analytics environments. They are commonly used by security teams to lock down access to corporate networks or specific VPNs.

- What problem does this SQL feature solve? It solves the problem of unauthorized network access to your Snowflake account.
- Why is it important in databases or analytics? It's important for security, compliance, and protecting sensitive data from being accessed outside of approved network environments.
- Where is it commonly used in real workflows? Security and operations teams use it to enforce strict access controls, often allowing only corporate VPNs or specific office IP ranges.

## Implementation Example

```sql
-- Create a network policy named 'my_network_policy'
-- allowing access from IP ranges 192.168.1.0/24 and 10.0.0.10
CREATE NETWORK POLICY my_network_policy
    ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.10');

-- Optionally, add a blocked IP list (e.g., a specific public IP that should never connect)
-- This is optional and often omitted if ALLOWED_IP_LIST is comprehensive.
ALTER NETWORK POLICY my_network_policy
    BLOCKED_IP_LIST = ('203.0.113.5');

-- Apply the network policy to your account
ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy;

-- Or apply the network policy to a specific user
ALTER USER jdoe SET NETWORK_POLICY = my_network_policy;

-- Unset the network policy from the account (to disable it)
-- ALTER ACCOUNT UNSET NETWORK_POLICY;
```

## Explanation of the Code

-   `CREATE NETWORK POLICY my_network_policy`: This statement initiates the creation of a new network policy and assigns it the name `my_network_policy`.
-   `ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.10')`: This clause defines the list of IP addresses or CIDR blocks from which connections to Snowflake are permitted. Any connection attempt originating from an IP address not in this list will be rejected if the policy is active.
-   `ALTER NETWORK POLICY my_network_policy BLOCKED_IP_LIST = ('203.0.113.5')`: This clause is optional. It defines a list of IP addresses or CIDR blocks from which connections to Snowflake are explicitly denied, even if they might otherwise be covered by an `ALLOWED_IP_LIST`. This acts as an explicit deny rule.
-   `ALTER ACCOUNT SET NETWORK_POLICY = my_network_policy`: This statement applies the `my_network_policy` policy to the entire Snowflake account. Once set at the account level, all users attempting to connect to this account will be subject to this policy unless overridden by a user-specific policy.
    -   What it does: Assigns the specified network policy to the Snowflake account.
    -   How it changes the result set: It doesn't change a result set in the traditional sense, but rather alters the security configuration of the account, affecting which network connections are allowed or denied.
-   `ALTER USER jdoe SET NETWORK_POLICY = my_network_policy`: This statement applies the `my_network_policy` policy specifically to the user `jdoe`. A user-level network policy overrides any account-level network policy for that particular user.
    -   What it does: Assigns the specified network policy to a specific user.
    -   How it changes the result set: Similar to the account-level setting, it modifies the security posture for the given user, controlling their network access to Snowflake.

## When to Use

1.  **Restricting access to corporate networks**: When you want to ensure that users can only access Snowflake from approved internal corporate networks or VPNs.
    *   *Example scenario*: Your company has sensitive financial data in Snowflake, and you need to prevent access from personal home networks or public Wi-Fi.
2.  **Compliance requirements**: For regulatory compliance (e.g., HIPAA, PCI DSS) that mandates strict control over data access points.
    *   *Example scenario*: An organization needs to demonstrate that access to customer health records (PHI) in Snowflake is strictly limited to authorized locations.
3.  **Enhancing security posture**: As a layer of defense against unauthorized access attempts, even if credentials are compromised.
    *   *Example scenario*: If a user's login credentials are stolen, a network policy can still block access if the attacker is not connecting from an allowed IP address.

## When Not to Use

1.  **Highly dynamic IP environments**: For users who frequently connect from changing public IP addresses (e.g., remote workers without a VPN, mobile users, or cloud functions with dynamic IPs).
    *   *Inefficient scenario*: Forcing all remote employees to use a VPN adds overhead if their work doesn't strictly require such a restriction, or if they need to connect from various locations.
2.  **Testing and development accounts with flexible access needs**: In non-production environments where the primary goal is quick and easy access for development and testing, and security restrictions can hinder productivity.
    *   *Inefficient scenario*: A development team needs to quickly test integrations from various cloud-hosted services, and constantly updating IP whitelists becomes a bottleneck.
3.  **When IP addresses are not the primary security concern**: If your main security concern is about unauthorized users rather than unauthorized network locations (e.g., strong authentication, role-based access control might be more effective).
    *   *Inefficient scenario*: Relying solely on network policies for an application that uses a fixed IP but has many users with varying permissions; granular RBAC is more appropriate here.

## Common Mistakes

1.  **Forgetting to apply the policy**: Creating a network policy but failing to activate it at either the account or user level, leaving the account unprotected.
2.  **Incorrect CIDR notation**: Specifying incorrect CIDR blocks (e.g., `/32` for a range, or a typo) which can inadvertently block legitimate users or leave gaps in protection.
3.  **Blocking yourself**: Accidentally creating a policy that excludes your current IP address, leading to a lockout from your own Snowflake account. This often requires contacting Snowflake Support to revert.
4.  **Overlapping `ALLOWED_IP_LIST` and `BLOCKED_IP_LIST`**: While `BLOCKED_IP_LIST` takes precedence, misunderstanding this hierarchy can lead to confusion about who has access.

## Expected Output

When a network policy is active, it affects the connection process. There is no direct SQL query output that shows the "result set" of an active network policy. Instead, its effect is observed during connection attempts.

*   **Successful Connection:** If an allowed IP attempts to connect, the connection proceeds normally.
*   **Blocked Connection:** If a disallowed IP attempts to connect, the connection will be rejected, typically with an error message indicating that the client IP address is not allowed by the network policy.

Example of expected behavior (not a SQL query output):

| Connection Attempt Source IP | Network Policy Status (my_network_policy) | Outcome         | Explanation                                                                         |
| :-------------------------- | :---------------------------------------- | :-------------- | :---------------------------------------------------------------------------------- |
| `192.168.1.50`              | Active                                    | Connection OK   | IP is within `192.168.1.0/24` (ALLOWED_IP_LIST).                                    |
| `10.0.0.10`                 | Active                                    | Connection OK   | IP is explicitly listed in `ALLOWED_IP_LIST`.                                       |
| `203.0.113.5`               | Active                                    | Connection Denied | IP is explicitly listed in `BLOCKED_IP_LIST` (takes precedence).                    |
| `172.16.0.1`                | Active                                    | Connection Denied | IP is not in `ALLOWED_IP_LIST` and not in `BLOCKED_IP_LIST`, so it's implicitly denied. |

**Explanation of Columns:**

-   **Connection Attempt Source IP**: The IP address from which a user or application is trying to connect to Snowflake.
-   **Network Policy Status (my_network_policy)**: Indicates whether the `my_network_policy` is currently applied and active at the account or user level.
-   **Outcome**: Whether the connection attempt was successful ("Connection OK") or rejected ("Connection Denied").
-   **Explanation**: A brief reason for the outcome based on the network policy rules.
