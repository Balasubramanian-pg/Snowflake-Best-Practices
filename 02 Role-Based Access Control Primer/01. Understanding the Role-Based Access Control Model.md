# 01. Understanding the Role Based Access Control Model

Role-Based Access Control (RBAC) is a security paradigm that restricts system access by assigning permissions to roles rather than directly to individual users. 
Users are then granted these roles, simplifying the management of privileges across an organization as personnel change and scale.

> [!IMPORTANT]  
> In a strict RBAC model, you should never grant object-level privileges (like `SELECT` or `INSERT`) directly to a user. Always grant privileges to a role, and then grant that role to the user.

# Key Points

*   **Separation of Duties:** RBAC ensures that users only have the permissions necessary to perform their specific job functions, strictly adhering to the principle of least privilege.
*   **Role Hierarchy:** Roles can inherit permissions from other roles, allowing for a structured and scalable access model (e.g., a senior analyst role inherits all privileges from a junior analyst role).
*   **Simplified Administration:** When a user changes departments or leaves the company, administrators only need to update their role assignments rather than auditing and revoking individual object-level permissions.
*   **Object-Level Security:** Privileges (such as `USAGE`, `READ`, or `WRITE`) are granted on securable objects (like databases, schemas, or tables) exclusively to roles, creating a clean security matrix.
*   **Auditing and Compliance:** RBAC provides a clear, auditable trail of who has access to what, making it significantly easier to meet regulatory compliance and enforce internal security policies.

# Implementation

The following Snowflake SQL example demonstrates how to create a role, assign it specific privileges, grant it to a user, and establish a role hierarchy.

```sql
-- Create a custom role for data analysts
CREATE ROLE data_analyst;

-- Grant foundational usage on the database and schema to the role
GRANT USAGE ON DATABASE sales_db TO ROLE data_analyst;
GRANT USAGE ON SCHEMA sales_db.public TO ROLE data_analyst;

-- Grant read-only access on all current tables to the role
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE data_analyst;

-- Assign the role to a specific user and build the hierarchy
GRANT ROLE data_analyst TO USER jsmith;
GRANT ROLE data_analyst TO ROLE sysadmin;
```

# What does the Query do?

Step 1 creates a new, dedicated role named `data_analyst` to encapsulate the specific permissions required for analytical tasks without tying them to a specific person.

Step 2 grants foundational `USAGE` privileges on the `sales_db` database and its `public` schema, which is a prerequisite before any table-level access can be utilized by the role.

Step 3 assigns the `SELECT` privilege on all existing tables within that schema to the `data_analyst` role, enabling read-only data access for anyone who assumes the role.

Step 4 assigns the `data_analyst` role to the user `jsmith` so they can query the data, and then grants it to the `sysadmin` role to establish a hierarchy where system administrators automatically inherit analyst privileges.

# When Should We Use This? 

*   When onboarding new employees or managing departmental transfers, allowing administrators to provision or revoke access instantly by modifying role assignments instead of tracking down individual database privileges.
*   When designing a scalable data warehouse architecture that requires strict adherence to compliance standards (like SOC2 or GDPR) by enforcing the principle of least privilege across all environments.
