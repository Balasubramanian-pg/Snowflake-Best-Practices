# Using CURRENT_USER in Masking Policies in Snowflake

This concept explains how to dynamically control data visibility in Snowflake using masking policies that leverage the `CURRENT_USER` function. Masking policies in Snowflake allow you to apply transformation rules to data in a table column to obscure it based on certain conditions, while `CURRENT_USER` helps make these conditions user-specific. This combination ensures that sensitive data is protected and only accessible to authorized users.

**Short Explanation**

Using `CURRENT_USER` within a Snowflake masking policy allows you to create dynamic rules that check the identity of the user querying the data. This means different users can see different versions of the same data, like a full value or a masked value, based on who they are. It solves the problem of fine-grained access control on a column level, ensuring data privacy and compliance without creating multiple views or tables. It's crucial for data governance and is commonly used in scenarios requiring role-based or user-based data redaction.

- **What problem does this SQL feature solve?** It solves the problem of dynamic, fine-grained, column-level security, enabling different users or roles to see different representations of sensitive data within the same table.
- **Why is it important in databases or analytics?** It's vital for data privacy, regulatory compliance (like GDPR, HIPAA), and internal data governance, ensuring sensitive information is protected while still allowing necessary access for authorized individuals.
- **Where is it commonly used in real workflows?** It's used in financial services (masking account numbers for non-finance roles), healthcare (masking patient IDs for non-medical staff), HR systems (masking salaries for non-HR users), and any scenario where sensitive PII needs conditional access.

## Implementation Example

```sql
-- Create a sample table with sensitive data
CREATE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    customer_email VARCHAR,
    sale_amount DECIMAL(10, 2)
);

-- Insert some sample data
INSERT INTO sales_data (order_id, customer_name, customer_email, sale_amount) VALUES
(1, 'Alice Smith', 'alice.s@example.com', 100.50),
(2, 'Bob Johnson', 'bob.j@example.com', 250.75),
(3, 'Charlie Brown', 'charlie.b@example.com', 50.00);

-- Create a masking policy that hides the full email unless the user is 'ANALYST_USER'
CREATE MASKING POLICY email_mask_policy AS (val VARCHAR) RETURNS VARCHAR ->
  CASE
    WHEN CURRENT_USER() = 'ANALYST_USER' THEN val
    ELSE '***MASKED***'
  END;

-- Apply the masking policy to the customer_email column
ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_mask_policy;

-- Test as a non-ANALYST_USER (e.g., assuming current user is 'DEVELOPER_USER')
-- SELECT order_id, customer_name, customer_email FROM sales_data;

-- Test as 'ANALYST_USER' (you would run this after logging in as 'ANALYST_USER' or using EXECUTE AS)
-- SELECT order_id, customer_name, customer_email FROM sales_data;
```

## Explanation of the Code

The example demonstrates setting up a masking policy that checks the `CURRENT_USER()` to decide whether to show the actual email or a masked string.

- `CREATE TABLE sales_data (...)`: This statement creates a simple table named `sales_data` with columns including `customer_email`, which will be masked.
- `INSERT INTO sales_data (...)`: Populates the `sales_data` table with sample records for demonstration.
- `CREATE MASKING POLICY email_mask_policy AS (val VARCHAR) RETURNS VARCHAR ->...`:
    - This defines a new masking policy called `email_mask_policy`.
    - `AS (val VARCHAR) RETURNS VARCHAR`: Specifies that the policy takes a `VARCHAR` input (the column value) and returns a `VARCHAR` (the masked or unmasked value).
    - `CASE WHEN CURRENT_USER() = 'ANALYST_USER' THEN val ELSE '***MASKED***' END`: This is the core logic.
        - `CURRENT_USER()`: A Snowflake function that returns the name of the user currently logged in and executing the query.
        - `='ANALYST_USER'`: Compares the current user's name to a specific user, `ANALYST_USER`.
        - `THEN val`: If the current user *is* `ANALYST_USER`, the original column value (`val`) is returned, meaning the email is shown unmasked.
        - `ELSE '***MASKED***'`: For any other user, the literal string `'***MASKED***'` is returned, hiding the original email.
- `ALTER TABLE sales_data MODIFY COLUMN customer_email SET MASKING POLICY email_mask_policy;`: This command applies the `email_mask_policy` to the `customer_email` column in the `sales_data` table. From this point on, any query on `customer_email` will invoke the policy.

## When to Use

1. **Column-level Access Control based on User Identity:** When specific users, not just roles, need direct access to unmasked data, or when testing specific user scenarios.
    * *Example Scenario:* A data auditor (`AUDITOR_USER`) needs to see all sensitive data for compliance checks, while all other users see masked data.
2. **Dynamic Data Redaction for Development/Testing Environments:** To ensure developers and testers working on non-production data can only see masked versions, unless they are explicitly authorized.
    * *Example Scenario:* In a `DEV` environment, only users with the `DEV_FULL_ACCESS_USER` can see actual customer names; everyone else sees `CUSTOMER_ID` or ``.[REDACTED]
3. **Ad-hoc User-Specific Overrides:** For temporary or specific situations where a particular user needs an exception to the general masking rules.
    * *Example Scenario:* A support technician (`SUPPORT_USER`) needs to temporarily view a customer's full address to resolve a shipping issue, but only when explicitly granted.

## When Not to Use

1. **Primary Reliance for Role-Based Access Control:** While `CURRENT_USER` works, `IS_ROLE_IN_SESSION` or `CURRENT_ROLE` is generally preferred for role-based masking as it's more scalable and manageable for a large number of users assigned to roles.
    * *Situation:* If your masking logic depends on roles (e.g., "HR role sees salaries, others don't"), use `CURRENT_ROLE()` or `IS_ROLE_IN_SESSION()` instead of listing every HR user with `CURRENT_USER()`.
2. **Performance-Critical Queries on Very Large Datasets:** Extremely complex masking policies, especially those involving `CURRENT_USER` with many `CASE` statements, might introduce minor overhead. If performance is absolutely paramount and simple masking is sufficient, a less dynamic approach might be considered.
    * *Situation:* A policy with dozens of `CURRENT_USER` checks for different user types on a column queried billions of times daily might be slightly less efficient than a simpler role-based check.
3. **Static Data Masking Requirements:** If the masking requirement is constant and doesn't depend on who is querying the data (e.g., always mask the middle 4 digits of a credit card number), then `CURRENT_USER` is unnecessary complexity.
    * *Situation:* A column always needs to be hashed or truncated regardless of the user; `CURRENT_USER` adds no value here.

## Common Mistakes

1. **Hardcoding Usernames without Proper Management:** Using specific usernames directly in policies (`CURRENT_USER() = 'ANALYST_USER'`) makes policies brittle. If the username changes or new users need access, the policy must be altered.
2. **Forgetting to Apply the Policy:** Creating the masking policy but forgetting to apply it to the column using `ALTER TABLE... SET MASKING POLICY...`. The data will remain unmasked.
3. **Not Understanding Role Hierarchy/Inheritance:** `CURRENT_USER()` returns the *current user name*, not the active role. If you intend to mask based on roles, use `CURRENT_ROLE()` or `IS_ROLE_IN_SESSION()`. Mixing these up leads to incorrect access control.
4. **Over-complicating Simple Policies:** Using `CURRENT_USER()` for every minor distinction when a more robust, role-based approach (`CURRENT_ROLE()`, `IS_ROLE_IN_SESSION()`) would be more scalable and maintainable.
5. **Lack of Testing with Different Users:** Not thoroughly testing the policy by actually logging in as different users (or using `EXECUTE AS USER`) to verify that the masking works as expected for both authorized and unauthorized individuals.

The resulting dataset will show the `order_id` and `customer_name` as they are stored. However, the `customer_email` column will display different values depending on the user who executes the `SELECT` statement.

**Example Table (when queried by `ANALYST_USER`):**

| order_id | customer_name | customer_email |
| :------- | :------------ | :------------------ |
| 1 | Alice Smith | alice.s@example.com |
| 2 | Bob Johnson | bob.j@example.com |
| 3 | Charlie Brown | charlie.b@example.com |

**Example Table (when queried by any user *other than* `ANALYST_USER`, e.g., `DEVELOPER_USER`):**

| order_id | customer_name | customer_email |
| :------- | :------------ | :------------- |
| 1 | Alice Smith | \*\*\*MASKED\*\*\* |
| 2 | Bob Johnson | \*\*\*MASKED\*\*\* |
| 3 | Charlie Brown | \*\*\*MASKED\*\*\* |

**Explanation of Columns:**

- `order_id`: The unique identifier for each order, displayed as stored.
- `customer_name`: The name of the customer, displayed as stored.
- `customer_email`: This column demonstrates the effect of the masking policy.
    - If the querying user is `ANALYST_USER`, the original email address is shown.
    -   If the querying user is *not* `ANALYYST_USER`, the email address is replaced with the string `'***MASKED***'`.
