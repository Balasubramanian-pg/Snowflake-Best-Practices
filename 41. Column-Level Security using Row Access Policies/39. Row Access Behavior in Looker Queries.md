# Row Access Behavior in Looker Queries in Snowflake

This concept describes how Snowflake's Row Access Policies, when applied to tables, interact with queries generated by Looker. It ensures that data accessed through Looker dashboards, reports, and explores adheres to the defined security policies, filtering rows based on user roles, attributes, or other criteria before the data is presented to the Looker user. This mechanism is crucial for implementing fine-grained access control and data governance within a self-service analytics environment.

**Short Explanation**

Row Access Behavior in Looker queries in Snowflake refers to how security policies in Snowflake restrict what data users see in Looker. Snowflake's row access policies automatically filter rows based on conditions, and Looker queries respect these filters without needing explicit WHERE clauses in the LookML.

- What problem does this SQL feature solve? It solves the problem of ensuring data security and compliance by enforcing row-level access control directly at the data source (Snowflake), regardless of the querying tool.
- Why is it important in databases or analytics? It's vital for governance, privacy, and regulatory compliance (like GDPR or HIPAA), ensuring that sensitive data is only visible to authorized individuals.
- Where is it commonly used in real workflows? It's used in organizations where different user groups or individuals should only see a subset of data in their analytics tools, such as sales teams only seeing their region's sales, or HR personnel only seeing employee data for their department.

## Implementation Example

```sql
-- 1. Create a table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR(100),
    region VARCHAR(50),
    sales_amount DECIMAL(10, 2),
    employee_id INT
);

INSERT INTO sales_data VALUES
(1, 'Laptop', 'East', 1200.00, 101),
(2, 'Mouse', 'West', 50.00, 102),
(3, 'Keyboard', 'East', 75.00, 101),
(4, 'Monitor', 'Central', 300.00, 103),
(5, 'Webcam', 'West', 60.00, 102);

-- 2. Create a row access policy function
CREATE OR REPLACE FUNCTION sales_access_policy(region VARCHAR)
RETURNS BOOLEAN
AS
$$
  CASE
    WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE
    WHEN CURRENT_ROLE() = 'EAST_SALES_ANALYST' AND region = 'East' THEN TRUE
    WHEN CURRENT_ROLE() = 'WEST_SALES_ANALYST' AND region = 'West' THEN TRUE
    ELSE FALSE
  END
$$;

-- 3. Apply the row access policy to the table
ALTER TABLE sales_data
ADD ROW ACCESS POLICY sales_access_policy ON (region);

-- 4. Grant necessary privileges and test with different roles (simulated Looker users)
-- Assume 'ADMIN_ROLE', 'EAST_SALES_ANALYST', 'WEST_SALES_ANALYST' roles exist and are granted to users.

-- Looker, when connected as 'EAST_SALES_ANALYST', will execute:
-- SELECT sale_id, product_name, region, sales_amount, employee_id FROM sales_data;
-- But the underlying query will be filtered by the policy.
```

## Explanation of the Code

This example demonstrates how to set up and visualize row access policy behavior.

- `CREATE OR REPLACE TABLE sales_data (...)`: This creates a sample table `sales_data` containing sales information, including a `region` column that will be used for access control.
- `INSERT INTO sales_data VALUES (...)`: Populates the `sales_data` table with example rows across different regions.
- `CREATE OR REPLACE FUNCTION sales_access_policy(region VARCHAR) RETURNS BOOLEAN AS $$...$$;`: This defines a SQL UDF (User-Defined Function) that acts as the row access policy.
    - It takes the `region` of a row as input.
    - Inside the `CASE` statement, it checks the `CURRENT_ROLE()` of the user executing the query.
    - If the user is `ADMIN_ROLE`, they see all rows (`TRUE`).
    - If the user is `EAST_SALES_ANALYST` AND the row's `region` is 'East', they see the row.
    - If the user is `WEST_SALES_ANALYST` AND the row's `region` is 'West', they see the row.
    - For any other role or region mismatch, the row is hidden (`FALSE`).
- `ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_access_policy ON (region);`: This command applies the `sales_access_policy` function to the `sales_data` table, specifically using the `region` column to evaluate the policy for each row.
- `SELECT ... FROM sales_data;` (simulated Looker query): When a user connected via Looker executes a query against `sales_data`, Snowflake automatically invokes the `sales_access_policy` function for each row, filtering the results before they reach Looker. Looker itself does not need to know about or apply the policy; it's enforced by Snowflake.

## When to Use

1.  **Strict Data Governance and Compliance:** When regulatory requirements (e.g., GDPR, HIPAA) dictate that specific users can only view certain data records, such as patient health records or personally identifiable information, based on their roles or attributes.
    *   *Scenario:* A healthcare provider needs to ensure that doctors only see their assigned patients' records, even when using a general analytics dashboard.
2.  **Multi-Tenant Architectures:** In a multi-tenant application where different customers' data resides in the same table, but each customer should only see their own data.
    *   *Scenario:* An SaaS company stores data for multiple clients in one large table. Each client's analytics dashboard in Looker should only display their own data.
3.  **Departmental or Regional Data Segmentation:** When different departments, regions, or teams within an organization should only have access to data relevant to their operational scope.
    *   *Scenario:* A global sales organization wants regional sales managers to see only the sales data for their specific region in Looker reports, not worldwide sales.

## When Not to Use

1.  **Overkill for Simple Column-Level Security:** If the requirement is simply to hide entire columns for certain users, rather than rows, column-level security (e.g., dynamic data masking or view-based restrictions) might be a simpler and more efficient approach.
    *   *Situation:* Users in finance shouldn't see customer email addresses, but they can see all sales transactions.
2.  **Performance Critical Workloads with Very Complex Policies:** While Snowflake's Row Access Policies are optimized, extremely complex policy functions involving multiple joins or subqueries within the policy itself, especially on very large tables, can introduce performance overhead.
    *   *Situation:* A policy function requires joining to several large mapping tables for every row, and the underlying data table has billions of records, leading to slow query times in Looker.
3.  **When Access Control is Managed Exclusively by the Application Layer:** If your application layer already handles all row-level security checks and dynamically generates SQL with `WHERE` clauses, adding a Snowflake Row Access Policy might be redundant and could potentially lead to conflicting policies or increased complexity.
    *   *Situation:* A custom internal application strictly enforces row-level security using its own business logic, and Looker is only used to visualize data for which the application has already pre-filtered access.

## Common Mistakes

1.  **Not Testing with Different Roles:** Developers often test the policy with an admin role or the role that created the policy, seeing all data, and fail to switch to the intended restricted roles to verify the filtering works as expected.
2.  **Incorrect Policy Logic:** Errors in the `CASE` statement of the policy function, leading to unintended data exposure or over-restriction (e.g., using `OR` instead of `AND`, or missing a role condition).
3.  **Performance Impact Overlooked:** Creating overly complex policy functions (e.g., with multiple subqueries or joins within the policy itself) that are executed for every row, leading to degraded query performance, especially on large tables.
4.  **Forgetting to Apply or Grant Privileges:** The policy function must be created, applied to the table (`ALTER TABLE ADD ROW ACCESS POLICY`), and the roles executing queries must have `SELECT` privileges on the table and `USAGE` on the policy function's schema and database.
5.  **Hardcoding Usernames/Roles:** Building policies that directly reference individual usernames instead of roles, making maintenance difficult as users join or leave. Using `CURRENT_ROLE()` or `IS_ROLE_IN_SESSION()` with well-managed roles is generally better.

## Expected Output

The resulting dataset in Looker will appear as if a `WHERE` clause was implicitly applied to the original table, based on the logged-in user's role and the row access policy. The columns and their types will remain the same as the base table, but the number of rows will be reduced for restricted users.

For a user with the `EAST_SALES_ANALYST` role querying `sales_data`, the output would look like this:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT | EMPLOYEE_ID |
| :------ | :----------- | :----- | :----------- | :---------- |
| 1       | Laptop       | East   | 1200.00      | 101         |
| 3       | Keyboard     | East   | 75.00        | 101         |

-   `SALE_ID`: Unique identifier for the sale.
-   `PRODUCT_NAME`: Name of the product sold.
-   `REGION`: The sales region where the transaction occurred. (Only 'East' visible for this user).
-   `SALES_AMOUNT`: The monetary value of the sale.
-   `EMPLOYEE_ID`: The ID of the employee who made the sale.
