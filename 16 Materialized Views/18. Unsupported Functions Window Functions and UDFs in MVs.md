# Unsupported Functions: Window Functions and UDFs in Materialized Views in Snowflake

Materialized Views (MVs) in Snowflake are pre-computed datasets derived from a query, stored for faster retrieval. While incredibly powerful for performance optimization, they come with certain limitations regarding the types of SQL constructs they can include in their definitions. Specifically, Window Functions and User-Defined Functions (UDFs) are generally unsupported directly within a Snowflake Materialized View definition. This restriction exists to ensure the efficient and automatic maintenance of MVs by Snowflake's background services.

**Short Explanation**

Snowflake Materialized Views (MVs) cannot directly include Window Functions (like `ROW_NUMBER()`, `RANK()`, `LAG()`) or User-Defined Functions (UDFs) in their defining query. This limitation helps Snowflake efficiently manage and update the pre-computed data in MVs, ensuring they remain performant and consistent with the base table. Using these functions would make the MV's content harder to incrementally update and optimize, potentially leading to higher maintenance costs and slower refresh times.

- **What problem does this SQL feature solve?** This isn't a "feature" solving a problem, but rather a *limitation* to prevent other problems. It ensures materialized views can be automatically and efficiently refreshed by Snowflake, providing consistent and fast query performance without manual intervention.
- **Why is it important in databases or analytics?** This restriction maintains the integrity and performance benefits of materialized views. Without it, the complexity of pre-calculating and incrementally updating results from non-deterministic functions or complex windowing logic would make MVs much less effective or even impossible to maintain automatically.
- **Where is it commonly used in real workflows?** This limitation primarily affects scenarios where users want to pre-compute complex analytical results (which might involve window functions) or integrate custom logic (UDFs) directly into a materialized view for performance. Instead, these operations typically need to be applied in a regular view or query built *on top* of a simpler materialized view.

## Implementation Example

Since these are *unsupported* features, there isn't an "implementation example" where they work. Instead, here's an example of what would fail and how you might work around it.

**Attempt (Will Fail):**

```sql
CREATE MATERIALIZED VIEW mv_daily_sales_rank AS
SELECT
    sale_date,
    product_id,
    sales_amount,
    ROW_NUMBER() OVER (PARTITION BY sale_date ORDER BY sales_amount DESC) as sales_rank
FROM
    sales_data;

-- This query would fail due to the window function
```

**Workaround (Recommended Approach):**

First, create a simple materialized view for the base data:

```sql
CREATE MATERIALIZED VIEW mv_daily_sales AS
SELECT
    sale_date,
    product_id,
    sales_amount
FROM
    sales_data;
```

Then, create a regular view (or query directly) on top of the materialized view to apply the window function or UDF:

```sql
CREATE VIEW v_daily_sales_with_rank AS
SELECT
    sale_date,
    product_id,
    sales_amount,
    ROW_NUMBER() OVER (PARTITION BY sale_date ORDER BY sales_amount DESC) as sales_rank
FROM
    mv_daily_sales;
```

## Explanation of the Code

**Workaround Query Explanation:**

The workaround involves two steps:

1. **`CREATE MATERIALIZED VIEW mv_daily_sales AS...`**:
    * **What it does:** This statement creates a materialized view named `mv_daily_sales`. It pre-computes and stores the `sale_date`, `product_id`, and `sales_amount` columns from the `sales_data` table.
    * **How it changes the result set:** It essentially creates a physical, pre-stored table containing the raw sales data. Snowflake automatically maintains this MV when `sales_data` changes.

2. **`CREATE VIEW v_daily_sales_with_rank AS...`**:
    * **What it does:** This statement creates a *regular* view named `v_daily_sales_with_rank`. This view doesn't store data; instead, it defines a query that will be executed whenever the view is queried. Here, it queries the `mv_daily_sales` (the materialized view) and applies a window function.
    * **How it changes the result set:**
        * `SELECT sale_date, product_id, sales_amount,...`: Specifies the columns to be retrieved, including the calculated `sales_rank`.
        * `FROM mv_daily_sales`: This is the base data source for the view. By using the materialized view, the initial data retrieval is fast because it's already pre-computed.
        * `ROW_NUMBER() OVER (PARTITION BY sale_date ORDER BY sales_amount DESC) as sales_rank`: This is the window function.
            * **What it does:** Assigns a unique, sequential integer to each row within groups (partitions) defined by `sale_date`. The numbering restarts for each new `sale_date`.
            * **How it changes the result set:** It adds a new column, `sales_rank`, which orders sales within each day from highest to lowest amount, assigning `1` to the top sale of the day, `2` to the second, and so on. This column is computed dynamically each time `v_daily_sales_with_rank` is queried.

## When to Use

1. **When you need pre-aggregated or filtered data for performance, but also require complex analytical calculations**: Create a simpler MV with the base data, then apply window functions or UDFs in a regular view or final query that selects from the MV. This leverages the MV's performance for the initial data retrieval.
    * **Example Scenario:** You frequently analyze daily sales, needing the top 10 products by revenue each day, and also apply a custom scoring UDF to each product. You'd MV the daily sales totals, then query that MV with the ranking window function and UDF.
2. **When a UDF performs complex, but often reused, business logic on a large dataset**: Materialize the raw or lightly transformed data, then call the UDF in subsequent queries or views. This ensures the UDF is executed on an already optimized (and potentially smaller) dataset.
    * **Example Scenario:** You have a UDF `calculate_customer_segment(customer_id, total_spend, last_purchase_date)` that takes time to run. You can create an MV for customer activity data, and then apply `calculate_customer_segment` to the MV, rather than the raw transactional tables.
3. **When you need to maintain data consistency and freshness with minimal maintenance effort**: Snowflake's automatic refresh of MVs works best when their definitions are simple and adhere to the supported syntax. Using a regular view on top allows for dynamic, complex calculations while still benefiting from the MV's underlying efficiency.
    * **Example Scenario:** A dashboard needs to show trending products over various time periods (using `LAG` or `LEAD`). The base product sales data can be in an MV, ensuring rapid access, while the trending calculations are performed by the dashboard's queries or an intermediate view.

## When Not to Use

1. **When the core purpose of the materialized view is solely to pre-compute the results of a window function or UDF that is computationally expensive on its own**: If the window function or UDF *itself* is the bottleneck and needs materialization, a Snowflake MV cannot directly help, as it prohibits these constructs. You might need to pre-process the data in an external ETL tool or use a standard table refreshed periodically.
    * **Inefficient Example:** Trying to create an MV that *only* calculates `PERCENT_RANK()` across a very large dataset, expecting the MV to store the rank values.
2. **When the UDF is non-deterministic and needs to be evaluated on each query for fresh results**: If a UDF's output changes every time it's called (e.g., uses `CURRENT_TIMESTAMP`), it fundamentally cannot be part of an MV definition, which relies on deterministic inputs for caching.
    * **Inefficient Example:** An MV definition including a UDF that generates a random number or uses current time to create a unique ID.
3. **When the complexity introduced by layering views on top of MVs becomes unmanageable**: While using a regular view on an MV is a valid workaround, too many layers of views or overly complex definitions can make debugging and maintenance difficult. Sometimes, a simpler, regularly scheduled `CREATE TABLE AS SELECT` (CTAS) might be a better fit if the refresh frequency is low and the calculations are truly static.
    * **Inefficient Example:** Building five layers of views, each with multiple window functions and UDFs, on top of a single base MV, causing confusion and performance degradation due to view-on-view processing.

## Common Mistakes

1. **Attempting to include Window Functions or UDFs directly in `CREATE MATERIALIZED VIEW` statements**: This is the most common mistake, leading to SQL compilation errors. Users often assume that MVs can materialize *any* query.
2. **Not understanding the difference between a Materialized View and a Regular View**: Confusing the capabilities leads to frustration. MVs are for pre-computed, implicitly managed data stores, while regular views are dynamic queries.
3. **Over-complicating the MV definition**: Trying to push too much logic into the MV, instead of keeping it simple for optimal performance and automatic maintenance, then layering complex logic on top with regular views.
4. **Ignoring the cost implications**: While MVs improve query performance, they incur storage costs and compute costs for automatic maintenance. Complex MVs, even without prohibited functions, can be expensive to maintain.[uDFk]
5. **Using UDFs that are not immutable/deterministic**: Even if UDFs were allowed, using non-deterministic ones would prevent effective materialization. Always consider the determinism of functions for MVs.

## Expected Output

When using the workaround, the `v_daily_sales_with_rank` view would produce a result set that includes the original columns from the `mv_daily_sales` plus a new column, `sales_rank`, calculated using the window function.

| SALE_DATE | PRODUCT_ID | SALES_AMOUNT | SALES_RANK |
| :--------- | :--------- | :----------- | :--------- |
| 2026-03-01 | P101 | 500.00 | 1 |
| 2026-03-01 | P105 | 350.50 | 2 |
| 2026-03-01 | P103 | 120.00 | 3 |
| 2026-03-02 | P202 | 780.25 | 1 |
| 2026-03-02 | P201 | 600.00 | 2 |

**Explanation of Columns:**

* **SALE_DATE:** The date of the sales transaction.
* **PRODUCT_ID:** The identifier for the product sold.
* **SALES_AMOUNT:** The total amount of sales for that product on that day.
*   **SALES_RANK:** A sequential rank assigned to each product's sales amount *within its `SALE_DATE` group*, ordered from the highest `SALES_AMOUNT` (rank 1) to the lowest. This column is derived from the `ROW_NUMBER()` window function.
