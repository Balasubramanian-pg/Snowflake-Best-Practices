# Diagnosing Overloaded Warehouses in Snowflake

Diagnosing overloaded warehouses in Snowflake involves identifying when a virtual warehouse is struggling to process its assigned workload efficiently. This is crucial for maintaining optimal performance, ensuring timely query execution, and managing costs effectively in a cloud data warehouse environment. Snowflake warehouses are designed to scale, but improper sizing or sudden workload spikes can lead to performance degradation if not monitored and addressed.

**Short Explanation**

Diagnosing overloaded warehouses means figuring out why your Snowflake compute resources are slowing down. It helps you pinpoint if too many queries, too large queries, or an undersized warehouse are causing performance issues. This is vital for ensuring queries run fast and efficiently, impacting everything from daily reporting to complex analytics. You'd commonly use this when users report slow dashboards, queries are timing out, or costs are unexpectedly high due to inefficient resource usage.

- **What problem does this SQL feature solve?** It helps identify performance bottlenecks related to compute resources (warehouses), preventing slow query execution and ensuring consistent analytical performance.
- **Why is it important in databases or analytics?** Proper warehouse sizing and monitoring are critical for managing cloud costs and meeting Service Level Agreements (SLAs) for data availability and query response times.
- **Where is it commonly used in real workflows?** Data engineers and platform administrators regularly use these diagnostic techniques to monitor Snowflake environments, troubleshoot performance complaints, and optimize resource allocation.

## Implementation Example

```sql
-- Query 1: Check warehouse load and queueing
SELECT
    warehouse_name,
    SUM(CASE WHEN query_status = 'QUEUED' THEN 1 ELSE 0 END) AS queued_queries,
    COUNT(query_id) AS total_queries_in_warehouse,
    AVG(execution_time) AS avg_execution_time,
    AVG(queued_overload_time) AS avg_queued_overload_time,
    AVG(queued_provisioning_time) AS avg_queued_provisioning_time
FROM
    snowflake.account_usage.query_history
WHERE
    start_time >= DATEADD(hour, -1, CURRENT_TIMESTAMP()) -- Last 1 hour
    AND warehouse_name IS NOT NULL
GROUP BY
    warehouse_name
ORDER BY
    queued_queries DESC;

-- Query 2: Identify long-running queries on a specific warehouse
SELECT
    query_id,
    query_text,
    user_name,
    warehouse_name,
    start_time,
    end_time,
    total_elapsed_time / 1000 AS total_elapsed_seconds,
    execution_time / 1000 AS execution_seconds,
    queued_overload_time / 1000 AS queued_overload_seconds
FROM
    snowflake.account_usage.query_history
WHERE
    warehouse_name = 'YOUR_OVERLOADED_WAREHOUSE_NAME'
    AND start_time >= DATEADD(hour, -1, CURRENT_TIMESTAMP())
    AND total_elapsed_time > 60000 -- Queries running longer than 60 seconds
ORDER BY
    total_elapsed_time DESC;
```

## Explanation of the Code

This example uses Snowflake's `ACCOUNT_USAGE` views to gather metrics for diagnosing warehouse overload.

**Query 1: Check warehouse load and queueing**

- `SELECT warehouse_name, SUM(CASE WHEN query_status = 'QUEUED' THEN 1 ELSE 0 END) AS queued_queries, COUNT(query_id) AS total_queries_in_warehouse, AVG(execution_time) AS avg_execution_time, AVG(queued_overload_time) AS avg_queued_overload_time, AVG(queued_provisioning_time) AS avg_queued_provisioning_time`: This clause selects the warehouse name and calculates several aggregate metrics:
    - `queued_queries`: Counts how many queries are currently in a 'QUEUED' status due to the warehouse being overloaded. A high number here is a strong indicator of overload.
    - `total_queries_in_warehouse`: Shows the total number of queries processed by each warehouse in the specified period.
    - `avg_execution_time`: Average time a query spent executing.
    - `avg_queued_overload_time`: Average time queries spent waiting in the queue because the warehouse was at full capacity. This is a key metric for overload.
    - `avg_queued_provisioning_time`: Average time queries spent waiting for the warehouse to provision resources (e.g., scale up).
- `FROM snowflake.account_usage.query_history`: Specifies the source of the data, which is Snowflake's built-in `QUERY_HISTORY` view, providing comprehensive details about all executed queries.
- `WHERE start_time >= DATEADD(hour, -1, CURRENT_TIMESTAMP()) AND warehouse_name IS NOT NULL`: Filters the data to only include queries from the last hour and ensures a warehouse name is associated. This focuses on recent activity.
- `GROUP BY warehouse_name`: Aggregates the results for each distinct virtual warehouse, allowing a per-warehouse performance overview.
- `ORDER BY queued_queries DESC`: Sorts the results to show warehouses with the most queued queries first, highlighting potential problem areas.

**Query 2: Identify long-running queries on a specific warehouse**

- `SELECT query_id, query_text, user_name, warehouse_name, start_time, end_time, total_elapsed_time / 1000 AS total_elapsed_seconds, execution_time / 1000 AS execution_seconds, queued_overload_time / 1000 AS queued_overload_seconds`: This clause selects detailed information about individual queries.
    - `query_text`: The actual SQL command executed, useful for understanding what the query was doing.
    - `total_elapsed_time`: The total time taken for the query, including queueing, compilation, and execution.
    - `execution_time`: The time spent actually executing the query.
    - `queued_overload_time`: Time spent waiting due to warehouse overload.
- `FROM snowflake.account_usage.query_history`: Same data source as Query 1.
- `WHERE warehouse_name = 'YOUR_OVERLOADED_WAREHOUSE_NAME' AND start_time >= DATEADD(hour, -1, CURRENT_TIMESTAMP()) AND total_elapsed_time > 60000`: Filters queries to a specific warehouse, within the last hour, and only those that took longer than 60 seconds to complete (adjust as needed). This helps pinpoint problematic queries.
- `ORDER BY total_elapsed_time DESC`: Sorts the results to show the longest-running queries first, making it easier to identify top contributors to warehouse load.

## When to Use

1.  **Sudden Performance Drops**: When users report that dashboards or reports are running significantly slower than usual, suggesting a recent change in workload or warehouse capacity.
    *   *Example Scenario*: Your daily sales report, which typically takes 5 minutes, now takes 30 minutes to complete. Checking `queued_overload_time` reveals a spike.
2.  **Increased Query Queueing**: If you observe a growing number of queries stuck in a "QUEUED" state, indicating the warehouse doesn't have enough resources to handle the concurrent workload.
    *   *Example Scenario*: Monitoring shows that your analytics warehouse frequently has 10+ queries in the queue, rather than immediately starting execution, pointing to resource contention.
3.  **Unexpected Cost Increases**: When Snowflake credits consumption for a warehouse dramatically increases without a corresponding increase in data processing, potentially due to inefficient query patterns or constant auto-scaling that's not effectively used.
    *   *Example Scenario*: A warehouse that usually consumes 100 credits a day is now consuming 300, and you suspect it's constantly scaling up and down or processing redundant queries due to overload.

## When Not to Use

1.  **Diagnosing Data Ingestion Issues**: These queries focus on warehouse compute performance. They won't help troubleshoot problems with data loading (e.g., using Snowpipe or COPY INTO) unless the ingestion itself is causing a warehouse overload.
    *   *Example Scenario*: Your data pipeline is failing to load new data into a table. The warehouse queries won't directly show why the `COPY INTO` command is failing if it's due to data format errors rather than warehouse capacity.
2.  **Troubleshooting External System Integrations**: If the bottleneck is upstream (e.g., a slow source system, network latency outside Snowflake, or an inefficient ETL tool), these queries will only show the queries arriving at Snowflake, not the root cause of delays before that.
    *   *Example Scenario*: A dashboard connected via Tableau is slow. If the Tableau server itself is under-resourced or the network connection to Snowflake is poor, these queries won't point to that external issue.
3.  **Identifying Data Quality Problems**: These diagnostic queries are about performance, not the correctness or integrity of your data.
    *   *Example Scenario*: Your reports show incorrect totals. While a slow query might exacerbate the issue, these diagnostic queries won't tell you if the underlying data has nulls, duplicates, or calculation errors.

## Common Mistakes

1.  **Ignoring `QUEUED_OVERLOAD_TIME`**: Many users focus solely on `execution_time` and miss `queued_overload_time`, which is the most direct indicator of a warehouse being unable to cope with concurrent demand.
2.  **Not Considering Query Concurrency**: An overloaded warehouse isn't always about query complexity; often, it's too many simple queries running simultaneously that exceed the warehouse's parallelism.
3.  **Applying a One-Size-Fits-All Warehouse**: Using a single warehouse size for all workloads (ETL, ad-hoc analytics, reporting) often leads to some workloads being starved or over-resourced. Different workloads have different performance profiles and require appropriate warehouse sizing and sometimes separate warehouses.
4.  **Failing to Understand Auto-Scaling Behavior**: Misinterpreting scaling up/down events. Sometimes, a warehouse scales up not because it's overloaded, but because of a specific query's resource demands or because of policies set on the warehouse. Understanding the interplay of auto-scaling and multi-cluster warehouses is key.

## Expected Output

The expected output for Query 1 (`Diagnosing Overloaded Warehouses in Snowflake`) would be a list of your virtual warehouses, showing how many queries are currently queued due to overload and their average performance metrics.

**Example Table for Query 1: Warehouse Load**

| WAREHOUSE_NAME | QUEUED_QUERIES | TOTAL_QUERIES_IN_WAREHOUSE | AVG_EXECUTION_TIME | AVG_QUEUED_OVERLOAD_TIME | AVG_QUEUED_PROVISIONING_TIME |
| :------------- | :------------- | :------------------------- | :----------------- | :----------------------- | :--------------------------- |
| ANALYTICS_WH   | 5              | 120                        | 5500               | 1200                     | 0                            |
| ETL_WH         | 0              | 30                         | 12000              | 0                        | 0                            |
| REPORTING_WH   | 1              | 80                         | 3200               | 300                      | 0                            |

-   **WAREHOUSE_NAME**: The name of the virtual warehouse.
-   **QUEUED_QUERIES**: The count of queries that were queued due to overload in the specified time frame. A non-zero value here indicates potential overload.
-   **TOTAL_QUERIES_IN_WAREHOUSE**: The total number of queries executed by this warehouse in the specified time frame.
-   **AVG_EXECUTION_TIME**: The average time (in milliseconds) queries spent executing on this warehouse.
-   **AVG_QUEUED_OVERLOAD_TIME**: The average time (in milliseconds) queries spent waiting in the queue because the warehouse was at full capacity. This is a critical metric for diagnosing overload.
-   **AVG_QUEUED_PROVISIONING_TIME**: The average time (in milliseconds) queries spent waiting for the warehouse to provision new resources during scaling events.

The expected output for Query 2 (`Identify long-running queries on a specific warehouse`) would be a detailed list of queries that took a long time to complete on a specific warehouse, sorted by total elapsed time.

**Example Table for Query 2: Long-Running Queries**

| QUERY_ID                             | QUERY_TEXT                                             | USER_NAME | WAREHOUSE_NAME | START_TIME                  | END_TIME                    | TOTAL_ELAPSED_SECONDS | EXECUTION_SECONDS | QUEUED_OVERLOAD_SECONDS |
| :----------------------------------- | :----------------------------------------------------- | :-------- | :------------- | :-------------------------- | :-------------------------- | :-------------------- | :---------------- | :---------------------- |
| 01b4e062-07a8-4c1d-...               | SELECT * FROM sales_data WHERE order_date >= '2020-01-01'... | DATA_ANALYST | ANALYTICS_WH   | 2026-03-14 10:30:00.000+0000 | 2026-03-14 10:32:15.000+0000 | 135                   | 130               | 5                       |
| 01b4e065-08b9-4d2e-...               | INSERT INTO aggregated_metrics SELECT SUM(...) FROM... | ETL_USER  | ANALYTICS_WH   | 2026-03-14 10:31:00.000+0000 | 2026-03-14 10:32:00.000+0000 | 60                    | 55                | 5                       |

-   **QUERY_ID**: Unique identifier for the query.
-   **QUERY_TEXT**: The actual SQL code that was executed.
-   **USER_NAME**: The user who ran the query.
-   **WAREHOUSE_NAME**: The warehouse on which the query was executed.
-   **START_TIME**: When the query started.
-   **END_TIME**: When the query finished.
-   **TOTAL_ELAPSED_SECONDS**: Total time from start to finish, including any queueing.
-   **EXECUTION_SECONDS**: Time spent actively executing the query.
-   **QUEUED_OVERLOAD_SECONDS**: Time spent in the queue specifically due to the warehouse being overloaded. A high value here for a long-running query suggests the warehouse was saturated when this query tried to run.
