# Tracking Warehouse Load Percentage in Snowflake

Tracking warehouse load percentage in Snowflake refers to monitoring how much of a virtual warehouse's computational capacity is being utilized by running queries. It's crucial for optimizing performance, managing costs, and ensuring that your data workloads are efficiently processed. By understanding load, you can right-size warehouses, identify bottlenecks, and prevent resource contention.

**Short Explanation**

Warehouse load percentage measures how busy a Snowflake virtual warehouse is, showing the proportion of its available computing power currently in use. This SQL feature solves the problem of resource underutilization or overutilization, which directly impacts query performance and cloud expenditure. It's important in databases and analytics because it helps database administrators and data engineers make informed decisions about warehouse sizing and concurrency settings, ensuring a balance between performance and cost. It's commonly used in real workflows for capacity planning, performance tuning, and cost management in data warehousing environments.

- **What problem does this SQL feature solve?** It solves the problem of inefficient resource allocation and utilization, helping to prevent slow queries due to an overloaded warehouse or wasted spending due to an underutilized one.
- **Why is it important in databases or analytics?** It's vital for maintaining consistent query performance, optimizing infrastructure costs, and scaling resources effectively to meet demand.
- **Where is it commonly used in real workflows?** It's used in monitoring dashboards, alert systems for performance deviations, and capacity planning exercises to determine optimal warehouse sizes for different workloads.

## Implementation Example

```sql
SELECT
    TO_TIMESTAMP_NTZ(TRUNC(START_TIME, 'MINUTE')) AS minute_start,
    WAREHOUSE_NAME,
    AVG(PERCENTAGE_SCALED_OUT) AS avg_scaled_out_percentage,
    AVG(QUERY_LOAD_PERCENT) AS avg_query_load_percentage,
    MAX(QUERY_LOAD_PERCENT) AS max_query_load_percentage
FROM
    SNOWFLAKE.ACCOUNT_USAGE.WAREHOUSE_LOAD_HISTORY
WHERE
    START_TIME >= DATEADD(hour, -24, CURRENT_TIMESTAMP())
    AND WAREHOUSE_NAME = 'MY_ANALYTICS_WAREHOUSE' -- Replace with your warehouse name
GROUP BY
    1, 2
ORDER BY
    minute_start DESC;
```

## Explanation of the Code

This query retrieves and aggregates warehouse load percentage data from Snowflake's `ACCOUNT_USAGE` schema for a specific warehouse over the last 24 hours.

-   **`SELECT TO_TIMESTAMP_NTZ(TRUNC(START_TIME, 'MINUTE')) AS minute_start, WAREHOUSE_NAME, AVG(PERCENTAGE_SCALED_OUT) AS avg_scaled_out_percentage, AVG(QUERY_LOAD_PERCENT) AS avg_query_load_percentage, MAX(QUERY_LOAD_PERCENT) AS max_query_load_percentage`**:
    -   **What it does:** This clause specifies the columns to be retrieved and calculated.
    -   **How it changes the result set:** It projects the start time truncated to the minute, the warehouse name, the average scaled-out percentage (for multi-cluster warehouses), the average query load percentage, and the maximum query load percentage for each minute. `TO_TIMESTAMP_NTZ` ensures the timestamp is without a timezone, and `TRUNC` rounds down the timestamp to the nearest minute, allowing for minute-by-minute aggregation.

-   **`FROM SNOWFLAKE.ACCOUNT_USAGE.WAREHOUSE_LOAD_HISTORY`**:
    -   **What it does:** This clause indicates the table from which the data is being fetched. `WAREHOUSE_LOAD_HISTORY` is a view in the `ACCOUNT_USAGE` schema that provides historical data on warehouse load.
    -   **How it changes the result set:** It defines the source of the data for the query.

-   **`WHERE START_TIME >= DATEADD(hour, -24, CURRENT_TIMESTAMP()) AND WAREHOUSE_NAME = 'MY_ANALYTICS_WAREHOUSE'`**:
    -   **What it does:** This clause filters the rows based on specified conditions.
    -   **How it changes the result set:** It restricts the data to records from the last 24 hours (`START_TIME >= DATEADD(hour, -24, CURRENT_TIMESTAMP())`) and for a particular warehouse named 'MY_ANALYTICS_WAREHOUSE'. This narrows down the historical data to a relevant time frame and specific resource.

-   **`GROUP BY 1, 2`**:
    -   **What it does:** This clause groups rows that have the same values in the specified columns into a summary row.
    -   **How it changes the result set:** It aggregates the `AVG` and `MAX` calculations for each unique `minute_start` and `WAREHOUSE_NAME` combination.

-   **`ORDER BY minute_start DESC`**:
    -   **What it does:** This clause sorts the result set based on the specified column(s).
    -   **How it changes the result set:** It arranges the results in descending order of `minute_start`, showing the most recent load data first.

## When to Use

1.  **Capacity Planning:** When you need to decide whether to scale up or down a warehouse. If a warehouse consistently shows high load percentages (e.g., >80%) during peak hours, it might indicate a need for a larger warehouse size or increased concurrency.
    *   *Example Scenario:* Reviewing monthly load history to determine if your "DailyBatch" warehouse needs to be upgraded from `MEDIUM` to `LARGE` before the next quarter's anticipated data growth.
2.  **Performance Troubleshooting:** When users report slow query performance. A sudden spike or sustained high load percentage can point to the warehouse being a bottleneck.
    *   *Example Scenario:* A dashboard refresh takes too long. You check the `WAREHOUSE_LOAD_HISTORY` and find `REPORTING_WH` hit 95% load during the refresh window, indicating resource contention.
3.  **Cost Optimization:** To identify underutilized warehouses. If a warehouse consistently has very low load percentages (e.g., <20%), you might be overpaying for its capacity.
    *   *Example Scenario:* Discovering that your `ADHOC_QUERY_WH` is rarely above 10% load, suggesting it could be scaled down to a smaller size or configured to auto-suspend more aggressively.

## When Not to Use

1.  **Real-time Individual Query Performance:** While load history helps understand overall warehouse health, it doesn't give granular insights into why a *specific* query was slow. For that, you'd use `QUERY_HISTORY` for detailed query profiles.
    *   *Situation:* A user complains about one specific query taking 30 minutes instead of 5. Looking at `WAREHOUSE_LOAD_HISTORY` might show overall high load, but `QUERY_HISTORY` for that specific query ID would reveal bottlenecks like disk spilling or poor join strategies.
2.  **Understanding Application-level Bottlenecks:** Warehouse load reflects database-level resource usage, not issues in the application layer (e.g., inefficient ETL logic, poorly optimized application code, network latency between application and Snowflake).
    *   *Situation:* Data ingestion is slow, but the `INGESTION_WH` shows low load. The problem likely lies in the ETL tool's configuration, source system extraction, or network, not the Snowflake warehouse itself.
3.  **Predicting Future Workload Patterns Without Context:** While historical load data is valuable, simply extrapolating without considering planned changes (new reports, increased user base, new data sources) can lead to inaccurate capacity predictions.
    *   *Situation:* Automatically scaling up a warehouse based purely on past peaks without realizing that a major data source is being retired next month, which will significantly reduce the workload.

## Common Mistakes

1.  **Ignoring `PERCENTAGE_SCALED_OUT` for Multi-cluster Warehouses:** For multi-cluster warehouses, `QUERY_LOAD_PERCENT` only reflects the load on *one* cluster. `PERCENTAGE_SCALED_OUT` indicates how many clusters are active. A low `QUERY_LOAD_PERCENT` with high `PERCENTAGE_SCALED_OUT` means your queries are effectively utilizing multiple clusters, which is good, not underutilization.
2.  **Focusing Only on Averages:** Using only `AVG(QUERY_LOAD_PERCENT)` can hide peak loads. A warehouse might average 50% load but hit 100% for short, critical periods, causing performance dips. Always look at `MAX` or 90th percentile values during relevant time windows.
3.  **Misinterpreting Low Load as Underutilization:** A low load percentage doesn't always mean a warehouse is too big. For interactive user-facing dashboards, you might intentionally provision a larger warehouse to ensure sub-second query response times, even if its average load is low. The goal isn't always 100% utilization, but optimal performance for the workload.

## Expected Output

The resulting dataset will show the average and maximum query load percentage for a specified Snowflake warehouse, broken down by minute for the last 24 hours (or the specified time frame), along with the average percentage of clusters scaled out for multi-cluster warehouses. Each row represents a one-minute interval for the specific warehouse.

| MINUTE\_START              | WAREHOUSE\_NAME         | AVG\_SCALED\_OUT\_PERCENTAGE | AVG\_QUERY\_LOAD\_PERCENT | MAX\_QUERY\_LOAD\_PERCENT |
| :------------------------- | :---------------------- | :--------------------------- | :------------------------ | :------------------------ |
| 2026-03-14 14:59:00.000    | MY\_ANALYTICS\_WAREHOUSE | 100.00                       | 78.50                     | 92.00                     |
| 2026-03-14 14:58:00.000    | MY\_ANALYTICS\_WAREHOUSE | 100.00                       | 65.25                     | 88.00                     |
| 2026-03-14 14:57:00.000    | MY\_ANALYTICS\_WAREHOUSE | 100.00                       | 52.00                     | 75.00                     |

-   **MINUTE\_START:** The start of the one-minute interval for which the load metrics are calculated.
-   **WAREHOUSE\_NAME:** The name of the virtual warehouse being monitored.
-   **AVG\_SCALED\_OUT\_PERCENTAGE:** For multi-cluster warehouses, this indicates the average percentage of clusters that were active during that minute (e.g., 100.00 means all clusters were active). For single-cluster warehouses, it's typically 100.00 if the warehouse was running.
-   **AVG\_QUERY\_LOAD\_PERCENT:** The average query load percentage across all active clusters during that one-minute interval.
-   **MAX\_QUERY\_LOAD\_PERCENT:** The maximum query load percentage observed across all active clusters within that one-minute interval. This helps identify peak usage.
