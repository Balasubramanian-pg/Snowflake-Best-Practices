# Restricting Create Privileges in Snowflake

Restricting `CREATE` privileges in Snowflake is a fundamental security practice that ensures users can only create objects (like databases, schemas, tables, views, or roles) when explicitly authorized to do so. This concept is crucial for maintaining data governance, adhering to the principle of least privilege, and preventing unauthorized schema modifications or resource consumption. It primarily involves carefully assigning `CREATE` privileges to specific roles and managing those roles effectively within Snowflake's Role-Based Access Control (RBAC) framework.

**Short Explanation**

This SQL feature solves the problem of uncontrolled object creation within a Snowflake account, which can lead to security vulnerabilities, disorganized environments, and wasted resources. It is important in databases and analytics to enforce strict control over who can build or modify the data infrastructure. This is commonly used in regulated industries or large organizations to maintain a clean, secure, and well-governed data platform.

- What problem does this SQL feature solve? It prevents unauthorized users from creating database objects, maintaining order, security, and resource control.
- Why is it important in databases or analytics? It's key for data governance, auditing, and ensuring only approved structures are deployed, which is vital for data integrity and compliance.
- Where is it commonly used in real workflows? In environments with multiple teams, strict change management processes, or compliance requirements (e.g., finance, healthcare), where uncontrolled creation could lead to chaos or security breaches.

## Implementation Example

Let's say we want to create a dedicated role called `DEVELOPER_ROLE` that can create tables within a specific schema, `DEV_DB.DEV_SCHEMA`, but cannot create new databases or schemas.

```sql
-- 1. Create a custom role for developers
CREATE ROLE DEVELOPER_ROLE;

-- 2. Grant the DEVELOPER_ROLE to a user (replace 'your_user' with an actual username)
GRANT ROLE DEVELOPER_ROLE TO USER your_user;

-- 3. Grant USAGE on the database to allow interaction with it
GRANT USAGE ON DATABASE DEV_DB TO ROLE DEVELOPER_ROLE;

-- 4. Grant USAGE on the schema to allow interaction with it
GRANT USAGE ON SCHEMA DEV_DB.DEV_SCHEMA TO ROLE DEVELOPER_ROLE;

-- 5. Grant CREATE TABLE privilege on the specific schema
GRANT CREATE TABLE ON SCHEMA DEV_DB.DEV_SCHEMA TO ROLE DEVELOPER_ROLE;

-- 6. (Optional) Revoke CREATE DATABASE/SCHEMA privileges from public/less privileged roles if they were inadvertently granted
REVOKE CREATE DATABASE ON ACCOUNT FROM PUBLIC;
REVOKE CREATE SCHEMA ON DATABASE DEV_DB FROM PUBLIC;

-- 7. (Optional) Create a default warehouse for the developer role
CREATE WAREHOUSE DEV_WH WITH WAREHOUSE_SIZE = 'XSMALL';
GRANT USAGE ON WAREHOUSE DEV_WH TO ROLE DEVELOPER_ROLE;
ALTER USER your_user SET DEFAULT_ROLE = DEVELOPER_ROLE DEFAULT_WAREHOUSE = DEV_WH;
```

## Explanation of the Code

- `CREATE ROLE DEVELOPER_ROLE;`: This statement creates a new, custom role named `DEVELOPER_ROLE`. Roles are fundamental to Snowflake's RBAC model, serving as containers for privileges.
- `GRANT ROLE DEVELOPER_ROLE TO USER your_user;`: This assigns the newly created `DEVELOPER_ROLE` to a specific user (`your_user`), allowing that user to assume the privileges associated with this role.
- `GRANT USAGE ON DATABASE DEV_DB TO ROLE DEVELOPER_ROLE;`: The `USAGE` privilege on a database permits a role to access the database and its objects, but not to create new objects within it directly.
- `GRANT USAGE ON SCHEMA DEV_DB.DEV_SCHEMA TO ROLE DEVELOPER_ROLE;`: Similarly, `USAGE` on a schema allows access to objects within that schema. Without `USAGE` on the parent database and schema, even `CREATE TABLE` would fail.
- `GRANT CREATE TABLE ON SCHEMA DEV_DB.DEV_SCHEMA TO ROLE DEVELOPER_ROLE;`: This is the core statement that grants the specific `CREATE TABLE` privilege, enabling users assuming `DEVELOPER_ROLE` to create tables only within `DEV_DB.DEV_SCHEMA`.
- `REVOKE CREATE DATABASE ON ACCOUNT FROM PUBLIC;` and `REVOKE CREATE SCHEMA ON DATABASE DEV_DB FROM PUBLIC;`: These are preventative measures. By default, the `PUBLIC` role often has broader permissions. Revoking these prevents unintended global or database-wide `CREATE` capabilities from less privileged roles.
- `CREATE WAREHOUSE DEV_WH...;` and `GRANT USAGE ON WAREHOUSE DEV_WH TO ROLE DEVELOPER_ROLE;`: This creates a virtual warehouse and grants usage to the `DEVELOPER_ROLE`, providing compute resources necessary for DDL/DML operations.
- `ALTER USER your_user SET DEFAULT_ROLE = DEVELOPER_ROLE DEFAULT_WAREHOUSE = DEV_WH;`: Sets the default role and warehouse for the user, so they automatically operate under `DEVELOPER_ROLE` and use `DEV_WH` upon logging in.

## When to Use

1. **Enforcing Principle of Least Privilege**: To ensure users only have the minimum permissions necessary for their tasks. For instance, a data analyst should only have `SELECT` privileges, not `CREATE TABLE`.
    * *Scenario*: A junior data analyst needs to query existing data but should not be able to create new tables, views, or modify the database schema.
2. **Maintaining Data Governance and Schema Integrity**: To prevent accidental or unauthorized schema changes in production environments.
    * *Scenario*: In a production data warehouse, only automated ETL processes or highly privileged administrators should have `CREATE` or `ALTER` privileges on schemas and tables, ensuring data consistency.
3. **Controlling Resource Consumption**: To limit who can create resource-intensive objects like large tables or external stages, which could impact billing or performance.
    * *Scenario*: Prevent ad-hoc creation of very large tables by non-approved users that could lead to unexpected credit usage.

## When Not to Use

1. **During Initial Development or Sandbox Environments**: Overly restrictive `CREATE` privileges can hinder rapid prototyping and experimentation.
    * *Situation*: In a dedicated development or sandbox account, developers might need full `CREATE` and `DROP` capabilities to iterate quickly on new data models or pipelines.
2. **For Roles Designed for Data Engineers or Architects**: These roles inherently require the ability to create and manage data objects.
    * *Situation*: A `DATA_ENGINEER` role is explicitly designed to build and maintain data pipelines, which necessitates `CREATE TABLE`, `CREATE VIEW`, `CREATE PIPE`, etc., across designated development and staging areas.
3. **If Granular Control Isn't Possible or Practical for the Use Case**: In very small teams or simple setups, the overhead of managing fine-grained `CREATE` privileges might outweigh the benefits.
    * *Situation*: A small startup with a single data administrator who performs all data-related tasks might use higher-level roles like `SYSADMIN` that already encompass `CREATE` privileges, as the risk is centralized.

## Common Mistakes

1. **Over-granting Privileges to `PUBLIC` Role**: The `PUBLIC` role is granted to all users by default. If `CREATE` privileges are granted to `PUBLIC`, any user can create objects, leading to an uncontrolled environment.
2. **Using `ACCOUNTADMIN` for Daily Operations**: The `ACCOUNTADMIN` role has all privileges, including `CREATE` on everything. Using it for routine tasks bypasses RBAC and increases security risk.
3. **Not Leveraging Role Hierarchy**: Failing to structure roles hierarchically can lead to repetitive privilege grants and make managing permissions complex and error-prone.
4. **Forgetting `USAGE` Grants**: Users need `USAGE` privileges on parent objects (database, schema) in addition to `CREATE` privileges on the child objects (table, view) to actually perform creation.

## Expected Output

The resulting dataset from successfully restricting `CREATE` privileges isn't a table in the traditional sense, but rather a state of the system where privilege assignments are controlled. When you query grants, you would see specific roles having `CREATE` capabilities on specific objects, as intended.

Here's what you might expect to see if you query the `GRANTS_TO_ROLES` view in Snowflake's `ACCOUNT_USAGE` schema after implementing the restrictions:

| GRANT_TO_ROLE | GRANTED_ON | GRANTED_TO | PRIVILEGE | NAME |
| :------------ | :--------- | :--------- | :----------- | :------------- |
| DEVELOPER_ROLE | DATABASE | ROLE | USAGE | DEV_DB |
| DEVELOPER_ROLE | SCHEMA | ROLE | USAGE | DEV_DB.DEV_SCHEMA |
| DEVELOPER_ROLE | SCHEMA | ROLE | CREATE TABLE | DEV_DB.DEV_SCHEMA |

**Explanation of Columns:**

- `GRANT_TO_ROLE`: The role that has been granted a privilege (e.g., `DEVELOPER_ROLE`).
- `GRANTED_ON`: The type of object the privilege is granted on (e.g., `DATABASE`, `SCHEMA`).
- `GRANTED_TO`: Indicates whether the privilege was granted to a `ROLE` or `USER`.
- `PRIVILEGE`: The specific action that the role is permitted to perform (e.g., `USAGE`, `CREATE TABLE`).
- `NAME`: The fully qualified name of the object on which the privilege is granted (e.g., `DEV_DB`, `DEV_DB.DEV_SCHEMA`).

This example table shows that `DEVELOPER_ROLE` has the necessary `USAGE` permissions to navigate `DEV_DB` and `DEV_DB.DEV_SCHEMA`, and crucially, the `CREATE TABLE` privilege specifically on `DEV_DB.DEV_SCHEMA`. It implies that `DEVELOPER_ROLE` cannot create databases, schemas, or tables in other schemas, thereby demonstrating the successful restriction of `CREATE` privileges.
