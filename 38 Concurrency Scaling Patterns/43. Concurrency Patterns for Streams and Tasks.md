# Concurrency Patterns for Streams and Tasks in Snowflake

Concurrency patterns for Streams and Tasks in Snowflake refer to how these two powerful features work together to manage and process data changes efficiently and in parallel, especially in real-time or near real-time data pipelines. Streams capture row-level changes on a table, and Tasks execute SQL statements on a schedule or when triggered, often to process these captured changes. Managing their concurrency is crucial for performance, cost-efficiency, and data integrity.

**Short Explanation**

This SQL feature solves the problem of keeping data warehouses and analytical tables up-to-date with changes from source systems without constantly reloading entire datasets. Streams track specific data modifications (inserts, updates, deletes), and Tasks automate the processing of these changes. It's important in databases and analytics because it enables efficient Change Data Capture (CDC), reduces processing latency, and optimizes resource consumption. It's commonly used in ETL/ELT pipelines, data synchronization, and real-time reporting to maintain fresh data with minimal overhead.

- **What problem does this SQL feature solve?** It solves the challenge of efficiently propagating data changes from source tables to target tables (e.g., data warehouse fact or dimension tables) without full table scans or complex external orchestration, enabling robust Change Data Capture (CDC).
- **Why is it important in databases or analytics?** It's vital for building scalable, near real-time data pipelines, reducing data latency, and optimizing compute costs by processing only changed data instead of entire datasets. This ensures analytical insights are based on fresh information.
- **Where is it commonly used in real workflows?** It's extensively used in ELT workflows for incrementally loading data into data marts, synchronizing operational databases with analytical ones, and creating historical audit trails for data changes.

## Implementation Example

```sql
-- 1. Source table (assume this table is updated frequently)
CREATE OR REPLACE TABLE raw.customers (
  customer_id INT PRIMARY KEY,
  name VARCHAR(200),
  email VARCHAR(200),
  status VARCHAR(20),
  updated_at TIMESTAMP_NTZ
);

-- 2. Create a stream on the source table to capture changes
CREATE OR REPLACE STREAM raw.customers_stream ON TABLE raw.customers;

-- 3. Target table for processed customer data
CREATE OR REPLACE TABLE analytics.dim_customers (
  customer_id INT PRIMARY KEY,
  name VARCHAR(200),
  email VARCHAR(200),
  status VARCHAR(20),
  updated_at TIMESTAMP_NTZ,
  cdc_action VARCHAR(10),
  cdc_timestamp TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP()
);

-- 4. Create a task to process changes from the stream into the dimension table
CREATE OR REPLACE TASK analytics.sync_customers_task
  WAREHOUSE = ETL_WH -- Specify a dedicated warehouse for ETL
  SCHEDULE = '5 MINUTE' -- Run every 5 minutes
  WHEN SYSTEM$STREAM_HAS_DATA('raw.customers_stream') -- Only run if the stream has data
AS
  MERGE INTO analytics.dim_customers AS tgt
  USING (
    SELECT
      customer_id,
      name,
      email,
      status,
      updated_at,
      METADATA$ACTION AS action,
      METADATA$ISUPDATE AS is_update
    FROM raw.customers_stream
  ) AS src ON tgt.customer_id = src.customer_id
  WHEN MATCHED AND src.action = 'DELETE' THEN
    DELETE
  WHEN MATCHED AND src.action = 'INSERT' AND src.is_update = TRUE THEN
    UPDATE SET
      tgt.name = src.name,
      tgt.email = src.email,
      tgt.status = src.status,
      tgt.updated_at = src.updated_at,
      tgt.cdc_action = 'UPDATE',
      tgt.cdc_timestamp = CURRENT_TIMESTAMP()
  WHEN NOT MATCHED AND src.action = 'INSERT' THEN
    INSERT (customer_id, name, email, status, updated_at, cdc_action, cdc_timestamp)
    VALUES (src.customer_id, src.name, src.email, src.status, src.updated_at, src.action, CURRENT_TIMESTAMP());

-- 5. Resume the task to activate it
ALTER TASK analytics.sync_customers_task RESUME;
```

## Explanation of the Code

This example sets up a CDC pipeline using a Stream and a Task to maintain a `dim_customers` table based on changes in a `raw.customers` table.

- **`CREATE OR REPLACE TABLE raw.customers (...)`**: This creates a `raw.customers` table, representing the source data that will be updated.
- **`CREATE OR REPLACE STREAM raw.customers_stream ON TABLE raw.customers`**: This creates a Stream named `customers_stream` on the `raw.customers` table.
    - **What it does**: A Stream tracks data manipulation language (DML) changes (inserts, updates, and deletes) made to its source table. It does not store a copy of the data but rather an offset, or pointer, to where changes were last consumed.
    - **How it changes the result set**: When queried, a Stream returns only the rows that have changed since the last time it was queried (or the last successful Task run that consumed it). Each row includes metadata columns like `METADATA$ACTION` (INSERT/DELETE) and `METADATA$ISUPDATE` (TRUE/FALSE for updates, which appear as a DELETE and INSERT pair).
- **`CREATE OR REPLACE TABLE analytics.dim_customers (...)`**: This creates the target `dim_customers` table, which will store the processed and consolidated customer data.
- **`CREATE OR REPLACE TASK analytics.sync_customers_task... AS...`**: This defines a Task named `sync_customers_task`.
    - **`WAREHOUSE = ETL_WH`**: Specifies the virtual warehouse that will execute the SQL statement within the Task.
        - **What it does**: A virtual warehouse provides the compute resources (CPU, memory) for running SQL queries.
        - **How it changes the result set**: It doesn't directly change the result set but influences the performance and cost of the Task execution.
    - **`SCHEDULE = '5 MINUTE'`**: Defines how often the Task should attempt to run.
        - **What it does**: Sets a fixed interval for the Task's execution.
        - **How it changes the result set**: Controls the frequency of data synchronization.
    - **`WHEN SYSTEM$STREAM_HAS_DATA('raw.customers_stream')`**: This condition ensures the Task only runs if the specified Stream contains new, unconsumed data.
        - **What it does**: Prevents the Task from running unnecessarily and consuming warehouse credits when there's no work to be done.[aSZU]
        - **How it changes the result set**: It optimizes resource usage but does not directly modify the data content.
    - **`MERGE INTO analytics.dim_customers AS tgt USING (...)`**: This is the core DML statement executed by the Task. It merges data from the Stream into the target dimension table.
        - **What it does**: The `MERGE` statement synchronizes data by performing `INSERT`, `UPDATE`, or `DELETE` operations on the target table based on conditions met with the source data (the Stream).
        - **How it changes the result set**:
            - **`WHEN MATCHED AND src.action = 'DELETE' THEN DELETE`**: If a row in the Stream has `METADATA$ACTION = 'DELETE'`, the corresponding row in the target table is deleted.
            - **`WHEN MATCHED AND src.action = 'INSERT' AND src.is_update = TRUE THEN UPDATE SET...`**: If a row in the Stream represents an update (appears as an INSERT with `METADATA$ISUPDATE = TRUE`), the corresponding row in the target table is updated.
            - **`WHEN NOT MATCHED AND src.action = 'INSERT' THEN INSERT (...) VALUES (...)`**: If a row in the Stream is a new insert and doesn't exist in the target table, it's inserted.
- **`ALTER TASK analytics.sync_customers_task RESUME`**: Tasks are created in a suspended state and must be explicitly resumed to start running.

## When to Use

1. **Change Data Capture (CDC)**: When you need to incrementally load data changes from an operational database into a data warehouse for analytics.
    * **Scenario**: A CRM system continuously updates customer records. You use a Stream on the source customer table and a Task to periodically merge these changes into a `DIM_CUSTOMER` table in your Snowflake data warehouse.
2. **Near Real-time Data Synchronization**: To keep disparate datasets synchronized or to provide fresh data for dashboards and reports.
    * **Scenario**: An e-commerce platform generates sales orders. A Stream captures new orders, and a Task immediately processes them into a sales aggregation table, providing up-to-the-minute sales figures.
3. **Building Task Graphs/Workflows**: For multi-step data processing where one step depends on the completion of another.
    * **Scenario**: After new raw data arrives and a Stream captures changes, a "stage" Task loads it into a staging table. A subsequent "transform" Task cleans and enriches this data, and a final "load" Task merges it into the final analytics table. These Tasks are chained as a task graph.

## When Not to Use

1. **Full Data Loads**: For initial data loads of an entire table where historical changes aren't relevant, or for very small tables where a full reload is trivial.
    * **Reason**: Streams track *changes*, and Tasks process them. For an initial full load, a simple `INSERT INTO... SELECT * FROM...` is more straightforward and efficient.
2. **Infrequently Updated Tables with Complex Logic**: If a source table is updated very rarely and the transformation logic is extremely complex or requires extensive lookups not easily done in a `MERGE` statement.
    * **Reason**: The overhead of maintaining a Stream and running a Task every interval (even if it's `WHEN SYSTEM$STREAM_HAS_DATA`) might outweigh the benefits. Simpler scheduled batch jobs might be more appropriate.
3. **When Dynamic Tables Suffice**: For simpler incremental pipelines where you only need to transform data and have it continuously updated, Snowflake's Dynamic Tables might be a more simplified solution.
    * **Reason**: Dynamic Tables abstract away the need for explicit Streams and Tasks for many common CDC patterns, offering a more declarative approach.

## Common Mistakes

1. **Not Resuming Tasks**: Tasks are created in a suspended state and will not run until `ALTER TASK... RESUME;` is executed.
2. **Ignoring Stream Staleness**: If a Stream is not consumed within the source table's data retention period (default 14 days), it goes stale. A stale Stream can no longer provide change data and must be recreated, leading to data loss.
3. **Lack of `WHEN SYSTEM$STREAM_HAS_DATA()`**: Running a Task on a schedule without this condition will cause it to execute even when no new data is available in the Stream, wasting warehouse credits.[aSZU]
4. **Improper Transaction Control**: When consuming Streams outside of Tasks (e.g., in stored procedures), failing to wrap the consumption logic in explicit `BEGIN...COMMIT` blocks can lead to the Stream offset advancing even if the subsequent DML fails, resulting in lost changes.
5. **Incorrect `METADATA$ACTION` Handling for Updates**: Updates in Streams appear as a `DELETE` and an `INSERT` pair. Incorrectly handling this in `MERGE` statements (e.g., only checking for 'INSERT' and not considering `METADATA$ISUPDATE = TRUE`) can lead to missed updates or duplicate data.

## Expected Output

The resulting dataset in `analytics.dim_customers` should reflect the current state of the `raw.customers` table, with additional metadata indicating the last CDC action and timestamp. Each row will represent a unique customer, kept up-to-date by the Stream and Task.

**Example Table: `analytics.dim_customers`**

| CUSTOMER_ID | NAME | EMAIL | STATUS | UPDATED_AT | CDC_ACTION | CDC_TIMESTAMP |
| :---------- | :--------- | :------------------- | :------- | :------------------ | :--------- | :-------------------------- |
| 101 | Alice | alice@example.com | Active | 2026-03-14 10:00:00 | INSERT | 2026-03-14 10:05:12.345678 |
| 102 | Bob | bob@example.com | Inactive | 2026-03-14 11:30:00 | UPDATE | 2026-03-14 11:35:45.987654 |
| 103 | Charlie | charlie@example.com | Active | 2026-03-14 12:45:00 | INSERT | 2026-03-14 12:50:01.123456 |

**Explanation of Columns:**

- **`CUSTOMER_ID`**: Unique identifier for the customer.
- **`NAME`**, **`EMAIL`**, **`STATUS`**, **`UPDATED_AT`**: Data fields from the source `raw.customers` table, reflecting their most current state.
- **`CDC_ACTION`**: Indicates the last type of change that occurred for this row ('INSERT' for new records, 'UPDATE' for modified records). A 'DELETE' action would remove the row from this target table.
-   **`CDC_TIMESTAMP`**: The timestamp when the `sync_customers_task` last processed a change for this specific customer record.
