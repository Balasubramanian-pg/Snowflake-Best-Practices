# Comparing Suspend vs. Suspend Immediate Data States in Snowflake

**Description:**
This topic explains the difference between the SUSPEND and SUSPEND IMMEDIATE data states in Snowflake, their implications on data usage, and best practices for managing these states.

## Short Explanation
**Overview:** Snowflake's SUSPEND and SUSPEND IMMEDIATE data states control data usage and billing.

**Problem Solved:** Managing data states to optimize costs and data accessibility.

**Importance:** Understanding data states is crucial for cost-effective data warehousing.

**Use Cases:**
- Cost optimization
- Data governance

### Typical Scenario
A company wants to manage its Snowflake data usage and costs by suspending unused databases or schemas.

### Core Snowflake SQL Objects Used
`ALTER DATABASE`, `ALTER SCHEMA`

### Code Source Execution
```sql
ALTER DATABASE my_database SET DATA STATE = 'SUSPEND';
```

### Example Execution Logic Step-by-Step
The SUSPEND data state allows data to be reactivated quickly, while SUSPEND IMMEDIATE makes data inaccessible until reactivated.

### Implementation Example
To suspend a database:
ALTER DATABASE my_database SET DATA STATE = 'SUSPEND';
To suspend a schema:
ALTER SCHEMA my_schema SET DATA STATE = 'SUSPEND IMMEDIATE';

### Explanation of the Code
- Step 1: Alter the database or schema to set its data state to SUSPEND or SUSPEND IMMEDIATE.
- Step 2: Verify the data state change using SHOW DATABASES or SHOW SCHEMAS.

### When to Use
- Use SUSPEND for data that needs to be reactivated quickly
- Use SUSPEND IMMEDIATE for data that is no longer needed

### When Not to Use
- Avoid using SUSPEND IMMEDIATE for data that may be needed soon
- Do not use SUSPEND for large datasets

### Common Mistakes
- Forgetting to reactivate suspended data
- Misunderstanding the implications of SUSPEND IMMEDIATE on data accessibility
