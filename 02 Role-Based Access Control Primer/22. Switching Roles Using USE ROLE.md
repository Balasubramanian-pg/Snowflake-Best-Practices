# 22. Switching Roles Using USE ROLE

The `USE ROLE` command dynamically changes the active security context for a user's current session, determining which database objects they can access and what actions they can perform. By switching roles, users adhere to the principle of least privilege, activating elevated or specific permissions only when necessary for a given task.

# Key Points

* Role-Based Access Control (RBAC) relies on `USE ROLE` to enforce security boundaries by ensuring users only operate with the privileges granted to their currently active role.
* A user can only switch to a role that has been explicitly granted to them; attempting to use an unauthorized role will result in an access control error.
* Changing the active role immediately updates the session's context, which instantly alters the visibility and accessibility of databases, schemas, and tables.
* While `USE ROLE` sets the primary active role, modern data platforms often support secondary roles (e.g., `USE SECONDARY ROLES ALL`) to aggregate privileges without requiring constant context switching.
* > [!IMPORTANT] Always revert to a lower-privileged role after completing administrative or sensitive tasks to minimize the risk of accidental data modification or deletion.

# Implementation

Provide a realistic Snowflake SQL example demonstrating the concept.

```sql
-- Check the currently active role
SELECT CURRENT_ROLE();

-- Switch to a highly privileged role to perform administrative tasks
USE ROLE SYSADMIN;

-- Perform an action requiring SYSADMIN privileges
CREATE OR REPLACE WAREHOUSE data_science_wh 
  WITH WAREHOUSE_SIZE = 'LARGE' 
  AUTO_SUSPEND = 300;

-- Revert to a standard functional role for daily querying
USE ROLE DATA_ANALYST;
```

# What does the Query do?

Explain the SQL step by step:

Step 1: The `SELECT CURRENT_ROLE();` function retrieves and displays the active role for the current session before any context changes are made.

Step 2: The `USE ROLE SYSADMIN;` command elevates the session's security context, granting the user the administrative privileges required for infrastructure management.

Step 3: The `CREATE OR REPLACE WAREHOUSE` statement provisions a new compute resource, an action that succeeds specifically because the newly activated `SYSADMIN` role possesses the necessary creation permissions.

Step 4: The `USE ROLE DATA_ANALYST;` command intentionally downgrades the session's privileges back to a standard operational role, safeguarding the environment against accidental administrative changes during subsequent queries.

# When Should We Use This? 

* When a user holds multiple roles (e.g., developer and administrator) and needs to temporarily elevate privileges to deploy code or provision resources before returning to standard read/write tasks.
* Within automated deployment scripts or CI/CD pipelines where specific execution steps require distinct security contexts (e.g., using `SECURITYADMIN` to grant access, then switching to `SYSADMIN` to create tables).
