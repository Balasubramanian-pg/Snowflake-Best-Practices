# Prefixing vs Suffixing Environment Identifiers in Snowflake

**Description:**
This topic discusses the best practices for prefixing or suffixing environment identifiers in Snowflake, such as development, testing, and production environments. It aims to provide guidance on how to organize and identify different environments in Snowflake. Proper environment identification is crucial for maintaining data integrity and preventing accidental changes to critical data.

## Short Explanation
**Overview:** Prefixing or suffixing environment identifiers helps to clearly distinguish between different environments in Snowflake.

**Problem Solved:** It solves the problem of confusing or overlapping environment names, reducing the risk of data corruption or unauthorized changes.

**Importance:** It matters in analytics and warehousing as it ensures data integrity, security, and compliance with organizational policies.

**Use Cases:**
- Development, testing, and production environments
- Data warehousing and business intelligence applications

### Typical Scenario
A company has multiple Snowflake accounts for development, testing, and production environments. They want to ensure that data in each environment is clearly identified and separated.

### Core Snowflake SQL Objects Used
`CREATE DATABASE`, `CREATE SCHEMA`

### Code Source Execution
```sql
CREATE DATABASE DEV_MY_DATABASE;
```

### Example Execution Logic Step-by-Step
In this example, 'DEV_' is a prefix indicating that this database belongs to the development environment. Similarly, suffixing can be done by adding '_DEV' to the end of the database name.

### Implementation Example
To implement prefixing or suffixing, you can create a naming convention that includes the environment identifier. For example:

CREATE DATABASE DEV_MY_DATABASE;
CREATE SCHEMA DEV_MY_SCHEMA;

or

CREATE DATABASE MY_DATABASE_DEV;
CREATE SCHEMA MY_SCHEMA_DEV;

- Step 2: Create a schema with a prefixed environment identifier

### When to Use
- When you have multiple environments with similar database and schema names
- When you want to clearly distinguish between environments

### When Not to Use
- When you have a single environment with unique database and schema names
- When you have a simple development setup with no need for environment separation

### Common Mistakes
- Using similar environment identifiers that can be easily confused
- Not consistently applying the prefix or suffix across all environments
