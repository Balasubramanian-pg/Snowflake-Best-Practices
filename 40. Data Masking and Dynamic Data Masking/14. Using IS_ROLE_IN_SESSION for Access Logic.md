# Combining Masking with Secure Data Sharing in Snowflake

This concept introduces how to leverage Snowflake's dynamic data masking capabilities in conjunction with Secure Data Sharing to provide controlled access to sensitive data across different accounts, ensuring data privacy and compliance without creating multiple copies of the data. It exists to enable organizations to share data securely and efficiently, allowing consumers to query the data while only seeing what they are authorized to see, based on their roles and policies.

**Short Explanation**

Combining masking with secure data sharing means you can share a table containing sensitive information with other Snowflake accounts, but the data consumers will only see a masked (hidden or altered) version of that sensitive data. This happens automatically based on policies applied to the table and the role of the user accessing it.

- **What problem does this SQL feature solve?** It solves the problem of sharing sensitive data with external parties or different departments while maintaining strict data privacy and compliance requirements (e.g., GDPR, HIPAA). It avoids data duplication and complex ETL processes for obfuscation.
- **Why is it important in databases or analytics?** It's crucial for enabling data collaboration and external analytics without compromising data security. It simplifies data governance and ensures consistent application of privacy rules.
- **Where is it commonly used in real workflows?** Financial institutions sharing customer data with partners, healthcare providers sharing patient data for research, or enterprises sharing internal data across business units with varying access needs.

## Implementation Example

```sql
-- Provider Account:
-- 1. Create a masking policy
CREATE MASKING POLICY email_mask_policy AS (val string) RETURNS string ->
  CASE
    WHEN CURRENT_ROLE() IN ('ANALYST', 'SALES_MANAGER') THEN val
    WHEN CURRENT_ROLE() = 'DATA_SCIENTIST' THEN '************'
    ELSE '##########' -- Default mask for others
  END;

-- 2. Create a table with sensitive data
CREATE TABLE customer_data (
    customer_id INT,
    customer_name VARCHAR,
    customer_email VARCHAR,
    customer_address VARCHAR
);

-- 3. Apply the masking policy to the sensitive column
ALTER TABLE customer_data MODIFY COLUMN customer_email SET MASKING POLICY email_mask_policy;

-- 4. Insert some sample data
INSERT INTO customer_data VALUES
    (1, 'Alice Smith', 'alice.smith@example.com', '123 Main St'),
    (2, 'Bob Johnson', 'bob.j@example.com', '456 Oak Ave'),
    (3, 'Charlie Brown', 'charlie.b@example.com', '789 Pine Rd');

-- 5. Create a secure share and add the table
CREATE SHARE my_secure_share;
GRANT USAGE ON DATABASE my_db TO SHARE my_secure_share;
GRANT USAGE ON SCHEMA my_db.public TO SHARE my_secure_share;
GRANT SELECT ON TABLE my_db.public.customer_data TO SHARE my_secure_share;
ALTER SHARE my_secure_share ADD ACCOUNTS = <consumer_account_locator>;

-- Consumer Account:
-- 1. Create a database from the share
CREATE DATABASE shared_customer_db FROM SHARE <provider_account_locator>.my_secure_share;

-- 2. Create roles for testing masking
CREATE ROLE ANALYST;
CREATE ROLE DATA_SCIENTIST;
GRANT ROLE ANALYST TO USER <your_user>;
GRANT ROLE DATA_SCIENTIST TO USER <your_user>;

-- 3. Query the shared table as different roles
-- Use role ANALYST;
-- SELECT customer_id, customer_email FROM shared_customer_db.public.customer_data;

-- Use role DATA_SCIENTIST;
-- SELECT customer_id, customer_email FROM shared_customer_db.public.customer_data;

-- Use role PUBLIC; (or any other role not explicitly granted in the policy)
-- SELECT customer_id, customer_email FROM shared_customer_db.public.customer_data;
```

## Explanation of the Code

This example demonstrates the end-to-end process from the provider setting up the share to the consumer accessing the masked data.

- `CREATE MASKING POLICY email_mask_policy AS (val string) RETURNS string -> ...`: This statement defines a dynamic data masking policy.
    - **What it does**: It specifies a SQL expression that Snowflake will use to transform the `val` (the original column value) based on the `CURRENT_ROLE()` of the user querying the data.
    - **How it changes the result set**: When this policy is applied to a column, every query on that column will have its output altered according to the `CASE` statement. Users with 'ANALYST' or 'SALES_MANAGER' roles see the original email. Users with 'DATA_SCIENTIST' see asterisks (`************`). All other roles see hashes (`##########`).
- `CREATE TABLE customer_data (...)`: This creates a standard table to hold customer information, including a sensitive `customer_email` column.
- `ALTER TABLE customer_data MODIFY COLUMN customer_email SET MASKING POLICY email_mask_policy;`: This command applies the previously defined `email_mask_policy` to the `customer_email` column in the `customer_data` table.
    - **What it does**: It links the masking policy to the specific column, ensuring that the policy is automatically enforced whenever the `customer_email` column is queried.
    - **How it changes the result set**: From this point onwards, any query on `customer_data.customer_email` will trigger the masking policy, transforming the output based on the querying user's role.
- `INSERT INTO customer_data VALUES (...)`: Populates the table with sample data for demonstration.
- `CREATE SHARE my_secure_share;`: This initializes a secure data share object.
- `GRANT USAGE ON DATABASE ... TO SHARE ...;`, `GRANT USAGE ON SCHEMA ... TO SHARE ...;`, `GRANT SELECT ON TABLE ... TO SHARE ...;`: These `GRANT` statements define which database objects (database, schema, table) from the provider account are included in `my_secure_share`.
    - **What it does**: They make the specified objects available for consumption through the secure share. Importantly, the masking policy applied to `customer_email` travels with the shared table.
    - **How it changes the result set**: When a consumer queries the `customer_data` table via the share, the masking policy is still active and enforced based on the consumer's role within their own account.
- `ALTER SHARE my_secure_share ADD ACCOUNTS = <consumer_account_locator>;`: This command adds a specific consumer account to the share, allowing them to access the shared data.
- **Consumer Account Steps**: These steps illustrate how a consumer would interact with the shared data.
    - `CREATE DATABASE shared_customer_db FROM SHARE ...;`: Creates a local database in the consumer account that points to the shared data.
    - `CREATE ROLE ...;` and `GRANT ROLE ... TO USER ...;`: Demonstrate creating and assigning roles within the consumer account that correspond to the roles referenced in the provider's masking policy.
    - `SELECT ... FROM shared_customer_db.public.customer_data;`: These are the actual queries from the consumer. The output for `customer_email` will vary depending on which role (`ANALYST`, `DATA_SCIENTIST`, or `PUBLIC`) is active when the query is run, showcasing the dynamic masking in action.

## When to Use

1.  **Sharing PII (Personally Identifiable Information) with partners:** A healthcare provider wants to share patient demographic data with a research institution, but needs to mask sensitive identifiers like email addresses or phone numbers for all researchers, while allowing certain high-privileged roles to see the full data for specific, approved use cases.
2.  **Cross-departmental data analysis with varying access levels:** An e-commerce company needs to share customer purchase history with both marketing and data science teams. Marketing might only need generalized customer locations, while data science needs full addresses for specific models, and sensitive PII like customer email should be masked for everyone except specific compliance officers.
3.  **Monetizing data securely through data marketplaces:** A data vendor offers a dataset containing user activity. To appeal to a broad audience while respecting privacy, they use masking policies to offer different tiers of data access – a basic tier with heavily masked PII, and a premium tier with less stringent masking for trusted partners.

## When Not to Use

1.  **When full, unmasked data is always required by all consumers:** If all consumers of the shared data genuinely need access to the original, unmasked values of all columns, then adding masking policies introduces unnecessary overhead and complexity without providing a benefit.
2.  **For extremely high-performance, low-latency applications on masked data:** While dynamic masking is efficient, it does involve runtime evaluation of the masking policy. For scenarios where every millisecond of query performance is critical and masking logic is complex, pre-masked copies (if security constraints allow) might offer slightly better performance, though this often comes with increased data management overhead.
3.  **When data governance is static and simple:** If the access rules are very simple (e.g., either full access or no access to an entire table) and do not depend on user roles or context, simpler access control mechanisms like `GRANT/REVOKE` permissions on tables might be sufficient without the added complexity of masking policies.

## Common Mistakes

1.  **Incorrect Role Configuration:** Developers often define masking policies that rely on specific roles (e.g., `CURRENT_ROLE()`), but then fail to ensure those roles exist and are correctly granted to users in the consumer accounts. This can lead to unexpected masking (e.g., everyone seeing the default masked value) or even errors.
2.  **Overly Complex Masking Logic:** Creating masking policies with very intricate `CASE` statements or subqueries can impact query performance, especially on large datasets. Policies should be kept as simple and efficient as possible.
3.  **Masking Policy Not Applied to Shared Data:** Forgetting to apply the masking policy to the sensitive column *before* sharing the table can result in the full, unmasked data being inadvertently shared. The policy must be active on the provider's table for it to take effect on the shared data.
4.  **Assuming Obfuscation is Encryption:** Masking policies obfuscate data (hide or transform it) but do not encrypt it at rest or in transit in a way that requires a decryption key. While effective for data privacy, it's not a substitute for proper encryption where required by compliance standards.

## Expected Output

The resulting dataset will contain all columns from the `customer_data` table. However, the `customer_email` column's values will be transformed based on the `CURRENT_ROLE()` of the user executing the `SELECT` query in the consumer account.

Here's an example of the expected output when different roles query the shared data:

**Querying as ANALYST (or SALES_MANAGER) role:**

| CUSTOMER_ID | CUSTOMER_NAME | CUSTOMER_EMAIL            | CUSTOMER_ADDRESS |
|-------------|---------------|---------------------------|------------------|
| 1           | Alice Smith   | alice.smith@example.com   | 123 Main St      |
| 2           | Bob Johnson   | bob.j@example.com         | 456 Oak Ave      |
| 3           | Charlie Brown | charlie.b@example.com     | 789 Pine Rd      |

**Querying as DATA_SCIENTIST role:**

| CUSTOMER_ID | CUSTOMER_NAME | CUSTOMER_EMAIL | CUSTOMER_ADDRESS |
|-------------|---------------|----------------|------------------|
| 1           | Alice Smith   | ************   | 123 Main St      |
| 2           | Bob Johnson   | ************   | 456 Oak Ave      |
| 3           | Charlie Brown | ************   | 789 Pine Rd      |

**Querying as PUBLIC (or any other role not specified in policy):**

| CUSTOMER_ID | CUSTOMER_NAME | CUSTOMER_EMAIL | CUSTOMER_ADDRESS |
|-------------|---------------|----------------|------------------|
| 1           | Alice Smith   | ##########     | 123 Main St      |
| 2           | Bob Johnson   | ##########     | 456 Oak Ave      |
| 3           | Charlie Brown | ##########     | 789 Pine Rd      |

- **CUSTOMER_ID**: The unique identifier for the customer, shown as original integer value.
- **CUSTOMER_NAME**: The customer's full name, shown as original string value.
- **CUSTOMER_EMAIL**: The customer's email address, dynamically masked based on the querying user's role.
- **CUSTOMER_ADDRESS**: The customer's physical address, shown as original string value.
