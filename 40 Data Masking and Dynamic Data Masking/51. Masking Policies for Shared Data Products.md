# Masking Policies for Shared Data Products in Snowflake

Masking policies in Snowflake provide a robust mechanism for data governance, allowing organizations to protect sensitive data by dynamically masking it based on a user's role or other conditions. When applied to shared data products, these policies ensure that consumers of the shared data only see the data they are authorized to view, without requiring separate data copies or complex ETL processes. It exists to enable secure data sharing while maintaining compliance and privacy requirements.

**Short Explanation**

Masking policies dynamically hide or transform sensitive data within a column based on who is querying it. This feature solves the problem of securely sharing data with different levels of access, ensuring that data consumers only see what's relevant and permitted for their roles. It's important for compliance (like GDPR, HIPAA), and is commonly used in data sharing scenarios where a data provider needs to share a single dataset with multiple consumers, each with varying data access requirements.

- What problem does this SQL feature solve? It solves the problem of controlling access to sensitive data within shared datasets without creating multiple versions of the data.
- Why is it important in databases or analytics? It's crucial for data governance, regulatory compliance, and enabling secure data collaboration.
- Where is it commonly used in real workflows? Data sharing between departments, external partners, or customers where different entities require different views of the same underlying sensitive data.

## Implementation Example

```sql
-- Create a table with sensitive data
CREATE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    customer_email VARCHAR,
    total_amount DECIMAL(10, 2),
    credit_card_number VARCHAR -- Sensitive column
);

-- Insert some sample data
INSERT INTO sales_data (order_id, customer_name, customer_email, total_amount, credit_card_number) VALUES
(101, 'Alice Wonderland', 'alice@example.com', 150.75, '1234-5678-9012-3456'),
(102, 'Bob The Builder', 'bob@example.com', 200.00, '9876-5432-1098-7654');

-- Create a masking policy
CREATE MASKING POLICY card_number_mask AS (val VARCHAR) RETURNS VARCHAR ->
    CASE
        WHEN CURRENT_ROLE() IN ('SECURITY_ADMIN', 'ANALYST_FULL_ACCESS') THEN val
        WHEN CURRENT_ROLE() = 'ANALYST_LIMITED_ACCESS' THEN 'XXXX-XXXX-XXXX-' || RIGHT(val, 4)
        ELSE '***********'
    END;

-- Apply the masking policy to the sensitive column
ALTER TABLE sales_data MODIFY COLUMN credit_card_number SET MASKING POLICY card_number_mask;

-- Grant usage on the table to a share
CREATE SHARE my_secure_share;
GRANT USAGE ON DATABASE my_db TO SHARE my_secure_share;
GRANT USAGE ON SCHEMA my_schema TO SHARE my_secure_share;
GRANT SELECT ON TABLE sales_data TO SHARE my_secure_share;

-- Grant roles to consumers (in reality, these roles would exist in consumer accounts)
-- Assuming CONSUMER_ANALYST is a role in the consumer account.
-- The masking policy logic uses CURRENT_ROLE() which executes in the context of the querying user.
-- So, consumer roles like CONSUMER_ANALYST might need mapping or specific logic in the policy.
-- For this example, we demonstrate the policy's logic within the provider account roles.
```

## Explanation of the Code

- **`CREATE TABLE sales_data (...)`**: This defines a simple table named `sales_data` to hold sales transaction information, including a `credit_card_number` column which we designate as sensitive.
- **`CREATE MASKING POLICY card_number_mask AS (val VARCHAR) RETURNS VARCHAR -> ...`**: This statement creates the actual masking policy.
    - `AS (val VARCHAR) RETURNS VARCHAR`: Specifies that the policy takes a `VARCHAR` input (the column value) and returns a `VARCHAR` (the masked value).
    - `CASE WHEN CURRENT_ROLE() IN ('SECURITY_ADMIN', 'ANALYST_FULL_ACCESS') THEN val`: If the user querying the data has the `SECURITY_ADMIN` or `ANALYST_FULL_ACCESS` role, they will see the original, unmasked `credit_card_number`.
    - `WHEN CURRENT_ROLE() = 'ANALYST_LIMITED_ACCESS' THEN 'XXXX-XXXX-XXXX-' || RIGHT(val, 4)`: Users with the `ANALYST_LIMITED_ACCESS` role will see the credit card number partially masked, revealing only the last four digits.
    - `ELSE '***********'`: For all other roles, the credit card number will be completely masked with asterisks.
- **`ALTER TABLE sales_data MODIFY COLUMN credit_card_number SET MASKING POLICY card_number_mask;`**: This command applies the `card_number_mask` policy to the `credit_card_number` column in the `sales_data` table. From this point forward, any query on this column will activate the policy.
- **`CREATE SHARE my_secure_share;`**: Initiates the creation of a secure share object.
- **`GRANT USAGE ON DATABASE my_db TO SHARE my_secure_share;`**, **`GRANT USAGE ON SCHEMA my_schema TO SHARE my_secure_share;`**, **`GRANT SELECT ON TABLE sales_data TO SHARE my_secure_share;`**: These statements grant the necessary permissions for the `sales_data` table to be included in the share. When data is shared this way, the masking policy travels with the shared object. The policy will be evaluated in the consumer's account context, using the consumer's roles to determine the masking behavior.

## When to Use

1.  **Regulatory Compliance**: When data contains personally identifiable information (PII) or protected health information (PHI) that needs to be restricted based on compliance requirements (e.g., GDPR, HIPAA).
    *   *Example Scenario*: A healthcare provider shares patient data with a research institution. The research institution's analysts need to see patient demographics, but PHI like patient names or medical record numbers must be fully masked unless they have specific research clearance roles.
2.  **Internal Data Sharing with Varying Access Levels**: Within a large organization, different departments or teams require access to the same dataset but with different levels of sensitivity.
    *   *Example Scenario*: A sales performance dashboard needs to show customer contact information to sales managers but only masked email addresses to regional analysts, and completely hide credit card numbers from everyone except a finance compliance team.
3.  **Secure Data Monetization**: When providing data products to external customers or partners where you need to offer tiered data access based on subscription levels or contractual agreements.
    *   *Example Scenario*: A financial data provider offers a dataset of transactional information. Basic subscribers might see aggregated or fully masked account numbers, while premium subscribers with higher security vetting can access partially masked account numbers for specific analysis.

## When Not to Use

1.  **Simple Row-Level Security (RLS)**: If your primary goal is to filter entire rows based on user access (e.g., a user can only see data for their region), masking policies are not the correct tool. RLS (using row access policies in Snowflake) is more appropriate.
    *   *Inefficient*: Applying a masking policy to a column when you actually want to hide the entire record if a condition isn't met.
2.  **Performance-Critical Large-Scale Aggregations**: While Snowflake masking policies are optimized, complex masking logic on columns heavily used in `GROUP BY` or `JOIN` clauses on extremely large tables might introduce minor performance overhead.
    *   *Inefficient*: Masking a column that is a foreign key used in millions of joins, especially if the masking logic is intricate.
3.  **Static Data Anonymization for Exports**: If the goal is a one-time anonymization of data for a static export (e.g., a CSV file for a data science project that doesn't need dynamic access control), performing a `CREATE TABLE AS SELECT` with explicit masking functions might be simpler and yield a static, anonymized dataset without the overhead of dynamic policies.
    *   *Inefficient*: Setting up a masking policy for a temporary or one-off anonymization task where the dynamic nature isn't required.

## Common Mistakes

1.  **Forgetting to Apply the Policy**: Creating a masking policy is only half the battle; it must be explicitly applied to the target column(s) using `ALTER TABLE ... MODIFY COLUMN ... SET MASKING POLICY ...`. Many developers create the policy but then wonder why it's not working.
2.  **Incorrect Role Evaluation**: Masking policies evaluate `CURRENT_ROLE()` in the context of the *querying user*. When sharing data, this means the roles refer to those in the *consumer's account*, not the provider's. Developers often create policies assuming provider-side roles will dictate masking in the consumer account.
3.  **Data Type Mismatch in Policy Logic**: The `RETURNS` data type of the masking policy must match the data type of the column it's applied to, or be implicitly castable. Returning a string when the column expects a number will lead to errors.
4.  **Overly Complex Policy Logic**: While powerful, overly complex `CASE` statements or user-defined functions (UDFs) within a masking policy can be difficult to manage, debug, and might introduce performance penalties.
5.  **Not Testing All Role Scenarios**: Developers might test with one or two roles but forget to test how the policy behaves for other roles, especially edge cases or default unprivileged users.

## Expected Output

When a user queries the `sales_data` table, the `credit_card_number` column will appear differently based on their active role.

For a user with the `SECURITY_ADMIN` or `ANALYST_FULL_ACCESS` role:

| ORDER_ID | CUSTOMER_NAME    | CUSTOMER_EMAIL    | TOTAL_AMOUNT | CREDIT_CARD_NUMBER     |
| -------- | ---------------- | ----------------- | ------------ | ---------------------- |
| 101      | Alice Wonderland | alice@example.com | 150.75       | 1234-5678-9012-3456    |
| 102      | Bob The Builder  | bob@example.com   | 200.00       | 9876-5432-1098-7654    |

For a user with the `ANALYST_LIMITED_ACCESS` role:

| ORDER_ID | CUSTOMER_NAME    | CUSTOMER_EMAIL    | TOTAL_AMOUNT | CREDIT_CARD_NUMBER     |
| -------- | ---------------- | ----------------- | ------------ | ---------------------- |
| 101      | Alice Wonderland | alice@example.com | 150.75       | XXXX-XXXX-XXXX-3456    |
| 102      | Bob The Builder  | bob@example.com   | 200.00       | XXXX-XXXX-XXXX-7654    |

For any other role (e.g., a default consumer role without specific access):

| ORDER_ID | CUSTOMER_NAME    | CUSTOMER_EMAIL    | TOTAL_AMOUNT | CREDIT_CARD_NUMBER     |
| -------- | ---------------- | ----------------- | ------------ | ---------------------- |
| 101      | Alice Wonderland | alice@example.com | 150.75       | ***********            |
| 102      | Bob The Builder  | bob@example.com   | 200.00       | ***********            |

The columns are:
- `ORDER_ID`: Unique identifier for each sales order.
- `CUSTOMER_NAME`: The name of the customer.
- `CUSTOMER_EMAIL`: The email address of the customer.
- `TOTAL_AMOUNT`: The total monetary value of the order.
- `CREDIT_CARD_NUMBER`: The credit card number used for the transaction, dynamically masked based on the querying user's role.
