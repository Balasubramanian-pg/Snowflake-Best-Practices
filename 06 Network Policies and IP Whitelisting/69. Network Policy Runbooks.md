# Network Policy Runbooks in Snowflake

Network Policies in Snowflake are a crucial security feature that allows administrators to control network access to their Snowflake account. They define a set of allowed or blocked IP addresses, ensuring that connections to Snowflake can only originate from trusted locations. "Runbooks" in this context would refer to documented procedures for creating, managing, and troubleshooting these network policies, ensuring consistent and secure implementation.

**Short Explanation**

Network Policies in Snowflake act as a digital gatekeeper, restricting inbound connections to your Snowflake environment based on the source IP address. They are essential for enhancing security by preventing unauthorized access attempts and complying with data governance regulations. You'll typically find them used to whitelist corporate network ranges, VPNs, or specific application servers that need to connect to Snowflake.

- **What problem does this SQL feature solve?** This feature solves the problem of uncontrolled network access, preventing malicious actors or unauthorized users from connecting to your Snowflake account from untrusted locations.
- **Why is it important in databases or analytics?** It's vital for data security, compliance (e.g., GDPR, HIPAA), and protecting sensitive data by limiting the attack surface, especially in cloud environments where Snowflake endpoints can be publicly accessible.
- **Where is it commonly used in real workflows?** Commonly used to enforce access from corporate offices, secure VPNs, specific cloud service IPs, or dedicated application servers, ensuring that all data interactions happen over approved and secure channels.

## Implementation Example

```sql
-- Step 1: Create a Network Rule for allowed IP addresses
CREATE OR REPLACE NETWORK RULE corporate_office_ips
  VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')
  TYPE = IPV4
  MODE = INGRESS
  COMMENT = 'Allowed IP addresses for corporate offices and a specific analyst workstation';

-- Step 2: Create a Network Policy that references the Network Rule
CREATE OR REPLACE NETWORK POLICY corporate_access_policy
  ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips')
  COMMENT = 'Network policy allowing access only from corporate office IPs';

-- Step 3: Activate the Network Policy for the account
ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;

-- Optional: Activate for a specific user (overrides account-level policy for this user)
-- ALTER USER your_username SET NETWORK_POLICY = corporate_access_policy;

-- To view active network policies
SHOW PARAMETERS LIKE 'network_policy' IN ACCOUNT;

-- To view network rules
SHOW NETWORK RULES;
```

## Explanation of the Code

This example demonstrates the modern approach using Network Rules, which are reusable objects for defining IP lists, then incorporating them into Network Policies.

- **`CREATE OR REPLACE NETWORK RULE corporate_office_ips ...`**:
    - **What it does**: This statement defines a new named network rule called `corporate_office_ips`. A Network Rule is a schema-level object that groups network identifiers (like IP addresses).
    - **How it changes the result set**: This is a DDL (Data Definition Language) statement that creates a metadata object; it does not operate on a result set.
    - `VALUE_LIST = ('203.0.113.0/24', '198.51.100.10')`: Specifies the list of allowed IP addresses or CIDR blocks.
    - `TYPE = IPV4`: Declares that the identifiers are IPv4 addresses.
    - `MODE = INGRESS`: Indicates that this rule applies to incoming connections to Snowflake.
    - `COMMENT = '...'`: Provides a descriptive comment for documentation.

- **`CREATE OR REPLACE NETWORK POLICY corporate_access_policy ...`**:
    - **What it does**: This statement creates a new network policy named `corporate_access_policy`. Network Policies reference one or more Network Rules to determine allowed or blocked access.
    - **How it changes the result set**: This is a DDL statement creating a metadata object; it does not operate on a result set.
    - `ALLOWED_NETWORK_RULE_LIST = ('corporate_office_ips')`: This clause specifies that connections matching the `corporate_office_ips` network rule are permitted. Any IPs not covered by an allowed rule are implicitly blocked.
    - `COMMENT = '...'`: Adds a descriptive comment.

- **`ALTER ACCOUNT SET NETWORK_POLICY = corporate_access_policy;`**:
    - **What it does**: This statement activates the `corporate_access_policy` for the entire Snowflake account. Once activated, all connections to this account must originate from IP addresses defined in the `corporate_office_ips` rule.
    - **How it changes the result set**: This is a DDL statement that modifies the account's security configuration; it does not operate on a result set.

- **`SHOW PARAMETERS LIKE 'network_policy' IN ACCOUNT;`**:
    - **What it does**: This command retrieves the currently active network policy for the account.
    - **How it changes the result set**: Returns a table-like result set showing account parameters, including the active network policy.

- **`SHOW NETWORK RULES;`**:
    - **What it does**: This command lists all defined network rules in the current account.
    - **How it changes the result set**: Returns a table-like result set detailing each network rule (name, type, value list, mode, comment, etc.).

## When to Use

1.  **Restricting Corporate Access**: When you need to ensure that all access to Snowflake originates from your company's internal network or approved VPNs.
    *   *Example Scenario*: A financial institution mandates that data analysts can only connect to Snowflake from their secure office network or a company-issued laptop connected to the corporate VPN.
2.  **Securing Partner/Client Access**: To provide controlled access to specific external partners or clients from their designated IP ranges.
    *   *Example Scenario*: A vendor needs to access a specific dataset in your Snowflake account, and you want to whitelist only their known, secure IP addresses to prevent broader access.
3.  **Compliance and Regulatory Requirements**: To meet industry-specific regulations (e.g., PCI DSS, HIPAA) that require strict control over network access to sensitive data.
    *   *Example Scenario*: A healthcare provider stores patient data in Snowflake and must ensure that only connections from HIPAA-compliant environments are allowed.

## When Not to Use

1.  **Public-Facing Applications**: If you are building a public application or website that needs to connect to Snowflake, network policies are not suitable for user-initiated connections as whitelisting every potential user's IP is impossible and impractical.
    *   *Reason*: Network policies are designed for controlled, known access points, not for open internet access. Other authentication/authorization mechanisms (like OAuth or service accounts with key pair authentication) are more appropriate.
2.  **Overly Broad IP Ranges**: Using very broad CIDR blocks (e.g., `0.0.0.0/0`) defeats the purpose of network policies and offers minimal security benefit.
    *   *Reason*: This essentially allows connections from anywhere, negating the access control. It only adds unnecessary complexity without improving security posture.
3.  **User-Level Policies for Large, Dynamic Teams**: While user-level policies exist, managing them for a large, frequently changing user base with varying IP addresses can become an administrative nightmare.
    *   *Reason*: It increases overhead and can lead to users being inadvertently locked out. Account-level policies or integration-specific policies are generally more manageable for most scenarios, or relying on identity providers with conditional access.

## Common Mistakes

1.  **Locking Yourself Out**: A very common mistake is activating a network policy without including your current IP address in the `ALLOWED_NETWORK_RULE_LIST`, which immediately blocks your access to the account.
2.  **Mismatched IP Addresses/CIDR Blocks**: Incorrectly formatting IP addresses or CIDR blocks can lead to legitimate users being blocked or, conversely, unintended access being granted.
3.  **Forgetting to Activate**: Creating a network policy but forgetting to activate it at the account, user, or integration level means it has no effect, leaving the account unsecured.
4.  **Overlapping or Conflicting Policies**: Applying conflicting policies at the account, user, and security integration levels can cause confusion or unintended access restrictions due to precedence rules (security integration > user > account).
5.  **Not Using Network Rules**: Before Network Rules were introduced, policies directly contained IP lists. Not migrating to or using Network Rules can lead to less flexible and less maintainable policy management, especially with complex IP requirements.

## Expected Output

After successfully implementing and querying for network policies, the expected output will be metadata tables providing information about the configured rules and policies.

**`SHOW PARAMETERS LIKE 'network_policy' IN ACCOUNT;`**

This query would show the active network policy name for the account.

| KEY            | VALUE                  | DEFAULT   | LEVEL   | DESCRIPTION                       |
| :------------- | :--------------------- | :-------- | :------ | :-------------------------------- |
| NETWORK_POLICY | corporate_access_policy| NULL      | ACCOUNT | Network policy for the account    |

**`SHOW NETWORK RULES;`**

This query would list the details of the created network rule.

| name                | type | mode    | value_list                                   | comment                                                      |
| :------------------ | :--- | :------ | :------------------------------------------- | :----------------------------------------------------------- |
| CORPORATE_OFFICE_IPS| IPV4 | INGRESS | ["203.0.113.0/24", "198.51.100.10"] | Allowed IP addresses for corporate offices and a specific analyst workstation |

**Explanation of Columns:**

-   **`name`**: The unique identifier of the network rule.
-   **`type`**: The type of network identifier (e.g., IPV4).
-   **`mode`**: Indicates if the rule applies to `INGRESS` (incoming) or `EGRESS` (outgoing) traffic.
-   **`value_list`**: The actual list of IP addresses or CIDR ranges defined in the rule.
-   **`comment`**: The descriptive text provided during rule creation.
