# Creating a Basic Row Access Policy in Snowflake

Row Access Policies in Snowflake provide a robust, centralized, and flexible way to implement row-level security. They allow you to define rules that determine which rows a user can see in a table or view, based on their role, user ID, or other attributes. This is crucial for data privacy, compliance, and multi-tenant environments where different users should only access specific subsets of data.

**Short Explanation**

A Row Access Policy acts like a filter on a table, dynamically showing users only the rows they are authorized to see, without altering the underlying data. It solves the problem of ensuring data privacy and compliance by restricting data access at a granular level. This is important for protecting sensitive information, meeting regulatory requirements, and building secure data applications. It's commonly used in scenarios where data needs to be segmented by department, region, customer, or any other access-control criteria.

- What problem does this SQL feature solve? It solves the problem of restricting access to specific rows of data based on user identity or other conditions, enhancing data privacy and security.
- Why is it important in databases or analytics? It's vital for implementing fine-grained access control, supporting compliance mandates (like GDPR, HIPAA), and enabling secure multi-tenant data sharing.
- Where is it commonly used in real workflows? It's used in financial services for customer data segregation, healthcare for patient record access, and SaaS applications for isolating tenant data.

## Implementation Example

```sql
-- 1. Create a simple table with sensitive data
CREATE OR REPLACE TABLE sales_data (
    sale_id INTEGER,
    customer_id INTEGER,
    region VARCHAR,
    amount DECIMAL(10, 2),
    sales_rep VARCHAR
);

-- Insert some sample data
INSERT INTO sales_data VALUES
(1, 101, 'East', 150.00, 'Alice'),
(2, 102, 'West', 200.00, 'Bob'),
(3, 103, 'East', 300.00, 'Alice'),
(4, 104, 'North', 120.00, 'Charlie'),
(5, 105, 'West', 250.00, 'Bob');

-- 2. Create the Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy
AS (sales_rep VARCHAR) RETURNS BOOLEAN ->
    -- Condition to allow access: current user is the sales rep or has 'ADMIN' role
    CURRENT_ROLE() = 'ACCOUNTADMIN' OR
    CURRENT_USER() = sales_rep;

-- 3. Apply the Row Access Policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (sales_rep);

-- 4. Verify access (as different users/roles)
-- As 'ACCOUNTADMIN' or 'Alice' (assuming 'Alice' is a user and her username is 'Alice'):
-- SELECT * FROM sales_data;

-- As 'Bob' (assuming 'Bob' is a user and his username is 'Bob'):
-- SELECT * FROM sales_data;

-- As 'Charlie' (assuming 'Charlie' is a user and his username is 'Charlie'):
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example demonstrates creating and applying a basic Row Access Policy:

- `CREATE OR REPLACE TABLE sales_data (...)`: This statement creates a sample table `sales_data` with columns like `sale_id`, `customer_id`, `region`, `amount`, and `sales_rep`. The `sales_rep` column will be used to determine row access.
- `INSERT INTO sales_data VALUES (...)`: Populates the `sales_data` table with sample records.
- `CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (sales_rep VARCHAR) RETURNS BOOLEAN -> ...`:
    - This is the core statement for defining the policy.
    - `sales_policy` is the name of our policy.
    - `AS (sales_rep VARCHAR)` specifies the column(s) from the target table that the policy will evaluate. Here, it takes the `sales_rep` column's value as an argument.
    - `RETURNS BOOLEAN ->` indicates that the policy function will return `TRUE` (allow access) or `FALSE` (deny access).
    - `CURRENT_ROLE() = 'ACCOUNTADMIN' OR CURRENT_USER() = sales_rep;`: This is the policy's logic. It allows a user to see a row if:
        - The `CURRENT_ROLE()` is `ACCOUNTADMIN` (allowing administrators full access).
        - OR the `CURRENT_USER()` (the authenticated user's login name) matches the `sales_rep` value for that row.
- `ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (sales_rep);`:
    - `ALTER TABLE`: This clause is used to modify an existing table.
    - `ADD ROW ACCESS POLICY sales_policy ON (sales_rep)`: This attaches the `sales_policy` to the `sales_data` table. The `ON (sales_rep)` part specifies the column in the `sales_data` table that will be passed as an argument to the `sales_policy` function. When a user queries `sales_data`, Snowflake internally calls `sales_policy` for each row, passing the `sales_rep` value from that row to the policy for evaluation. If the policy returns `TRUE`, the row is included in the result set; otherwise, it's filtered out.
- `SELECT * FROM sales_data;` (under different users/roles): This demonstrates how the `SELECT` clause will return different results based on the `CURRENT_USER()` and `CURRENT_ROLE()`, as defined by the active row access policy. The `FROM` clause refers to the table, and the policy transparently filters rows before they are returned.

## When to Use

1.  **Multi-tenant Applications:** When different customers or tenants share the same database tables but should only see their own data.
    *   *Scenario:* A SaaS application stores data for multiple companies in a single `transactions` table. A row access policy ensures each company's users only see their company's transactions.
2.  **Regulatory Compliance (e.g., GDPR, HIPAA):** To restrict access to sensitive data based on specific user roles or attributes required by privacy regulations.
    *   *Scenario:* A healthcare provider stores patient records. A policy might limit doctors to only see records of their assigned patients, and administrative staff to only see billing information, not medical details.
3.  **Internal Departmental Data Segregation:** To ensure employees only access data relevant to their department or region.
    *   *Scenario:* A global sales team uses a `sales_leads` table. A policy could be set up so that sales managers in Europe only see leads from European countries, and US managers only see US leads.

## When Not to Use

1.  **When Column-Level Security is Sufficient:** If access needs to be restricted only to certain columns, not rows, then column-level security (e.g., masking policies) might be a simpler and more performant solution.
    *   *Situation:* You want to hide a credit card number column from most users but show all rows. A masking policy on that specific column would be better.
2.  **For Simple Table/View Access Control:** If the goal is just to grant or deny access to an entire table or view, standard `GRANT` and `REVOKE` permissions are more appropriate and less overhead.
    *   *Situation:* A user should either see all data in a table or none. `GRANT SELECT ON TABLE` is the correct approach.
3.  **Performance Critical Workloads with Very Complex Policies:** While Snowflake's row access policies are optimized, excessively complex policies that perform many lookups or calculations per row can introduce performance overhead, especially on very large tables with high query concurrency.
    *   *Situation:* A policy requires joining to several large lookup tables for each row or executing complex subqueries. Simpler designs or pre-filtered views might be more efficient.

## Common Mistakes

1.  **Forgetting to Apply the Policy:** Defining the policy is only half the battle; it must be applied to the table using `ALTER TABLE ADD ROW ACCESS POLICY`. Forgetting this step means the policy has no effect.
2.  **Incorrect Policy Logic:** Errors in the `RETURNS BOOLEAN` expression can lead to unintended data exposure (granting too much access) or data hiding (denying legitimate access). Thorough testing with different users is crucial.
3.  **Performance Impact with Suboptimal Policy Design:** Using non-performant functions or complex subqueries within the policy definition can slow down queries on the protected table. Policies should be as simple and efficient as possible.
4.  **Not Handling `NULL` Values:** If the policy column (`sales_rep` in our example) can contain `NULL`s, the policy logic must explicitly account for them, otherwise rows with `NULL`s might be unintentionally filtered or exposed.
5.  **Overlapping Policies/Masking Policies:** While policies can coexist, careful design is needed when applying both row access policies and column masking policies to the same table to ensure expected behavior and avoid conflicts.

## Expected Output

The resulting dataset will contain only the rows that satisfy the conditions defined in the `sales_policy` for the `CURRENT_USER()` or `CURRENT_ROLE()`.

**Example Table (when queried by user 'Alice' or 'ACCOUNTADMIN'):**

| SALE_ID | CUSTOMER_ID | REGION | AMOUNT | SALES_REP |
| :------ | :---------- | :----- | :----- | :-------- |
| 1       | 101         | East   | 150.00 | Alice     |
| 3       | 103         | East   | 300.00 | Alice     |
| 2       | 102         | West   | 200.00 | Bob       |
| 5       | 105         | West   | 250.00 | Bob       |
| 4       | 104         | North  | 120.00 | Charlie   |

**Example Table (when queried by user 'Bob'):**

| SALE_ID | CUSTOMER_ID | REGION | AMOUNT | SALES_REP |
| :------ | :---------- | :----- | :----- | :-------- |
| 2       | 102         | West   | 200.00 | Bob       |
| 5       | 105         | West   | 250.00 | Bob       |

**Explanation of Columns:**
- `SALE_ID`: Unique identifier for each sale.
- `CUSTOMER_ID`: Identifier for the customer involved in the sale.
- `REGION`: The geographical region where the sale occurred.
- `AMOUNT`: The monetary value of the sale.
- `SALES_REP`: The name of the sales representative responsible for the sale. This is the column used by the policy to determine access.
