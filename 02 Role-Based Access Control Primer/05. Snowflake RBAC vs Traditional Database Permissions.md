# 05. Snowflake RBAC vs Traditional Database Permissions

Snowflake's Role-Based Access Control (RBAC) centralizes security by assigning privileges exclusively to roles rather than directly to users, unlike traditional databases that often rely on direct user-level grants.
This hierarchical approach allows roles to inherit permissions from other roles, drastically simplifying access management, enforcing least privilege, and streamlining auditing at scale.

# Key Points

* **No Direct User Grants:** In Snowflake, privileges on objects (tables, schemas) are granted exclusively to roles, whereas traditional databases often allow direct `GRANT SELECT ON table TO user`.
* **Role Hierarchy and Inheritance:** Snowflake supports a robust role hierarchy where higher-level roles automatically inherit all privileges of the lower-level roles granted to them.
* **Separation of Ownership and Access:** Snowflake distinguishes between object ownership (the `OWNERSHIP` privilege) and access privileges, allowing centralized management without tying object lifecycle to a specific user account.
* **System-Defined Roles:** Snowflake provides built-in roles (`ACCOUNTADMIN`, `SECURITYADMIN`, `SYSADMIN`, `USERADMIN`, `PUBLIC`) with specific administrative boundaries, enforcing a secure-by-default architecture.
* **Context-Dependent Execution:** Users in Snowflake operate under a single active role per session, preventing accidental privilege escalation, unlike traditional databases where all user privileges are active simultaneously.

# Implementation

> [!IMPORTANT]  
> Always use `SECURITYADMIN` or `USERADMIN` to manage roles and users, and `SYSADMIN` to manage database objects and their grants. Never grant privileges directly to users.

```sql
USE ROLE SECURITYADMIN;

CREATE ROLE data_analyst;
CREATE ROLE senior_data_analyst;

GRANT ROLE data_analyst TO ROLE senior_data_analyst;
GRANT ROLE senior_data_analyst TO USER alice_smith;

USE ROLE SYSADMIN;

GRANT USAGE ON DATABASE sales_db TO ROLE data_analyst;
GRANT USAGE ON SCHEMA sales_db.public TO ROLE data_analyst;
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE data_analyst;
```

# What does the Query do?

Step 1 explanation: Switches to the `SECURITYADMIN` role to create two custom roles (`data_analyst` and `senior_data_analyst`) representing different organizational job functions.

Step 2 explanation: Establishes a role hierarchy by granting the `data_analyst` role to the `senior_data_analyst` role, ensuring the senior role automatically inherits all base analyst privileges.

Step 3 explanation: Assigns the top-level `senior_data_analyst` role to a specific user (`alice_smith`), granting her access to the entire inherited privilege chain without assigning direct object permissions.

Step 4 explanation: Switches to the `SYSADMIN` role to grant actual database, schema, and table-level read privileges to the base `data_analyst` role, which seamlessly flows up the hierarchy to the senior role.

# When Should We Use This? 

* When designing a scalable security architecture where employee onboarding requires assigning a single job-function role rather than managing hundreds of individual table grants.
* When compliance and auditing requirements mandate strict separation of duties and explicit session contexts for data access.
