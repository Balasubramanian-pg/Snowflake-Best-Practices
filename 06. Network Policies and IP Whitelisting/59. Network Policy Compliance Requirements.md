# Network Policy Compliance Requirements in Snowflake

Network policies in Snowflake define which network locations are allowed to access your Snowflake account, enhancing security by restricting access to authorized IP addresses. They exist to enforce a strong security perimeter, ensuring that only trusted networks can connect to your data warehouse.

**Short Explanation**

Network policies in Snowflake act as firewalls, controlling inbound access to your Snowflake account. They solve the problem of unauthorized network access by allowing you to whitelist specific IP addresses or ranges. This is crucial for maintaining data security and compliance, especially when dealing with sensitive information or regulatory requirements. They are commonly used in enterprise environments to restrict access to corporate networks, VPNs, or specific cloud environments.

- **What problem does this SQL feature solve?** It prevents unauthorized network access to a Snowflake account, ensuring that only specified IP addresses or ranges can connect.
- **Why is it important in databases or analytics?** It's vital for data security, compliance with regulations (like GDPR, HIPAA), and reducing the attack surface by limiting potential entry points.
- **Where is it commonly used in real workflows?** Enterprises use it to restrict access to their internal corporate networks, secure cloud environments, or specific data centers where analytical tools or users reside.

## Implementation Example

```sql
-- Create a network policy allowing access from specific IP addresses
CREATE NETWORK POLICY my_corporate_network_policy
    ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.100')
    COMMENT = 'Network policy for corporate office and specific VPN IP.';

-- Activate the network policy for the account
ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;

-- (Optional) Deactivate the network policy for the account
-- ALTER ACCOUNT UNSET NETWORK_POLICY;
```

## Explanation of the Code

This example demonstrates how to create and activate a network policy in Snowflake.

- `CREATE NETWORK POLICY my_corporate_network_policy`: This statement initializes the creation of a new network policy named `my_corporate_network_policy`.
    - **What it does**: Defines a new security object that specifies allowed network locations.
    - **How it changes the result set**: This is a DDL (Data Definition Language) statement; it doesn't return a result set but creates a metadata object in Snowflake.
- `ALLOWED_IP_LIST = ('192.168.1.0/24', '10.0.0.100')`: This clause specifies the list of IP addresses or CIDR ranges that are permitted to connect to the Snowflake account.
    - **What it does**: Defines the whitelist of allowed networks. Any connection attempts from IPs not in this list will be rejected.
    - **How it changes the result set**: Not applicable, as it's part of the policy definition.
- `COMMENT = 'Network policy for corporate office and specific VPN IP.'`: An optional clause to add a descriptive comment to the network policy.
    - **What it does**: Provides context or documentation for the policy.
    - **How it changes the result set**: Not applicable.
- `ALTER ACCOUNT SET NETWORK_POLICY = my_corporate_network_policy;`: This statement applies the previously created network policy to the entire Snowflake account.
    - **What it does**: Activates the `my_corporate_network_policy` at the account level, making its rules effective for all users and connections to that account.
    - **How it changes the result set**: Not applicable; it modifies account-level configuration.
- `ALTER ACCOUNT UNSET NETWORK_POLICY;`: This (commented-out) statement would deactivate any active account-level network policy.
    - **What it does**: Removes the account-level network policy, allowing connections from any IP address again (unless user-level policies are in place).
    - **How it changes the result set**: Not applicable.

## When to Use

1.  **Enhancing Security Posture**: When you need to restrict access to your Snowflake account only from known, trusted corporate networks or VPNs to minimize external attack vectors.
    *   *Example Scenario*: A company wants to ensure that employees can only access Snowflake from their office network or via the corporate VPN.
2.  **Meeting Compliance Requirements**: For industries with strict data governance or regulatory compliance (e.g., PCI DSS, HIPAA, GDPR) that mandate network access controls.
    *   *Example Scenario*: A healthcare provider needs to comply with HIPAA regulations by limiting access to patient data in Snowflake to specific secure environments.
3.  **Preventing Unauthorized Access from Compromised Credentials**: Even if user credentials are leaked, a network policy can prevent attackers from logging in if they are not on an allowed network.
    *   *Example Scenario*: An employee's laptop is stolen, and their Snowflake credentials are compromised, but the attacker cannot log in because they are outside the allowed corporate network.

## When Not to Use

1.  **Open Access for Development or Testing**: When you need broad, unrestricted access for a short-term development or testing environment where security is less critical, and setting up specific IPs is cumbersome.
    *   *Example Scenario*: A small development team is rapidly prototyping a new application and needs to connect from various temporary locations without strict IP whitelisting.
2.  **Public-facing Applications with Dynamic IPs**: For applications hosted in cloud environments with highly dynamic or unpredictable outbound IP addresses, or those meant for broad public access (though Snowflake is rarely used directly by public-facing apps).
    *   *Example Scenario*: A web application that dynamically scales across many cloud instances with frequently changing public IP addresses, making a static `ALLOWED_IP_LIST` impractical.
3.  **Troubleshooting Connectivity Issues**: Temporarily disabling or loosening a network policy might be necessary to diagnose connectivity problems, but it should be re-enabled promptly.
    *   *Example Scenario*: Users are reporting connection errors, and the administrator suspects the network policy might be blocking legitimate traffic; a temporary deactivation helps isolate the issue.

## Common Mistakes

1.  **Forgetting to Activate the Policy**: Creating a network policy but not activating it at the account or user level means it has no effect, leaving the account unsecured.
2.  **Incorrect IP/CIDR Specification**: Using incorrect IP addresses or CIDR ranges, leading to legitimate users being blocked or unintended IPs gaining access.
3.  **Overly Restrictive Policies**: Implementing a policy that blocks essential services or legitimate user access (e.g., blocking Snowflake's own internal services or required cloud IPs).
4.  **No `BLOCKED_IP_LIST` for Exceptions**: Not understanding that `ALLOWED_IP_LIST` is a whitelist, and thinking `BLOCKED_IP_LIST` is needed to block *within* the allowed range (it's for blocking when `ALLOWED_IP_LIST` isn't used or for user-level policies).

## Expected Output

Since the commands for creating and altering network policies are DDL statements, they do not produce a traditional data set. Instead, they confirm the successful execution of the command.

Upon successful execution, the output would typically be a message indicating the operation was completed.

| status                          |
| :------------------------------ |
| Statement executed successfully |

This output confirms that the `CREATE NETWORK POLICY` or `ALTER ACCOUNT SET NETWORK_POLICY` command was processed by Snowflake, and the network policy is now defined and/or active according to the SQL statement. No actual data rows are returned, as it's a configuration change.
