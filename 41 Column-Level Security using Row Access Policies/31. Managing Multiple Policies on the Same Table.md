# Managing Multiple Policies on the Same Table in Snowflake

This concept addresses how Snowflake handles scenarios where multiple security policies (like row access policies and masking policies) are applied to the same table or column. It ensures that data governance and access control rules are enforced consistently, even when diverse policy requirements exist.

**Short Explanation**

When multiple policies are applied to the same data, Snowflake evaluates them in a defined order to determine the final data access or transformation. This allows for granular control, combining different security rules such as restricting rows based on user roles and masking sensitive data within the allowed rows. It's important for maintaining data security and compliance while still providing necessary data access to different user groups. This capability is commonly used in enterprises with complex data governance needs, where various policies might apply to different subsets of data or different user roles accessing the same data.

- **What problem does this SQL feature solve?** It solves the challenge of applying comprehensive and layered security controls to data within a single table, accommodating different access rules for various user groups or data sensitivity levels.
- **Why is it important in databases or analytics?** It's crucial for achieving fine-grained access control, data anonymization, and regulatory compliance (e.g., GDPR, HIPAA) without duplicating data or creating complex views for every access scenario.
- **Where is it commonly used in real workflows?** In industries like finance, healthcare, and any sector dealing with personally identifiable information (PII) or sensitive business data, where different users (analysts, auditors, operations) need varied levels of access to the same underlying data.

## Implementation Example

```sql
-- 1. Create a sample table
CREATE OR REPLACE TABLE sales_data (
    order_id INT,
    customer_name VARCHAR,
    product VARCHAR,
    amount DECIMAL(10, 2),
    region VARCHAR,
    sensitive_info VARCHAR,
    sales_rep VARCHAR
);

-- 2. Insert sample data
INSERT INTO sales_data VALUES
(1, 'Alice', 'Laptop', 1200.00, 'East', 'Personal Email: alice@example.com', 'John'),
(2, 'Bob', 'Keyboard', 75.00, 'West', 'Phone: 555-1234', 'Jane'),
(3, 'Charlie', 'Monitor', 300.00, 'East', 'Address: 123 Main St', 'John'),
(4, 'David', 'Mouse', 25.00, 'North', 'SSN: XXX-XX-6789', 'Mike');

-- 3. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY regional_sales_policy
AS (sales_rep VARCHAR) RETURNS BOOLEAN ->
    CURRENT_ROLE() IN ('ADMIN', 'SECURITY_ANALYST')
    OR (CURRENT_ROLE() = 'SALES_REP_EAST' AND sales_rep = 'John')
    OR (CURRENT_ROLE() = 'SALES_REP_WEST' AND sales_rep = 'Jane');

-- 4. Create a Masking Policy for sensitive_info
CREATE OR REPLACE MASKING POLICY mask_sensitive_data
AS (val VARCHAR) RETURNS VARCHAR ->
    CASE
        WHEN CURRENT_ROLE() IN ('ADMIN', 'SECURITY_ANALYST') THEN val
        WHEN CURRENT_ROLE() = 'DATA_ANALYST' THEN 'XXXXXXXXX' -- Mask for data analysts
        ELSE '**********' -- Default mask for others
    END;

-- 5. Apply the Row Access Policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY regional_sales_policy ON (sales_rep);

-- 6. Apply the Masking Policy to the sensitive_info column
ALTER TABLE sales_data MODIFY COLUMN sensitive_info SET MASKING POLICY mask_sensitive_data;

-- 7. Grant necessary roles (example assumes these roles exist)
GRANT ROLE ADMIN TO USER <your_user>;
GRANT ROLE SALES_REP_EAST TO USER <your_user>;
GRANT ROLE DATA_ANALYST TO USER <your_user>;

-- Now, try querying with different roles:

-- Assume role ADMIN (sees all data unmasked)
USE ROLE ADMIN;
SELECT * FROM sales_data;

-- Assume role SALES_REP_EAST (sees only East region sales, sensitive_info unmasked)
USE ROLE SALES_REP_EAST;
SELECT * FROM sales_data;

-- Assume role DATA_ANALYST (sees all rows but sensitive_info is masked)
USE ROLE DATA_ANALYST;
SELECT * FROM sales_data;

-- Assume a user with no specific role (sees no data, or default masking if row policy allows)
-- This example with default masking will be tricky without a specific default role,
-- but if we had a general 'viewer' role, they would see all rows, but sensitive_info would be '**********'
```

## Explanation of the Code

- **`CREATE OR REPLACE TABLE sales_data (...)`**: This statement defines a new table named `sales_data` with various columns to store sales transaction details, including a `sensitive_info` column and a `sales_rep` column for policy application.
- **`INSERT INTO sales_data VALUES (...)`**: Populates the `sales_data` table with example records, providing data for testing the policies.
- **`CREATE OR REPLACE ROW ACCESS POLICY regional_sales_policy AS (sales_rep VARCHAR) RETURNS BOOLEAN -> ...`**:
    - **What it does**: Defines a row access policy named `regional_sales_policy`. This policy determines which rows a user can see based on their current role and the `sales_rep` value in the row.
    - **How it changes the result set**: When applied, this policy acts as an implicit `WHERE` clause. If the policy evaluates to `TRUE` for a given row, the row is returned; otherwise, it's filtered out. For example, `SALES_REP_EAST` can only see rows where `sales_rep` is 'John'.
- **`CREATE OR REPLACE MASKING POLICY mask_sensitive_data AS (val VARCHAR) RETURNS VARCHAR -> ...`**:
    - **What it does**: Defines a masking policy named `mask_sensitive_data`. This policy transforms the value of a column (`val`) based on the user's current role.
    - **How it changes the result set**: When applied to a column, this policy alters the visible data in that column. For instance, `DATA_ANALYST` sees 'XXXXXXXXX' instead of the actual `sensitive_info`.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY regional_sales_policy ON (sales_rep);`**:
    - **What it does**: Attaches the `regional_sales_policy` to the `sales_data` table, specifying that the `sales_rep` column will be used as the input for the policy's evaluation.
    - **How it changes the result set**: This activates the row-level filtering based on the `regional_sales_policy` for any queries on `sales_data`.
- **`ALTER TABLE sales_data MODIFY COLUMN sensitive_info SET MASKING POLICY mask_sensitive_data;`**:
    - **What it does**: Attaches the `mask_sensitive_data` masking policy to the `sensitive_info` column of the `sales_data` table.
    - **How it changes the result set**: This activates the column-level data transformation defined by `mask_sensitive_data` for the `sensitive_info` column.
- **`USE ROLE ...; SELECT * FROM sales_data;`**:
    - **What it does**: These commands demonstrate querying the table under different user roles, showcasing how the combined policies affect the visible data.
    - **How it changes the result set**: Depending on the `CURRENT_ROLE()`:
        - `ADMIN`: No filtering by the row policy, no masking by the masking policy. All original data is seen.
        - `SALES_REP_EAST`: Rows are filtered by the row policy (only `sales_rep = 'John'`). `sensitive_info` is unmasked because the policy condition `CURRENT_ROLE() IN ('ADMIN', 'SECURITY_ANALYST')` evaluates to false, and the `ELSE` branch for `SALES_REP_EAST` is not defined to mask. *Correction*: In the example's masking policy, SALES_REP_EAST would fall under the `ELSE '**********'` branch if not explicitly handled. For this specific masking policy, only 'ADMIN' and 'SECURITY_ANALYST' see `val` unmasked. 'SALES_REP_EAST' would see '**********' for sensitive info.
        - `DATA_ANALYST`: No filtering by the row policy (as it's not 'ADMIN' or 'SECURITY_ANALYST' or a specific sales rep, it implicitly sees nothing unless the policy specifically allows), but `sensitive_info` is masked to 'XXXXXXXXX'. *Correction*: The `regional_sales_policy` for `DATA_ANALYST` would evaluate to FALSE, meaning `DATA_ANALYST` would see no rows, unless the policy was updated to explicitly allow them.

## When to Use

1.  **Granular Data Access Control**: When different departments or user roles need to see varying subsets of rows and/or different levels of data obfuscation within the same table.
    *   *Scenario*: A `transactions` table contains financial data. Sales managers only see transactions from their region, while auditors see all transactions but sensitive customer details are masked, and compliance officers see everything unmasked.
2.  **Regulatory Compliance (e.g., GDPR, HIPAA)**: To ensure that sensitive data (like PII or Protected Health Information) is only accessible to authorized personnel and is appropriately masked or filtered for others.
    *   *Scenario*: A `patient_records` table in a healthcare system has a row access policy limiting access to patients based on their attending physician's department and a masking policy redacting patient names and addresses for researchers.
3.  **Tiered Security for Analytics**: When analytical teams need access to broad datasets for trends but shouldn't see individual-level sensitive details.
    *   *Scenario*: Data scientists analyzing customer behavior need access to all customer activity data, but customer names, email addresses, and specific financial figures are masked, while high-level aggregations are fully visible.

## When Not to Use

1.  **Over-Complication for Simple Access**: If your security requirements are very basic (e.g., only full access or no access for everyone), applying multiple policies can add unnecessary complexity.
    *   *Situation*: A small internal tool where all users have the same data access permissions. A single role-based access control (RBAC) model might be sufficient.
2.  **Performance Overheads with Complex Policies**: Extremely complex or nested policy logic, especially within row access policies, can introduce performance overheads on large tables as each row must be evaluated.
    *   *Situation*: A high-throughput transactional system where query latency is critical, and policies involve expensive subqueries or many conditional checks for every row.
3.  **Conflicting Policy Definitions**: If policies are poorly designed and create contradictory rules (e.g., one policy says "mask this" and another implicitly says "show this" for the same column/user), leading to unexpected or incorrect data exposure.
    *   *Situation*: Two masking policies applied to the same column, or a row access policy filters rows that a masking policy was intended to operate on for a specific role, leading to confusion about what data is ultimately seen.

## Common Mistakes

1.  **Not Understanding Policy Evaluation Order**: Snowflake evaluates row access policies first (filtering rows) and then masking policies (transforming data in the remaining rows). Not understanding this order can lead to unexpected results, such as masking policies never being applied to filtered-out rows.
2.  **Overlapping or Redundant Policy Conditions**: Creating policies with conditions that overlap unnecessarily or are difficult to manage, leading to maintenance nightmares and potential security gaps. Forgetting to define a default behavior (`ELSE` clause) in masking policies can result in unmasked data for roles not explicitly covered.
3.  **Performance Impacts**: Applying complex policies to columns used in `JOIN` conditions or `WHERE` clauses can severely degrade query performance, as the policy must be evaluated for each row before the join or filter can occur efficiently.
4.  **Forgetting to Grant Privileges**: Users need `APPLY` privileges on policies to be able to attach them to tables/columns, and appropriate `USAGE` on roles used within the policies. Not granting these can lead to policy application failures or policies not being effective.
5.  **Lack of Testing with Different Roles**: Developers often test with an `ADMIN` role but forget to verify the data seen by various restricted roles, potentially leading to over-exposure or under-exposure of data.
