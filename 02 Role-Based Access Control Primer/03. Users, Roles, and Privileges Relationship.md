# 03. Users, Roles, and Privileges Relationship

The relationship between users, roles, and privileges forms the foundation of Role-Based Access Control (RBAC), ensuring secure and manageable access to system resources. Privileges are granted to roles, and roles are assigned to users, creating a scalable security architecture that separates identity management from access rights.

> [!IMPORTANT]  
> Never grant privileges directly to users. Always grant privileges to roles and assign those roles to users to maintain a scalable, auditable, and easily maintainable security model.

# Key Points

* **Users** are individual identities (people, applications, or service accounts) that authenticate into the system but do not hold direct permissions to data objects.
* **Roles** act as logical containers or job functions that group related privileges together and can be nested to create a hierarchical access model.
* **Privileges** are specific, granular permissions (e.g., `SELECT`, `INSERT`, `USAGE`) that dictate exactly what actions can be performed on specific securable objects.
* **Role-Based Access Control (RBAC)** simplifies administration by allowing administrators to modify a role's privileges once, instantly updating access for all users assigned to that role.
* **Principle of Least Privilege** is achieved by assigning users only the specific roles necessary to perform their required tasks, minimizing potential security risks.

# Implementation

Provide a realistic Snowflake SQL example demonstrating the concept.

```sql
-- Step 1: Create the role
CREATE ROLE data_analyst_role;

-- Step 2: Grant privileges to the role
GRANT USAGE ON DATABASE analytics_db TO ROLE data_analyst_role;
GRANT USAGE ON SCHEMA analytics_db.public TO ROLE data_analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics_db.public TO ROLE data_analyst_role;

-- Step 3: Create the user
CREATE USER jdoe 
    PASSWORD = 'StrongPassword123!' 
    DEFAULT_ROLE = data_analyst_role 
    MUST_CHANGE_PASSWORD = TRUE;

-- Step 4: Assign the role to the user
GRANT ROLE data_analyst_role TO USER jdoe;
```

# What does the Query do?

Explain the SQL step by step:

Step 1 creates a new logical container called `data_analyst_role` that will hold specific permissions for analytical tasks.

Step 2 grants the necessary granular privileges (`USAGE` on the database and schema, and `SELECT` on all tables) directly to the role, establishing what the role is allowed to do.

Step 3 provisions a new user identity (`jdoe`) in the system, sets their default role to ensure they have the correct context upon logging in, and enforces a password change.

Step 4 establishes the final relationship by assigning the `data_analyst_role` to the user `jdoe`, effectively granting the user all privileges contained within that role.

# When Should We Use This? 

* When onboarding new employees or service accounts, allowing administrators to assign a predefined job-function role rather than manually configuring dozens of individual table permissions.
* When designing a scalable security architecture where access requirements change frequently, ensuring that adding a new table only requires updating the role's privileges once for all associated users.
