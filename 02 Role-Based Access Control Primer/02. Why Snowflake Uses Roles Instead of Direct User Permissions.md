# 02. Why Snowflake Uses Roles Instead of Direct User Permissions

Snowflake utilizes Role-Based Access Control (RBAC) to decouple users from privileges, ensuring that access rights are granted to roles rather than individual users.
This architectural design simplifies security administration, enables hierarchical privilege inheritance, and ensures scalable access management as organizations grow.

# Key Points

*   **Simplified Administration:** Managing permissions at the role level prevents the administrative nightmare of updating individual user privileges when employees change teams or leave the company.
*   **Role Hierarchies:** Roles can be nested (granted to other roles), allowing higher-level roles to automatically inherit the privileges of lower-level roles without redundant grants.
*   **Principle of Least Privilege:** Users can switch between different roles during a session, ensuring they only use the specific permissions required for the task at hand.
*   **Centralized Auditing:** Tracking access and auditing security compliance is significantly easier when evaluating a finite set of roles rather than hundreds or thousands of individual users.
*   **Separation of Duties:** System-defined roles (like `SECURITYADMIN` and `SYSADMIN`) enforce a clear separation between user management, security policies, and object creation.

# Implementation

> [!IMPORTANT]  
> Snowflake strictly enforces RBAC. You cannot grant object privileges (like `SELECT` or `USAGE`) directly to a user account; you must grant them to a role, and then grant that role to the user.

```sql
-- Create a dedicated role for data analysts
CREATE ROLE data_analyst_role;

-- Grant usage on the database and schema to the role
GRANT USAGE ON DATABASE sales_db TO ROLE data_analyst_role;
GRANT USAGE ON SCHEMA sales_db.public TO ROLE data_analyst_role;

-- Grant read-only access to all current and future tables in the schema
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE data_analyst_role;
GRANT SELECT ON FUTURE TABLES IN SCHEMA sales_db.public TO ROLE data_analyst_role;

-- Assign the role to a specific user
GRANT ROLE data_analyst_role TO USER jsmith;
```

# What does the Query do?

*   **Step 1:** Creates a new custom role named `data_analyst_role` to act as a centralized container for specific analytical privileges.
*   **Step 2:** Grants foundational `USAGE` privileges on the `sales_db` database and its `public` schema to the role, which is a strict prerequisite before any objects inside can be queried.
*   **Step 3:** Grants `SELECT` (read-only) permissions on all existing and future tables within the schema to the role, ensuring automatic access management as new tables are created.
*   **Step 4:** Assigns the `data_analyst_role` to the user `jsmith`, granting them all the accumulated privileges of the role without assigning a single permission directly to their user account.

# When Should We Use This? 

*   When onboarding new employees or managing team transfers, allowing administrators to simply assign or revoke a single role rather than modifying dozens of individual table permissions.
*   When designing a scalable security architecture that requires strict compliance auditing, separation of duties, and hierarchical access control across different business units.
