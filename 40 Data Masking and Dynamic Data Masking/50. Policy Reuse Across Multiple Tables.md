# Policy Reuse Across Multiple Tables in Snowflake

Policy reuse across multiple tables in Snowflake refers to the ability to define a single security policy (like a Row Access Policy or a Masking Policy) once and then apply it to multiple tables or columns, rather than creating a separate policy for each individual data asset. This streamlines data governance, reduces administrative overhead, and ensures consistent security enforcement across your data estate.

**Short Explanation**

Imagine you have a rule, like "only managers can see employee salaries," or "salespeople only see customers in their region." Instead of writing that rule for every single table that has employee salaries or customer regions, you write it once. Then, you tell Snowflake to apply that one rule to all the relevant tables.

- **What problem does this SQL feature solve?** It solves the problem of redundant policy creation and management, ensuring consistent data security rules are applied everywhere they're needed without manual, table-by-table configuration.[axpk]
- **Why is it important in databases or analytics?** It's crucial for scalable data governance, especially in large organizations with many tables and columns containing sensitive data. It simplifies auditing, reduces the risk of human error, and ensures compliance with regulations by standardizing security enforcement.
- **Where is it commonly used in real workflows?** It's used in scenarios requiring row-level security (e.g., restricting sales data by region) and column-level security (e.g., masking PII like email addresses or credit card numbers) across many departmental tables.[axpk]

## Implementation Example

Here's an example using a Row Access Policy to restrict data visibility based on a user's role and a `REGION` column, applied to multiple sales tables.

```sql
-- 1. Create a mapping table for roles and regions
CREATE OR REPLACE TABLE demo.governance.region_roles_mapping (
    region VARCHAR,
    role_name VARCHAR
);

INSERT INTO demo.governance.region_roles_mapping VALUES
    ('East', 'SALES_EAST_ANALYST'),
    ('West', 'SALES_WEST_ANALYST'),
    ('North', 'SALES_NORTH_ANALYST');

-- 2. Create the Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY demo.governance.region_policy
  ON (region VARCHAR)
  RETURNS BOOLEAN ->
    'SYSADMIN' = CURRENT_ROLE() OR -- SYSADMIN can see all
    EXISTS (
        SELECT 1 FROM demo.governance.region_roles_mapping
        WHERE region = region_policy.region
        AND role_name = CURRENT_ROLE()
    );

-- 3. Apply the policy to multiple tables
-- Assume 'sales_data_east' and 'sales_data_west' both have a 'region' column
ALTER TABLE demo.sales.sales_data_east ADD ROW ACCESS POLICY demo.governance.region_policy ON (region);
ALTER TABLE demo.sales.sales_data_west ADD ROW ACCESS POLICY demo.governance.region_policy ON (region);

-- 4. Test (as different roles)
-- USE ROLE SALES_EAST_ANALYST;
-- SELECT * FROM demo.sales.sales_data_east;
-- SELECT * FROM demo.sales.sales_data_west;
```

## Explanation of the Code

- **`CREATE OR REPLACE TABLE demo.governance.region_roles_mapping (...)`**: This statement creates a lookup table that associates specific data regions with the roles allowed to view data from those regions. It's a common pattern to manage permissions external to the policy itself.
- **`INSERT INTO demo.governance.region_roles_mapping VALUES (...)`**: Populates the mapping table with sample data, linking 'East' to `SALES_EAST_ANALYST`, 'West' to `SALES_WEST_ANALYST`, etc.
- **`CREATE OR REPLACE ROW ACCESS POLICY demo.governance.region_policy ON (region VARCHAR) RETURNS BOOLEAN ->...`**: This is where the reusable policy is defined.
    - **`ON (region VARCHAR)`**: Specifies the input signature of the policy. The policy expects a column named `region` of type `VARCHAR` from any table it's applied to. The actual column name on the table doesn't have to be `region`, but the data type must match.
    - **`RETURNS BOOLEAN ->`**: Indicates that the policy will return a boolean value (TRUE or FALSE) for each row, determining if it should be visible.
    - **`'SYSADMIN' = CURRENT_ROLE()`**: Allows the `SYSADMIN` role to bypass the policy and see all rows. This is a common administrative exemption.
    - **`OR EXISTS (SELECT 1 FROM demo.governance.region_roles_mapping WHERE region = region_policy.region AND role_name = CURRENT_ROLE())`**: This is the core logic. For any role other than `SYSADMIN`, a row is visible only if the `region` value in that row exists in the `region_roles_mapping` table for the `CURRENT_ROLE()`.
- **`ALTER TABLE... ADD ROW ACCESS POLICY... ON (region)`**: This command applies the `region_policy` to specific tables (`sales_data_east`, `sales_data_west`), specifying that the policy should use the `region` column from those tables as its input. This is the "reuse" part: the same policy definition is attached to multiple tables.
- **`SELECT * FROM...`**: This is a placeholder for querying the tables.
    - **How it changes the result set**: When a user queries a table with this policy applied, Snowflake dynamically evaluates the policy for each row. Only rows for which the policy expression returns `TRUE` are included in the final result set visible to the user.

## When to Use

1. **Consistent Row-Level Security**: When you need to enforce the same row-filtering logic across multiple tables that share a common attribute (e.g., `REGION_ID`, `DEPARTMENT_CODE`).
    * **Example Scenario**: A global company wants to ensure that employees in each country or region can only view customer data pertinent to their geographical area, across various sales, marketing, and support tables. A single row access policy can filter all these tables based on a `COUNTRY` or `REGION` column.
2. **Standardized Data Masking**: When certain sensitive columns (e.g., `SSN`, `EMAIL`, `CREDIT_CARD_NUMBER`) need to be masked consistently across many tables.
    * **Example Scenario**: A financial institution needs to mask credit card numbers for all users except specific compliance officers. A single masking policy can be applied to all `CREDIT_CARD_NUMBER` columns across customer, transaction, and payment tables.
3. **Tag-Based Governance**: When leveraging Snowflake's object tagging to automatically apply policies to tagged objects (databases, schemas, tables, or columns) based on their classification (e.g., 'PII', 'CONFIDENTIAL').
    * **Example Scenario**: A data steward tags all columns containing personally identifiable information (PII) with a 'PII' tag. A pre-defined tag-based masking policy automatically masks these columns for non-privileged roles, even as new PII columns are added or discovered.

## When Not to Use

1. **Highly Specific, Unique Logic**: If a table requires a very specific, one-off security logic that doesn't apply to any other tables, creating a reusable policy might be overkill or less efficient than an inline view or specific grants.
    * **Situation**: A temporary table created for a very unique, short-lived analytical task that needs specialized, non-standard filtering.
2. **Performance Critical Workloads with Complex Policies**: While Snowflake optimizes policies, extremely complex policies involving many joins or subqueries within the policy logic, applied to very large, frequently queried tables, could introduce minor performance overhead.
    * **Situation**: A real-time dashboard or high-volume API endpoint that requires sub-millisecond response times and where the policy logic is excessively complex, potentially leading to noticeable latency.
3. **Policy Conflicts or Overlapping Requirements**: If different tables, despite sharing a common column, have fundamentally contradictory or non-overlapping security requirements that cannot be resolved within a single flexible policy, then separate policies might be clearer and safer.
    * **Situation**: Two tables both have a `STATUS` column, but one requires filtering for 'ACTIVE' status, while the other requires filtering for 'PENDING' status, and these requirements are driven by entirely different business rules and roles.

## Common Mistakes

1. **Ignoring Policy Precedence**: For columns, if both a Row Access Policy and a Masking Policy are applied, the Row Access Policy is evaluated first. For tag-based policies, direct column policies take precedence over tag policies. Not understanding this can lead to unexpected data exposure or over-masking.
2. **Lack of Testing with Different Roles**: Policies need to be thoroughly tested with all relevant roles to ensure they are working as intended and not inadvertently exposing sensitive data or overly restricting access.
3. **Complex Policy Logic**: Over-complicating the policy definition with too many conditions, joins, or subqueries can make it hard to read, maintain, and debug, and might impact query performance. Keep policies as simple and efficient as possible.
4. **Not Granting Proper Privileges**: The role creating the policy, applying the policy, and managing the mapping tables all need appropriate privileges. Missing grants can lead to errors or policies not being applied correctly.

## Expected Output

When a user with the `SALES_EAST_ANALYST` role queries `demo.sales.sales_data_east` and `demo.sales.sales_data_west`, the expected output will show only rows where the `REGION` column is 'East'.

Here's an example of what the user with `SALES_EAST_ANALYST` would see after querying `SELECT * FROM demo.sales.sales_data_east;`:

| CUSTOMER_ID | ORDER_ID | REGION | SALES_AMOUNT |
| :---------- | :------- | :----- | :----------- |
| 101 | 5001 | East | 1500.00 |
| 105 | 5003 | East | 2100.50 |

**Explanation of Columns:**
- `CUSTOMER_ID`: A unique identifier for the customer.
- `ORDER_ID`: A unique identifier for the sales order.
- `REGION`: The geographical region associated with the sales record. This column is used by the `region_policy` to filter results based on the user's role.
- `SALES_AMOUNT`: The total amount of the sale.

If the `SALES_EAST_ANALYST` role queries `demo.sales.sales_data_west`, they would see an empty result set, as the policy would filter out all rows not belonging to the 'East' region.
