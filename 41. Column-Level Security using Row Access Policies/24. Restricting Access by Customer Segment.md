# Restricting Access by Customer Segment in Snowflake

This concept addresses the crucial need to control which users can view specific data based on their customer segment. It ensures data privacy, compliance with regulations, and prevents unauthorized access to sensitive information, especially in multi-tenant environments or businesses with diverse customer bases.

**Short Explanation**

Restricting access by customer segment means setting up rules so that different groups of users (e.g., sales teams, marketing teams, or analysts) can only see data belonging to the specific customer segments they are authorized for. This prevents a user from seeing data for customer segments they shouldn't have access to, even if they have general access to the table.

- **What problem does this SQL feature solve?** It solves the problem of granular data access control, ensuring that sensitive customer data is only visible to authorized personnel based on their roles or the customer segments they manage.
- **Why is it important in databases or analytics?** It's critical for data governance, regulatory compliance (like GDPR or CCPA), and maintaining data integrity and confidentiality in complex data ecosystems.
- **Where is it commonly used in real workflows?** It's used in data warehousing, analytics platforms, CRM systems, and any application where different internal or external stakeholders need to interact with a subset of customer data.

## Implementation Example

```sql
CREATE OR REPLACE ROW ACCESS POLICY customer_segment_policy AS
(customer_segment VARCHAR) RETURNS BOOLEAN ->
  CURRENT_ROLE() IN ('ACCOUNT_ADMIN', 'SECURITY_ADMIN') OR
  EXISTS (
    SELECT 1 FROM your_data_sharing_db.information_schema.enabled_roles
    WHERE
      (role_name = 'SALES_EAST' AND customer_segment = 'East') OR
      (role_name = 'SALES_WEST' AND customer_segment = 'West') OR
      (role_name = 'MARKETING_B2B' AND customer_segment = 'B2B') OR
      (role_name = 'MARKETING_B2C' AND customer_segment = 'B2C') OR
      (role_name = CURRENT_ROLE() AND CURRENT_ROLE() = 'ANALYST_GLOBAL') -- Analyst can see all
  );

ALTER TABLE customers ADD ROW ACCESS POLICY customer_segment_policy ON (segment);
```

## Explanation of the Code

This code creates a Row Access Policy in Snowflake and then applies it to a table.

- `CREATE OR REPLACE ROW ACCESS POLICY customer_segment_policy AS (customer_segment VARCHAR) RETURNS BOOLEAN -> ...`: This defines a new row access policy named `customer_segment_policy`. It takes one argument, `customer_segment` (which will be mapped to a column in the table), and returns a `BOOLEAN` value. If the policy returns `TRUE`, the row is visible; if `FALSE`, it's hidden.
- `CURRENT_ROLE() IN ('ACCOUNT_ADMIN', 'SECURITY_ADMIN')`: This part of the policy grants full visibility to users with the `ACCOUNT_ADMIN` or `SECURITY_ADMIN` roles, acting as an override for administrative purposes.
- `EXISTS (SELECT 1 FROM your_data_sharing_db.information_schema.enabled_roles WHERE ...)`: This is the core logic. It checks if the `CURRENT_ROLE()` (the role of the user querying the table) matches certain conditions to determine access.
    - `(role_name = 'SALES_EAST' AND customer_segment = 'East')`: This condition allows users with the `SALES_EAST` role to see rows where the `customer_segment` column is 'East'.
    - `(role_name = 'SALES_WEST' AND customer_segment = 'West')`: Similar to above, but for `SALES_WEST` and 'West' segment.
    - `(role_name = 'MARKETING_B2B' AND customer_segment = 'B2B')`: Allows `MARKETING_B2B` role to see 'B2B' segment data.
    - `(role_name = 'MARKETING_B2C' AND customer_segment = 'B2C')`: Allows `MARKETING_B2C` role to see 'B2C' segment data.
    - `(role_name = CURRENT_ROLE() AND CURRENT_ROLE() = 'ANALYST_GLOBAL')`: This is an example of a role (`ANALYST_GLOBAL`) that can see all segments. By checking `CURRENT_ROLE()` against itself, it effectively allows any row for this specific role.
- `ALTER TABLE customers ADD ROW ACCESS POLICY customer_segment_policy ON (segment);`: This statement applies the `customer_segment_policy` to the `customers` table, specifically linking the `customer_segment` argument in the policy to the `segment` column in the `customers` table.

## When to Use

1.  **Multi-tenant applications:** When different client organizations or departments share the same database tables but must only see their own data.
    *   *Example scenario:* A SaaS platform where each customer company has its own data, and users from one company should never see data from another.
2.  **Regulatory compliance:** To meet data privacy regulations that mandate restricted access to sensitive data based on user roles or data classification.
    *   *Example scenario:* A healthcare provider needing to restrict patient record access to only authorized medical staff based on their specialty or assigned patient list.
3.  **Internal departmental segregation:** When different internal teams (e.g., regional sales, product marketing) need access to a common dataset but only for their specific areas of responsibility.
    *   *Example scenario:* A national sales team where regional managers should only see sales figures and customer details for their assigned region.

## When Not to Use

1.  **Simple table-level permissions are sufficient:** If access control only needs to be at the table or schema level (i.e., users either see all rows or none), standard `GRANT SELECT` statements are simpler and more efficient.
    *   *Inefficient scenario:* Restricting access to an entire 'audit_logs' table by user, when a simple `GRANT SELECT ON TABLE audit_logs TO ROLE auditor_role;` would suffice.
2.  **Performance is extremely critical for large tables with complex policies:** Row access policies add a filter to every query, which can introduce performance overhead. If the policy logic is very complex or the table is extremely large and queried frequently, the overhead might be noticeable.
    *   *Inefficient scenario:* Applying a row access policy with multiple complex subqueries to a fact table containing billions of rows, where every query becomes significantly slower.
3.  **Data separation is achieved via separate tables or schemas:** If customer data is already physically separated into different tables or schemas per segment, then row access policies on a single table might be redundant or unnecessary.
    *   *Redundant scenario:* Having separate `customers_east` and `customers_west` tables, but then also applying a row access policy to a unified `customers` view that combines them, when direct table access could be managed.

## Common Mistakes

1.  **Forgetting to grant policy privileges:** Users might create a policy but forget to grant the necessary `APPLY ROW ACCESS POLICY` privilege to the roles that need to use it.
2.  **Overly complex policy logic:** Creating policies with many `OR` conditions or subqueries can make them difficult to debug, maintain, and can negatively impact query performance.
3.  **Not testing with different roles:** Developers often test with an `ACCOUNT_ADMIN` role (which bypasses policies) and assume the policy works, failing to test with the actual user roles.
4.  **Incorrect column mapping:** Mapping the policy argument to the wrong column in the `ALTER TABLE` statement, leading to incorrect filtering or no filtering at all.
5.  **Hardcoding values instead of using functions/roles:** Statically embedding role names or segment values directly in the policy without considering future changes or dynamic role assignments.

## Expected Output

When a user with the `SALES_EAST` role queries the `customers` table, they will only see rows where the `segment` column is 'East'. All other rows will be invisibly filtered out.

**Example Table `customers` (as seen by `SALES_EAST` role):**

| customer_id | name        | segment | revenue |
|-------------|-------------|---------|---------|
| C001        | Alice Inc.  | East    | 15000   |
| C004        | David Corp. | East    | 22000   |
| C007        | Frank Ltd.  | East    | 18000   |

- **customer_id:** A unique identifier for the customer.
- **name:** The name of the customer.
- **segment:** The customer segment (e.g., 'East', 'West', 'B2B', 'B2C'), which is used by the row access policy for filtering.
- **revenue:** Financial data related to the customer.
