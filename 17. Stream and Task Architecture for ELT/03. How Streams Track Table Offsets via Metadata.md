# How Streams Track Table Offsets via Metadata
Streams track table offsets via metadata to ensure data consistency and accuracy during data replication and integration processes. This is achieved by storing metadata about the current position of the stream in the table, allowing the stream to resume from the correct position in case of failures or restarts. The metadata includes information such as the last read offset, last written offset, and any pending transactions. By tracking table offsets via metadata, streams can ensure that data is processed in the correct order and that no data is lost or duplicated. This is particularly important in distributed systems where data is being processed in parallel across multiple nodes.

## Implementation Example
```sql
CREATE TABLE metadata (
    id SERIAL PRIMARY KEY,
    stream_name VARCHAR(50),
    table_name VARCHAR(50),
    last_read_offset BIGINT,
    last_written_offset BIGINT,
    pending_transactions JSONB
);

INSERT INTO metadata (stream_name, table_name, last_read_offset, last_written_offset, pending_transactions)
VALUES ('my_stream', 'my_table', 0, 0, '[]'::jsonb);

CREATE FUNCTION update_metadata(p_stream_name VARCHAR(50), p_table_name VARCHAR(50), p_last_read_offset BIGINT, p_last_written_offset BIGINT, p_pending_transactions JSONB)
RETURNS VOID AS $$
BEGIN
    UPDATE metadata
    SET last_read_offset = p_last_read_offset,
        last_written_offset = p_last_written_offset,
        pending_transactions = p_pending_transactions
    WHERE stream_name = p_stream_name AND table_name = p_table_name;
END;
$$ LANGUAGE plpgsql;
```
Few examples and alternatives of SQL Code include using a separate table for each stream, or using a single table with a composite primary key.

## Explanation of the Code
* The `metadata` table is created to store information about the current position of each stream in the table.
* The `id` column is a unique identifier for each row in the table.
* The `stream_name` and `table_name` columns identify the stream and table that the metadata is for.
* The `last_read_offset` and `last_written_offset` columns store the last read and written offsets for the stream.
* The `pending_transactions` column stores any pending transactions for the stream.
* The `update_metadata` function is used to update the metadata for a given stream and table.
* The function takes in the stream name, table name, last read offset, last written offset, and pending transactions as parameters.

## When to use
This approach is useful in the following use cases:
* **Data replication**: When replicating data from one table to another, tracking table offsets via metadata ensures that data is processed in the correct order and that no data is lost or duplicated.
* **Data integration**: When integrating data from multiple sources, tracking table offsets via metadata ensures that data is processed in the correct order and that no data is lost or duplicated.

## When not to use
This approach is not suitable for the following use cases:
* **Real-time data processing**: When processing data in real-time, tracking table offsets via metadata may introduce additional latency and overhead.
* **Simple data transfer**: When transferring data from one table to another without any processing or transformation, tracking table offsets via metadata may not be necessary.

## Common Mistakes
* Not updating the metadata correctly after processing data.
* Not handling failures and restarts correctly.
* Not considering the impact of pending transactions on the metadata.
* Not using the correct data types for the metadata columns.

## Expected Output
The expected output of the `metadata` table would be:
| id | stream_name | table_name | last_read_offset | last_written_offset | pending_transactions |
| --- | --- | --- | --- | --- | --- |
| 1 | my_stream | my_table | 100 | 100 | [] |
The `last_read_offset` and `last_written_offset` columns would be updated as data is processed, and the `pending_transactions` column would store any pending transactions for the stream.
