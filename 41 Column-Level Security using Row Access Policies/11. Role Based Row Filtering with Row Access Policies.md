# Role Based Row Filtering with Row Access Policies in Snowflake

Role-based row filtering with Row Access Policies in Snowflake is a security mechanism that allows organizations to control which rows a user can see in a table based on their assigned role. It ensures data privacy and compliance by applying a dynamic filter at query time, without modifying the underlying data or requiring complex view management. This feature exists to provide fine-grained, centralized access control over sensitive data, allowing different users to see different subsets of the same table.

**Short Explanation**

Row Access Policies in Snowflake let you set rules that automatically hide certain rows from users based on their role. This solves the problem of needing to show different data to different people from the same table, ensuring data privacy and compliance without creating multiple versions of the table. It's important in databases and analytics for maintaining strict data governance, especially in industries with sensitive information, and is commonly used in scenarios like financial reporting, healthcare data access, and multi-tenant applications to restrict what specific users or departments can view.

-   **What problem does this SQL feature solve?** It solves the problem of implementing fine-grained, dynamic row-level security, ensuring users only see the data they are authorized for, based on their role.
-   **Why is it important in databases or analytics?** It's crucial for data governance, regulatory compliance (e.g., GDPR, HIPAA), and protecting sensitive information by centralizing access control logic and simplifying data sharing.
-   **Where is it commonly used in real workflows?** It's used in scenarios like multi-tenant applications where each tenant should only see their own data, financial systems restricting access to specific transactions, or HR systems limiting employee data visibility to relevant managers.

## Implementation Example

```sql
-- 1. Create a sample table with sensitive data
CREATE TABLE sales_data (
    sale_id INT,
    product_category VARCHAR,
    region VARCHAR,
    sale_amount DECIMAL(10, 2),
    customer_segment VARCHAR
);

INSERT INTO sales_data VALUES
(1, 'Electronics', 'North', 1500.00, 'Premium'),
(2, 'Clothing', 'South', 300.00, 'Standard'),
(3, 'Electronics', 'West', 2000.00, 'Premium'),
(4, 'Home Goods', 'East', 500.00, 'Standard'),
(5, 'Electronics', 'North', 1200.00, 'Standard'),
(6, 'Clothing', 'West', 450.00, 'Premium');

-- 2. Create roles for different access levels
CREATE ROLE sales_north_analyst;
CREATE ROLE sales_west_analyst;
CREATE ROLE sales_all_analyst;

-- 3. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy
AS (region_col VARCHAR) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'SALES_NORTH_ANALYST' THEN region_col = 'North'
        WHEN CURRENT_ROLE() = 'SALES_WEST_ANALYST'  THEN region_col = 'West'
        WHEN CURRENT_ROLE() = 'SALES_ALL_ANALYST'   THEN TRUE
        ELSE FALSE
    END;

-- 4. Apply the policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region);

-- 5. Grant necessary permissions to roles
GRANT SELECT ON TABLE sales_data TO ROLE sales_north_analyst;
GRANT SELECT ON TABLE sales_data TO ROLE sales_west_analyst;
GRANT SELECT ON TABLE sales_data TO ROLE sales_all_analyst;

-- 6. Test the policy with different roles
-- Test 1: As SALES_NORTH_ANALYST
-- USE ROLE SALES_NORTH_ANALYST;
-- SELECT * FROM sales_data;

-- Test 2: As SALES_WEST_ANALYST
-- USE ROLE SALES_WEST_ANALYST;
-- SELECT * FROM sales_data;

-- Test 3: As SALES_ALL_ANALYST
-- USE ROLE SALES_ALL_ANALYST;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

The example demonstrates how to implement row-level security based on the user's role.

-   **`CREATE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to simulate a business dataset with a `region` column that will be used for filtering.
-   **`INSERT INTO sales_data VALUES (...)`**: This populates the `sales_data` table with example sales records across different regions.
-   **`CREATE ROLE sales_north_analyst; ...`**: These statements create three distinct roles (`sales_north_analyst`, `sales_west_analyst`, `sales_all_analyst`) that will have different levels of data access.
-   **`CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy AS (region_col VARCHAR) RETURNS BOOLEAN -> ...`**: This is the core of the row-level security.
    -   `CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy`: Defines a new row access policy named `sales_region_policy`.
    -   `AS (region_col VARCHAR) RETURNS BOOLEAN`: Specifies that the policy takes one argument, `region_col` (which will be mapped to a column in the table), and returns a boolean value.
    -   `-> CASE WHEN CURRENT_ROLE() = 'SALES_NORTH_ANALYST' THEN region_col = 'North' ... ELSE FALSE END`: This is the SQL expression that defines the filtering logic. It checks the `CURRENT_ROLE()` of the user querying the table:
        -   If the role is `SALES_NORTH_ANALYST`, it returns `TRUE` only for rows where `region_col` is 'North'.
        -   If the role is `SALES_WEST_ANALYST`, it returns `TRUE` only for rows where `region_col` is 'West'.
        -   If the role is `SALES_ALL_ANALYST`, it returns `TRUE` for all rows (no filter).
        -   For any other role, it returns `FALSE`, effectively showing no rows.
-   **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region);`**: This statement applies the `sales_region_policy` to the `sales_data` table, specifically linking the policy's `region_col` argument to the `region` column of the table. From this point on, any `SELECT` query on `sales_data` will automatically have this policy applied.
-   **`GRANT SELECT ON TABLE sales_data TO ROLE sales_north_analyst; ...`**: These statements grant the necessary `SELECT` privilege on the `sales_data` table to each of the created roles.
-   **`USE ROLE ...; SELECT * FROM sales_data;`**: These commented-out lines demonstrate how to switch roles and then query the table to observe the policy in action, showing different results based on the active role.

## When to Use

1.  **Multi-tenant Data Segmentation**: When you have a single table storing data for multiple tenants (e.g., customers, departments), and each tenant should only see their own data.
    *   **Example Scenario**: An e-commerce platform stores orders for all its vendors in one `ORDERS` table. A row access policy ensures Vendor A can only see orders where `vendor_id = 'Vendor A ID'`, and Vendor B only sees orders where `vendor_id = 'Vendor B ID'`.
2.  **Regulatory Compliance (e.g., HIPAA, GDPR)**: To restrict access to sensitive personal or health information based on user roles and consent.
    *   **Example Scenario**: A healthcare database contains patient records. Doctors might see full records for their patients, while billing staff only see billing-related information, and researchers see anonymized data or aggregated statistics, achieved by filtering rows based on a `patient_consent_level` column and the user's role.
3.  **Internal Departmental Data Access**: To manage internal access to data, ensuring that only employees in specific departments can view relevant operational data.
    *   **Example Scenario**: A company's `EMPLOYEE_PERFORMANCE` table contains confidential data. A policy allows only HR managers to see all employee data, while team leads can only see data for their direct reports, based on matching `manager_id` or `department_id` with the querying user's role.

## When Not to Use

1.  **Column-level Security Needs**: When the requirement is to hide specific columns rather than entire rows. Row Access Policies only filter rows; they cannot mask or hide columns.
    *   **Inefficiency**: Applying a row access policy for a column-level security need would be complex and inefficient, possibly requiring multiple policies or views instead of simpler Column-level Security or Dynamic Data Masking.
2.  **Performance-critical Queries with Simple Filters**: For very high-volume, performance-critical queries where a simple `WHERE` clause on an indexed column would suffice and role-based filtering is not strictly necessary.
    *   **Overhead**: Row access policies introduce a slight overhead due to the function evaluation at query time. For basic filters that don't depend on the user's role, a standard `WHERE` clause is more performant.
3.  **When Data Access Logic is Highly Dynamic or User-Driven**: If the access logic changes frequently and is determined by user-specific, granular permissions that are not easily mapped to a fixed set of roles.
    *   **Complexity**: The policy logic can become overly complex and difficult to manage if it needs to account for too many dynamic variables or individual user exceptions rather than stable role-based rules.

## Common Mistakes

1.  **Not granting `SELECT` on the table to the role**: Even with a policy, if the user's role doesn't have `SELECT` privilege on the table, they won't be able to query it at all.
2.  **Overly complex policy logic**: Writing policies with too many `CASE` statements or complex subqueries can lead to performance degradation and make the policy hard to debug and maintain.
3.  **Forgetting to test with different roles**: Developers often test with an admin role or the role that created the policy, which might not be subject to the policy (or sees all data), leading to a false sense of security. Always test explicitly with the intended restricted roles.
4.  **Not understanding policy precedence**: If multiple policies are applied or combined, understanding their interaction and precedence is crucial to avoid unintended data exposure or denial of access.
5.  **Policy parameter mismatch**: Mapping the policy argument to the wrong table column, or using a data type for the policy argument that doesn't match the table column's data type.

## Expected Output

The resulting dataset will vary based on the `CURRENT_ROLE()` executing the `SELECT` query. The policy dynamically filters rows *before* they are returned to the user, so the user only sees authorized rows.

For example, if `SALES_NORTH_ANALYST` queries `sales_data`:

| SALE_ID | PRODUCT_CATEGORY | REGION | SALE_AMOUNT | CUSTOMER_SEGMENT |
| :------ | :--------------- | :----- | :---------- | :--------------- |
| 1       | Electronics      | North  | 1500.00     | Premium          |
| 5       | Electronics      | North  | 1200.00     | Standard         |

-   **SALE_ID**: Unique identifier for the sale.
-   **PRODUCT_CATEGORY**: The type of product sold.
-   **REGION**: The geographical region where the sale occurred. This is the column filtered by the policy.
-   **SALE_AMOUNT**: The monetary value of the sale.
-   **CUSTOMER_SEGMENT**: The category of the customer.

If `SALES_WEST_ANALYST` queries `sales_data`, they would only see:

| SALE_ID | PRODUCT_CATEGORY | REGION | SALE_AMOUNT | CUSTOMER_SEGMENT |
| :------ | :--------------- | :----- | :---------- | :--------------- |
| 3       | Electronics      | West   | 2000.00     | Premium          |
| 6       | Clothing         | West   | 450.00      | Premium          |

If `SALES_ALL_ANALYST` queries `sales_data`, they would see all six rows, as their role is permitted to see all data.
