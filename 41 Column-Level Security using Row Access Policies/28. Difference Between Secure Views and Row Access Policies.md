# Combining Row Access Policies with RBAC in Snowflake

This concept explores how to integrate Snowflake's Row Access Policies (RAPs) with Role-Based Access Control (RBAC) to create a robust and fine-grained data security model. It allows organizations to restrict data visibility at the row level based on a user's role, ensuring that users only see the data they are authorized to access, while leveraging the established hierarchical nature of RBAC.

**Short Explanation**

Combining Row Access Policies with RBAC means using a user's assigned roles to determine which rows of data they can see. This feature solves the problem of needing to hide specific data rows from certain users, even if they have access to the table, ensuring data privacy and compliance. It's important in databases and analytics for implementing highly granular security without duplicating data or creating complex views for every access scenario. This combination is commonly used in multi-tenant applications, healthcare systems, financial institutions, and any environment where data access must be strictly controlled based on user attributes or roles.

- What problem does this SQL feature solve? It solves the problem of implementing fine-grained, row-level security dynamically, ensuring that data access is restricted based on a user's role without hardcoding permissions into applications or views.
- Why is it important in databases or analytics? It's crucial for compliance with regulations (like GDPR, HIPAA), protecting sensitive information, and preventing unauthorized data exposure, all while maintaining a single source of truth for the data.
- Where is it commonly used in real workflows? It's used in scenarios like sales teams only seeing their own region's sales data, HR staff only accessing employee records for their department, or customers in a multi-tenant application only seeing their own company's data.

## Implementation Example

```sql
-- 1. Create a table with sensitive data
CREATE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sales_amount DECIMAL(10, 2),
    sensitive_info VARCHAR
);

-- Insert some sample data
INSERT INTO sales_data VALUES
(1, 'Laptop', 'North', 1500.00, 'Confidential A'),
(2, 'Mouse', 'South', 25.00, 'Confidential B'),
(3, 'Keyboard', 'East', 75.00, 'Confidential C'),
(4, 'Monitor', 'North', 300.00, 'Confidential D'),
(5, 'Webcam', 'West', 50.00, 'Confidential E');

-- 2. Create roles for different regions
CREATE ROLE north_sales_role;
CREATE ROLE south_sales_role;
CREATE ROLE east_sales_role;

-- Grant SELECT on the table to these roles
GRANT SELECT ON TABLE sales_data TO ROLE north_sales_role;
GRANT SELECT ON TABLE sales_data TO ROLE south_sales_role;
GRANT SELECT ON TABLE sales_data TO ROLE east_sales_role;

-- 3. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_col VARCHAR) RETURNS BOOLEAN ->
    CASE
        WHEN CURRENT_ROLE() = 'NORTH_SALES_ROLE' AND region_col = 'North' THEN TRUE
        WHEN CURRENT_ROLE() = 'SOUTH_SALES_ROLE' AND region_col = 'South' THEN TRUE
        WHEN CURRENT_ROLE() = 'EAST_SALES_ROLE' AND region_col = 'East' THEN TRUE
        WHEN IS_ROLE_IN_SESSION('ACCOUNTADMIN') THEN TRUE -- Allow account admin to see all
        ELSE FALSE
    END;

-- 4. Apply the Row Access Policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);

-- 5. Create users and grant roles
CREATE USER user_north PASSWORD='Password123' DEFAULT_ROLE=north_sales_role;
CREATE USER user_south PASSWORD='Password123' DEFAULT_ROLE=south_sales_role;
CREATE USER user_admin PASSWORD='Password123' DEFAULT_ROLE=accountadmin;

GRANT ROLE north_sales_role TO USER user_north;
GRANT ROLE south_sales_role TO USER user_south;
GRANT ROLE accountadmin TO USER user_admin;

-- Test as user_north (should only see 'North' data)
-- USE ROLE NORTH_SALES_ROLE;
-- SELECT * FROM sales_data;

-- Test as user_south (should only see 'South' data)
-- USE ROLE SOUTH_SALES_ROLE;
-- SELECT * FROM sales_data;

-- Test as user_admin (should see all data)
-- USE ROLE ACCOUNTADMIN;
-- SELECT * FROM sales_data;
```

## Explanation of the Code

- **`CREATE TABLE sales_data (...)`**: This statement creates a sample table named `sales_data` to hold sales information, including a `region` column which will be used for row-level filtering.
- **`INSERT INTO sales_data VALUES (...)`**: This populates the `sales_data` table with example records across different regions.
- **`CREATE ROLE ...`**: These statements create distinct Snowflake roles (e.g., `north_sales_role`) representing different access groups.
- **`GRANT SELECT ON TABLE sales_data TO ROLE ...`**: This grants the basic `SELECT` privilege on the `sales_data` table to each regional role. Without a Row Access Policy, users with these roles would see all rows.
- **`CREATE OR REPLACE ROW ACCESS POLICY sales_policy AS (region_col VARCHAR) RETURNS BOOLEAN -> ...`**:
    - This is the core of the row-level security. It defines a policy named `sales_policy`.
    - `AS (region_col VARCHAR)`: Specifies that the policy will receive one argument, `region_col`, which will hold the value of the `region` column from the table.
    - `RETURNS BOOLEAN -> ...`: Indicates that the policy function will return a boolean value (`TRUE` to show the row, `FALSE` to hide it).
    - `CASE WHEN CURRENT_ROLE() = 'NORTH_SALES_ROLE' AND region_col = 'North' THEN TRUE ...`: This is the logic that combines RBAC with row access. It checks the role of the user (`CURRENT_ROLE()`) and compares it to the `region_col` value for each row. If the role matches the region (e.g., `NORTH_SALES_ROLE` trying to access `'North'` region data), it returns `TRUE`, allowing the row to be seen.
    - `WHEN IS_ROLE_IN_SESSION('ACCOUNTADMIN') THEN TRUE`: This line explicitly allows the `ACCOUNTADMIN` role to bypass the policy and see all rows, which is a common practice for administrative oversight.
    - `ELSE FALSE`: If none of the conditions are met, the row is hidden.
- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_policy ON (region);`**: This statement applies the `sales_policy` to the `sales_data` table, specifically linking it to the `region` column. From this point forward, any `SELECT` query on `sales_data` will implicitly invoke this policy.
- **`CREATE USER ...`**: These statements create sample users.
- **`GRANT ROLE ... TO USER ...`**: This assigns the previously created roles to the respective users, establishing their access permissions within the RBAC framework.

## When to Use

1.  **Multi-tenant Applications**: When different customers share the same underlying data table but should only see their own data.
    *   *Example Scenario*: An e-commerce platform where `orders` data for multiple vendors is stored in one table. Each vendor's user should only see their specific orders.
2.  **Regulatory Compliance (e.g., HIPAA, GDPR)**: To ensure sensitive data (like patient health information or personal identifiable information) is only visible to authorized personnel.
    *   *Example Scenario*: A healthcare database where doctors can only see records of their own patients, and administrative staff only see aggregated, anonymized data.
3.  **Internal Departmental Segregation**: To limit employee access to data relevant to their department or geographical region.
    *   *Example Scenario*: A large corporation where sales managers in the "East" region can only view sales figures for the East, while "West" managers only see West figures, all within the same `company_sales` table.

## When Not to Use

1.  **Basic Table/Schema Level Permissions Suffice**: If access control is only needed at a broader level (e.g., a user either sees all data in a table or none).
    *   *Situation*: A small team where everyone has full access to project data, and there's no need for row-specific restrictions. Using RAPs would add unnecessary complexity.
2.  **Performance-Critical OLTP Workloads with High Latency Sensitivity**: Row access policies add overhead, as they are evaluated for every query. For extremely high-throughput, low-latency transactional systems, this overhead might be unacceptable.
    *   *Situation*: A real-time bidding platform where every millisecond counts, and the security requirements don't warrant row-level filtering based on roles.
3.  **Complex, Deeply Nested Policy Logic**: If the policy logic becomes extremely convoluted with many nested `CASE` statements and `OR` conditions involving multiple columns and external lookups, it can be difficult to maintain, debug, and might lead to performance degradation.
    *   *Situation*: A policy that needs to check 10 different attributes from 5 different lookup tables for each row to determine visibility, making the policy function very expensive to execute.

## Common Mistakes

1.  **Forgetting to Grant Privileges to the Policy Owner**: The role that owns the Row Access Policy must have `SELECT` privileges on any tables or secure views referenced within the policy's body (e.g., if the policy references a lookup table).
2.  **Not Handling All Roles/Conditions in Policy Logic**: If a policy doesn't explicitly define access for a certain role or scenario, users in that role might unexpectedly see no data or all data, leading to security gaps or access denial issues. Including a default `ELSE FALSE` or an explicit admin bypass is crucial.
3.  **Overlooking Performance Impacts**: Applying RAPs to very large tables with frequent queries, especially with complex policy logic, can introduce noticeable query latency if not optimized. Performance should be tested.
4.  **Applying Policies to Incorrect Columns**: Attaching the policy to the wrong column (e.g., applying a regional policy on a `product_id` column instead of a `region` column).
5.  **Not Testing with Different Roles**: Developers often test with high-privileged roles (like `ACCOUNTADMIN`), which bypass policies. It's essential to thoroughly test with low-privileged, specific roles to ensure the policy behaves as intended.

## Expected Output

The resulting dataset will vary based on the `CURRENT_ROLE()` of the user executing the `SELECT` statement.

If `user_north` (with `NORTH_SALES_ROLE`) queries `sales_data`:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT | SENSITIVE_INFO |
| :------ | :----------- | :----- | :----------- | :------------- |
| 1       | Laptop       | North  | 1500.00      | Confidential A |
| 4       | Monitor      | North  | 300.00       | Confidential D |

*Explanation*: Only rows where the `REGION` column is 'North' are returned, because the `NORTH_SALES_ROLE` is only permitted to see data from that region according to the `sales_policy`.

If `user_south` (with `SOUTH_SALES_ROLE`) queries `sales_data`:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT | SENSITIVE_INFO |
| :------ | :----------- | :----- | :----------- | :------------- |
| 2       | Mouse        | South  | 25.00        | Confidential B |

*Explanation*: Only rows where the `REGION` column is 'South' are returned, as per the policy logic for `SOUTH_SALES_ROLE`.

If `user_admin` (with `ACCOUNTADMIN` role) queries `sales_data`:

| SALE_ID | PRODUCT_NAME | REGION | SALES_AMOUNT | SENSITIVE_INFO |
| :------ | :----------- | :----- | :----------- | :------------- |
| 1       | Laptop       | North  | 1500.00      | Confidential A |
| 2       | Mouse        | South  | 25.00        | Confidential B |
| 3       | Keyboard     | East   | 75.00        | Confidential C |
| 4       | Monitor      | North  | 300.00       | Confidential D |
| 5       | Webcam       | West   | 50.00        | Confidential E |

*Explanation*: All rows are returned because the `ACCOUNTADMIN` role is explicitly granted full access within the `sales_policy`.

The columns in the output will always be the same as the underlying table, but the number of rows (and the specific data within those rows) will be filtered based on the active role and the logic defined in the Row Access Policy.
