# Using Policies in Data Sharing Scenarios in Snowflake

This concept explores how Snowflake data governance features, specifically policies (like row access policies and masking policies), can be effectively applied to data shares. This allows data providers to maintain fine-grained control over the data being shared with consumers, ensuring compliance and security even when data leaves the provider's direct environment. It exists to enable secure, controlled, and compliant data sharing without compromising data privacy or regulatory requirements.

**Short Explanation**

Using policies in data sharing scenarios allows data providers to dynamically restrict access to rows (row access policies) or mask sensitive columns (masking policies) for data consumers. This means the provider can share a single dataset, but different consumers see different versions of the data based on their roles or other attributes.

- **What problem does this SQL feature solve?** It solves the problem of needing to share data securely and compliantly, where different consumers might have different access rights or requirements for sensitive information, without having to create multiple copies or views of the data.
- **Why is it important in databases or analytics?** It's crucial for maintaining data governance, privacy, and regulatory compliance (like GDPR, HIPAA) when distributing data to various internal or external stakeholders, enabling secure collaboration.
- **Where is it commonly used in real workflows?** It's used in industries like healthcare for sharing patient data while masking PII, in financial services for sharing transaction data with different levels of detail, or in any multi-tenant application where different customers access a shared data pool with specific restrictions.

## Implementation Example

```sql
-- 1. Create a Masking Policy (Data Provider side)
CREATE MASKING POLICY email_mask_policy AS (val STRING) RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() = 'ANALYST_ROLE' THEN val
    WHEN CURRENT_ROLE() = 'SALES_ROLE' THEN '******' || SUBSTR(val, INSTR(val, '@'))
    ELSE '**********'
  END;

-- 2. Apply the Masking Policy to a column (Data Provider side)
ALTER TABLE sales_db.public.customer
  MODIFY COLUMN email SET MASKING POLICY email_mask_policy;

-- 3. Create a Row Access Policy (Data Provider side)
CREATE ROW ACCESS POLICY region_access_policy AS (region STRING) RETURNS BOOLEAN ->
  CASE
    WHEN CURRENT_ROLE() = 'GLOBAL_ADMIN' THEN TRUE
    WHEN CURRENT_ROLE() = 'NORTH_AMERICA_SALES' THEN region = 'NA'
    WHEN CURRENT_ROLE() = 'EUROPE_SALES' THEN region = 'EU'
    ELSE FALSE
  END;

-- 4. Apply the Row Access Policy to a table (Data Provider side)
ALTER TABLE sales_db.public.customer
  ADD ROW ACCESS POLICY region_access_policy ON (customer_region);

-- 5. Grant necessary privileges for policies (Data Provider side)
GRANT APPLY MASKING POLICY ON ACCOUNT TO DATA_SHARING_ROLE;
GRANT APPLY ROW ACCESS POLICY ON ACCOUNT TO DATA_SHARING_ROLE;

-- 6. Create a secure view that references the policy-protected table (Data Provider side)
CREATE SECURE VIEW sales_db.public.customer_share_view AS
SELECT customer_id, customer_name, email, customer_region FROM sales_db.public.customer;

-- 7. Create a share and add the secure view (Data Provider side)
CREATE SHARE customer_data_share;
GRANT USAGE ON DATABASE sales_db TO SHARE customer_data_share;
GRANT USAGE ON SCHEMA sales_db.public TO SHARE customer_data_share;
GRANT SELECT ON VIEW sales_db.public.customer_share_view TO SHARE customer_data_share;

-- 8. Add consumer account to the share (Data Provider side)
ALTER SHARE customer_data_share ADD ACCOUNTS = <consumer_account_locator>;

-- 9. (Consumer side) Create database from share
CREATE DATABASE shared_customer_data FROM SHARE <provider_account_locator>.customer_data_share;

-- 10. (Consumer side) Grant access to roles
GRANT IMPORTED PRIVILEGES ON DATABASE shared_customer_data TO ANALYST_ROLE;
GRANT IMPORTED PRIVILEGES ON DATABASE shared_customer_data TO SALES_ROLE;

-- 11. (Consumer side) Query the shared view as different roles
-- Use role ANALYST_ROLE;
-- SELECT * FROM shared_customer_data.public.customer_share_view; -- Email visible, all regions
-- Use role SALES_ROLE;
-- SELECT * FROM shared_customer_data.public.customer_share_view; -- Email masked, no rows if region policy applies to sales role
```

## Explanation of the Code

This example demonstrates how policies affect data shared through secure views.

- **`CREATE MASKING POLICY email_mask_policy`**: This DDL statement defines a masking policy named `email_mask_policy`.
    - **What it does**: It specifies logic for how the `val` (column value) should be transformed based on the `CURRENT_ROLE()` of the user querying the data.
    - **How it changes the result set**: When applied, it dynamically masks the `email` column, showing the full email only to `ANALYST_ROLE`, a partial mask to `SALES_ROLE`, and a full mask to others.

- **`ALTER TABLE sales_db.public.customer MODIFY COLUMN email SET MASKING POLICY email_mask_policy`**:
    - **What it does**: This applies the `email_mask_policy` to the `email` column of the `customer` table.
    - **How it changes the result set**: From this point on, any query directly on `customer.email` or on a view built on it will have the masking policy enforced.

- **`CREATE ROW ACCESS POLICY region_access_policy`**: This DDL statement defines a row access policy named `region_access_policy`.
    - **What it does**: It defines a boolean condition based on the `region` column that determines whether a row should be returned.
    - **How it changes the result set**: It filters rows based on the `CURRENT_ROLE()`, allowing `GLOBAL_ADMIN` to see all rows, `NORTH_AMERICA_SALES` to see only 'NA' region rows, `EUROPE_SALES` to see 'EU' region rows, and others to see no rows.

- **`ALTER TABLE sales_db.public.customer ADD ROW ACCESS POLICY region_access_policy ON (customer_region)`**:
    - **What it does**: This applies the `region_access_policy` to the `customer` table, referencing the `customer_region` column.
    - **How it changes the result set**: Any query on the `customer` table or a view built on it will have rows filtered according to this policy.

- **`CREATE SECURE VIEW sales_db.public.customer_share_view AS SELECT ...`**:
    - **What it does**: This creates a secure view that exposes specific columns from the `customer` table. Secure views encapsulate underlying policies, meaning the consumer doesn't see the base table or the policy definitions.
    - **How it changes the result set**: The view itself doesn't directly change the result set from the base table, but it's the gateway through which policies applied to the base table are enforced for consumers.

- **`CREATE SHARE customer_data_share; GRANT USAGE ON DATABASE/SCHEMA/VIEW TO SHARE; ALTER SHARE ... ADD ACCOUNTS`**:
    - **What it does**: These commands create a data share, grant necessary privileges for the database, schema, and the secure view to the share, and then add a consumer account to receive the share.
    - **How it changes the result set**: These actions make the shared view accessible to the consumer account, but the data seen by the consumer will still be subject to the policies defined by the provider based on the roles assigned to the consumer.

- **`CREATE DATABASE shared_customer_data FROM SHARE ...; GRANT IMPORTED PRIVILEGES ... TO ANALYST_ROLE/SALES_ROLE`**:
    - **What it does**: On the consumer side, these commands create a database from the shared data and grant roles within the consumer account access to this shared database.
    - **How it changes the result set**: When a consumer user queries the `customer_share_view` through these roles, the policies defined by the provider will evaluate against the `CURRENT_ROLE()` in the *consumer's account*, dynamically adjusting the visible data.

## When to Use

1.  **Compliance-driven data sharing**: When sharing sensitive data (e.g., PII, financial records, health information) with various external partners or internal departments, and strict regulatory compliance (GDPR, HIPAA, CCPA) requires varying levels of access or anonymization based on the consumer's authorization.
    *   *Example scenario:* A healthcare provider shares patient demographic data with a research institution but needs to mask patient names and IDs while allowing researchers to see anonymized health conditions.
2.  **Multi-tenant applications or platforms**: For a product that serves multiple customers (tenants) using a single underlying dataset, but each customer should only see their own data, or specific aggregated/masked views of shared data.
    *   *Example scenario:* A SaaS analytics platform offers services to multiple companies. Each company uses the same dashboard structure, but behind the scenes, row access policies ensure they only see data relevant to their own organization.
3.  **Tiered data access for internal users**: Within a large organization, sharing a central dataset where different internal teams (e.g., sales, marketing, finance, legal) require access to the same base data but with different restrictions or levels of detail.
    *   *Example scenario:* A global sales team shares a customer database. Regional sales managers can only see customer data for their specific region (row access policy), while marketing might see all customer data but with sensitive contact information masked (masking policy).

## When Not to Use

1.  **Simple, unrestricted data sharing**: If the data being shared is non-sensitive and all consumers require full, unrestricted access to the exact same dataset, adding policies introduces unnecessary complexity and overhead.
    *   *Inefficiency:* Creating and maintaining policies for public, non-sensitive data adds administrative burden without security benefits.
2.  **Performance-critical queries where policies cause significant overhead**: While Snowflake policies are highly optimized, extremely complex policy logic or a very large number of overlapping policies on frequently queried large tables could potentially introduce minor performance overhead. If raw speed is paramount and security can be handled at a higher architectural level (e.g., separate databases for different access levels), policies might be overkill.
    *   *Inefficiency:* For reporting systems demanding sub-second query times on petabytes of data with minimal security concerns, the slight overhead of policy evaluation might be avoided.
3.  **When data transformation or aggregation is the primary goal**: If the goal is not just to restrict data but to fundamentally transform, aggregate, or derive new data points for consumers, then a separate ETL/ELT process or materialized views might be more appropriate. Policies are for *filtering* and *masking*, not for arbitrary data manipulation.
    *   *Wrong approach:* If consumers need aggregated sales totals by region, rather than row-level access, policies aren't the right tool; aggregated views or tables are.

## Common Mistakes

1.  **Forgetting to grant `APPLY MASKING POLICY` or `APPLY ROW ACCESS POLICY` privileges**: The role responsible for creating the share (or the owner role of the database/schema containing the policy-protected objects) must have these global privileges to ensure the policies are enforced for shared objects. Without them, policies won't work correctly for shared data.
2.  **Not using a secure view for sharing**: When sharing data with policies, always share a *secure view* built on top of the policy-protected tables. If you share the base table directly, the policy definitions are visible to the consumer, compromising the security by obscurity aspect and potentially revealing sensitive logic.
3.  **Incorrectly assuming policies evaluate on the provider's context**: Policies (`CURRENT_ROLE()`, `CURRENT_USER()`, `INVOKER_ROLE()`, etc.) are evaluated *in the consumer's account context* when the consumer queries the shared data. A common mistake is to define policies expecting `CURRENT_ROLE()` to refer to a role in the provider's account. Providers must communicate the expected role names or consumer-specific attributes that their policies rely on.
4.  **Over-complicating policy logic**: Creating overly complex `CASE` statements or subqueries within policies can make them hard to debug, maintain, and potentially impact performance. Keep policy logic as simple and efficient as possible.
5.  **Forgetting to grant `SELECT` on the secure view to the share**: If the `SELECT` privilege on the secure view is not granted to the share, the consumer will not be able to access the shared data.

## Expected Output

The resulting dataset will vary dynamically based on the role used by the consumer to query the `customer_share_view`. The number of rows returned and the visibility of the `email` column will change.

### Example Table (when queried by `ANALYST_ROLE` in the consumer account)

This role has full access to emails and all regions.

| CUSTOMER_ID | CUSTOMER_NAME | EMAIL                 | CUSTOMER_REGION |
|-------------|---------------|-----------------------|-----------------|
| 101         | Alice Johnson | alice.j@example.com   | NA              |
| 102         | Bob Smith     | bob.s@test.org        | EU              |
| 103         | Carol White   | carol.w@business.net  | NA              |

-   **CUSTOMER_ID**: Unique identifier for the customer (fully visible).
-   **CUSTOMER_NAME**: Name of the customer (fully visible).
-   **EMAIL**: Email address of the customer (fully visible as per `ANALYST_ROLE` logic in `email_mask_policy`).
-   **CUSTOMER_REGION**: Geographic region of the customer (fully visible as per `GLOBAL_ADMIN` or equivalent logic, or `ANALYST_ROLE` logic in `region_access_policy` if analyst can see all).

### Example Table (when queried by `SALES_ROLE` in the consumer account, assuming `SALES_ROLE` can only see 'NA' region)

This role has masked emails and filtered rows.

| CUSTOMER_ID | CUSTOMER_NAME | EMAIL               | CUSTOMER_REGION |
|-------------|---------------|---------------------|-----------------|
| 101         | Alice Johnson | ******@example.com  | NA              |
| 103         | Carol White   | ******@business.net | NA              |

-   **CUSTOMER_ID**: Unique identifier for the customer (fully visible).
-   **CUSTOMER_NAME**: Name of the customer (fully visible).
-   **EMAIL**: Email address of the customer (partially masked as per `SALES_ROLE` logic in `email_mask_policy`).
-   **CUSTOMER_REGION**: Geographic region of the customer (filtered to only 'NA' region as per `SALES_ROLE` logic in `region_access_policy`).
