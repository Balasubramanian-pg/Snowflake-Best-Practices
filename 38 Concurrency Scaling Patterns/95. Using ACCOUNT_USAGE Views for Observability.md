# Using ACCOUNT_USAGE Views for Observability in Snowflake

The `ACCOUNT_USAGE` schema in Snowflake provides a set of views that expose metadata about your Snowflake account's usage, performance, and security. It exists to give account administrators and users insights into how their Snowflake resources are being consumed and who is doing what within the account. This is crucial for cost management, performance optimization, and auditing.

**Short Explanation**

`ACCOUNT_USAGE` views are like a detailed logbook for your entire Snowflake account, containing information about queries, storage, users, and more. This SQL feature solves the problem of understanding resource consumption, identifying performance bottlenecks, and maintaining security compliance within Snowflake. It's important for database administrators, cost managers, and security officers to optimize costs, improve query performance, and audit activities. It's commonly used in real workflows for building custom dashboards, reporting on spending, and analyzing query patterns.

- What problem does this SQL feature solve? It solves the problem of gaining comprehensive visibility into Snowflake account activity, resource consumption, and performance, which is not readily available through standard database views.
- Why is it important in databases or analytics? It's important for effective governance, cost control, performance tuning, and security auditing of a Snowflake environment, allowing data professionals to make informed decisions.
- Where is it commonly used in real workflows? It's used for creating custom usage reports, monitoring credit consumption, identifying long-running or expensive queries, tracking data storage growth, and auditing user access patterns.

## Implementation Example

```sql
SELECT
    query_id,
    query_text,
    user_name,
    role_name,
    warehouse_name,
    execution_status,
    total_elapsed_time / 1000 AS total_elapsed_seconds,
    credits_used_cloud_services
FROM
    SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    AND execution_status = 'SUCCESS'
    AND total_elapsed_seconds > 60
ORDER BY
    total_elapsed_seconds DESC
LIMIT 10;
```

## Explanation of the Code

This query retrieves information about the 10 longest-running successful queries from the past 7 days, helping to identify potential performance issues.

- `SELECT`: Specifies the columns to be retrieved from the `QUERY_HISTORY` view.
  - What it does: It selects specific attributes related to each query.
  - How it changes the result set: It defines the schema of the output table.
- `FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY`: Indicates that the data is sourced from the `QUERY_HISTORY` view within the `ACCOUNT_USAGE` schema of the `SNOWFLAKE` shared database.
  - What it does: Specifies the data source.
  - How it changes the result set: It determines which raw data records are considered for the query.
- `WHERE`: Filters the records based on specified conditions.
  - `start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())`: Filters for queries that started within the last 7 days.
  - `execution_status = 'SUCCESS'`: Includes only queries that completed successfully.
  - `total_elapsed_seconds > 60`: Filters for queries that took longer than 60 seconds (after converting milliseconds to seconds).
  - What it does: Narrows down the dataset to only relevant queries.
  - How it changes the result set: It removes rows that do not meet the criteria.
- `ORDER BY total_elapsed_seconds DESC`: Sorts the results by the `total_elapsed_seconds` in descending order.
  - What it does: Arranges the filtered rows based on execution time.
  - How it changes the result set: It reorders the rows, placing the longest-running queries at the top.
- `LIMIT 10`: Restricts the output to the top 10 rows after ordering.
  - What it does: Returns only a specified number of rows.
  - How it changes the result set: It truncates the result set to the top 10 longest queries.

## When to Use

1.  **Cost Management and Optimization:** When you need to understand where your Snowflake credits are being consumed. For example, regularly querying `QUERY_HISTORY` to identify expensive warehouses or inefficient queries and then optimizing them to reduce costs.
2.  **Performance Monitoring and Tuning:** To pinpoint slow-running queries or warehouses under stress. For instance, using `QUERY_HISTORY` to find queries exceeding a certain execution time and then analyzing their `query_text` to suggest indexing or rewriting.
3.  **Auditing and Security:** To track user activity, access patterns, and data manipulation for compliance or security investigations. A scenario would be monitoring the `LOGIN_HISTORY` and `QUERY_HISTORY` views to detect unusual login attempts or unauthorized data access.

## When Not to Use

1.  **Real-time Operational Monitoring:** `ACCOUNT_USAGE` views are near real-time, typically updated every few hours. For immediate, second-by-second operational monitoring (e.g., tracking active queries for immediate intervention), a different approach like `INFORMATION_SCHEMA` functions (`TABLE(INFORMATION_SCHEMA.QUERY_HISTORY())`) or Snowflake's web interface is more suitable.
2.  **User-facing Application Data:** These views are for internal account management, not for serving data to external applications or end-users. Using them for this purpose would be inefficient and expose internal metadata.
3.  **Very Granular Performance Debugging:** While `ACCOUNT_USAGE` helps identify slow queries, it doesn't provide the in-depth, step-by-step execution details needed for deep performance debugging (e.g., breakdown of each operator in a query plan). For such detailed analysis, `EXPLAIN` plans and profiling within the Snowflake UI or specific `INFORMATION_SCHEMA` functions are better.

## Common Mistakes

1.  **Over-reliance on `ACCOUNT_USAGE` for real-time issues:** Expecting immediate updates or real-time insights from `ACCOUNT_USAGE` views, leading to delayed responses to critical operational problems.
2.  **Ignoring data retention policies:** `ACCOUNT_USAGE` views have specific data retention periods (e.g., `QUERY_HISTORY` retains data for 365 days). For long-term historical analysis beyond these limits, data needs to be extracted and stored elsewhere.
3.  **Querying without filters:** Running `SELECT *` on large `ACCOUNT_USAGE` views (like `QUERY_HISTORY` without time or status filters) can be expensive and slow, consuming unnecessary warehouse resources. Always filter by time and other relevant criteria.
