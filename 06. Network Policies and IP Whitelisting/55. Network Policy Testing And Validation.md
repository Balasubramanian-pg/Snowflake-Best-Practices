# Network Policy Testing And Validation in Snowflake

Network Policy Testing and Validation in Snowflake involves ensuring that the access rules defined within your network policies function as intended, preventing unauthorized connections while allowing legitimate users and applications to access your Snowflake environment. This is a critical security practice to maintain data integrity and compliance.

**Short Explanation**

This SQL feature ensures that your Snowflake network policies, which restrict access based on IP addresses or network rules, are working correctly. It solves the problem of inadvertently locking out legitimate users or, conversely, leaving your account exposed. It's important for maintaining a secure data environment by verifying that only approved networks can connect, and it's commonly used during initial setup, policy modifications, and security audits to prevent access issues.

- What problem does this SQL feature solve? It prevents unintended access restrictions or security vulnerabilities by verifying network policy configurations before or after implementation.
- Why is it important in databases or analytics? It safeguards sensitive data by ensuring that only authorized network locations can connect to your Snowflake instance, crucial for compliance and data protection.
- Where is it commonly used in real workflows? During the deployment of new network policies, when modifying existing ones, and as part of regular security reviews and audits.

## Implementation Example

```sql
-- Step 1: Create a network rule for allowed IPs (e.g., your office IP range)
CREATE OR REPLACE NETWORK RULE my_security_schema.office_ip_range_allow
  TYPE = IPV4
  VALUE_LIST = ('192.168.1.0/24', '203.0.113.45') -- Example: Office LAN, specific remote IP
  MODE = INGRESS
  COMMENT = 'Allowed IP ranges for corporate office and specific users';

-- Step 2: Create a network policy referencing the allowed network rule
CREATE OR REPLACE NETWORK POLICY my_company_network_policy
  ALLOWED_NETWORK_RULE_LIST = ('my_security_schema.office_ip_range_allow')
  COMMENT = 'Network policy to restrict access to company-approved IPs';

-- Step 3: Activate the network policy for the account
ALTER ACCOUNT SET NETWORK_POLICY = my_company_network_policy;

-- Step 4: Use the Network Policy Advisor to simulate and validate (non-disruptive)
-- First, get historical ingress traffic data
CALL SNOWFLAKE.ACCOUNT_USAGE.RECOMMEND_NETWORK_POLICY();

-- Then, evaluate a candidate policy based on the recommendation or your own definition
-- (Replace 'MY_CANDIDATE_POLICY_NAME' with your desired policy or the one from RECOMMEND_NETWORK_POLICY)
CALL SNOWFLAKE.ACCOUNT_USAGE.EVALUATE_CANDIDATE_NETWORK_POLICY(
    'MY_CANDIDATE_POLICY_NAME', -- Or the name of the policy you want to evaluate
    'ACCOUNT' -- Evaluate for account level
);

-- Step 5: (Optional) Deactivate the policy if issues are found or for maintenance
-- ALTER ACCOUNT UNSET NETWORK_POLICY;
```

## Explanation of the Code

This example demonstrates a typical workflow for creating, activating, and validating a network policy.

- `CREATE OR REPLACE NETWORK RULE my_security_schema.office_ip_range_allow...`:
    - **What it does:** Defines a schema-level object called a "network rule" that encapsulates a list of IP addresses or CIDR blocks.
    - **How it changes the result set:** It doesn't directly change a result set but creates a reusable security object in your Snowflake environment. The `TYPE` specifies `IPV4` for IPv4 addresses, `VALUE_LIST` contains the actual IP addresses or ranges, and `MODE = INGRESS` indicates it applies to incoming connections.
- `CREATE OR REPLACE NETWORK POLICY my_company_network_policy...`:
    - **What it does:** Creates a network policy object. Network policies refer to network rules to define what IP addresses are allowed or blocked.
    - **How it changes the result set:** Similar to network rules, this creates a security object. `ALLOWED_NETWORK_RULE_LIST` specifies which network rules contain the IP addresses permitted to access Snowflake.
- `ALTER ACCOUNT SET NETWORK_POLICY = my_company_network_policy;`:
    - **What it does:** Activates the `my_company_network_policy` at the account level. This makes the policy effective for all users and connections to the Snowflake account unless overridden by a user-specific or integration-specific policy.[AELq]
    - **How it changes the result set:** This is a DDL statement that modifies account parameters; it doesn't return a traditional result set but applies the security control.
- `CALL SNOWFLAKE.ACCOUNT_USAGE.RECOMMEND_NETWORK_POLICY();`:
    - **What it does:** This system stored procedure analyzes historical ingress traffic data and recommends a candidate network policy based on observed connection patterns.
    - **How it changes the result set:** It returns a table with recommended SQL syntax for creating network rules and policies, helping you design effective policies.
- `CALL SNOWFLAKE.ACCOUNT_USAGE.EVALUATE_CANDIDATE_NETWORK_POLICY('MY_CANDIDATE_POLICY_NAME', 'ACCOUNT');`:
    - **What it does:** Simulates the effect of a given candidate network policy (identified by name) on historical traffic without actually activating it.
    - **How it changes the result set:** It returns a table showing which IP addresses would be allowed or blocked by the candidate policy, allowing you to validate its behavior.
- `ALTER ACCOUNT UNSET NETWORK_POLICY;`:
    - **What it does:** Deactivates the network policy set at the account level, reverting to a state where there are no account-level network restrictions.
    - **How it changes the result set:** This DDL statement modifies account parameters.

## When to Use

1. **Initial Deployment of Network Policies:** When you first set up network policies in Snowflake, use testing and validation to ensure that your corporate IP ranges, VPN endpoints, or specific client applications can connect without issues. For example, after creating a policy for your organization's VPN, validate it by attempting connections from within and outside the VPN.
2. **Modifying Existing Network Policies:** Any change to an `ALLOWED_IP_LIST`, `BLOCKED_IP_LIST`, or the `ALLOWED_NETWORK_RULE_LIST`/`BLOCKED_NETWORK_RULE_LIST` should be validated. Use the Network Policy Advisor to simulate changes before applying them to production to prevent accidental lockouts.
3. **Regular Security Audits and Compliance Checks:** Periodically validate network policies to ensure they align with current security requirements and that no new, unauthorized access points have emerged. This might involve reviewing logs in conjunction with policies or re-running simulations.

## When Not to Use

1. **For Outbound Traffic Control:** Network policies in Snowflake primarily control *inbound* access to the Snowflake service. They are not designed to control outbound traffic from Snowflake to external network destinations (e.g., UDFs calling external APIs). For outbound control, use external network access with `EGRESS` mode network rules.
2. **As a Replacement for Other Security Controls:** Network policies are a crucial layer but should not be the sole security measure. They complement other controls like user authentication (MFA), role-based access control (RBAC), and data masking, not replace them.
3. **For Granular Object-Level Access:** Network policies operate at the account, user, or integration level to restrict *connection* based on network origin. They do not control access to specific databases, schemas, tables, or columns within Snowflake; RBAC is used for that.

## Common Mistakes

1. **Accidentally Locking Out All Users:** A common mistake is creating or activating a network policy with an empty `ALLOWED_IP_LIST` (or an `ALLOWED_NETWORK_RULE_LIST` that resolves to no IPs) without including at least one trusted IP address. This can lead to all users, including administrators, being unable to access the account.
2. **Not Including All Necessary IP Ranges:** Forgetting to include all relevant IP addresses or CIDR blocks for legitimate users, applications, or services (e.g., ETL tools, BI platforms) in the allowed list, leading to connectivity issues for those components.
3. **Relying Only on Account-Level Policies for Specific Needs:** While convenient, an account-level network policy applies globally. For specific users or security integrations requiring different access rules, failing to leverage user-level or integration-level network policies can lead to overly permissive or restrictive access for certain groups.[AELq]

## Expected Output

The `EVALUATE_CANDIDATE_NETWORK_POLICY` stored procedure, when used for validation, will return a table indicating the outcome of the simulation. This table helps you understand which IP addresses would be allowed or blocked by the proposed policy.

| IP_ADDRESS | MATCHED_NETWORK_RULE | ALLOWED |
| :--------------- | :------------------- | :------ |
| 192.168.1.10 | OFFICE_IP_RANGE_ALLOW| TRUE |
| 203.0.113.45 | OFFICE_IP_RANGE_ALLOW| TRUE |
| 1.2.3.4 | NULL | FALSE |

**Explanation of Columns:**

- **`IP_ADDRESS`**: The IP address that was simulated against the network policy.
- **`MATCHED_NETWORK_RULE`**: The name of the network rule that this IP address matched, if any. If `NULL`, it means the IP did not match any rule in the policy.
-   **`ALLOWED`**: A boolean value (`TRUE`/`FALSE`) indicating whether the IP address would be allowed access by the network policy. `TRUE` means access is granted, `FALSE` means it would be denied.
