# Using CURRENT_USER in Row Access Policies in Snowflake

This concept involves leveraging the `CURRENT_USER()` function within Snowflake Row Access Policies to dynamically filter rows based on the identity of the user executing a query. It's a powerful security mechanism for implementing fine-grained access control, ensuring that users only see the data they are authorized to view. This prevents unauthorized data exposure and helps in achieving data governance and compliance requirements.

**Short Explanation**

Using `CURRENT_USER()` in Row Access Policies allows you to restrict data visibility at the row level, making access dependent on who is querying the data. This feature solves the problem of needing to apply different security filters for different users on the same table without creating multiple views or tables. It's important in databases and analytics for enforcing data privacy, complying with regulations (like GDPR or HIPAA), and ensuring that sensitive information is only accessible to authorized personnel. It's commonly used in multi-tenant applications, financial services, healthcare, and any scenario where data access needs to be granularly controlled based on user roles or attributes.

- **What problem does this SQL feature solve?** It solves the problem of implementing dynamic row-level security, where data access is controlled based on the identity of the user querying the table, without duplicating data or creating numerous views.
- **Why is it important in databases or analytics?** It's crucial for data privacy, compliance (e.g., GDPR, HIPAA), and ensuring that users only see the data relevant and authorized for them, enhancing data governance.
- **Where is it commonly used in real workflows?** Multi-tenant data platforms, financial reporting, healthcare data analysis, and any application requiring strict, user-specific data isolation.

## Implementation Example

```sql
-- 1. Create a sample table
CREATE OR REPLACE TABLE sales_data (
    sale_id INT,
    product_name VARCHAR,
    region VARCHAR,
    sale_amount DECIMAL(10, 2),
    salesperson_email VARCHAR
);

-- 2. Insert some sample data
INSERT INTO sales_data (sale_id, product_name, region, sale_amount, salesperson_email) VALUES
    (1, 'Laptop', 'East', 1200.00, 'john.doe@example.com'),
    (2, 'Mouse', 'West', 25.00, 'jane.smith@example.com'),
    (3, 'Keyboard', 'East', 75.00, 'john.doe@example.com'),
    (4, 'Monitor', 'Central', 300.00, 'alice.jones@example.com'),
    (5, 'Webcam', 'West', 50.00, 'jane.smith@example.com');

-- 3. Create a Row Access Policy
CREATE OR REPLACE ROW ACCESS POLICY sales_rap
AS (salesperson_email_col VARCHAR) RETURNS BOOLEAN ->
  CURRENT_ROLE() = 'ACCOUNTADMIN' OR salesperson_email_col = CURRENT_USER();

-- 4. Apply the Row Access Policy to the table
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_rap ON (salesperson_email);

-- 5. (Optional) Grant necessary privileges if users are not ACCOUNTADMIN
-- GRANT SELECT ON TABLE sales_data TO ROLE sales_analyst;
-- GRANT APPLY ROW ACCESS POLICY ON ACCOUNT TO ROLE sales_admin;

-- 6. Test as a specific user (e.g., 'john.doe@example.com')
-- This would typically be run by a user logged in as 'john.doe@example.com'
-- SELECT * FROM sales_data;
```

## Explanation of the Code

This example demonstrates how to create a `sales_data` table, define a `sales_rap` Row Access Policy, and apply it to the table, restricting rows based on the `salesperson_email` column and the `CURRENT_USER()`.

- **`CREATE OR REPLACE TABLE sales_data (...)`**:
    - **What it does**: Defines a new table named `sales_data` with columns for sale details and the email of the salesperson.
    - **How it changes the result set**: This clause creates the structure that will hold our data, acting as the base for all subsequent operations.

- **`INSERT INTO sales_data (...) VALUES (...)`**:
    - **What it does**: Populates the `sales_data` table with sample records, including different salespeople emails.
    - **How it changes the result set**: Adds specific rows to the table, which will then be subject to the Row Access Policy.

- **`CREATE OR REPLACE ROW ACCESS POLICY sales_rap AS (salesperson_email_col VARCHAR) RETURNS BOOLEAN -> CURRENT_ROLE() = 'ACCOUNTADMIN' OR salesperson_email_col = CURRENT_USER();`**:
    - **What it does**: This is the core of the dynamic access control. It defines a policy named `sales_rap` that takes a `salesperson_email_col` as input. The policy returns `TRUE` if the current user has the `ACCOUNTADMIN` role, or if the email in the `salesperson_email_col` matches the email of the `CURRENT_USER()` executing the query. If it returns `TRUE`, the row is visible; otherwise, it's hidden.
    - **How it changes the result set**: This clause doesn't directly change a result set, but it defines the logic that will filter rows when applied to a table.

- **`ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_rap ON (salesperson_email);`**:
    - **What it does**: This statement applies the `sales_rap` policy to the `sales_data` table. The `ON (salesperson_email)` clause specifies which column from the `sales_data` table should be passed as the argument `salesperson_email_col` to the policy.
    - **How it changes the result set**: When a user queries `sales_data`, this clause ensures that the `sales_rap` policy is evaluated for each row, effectively filtering the rows returned to the user based on the policy's logic.

- **`CURRENT_USER()`**:
    - **What it does**: This Snowflake function returns the login name of the user who is currently executing the SQL statement.
    - **How it changes the result set**: Within the policy, it provides the dynamic element that allows row visibility to change based on the querying user's identity.

- **`CURRENT_ROLE()`**:
    - **What it does**: This Snowflake function returns the active role for the current session.
    - **How it changes the result set**: In this policy, it allows users with the `ACCOUNTADMIN` role to bypass the row-level filtering, providing a back-door for administrative access.

## When to Use

1.  **Multi-tenant applications**: If you have a single `customers` table containing data for multiple companies, you can use `CURRENT_USER()` in a Row Access Policy to ensure each company's users only see their own customer data.
    *   *Example scenario*: A SaaS provider stores customer records for multiple businesses in one table. A policy could be `customer_id = (SELECT customer_id FROM user_mapping WHERE user_email = CURRENT_USER())`.
2.  **Sensitive data compartmentalization**: In healthcare or finance, where specific users or roles are only allowed to see data related to their patients or clients.
    *   *Example scenario*: A policy on a `patient_records` table ensures that doctors only see records for their assigned patients, or financial advisors only see portfolios of their clients, using `CURRENT_USER()` to match against an assigned ID.
3.  **Regional data access restrictions**: When data access needs to be limited based on the user's region or territory, and users are identified by an email or ID that includes regional information.
    *   *Example scenario*: A `sales_territory` table is filtered so that regional managers only view sales data from their specific region, where the policy checks `region = (SELECT user_region FROM user_profile WHERE user_email = CURRENT_USER())`.

## When Not to Use

1.  **Static filtering requirements**: If data access policies are static and don't depend on the querying user's identity, a simple `WHERE` clause in a view might be more straightforward and easier to manage than a Row Access Policy.
    *   *Situation*: All users should only see "active" records. A view `CREATE VIEW active_records AS SELECT * FROM data WHERE status = 'active';` is sufficient.
2.  **Performance-critical, high-volume transactional tables with complex policies**: While Snowflake optimizes policies, extremely complex policies involving multiple subqueries or joins within the policy, especially on very large tables with high query concurrency, can introduce overhead.
    *   *Situation*: A policy that involves joining several mapping tables to determine user access on a table with billions of rows, frequently updated. A materialized view or pre-filtered tables might be better.
3.  **When column-level security is needed exclusively**: Row Access Policies operate at the row level. If the requirement is to hide specific columns regardless of the row, Column-Level Security (Masking Policies) should be used instead or in conjunction.
    *   *Situation*: Certain users should never see the `social_security_number` column, even if they have access to the row. A masking policy would be more appropriate for this specific column.

## Common Mistakes

1.  **Not testing the policy with different users**: Developers often test policies with `ACCOUNTADMIN` or the user who created the policy, which might have elevated privileges that bypass the policy logic (as seen in our example with `CURRENT_ROLE() = 'ACCOUNTADMIN'`). This can lead to a false sense of security.
2.  **Overlooking performance impacts of complex policies**: Policies with inefficient subqueries or joins can lead to slower query execution, especially on large tables. It's crucial to optimize the policy logic and consider the performance implications.
3.  **Forgetting to grant necessary privileges**: Users need `SELECT` privileges on the table, and potentially `APPLY ROW ACCESS POLICY` on the account/schema/table if they are managing policies, but *not* necessarily on the policy itself to simply query the table. Incorrect grants can lead to "permission denied" errors.
4.  **Misunderstanding `CURRENT_USER()` vs. `CURRENT_ROLE()`**: Confusing these functions can lead to policies that don't enforce the intended access control. `CURRENT_USER()` identifies the logged-in user, while `CURRENT_ROLE()` identifies the active role, which might change during a session.
