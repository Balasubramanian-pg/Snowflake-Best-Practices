# Resource Monitor Access Control in Snowflake

Resource Monitors in Snowflake are essential tools for managing and controlling credit consumption by virtual warehouses. Access control for these monitors determines who can create, view, modify, and assign them, ensuring that cost governance is properly managed within an organization. It's about securing who has the power to define and enforce spending limits.

**Short Explanation**

Resource Monitor Access Control dictates which roles and users can interact with Snowflake Resource Monitors. This ensures that only authorized personnel can set credit quotas, define thresholds for notifications or warehouse suspensions, and assign these monitors to warehouses or the entire account. It helps prevent unauthorized changes to cost management policies and maintains financial accountability.

- **What problem does this SQL feature solve?** It solves the problem of uncontrolled credit spending and ensures that only designated administrators can configure cost governance policies in Snowflake.
- **Why is it important in databases or analytics?** It's crucial for cost management and security, preventing accidental or malicious changes to resource limits that could lead to unexpected cloud expenses or operational disruptions.
- **Where is it commonly used in real workflows?** It's used by Snowflake administrators and financial controllers to implement budget controls, enforce spending policies, and manage the operational efficiency of compute resources.

## Implementation Example

```sql
-- Assume ACCOUNTADMIN role to create the resource monitor
USE ROLE ACCOUNTADMIN;

-- Create a resource monitor
CREATE OR REPLACE RESOURCE MONITOR WEEKLY_BUDGET_MONITOR
    WITH CREDIT_QUOTA = 1000
    FREQUENCY = WEEKLY
    START_DATE = '2026-03-10'
    TRIGGERS
        ON 75 PERCENT DO NOTIFY
        ON 90 PERCENT DO SUSPEND
        ON 100 PERCENT DO SUSPEND_IMMEDIATE;

-- Grant MONITOR privilege to a custom finance role
GRANT MONITOR ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR TO ROLE FINANCE_ANALYST_ROLE;

-- Grant MODIFY privilege to a custom operations role
GRANT MODIFY ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR TO ROLE OPERATIONS_ADMIN_ROLE;

-- Assign the resource monitor to a specific warehouse
-- Note: Only ACCOUNTADMIN or a role with MODIFY privilege on the warehouse can do this
ALTER WAREHOUSE ANALYTICS_WH SET RESOURCE_MONITOR = WEEKLY_BUDGET_MONITOR;

-- Verify privileges (can be run by a role with appropriate monitoring privileges on roles)
SHOW GRANTS ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR;
```

## Explanation of the Code

- `USE ROLE ACCOUNTADMIN;`: This command sets the current context to the `ACCOUNTADMIN` role. This role is typically required to create resource monitors, as they are account-level objects.[UhBL]
- `CREATE OR REPLACE RESOURCE MONITOR WEEKLY_BUDGET_MONITOR...;`: This statement creates a new resource monitor named `WEEKLY_BUDGET_MONITOR` with a credit quota of 1000 credits, resetting weekly, and defines actions (notify, suspend, suspend immediately) at different usage thresholds.
    - `CREATE OR REPLACE RESOURCE MONITOR`: Defines the DDL statement to create or replace a resource monitor.
    - `WITH CREDIT_QUOTA = 1000`: Sets the maximum number of credits allowed for the monitor within the specified frequency.
    - `FREQUENCY = WEEKLY`: Determines how often the credit usage resets (e.g., daily, weekly, monthly, yearly, never).
    - `START_DATE = '2026-03-10'`: Specifies the date from which the monitor starts tracking credit usage.
    - `TRIGGERS ON... DO...`: Defines actions to be taken when credit usage reaches a certain percentage of the quota.
- `GRANT MONITOR ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR TO ROLE FINANCE_ANALYST_ROLE;`: This command grants the `MONITOR` privilege on the `WEEKLY_BUDGET_MONITOR` to the `FINANCE_ANALYST_ROLE`.
    - `GRANT <privilege> ON <object_type> <object_name> TO ROLE <role_name>;`: This is the standard Snowflake syntax for granting privileges.
    - `MONITOR`: Allows the grantee to view the resource monitor's configuration and current usage, but not modify it.
- `GRANT MODIFY ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR TO ROLE OPERATIONS_ADMIN_ROLE;`: This grants the `MODIFY` privilege, allowing `OPERATIONS_ADMIN_ROLE` to alter properties of the resource monitor, such as changing the quota or triggers.
    - `MODIFY`: Enables altering any properties of a resource monitor, like changing the monthly credit quota.
- `ALTER WAREHOUSE ANALYTICS_WH SET RESOURCE_MONITOR = WEEKLY_BUDGET_MONITOR;`: This statement assigns the newly created `WEEKLY_BUDGET_MONITOR` to a specific virtual warehouse called `ANALYTICS_WH`. A warehouse can only be assigned to a single resource monitor below the account level.[XoTL]
    - `ALTER WAREHOUSE`: Modifies properties of a virtual warehouse.
    - `SET RESOURCE_MONITOR =...`: Specifies which resource monitor should manage this warehouse's credit usage.
- `SHOW GRANTS ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR;`: This command displays all grants on the `WEEKLY_BUDGET_MONITOR`, allowing verification of access control settings.
    - `SHOW GRANTS ON...`: Retrieves information about privileges granted on a specific object.

## When to Use

1. **Enforcing Budget Limits:** When an organization needs to strictly control spending on Snowflake compute resources for specific departments, projects, or the entire account.
    * *Example Scenario:* A data science team has a monthly budget for their GPU-enabled warehouse. A resource monitor with `MODIFY` access for their lead and `MONITOR` access for finance is set to suspend the warehouse if 90% of the budget is hit, sending notifications to both.
2. **Delegating Cost Management:** When the `ACCOUNTADMIN` role needs to delegate the responsibility of managing credit consumption to other roles without giving them full administrative privileges.
    * *Example Scenario:* The `ACCOUNTADMIN` creates resource monitors but grants `MODIFY` privilege to `DATA_PLATFORM_ADMIN` role, allowing them to adjust quotas and triggers for various warehouses as needed, while `ACCOUNTADMIN` retains ultimate oversight.
3. **Auditing and Compliance:** To ensure that access to critical cost-control mechanisms is restricted and auditable, aligning with security and governance policies.
    * *Example Scenario:* A compliance officer needs to regularly review who has `MONITOR` or `MODIFY` access on all resource monitors to ensure that changes to cost policies are only made by authorized personnel, and finance can view usage trends.

## When Not to Use

1. **Over-delegation of `ACCOUNTADMIN`:** Avoid granting `ACCOUNTADMIN` to multiple users solely for resource monitor management, as this role has extensive privileges beyond just cost control.
    * *Reason:* The `ACCOUNTADMIN` role has global privileges and should be used sparingly. Granting `MONITOR` or `MODIFY` on specific resource monitors to custom roles is a more secure and granular approach.
2. **Granting `OWNERSHIP` haphazardly:** `OWNERSHIP` privilege gives full control, including the ability to drop the resource monitor. This should be carefully managed.
    * *Reason:* If `OWNERSHIP` is granted to a role that doesn't strictly need it, it introduces a security risk where a resource monitor (and thus cost controls) could be inadvertently or maliciously removed.
3. **Ignoring the role hierarchy for grants:** Directly granting privileges to individual users instead of roles breaks the RBAC model and can lead to privilege sprawl.
    * *Reason:* Snowflake's RBAC model is designed around roles. Granting directly to users makes management complex, auditing difficult, and increases the risk of unintended access or permission conflicts.

## Common Mistakes

1. **Assuming `ACCOUNTADMIN` is the only way to manage:** Many new users assume only `ACCOUNTADMIN` can interact with resource monitors, leading to over-privileging users.
2. **Not granting `MONITOR` privilege:** Users responsible for tracking costs might not have `MONITOR` access, preventing them from viewing resource monitor usage and receiving notifications.
3. **Misunderstanding `MODIFY` vs. `OWNERSHIP`:** Confusing `MODIFY` (alter properties) with `OWNERSHIP` (full control, including dropping) can lead to unintended security vulnerabilities or loss of resource monitors.
4. **Forgetting to assign resource monitors to warehouses:** A common oversight is creating a resource monitor but failing to assign it to the relevant warehouses or the account, rendering it ineffective.
5. **Inconsistent application of access control:** Applying granular access control to other Snowflake objects but neglecting it for resource monitors, creating a gap in governance.

## Expected Output

The `SHOW GRANTS ON RESOURCE MONITOR WEEKLY_BUDGET_MONITOR;` command would return a dataset showing the privileges granted on the specified resource monitor.

| created_on | privilege | granted_on | name | granted_to_type | granted_to_name | grant_option | granted_by |
| :------------------------ | :-------- | :------------- | :-------------------- | :-------------- | :---------------- | :----------- | :------------ |
| 2026-03-14 10:30:00.000 +0000 | OWNERSHIP | RESOURCE MONITOR | WEEKLY_BUDGET_MONITOR | ROLE | ACCOUNTADMIN | true | ACCOUNTADMIN |
| 2026-03-14 10:35:00.000 +0000 | MONITOR | RESOURCE MONITOR | WEEKLY_BUDGET_MONITOR | ROLE | FINANCE_ANALYST_ROLE | false | ACCOUNTADMIN |
| 2026-03-14 10:40:00.000 +0000 | MODIFY | RESOURCE MONITOR | WEEKLY_BUDGET_MONITOR | ROLE | OPERATIONS_ADMIN_ROLE | false | ACCOUNTADMIN |

**Explanation of the columns:**

- `created_on`: Timestamp when the grant was made.
- `privilege`: The specific privilege granted (e.g., `OWNERSHIP`, `MONITOR`, `MODIFY`).
- `granted_on`: The type of object on which the privilege was granted (e.g., `RESOURCE MONITOR`).
- `name`: The name of the object (e.g., `WEEKLY_BUDGET_MONITOR`).
- `granted_to_type`: The type of entity receiving the grant (e.g., `ROLE`).
- `granted_to_name`: The name of the role or user receiving the grant.
- `grant_option`: Indicates if the grantee can further grant this privilege (`true` or `false`).
-   `granted_by`: The role that performed the grant operation.
