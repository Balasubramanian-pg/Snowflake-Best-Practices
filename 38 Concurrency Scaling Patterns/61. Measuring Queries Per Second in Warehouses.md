# Measuring Queries Per Second in Warehouses in Snowflake

Measuring Queries Per Second (QPS) in Snowflake warehouses is a key performance indicator (KPI) used to understand the workload and efficiency of your compute resources. It represents the number of SQL queries executed by a specific warehouse per second, providing insights into how actively the warehouse is being utilized and its capacity to handle concurrent requests. This metric helps in capacity planning, cost optimization, and identifying potential bottlenecks.

**Short Explanation**

QPS indicates how many queries a Snowflake warehouse can process in one second. It addresses the problem of understanding warehouse performance and load, which is crucial for managing costs and ensuring optimal query execution times. This metric is important in database performance monitoring and analytics to scale resources effectively and avoid over-provisioning or under-provisioning. It's commonly used in monitoring dashboards, performance tuning efforts, and capacity planning discussions within Snowflake environments.

- **What problem does this SQL feature solve?** It helps database administrators and performance engineers quantify the workload on a Snowflake warehouse, allowing them to assess its efficiency and identify if it's over or under-utilized.
- **Why is it important in databases or analytics?** Understanding QPS is vital for cost management (paying for compute time), performance tuning (ensuring queries run fast), and resource allocation (sizing warehouses appropriately).
- **Where is it commonly used in real workflows?** It's used in daily performance monitoring, post-incident analysis, capacity planning for peak loads, and optimizing Snowflake credit consumption.

## Implementation Example

```sql
SELECT
    warehouse_name,
    TO_DATE(start_time) AS query_date,
    COUNT(query_id) AS total_queries,
    COUNT(query_id) / SUM(DATEDIFF(SECOND, start_time, end_time)) AS queries_per_second_approx,
    SUM(DATEDIFF(SECOND, start_time, end_time)) AS total_execution_time_seconds
FROM
    snowflake.account_usage.query_history
WHERE
    start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP()) -- Last 7 days
    AND query_type IN ('SELECT', 'INSERT', 'UPDATE', 'DELETE', 'MERGE', 'CREATE', 'ALTER', 'DROP') -- Filter for DDL/DML queries
GROUP BY
    1, 2
ORDER BY
    query_date DESC, warehouse_name;
```

## Explanation of the Code

This query analyzes the `query_history` view to calculate an approximate QPS per warehouse per day.

- `SELECT warehouse_name, TO_DATE(start_time) AS query_date, COUNT(query_id) AS total_queries, COUNT(query_id) / SUM(DATEDIFF(SECOND, start_time, end_time)) AS queries_per_second_approx, SUM(DATEDIFF(SECOND, start_time, end_time)) AS total_execution_time_seconds`:
    - **What it does:** This clause selects the warehouse name, truncates the query start time to just the date, counts the total number of queries, calculates an approximate QPS, and sums the total execution time.
    - **How it changes the result set:** It defines the columns that will be returned in the final output, providing aggregated metrics rather than individual query details.

- `FROM snowflake.account_usage.query_history`:
    - **What it does:** Specifies the source of the data, which is the `query_history` view in the `ACCOUNT_USAGE` schema, containing metadata about all queries executed in the account.
    - **How it changes the result set:** It determines the dataset from which all subsequent operations (filtering, grouping, aggregation) will draw their data.

- `WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP()) AND query_type IN ('SELECT', 'INSERT', 'UPDATE', 'DELETE', 'MERGE', 'CREATE', 'ALTER', 'DROP')`:
    - **What it does:** This clause filters the `query_history` to include only queries from the last 7 days and only specific types of DML/DDL queries, excluding internal Snowflake commands.
    - **How it changes the result set:** It significantly reduces the number of rows being processed, focusing the analysis on recent, user-initiated queries.

- `GROUP BY 1, 2`:
    - **What it does:** Groups the results by `warehouse_name` and `query_date`.
    - **How it changes the result set:** It aggregates the `COUNT` and `SUM` functions for each unique combination of warehouse and date, allowing for daily QPS calculation per warehouse.

- `ORDER BY query_date DESC, warehouse_name`:
    - **What it does:** Sorts the final result set by date in descending order, then by warehouse name.
    - **How it changes the result set:** Organizes the output for easier readability, showing the most recent data first.

## When to Use

1.  **Capacity Planning:** When you need to decide if your current warehouse size is sufficient for anticipated future load or if you need to scale up/down.
    *   *Scenario:* Reviewing QPS trends before a major marketing campaign or a data migration to ensure the warehouse can handle the expected surge in queries.
2.  **Cost Optimization:** To identify warehouses that might be over-provisioned (low QPS for their size) or under-provisioned (high QPS leading to queuing).
    *   *Scenario:* Analyzing QPS over a month to see if a large warehouse could be resized to a smaller one during off-peak hours, saving Snowflake credits.
3.  **Performance Monitoring & Troubleshooting:** As a baseline metric to detect unusual activity or performance degradation. A sudden drop in QPS on a busy warehouse might indicate an issue.
    *   *Scenario:* A support team notices a significant drop in application performance, and checking QPS for the associated warehouse reveals an unexpected decrease, prompting further investigation.

## When Not to Use

1.  **For precise, real-time QPS:** The `ACCOUNT_USAGE` views have a latency of up to 3 hours. For real-time monitoring, Snowflake's `SYSTEM$GET_QUERY_OPERATOR_STATS` or third-party monitoring tools are better.
    *   *Situation:* Monitoring a critical application where immediate alerts on QPS spikes or drops are required.
2.  **When focusing solely on query complexity:** QPS doesn't differentiate between simple and complex queries. A high QPS could mean many simple queries, not necessarily a high compute load.
    *   *Situation:* Trying to understand why a warehouse is consuming many credits when QPS is only moderately high; in this case, average query execution time or scan rates might be more relevant.
3.  **To attribute QPS to individual users/roles directly from this query:** While `query_history` contains user information, this aggregated QPS calculation doesn't easily break down by user or role without further modifications.
    *   *Situation:* Needing to identify which specific users or applications are generating the most queries to understand workload distribution.

## Common Mistakes

1.  **Ignoring `query_type` filters:** Not filtering `query_history` by `query_type` can include internal Snowflake commands, making the QPS misleadingly high and not reflective of user-initiated workload.
2.  **Misinterpreting average execution time as total duration:** The `DATEDIFF` calculation used for the QPS denominator needs to be `SUM(DATEDIFF(SECOND, start_time, end_time))` (total seconds queries were actively running) and not just `DATEDIFF` between the earliest and latest query.
3.  **Relying on `ACCOUNT_USAGE` for real-time data:** As mentioned, `ACCOUNT_USAGE` has latency. Assuming the data is perfectly up-to-the-minute can lead to incorrect immediate decisions.
4.  **Not considering warehouse size:** A higher QPS on a smaller warehouse might indicate it's working harder and efficiently, while the same QPS on a much larger warehouse might suggest underutilization. QPS needs context.

## Expected Output

The resulting dataset will show the daily approximate Queries Per Second for each Snowflake warehouse, along with the total number of queries and total execution time for that day.

| WAREHOUSE_NAME | QUERY_DATE | TOTAL_QUERIES | QUERIES_PER_SECOND_APPROX | TOTAL_EXECUTION_TIME_SECONDS |
|----------------|------------|---------------|---------------------------|------------------------------|
| ANALYTICS_WH   | 2026-03-13 | 15000         | 0.50                      | 30000                        |
| REPORTING_WH   | 2026-03-13 | 5000          | 0.25                      | 20000                        |
| ETL_WH         | 2026-03-12 | 20000         | 0.67                      | 30000                        |

- **WAREHOUSE_NAME:** The name of the Snowflake virtual warehouse.
- **QUERY_DATE:** The specific date for which the metrics are calculated.
- **TOTAL_QUERIES:** The sum of all qualifying queries executed by that warehouse on that date.
- **QUERIES_PER_SECOND_APPROX:** The calculated approximate queries per second for that warehouse on that date.
- **TOTAL_EXECUTION_TIME_SECONDS:** The total cumulative execution time (in seconds) of all qualifying queries run by that warehouse on that date.
