# Tracking Masked Queries Using QUERY_HISTORY in Snowflake

This concept explains how to identify and monitor queries that interact with masked data in Snowflake using the `QUERY_HISTORY` view. Data masking is a crucial security feature, and tracking its usage helps ensure compliance and understand data access patterns.

**Short Explanation**

This SQL feature helps administrators understand when sensitive, masked data is being accessed. It's important for auditing and security compliance, ensuring that data masking policies are effectively enforced and monitored. You would commonly use this in security audits, compliance reporting, and monitoring data access for sensitive information.

- What problem does this SQL feature solve? It solves the problem of not knowing which queries are accessing columns protected by data masking policies.
- Why is it important in databases or analytics? It's vital for data governance, regulatory compliance (like GDPR, HIPAA), and maintaining the security of sensitive data.
- Where is it commonly used in real workflows? Security and compliance teams use it for auditing data access, identifying potential policy violations, and reporting on data protection measures.

## Implementation Example

```sql
SELECT
    query_id,
    query_text,
    user_name,
    start_time,
    end_time,
    execution_status,
    masked_data_access_status -- This is the key column for tracking masked data access
FROM
    snowflake.account_usage.query_history
WHERE
    start_time >= DATEADD(hour, -24, CURRENT_TIMESTAMP())
    AND masked_data_access_status IS NOT NULL
ORDER BY
    start_time DESC;
```

## Explanation of the Code

This query retrieves information about recent queries from the `QUERY_HISTORY` view, specifically filtering for those that interacted with masked data.

- **`SELECT query_id, query_text, user_name, start_time, end_time, execution_status, masked_data_access_status`**: This clause specifies the columns we want to retrieve.
    - `query_id`: A unique identifier for each query.
    - `query_text`: The actual SQL text of the executed query.
    - `user_name`: The user who executed the query.
    - `start_time`: When the query execution began.
    - `end_time`: When the query execution finished.
    - `execution_status`: The final status of the query (e.g., SUCCESS, FAILED).
    - `masked_data_access_status`: This column is crucial. It indicates if the query accessed data protected by a masking policy. It can show values like 'SUCCESSFUL_MASKING_POLICY_APPLIED' or 'NO_MASKING_POLICY_APPLIED' (if the query accessed a column that *could* have been masked but wasn't, or it accessed a masked column for which the user had the unmasked view). For our purpose, we look for non-NULL values to find relevant interactions.
- **`FROM snowflake.account_usage.query_history`**: This specifies the source of our data, which is the `QUERY_HISTORY` view in the `ACCOUNT_USAGE` schema. This view provides a comprehensive record of all executed queries in your Snowflake account.
- **`WHERE start_time >= DATEADD(hour, -24, CURRENT_TIMESTAMP())`**: This filters the results to only include queries executed within the last 24 hours, making the analysis more current and manageable.
- **`AND masked_data_access_status IS NOT NULL`**: This is the core filter for our goal. It narrows down the results to only those queries where the `masked_data_access_status` column has a value, indicating that the query *interacted* with a column that either had a masking policy applied or was evaluated against one. This helps identify actual masking policy engagements.
- **`ORDER BY start_time DESC`**: This sorts the results by the query start time in descending order, showing the most recent queries first.

## When to Use

1.  **Auditing and Compliance:** To regularly check if sensitive data is being accessed as expected and if masking policies are correctly applied.
    *   *Example Scenario:* A monthly compliance report requires showing that all access to the `SSN` column in the `EMPLOYEE` table resulted in masked data being returned, except for authorized HR personnel.
2.  **Troubleshooting Masking Policies:** To diagnose why a masking policy might not be behaving as expected or why a user is seeing unmasked data when they shouldn't.
    *   *Example Scenario:* A user reports seeing unmasked credit card numbers. You can query `QUERY_HISTORY` to see their recent queries, the `masked_data_access_status`, and the exact `query_text` to understand how the data was accessed.
3.  **Security Monitoring:** To proactively monitor for unusual or unauthorized access patterns involving masked data.
    *   *Example Scenario:* An alert system triggers if an unusually high number of queries from a non-privileged user show `masked_data_access_status` indicating an attempt to access masked data, even if the masking was successful.

## When Not to Use

1.  **Real-time Enforcement:** `QUERY_HISTORY` is for auditing past queries, not for actively enforcing or preventing access in real-time. Masking policies themselves handle real-time enforcement.
    *   *Inefficient:* Trying to use `QUERY_HISTORY` to block a query *as it's happening* based on masking status.
2.  **Performance Tuning of Masking Policies:** While you can see query execution times, `QUERY_HISTORY` doesn't provide granular details about the performance impact *specifically* of the masking policy itself.
    *   *Inefficient:* Relying solely on `QUERY_HISTORY` to optimize the UDFs (User-Defined Functions) used within your masking policies.
3.  **Detailed Column-Level Masking Information:** `masked_data_access_status` tells you if masking *occurred* for the query, but not *which specific columns* were masked within that query or the exact masking function applied to each column.
    *   *Inefficient:* Trying to determine the masking logic applied to `column_A` versus `column_B` within a single query from this view alone.

## Common Mistakes

1.  **Ignoring `masked_data_access_status IS NOT NULL`**: Only filtering by columns that have masking policies defined, without checking this status, will return all queries on those columns, not just those where masking was relevant.
2.  **Not understanding the difference between `NULL` and 'NO_MASKING_POLICY_APPLIED'**: A `NULL` value means the query did not interact with any column that has a masking policy applied. 'NO_MASKING_POLICY_APPLIED' means the query *did* interact with a column that *could* have been masked, but for some reason (e.g., policy is inactive, or the user is the policy owner), the masking policy was not applied to the result for that user.
3.  **Using `ACCOUNT_USAGE` for high-frequency checks**: `ACCOUNT_USAGE` views have a latency of up to 3 hours. For near real-time monitoring, `INFORMATION_SCHEMA.QUERY_HISTORY` is available (with different columns and retention), but even that is not truly real-time.
4.  **Misinterpreting `masked_data_access_status` as an error indicator**: A 'SUCCESSFUL_MASKING_POLICY_APPLIED' status is a positive indication that the policy worked as intended, not a sign of an issue.

## Expected Output

The resulting dataset will be a historical record of queries, specifically highlighting those that interacted with masked data. Each row represents a single query.

| QUERY_ID                             | QUERY_TEXT                               | USER_NAME  | START_TIME                | END_TIME                  | EXECUTION_STATUS | MASKED_DATA_ACCESS_STATUS        |
| :----------------------------------- | :--------------------------------------- | :--------- | :------------------------ | :------------------------ | :--------------- | :------------------------------- |
| 123a...                              | SELECT ssn FROM employee_data;           | analyst_01 | 2026-03-13 10:00:00.000 +0000 | 2026-03-13 10:00:00.100 +0000 | SUCCESS          | SUCCESSFUL_MASKING_POLICY_APPLIED |
| 456b...                              | SELECT * FROM sensitive_table;           | hr_admin   | 2026-03-13 10:05:00.000 +0000 | 2026-03-13 10:05:00.250 +0000 | SUCCESS          | NO_MASKING_POLICY_APPLIED        |
| 789c...                              | SELECT name, address FROM customer_info; | auditor_01 | 2026-03-13 10:15:00.000 +0000 | 2026-03-13 10:15:00.080 +0000 | SUCCESS          | SUCCESSFUL_MASKING_POLICY_APPLIED |

- `QUERY_ID`: A unique identifier for the query.
- `QUERY_TEXT`: The actual SQL command executed.
- `USER_NAME`: The user who ran the query.
- `START_TIME`, `END_TIME`: Timestamps for when the query started and finished.
- `EXECUTION_STATUS`: Indicates if the query completed successfully or failed.
- `MASKED_DATA_ACCESS_STATUS`: This column clearly shows if a masking policy was applied (`SUCCESSFUL_MASKING_POLICY_APPLIED`) or not applied, despite interacting with a masked column (`NO_MASKING_POLICY_APPLIED`).
